3   The Token System

What Is a Token?

A token is the fundamental unit of meaning produced by lexical analysis. Instead of passing raw characters from the lexer to the parser, a compiler passes a stream of tokens — atomic units that represent keywords, identifiers, operators, literals, and punctuation. The full set of tokens a language defines is its lexical vocabulary.

Every token carries at least two pieces of information:

  • Kind: what kind of token it is (a keyword, an operator, an identifier, etc.)
  • Span: where in the source it came from (byte offsets and line/column positions)

Some tokens also carry a value — the actual text of an identifier, the digits of a number literal, the content of a string. The token kind tells the parser what to expect next; the span tells it where to report errors; the value tells it the concrete data.

Source: let x: i32 = 42;

Token stream:
Let Ident("x") Colon I32 Eq NumLiteral("42") Semicolon Eof

Notice what happened: whitespace is gone, let became a keyword token (not an identifier), i32 became a primitive-type token (not an identifier), and 42 became a number literal with its raw text preserved. The parser will work with these 8 tokens rather than 17 raw characters.

Tradeoffs & Design Space

Keywords: Enum Variants vs. Keyword Table

Every programming language has a fixed vocabulary of reserved words — keywords that have special meaning and cannot be used as identifiers. A compiler designer must decide when keywords are recognized:

  • Lexer-based keyword recognition: The lexer identifies keywords during tokenization and produces distinct token variants (e.g., Token::Func vs. Token::Ident("main")). This is the most common approach in hand-written compilers. It simplifies the parser because the parser matches on fixed enum variants rather than comparing string values.
  • Parser-based keyword recognition: The lexer produces Ident("func") for everything, and the parser matches the identifier string against a keyword table. This is more flexible for languages where keyword status depends on context, but adds string comparisons to the parser and makes keyword errors harder to produce.

Most modern compilers, including Ignis, use the lexer approach: a dedicated function maps every recognized keyword string to its token variant. Anything that doesn't match falls through to Token::Ident.

Spanned Tokens: Bundled vs. Separate

Source locations can be attached to tokens in two ways:

  • Bundled: Every token struct contains a Span field. This is ergonomic — every token knows where it came from without additional bookkeeping.
  • Separate array: Tokens and spans are stored in parallel arrays, or the parser queries a separate position map. This can save memory when spans are sparse, but adds complexity.

Ignis bundles them into a SpannedToken struct.

Token Values: Stored Strings vs. Interned IDs

Identifier and literal tokens carry a payload. The classic approaches:

  • Owned strings (String): Simple, self-contained, no lifetime management. Each token owns its value. This is what Ignis uses.
  • Source slices (&'a str): Zero-copy, but ties the token's lifetime to the source string. More memory-efficient but more complex.
  • Interned strings (Symbol): Each unique string gets a numeric ID. Fast equality checks and low memory, but requires a global string interner.

Ignis uses owned String values for simplicity — the compiler is not yet performance-critical enough to justify interning.

Operators: Distinct Variants vs. Generic

Operators can be encoded as Token::Plus, Token::Minus, etc. (one variant per operator) or as Token::Op("+") (a single variant with a string payload). The distinct-variant approach enables exhaustive matching in the parser and better compile-time checking of operator handling. Ignis uses distinct variants for every operator.

Ignis Implementation: The Token Enum

Ignis defines every token variant in a single Token enum in src/token.rs. Let's walk through the categories:

src/token.rs — Token enum (keywords)
pub enum Token {
    // Keywords
    Func, Let, Return,
    If, Else, While, For,
    Struct, Enum, Spec, Impl,
    Import, Asm, Cast,
    Pub, Static, Typedef, Noprefix,
    True, False,

Every keyword in the language gets its own variant. When the parser sees Token::Func, it knows immediately that a function declaration is beginning — no string comparison needed.

Primitive types are also encoded as tokens rather than as identifiers that get resolved later:

src/token.rs — Primitive type tokens
    // Primitive types
    Void, Bool, Char,
    Usize, Isize,
    U64, I64, U32, I32,
    U16, I16, U8, I8,

This is a design choice worth noting: i32 is not an identifier — it's a built-in primitive type. Encoding it as a distinct token means the parser can handle type annotations without consulting a symbol table. It also means i32 cannot be used as a variable name, which matches most languages' treatment of primitive type names.

Literal tokens carry their values as payloads:

    // Literals
    NumLiteral(String),
    StrLiteral(String),
    CharLiteral(u8),

    // Identifiers
    Ident(String),

The NumLiteral stores the raw digit string ("42", "0x1A", "1_000") without parsing it into a number. The StrLiteral stores the resolved content (escape sequences already processed). The CharLiteral stores a single byte. The Ident variant stores the full identifier text.

Operators use a flat set of variants — no nesting, no generic Op("+"):

    // Operators
    Plus, Minus, Star, Slash, Percent,
    Amp, Pipe, Caret, Tilde, Bang,
    Eq, EqEq, BangEq,
    Lt, Gt, LtEq, GtEq,
    AmpAmp, PipePipe,
    LtLt, GtGt,
    Arrow, Dot, At,

Notice that -> (arrow) gets its own token rather than being two separate tokens Minus Gt. This is important: the lexer does the work of recognizing multi-character operators so the parser doesn't have to re-assemble them.

Delimiters and structural punctuation:

    // Delimiters
    OpenParen, CloseParen,
    OpenBrace, CloseBrace,
    OpenBracket, CloseBracket,
    Semicolon, Comma, Colon,

    // Special
    Eof,
}

Token::Eof is not optional — it is always the final token in every token stream, including for empty source files. This eliminates special-case null checks in the parser: every iteration of the parse loop gets a valid token to examine.

The SpannedToken Wrapper

A token without source location information is nearly useless for error reporting. Ignis pairs every token with its Span in a SpannedToken struct:

src/token.rs — SpannedToken
pub struct SpannedToken {
    pub token: Token,
    pub span: Span,
}

impl SpannedToken {
    pub fn new(token: Token, span: Span) -> Self {
        SpannedToken { token, span }
    }
}

This is intentionally minimal — a simple bundle of (Token, Span) with a constructor. The lexer fills in the span after consuming characters, and the parser accesses spanned_token.token and spanned_token.span directly.

The Keyword Classifier: keyword_token()

The bridge between the lexer's raw identifier string and the token enum is a single function. It is the single source of truth for what counts as a keyword:

src/token.rs — keyword_token()
pub fn keyword_token(word: &str) -> Option<Token> {
    match word {
        "func" => Some(Token::Func),
        "Func" => Some(Token::Func),
        "let" => Some(Token::Let),
        "return" => Some(Token::Return),
        "if" => Some(Token::If),
        "else" => Some(Token::Else),
        "while" => Some(Token::While),
        "for" => Some(Token::For),
        "struct" => Some(Token::Struct),
        "enum" => Some(Token::Enum),
        "spec" => Some(Token::Spec),
        "impl" => Some(Token::Impl),
        "import" => Some(Token::Import),
        "asm" => Some(Token::Asm),
        "cast" => Some(Token::Cast),
        "pub" => Some(Token::Pub),
        "static" => Some(Token::Static),
        "typedef" => Some(Token::Typedef),
        "noprefix" => Some(Token::Noprefix),
        "true" => Some(Token::True),
        "false" => Some(Token::False),
        "void" => Some(Token::Void),
        "bool" => Some(Token::Bool),
        "char" => Some(Token::Char),
        "usize" => Some(Token::Usize),
        "isize" => Some(Token::Isize),
        "u64" => Some(Token::U64),
        "i64" => Some(Token::I64),
        "u32" => Some(Token::U32),
        "i32" => Some(Token::I32),
        "u16" => Some(Token::U16),
        "i16" => Some(Token::I16),
        "u8" => Some(Token::U8),
        "i8" => Some(Token::I8),
        _ => None,
    }
}

This function is a pure lookup: given a string, return the token variant or None. A few notable properties:

  • Case-sensitive matching: "func" and "Func" both map to Token::Func. This is explicit — only specific case variants are recognized.
  • Primitive types are keywords: "i32", "bool", etc. are classified here rather than in the lexer's identifier logic. This keeps the type-classification logic in one place.
  • No fallthrough to Ident: The function returns Option, leaving the fallback decision to the caller. The lexer calls keyword_token(&ident).unwrap_or(Token::Ident(ident)) — if it's not a keyword, it's an identifier.
The keyword function as an API boundary: keyword_token is the only function the lexer imports from the token module. Adding a new keyword requires exactly two changes: (1) add a variant to the Token enum, and (2) add a match arm to keyword_token. The lexer never needs to change. This is a clean separation of concerns between the token vocabulary and the tokenization algorithm.

Example: Tracing a Source Line Through the Token System

Consider what happens when the lexer encounters this source line:

let counter: i32 = 42;

The token system classifies each piece as follows:

Source Textkeyword_token resultFinal TokenWhy
letSome(Token::Let)LetMatches the keyword table
counterNoneIdent("counter")Not a keyword → fall through to Ident
:N/AColonSingle-character delimiter, lexed directly
i32Some(Token::I32)I32Matches the keyword table as a primitive type
=N/AEqSingle-character operator
42N/ANumLiteral("42")Starts with digit → lexed as number, not identifier
;N/ASemicolonSingle-character delimiter

The key insight: the lexer doesn't need to know what let means. It just knows that the string "let" corresponds to the Let token variant. The semantic meaning — that let introduces a variable declaration — is entirely the parser's domain.

Token Enum Design Patterns

Derived Traits

#[derive(Debug, Clone, PartialEq)]
pub enum Token { ... }

Three derives cover the needs of the entire compiler pipeline:

  • Debug: Enables format!("{:?}", token) for debugging and error messages.
  • Clone: Tokens need to be clonable because the parser may need to inspect a token, then put it back (for backtracking or lookahead).
  • PartialEq: Enables direct equality comparisons like token == Token::Eof, which is essential for parser control flow.

The SpannedToken Test

#[test]
fn test_spanned_token() {
    let span = Span::new(0, 4, 1, 1);
    let st = SpannedToken::new(Token::Func, span);
    assert_eq!(st.token, Token::Func);
    assert_eq!(st.span, span);
}

This tests the basic bundling invariant: a SpannedToken correctly associates its token variant and source location.

Testing the Keyword Table

#[test]
fn test_keywords() {
    assert_eq!(keyword_token("func"), Some(Token::Func));
    assert_eq!(keyword_token("let"), Some(Token::Let));
    // ... every keyword tested
}

#[test]
fn test_non_keyword() {
    assert_eq!(keyword_token("foo"), None);
    assert_eq!(keyword_token("bar"), None);
}

The test module verifies every keyword mapping and confirms that non-keyword strings return None. This is defense-in-depth: if someone adds a new keyword variant to the enum but forgets to add it to keyword_token, the keyword would silently be treated as an identifier at runtime. The tests catch that.

Key Takeaways

  1. Tokens are the parser's vocabulary. The parser never sees raw characters — only atomic token units with kind, value, and location.
  2. Keyword classification happens in the lexer, through a single function that maps strings to enum variants. This keeps the parser free of string comparisons.
  3. Every token carries its span via the SpannedToken wrapper. No token is ever created without source location information.
  4. Tokens use owned data (String, u8) rather than references or interned IDs. This is simple, correct, and sufficient for a compiler of this scale.
  5. The Token enum is flat — distinct variants for every keyword, operator, and delimiter. This maximizes the type system's ability to check exhaustiveness in match statements throughout the compiler.
