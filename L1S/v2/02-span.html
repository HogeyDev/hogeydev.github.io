2   Source Locations — Span

Why Every Token Needs a Location

When a compiler reports an error, it must tell the programmer exactly where the problem is. A message like "type mismatch" is useless; "type mismatch at line 14:23" is actionable. This requirement means that every token and every AST node must carry its source location—the position in the original source file where it came from.

Source locations serve two purposes:

  • Error reporting: Pointing to the exact region of code that caused the error.
  • Debug info: Mapping generated code back to source lines (DWARF, line tables).

Even a compiler that never produces errors (theoretically impossible) needs source locations for debugging output. They are the foundation of the entire diagnostic system.

Representing a Source Location

There are several common approaches to representing source locations:

  • Byte offsets only (e.g., start: usize, end: usize). Minimum information needed to extract the relevant source text. Used by many compilers internally. Fast, compact, but not human-readable.
  • Line and column only. Human-readable, but ambiguous if the same text appears on multiple lines. Cannot reliably extract the source snippet without reprocessing.
  • Byte offsets + line/col (the most common approach). Stores both machine-oriented offsets and human-oriented line/column numbers. Slightly larger struct but avoids recomputing line/col from offsets (which requires scanning the source).
  • File + offset. For multi-file compilers, you also need a file ID or path.

Ignis uses the third approach: store byte offsets for machine use and line/col for display. This is the same design as rustc's Span (though rustc adds file IDs for incremental compilation).

Design Decisions for Span

A few questions every compiler designer must answer about source locations:

0-based vs 1-based? Line and column numbers are conventionalized as 1-based in virtually every editor and compiler. The first character of the first line is line 1, col 1. Byte offsets remain 0-based because they are array indices into the source string.

Start and end, or start and length? Storing start and end is simpler for slicing and error highlighting (you just do &source[span.start..span.end]). Storing start and length is slightly more compact. Ignis uses start/end.

What about multi-file programs? L1S is currently a single-file compiler. If multi-file support were added, the Span would need a file ID field. The rest of the pipeline would be unaffected because spans are opaque—they are created by the lexer and consumed by the diagnostic system.

Copy vs move semantics? Source locations are tiny (four integers) and are copied constantly. The span type should be Copy to avoid unnecessary clone calls.

The Span Struct in Ignis

Ignis bundles these four values into a Span struct. Here is how the general pattern of a source location plays out in Ignis:

src/span.rs — the Span struct
pub struct Span {
    pub start: usize,   // byte offset of the first character
    pub end: usize,     // byte offset past the last character
    pub line: u32,      // 1-based line number
    pub col: u32,       // 1-based column number
}

Each field has a precise role:

  • start and end are byte offsets into the source string. The substring &source[span.start..span.end] gives you the exact source text for error underlining.
  • line and col are for human-readable messages. They are 1-based, matching common editor convention. line: 1, col: 1 means "the very first character of the file."
Why u32 for line/col? A file would need to be 4 billion lines or columns long to overflow a u32. usize for offsets is necessary because strings can be arbitrarily large on 64-bit systems. Using a smaller type for line/col keeps the struct compact (16 bytes vs 24 or 32).

Construction and Constants

The general pattern for creating spans is through constructors. Ignis provides two: one for real positions and one for synthetic/"unknown" positions:

src/span.rs — constructors
impl Span {
    pub fn new(start: usize, end: usize, line: u32, col: u32) -> Self {
        Span { start, end, line, col }
    }

    pub fn dummy() -> Self {
        Span { start: 0, end: 0, line: 0, col: 0 }
    }
}

Span::new is the workhorse constructor, called by the lexer every time it emits a token. Span::dummy is useful for built-in or synthetic AST nodes that don't correspond to any source text—for example, implicit return values or compiler-generated temporaries. A dummy span has all zeros, which is distinguishable from any real span (real spans start at line ≥ 1, col ≥ 1).

Never display a dummy span. Code that prints spans should check for dummy and display something like "<built-in>" instead of "0:0". Ignis currently does not have this check, so a bug that leaks a dummy span would display as the confusing "0:0".

Merging Spans: The merge Function

When constructing a larger AST node from smaller pieces, you often want the span of the whole node to cover all its children. For example, a binary expression a + b should have a span starting at a and ending at b, so error messages about the whole expression can underline both operands and the operator.

This is called span merging, and it is a common utility:

src/span.rs — merge
    pub fn merge(a: Span, b: Span) -> Span {
        let start = a.start.min(b.start);
        let end = a.end.max(b.end);
        let line = a.line.min(b.line);
        let col = if a.line < b.line { a.col } else { a.col.min(b.col) };
        Span { start, end, line, col }
    }

The logic:

  • start = the minimum of the two start offsets (earliest byte).
  • end = the maximum of the two end offsets (latest byte).
  • line = the minimum line number (earliest line).
  • col = uses the column from the span on the earlier line, or the minimum column if both are on the same line.

This produces a span that covers both a and b and points to the beginning of the combined region. The column heuristic ensures that when merging two spans on different lines, the merged span shows the beginning of the construct (line 1's column) rather than a random later column.

Why Not Point to the Source Directly?

A common question is: why store byte offsets and line/col at all? Why not store a &str slice directly (a pointer + length into the source)?

The answer is lifetime complexity. A span is used pervasively throughout the compiler—in tokens, AST nodes, error messages, and the diagnostic printer. If spans borrowed the source string, every data structure that contains a span would need a lifetime parameter, making the code dramatically more complex. By storing concrete integers, spans are self-contained, Copy, and lifetime-free. The diagnostic printer can reconstruct the source text by indexing into the original String using the byte offsets.

Display Format

Spans are displayed in the standard line:col format familiar from every compiler:

src/span.rs — Display impl
impl fmt::Display for Span {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}:{}", self.line, self.col)
    }
}

This means println!("{}", span) prints 14:23, which matches the format used by GCC, Clang, and rustc. The byte offsets are not displayed in user-facing messages but are used internally for source slicing in the diagnostic printer.

Derived Traits

The span type derives several traits to make it ergonomic to use throughout the compiler:

src/span.rs — derived traits
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct Span { /* ... */ }
  • Debug: Enables {:?} formatting for debugging.
  • Clone, Copy: Span is a small POD type (four integers). Making it Copy avoids explicit .clone() calls everywhere.
  • PartialEq, Eq: Lets you compare spans, useful in tests and for deduplication.
  • Hash: Enables using spans in hash-based data structures (e.g., caching results keyed by source location).
Copy vs Clone for spans. Because a Span is just 4 integers (16 bytes on 64-bit), copying it is cheaper than a pointer chase for a heap-allocated equivalent. Some compilers use an interned span table (store all spans in an arena and pass indices), but that complexity is unnecessary at this scale. Copy semantics are the right call for a toy compiler.

How Spans Flow Through the Pipeline

Spans are created in the lexer and flow through every subsequent phase:

  1. Lexer creates a Span for each token by recording pos, line, col before consuming the token and again after.
  2. Parser attaches spans to AST nodes. It calls Span::merge to construct spans for composite nodes (e.g., a binary expression's span covers both operands and the operator).
  3. Type checker uses spans from AST nodes when producing type error diagnostics.
  4. Codegen does not use spans directly, but the diagnostic printer uses them to extract source lines for error messages.

In short, a span is created once by the lexer and read-only throughout the rest of the compiler. This immutability is important: spans can be freely shared and copied without worrying about ownership or aliasing.

Testing Span

Span's logic is simple enough that tests are straightforward. Here is how the general pattern of testing a value type plays out in Ignis:

src/span.rs — tests
#[test]
fn test_span_new() {
    let s = Span::new(10, 20, 5, 3);
    assert_eq!(s.start, 10);
    assert_eq!(s.end, 20);
    assert_eq!(s.line, 5);
    assert_eq!(s.col, 3);
}

#[test]
fn test_span_dummy() {
    let s = Span::dummy();
    assert_eq!(s.start, 0);
    assert_eq!(s.end, 0);
}

#[test]
fn test_span_merge() {
    let a = Span::new(5, 10, 2, 3);
    let b = Span::new(15, 25, 4, 7);
    let m = Span::merge(a, b);
    assert_eq!(m.start, 5);  // min of 5, 15
    assert_eq!(m.end, 25);   // max of 10, 25
    assert_eq!(m.line, 2);   // min of 2, 4
}

#[test]
fn test_span_merge_same_line() {
    let a = Span::new(5, 10, 2, 3);
    let b = Span::new(12, 18, 2, 8);
    let m = Span::merge(a, b);
    assert_eq!(m.start, 5);
    assert_eq!(m.end, 18);
    assert_eq!(m.line, 2);  // same line
}

#[test]
fn test_span_display() {
    let s = Span::new(0, 0, 3, 14);
    assert_eq!(format!("{}", s), "3:14");
}

#[test]
fn test_span_copy() {
    let a = Span::new(1, 2, 1, 1);
    let b = a;  // copy, not move
    assert_eq!(a, b);  // both still valid
}

#[test]
fn test_span_merge_reverse_order() {
    let a = Span::new(20, 30, 5, 10);
    let b = Span::new(5, 15, 2, 1);
    let m = Span::merge(a, b);
    assert_eq!(m.start, 5);   // b is earlier
    assert_eq!(m.end, 30);    // a ends later
    // Result: 5..30, line 2
}

The tests cover construction, dummy creation, merge (both same-line and cross-line), display formatting, copy semantics, and order-independence of merge. Running them:

$ cargo test -- span

Common Pitfalls with Source Locations

  1. Off-by-one in byte offsets: The end offset should be the byte after the last character, so that &source[start..end] extracts a contiguous slice. If end points to the last character, the slice is one byte short.
  2. Mixing 0-based and 1-based: Easy to accidentally display 0-based line numbers. Always initialize line/col to 1.
  3. Forgetting to update line on newlines: The column must reset to 1 after a newline, not just keep incrementing.
  4. Merge asymmetry: Make sure Span::merge(a, b) == Span::merge(b, a) for all valid inputs. The Ignis implementation is symmetric because it uses min/max.
