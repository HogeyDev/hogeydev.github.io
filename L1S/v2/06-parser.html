6   The Parser

What is Parsing?

Parsing is the process of converting a flat stream of tokens into a structured tree that represents the grammatical structure of the program. Where the lexer grouped characters into tokens, the parser groups tokens into larger constructs: declarations, statements, and expressions.

There are two main families of parsing algorithms: top-down (recursive descent) and bottom-up (LR, LALR). Recursive descent is the most popular for hand-written compilers because it maps directly to the grammar, gives great error messages, and is straightforward to debug. Bottom-up parsers (like Yacc/Bison) are typically generated from a grammar specification and can handle a wider class of grammars, but produce opaque error messages and are harder to extend.

Within recursive descent, expression parsing has its own sub-problem: handling operator precedence without encoding it in a cascade of functions. Two approaches dominate: Pratt parsing (top-down operator precedence) and precedence climbing. Both use numeric binding powers to decide which operator binds tighter. Pratt is more general — it handles prefix, postfix, and infix operators uniformly — while precedence climbing is simpler but typically only handles infix.

Tradeoffs & Design Space

Recursive descent vs generated parsers:

  • Recursive descent: Each grammar rule becomes a function. Easy to debug (step through in a debugger), easy to extend, trivial to add error recovery. Downside: left-recursive grammar rules cause infinite recursion (must be rewritten as right-recursive loops).
  • Generated (Yacc/Bison, ANTLR): Grammar is declarative; parser is generated. Can handle LR(1) grammars including left recursion. Downside: poor error messages, hard to debug, generated code is opaque, adding error recovery requires grammar-level hacks.

For a pedagogical compiler like L1S, recursive descent is the obvious choice. The grammar is small enough that hand-writing a parser is straightforward, and the error recovery patterns are educational.

Pratt vs precedence climbing for expressions:

  • Pratt: A single loop reads a prefix, then repeatedly checks for infix operators. Each operator has a binding power. Prefix and infix handling are unified. Adding a new operator means adding an entry to the binding power table and a match arm.
  • Precedence climbing: Similar but typically only handles infix operators. Prefix/postfix must be handled separately.
  • Function cascade: The traditional approach where each precedence level is a separate function (parse_add_expr calls parse_mul_expr, etc.). Easy to understand but verbose: adding a new precedence level requires inserting a new function in the chain.

L1S uses a Pratt parser for expressions, combined with recursive descent for declarations and statements. This hybrid is common in production compilers (rustc, GCC, Clang all use variants of this approach).

Ignis Implementation

Ignis uses a hand-written recursive descent parser with a Pratt expression parser. The Parser struct wraps the token stream and a diagnostics accumulator, providing cursor operations that every parse method builds on:

src/parser.rs — Parser struct
pub struct Parser {
    tokens: Vec<SpannedToken>,
    pos: usize,
    diag: Diagnostics,
}

impl Parser {
    pub fn new(tokens: Vec<SpannedToken>, diag: Diagnostics) -> Self {
        Parser { tokens, pos: 0, diag }
    }

    pub fn into_diagnostics(self) -> Diagnostics {
        self.diag
    }
}

The three fields are all a recursive descent parser needs: an owned token list, a cursor into it, and a place to record errors.

Cursor Operations: The Parser's Vocabulary

Every recursive descent parser is built from a handful of primitives for looking at and consuming tokens. These are the building blocks that higher-level parse methods compose:

src/parser.rs — cursor operations
fn is_at_end(&self) -> bool {
    self.peek_token() == Token::Eof
}

fn peek(&self) -> &SpannedToken {
    &self.tokens[self.pos]
}

fn peek_token(&self) -> Token {
    if self.pos >= self.tokens.len() { return Token::Eof; }
    self.tokens[self.pos].token.clone()
}

fn peek_nth(&self, n: usize) -> Token {
    let idx = self.pos + n;
    if idx >= self.tokens.len() { return Token::Eof; }
    self.tokens[idx].token.clone()
}

fn advance(&mut self) -> &SpannedToken {
    let t = &self.tokens[self.pos];
    self.pos += 1;
    t
}

The three consumption patterns — check, match_token, and expect — form a taxonomy of lookahead strategies:

fn check(&self, tok: &Token) -> bool {
    self.peek_token() == *tok         // pure predicate, cursor unchanged
}

fn match_token(&mut self, tok: &Token) -> bool {
    if self.check(tok) { self.advance(); true }       // consume if matches
    else { false }
}

fn expect(&mut self, tok: &Token) -> Result<Span, Span> {
    if self.check(tok) { ... Ok(span) }           // consume or signal error
    else { Err(self.current_span()) }
}

check is a pure predicate — look but don't touch. match_token is for optional syntax (like commas between parameters): consume if present, no error if absent. expect is for required syntax: consume on success, return an error span on failure so the caller can synthesize a placeholder.

Why return Result for expect? The caller uses unwrap_or_else to supply a synthetic value (a dummy span, an empty name, etc.). The error is already recorded in the diagnostics by the caller's choice. This keeps expect pure — it reports the failure; the caller decides how to recover.

Error Recovery Pattern

The most important design principle in the Ignis parser: never stop after one error. Every parse method can fail gracefully by inserting a synthetic value and continuing. Consider parse_func_decl:

fn parse_func_decl(&mut self, pub_: bool) -> AstFuncDecl {
    let span = self.expect(&Token::Func).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenParen).unwrap_or_else(|_| Span::dummy());
    let return_type = self.parse_type();
    let params = if self.match_token(&Token::Comma) { self.parse_params() }
                 else { Vec::new() };
    self.expect(&Token::CloseParen).unwrap_or_else(|_| Span::dummy());
    let body = self.parse_block();
    AstFuncDecl { span, pub_, name, return_type, params, body }
}

If the function name is missing, the parser inserts "_" as a synthetic name and continues. If a closing paren is missing, it gets a dummy span. The error is already recorded in self.diag, but parsing proceeds to find more errors. This maximizes the number of issues reported in a single compilation.

Every loop also has an infinite loop guard. If a bug causes the parser to loop without advancing (e.g., a malformed declaration returns None every iteration), a panic fires after 10,000 iterations:

fn parse_program(&mut self) -> AstProgram {
    let mut decls = Vec::new();
    let mut guard = 0u32;
    while !self.is_at_end() {
        guard += 1;
        if guard > 10000 { panic!("INFINITE LOOP"); }
        if let Some(decl) = self.parse_declaration() {
            decls.push(decl);
        } else {
            self.advance(); // force progress on parse failure
        }
    }
    AstProgram { decls }
}

The guard paired with self.advance() on failure is the double lock. Even if parse_declaration returns None without consuming anything, the force-advance guarantees the loop makes progress.

Declaration Parsing

Declarations are the top-level constructs of a program: functions, structs, enums, specs, impls, typedefs, imports, and static variables. The dispatch is a single match on the current token, with pub handled as an optional prefix modifier:

fn parse_declaration(&mut self) -> Option<AstDecl> {
    let is_pub = self.match_token(&Token::Pub);
    match self.peek_token() {
        Token::Struct   => Some(AstDecl::Struct(self.parse_struct_decl(is_pub))),
        Token::Enum     => Some(AstDecl::Enum(self.parse_enum_decl(is_pub))),
        Token::Spec     => Some(AstDecl::Spec(self.parse_spec_decl(is_pub))),
        Token::Impl     => Some(AstDecl::Impl(self.parse_impl_decl())),
        Token::Func     => Some(AstDecl::Func(self.parse_func_decl(is_pub))),
        Token::Typedef  => Some(AstDecl::Typedef(self.parse_typedef_decl(is_pub))),
        Token::Import   => Some(AstDecl::Import(self.parse_import())),
        Token::Static   => Some(AstDecl::StaticVar(self.parse_static_var_decl())),
        _ => { if is_pub { self.error("expected declaration after 'pub'", ...); } None }
    }
}

The pub keyword is consumed greedily by match_token, then passed to the sub-parser. Declarations like impl and import reject it by emitting an error. This pattern keeps the dispatch clean and avoids duplicating the pub-handling logic in each sub-parser.

Each declaration follows the same template: consume a keyword, parse a name, parse delimiters and body, return the AST node. For example, a struct declaration:

fn parse_struct_decl(&mut self, pub_: bool) -> AstStructDecl {
    let span = self.expect(&Token::Struct).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenBrace).unwrap_or_else(|_| Span::dummy());
    let fields = self.parse_fields();
    self.expect(&Token::CloseBrace).unwrap_or_else(|_| Span::dummy());
    AstStructDecl { span, pub_, name, fields }
}

Ignis uses a non-standard function declaration syntax: func main(i32, argc: i32, argv: @char) { ... }. The return type appears first inside the parentheses, followed by optional named parameters after a comma. This means func main(i32) { } declares a function taking no parameters returning i32, while func main(i32, x: i32, y: i32) { } adds named parameters.

Why return type first? In most C-like languages, a bare type with no name (like i32) would be illegal. Ignis uses this to distinguish between "zero parameters" (just a return type) and "named parameters" (comma, then name:type pairs). It's unusual but keeps the grammar compact.

Statement Parsing

Statements are the executable units inside function bodies. Each statement type starts with a distinct keyword, making dispatch a simple match:

fn parse_statement(&mut self) -> Option<AstStmt> {
    match self.peek_token() {
        Token::Return    => Some(AstStmt::Return(self.parse_return())),
        Token::Let       => Some(AstStmt::VarDecl(self.parse_var_decl())),
        Token::If        => Some(AstStmt::If(self.parse_if())),
        Token::While     => Some(AstStmt::While(self.parse_while())),
        Token::For       => Some(AstStmt::For(self.parse_for())),
        Token::Asm       => Some(AstStmt::Asm(self.parse_asm())),
        Token::OpenBrace => Some(AstStmt::Block(self.parse_block())),
        Token::Import    => Some(AstStmt::Import(self.parse_import())),
        _ => { // expression statement: any expression followed by semicolon
            if self.is_at_end() { return None; }
            let expr = self.parse_expr();
            self.expect(&Token::Semicolon).unwrap_or_else(|_| Span::dummy());
            Some(AstStmt::Expr(expr))
        }
    }
}

The fallthrough case handles expression statements — any expression followed by a semicolon. This is how foo(); (function call as a statement) and x = 42; (assignment as a statement) work. The if/while/let keywords are reserved, so there's no ambiguity: an identifier starting a statement can only be an expression statement.

The else if chain is handled recursively in parse_if:

fn parse_if(&mut self) -> AstIf {
    // ... parse condition and then_block ...
    let else_branch = if self.match_token(&Token::Else) {
        match self.peek_token() {
            Token::If => Some(Box::new(AstStmt::If(self.parse_if()))),
            _ => Some(Box::new(AstStmt::Block(self.parse_block()))),
        }
    } else { None };
    AstIf { span, cond, then_block, else_branch }
}

When else is followed by if, the parser recursively calls parse_if, creating a nested chain. The statement if (a) {} else if (b) {} else {} becomes If(cond=a, else=Some(If(cond=b, else=Some(Block)))).

Pratt Expression Parsing

Expressions are the most elegant part of the parser. Instead of encoding precedence in a cascade of functions, Pratt parsing uses binding powers — numeric values assigned to each operator that determine how tightly it binds its operands:

Operator binding powers (lower = looser, higher = tighter):

= (assignment) ........... 10
|| (logical or) .......... 20
&& (logical and) ......... 30
== != (equality) ........... 40
< > <= >= (comparison) ..... 50
| (bitwise or) .......... 60
^ (bitwise xor) ......... 70
& (bitwise and) ......... 80
<< >> (shift) .............. 90
+ - (addition) ........... 100
* / % (multiplication) .... 110
( [ { . -> (postfix) ...... 130

The core algorithm is just two functions, parse_expr_bp and the binding power table infix_bp:

fn parse_expr(&mut self) -> AstExpr {
    self.parse_expr_bp(0)       // start with minimum binding power 0
}

fn parse_expr_bp(&mut self, min_bp: u8) -> AstExpr {
    let mut lhs = self.parse_prefix();       // literal, ident, unary, parens
    loop {
        let bp = self.infix_bp();             // peek at next operator's power
        if bp == 0 || bp < min_bp { break; } // stop if looser than minimum
        lhs = self.parse_infix(lhs, bp);       // consume operator and RHS
    }
    lhs
}

fn infix_bp(&self) -> u8 {
    match self.peek_token() {
        Token::Eq => 10,
        Token::PipePipe => 20,
        Token::AmpAmp => 30,
        Token::EqEq | Token::BangEq => 40,
        Token::Lt | Token::Gt | Token::LtEq | Token::GtEq => 50,
        Token::Pipe => 60,
        Token::Caret => 70,
        Token::Amp => 80,
        Token::LtLt | Token::GtGt => 90,
        Token::Plus | Token::Minus => 100,
        Token::Star | Token::Slash | Token::Percent => 110,
        Token::OpenParen | Token::OpenBracket | Token::OpenBrace
            | Token::Dot | Token::Arrow => 130,
        _ => 0,  // not an infix operator: stop
    }
}

The algorithm works in a tight loop:

  1. Parse a prefix expression (integer literal, string, identifier, parenthesized expression, unary operator like -, !, @).
  2. Peek at the next token. Look up its binding power. Zero means "not an infix operator" — stop here.
  3. Compare the operator's binding power with the current minimum. If the operator binds tighter (bp >= min_bp), parse it as an infix (which recursively calls parse_expr_bp with a higher minimum for the right-hand side).
  4. If the operator binds looser, stop and return the current lhs.

Left vs right associativity is controlled by the bp value passed to the recursive call:

  • Left-associative operators (+, *, ==): the RHS is parsed with bp + 1, meaning 1 + 2 + 3 becomes ((1 + 2) + 3).
  • Right-associative operators (=): the RHS is parsed with bp (not bp + 1), so x = y = 0 becomes x = (y = 0).

Unary operators (-, !, ~, &, @) all use binding power 120 for their operand — higher than all binary operators. This means -x * y parses as (-x) * y, not -(x * y).

The parse_infix method handles both binary operators and postfix syntax (function calls, array indexing, field access, struct initialization) by matching on the current token:

fn parse_infix(&mut self, lhs: AstExpr, bp: u8) -> AstExpr {
    match self.peek_token() {
        // Postfix operators (binding power 130)
        Token::OpenParen   => { /* function call: foo(args) */ }
        Token::OpenBracket => { /* array index: a[i] */ }
        Token::OpenBrace   => { /* struct init: Point { x: 1 } */ }
        Token::Dot         => { /* field access: p.x */ }
        Token::Arrow       => { /* pointer field: p->x */ }

        // Binary operators follow the same pattern:
        Token::Plus  => { self.advance();
                         let rhs = self.parse_expr_bp(bp + 1);
                         AstExpr::Binary(BinaryOp::Add, Box::new(lhs), Box::new(rhs), op_span) }
        // ... Star, Slash, Percent, Minus, Amp, Pipe, Caret, EqEq, BangEq, ...

        // Assignment is right-associative: use bp, not bp + 1
        Token::Eq   => { self.advance();
                         let rhs = self.parse_expr_bp(bp); // no +1
                         AstExpr::Assign(Box::new(lhs), Box::new(rhs), op_span) }
        _ => lhs,
    }
}

Postfix operators like (, [, . have the highest binding power (130), so they always bind tighter than any binary operator. The expression a[i] + 1 correctly parses as (a[i]) + 1 rather than a[(i) + 1].

Type Parsing

Types form their own mini-language that is parsed by a single recursive method. This is possible because Ignis types use prefix syntax consistently:

fn parse_type(&mut self) -> AstType {
    // [expr] type  → array type (prefix syntax)
    if self.match_token(&Token::OpenBracket) {
        let size = self.parse_expr();
        self.expect(&Token::CloseBracket);
        let inner = self.parse_type();
        return AstType::Array(Box::new(size), Box::new(inner), span);
    }
    // @type  → pointer type (prefix syntax)
    if self.match_token(&Token::At) {
        let inner = self.parse_type();
        return AstType::Ptr(Box::new(inner), span);
    }
    // Func<ret, args...>  → function type
    if self.match_token(&Token::Func) {
        // parse Func<ret_type, arg1_type, arg2_type, ...>
    }
    // Primitive or named type (base case)
    match self.peek_token() {
        Token::I32  => { self.advance(); AstType::Prim(PrimType::I32, span) }
        Token::Bool => { self.advance(); AstType::Prim(PrimType::Bool, span) }
        // ... all primitives ...
        Token::Ident(ref name) => { self.advance(); AstType::Named(name.clone(), span) }
        _ => { self.error("expected type", span); AstType::Prim(PrimType::Void, span) }
    }
}

The recursive calls handle nesting naturally: @[10]@i32 is a pointer to an array of ten pointers to i32. Each prefix operator (@, [n], Func<...>) recursively calls parse_type for the element type. Named types and primitives are the base cases that terminate the recursion.

Traced Example: Parsing 1 + 2 * 3

Let's trace the Pratt parser on 1 + 2 * 3. The token stream: NumLiteral("1"), Plus, NumLiteral("2"), Star, NumLiteral("3").

The key question is: does the parser produce (1 + 2) * 3 or 1 + (2 * 3)? The answer depends on the binding powers of + (100) and * (110). Since * has a higher binding power, it grabs the 2 before + can:

Traced Example: Parsing a Function Declaration

Given func main(i32, argc: i32, argv: @char) { return argc; }:

  1. parse_program loops, calls parse_declaration.
  2. parse_declaration sees no pub, matches Token::Func.
  3. parse_func_decl: consumes func keyword, reads Ident("main").
  4. Consumes OpenParen, calls parse_type() → AstType::Prim(I32).
  5. Sees Comma → calls parse_params which reads argc: i32, argv: @char (two params).
  6. Consumes CloseParen, calls parse_block which parses { return argc; }.
  7. Returns AstFuncDecl { name: "main", return_type: I32, params: [...], body: Block(...) }.

The parser produces a structured tree from the flat token stream:

AstProgram {
    decls: [
        AstFuncDecl {
            name: "main",
            return_type: I32,
            params: [
                AstParam { name: "argc", type_: I32 },
                AstParam { name: "argv", type_: Ptr(Char) },
            ],
            body: AstBlock {
                stmts: [
                    AstStmt::Return(AstExpr::Ident("argc"))
                ]
            }
        }
    ]
}

Traced Example: else if Chaining

Given if (a) { f1(); } else if (b) { f2(); } else { f3(); }:

  1. parse_if consumes if, parses condition a, parses then_block.
  2. Sees else, peeks at next token. It's if (not {), so it recursively calls parse_if.
  3. The recursive call parses if (b) { f2(); } else { f3(); } as a nested AstIf.

The resulting tree: If(cond=a, then=Block(f1), else=If(cond=b, then=Block(f2), else=Block(f3))). This recursive representation is elegant and matches how the parser naturally handles else if without special-casing.

Common Pitfalls

  1. Infinite loops: Every while loop must guarantee progress. Always pair a guard counter with force-advance on failure. Test with using cargo test to ensure the guards don't fire.
  2. Forgetting to advance: check doesn't consume. match_token does. Mixing them up is the most common bug. Use match_token for optional syntax, check for lookahead decisions, expect for required syntax.
  3. Pratt off-by-one: Left-associative operators use bp + 1 for the RHS. Right-associative operators use bp. Getting this wrong silently produces wrong parse trees. Always test with 1 + 2 + 3 (left-associate) and x = y = 0 (right-associate).
  4. Error recovery swallowing tokens: Synthetic values let the parser continue, but the next call might be confused by the missing token. The guard counter catches this. If a test hangs, check the guard fires.
Adding a new statement or operator: You need to touch exactly these places: add a Token variant (if new syntax), add an AstStmt or AstExpr variant, add the parsing method, add an infix_bp entry (for operators), add dispatch in parse_statement or parse_infix. That's it — the Pratt loop handles the rest.
