15   Assembly & Linking

From Assembly Text to Executable Binary

After code generation, the compiler has produced a string of assembly text — human-readable mnemonics like mov, add, and jmp. But this is not yet machine code. Two more tools are needed to turn this text into a running executable: an assembler and a linker.

The assembler translates assembly mnemonics into machine code bytes, producing an object file (.o). The linker resolves symbolic references between object files, relocates code and data to their final memory addresses, and produces an executable binary. Most compilers do not ship their own assembler or linker — instead, they invoke system tools like nasm (for assembly) and ld or cc (for linking).

The assembly-to-executable pipeline:

program.asm           // NASM assembly text (~100 bytes for a trivial program)
    │
    ▼  nasm -f elf64
    │
program.o              // ELF64 object file (machine code + metadata)
    │
    ▼  cc -no-pie
    │
program                // ELF64 executable (runnable binary)

Assemblers: How Text Becomes Machine Code

An assembler performs a conceptually simple translation: each assembly mnemonic and its operands map to a specific sequence of machine code bytes. For example, on x86-64:

AssemblyMachine Code (hex)Meaning
push rbp55Push base pointer onto stack
mov rbp, rsp48 89 e5Copy stack pointer to base pointer
retc3Return from function

Assemblers differ in their approach to this translation:

  • One-pass assemblers: Read the source once, emitting code as they go. Forward references (jumping to a label defined later) require a fixup table that is patched after the first pass. NASM uses this approach.
  • Two-pass assemblers: First pass builds a symbol table of all labels and their offsets; second pass emits code using the now-known addresses. MASM (Microsoft Assembler) uses this.
  • Multi-pass assemblers: Additional passes enable optimizations like jump-sizing (choosing between short and near jumps).

The assembler also produces an object file in a standard format (ELF64 on Linux, Mach-O on macOS, PE on Windows). An object file contains not just machine code, but also metadata: relocation entries (places where addresses need to be filled in), a symbol table (defined and undefined symbols), and section headers.

Linkers: Resolving References

The linker's job is to combine one or more object files into a single executable. It performs two main tasks:

  1. Symbol resolution: Match undefined symbols (e.g., a call to main from the C runtime startup code) to their definitions in other object files or libraries.
  2. Relocation: Adjust the code and data addresses in each object file to reflect the final memory layout of the executable. A jump to main that was assembled as "jump to offset 0" gets patched to the actual address of main in the final binary.

Modern compilers rarely invoke the linker (ld) directly. Instead, they invoke the C compiler frontend (cc or gcc), which knows how to pass the right flags to the linker, including the necessary C runtime startup files (crt0.o, crti.o, crtn.o) that set up _start and call main.

How Ignis Invokes the Toolchain

Ignis's compiler driver (main.rs) handles assembly and linking with three sequential steps: write the assembly file, invoke NASM, and invoke the C compiler as a linker frontend.

src/main.rs — Assembly, linking, and output
    // Step 1: Write .asm file
    fs::write(&asm_path, &asm).unwrap_or_else(|e| {
        eprintln!("error writing {}: {}", asm_path, e);
        std::process::exit(1);
    });

    // Step 2: Assemble with NASM
    let status = Command::new("nasm")
        .args(&["-f", "elf64", "-o", &o_path, &asm_path])
        .status()
        .unwrap_or_else(|e| { /* handle error */ });

    // Step 3: Link with cc
    let status = Command::new("cc")
        .args(&["-o", &bin_path, &o_path, "-no-pie"])
        .status()
        .unwrap_or_else(|e| { /* handle error */ });

Step 1: Writing the Assembly File

The code generator's output (a String) is written to a .asm file alongside the intended output binary. If the input was program.is, the assembly file is program.asm. The .asm file is deliberately kept after compilation so developers can inspect the generated assembly for debugging.

Step 2: Invoking NASM

NASM (Netwide Assembler) is invoked with three flags:

  • -f elf64: Produce an ELF64 object file, the standard format on Linux x86-64.
  • -o <path>: Where to write the output object file.
  • The input .asm file: The assembly source.

The driver uses Command::status() to run NASM and wait for it to complete. If NASM is not installed (unwrap_or_else catches the io::Error) or if NASM reports assembly errors (status.success() returns false), the driver prints an error and exits with code 1.

NASM is required. Install it with your package manager: 
# Debian/Ubuntu
$ sudo apt install nasm
# Fedora
$ sudo dnf install nasm
# macOS
$ brew install nasm

Step 3: Linking with cc

Using cc (the C compiler, typically gcc or clang) as a linker frontend is a common trick in small compilers. The C compiler knows how to invoke the linker (ld) with all the platform-specific flags needed:

  • The correct C runtime startup files (crt0.o, etc.) are linked automatically. These provide the _start entry point that the kernel calls, which in turn calls our main.
  • Standard library paths are set up correctly.
  • The correct dynamic linker (ld-linux-x86-64.so.2) is specified.

Three flags are passed to cc:

  • -o <binary>: Output executable path.
  • <.o file>: Input object file.
  • -no-pie: Disable position-independent executable generation. L1S's generated assembly uses absolute addresses in some cases (e.g., section .data globals), which are incompatible with PIE. Without this flag, the linker would fail with relocation errors.

Step 4 & 5: Cleanup and Confirmation

After successful linking, the intermediate .o object file is deleted. The .asm file is kept. A success message is printed to stdout:

$ ignis program.is -o myprog
wrote myprog

Output Path Resolution

The driver supports two modes for determining the output binary name:

$ ignis program.is                  # produces ./program
$ ignis program.is -o mybinary     # produces ./mybinary

Without -o, the input filename's extension is stripped using Path::with_extension(""). program.is becomes program. Both .asm and .o paths are derived from this base name.

Error Handling in the Driver

The compiler driver must handle failures at every stage. The strategy is straightforward: check for errors after each operation, print a descriptive message to stderr, and exit with code 1. Here is the complete error handling map:

StageFailure ModeUser Sees
CLI parsingargs.len() < 2usage: ignis <file.is> [-o <output>]
File readFile not found, permission deniederror reading program.is: No such file or directory
ParsingSyntax errorsFormatted diagnostic output

LinkerUndefined symbolsLinker error output (exits 1)

Note that lexer errors are currently collected but silently discarded (_lex_errors). This is a known limitation — in a production compiler, lex errors would be reported alongside parse errors.

The ABI Module

The abi.rs module provides small helper functions used across code generation and IR building:

src/abi.rs — ABI helpers
pub fn type_size(types: &TypeArena, t: TypeId) -> u64 {
    types.size_of(t)
}
pub fn type_align(types: &TypeArena, t: TypeId) -> u64 {
    types.align_of(t)
}
pub fn stack_align(size: u64) -> u64 {
    (size + 15) & !15
}

type_size and type_align are thin wrappers over TypeArena methods, providing an ABI-focused interface. stack_align rounds up to the next 16-byte boundary, matching the x86-64 SysV ABI requirement that the stack be 16-byte aligned at function call boundaries.

Testing the Build Pipeline End-to-End

Since assembly and linking involve external tools, the most effective testing approach is end-to-end compilation and execution:

$ cargo build --release
$ echo "func main(i32) { return 42; }" > /tmp/test.is
$ ./target/release/ignis /tmp/test.is
wrote /tmp/test
$ /tmp/test; echo $?
42

If the process fails, inspect the intermediate files:

$ cat /tmp/test.asm                  # examine generated assembly
$ nasm -f elf64 /tmp/test.asm       # isolate NASM errors
$ cc -o /tmp/test /tmp/test.o -no-pie  # isolate linker errors

This manual debug chain is essential for distinguishing between bugs in code generation (wrong assembly syntax) and bugs in assembly/linking (missing tools, wrong flags).

Common Pitfalls

  1. NASM not installed: The error message is error invoking assembler: No such file or directory. Install NASM via your package manager.
  2. Missing main: If no func main(i32) is defined, the linker will report an undefined reference to main. Every L1S program needs an entry point.
  3. PIE incompatibility: Without -no-pie, modern linkers default to producing position-independent executables, which require position-independent code (PIC). L1S's assembly is not PIC-compliant (it uses absolute addresses in .data), so -no-pie is required.
  4. Output path edge cases: Path::with_extension("") works for most cases but can produce surprising results for inputs like .is (hidden file) or filenames with multiple dots.
  5. Temporary file deletion: If the linker succeeds, the .o file is deleted. If you want to inspect it, compile with -o and catch it before deletion, or modify the driver to keep it (e.g., with a --keep flag).
