What is a Compiler?

A compiler is a program that translates source code written in one language (the source language) into another language (the target language), typically machine code. The fundamental challenge is that humans think in high-level abstractions—functions, loops, variables, types—while CPUs operate on raw bytes, registers, and jump instructions. A compiler bridges this gap through a series of transformations, each step removing one layer of abstraction.

Not every language implementation is a compiler. The design space includes:

  • Interpreters execute source code directly without translation. They offer fast startup and easy introspection but slower execution. Examples: Python, Bash.
  • Just-In-Time (JIT) compilers translate code at runtime, blending compilation speed with execution speed. Examples: Java HotSpot, V8 (JavaScript).
  • Ahead-Of-Time (AOT) compilers translate entirely before execution. Startup is instant but the compilation itself takes time. Examples: C, Rust, Go.
  • Transpilers translate between high-level languages (e.g., TypeScript to JavaScript).

L1S is an AOT compiler with a traditional multi-phase pipeline. It translates a C-like language into x86-64 assembly, then uses NASM and the system linker to produce a standalone executable.

Why build a compiler? You learn exactly how high-level language features map to machine code. There is no magic—just a series of well-defined transformations. After building one, you will never look at if, while, or function calls the same way again.

The --release flag is important: the compiler itself benefits from optimization, especially on larger input programs.

Running the Compiler

The compiler takes a single source file and an optional output path:

$ cat hello.is
func main(i32) {
    return 42;
}

$ ./target/release/ignis hello.is
wrote hello
$ ./hello
$ echo $?
42

The intermediate hello.asm file is left on disk for inspection. You can open it to see exactly what x86-64 instructions the compiler generated for your program.

If no -o flag is given, the output binary gets the same name as the source file without the .is extension. Under the hood, the compiler produces a .asm file, invokes NASM to create a .o, then invokes cc to link it into an executable.

The Source Language — Ignis

L1S compiles a language called Ignis (Latin for "fire"). It is a minimal, statically-typed, C-like language with:

  • Functions with explicit parameter types and return types
  • i32, bool, char, pointer (@T), and array ([N]T) types
  • let bindings with explicit type annotations
  • if/else, while, return
  • Integer literals in decimal, hex, binary, and octal with underscore separators
  • Integer literals in decimal, hex, binary, and octal with underscore separators
  • Strings, characters, comments

Here is a complete Ignis program:

func gcd(i32, a: i32, b: i32) {
    if b == 0 {
        return a;
    }
    return gcd(b, a % b);
}

func main(i32) {
    let x: i32 = gcd(42, 56);
    return x;
}

The compiler pipeline will transform this into x86-64 assembly, then into an executable.

Project Structure

Every compiler pipeline has roughly the same logical phases. The L1S source tree is organized to mirror this pipeline directly. Each phase lives in its own module:

src/main.rs — module declarations and pipeline orchestration
mod span;       // Source locations (byte offsets, line/col)
mod token;     // Token definitions (keywords, operators, literals)
mod diag;      // Error reporting and diagnostics
mod lexer;     // Lexical analysis (source → tokens)
mod ast;       // Abstract syntax tree definitions
mod parser;    // Syntactic analysis (tokens → AST)
mod types;     // Type system (arena-allocated types)
mod symbol;    // Symbol table (scope-chain name resolution)
mod typeck;    // Semantic analysis / type checking
mod ir;        // Intermediate representation definitions
mod ir_build;  // IR construction (AST → IR)
mod ir_opt;    // IR optimization passes
mod abi;       // Calling convention and ABI details
mod regalloc;  // Register allocation
mod codegen;   // x86-64 code generation (IR → assembly)

Reading top to bottom, the module list tells the story of the compiler. Early modules (span, token, diag) are foundational data types used everywhere. Middle modules (lexer, parser, typeck) perform analysis. Late modules (ir_build, ir_opt, regalloc, codegen) perform synthesis and code generation. The abi module sits at the boundary between IR and codegen, defining how types are laid out in memory and how function calls pass arguments.

The Compiler Driver

The main function in main.rs is the driver—it orchestrates the entire pipeline. Here is how the general pattern of a multi-phase compiler driver plays out in Ignis:

src/main.rs — the pipeline driver
fn main() {
    // 1. Parse CLI arguments
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: ignis <file.is> [-o <output>]");
        std::process::exit(1);
    }

    // 2. Read source file
    let source = fs::read_to_string(&args[1]).unwrap_or_else(|e| { /* ... */ });

    // 3. Lexical analysis: source → tokens
    let mut lexer = lexer::Lexer::new(&source);
    let (tokens, _lex_errors) = lexer.tokenize();

    // 4. Syntactic analysis: tokens → AST
    let diag = diag::Diagnostics::new();
    let mut parser = parser::Parser::new(tokens, diag);
    let mut prog = parser.parse_program();
    let mut diag = parser.into_diagnostics();

    // 5. Error check before proceeding
    if diag.has_errors() { diag.print_all(); std::process::exit(1); }

    // 6. Semantic analysis: type checking
    let mut types = types::TypeArena::new();
    let mut symbols = symbol::SymbolTable::new();
    let mut tc = typeck::TypeChecker::new(&mut types, &mut symbols, &mut diag);
    tc.check_program(&mut prog);
    let annotations = tc.annotations.clone();
    let has_errors = tc.diag.has_errors();
    drop(tc);
    if has_errors { diag.print_all(); std::process::exit(1); }

    // 7. IR construction and optimization
    let builder = ir_build::IrBuilder::new(&types, &annotations, &mut diag);
    let mut ir_module = builder.build(&prog);
    ir_opt::optimize(&mut ir_module);

    // 8. Code generation and linking
    let mut codegen = codegen::Codegen::new(&types);
    let asm = codegen.generate(&ir_module);

    // 9. Write assembly, assemble, link
    // ... (path resolution, nasm, cc invocations)
}

The structure is linear and explicit. Each phase transforms its input into the next phase's input. Early-exit checks after parsing and type checking ensure that malformed programs fail fast with clear error messages rather than producing garbage IR or assembly.

Pipeline design philosophy: The driver does nothing clever. It is a straight-line sequence of function calls with error checks. This simplicity is deliberate—when something goes wrong, you can trace the exact path through the driver to find the bug. Every compiler phase is a pure-ish function: given valid input, it produces deterministic output or a list of errors.

The Idea of a Compiler Pipeline

Every compiler, regardless of language or target, is fundamentally a sequence of transformations. Source text enters one end, machine code exits the other, and each intermediate step operates on a progressively lower-level representation. This sequence is called the compiler pipeline.

The key insight is separation of concerns. Rather than one enormous function that goes from source to machine code (a "compiler-monolith"), we decompose the problem into phases. Each phase has a well-defined input, a well-defined output, and a single responsibility. This makes the compiler easier to reason about, test, and extend.

Most compilers divide into two major halves:

  • Frontend — Analyzes the source program: lexing, parsing, semantic analysis. Produces an annotated AST (abstract syntax tree with type information).
  • Backend — Synthesizes target code: instruction selection, register allocation, assembly emission. Produces machine code or assembly.

Between them often sits an Intermediate Representation (IR)—a data structure that is independent of both the source language and the target architecture. IRs are what make retargetable compilers possible: change the frontend to support a new language, change the backend to support a new CPU, keep the IR and its optimization passes unchanged.

Pipeline Design Tradeoffs

How many phases should a compiler have? The answer depends on what you are optimizing for:

  • Single-pass compilers (Pascal, early C) emit code as they parse. They are fast and use little memory, but the language must be designed so that every construct can be compiled without looking ahead. You cannot use a function before declaring it. These are rare today.
  • Two-pass compilers (many toy compilers) separate analysis from code generation. The first pass builds a data structure (often an AST), the second emits code from it. Simple and effective, but optimizations are limited.
  • Multi-pass compilers (LLVM, GCC, Rust, modern C) have many phases: lexing, parsing, name resolution, type checking, several IR transformations, instruction selection, register allocation, peephole optimization. Each pass is small and testable. This is the dominant model.

There is also a stylistic choice around error handling: some compilers abort on the first error, while others collect as many errors as possible in each phase before giving up. Modern compilers favor the latter—it is deeply frustrating to fix one typo only to discover another one on the next compilation.

The driver running nasm and cc is technically outside the compiler proper, but it is the final step that makes a usable tool.

What Data Flows Between Phases?

Each phase transforms its input into a new representation. These representations form a hierarchy of increasing specificity and decreasing abstraction:

Source text → a flat string of characters. The lexer cannot use this directly for any analysis beyond character grouping.

Tokens → a flat sequence of tagged values. The parser consumes these to build structure. Tokens carry Spans so errors can point back to the original source.

AST → a tree reflecting the grammatical structure of the program. After type checking, every expression node carries its resolved type.

IR → a control-flow graph of basic blocks containing low-level instructions. Virtual registers are infinite (the register allocator later maps them to real registers). This representation makes optimizations straightforward to implement and reason about.

Assembly → a string of x86-64 mnemonics. This is the final output of the compiler; nasm turns it into machine code bytes.

Internal vs External Phases

Some compiler phases are truly internal (they exist only within the compiler binary), while others rely on external tools. In Ignis, phases 1–7 are internal Rust code. Phase 8—assembling and linking—delegates to NASM and the system C compiler. This is a pragmatic choice: writing an assembler and linker from scratch would add thousands of lines without teaching anything new about compilation.

Many real compilers do the same. GCC emits assembly and calls an external assembler (as) and linker (ld). LLVM can emit assembly, object code, or use an integrated assembler. The boundary between "the compiler" and "the toolchain" is fluid and depends on your goals.

The advantage of using NASM is that the generated assembly is human-readable. You can open the .asm file and see exactly what instructions the compiler produced. This is invaluable for debugging and learning.

How the Pipeline is Wired in Ignis

The general pattern of a pipeline driver in Ignis is a straight-line sequence in main.rs. Let's walk through each call:

src/main.rs — the pipeline driver (annotated)
// === PHASE 1: LEXER ===
// The lexer takes a &str and returns (tokens, errors).
// Errors are collected but not fatal - we keep going.
let mut lexer = lexer::Lexer::new(&source);
let (tokens, _lex_errors) = lexer.tokenize();

// === PHASE 2: PARSER ===
// The parser builds an AST from the token stream.
// It owns the Diagnostics object and accumulates errors.
let diag = diag::Diagnostics::new();
let mut parser = parser::Parser::new(tokens, diag);
let mut prog = parser.parse_program();
let mut diag = parser.into_diagnostics();

// Error checkpoint: abort if parsing failed.
if diag.has_errors() {
    diag.print_all();
    std::process::exit(1);
}

// === PHASE 3: TYPE CHECKER ===
// The type checker walks the AST, resolves types, and
// annotates every expression with its type.
let mut types = types::TypeArena::new();
let mut symbols = symbol::SymbolTable::new();
let mut tc = typeck::TypeChecker::new(
    &mut types, &mut symbols, &mut diag);
tc.check_program(&mut prog);
let annotations = tc.annotations.clone();
let has_errors = tc.diag.has_errors();
drop(tc);

// Error checkpoint: type errors abort before codegen.
if has_errors {
    diag.print_all();
    std::process::exit(1);
}

// === PHASE 4: IR BUILDER ===
// Lowers the checked AST into a control-flow-graph IR.
let builder = ir_build::IrBuilder::new(
    &types, &annotations, &mut diag);
let mut ir_module = builder.build(&prog);

// === PHASE 5: OPTIMIZER ===
// Runs constant folding, dead code elimination, and
// copy propagation on the IR.
ir_opt::optimize(&mut ir_module);

// === PHASE 6 (implicit) & 7: REG ALLOC + CODEGEN ===
// The codegen handles register allocation internally.
let mut codegen = codegen::Codegen::new(&types);
let asm = codegen.generate(&ir_module);

// === PHASE 8: ASSEMBLE & LINK ===
// Write .asm file, call nasm, call cc, clean up.
fs::write(&asm_path, &asm);
Command::new("nasm").args(&["-f", "elf64", "-o", &o_path, &asm_path]);
Command::new("cc").args(&["-o", &bin_path, &o_path, "-no-pie"]);

Notice the two error checkpoints: one after parsing and one after type checking. If either produces errors, the compiler prints diagnostics and exits before attempting code generation. This means the backend never has to handle malformed input—a significant simplification.

Why two separate error checkpoints? Parse errors indicate the program is not even syntactically valid. Trying to type-check a syntactically invalid program would produce confusing cascading errors. By checking after parsing, we report only syntax errors. Then we type-check, and report only semantic errors. The user gets clean, separated error output.

Phase Granularity: Combined vs Separate

Some compilers combine adjacent phases. For example, a lexer and parser are often fused into a single pass because they are tightly coupled: the parser calls the lexer on demand (this is called a lexer with parser feedback or a scannerless parser). Ignis keeps them separate, which means the lexer produces a complete token list before the parser starts. This has two advantages:

  • Testability: You can test the lexer in isolation by inspecting its token output.
  • Simplicity: The parser does not need to worry about character-level details.

The cost is memory: the entire token list is stored before parsing begins. For a small language like Ignis this is negligible. For languages with millions of lines of code, some compilers use a two-lexer approach (one fast pre-lexer for the parser's lookahead needs) or re-lex on demand.

Similarly, the register allocator and code generator could be fused. Ignis keeps them separate because register allocation is a well-defined problem (mapping virtual registers to real ones) that benefits from its own dedicated pass and data structures.

Error Handling Philosophy

Different phases handle errors differently, reflecting their position in the pipeline:

  • Lexer: Never crashes. Collects errors in a Vec<LexError> and continues producing tokens. Invalid characters are skipped.
  • Parser: Never crashes. Uses the Diagnostics system to record errors and employs error recovery strategies (e.g., token synchronization) to continue parsing.
  • Type Checker: Never crashes. Records every type error and continues checking other parts of the program.
  • Backend (IR Builder, Codegen): If the frontend produced valid output, the backend should never encounter errors. If it does, it is a compiler bug, not a user error.

This layered approach means a user sees as many errors as possible in a single compilation run. Lexer errors might produce garbled tokens that cause parser errors, so the driver currently stops after parsing if there are lex errors. A production compiler would filter errors to avoid cascading noise, but the infrastructure to collect and report them is in place.

What About the Diagnostics Module?

You may have noticed mod diag in the module list but no dedicated pipeline phase for it. That is because diagnostics is not a phase—it is a library used by multiple phases. The parser, type checker, and IR builder all report errors through the same Diagnostics object. Separating error collection from error reporting means each phase can focus on its primary job and simply call diag.report(...) when something goes wrong.

This is the same pattern used in rustc (which has a Session object for diagnostics) and LLVM (which has DiagnosticHandler). It is another example of separation of concerns within the pipeline architecture.

Why This Design Works for Learning

The eight-phase pipeline is more phases than strictly necessary (a minimal compiler could do it in four: lex, parse, codegen, link). But the extra phases—type checking, IR building, optimization, register allocation—each teach a distinct concept that appears in every production compiler. By separating them, you can study each in isolation:

  • Lexing teaches finite-state machines and character-level scanning.
  • Parsing teaches grammar, recursion, and tree construction.
  • Type checking teaches constraint satisfaction and semantic analysis.
  • IR teaches intermediate representations and control-flow graphs.
  • Optimization teaches program analysis and transformation.
  • Register allocation teaches graph coloring and resource constraints.
  • Code generation teaches instruction selection and calling conventions.

Each phase is small enough (typically 200–500 lines of Rust) to read in one sitting. The total compiler is under 5,000 lines of code—small enough to fit in your head, large enough to be a real compiler.

Summary

The Ignis pipeline is a linear sequence of well-defined transformations, each building on the previous. The frontend (lexer, parser, type checker) analyzes the program and reports errors. The backend (IR builder, optimizer, codegen) synthesizes assembly. The driver (assembly + linking) produces the final executable. This structure is not unique to Ignis—it follows the same architectural principles as GCC, LLVM, and rustc, though at a much smaller scale that fits in a single afternoon of reading.

2   Source Locations — Span

Why Every Token Needs a Location

When a compiler reports an error, it must tell the programmer exactly where the problem is. A message like "type mismatch" is useless; "type mismatch at line 14:23" is actionable. This requirement means that every token and every AST node must carry its source location—the position in the original source file where it came from.

Source locations serve two purposes:

  • Error reporting: Pointing to the exact region of code that caused the error.
  • Debug info: Mapping generated code back to source lines (DWARF, line tables).

Even a compiler that never produces errors (theoretically impossible) needs source locations for debugging output. They are the foundation of the entire diagnostic system.

Representing a Source Location

There are several common approaches to representing source locations:

  • Byte offsets only (e.g., start: usize, end: usize). Minimum information needed to extract the relevant source text. Used by many compilers internally. Fast, compact, but not human-readable.
  • Line and column only. Human-readable, but ambiguous if the same text appears on multiple lines. Cannot reliably extract the source snippet without reprocessing.
  • Byte offsets + line/col (the most common approach). Stores both machine-oriented offsets and human-oriented line/column numbers. Slightly larger struct but avoids recomputing line/col from offsets (which requires scanning the source).
  • File + offset. For multi-file compilers, you also need a file ID or path.

Ignis uses the third approach: store byte offsets for machine use and line/col for display. This is the same design as rustc's Span (though rustc adds file IDs for incremental compilation).

Design Decisions for Span

A few questions every compiler designer must answer about source locations:

0-based vs 1-based? Line and column numbers are conventionalized as 1-based in virtually every editor and compiler. The first character of the first line is line 1, col 1. Byte offsets remain 0-based because they are array indices into the source string.

Start and end, or start and length? Storing start and end is simpler for slicing and error highlighting (you just do &source[span.start..span.end]). Storing start and length is slightly more compact. Ignis uses start/end.

What about multi-file programs? L1S is currently a single-file compiler. If multi-file support were added, the Span would need a file ID field. The rest of the pipeline would be unaffected because spans are opaque—they are created by the lexer and consumed by the diagnostic system.

Copy vs move semantics? Source locations are tiny (four integers) and are copied constantly. The span type should be Copy to avoid unnecessary clone calls.

The Span Struct in Ignis

Ignis bundles these four values into a Span struct. Here is how the general pattern of a source location plays out in Ignis:

src/span.rs — the Span struct
pub struct Span {
    pub start: usize,   // byte offset of the first character
    pub end: usize,     // byte offset past the last character
    pub line: u32,      // 1-based line number
    pub col: u32,       // 1-based column number
}

Each field has a precise role:

  • start and end are byte offsets into the source string. The substring &source[span.start..span.end] gives you the exact source text for error underlining.
  • line and col are for human-readable messages. They are 1-based, matching common editor convention. line: 1, col: 1 means "the very first character of the file."
Why u32 for line/col? A file would need to be 4 billion lines or columns long to overflow a u32. usize for offsets is necessary because strings can be arbitrarily large on 64-bit systems. Using a smaller type for line/col keeps the struct compact (16 bytes vs 24 or 32).

Construction and Constants

The general pattern for creating spans is through constructors. Ignis provides two: one for real positions and one for synthetic/"unknown" positions:

src/span.rs — constructors
impl Span {
    pub fn new(start: usize, end: usize, line: u32, col: u32) -> Self {
        Span { start, end, line, col }
    }

    pub fn dummy() -> Self {
        Span { start: 0, end: 0, line: 0, col: 0 }
    }
}

Span::new is the workhorse constructor, called by the lexer every time it emits a token. Span::dummy is useful for built-in or synthetic AST nodes that don't correspond to any source text—for example, implicit return values or compiler-generated temporaries. A dummy span has all zeros, which is distinguishable from any real span (real spans start at line ≥ 1, col ≥ 1).

Never display a dummy span. Code that prints spans should check for dummy and display something like "<built-in>" instead of "0:0". Ignis currently does not have this check, so a bug that leaks a dummy span would display as the confusing "0:0".

Merging Spans: The merge Function

When constructing a larger AST node from smaller pieces, you often want the span of the whole node to cover all its children. For example, a binary expression a + b should have a span starting at a and ending at b, so error messages about the whole expression can underline both operands and the operator.

This is called span merging, and it is a common utility:

src/span.rs — merge
    pub fn merge(a: Span, b: Span) -> Span {
        let start = a.start.min(b.start);
        let end = a.end.max(b.end);
        let line = a.line.min(b.line);
        let col = if a.line < b.line { a.col } else { a.col.min(b.col) };
        Span { start, end, line, col }
    }

The logic:

  • start = the minimum of the two start offsets (earliest byte).
  • end = the maximum of the two end offsets (latest byte).
  • line = the minimum line number (earliest line).
  • col = uses the column from the span on the earlier line, or the minimum column if both are on the same line.

This produces a span that covers both a and b and points to the beginning of the combined region. The column heuristic ensures that when merging two spans on different lines, the merged span shows the beginning of the construct (line 1's column) rather than a random later column.

Why Not Point to the Source Directly?

A common question is: why store byte offsets and line/col at all? Why not store a &str slice directly (a pointer + length into the source)?

The answer is lifetime complexity. A span is used pervasively throughout the compiler—in tokens, AST nodes, error messages, and the diagnostic printer. If spans borrowed the source string, every data structure that contains a span would need a lifetime parameter, making the code dramatically more complex. By storing concrete integers, spans are self-contained, Copy, and lifetime-free. The diagnostic printer can reconstruct the source text by indexing into the original String using the byte offsets.

Display Format

Spans are displayed in the standard line:col format familiar from every compiler:

src/span.rs — Display impl
impl fmt::Display for Span {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}:{}", self.line, self.col)
    }
}

This means println!("{}", span) prints 14:23, which matches the format used by GCC, Clang, and rustc. The byte offsets are not displayed in user-facing messages but are used internally for source slicing in the diagnostic printer.

Derived Traits

The span type derives several traits to make it ergonomic to use throughout the compiler:

src/span.rs — derived traits
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct Span { /* ... */ }
  • Debug: Enables {:?} formatting for debugging.
  • Clone, Copy: Span is a small POD type (four integers). Making it Copy avoids explicit .clone() calls everywhere.
  • PartialEq, Eq: Lets you compare spans, useful in tests and for deduplication.
  • Hash: Enables using spans in hash-based data structures (e.g., caching results keyed by source location).
Copy vs Clone for spans. Because a Span is just 4 integers (16 bytes on 64-bit), copying it is cheaper than a pointer chase for a heap-allocated equivalent. Some compilers use an interned span table (store all spans in an arena and pass indices), but that complexity is unnecessary at this scale. Copy semantics are the right call for a toy compiler.

How Spans Flow Through the Pipeline

Spans are created in the lexer and flow through every subsequent phase:

  1. Lexer creates a Span for each token by recording pos, line, col before consuming the token and again after.
  2. Parser attaches spans to AST nodes. It calls Span::merge to construct spans for composite nodes (e.g., a binary expression's span covers both operands and the operator).
  3. Type checker uses spans from AST nodes when producing type error diagnostics.
  4. Codegen does not use spans directly, but the diagnostic printer uses them to extract source lines for error messages.

In short, a span is created once by the lexer and read-only throughout the rest of the compiler. This immutability is important: spans can be freely shared and copied without worrying about ownership or aliasing.

Common Pitfalls with Source Locations

  1. Off-by-one in byte offsets: The end offset should be the byte after the last character, so that &source[start..end] extracts a contiguous slice. If end points to the last character, the slice is one byte short.
  2. Mixing 0-based and 1-based: Easy to accidentally display 0-based line numbers. Always initialize line/col to 1.
  3. Forgetting to update line on newlines: The column must reset to 1 after a newline, not just keep incrementing.
  4. Merge asymmetry: Make sure Span::merge(a, b) == Span::merge(b, a) for all valid inputs. The Ignis implementation is symmetric because it uses min/max.

3   The Token System

What Is a Token?

A token is the fundamental unit of meaning produced by lexical analysis. Instead of passing raw characters from the lexer to the parser, a compiler passes a stream of tokens — atomic units that represent keywords, identifiers, operators, literals, and punctuation. The full set of tokens a language defines is its lexical vocabulary.

Every token carries at least two pieces of information:

  • Kind: what kind of token it is (a keyword, an operator, an identifier, etc.)
  • Span: where in the source it came from (byte offsets and line/column positions)

Some tokens also carry a value — the actual text of an identifier, the digits of a number literal, the content of a string. The token kind tells the parser what to expect next; the span tells it where to report errors; the value tells it the concrete data.

Source: let x: i32 = 42;

Token stream:
Let Ident("x") Colon I32 Eq NumLiteral("42") Semicolon Eof

Notice what happened: whitespace is gone, let became a keyword token (not an identifier), i32 became a primitive-type token (not an identifier), and 42 became a number literal with its raw text preserved. The parser will work with these 8 tokens rather than 17 raw characters.

Tradeoffs & Design Space

Keywords: Enum Variants vs. Keyword Table

Every programming language has a fixed vocabulary of reserved words — keywords that have special meaning and cannot be used as identifiers. A compiler designer must decide when keywords are recognized:

  • Lexer-based keyword recognition: The lexer identifies keywords during tokenization and produces distinct token variants (e.g., Token::Func vs. Token::Ident("main")). This is the most common approach in hand-written compilers. It simplifies the parser because the parser matches on fixed enum variants rather than comparing string values.
  • Parser-based keyword recognition: The lexer produces Ident("func") for everything, and the parser matches the identifier string against a keyword table. This is more flexible for languages where keyword status depends on context, but adds string comparisons to the parser and makes keyword errors harder to produce.

Most modern compilers, including Ignis, use the lexer approach: a dedicated function maps every recognized keyword string to its token variant. Anything that doesn't match falls through to Token::Ident.

Spanned Tokens: Bundled vs. Separate

Source locations can be attached to tokens in two ways:

  • Bundled: Every token struct contains a Span field. This is ergonomic — every token knows where it came from without additional bookkeeping.
  • Separate array: Tokens and spans are stored in parallel arrays, or the parser queries a separate position map. This can save memory when spans are sparse, but adds complexity.

Ignis bundles them into a SpannedToken struct.

Token Values: Stored Strings vs. Interned IDs

Identifier and literal tokens carry a payload. The classic approaches:

  • Owned strings (String): Simple, self-contained, no lifetime management. Each token owns its value. This is what Ignis uses.
  • Source slices (&'a str): Zero-copy, but ties the token's lifetime to the source string. More memory-efficient but more complex.
  • Interned strings (Symbol): Each unique string gets a numeric ID. Fast equality checks and low memory, but requires a global string interner.

Ignis uses owned String values for simplicity — the compiler is not yet performance-critical enough to justify interning.

Operators: Distinct Variants vs. Generic

Operators can be encoded as Token::Plus, Token::Minus, etc. (one variant per operator) or as Token::Op("+") (a single variant with a string payload). The distinct-variant approach enables exhaustive matching in the parser and better compile-time checking of operator handling. Ignis uses distinct variants for every operator.

This function is a pure lookup: given a string, return the token variant or None. A few notable properties:

  • Case-sensitive matching: "func" and "Func" both map to Token::Func. This is explicit — only specific case variants are recognized.
  • Primitive types are keywords: "i32", "bool", etc. are classified here rather than in the lexer's identifier logic. This keeps the type-classification logic in one place.
  • No fallthrough to Ident: The function returns Option, leaving the fallback decision to the caller. The lexer calls keyword_token(&ident).unwrap_or(Token::Ident(ident)) — if it's not a keyword, it's an identifier.
The keyword function as an API boundary: keyword_token is the only function the lexer imports from the token module. Adding a new keyword requires exactly two changes: (1) add a variant to the Token enum, and (2) add a match arm to keyword_token. The lexer never needs to change. This is a clean separation of concerns between the token vocabulary and the tokenization algorithm.

Key Takeaways

  1. Tokens are the parser's vocabulary. The parser never sees raw characters — only atomic token units with kind, value, and location.
  2. Keyword classification happens in the lexer, through a single function that maps strings to enum variants. This keeps the parser free of string comparisons.
  3. Every token carries its span via the SpannedToken wrapper. No token is ever created without source location information.
  4. Tokens use owned data (String, u8) rather than references or interned IDs. This is simple, correct, and sufficient for a compiler of this scale.
  5. The Token enum is flat — distinct variants for every keyword, operator, and delimiter. This maximizes the type system's ability to check exhaustiveness in match statements throughout the compiler.

4   The Lexer

4.1   What Is Lexical Analysis?

Concept. Every compiler begins by reading a stream of characters — the raw source file. The first transformation is to group these characters into meaningful atoms called tokens. This phase is lexical analysis (lexing or tokenizing).

Think of it as splitting a sentence into words:

Input: func main(i32) { return 42; }

Output:
Func   Ident("main")   OpenParen   I32   CloseParen   OpenBrace   Return   NumLiteral("42")   Semicolon   CloseBrace   Eof

The lexer does not understand semantics — it doesn't know that main is a function name or that 42 is an integer literal. It merely classifies spans of characters into token categories. The parser consumes these tokens to build structure.

The fundamental lexer loop is universal:

  1. Skip whitespace and comments
  2. Look at the next character
  3. Decide what kind of token it starts
  4. Consume characters belonging to that token
  5. Emit the token with its source location
  6. Repeat until EOF

Lexing is a deterministic finite automaton (DFA) in practice: at each step, the current character determines the next state. There is no backtracking and no ambiguity once the character class is known.

4.2   Design Space: How to Build a Lexer

Tradeoffs. There are three main approaches to building a lexer:

ApproachProsCons
Generator (lex/flex) Declarative regex rules, fast generated C code, battle-tested External tool, build dependency, opaque generated code, poor error messages
Hand-written DFA table Fastest possible, small code, data-driven Hard to maintain, table generation is error-prone, worse for simple languages
Hand-written recursive Easy to read and debug, exact error positions, no build deps, flexible Slightly slower per character, more code to write

L1S uses a hand-written recursive lexer. For a language with ~40 token types, this is the sweet spot. The lexer is about 400 lines of straightforward Rust with no external dependencies. Every character is examined exactly once (O(n)), and the performance difference from a generated lexer is negligible for any realistic input file.

Why not flex? Flex-generated lexers excel for languages with many token patterns (C's preprocessor, FORTRAN). For a simple C-like language, the complexity of integrating a code generator into the build pipeline outweighs the benefits. Hand-written lexers are the norm in modern compilers: Rust, Go, Zig, and Clang all use hand-written lexers.

4.3   Core Types

Before the lexer itself, we need the types it produces. The three building blocks are Span, Token, and SpannedToken.

Span encodes a source location: byte offsets plus line/column numbers. Every token carries one, enabling precise error messages downstream.

Token is a flat enum with a variant for every keyword, operator, delimiter, and literal in the language. The full enum is defined in src/token.rs and contains ~50 variants.

4.4   The Lexer Struct & Position Model

Concept. A lexer is a stateful transducer. It holds a reference to the source text, a cursor into that text, and accumulates tokens and errors as it advances. The position model must handle two concerns: byte offsets (for slicing the source) and human-readable coordinates (for error messages).

Tradeoff: Byte offsets vs character indices. Rust's str::CharIndices yields (usize, char) pairs, giving us byte offsets automatically. Some lexers store character indices and convert to byte offsets only when needed. Byte offsets are more general — they let us slice the source directly — and Rust's UTF-8 guarantees make them equally convenient.

src/lexer.rs — Lexer struct
pub struct Lexer<'a> {
    source: &str,
    chars: Peekable<CharIndices<'a>>,
    pos: usize,
    line: u32,
    col: u32,
    errors: Vec<LexError>,
}

The fields in detail:

  • source — Borrowed input string. Never modified, only referenced.
  • chars — A Peekable<CharIndices>. CharIndices yields (byte_offset, char) pairs. Peekable gives us one character of lookahead without consuming it, which is essential for multi-character operators like ==.
  • pos — Current byte offset. Monotonically increases as characters are consumed.
  • line, col — 1-based line and column numbers, updated by advance().
  • errors — Accumulated errors. The lexer never panics; it records the error and continues.

Tradeoff: Lifetime parameter. The 'a lifetime ties the lexer to the source string. This means the lexer borrows the input rather than owning it. An alternative would be to copy the source into an owned String, but borrowing avoids an allocation and keeps the API zero-cost.

Position tracking is centralized in a single method:

fn advance(&mut self) -> Option<char> {
    match self.chars.next() {
        Some((_, c)) => {
            let len = c.len_utf8();
            self.pos += len;
            if c == '\n' {
                self.line += 1;
                self.col = 1;
            } else {
                self.col += 1;
            }
            Some(c)
        }
        None => None,
    }
}

Having all position updates in one place is essential for correctness. Every character consumed anywhere in the lexer goes through advance(), so pos, line, and col are always consistent. The complement is peek(), which looks ahead without consuming:

fn peek(&mut self) -> Option<char> {
    self.chars.peek().map(|&(_, c)| c)
}
UTF-8 safety. CharIndices yields complete Unicode characters. The len_utf8() call returns the byte width of the character (1–4 bytes), so pos always lands on a valid UTF-8 boundary. Emoji, CJK characters, and accented letters in comments or string literals are handled correctly without any special logic.

4.5   The Main Loop: tokenize()

Concept. Every lexer's entry point is a loop that drives the state machine: skip ignorable input, lex one token, emit it. The loop must handle three outcomes: a valid token (emit), an unrecognized character (record error, skip), or end of input (emit EOF and stop).

Tradeoff: Error accumulation vs Result. L1S collects errors in a Vec<LexError> and returns them alongside the tokens. The alternative would be to return Result<Vec<SpannedToken>>, stopping at the first error. Error accumulation is friendlier: the user sees all errors in one compilation, not one at a time with recompilation between fixes.

src/lexer.rs — tokenize()
pub fn tokenize(&mut self) -> (Vec<SpannedToken>, Vec<LexError>) {
    let mut tokens = Vec::new();

    loop {
        self.skip_whitespace();
        self.skip_comment();
        self.skip_whitespace();

        if self.chars.peek().is_none() {
            break;
        }

        let start_pos = self.pos;
        let start_line = self.line;
        let start_col = self.col;

        let token = self.lex_token();
        let end_pos = self.pos;
        let span = Span::new(start_pos, end_pos, start_line, start_col);

        match token {
            Some(tok) => {
                tokens.push(SpannedToken::new(tok, span));
                if tok == Token::Eof { break; }
            }
            None => {
                if let Some((_, c)) = self.chars.next() {
                    let len = c.len_utf8();
                    self.pos += len;
                    self.col += 1;
                    let span = Span::new(start_pos, self.pos, start_line, start_col);
                    self.errors.push(LexError::new(
                        format!("unexpected character '{}'", c), span));
                }
            }
        }
    }

    if tokens.last().map(|t| &t.token) != Some(&Token::Eof) {
        let span = Span::new(self.pos, self.pos, self.line, self.col);
        tokens.push(SpannedToken::new(Token::Eof, span));
    }

    (tokens, std::mem::take(&self.errors))
}

Key design decisions in this loop:

  • Whitespace/comments before every token — we skip twice (before and after comments), which handles the edge case of whitespace between // and the following token.
  • Span captured before and after lexing — the start position is saved before lex_token() runs; the end position is read after. This gives accurate bounds for every token.
  • EOF guarantee — the loop ensures at least one Token::Eof is emitted, even for empty input. The parser depends on this sentinel to avoid checking bounds.
  • std::mem::take — swaps self.errors with an empty vec, leaving the lexer in a clean state for potential reuse.

4.6   The Comment Ambiguity

Concept. Many languages use the same starting character for comments and operators: / starts both division (/) and line comments (//). The lexer cannot know which it is from the first character alone — it must look at the next character to decide.

Tradeoff: Save/restore vs two-pass. One approach is to lex / in lex_token() only after comments have been pre-skipped. But comments are supposed to be invisible to later stages, so they need to be consumed before token dispatch. L1S uses a save/restore pattern: tentatively consume the /, peek at the next character, and either consume the comment or restore the saved state so lex_token() sees / as a division operator.

fn skip_comment(&mut self) {
    if self.peek() == Some('/') {
        let saved = self.chars.clone();
        let saved_pos = self.pos;
        let saved_line = self.line;
        let saved_col = self.col;

        self.advance();
        if self.peek() == Some('/') {
            while let Some(c) = self.peek() {
                if c == '\n' { break; }
                self.advance();
            }
        } else {
            self.chars = saved;
            self.pos = saved_pos;
            self.line = saved_line;
            self.col = saved_col;
        }
    }
}

The iterator is cloneable because Peekable<CharIndices> is backed by a slice iterator, which is Clone. When / is followed by something other than /, we restore the entire lexer state, and the / is lexed as Token::Slash during the next call to lex_token().

Performance note. Restoring the iterator means we'll re-examine the / character. In a correct lexer this is unavoidable — you must peek ahead to disambiguate. The cost is minimal because this pattern only fires when the character is / (rare in typical source code).

4.7   Operator Dispatch

Concept. A lexer's dispatch table maps the first character of a potential token to either a single token or a more complex handler. Operators fall into three categories: single-character (+, *, (), two-character (==, !=, &&), and compound starting with a prefix that matches a single operator (- can be ->).

Tradeoff: Character matching vs table-driven. A match expression on the first character is the simplest approach. For a language with ~30 operators it's perfectly readable. An alternative is a lookup table from (char, Option<char>) to Token, but this sacrifices clarity for negligible speed gain.

fn lex_token(&mut self) -> Option<Token> {
    let c = self.peek()?;

    match c {
        '+' => { self.advance(); Some(Token::Plus) }
        '-' => {
            self.advance();
            if self.peek() == Some('>') {
                self.advance();
                Some(Token::Arrow)
            } else {
                Some(Token::Minus)
            }
        }
        '(' => { self.advance(); Some(Token::OpenParen) }
        ')' => { self.advance(); Some(Token::CloseParen) }
        '{' => { self.advance(); Some(Token::OpenBrace) }
        '}' => { self.advance(); Some(Token::CloseBrace) }
        '&' => {
            self.advance();
            if self.peek() == Some('&') {
                self.advance();
                Some(Token::AmpAmp)
            } else {
                Some(Token::Amp)
            }
        }
        '=' => {
            self.advance();
            if self.peek() == Some('=') {
                self.advance();
                Some(Token::EqEq)
            } else {
                Some(Token::Eq)
            }
        }
        '"'  => Some(self.lex_string()),
        '\'' => Some(self.lex_char()),
        '0'..='9' => Some(self.lex_number()),
        c if is_ident_start(c) => Some(self.lex_identifier_or_keyword()),
        _ => None,
    }
}

The pattern for multi-character operators is consistent: advance past the first character, peek at the second, and either return the compound token or fall back to the single-character one. This pattern repeats for !/!=, </<=/<<, >/>=/>>, and |/||.

4.8   Number Lexing

Concept. A numeric literal is a sequence of digits, optionally with a radix prefix. The lexer must distinguish decimal, hexadecimal (0x), binary (0b), and octal (0o) literals, and allow underscores as visual separators.

Tradeoff: Validate now or later? L1S lexes numbers as raw strings without validating their content. 0xGG is happily lexed as NumLiteral("0xGG"). The parser or type checker is responsible for rejecting invalid values. This keeps the lexer simple and decouples concerns. The alternative is to parse and validate the number in the lexer, which means error messages about invalid literals come from the lexer instead of being unified with other diagnostics.

fn lex_number(&mut self) -> Token {
    let mut raw = String::new();

    if self.peek() == Some('0') {
        self.advance();
        raw.push('0');
        match self.peek() {
            Some('x' | 'X') => {
                self.advance(); raw.push('x');
                while let Some(c) = self.peek() {
                    if c.is_ascii_hexdigit() || c == '_' {
                        self.advance(); raw.push(c);
                    } else { break; }
                }
            }
            Some('b' | 'B') => { /* binary: 0-1 and _ */ }
            Some('o' | 'O') => { /* octal: 0-7 and _ */ }
            _ => { /* decimal starting with 0 */ }
        }
    } else {
        while let Some(c) = self.peek() {
            if c.is_ascii_digit() || c == '_' {
                self.advance(); raw.push(c);
            } else { break; }
        }
    }
    Token::NumLiteral(raw)
}
Underscore behavior. Underscores are accepted anywhere in a number: 1_000, 0x_FF_EE, 0b_1010_1111. They are included verbatim in the raw string and must be stripped during parsing. This is the simplest approach — validating underscore placement is left to a later stage.

4.9   String and Character Lexing

Concept. String literals require a stateful sub-loop: enter on ", consume characters until the closing ", handling escape sequences (like \n, \t, \xHH) along the way. Character literals follow the same pattern but produce a single byte.

Tradeoff: Resolve escapes or keep raw? L1S resolves escape sequences during lexing, so "\n" becomes StrLiteral("\n") (one newline character), not StrLiteral("\\n"). This means downstream stages never need to understand escape sequences. The alternative is to keep the raw string and resolve escapes on demand, which is more flexible but pushes complexity into every consumer of string tokens.

fn lex_string(&mut self) -> Token {
    let mut content = String::new();
    self.advance(); // consume opening '"'

    loop {
        match self.peek() {
            None => {
                self.errors.push(LexError::new("unclosed string literal", span));
                break;
            }
            Some('"') => { self.advance(); break; }
            Some('\\') => {
                self.advance();
                match self.advance() {
                    Some('n') => content.push('\n'),
                    Some('t') => content.push('\t'),
                    Some('\\') => content.push('\\'),
                    Some('"') => content.push('"'),
                    Some('0') => content.push('\0'),
                    Some('x') => {
                        let hex = self.lex_hex_escape();
                        content.push(hex as char);
                    }
                    Some(c) => {
                        // invalid escape: record error, push char anyway
                        content.push(c);
                    }
                    None => break,
                }
            }
            Some(_) => {
                self.advance().map(|c| content.push(c));
            }
        }
    }
    Token::StrLiteral(content)
}

Character lexing is the same pattern but constrained to a single character and producing CharLiteral(u8). Both share lex_hex_escape() for \xHH sequences:

fn lex_hex_escape(&mut self) -> u8 {
    let mut hex = String::with_capacity(2);
    for _ in 0..2 {
        match self.peek() {
            Some(c) if c.is_ascii_hexdigit() => {
                self.advance(); hex.push(c);
            }
            _ => break,
        }
    }
    u8::from_str_radix(&hex, 16).unwrap_or(0)
}

4.10   Identifiers and Keywords

Concept. Identifiers and keywords share the same character class: they start with a letter or underscore and continue with letters, digits, and underscores. The lexer consumes the entire word, then checks whether it matches a reserved keyword. This is the simplest and most common approach.

Tradeoff: Keyword-as-identifier vs keyword filtering. Some lexers check for keywords character-by-character during consumption. L1S uses the simpler "consume then classify" approach: read the whole identifier, then look it up in a keyword table. This separates the concerns of "what is an identifier" (a simple regex) from "what is a keyword" (a set membership test).

fn lex_identifier_or_keyword(&mut self) -> Token {
    let mut ident = String::new();
    while let Some(c) = self.peek() {
        if c.is_alphanumeric() || c == '_' {
            self.advance();
            ident.push(c);
        } else { break; }
    }
    keyword_token(&ident).unwrap_or(Token::Ident(ident))
}

The keyword dispatch is a single function in token.rs that maps string names to token variants:

pub fn keyword_token(word: &str) -> Option<Token> {
    match word {
        "func"   => Some(Token::Func),
        "let"    => Some(Token::Let),
        "return" => Some(Token::Return),
        "if"     => Some(Token::If),
        "else"   => Some(Token::Else),
        "true"   => Some(Token::True),
        "false"  => Some(Token::False),
        "i32"    => Some(Token::I32),
        "u32"    => Some(Token::U32),
        // ... 30+ more entries for all keywords and type names
        _ => None,
    }
}

Adding a new keyword requires exactly two changes: add a variant to the Token enum, and add a match arm to keyword_token. The lexer never changes. This separation of concerns is the hallmark of a well-structured lexer.

4.11   Error Recovery Strategy

Concept. A lexer for a production compiler must handle malformed input gracefully. The goal is to report as many errors as possible in a single pass, so the user doesn't have to repeatedly compile to discover each mistake.

Tradeoffs in error recovery:

  • Panic on first error — Simplest, but terrible UX. One missing quote hides every subsequent error.
  • Synchronized recovery — Skip tokens until a synchronization point (like ; or }). Better for the parser; overkill for the lexer.
  • Record-and-continue — Record the error, skip the offending character(s), carry on. This is what L1S does.

L1S never panics. Every error path records a LexError and advances past the bad input. The three error categories are:

ErrorRecovery
Unrecognized character (e.g., $)Consume and skip it
Unclosed string literalClose it at EOF, return partial content
Invalid or unterminated escape sequencePush the raw character and continue
Never crash, never hang. Every loop in the lexer has an exit condition that is guaranteed to be reached: the iterator runs out of characters. There are no recursive descent spirals or unbounded lookahead loops. This is a property worth verifying for any lexer you write.

4.14   Common Pitfalls

  1. Line/column off by one. Initialize line: 1, col: 1. On newline, increment line and reset col to 1. Test with multi-line input early.
  2. Comment/division ambiguity. Without the save/restore pattern, a lone / is consumed by skip_comment and lost. Always test x / y alongside // comment.
  3. Unclosed strings hanging the lexer. Ensure every sub-loop has an EOF exit. The string lexer must handle "hello (no closing quote) without entering an infinite loop.
  4. Multi-byte UTF-8 confusion. Never index into the source string by character position. Always use CharIndices or equivalent. A byte offset that lands in the middle of a multi-byte character is a crash waiting to happen.
  5. Identifier starting with a digit. 123foo should NOT be an identifier. If lex_number accepts non-digit characters (as L1S currently does), the error is caught later. Decide deliberately where to validate.
  6. Overlapping multi-character sequences. >> and >=> can be ambiguous if you don't handle them left-to-right. L1S always peeks at the immediate next character and greedily forms the longest valid token.

5   The Abstract Syntax Tree

What Is an Abstract Syntax Tree?

After the lexer converts source text into a flat stream of tokens, the parser's job is to impose structure on that stream. The result is an Abstract Syntax Tree (AST) — a tree data structure where each node represents a grammatical construct in the source program.

The word "abstract" is critical: the AST discards purely syntactic details that are not needed for later compiler stages. Semicolons, parentheses (when used for grouping rather than function calls), braces, and whitespace all vanish in the AST. What remains is the essential structure: declarations contain statements, statements contain expressions, expressions contain sub-expressions.

Source: return (2 + 3) * 4;

Token stream (flat):
Return OpenParen NumLiteral("2") Plus NumLiteral("3") CloseParen Star NumLiteral("4") Semicolon

AST (tree):
Return
  Binary(Mul)
    Binary(Add)
      Int("2")
      Int("3")
    Int("4")

The parentheses, semicolon, and the flat token ordering are gone. What remains is a tree that directly reflects the grammatical structure: a return statement containing a multiplication expression containing an addition expression.

Tradeoffs & Design Space

AST vs. Concrete Syntax Tree (CST)

A Concrete Syntax Tree (also called a parse tree) preserves every detail from the source text — every keyword, every parenthesis, every semicolon gets a node. An AST is a lossy compression of the CST. The tradeoff:

  • CST: Easier to generate (just mirror the grammar rules), and can reproduce exact source text from the tree. But CSTs are large, verbose, and force later stages to navigate through irrelevant syntactic noise.
  • AST: Smaller, more semantic, and easier for analysis passes (type checking, IR generation) to traverse. The parser works harder, but every downstream pass is simpler.

Ignis uses a pure AST — no syntax nodes survive into the tree. The parser discards grouping parens, delimiters, and separators as it builds the tree.

Sum Types vs. Trait Hierarchy

In typed functional languages (Rust, Haskell, OCaml), AST nodes are naturally represented as sum types (enums). Each category of node — expression, statement, declaration — is one enum with variants for each kind. In OOP languages (Java, C++), ASTs typically use an abstract base class with virtual methods (the visitor pattern).

  • Sum types: Exhaustive matching is checked by the compiler. If you add a new expression variant, every match on expressions must handle it (or explicitly use a wildcard). This is safer and more maintainable.
  • Visitor pattern: More extensible (you can add new operations without modifying existing classes) but lacks exhaustiveness guarantees.

Ignis uses Rust enums for all AST node categories.

Span on Every Node

Every AST node must carry its source location so that errors in later stages (type checking, code generation) can point back to the relevant source code. The design options:

  • Span field on every variant: Every variant of an enum carries its own Span. This is verbose but explicit.
  • Common struct wrapper: struct SpannedNode { node: AstExpr, span: Span }. This factors out the span but adds an indirection layer.
  • Span on a subset of nodes: Only nodes that can produce errors carry spans. This saves memory but risks missing error locations.

Ignis puts a Span field in every variant of every AST enum. Every expression, every statement, every type node knows its source location.

Box vs. Arena Allocation

AST nodes reference each other recursively — an expression can contain sub-expressions. In Rust, this requires indirection:

  • Box<AstExpr>: Heap-allocated, simple, each node is independently owned. This is what Ignis uses.
  • Arena allocation: All nodes are allocated in a contiguous arena with &'arena references between them. More memory-efficient and cache-friendly, but requires a lifetime parameter on the entire AST and makes mutation harder.

Ignis uses Box for recursive nodes — simple and correct, at the cost of individual heap allocations per node.

Ignis Implementation: The AST Hierarchy

The AST is defined in src/ast.rs. It is organized into four levels: types, expressions, statements, and declarations. A program is a vector of top-level declarations.

Level 1: Types (AstType)

A type in Ignis is either a primitive, a named type, or a compound type built from other types:

src/ast.rs — BinaryOp, PrimType, AstType
pub enum BinaryOp {
    Add, Sub, Mul, Div, Mod,
    And, Or, Xor,
    Shl, Shr,
    Eq, Ne, Lt, Gt, Le, Ge,
    LogicAnd, LogicOr,
}

pub enum PrimType {
    Void, Bool, Char, Usize, Isize,
    U64, I64, U32, I32, U16, I16, U8, I8,
}

pub enum AstType {
    Prim(PrimType, Span),
    Named(String, Span),
    Ptr(Box<AstType>, Span),
    Array(Box<AstExpr>, Box<AstType>, Span),
    Func(Vec<AstType>, Box<AstType>, Span),
    Spec(String, Span),
}

BinaryOp and PrimType are standalone enums shared across the AST. AstType represents type annotations in the tree:

  • Prim: A built-in type like i32 or bool. The token Token::I32 becomes AstType::Prim(PrimType::I32, ...).
  • Named: A user-defined type referenced by name, like MyStruct. The parser cannot resolve names — that happens during type checking.
  • Ptr: A pointer type like *i32. Boxed because it wraps another type recursively.
  • Array: A fixed-size array type like [42]i32. Contains an expression for the size and a type for the element type.
  • Func: A function type like (i32, bool) -> void. Contains parameter types and a return type.
  • Spec: A trait/spec type reference, used in impl declarations.

The span() method extracts the span from any type variant:

impl AstType {
    pub fn span(&self) -> Span {
        match self {
            AstType::Prim(_, s)
            | AstType::Named(_, s)
            | AstType::Ptr(_, s)
            | AstType::Spec(_, s) => *s,
            AstType::Array(_, _, s)
            | AstType::Func(_, _, s) => *s,
        }
    }
}

This method is used throughout the compiler whenever an error needs to point to a type annotation.

Level 2: Expressions (AstExpr)

Expressions are the heart of the AST — they represent computations that produce values. Ignis defines 17 expression variants:

src/ast.rs — AstExpr
pub enum AstExpr {
    Int(String, Span),
    Str(String, Span),
    Char(u8, Span),
    Bool(bool, Span),
    Ident(String, Span),
    Deref(Box<AstExpr>, Span),
    AddrOf(Box<AstExpr>, Span),

    Assign(Box<AstExpr>, Box<AstExpr>, Span),
    Call(Box<AstExpr>, Vec<AstExpr>, Span),
    Index(Box<AstExpr>, Box<AstExpr>, Span),
    StructInit(Box<AstExpr>, Vec<AstMemberInit>, Span),
    Field(Box<AstExpr>, String, Span),
    Arrow(Box<AstExpr>, String, Span),
    Cast(Box<AstType>, Box<AstExpr>, Span),
    Paren(Box<AstExpr>, Span),
}

Let's examine the categories:

Literals (Int, Str, Char, Bool): These are leaf nodes — they have no children. The integer literal stores its raw string (not a parsed number) for the same reason the token does: parsing the numeric value is deferred until it's needed. The boolean literal is resolved from true/false keywords into an actual bool.

Identifiers (Ident): References to variables, functions, or types by name. Name resolution happens during type checking, not parsing.

Unary operators (Deref, AddrOf, Neg, Not, BitNot): Each unary operator gets its own variant rather than a generic Unary(UnaryOp, ...). This makes pattern matching in later passes more explicit.

Binary operators (Binary): A single variant covers all binary operations, parameterized by the BinaryOp enum. This is more compact than one variant per operator and makes it easy to iterate over all binary expressions. The operator semantics (associativity, precedence) are handled by the parser and do not appear in the AST.

Compound expressions (Call, Index, StructInit, Field, Arrow): These represent operations with multiple sub-expressions. A function call has a callee and argument list. A struct initializer has a type name and member initializers. Field access has a base expression and a field name.

Type operations (Cast): A cast expression like cast(i32, x) carries both the target type and the source expression.

Grouping (Paren): Parenthesized expressions are preserved in the AST. This is intentional: while (2 + 3) and 2 + 3 produce the same tree after evaluation, preserving parentheses allows the compiler to produce better error messages (“unnecessary parentheses” warnings, for example).

Level 3: Statements (AstStmt)

Statements are the executable units of a program. Unlike expressions, statements do not produce values:

    While(AstWhile),

Each statement variant wraps a dedicated struct rather than inlining fields directly into the enum. This keeps variant payloads named and makes the code self-documenting:

pub struct AstVarDecl {
    pub span: Span,
    pub name: String,
    pub type_: Option<AstType>,
    pub init: Option<AstExpr>,
}

pub struct AstIf {
    pub span: Span,
    pub cond: AstExpr,
    pub then_block: AstBlock,
    pub else_branch: Option<Box<AstStmt>>,
}

pub struct AstBlock {
    pub span: Span,
    pub stmts: Vec<AstStmt>,
}

Key design decisions in the statement hierarchy:

  • Variable declarations (AstVarDecl) have optional type annotation and optional initializer. Both are Option because let x: i32 (type, no init), let x = 42 (init, no type), and let x: i32 = 42 (both) are all valid.
  • If statements have an optional else branch stored as Option<Box<AstStmt>>. The box is necessary because AstStmt is a recursive type. The else branch can itself be another If (producing an else if chain) or any other statement type.
  • Expression statements (AstStmt::Expr) wrap an expression in a statement. This is how function calls used as statements (foo();) are represented — the expression Call is wrapped in an Expr statement.
  • Blocks are first-class statement nodes. A block has its own span and contains a vector of statements. Blocks appear in function bodies, if/while/for bodies, and can be nested arbitrarily.

At the top level of a program, declarations introduce named entities — functions, structs, enums, traits, implementations, type aliases, static variables, and imports:

  • Visibility (pub_: bool): Each declaration carries its visibility. The trailing underscore on pub_ is the Rust convention for avoiding the reserved keyword pub.
  • Function declarations always have a body block, even if empty. Function signatures (for traits) use AstFuncSig instead, which has no body.
  • Enum variants are simple strings with optional explicit values (variant_values: Option<Vec<u64>>). The noprefix flag controls whether variant names are prefixed with the enum name in code generation.
  • Import paths are represented as Vec<String> — e.g., import std.io becomes AstImport { path: ["std", "io"] }.

The Program Root

pub struct AstProgram {
    pub decls: Vec<AstDecl>,
}

A program is just a list of declarations. The order matters — later compiler stages process declarations in order, top-to-bottom. There is no explicit module node or file node; the program struct is the root.

Why no root span on AstProgram? The program's span would be the entire source file, which is not particularly useful for error reporting. Errors are always reported at specific declarations, statements, or expressions inside the program. The root struct exists only to hold the declaration list.

  • Resolved: i32 tokens became AstType::Prim(PrimType::I32). The + operator became BinaryOp::Add. The return keyword became an AstStmt::Return wrapper.
  • Structured: The flat token stream Func Ident("add") OpenParen I32 Ident("a") ... became a tree with explicit parent-child relationships.

Pattern: The span() Method Across Node Types

Every AST node type that can appear in an error message implements a span() method. The pattern is consistent:

impl AstExpr {
    pub fn span(&self) -> Span {
        match self {
            AstExpr::Int(_, s)
            | AstExpr::Str(_, s)
            // ... leaf nodes
            | AstExpr::Paren(_, s) => *s,
            AstExpr::Binary(_, _, _, s)
            | AstExpr::Call(_, _, s)
            // ... compound nodes
            | AstExpr::Cast(_, _, s) => *s,
        }
    }
}

The span is always the last field in every variant. This is a convention, not a Rust requirement, but it makes the pattern easy to follow: every variant's span is in the same position.

Testing the AST Structure

The AST module does not have its own test module — the AST types are tested implicitly through the parser and downstream passes. However, the design ensures correctness at the type level:

  • Recursive types use Box, which prevents unbounded stack allocation.
  • Optional fields use Option, making missing data explicit at the type level rather than using sentinel values.
  • Enum variants with the wrong field types are caught at compile time — you cannot accidentally put an expression where a statement is expected.

Key Takeaways

  1. The AST is a semantic tree — it discards syntactic sugar (semicolons, grouping parens, braces) and preserves only the grammatical structure needed for analysis and code generation.
  2. Every node carries its span via a span() method. No AST node is ever created without source location information.
  3. Sum types (enums) provide exhaustiveness checking. Adding a new expression variant forces the compiler to check every match on expressions.
  4. Boxed recursion (Box<AstExpr>) enables tree-shaped data with deterministic ownership and no garbage collection.
  5. The hierarchy is four levels deep: types → expressions → statements → declarations. Each level builds on the previous, and the root is a flat list of declarations.

6   The Parser

What is Parsing?

Parsing is the process of converting a flat stream of tokens into a structured tree that represents the grammatical structure of the program. Where the lexer grouped characters into tokens, the parser groups tokens into larger constructs: declarations, statements, and expressions.

There are two main families of parsing algorithms: top-down (recursive descent) and bottom-up (LR, LALR). Recursive descent is the most popular for hand-written compilers because it maps directly to the grammar, gives great error messages, and is straightforward to debug. Bottom-up parsers (like Yacc/Bison) are typically generated from a grammar specification and can handle a wider class of grammars, but produce opaque error messages and are harder to extend.

Within recursive descent, expression parsing has its own sub-problem: handling operator precedence without encoding it in a cascade of functions. Two approaches dominate: Pratt parsing (top-down operator precedence) and precedence climbing. Both use numeric binding powers to decide which operator binds tighter. Pratt is more general — it handles prefix, postfix, and infix operators uniformly — while precedence climbing is simpler but typically only handles infix.

Tradeoffs & Design Space

Recursive descent vs generated parsers:

  • Recursive descent: Each grammar rule becomes a function. Easy to debug (step through in a debugger), easy to extend, trivial to add error recovery. Downside: left-recursive grammar rules cause infinite recursion (must be rewritten as right-recursive loops).
  • Generated (Yacc/Bison, ANTLR): Grammar is declarative; parser is generated. Can handle LR(1) grammars including left recursion. Downside: poor error messages, hard to debug, generated code is opaque, adding error recovery requires grammar-level hacks.

For a pedagogical compiler like L1S, recursive descent is the obvious choice. The grammar is small enough that hand-writing a parser is straightforward, and the error recovery patterns are educational.

Pratt vs precedence climbing for expressions:

  • Pratt: A single loop reads a prefix, then repeatedly checks for infix operators. Each operator has a binding power. Prefix and infix handling are unified. Adding a new operator means adding an entry to the binding power table and a match arm.
  • Precedence climbing: Similar but typically only handles infix operators. Prefix/postfix must be handled separately.
  • Function cascade: The traditional approach where each precedence level is a separate function (parse_add_expr calls parse_mul_expr, etc.). Easy to understand but verbose: adding a new precedence level requires inserting a new function in the chain.

L1S uses a Pratt parser for expressions, combined with recursive descent for declarations and statements. This hybrid is common in production compilers (rustc, GCC, Clang all use variants of this approach).

Ignis Implementation

Ignis uses a hand-written recursive descent parser with a Pratt expression parser. The Parser struct wraps the token stream and a diagnostics accumulator, providing cursor operations that every parse method builds on:

Cursor Operations: The Parser's Vocabulary

Every recursive descent parser is built from a handful of primitives for looking at and consuming tokens. These are the building blocks that higher-level parse methods compose:

src/parser.rs — cursor operations
fn is_at_end(&self) -> bool {
    self.peek_token() == Token::Eof
}

fn peek(&self) -> &SpannedToken {
    &self.tokens[self.pos]
}

fn peek_token(&self) -> Token {
    if self.pos >= self.tokens.len() { return Token::Eof; }
    self.tokens[self.pos].token.clone()
}

fn peek_nth(&self, n: usize) -> Token {
    let idx = self.pos + n;
    if idx >= self.tokens.len() { return Token::Eof; }
    self.tokens[idx].token.clone()
}

fn advance(&mut self) -> &SpannedToken {
    let t = &self.tokens[self.pos];
    self.pos += 1;
    t
}

The three consumption patterns — check, match_token, and expect — form a taxonomy of lookahead strategies:

fn check(&self, tok: &Token) -> bool {
    self.peek_token() == *tok         // pure predicate, cursor unchanged
}

fn match_token(&mut self, tok: &Token) -> bool {
    if self.check(tok) { self.advance(); true }       // consume if matches
    else { false }
}

fn expect(&mut self, tok: &Token) -> Result<Span, Span> {
    if self.check(tok) { ... Ok(span) }           // consume or signal error
    else { Err(self.current_span()) }
}

check is a pure predicate — look but don't touch. match_token is for optional syntax (like commas between parameters): consume if present, no error if absent. expect is for required syntax: consume on success, return an error span on failure so the caller can synthesize a placeholder.

Why return Result for expect? The caller uses unwrap_or_else to supply a synthetic value (a dummy span, an empty name, etc.). The error is already recorded in the diagnostics by the caller's choice. This keeps expect pure — it reports the failure; the caller decides how to recover.

Error Recovery Pattern

The most important design principle in the Ignis parser: never stop after one error. Every parse method can fail gracefully by inserting a synthetic value and continuing. Consider parse_func_decl:

fn parse_func_decl(&mut self, pub_: bool) -> AstFuncDecl {
    let span = self.expect(&Token::Func).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenParen).unwrap_or_else(|_| Span::dummy());
    let return_type = self.parse_type();
    let params = if self.match_token(&Token::Comma) { self.parse_params() }
                 else { Vec::new() };
    self.expect(&Token::CloseParen).unwrap_or_else(|_| Span::dummy());
    let body = self.parse_block();
    AstFuncDecl { span, pub_, name, return_type, params, body }
}

If the function name is missing, the parser inserts "_" as a synthetic name and continues. If a closing paren is missing, it gets a dummy span. The error is already recorded in self.diag, but parsing proceeds to find more errors. This maximizes the number of issues reported in a single compilation.

Every loop also has an infinite loop guard. If a bug causes the parser to loop without advancing (e.g., a malformed declaration returns None every iteration), a panic fires after 10,000 iterations:

fn parse_program(&mut self) -> AstProgram {
    let mut decls = Vec::new();
    let mut guard = 0u32;
    while !self.is_at_end() {
        guard += 1;
        if guard > 10000 { panic!("INFINITE LOOP"); }
        if let Some(decl) = self.parse_declaration() {
            decls.push(decl);
        } else {
            self.advance(); // force progress on parse failure
        }
    }
    AstProgram { decls }
}

The guard paired with self.advance() on failure is the double lock. Even if parse_declaration returns None without consuming anything, the force-advance guarantees the loop makes progress.

Declaration Parsing

Declarations are the top-level constructs of a program: functions, structs, enums, specs, impls, typedefs, imports, and static variables. The dispatch is a single match on the current token, with pub handled as an optional prefix modifier:

fn parse_declaration(&mut self) -> Option<AstDecl> {
    let is_pub = self.match_token(&Token::Pub);
    match self.peek_token() {
        Token::Struct   => Some(AstDecl::Struct(self.parse_struct_decl(is_pub))),
        Token::Enum     => Some(AstDecl::Enum(self.parse_enum_decl(is_pub))),
        Token::Spec     => Some(AstDecl::Spec(self.parse_spec_decl(is_pub))),
        Token::Impl     => Some(AstDecl::Impl(self.parse_impl_decl())),
        Token::Func     => Some(AstDecl::Func(self.parse_func_decl(is_pub))),
        Token::Typedef  => Some(AstDecl::Typedef(self.parse_typedef_decl(is_pub))),
        Token::Import   => Some(AstDecl::Import(self.parse_import())),
        Token::Static   => Some(AstDecl::StaticVar(self.parse_static_var_decl())),
        _ => { if is_pub { self.error("expected declaration after 'pub'", ...); } None }
    }
}

The pub keyword is consumed greedily by match_token, then passed to the sub-parser. Declarations like impl and import reject it by emitting an error. This pattern keeps the dispatch clean and avoids duplicating the pub-handling logic in each sub-parser.

Each declaration follows the same template: consume a keyword, parse a name, parse delimiters and body, return the AST node. For example, a struct declaration:

fn parse_struct_decl(&mut self, pub_: bool) -> AstStructDecl {
    let span = self.expect(&Token::Struct).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenBrace).unwrap_or_else(|_| Span::dummy());
    let fields = self.parse_fields();
    self.expect(&Token::CloseBrace).unwrap_or_else(|_| Span::dummy());
    AstStructDecl { span, pub_, name, fields }
}

Ignis uses a non-standard function declaration syntax: func main(i32, argc: i32, argv: @char) { ... }. The return type appears first inside the parentheses, followed by optional named parameters after a comma. This means func main(i32) { } declares a function taking no parameters returning i32, while func main(i32, x: i32, y: i32) { } adds named parameters.

Why return type first? In most C-like languages, a bare type with no name (like i32) would be illegal. Ignis uses this to distinguish between "zero parameters" (just a return type) and "named parameters" (comma, then name:type pairs). It's unusual but keeps the grammar compact.

Statement Parsing

Statements are the executable units inside function bodies. Each statement type starts with a distinct keyword, making dispatch a simple match:

fn parse_statement(&mut self) -> Option<AstStmt> {
    match self.peek_token() {
        Token::Return    => Some(AstStmt::Return(self.parse_return())),
        Token::Let       => Some(AstStmt::VarDecl(self.parse_var_decl())),
        Token::If        => Some(AstStmt::If(self.parse_if())),
        Token::While     => Some(AstStmt::While(self.parse_while())),
        Token::For       => Some(AstStmt::For(self.parse_for())),
        Token::Asm       => Some(AstStmt::Asm(self.parse_asm())),
        Token::OpenBrace => Some(AstStmt::Block(self.parse_block())),
        Token::Import    => Some(AstStmt::Import(self.parse_import())),
        _ => { // expression statement: any expression followed by semicolon
            if self.is_at_end() { return None; }
            let expr = self.parse_expr();
            self.expect(&Token::Semicolon).unwrap_or_else(|_| Span::dummy());
            Some(AstStmt::Expr(expr))
        }
    }
}

The fallthrough case handles expression statements — any expression followed by a semicolon. This is how foo(); (function call as a statement) and x = 42; (assignment as a statement) work. The if/while/let keywords are reserved, so there's no ambiguity: an identifier starting a statement can only be an expression statement.

The else if chain is handled recursively in parse_if:

fn parse_if(&mut self) -> AstIf {
    // ... parse condition and then_block ...
    let else_branch = if self.match_token(&Token::Else) {
        match self.peek_token() {
            Token::If => Some(Box::new(AstStmt::If(self.parse_if()))),
            _ => Some(Box::new(AstStmt::Block(self.parse_block()))),
        }
    } else { None };
    AstIf { span, cond, then_block, else_branch }
}

When else is followed by if, the parser recursively calls parse_if, creating a nested chain. The statement if (a) {} else if (b) {} else {} becomes If(cond=a, else=Some(If(cond=b, else=Some(Block)))).

Pratt Expression Parsing

Expressions are the most elegant part of the parser. Instead of encoding precedence in a cascade of functions, Pratt parsing uses binding powers — numeric values assigned to each operator that determine how tightly it binds its operands:

Operator binding powers (lower = looser, higher = tighter):

= (assignment) ........... 10
|| (logical or) .......... 20
&& (logical and) ......... 30
== != (equality) ........... 40
< > <= >= (comparison) ..... 50
| (bitwise or) .......... 60
^ (bitwise xor) ......... 70
& (bitwise and) ......... 80
<< >> (shift) .............. 90
+ - (addition) ........... 100
* / % (multiplication) .... 110
( [ { . -> (postfix) ...... 130

The core algorithm is just two functions, parse_expr_bp and the binding power table infix_bp:

fn parse_expr(&mut self) -> AstExpr {
    self.parse_expr_bp(0)       // start with minimum binding power 0
}

fn parse_expr_bp(&mut self, min_bp: u8) -> AstExpr {
    let mut lhs = self.parse_prefix();       // literal, ident, unary, parens
    loop {
        let bp = self.infix_bp();             // peek at next operator's power
        if bp == 0 || bp < min_bp { break; } // stop if looser than minimum
        lhs = self.parse_infix(lhs, bp);       // consume operator and RHS
    }
    lhs
}

fn infix_bp(&self) -> u8 {
    match self.peek_token() {
        Token::Eq => 10,
        Token::PipePipe => 20,
        Token::AmpAmp => 30,
        Token::EqEq | Token::BangEq => 40,
        Token::Lt | Token::Gt | Token::LtEq | Token::GtEq => 50,
        Token::Pipe => 60,
        Token::Caret => 70,
        Token::Amp => 80,
        Token::LtLt | Token::GtGt => 90,
        Token::Plus | Token::Minus => 100,
        Token::Star | Token::Slash | Token::Percent => 110,
        Token::OpenParen | Token::OpenBracket | Token::OpenBrace
            | Token::Dot | Token::Arrow => 130,
        _ => 0,  // not an infix operator: stop
    }
}

The algorithm works in a tight loop:

  1. Parse a prefix expression (integer literal, string, identifier, parenthesized expression, unary operator like -, !, @).
  2. Peek at the next token. Look up its binding power. Zero means "not an infix operator" — stop here.
  3. Compare the operator's binding power with the current minimum. If the operator binds tighter (bp >= min_bp), parse it as an infix (which recursively calls parse_expr_bp with a higher minimum for the right-hand side).
  4. If the operator binds looser, stop and return the current lhs.

Left vs right associativity is controlled by the bp value passed to the recursive call:

  • Left-associative operators (+, *, ==): the RHS is parsed with bp + 1, meaning 1 + 2 + 3 becomes ((1 + 2) + 3).
  • Right-associative operators (=): the RHS is parsed with bp (not bp + 1), so x = y = 0 becomes x = (y = 0).

Unary operators (-, !, ~, &, @) all use binding power 120 for their operand — higher than all binary operators. This means -x * y parses as (-x) * y, not -(x * y).

The parse_infix method handles both binary operators and postfix syntax (function calls, array indexing, field access, struct initialization) by matching on the current token:

fn parse_infix(&mut self, lhs: AstExpr, bp: u8) -> AstExpr {
    match self.peek_token() {
        // Postfix operators (binding power 130)
        Token::OpenParen   => { /* function call: foo(args) */ }
        Token::OpenBracket => { /* array index: a[i] */ }
        Token::OpenBrace   => { /* struct init: Point { x: 1 } */ }
        Token::Dot         => { /* field access: p.x */ }
        Token::Arrow       => { /* pointer field: p->x */ }

        // Binary operators follow the same pattern:
        Token::Plus  => { self.advance();
                         let rhs = self.parse_expr_bp(bp + 1);
                         AstExpr::Binary(BinaryOp::Add, Box::new(lhs), Box::new(rhs), op_span) }
        // ... Star, Slash, Percent, Minus, Amp, Pipe, Caret, EqEq, BangEq, ...

        // Assignment is right-associative: use bp, not bp + 1
        Token::Eq   => { self.advance();
                         let rhs = self.parse_expr_bp(bp); // no +1
                         AstExpr::Assign(Box::new(lhs), Box::new(rhs), op_span) }
        _ => lhs,
    }
}

Postfix operators like (, [, . have the highest binding power (130), so they always bind tighter than any binary operator. The expression a[i] + 1 correctly parses as (a[i]) + 1 rather than a[(i) + 1].

Type Parsing

Types form their own mini-language that is parsed by a single recursive method. This is possible because Ignis types use prefix syntax consistently:

fn parse_type(&mut self) -> AstType {
    // [expr] type  → array type (prefix syntax)
    if self.match_token(&Token::OpenBracket) {
        let size = self.parse_expr();
        self.expect(&Token::CloseBracket);
        let inner = self.parse_type();
        return AstType::Array(Box::new(size), Box::new(inner), span);
    }
    // @type  → pointer type (prefix syntax)
    if self.match_token(&Token::At) {
        let inner = self.parse_type();
        return AstType::Ptr(Box::new(inner), span);
    }
    // Func<ret, args...>  → function type
    if self.match_token(&Token::Func) {
        // parse Func<ret_type, arg1_type, arg2_type, ...>
    }
    // Primitive or named type (base case)
    match self.peek_token() {
        Token::I32  => { self.advance(); AstType::Prim(PrimType::I32, span) }
        Token::Bool => { self.advance(); AstType::Prim(PrimType::Bool, span) }
        // ... all primitives ...
        Token::Ident(ref name) => { self.advance(); AstType::Named(name.clone(), span) }
        _ => { self.error("expected type", span); AstType::Prim(PrimType::Void, span) }
    }
}

The recursive calls handle nesting naturally: @[10]@i32 is a pointer to an array of ten pointers to i32. Each prefix operator (@, [n], Func<...>) recursively calls parse_type for the element type. Named types and primitives are the base cases that terminate the recursion.

Traced Example: else if Chaining

Given if (a) { f1(); } else if (b) { f2(); } else { f3(); }:

  1. parse_if consumes if, parses condition a, parses then_block.
  2. Sees else, peeks at next token. It's if (not {), so it recursively calls parse_if.
  3. The recursive call parses if (b) { f2(); } else { f3(); } as a nested AstIf.

The resulting tree: If(cond=a, then=Block(f1), else=If(cond=b, then=Block(f2), else=Block(f3))). This recursive representation is elegant and matches how the parser naturally handles else if without special-casing.

Common Pitfalls

  1. Infinite loops: Every while loop must guarantee progress. Always pair a guard counter with force-advance on failure. Test with using cargo test to ensure the guards don't fire.
  2. Forgetting to advance: check doesn't consume. match_token does. Mixing them up is the most common bug. Use match_token for optional syntax, check for lookahead decisions, expect for required syntax.
  3. Pratt off-by-one: Left-associative operators use bp + 1 for the RHS. Right-associative operators use bp. Getting this wrong silently produces wrong parse trees. Always test with 1 + 2 + 3 (left-associate) and x = y = 0 (right-associate).
  4. Error recovery swallowing tokens: Synthetic values let the parser continue, but the next call might be confused by the missing token. The guard counter catches this. If a test hangs, check the guard fires.
Adding a new statement or operator: You need to touch exactly these places: add a Token variant (if new syntax), add an AstStmt or AstExpr variant, add the parsing method, add an infix_bp entry (for operators), add dispatch in parse_statement or parse_infix. That's it — the Pratt loop handles the rest.

7   The Type System

What Is a Type System?

A type system is a set of rules that assigns a type to every expression in a program and checks that operations are applied to compatible types. If an expression 1 + true reaches the type checker, it should be rejected because addition is not defined between an integer and a boolean.

At a minimum, a compiler's internal type representation needs to support:

  • Equality checking: Is @i32 the same type as @i32? Type checking requires comparing types constantly.
  • Structural queries: What are the fields of this struct? What are the parameter types of this function?
  • Size and alignment: How many bytes does this type occupy? Where should it be aligned in memory?
  • Forward references: A type like Node with field next: @Node references itself before it's fully defined.

Tradeoffs: Type Representation Strategies

Non-interned types (tree-based): Each type is a boxed tree node. Ptr(I32) is a heap-allocated struct. Equality requires recursive traversal: compare Ptr discriminant, then recurse into the pointee. Simple to implement, but equality is O(n) in the size of the type, and pointer equality doesn't work because each occurrence creates a new allocation.

Interned types (arena-based): Each unique type is stored once in a vector, referred to by a lightweight integer ID. Equality is a single integer comparison. Building a type checks whether it already exists (dedup). This is the approach used by rustc (TyS/TyCtxt), MLKit, and many production compilers.

Nominal vs structural: In a nominal system, two types are equal only if they have the same name. In a structural system, two types are equal if they have the same shape. Ignis uses structural equality for primitives and compound types, but for named types (structs, enums) it falls back to nominal identity: two Point types are equal only if they refer to the same struct definition.

Ignis uses an interned type arena with structural deduplication for compound types. This gives O(1) equality for the common case (comparing TypeId values) while keeping the implementation simple.

Ignis Implementation

TypeId — The Handle

Every type in the Ignis compiler is referred to by a TypeId, a lightweight newtype over u32. It's Copy, so passing it around is a single register move:

src/types.rs — TypeId
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct TypeId(u32);

impl TypeId {
    pub fn new(id: u32) -> Self { TypeId(id) }
    pub fn index(self) -> usize { self.0 as usize }
}

impl fmt::Display for TypeId {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {

    }
}

The Display impl prints TypeId as %0, %1, etc., making debug output compact and readable. The Eq and Hash derives mean TypeId can be used directly as a hash map key (useful if the O(n) linear scan ever needs replacing).

TypeId is the currency of the type system. Every function that deals with types takes and returns TypeId — never a raw &TypeNode. This indirection is what enables structural sharing and O(1) equality.

TypeNode — The Type Representation

The actual type data lives in the TypeNode enum. Each variant represents one kind of type that can exist in the compiler's internal model:

src/types.rs — TypeNode enum
#[derive(Debug, Clone)]
pub enum TypeNode {
    // Primitives: standalone variants, no wrapper
    Void, Bool, Char,
    U8, I8, U16, I16, U32, I32, U64, I64,
    Usize, Isize,

    // Compound types use TypeId references for structural sharing
    Ptr(TypeId),               // @T — pointer to T
    Array(u64, TypeId),         // [N]T — array of N x T
    Func(Vec<TypeId>, TypeId), // Func<ret, arg1, ...>

    // Named and alias types: carry name + resolved TypeId
    Named(String, TypeId),
    Alias(String, TypeId),

    // User-defined types: ID indexes into separate StructInfo/EnumInfo vectors
    Struct(TypeId),
    Enum(TypeId),

    // Spec (interface) type: carries method signatures
    Spec(String, Vec<(String, TypeId)>),
}

Key design decisions in TypeNode:

  • Primitives are standalone variants, not nested inside a Prim(PrimType) wrapper. This eliminates one level of indirection when matching on types — matching TypeNode::I32 directly is faster than matching TypeNode::Prim(x) and then matching x.
  • Ptr and Array hold TypeIds, not boxed TypeNodes. This is the key to structural sharing. The i32 inside Ptr(i32) is the same TypeId(%1) as every other reference to i32 in the program.
  • Struct and Enum store a separate TypeId that indexes into the arena's parallel structs/enums vectors. This avoids making TypeNode carry potentially large field/variant lists.
  • Named and Alias carry a TypeId alongside the name. Initially this is a placeholder; it gets resolved to the actual type as the compiler encounters the type's definition.

TypeArena — The Intern Pool

The arena is the central store that owns all type nodes. It comprises three vectors:

src/types.rs — TypeArena
#[derive(Debug)]
pub struct TypeArena {
    types: Vec<TypeNode>,      // the DAG of type nodes
    structs: Vec<StructInfo>,  // struct metadata (name, fields, size, align)
    enums: Vec<EnumInfo>,       // enum metadata (name, variants, noprefix)
}

The intern operation is the primary insertion path. It checks whether an equivalent node already exists, and returns the existing ID if so — this is structural deduplication:

impl TypeArena {
    pub fn intern(&mut self, node: TypeNode) -> TypeId {
        match self.lookup(&node) {
            Some(tid) => tid,           // already exists: return existing ID
            None => {                   // new type: assign next ID, push
                let id = TypeId(self.types.len() as u32);
                self.types.push(node);
                id
            }
        }
    }

    pub fn intern_no_dedup(&mut self, node: TypeNode) -> TypeId {
        let id = TypeId(self.types.len() as u32);
        self.types.push(node);
        id
    }
}

intern performs a linear scan (O(n) in the number of types). For small compilers this is fine. If performance ever becomes an issue, the linear scan can be replaced with a hash map keyed on a discriminant + inner IDs. But the O(1) equality (TypeId == TypeId) is a permanent benefit regardless.

intern_no_dedup skips the lookup entirely. This is used for Named types ("this is Point even if another type has the same shape") and for types that will be mutated later (placeholder structs).

Structural Equality: nodes_equal

The dedup logic depends on nodes_equal, a recursive function that compares two TypeNodes structurally. This function is the heart of the type system's equality semantics:

src/types.rs — nodes_equal
fn nodes_equal(a: &TypeNode, b: &TypeNode, arena: &TypeArena) -> bool {
    match (a, b) {
        // Primitives: same discriminant = equal
        (TypeNode::Void, TypeNode::Void)
        | (TypeNode::Bool, TypeNode::Bool)
        | (TypeNode::I32, TypeNode::I32) => true,

        // Compounds: compare inner TypeIds (already deduplicated)
        (TypeNode::Ptr(a), TypeNode::Ptr(b)) => a == b,
        (TypeNode::Array(na, ta), TypeNode::Array(nb, tb)) => na == nb && ta == tb,
        (TypeNode::Func(a_args, a_ret), TypeNode::Func(b_args, b_ret)) => {
            a_ret == b_ret && a_args.len() == b_args.len()
            && a_args.iter().zip(b_args).all(|(x, y)| x == y)
        }

        // Named/Alias: compare by name (the name is the identity)
        (TypeNode::Named(a, _), TypeNode::Named(b, _)) => a == b,
        (TypeNode::Alias(a, _), TypeNode::Alias(b, _)) => a == b,

        // Struct/Enum: compare by ID (not the field contents)
        (TypeNode::Struct(a), TypeNode::Struct(b)) => a == b,
        (TypeNode::Enum(a), TypeNode::Enum(b)) => a == b,

        // Spec: compare by name
        (TypeNode::Spec(a, _), TypeNode::Spec(b, _)) => a == b,

        _ => false,
    }
}

The crucial insight: comparing compound types like Ptr(a) == Ptr(b) is a single u32 comparison (a == b) because the TypeIds inside them have already been deduplicated. When you build Ptr(i32), the i32 TypeId is already unique across the arena. So nodes_equal never needs to recurse into inner types — it just compares their IDs. Equality is transitive through the arena.

Structural vs nominal: For primitives and compounds (Ptr, Array, Func), Ignis uses structural equality: Ptr(i32) is the same type wherever it appears. For structs and enums, equality is nominal: Struct(%0) and Struct(%1) are different even if the two structs have identical fields. The Struct/Enum equality by ID reflects this.

Size and Alignment

Every type must answer two critical layout questions: how many bytes it occupies (size_of) and what boundary it must be aligned to (align_of). These are computed by recursive functions on TypeNode:

src/types.rs — size_of and align_of
pub fn size_of(&self, id: TypeId) -> u64 {
    match self.get(id) {
        TypeNode::Void => 0,
        TypeNode::Bool | TypeNode::Char | TypeNode::U8 | TypeNode::I8 => 1,
        TypeNode::U16 | TypeNode::I16 => 2,
        TypeNode::U32 | TypeNode::I32 => 4,
        TypeNode::U64 | TypeNode::I64 | TypeNode::Usize | TypeNode::Isize => 8,
        TypeNode::Ptr(_) => 8,
        TypeNode::Array(n, elem) => n * self.size_of(*elem),
        TypeNode::Struct(sid) => self.structs.get(sid.index()).map_or(0, |s| s.size),
        _ => 0,
    }
}

pub fn align_of(&self, id: TypeId) -> u64 {
    match self.get(id) {
        TypeNode::Void => 1,
        TypeNode::Bool | TypeNode::Char | TypeNode::U8 | TypeNode::I8 => 1,
        TypeNode::U16 | TypeNode::I16 => 2,
        TypeNode::U32 | TypeNode::I32 => 4,
        TypeNode::U64 | TypeNode::I64 | TypeNode::Usize | TypeNode::Isize => 8,
        TypeNode::Ptr(_) => 8,
        TypeNode::Array(_, elem) => self.align_of(*elem),
        TypeNode::Struct(sid) => self.structs.get(sid.index()).map_or(1, |s| s.align),
        _ => 1,
    }
}
ABI sizes without padding: These are ABI sizes. For structs, the size and alignment are computed externally by the struct layout pass and stored in StructInfo. The size_of/align_of methods just read precomputed values. For arrays, size is count * element_size with no trailing padding. Real compilers must handle struct field padding, but Ignis keeps it simple.

The arena provides accessors that route from a TypeId through the TypeNode to the metadata:

pub fn add_struct(&mut self, info: StructInfo) -> TypeId {
    let id = TypeId(self.structs.len() as u32);
    let tid = self.intern(TypeNode::Struct(id));
    self.structs.push(info);
    tid
}

pub fn get_struct(&self, id: TypeId) -> Option<&StructInfo> {
    if let TypeNode::Struct(sid) = self.get(id) {
        self.structs.get(sid.index())
    } else { None }
}

The add_struct method assigns the struct a TypeId first, interns the Struct(id) node, then pushes the metadata. This ordering is important: any TypeNode::Struct(id) that appears before the info is pushed will return a placeholder until the struct is fully defined.

Forward References and Placeholders

Ignis allows types to reference other types that are defined later in the source. A self-referential struct like Node with a pointer to itself requires special handling:

struct Node {
    value: i32;
    next: @Node;  // forward reference to Node itself
}

To support this, the type resolution phase first creates placeholder entries with empty fields, then fills them in:

pub fn add_struct_placeholder(&mut self) -> TypeId {
    let id = TypeId(self.structs.len() as u32);
    let info = StructInfo {
        name: String::new(), fields: Vec::new(), size: 0, align: 1,
    };
    let tid = self.intern(TypeNode::Struct(id));
    self.structs.push(info);
    tid
}

The placeholder creates a struct entry with empty fields and size 0. The compiler later fills in the name, fields, size, and alignment using get_struct_mut. Any type that references the placeholder struct gets the correct TypeId immediately — only the field/offset data is deferred. This means structs can reference their own TypeId in field types without a chicken-and-egg problem.

Traced Example: Building a Function Type

Let's trace the construction of Func<i32, i32, void> — a function taking two i32 parameters and returning void. This example shows how deduplication works in practice:

In code:

let mut arena = TypeArena::new();

let i32  = arena.intern(TypeNode::I32);   // %1
let ft   = arena.intern(TypeNode::Func(vec![i32, i32], void));  // %2

Notice in step 3: interning I32 a second time does not create a new entry. It returns %1 again. This means the Func node at %2 stores both parameter slots as %1 — the same TypeId. The arena forms a DAG (directed acyclic graph), not a tree. The i32 node is shared between the two parameter positions.

If we rebuild the same function type from scratch:

let ft2 = arena.intern(TypeNode::Func(vec![i32, i32], void));
assert_eq!(ft, ft2);  // O(1) comparison of two u32 values

The assertion passes because nodes_equal compares the inner TypeIds, which are already deduplicated. No recursive traversal of the type tree is needed.

Traced Example: Struct Layout

Consider building a Point struct:

struct Point { x: i32; y: i32; }

The compiler processes this roughly as:

  1. Placeholder pass: arena.add_struct_placeholder() reserves ID %3 for Point with empty fields. The struct name is temporarily "".
  2. Field resolution: The compiler resolves x: i32 → FieldInfo { name: "x", type_: %1 (I32), offset: 0 } and y: i32 → FieldInfo { name: "y", type_: %1, offset: 4 }.
  3. Finalize: Using get_struct_mut, the compiler sets name = "Point", size = 8, align = 4, and fills in the fields.
  4. Query: arena.size_of(%3) returns 8 (the stored size). arena.get(%3) returns &TypeNode::Struct(TypeId(0)) (the struct ID).

The @Point type (pointer to Point) is then built as arena.intern(TypeNode::Ptr(%3)), giving a unique TypeId for Ptr(Point). Every occurrence of @Point in the source maps to this same TypeId.

Common Pitfalls

  1. Missing dedup match arm: If you add a new TypeNode variant and forget to add it to nodes_equal, the match falls through to _ => false. Every instance of the new type will be considered unique — defeating deduplication silently. Always add a match arm, even if it's just comparing inner IDs.
  2. Circular types without placeholders: A struct that references itself (like next: @Node) requires pre-allocating a TypeId via add_struct_placeholder. Without it, building the struct's field types would need the struct's own TypeNode, which doesn't exist yet.
  3. Size/alignment for incomplete types: Calling size_of on a Struct before its StructInfo is populated returns 0. This can cause subtle bugs in code generation. Always finalize structs before querying layout.
  4. Display format confusion: TypeId displays as %0, %1, etc. When debugging, print the TypeNode alongside the ID: eprintln!("%{} = {:?}", id, arena.get(id)).

}

8   The Symbol Table

Concept: Name → Definition Mapping

A compiler needs to answer one question more than any other: what does this name refer to? When the parser produces an identifier node like x or main, the compiler must resolve it to a specific declaration — a variable, a function, a type, or an enum variant. The data structure that handles this is the symbol table: a mapping from names to their definitions.

Consider this program:

func main(i32) {
    let x: i32 = 1;
    {
        let x: i32 = 2;  // shadows outer x
    }
}

The identifier x appears three times: two definitions (the outer and inner declarations) and one use (inside the inner block for the initializer 2). The use of x in the initializer 2 refers to the inner definition, not the outer one. This is the essence of lexical scoping: a name refers to the innermost enclosing definition.

Tradeoffs & Design Space

Flat map vs. scope chain. The simplest approach is a single flat HashMap<String, Entry>. It works for languages with no nesting (some assembly dialects) but breaks badly for structured programming. You would need to mangle names (e.g., _main_inner_x) to handle shadowing, which produces confusing error messages and makes semantic analysis impossible.

The standard solution is a scope chain: a linked list of nested maps. Each scope has a parent pointer. Name lookup starts at the innermost scope and walks outward. This directly mirrors how programming languages handle nested blocks, function bodies, and module boundaries.

Persistent vs. ephemeral scopes. Some compilers (like GCC's early C frontend) use a tree of scopes that persists for the entire compilation. Others allocate scopes temporarily during parsing and type checking, discarding them afterward. Persisting scopes enables querying between functions (e.g., for module-level analysis), at the cost of memory. L1S takes a middle path: scopes are stored in a Vec and never deallocated, but only the chain from current to root is active.

What does each symbol entry store? At minimum, the entry needs a type and a declaration span. Some compilers also store mutability, visibility, alignment hints, or linker symbols. L1S keeps it minimal: a SymbolKind enum that discriminates between variables, functions, types, enum variants, specs, modules, and statics.

Ignis Implementation

Ignis implements the classic scope chain pattern. The core types are Scope, ScopeId, and SymbolTable:

src/symbol.rs — scope chain infrastructure
pub struct Scope {
    pub symbols: HashMap<String, SymbolEntry>,
    pub parent: Option<ScopeId>,
}

pub struct SymbolTable {
    scopes: Vec<Scope>,
    current: ScopeId,
}

ScopeId is a newtype over u32 that indexes into the scopes vector. The root scope (index 0) has parent: None and holds all global declarations. push_scope appends a new scope whose parent is the current scope, then updates current. pop_scope follows the parent pointer back:

pub fn push_scope(&mut self) -> ScopeId {
    let id = ScopeId(self.scopes.len() as u32);
    self.scopes.push(Scope {
        symbols: HashMap::new(),
        parent: Some(self.current),
    });
    self.current = id;
    id
}

pub fn pop_scope(&mut self) {
    if let Some(parent) = self.scopes[self.current.0 as usize].parent {
        self.current = parent;
    }
}

The critical insight: scopes are never removed from the vector. This means ScopeId values remain valid for the lifetime of the table, enabling persistent queries. But only the chain from current to root is reachable via lookup.

Lookup walks the chain:

src/symbol.rs — scope-walking lookup
pub fn lookup(&self, name: &str) -> Option<&SymbolEntry> {
    let mut sid = Some(self.current);
    while let Some(id) = sid {
        let scope = &self.scopes[id.0 as usize];
        if let Some(entry) = scope.symbols.get(name) {
            return Some(entry);
        }
        sid = scope.parent;
    }
    None
}

The loop terminates because the root scope has parent: None. This is O(d) where d is the nesting depth — typically a small constant in practice.

Two insertion paths: insert adds to the current scope (used for local variables and parameters). insert_global always adds to scope 0 (used for top-level declarations like functions and structs). The split ensures that a function body's temporary scopes don't accidentally capture global declarations.

Symbol entries carry four fields:

pub struct SymbolEntry {
    pub name: String,
    pub kind: SymbolKind,
    pub span: Span,
    pub is_pub: bool,
}

The span enables "previously defined here" error diagnostics. The kind discriminates via the SymbolKind enum: Variable(TypeId), Function(TypeId), TypeDecl(TypeId), Spec(Vec<(String, TypeId)>), EnumVariant(u64), Module(Vec<String>), and StaticVar(TypeId).

Why an enum and not traits? An enum is closed and exhaustive: the type checker can match on every variant and the compiler ensures no case is missed. A trait-based approach would be more extensible but would require dynamic dispatch or downcasting. For a fixed set of symbol kinds, an enum is simpler and faster.

Example: Tracing Nested Scope Resolution

Let's trace the symbol table state for this program:

What Does a SymbolEntry Hold?

Each symbol entry stores the name, its kind, the source span of its declaration, and whether it's public. The SymbolKind enum captures every category of named entity in the language:

src/symbol.rs — SymbolKind and SymbolEntry
pub enum SymbolKind {
    Variable(TypeId),              // Local variable or parameter
    Function(TypeId),              // Function (full function type)
    TypeDecl(TypeId),              // Type name (struct, enum, typedef)
    Spec(Vec<(String, TypeId)>), // Spec/trait with method signatures
    EnumVariant(u64),              // Enum variant with its discriminant
    Module(Vec<String>),          // Module path for imports
    StaticVar(TypeId),             // Global/static variable
}

pub struct SymbolEntry {
    pub name: String,
    pub kind: SymbolKind,
    pub span: Span,
    pub is_pub: bool,
}

Each variant carries precisely the data needed for that kind: variables carry a TypeId so the type checker can verify assignments and expressions; functions carry their full function type (parameters + return type) so call sites can be validated; enum variants carry their numeric discriminant so they can be used as integer constants in the IR.

The SymbolTable API

The full public API is minimal but complete:

  • new() — Creates the root scope with ScopeId(0).
  • push_scope() / pop_scope() — Enter and exit a nested scope. Returns the new ScopeId.
  • insert(name, kind, span, is_pub) — Add to the current scope. Returns Err on duplicate.
  • insert_global(name, kind, span, is_pub) — Always add to the root scope, regardless of current nesting.
  • lookup(name) — Walk the scope chain from current to root. Returns the innermost match.
  • lookup_in_scope(name, scope_id) — Check only a specific scope (no walking). Used for targeted queries.
  • current_scope() — Returns the current ScopeId for saving/restoring.

Duplicate Detection & Error Reporting

insert checks for duplicates within the same scope and returns Err(SymbolError::AlreadyDefined) with the previous declaration's span. This is important: shadowing is allowed across scopes (an inner let x hides an outer one), but duplicate names in the same scope is an error. The type checker uses this to produce precise messages like error: redefinition of 'x' pointing at both the new and previous definitions.

Debugging scope chains. When the type checker reports "undefined identifier" for something you know exists, the scope chain is likely wrong. A quick debug helper walks the chain and prints each scope's keys: 
fn dump_scopes(&self) {
    let mut sid = Some(self.current);
    while let Some(id) = sid {
        println!("scope {}: {:?}", id.0, self.scopes[id.0 as usize].symbols.keys());
        sid = self.scopes[id.0 as usize].parent;
    }
}
The most common bugs are forgetting to push_scope at function entry or pop_scope after a block.

Common Pitfalls

  1. Scope leak from missing push/pop. If the type checker does not push a scope in check_func, parameters leak into the enclosing scope and can shadow globals. Every block (function body, if/while body, nested braces) must create a new scope.
  2. Mutable borrow conflicts. lookup borrows &self but insert borrows &mut self. You cannot interleave them in the same iteration. The design separates reads and writes cleanly: collect phase inserts everything, check phase only reads.
  3. Using insert where insert_global is needed. If the type checker is inside a function body and calls insert on a top-level declaration, it goes into the function's scope instead of the root. Use insert_global for anything that should be visible everywhere.
Alternatives not taken. Some compilers use a persistent tree of scopes with explicit ScopeId navigation, enabling import resolution across files. L1S's flat vector approach works well for single-file compilation. Multi-file compilation would need a more sophisticated module system.

9   Type Checking

Concept: Ensuring Programs Are Type-Safe

Type checking is the phase where the compiler verifies that the program respects the language's type rules. Every expression must have a known, consistent type. Operations must be applied to compatible operands. Function calls must match their declarations. A type error is not a "maybe it works" situation — it's a guarantee of undefined behavior at runtime.

The core questions a type checker answers:

  • What type does this expression produce?
  • Are the operands of + numeric?
  • Does the condition of if have type bool?
  • Do the arguments to this function match its parameter types?
  • Does the return value match the function's declared return type?

Tradeoffs & Design Space

Single-pass vs. multi-pass. A single-pass type checker processes declarations in order. This is simple but cannot handle forward references — a function called before it's defined will fail. Multi-pass solves this: the first pass collects all signatures, and later passes check bodies. L1S uses the multi-pass approach, which is also how C, C++, and Rust work (forward declarations or whole-module analysis).

Structural vs. nominal typing. Structural typing (TypeScript, Go) considers two types equal if they have the same shape. Nominal typing (C, Java, Rust) considers types equal only if they have the same name. L1S uses structural equality through its interned type arena: two i32 types are the same TypeId regardless of how they were written. Type aliases are transparently resolved.

Explicit annotations vs. inference. Requiring type annotations everywhere (like Java before local-variable type inference) is verbose but simple. Full Hindley-Milner inference (like Haskell, OCaml) is powerful but complex. L1S requires explicit type annotations on all variable declarations, which keeps the checker straightforward.

Annotation map. The type checker produces a map from Span (source location) to TypeId. This annotation map is the interface between type checking and later compiler phases: the IR builder consults it to know what type every expression has, without re-analyzing the entire program.

Ignis Implementation: Four-Phase Type Checking

Ignis's TypeChecker struct orchestrates four sequential phases:

Phase 1 (Collect): Register all declarations in the symbol table. Functions get their full Func type. Structs and enums get placeholder TypeIds so they can be referenced before their bodies are checked.
Phase 2 (Struct/Enum): Compute field offsets, sizes, alignment for structs. Compute variant values for enums. Register enum variants as global symbols.
Phase 3 (Functions): Type-check every function body. Push/pop scopes for parameters and blocks. Validate every expression and statement.
Phase 4 (Impls): Verify that each impl block satisfies the signatures declared in its corresponding spec.

The driver method check_program iterates the cloned declarations list four times:

src/typeck.rs — check_program orchestrator
pub fn check_program(&mut self, prog: &mut AstProgram) {
    let decls = prog.decls.clone();
    for decl in &decls { self.collect_decl(decl); }         // Phase 1
    for decl in &decls {
        match decl {
            AstDecl::Struct(s) => self.check_struct(s),
            AstDecl::Enum(e) => self.check_enum(e),         // Phase 2
            _ => {}
        }
    }
    for decl in &decls {                                  // Phase 3
        if let AstDecl::Func(f) = decl { self.check_func(f); }
    }
    for decl in &decls {                                  // Phase 4
        if let AstDecl::Impl(imp) = decl { self.check_impl(imp); }
    }
}

The clone is necessary because the checker mutates the type arena and symbol table, and Rust's borrow checker would not allow iterating &prog.decls while holding &mut self.

Phase 1: Collection with Placeholders

collect_decl registers every declaration in the symbol table. For structs and enums, it creates a placeholder — a TypeId with empty fields that will be filled in during Phase 2:

AstDecl::Struct(s) => {
    let tid = self.types.add_struct_placeholder();
    let _ = self.symbols.insert_global(&s.name, SymbolKind::TypeDecl(tid), s.span, s.pub_);
}

Why placeholders? Without them, a struct like Node with a field @Node (pointer to itself) would fail: Node would need to be fully defined before it could be referenced. The placeholder creates the TypeId early, allowing any declaration to reference it. This is the same trick C uses with forward struct declarations.

Phase 2: Struct Layout with Alignment

check_struct computes the byte offset of every field, respecting alignment constraints. The padding formula (offset + align - 1) / align * align rounds offset up to the next multiple of align:

src/typeck.rs — struct field layout with padding
let mut offset: u64 = 0;
let mut max_align: u64 = 1;
for f in &s.fields {
    let ft = self.ast_type_to_id(&f.type_);
    let align = self.types.align_of(ft);
    let padded = (offset + align - 1) / align * align;
    offset = padded;
    fields.push(FieldInfo { name: f.name.clone(), type_: ft, offset });
    offset += self.types.size_of(ft);
    if align > max_align { max_align = align; }
}
let final_size = (offset + max_align - 1) / max_align * max_align;

This matches C's struct layout: fields are aligned to their natural alignment, and the struct's total size is rounded up to the alignment of its most-aligned field. An i32 (alignment 4) followed by an i8 (alignment 1) produces a struct of size 8, with 3 bytes of trailing padding.

Phase 3: Function Body Checking

Function checking creates a new scope, inserts parameters as variables, checks the body, and restores the enclosing scope:

fn check_func(&mut self, f: &AstFuncDecl) {
    self.current_func_return = Some(self.ast_type_to_id(&f.return_type));
    self.symbols.push_scope();
    for p in &f.params {
        let pt = self.ast_type_to_id(&p.type_);
        let _ = self.symbols.insert(&p.name, SymbolKind::Variable(pt), p.span, false);
    }
    self.check_block(&f.body);
    self.symbols.pop_scope();
    self.current_func_return = None;
}

The current_func_return field is a stack (well, just a single Option, since L1S doesn't support nested functions) that the return statement checker consults. Every return expression is checked against this expected type.

Expression checking is recursive. The outer check_expr calls check_expr_inner and records the result:

fn check_expr(&mut self, expr: &AstExpr) -> TypeId {
    let t = self.check_expr_inner(expr);
    self.annotations.insert(expr.span(), t);  // save for IR builder
    t
}

The annotations map is critical: the IR builder needs to know the type of every expression, and re-checking would be wasteful and complex. By recording the type at the expression's source span, the IR builder has a cheap lookup table.

Statement Checking in Detail

The check_stmt method dispatches on each statement variant. Variable declarations resolve the type (or error if omitted), check the initializer if present, and insert the variable into the current scope:

AstStmt::VarDecl(v) => {
    let t = if let Some(ref type_) = v.type_ {
        self.ast_type_to_id(type_)
    } else {
        self.diag.error("type inference not yet supported", Some(v.span));
        self.types.intern(TypeNode::Void)
    };
    if let Some(ref init) = v.init {
        let init_type = self.check_expr(init);
        if !self.types_equal(init_type, t) {
            self.diag.error(
                format!("type mismatch: expected {}, got {}",
                    self.type_str(t), self.type_str(init_type)),
                Some(v.span));
        }
    }
    let _ = self.symbols.insert(&v.name, SymbolKind::Variable(t), v.span, false);
}

Control flow statements (if, while, for) check that their conditions are boolean. Return statements check against current_func_return. Block statements push/pop scopes, ensuring local variables don't leak.

Phase 4: Impl Checking

The final phase validates impl blocks against their corresponding spec declarations. For each method in the spec, the checker verifies that the impl provides an implementation with a matching signature. This is conceptually similar to Rust's trait impl checking, though L1S's spec system is simpler.

Type Resolution Helpers

Several helper methods support the checker:

  • ast_type_to_id: Converts an AstType (the parser's tree-based type representation) into an interned TypeId. Named types are looked up in the symbol table and must be SymbolKind::TypeDecl.
  • types_equal: Compares two types for equality, resolving type aliases transparently via resolve_type. This means typedef MyInt = i32 makes MyInt and i32 interchangeable.
  • is_numeric: Returns true for any integer type (including char). Used by arithmetic operators to reject non-numeric operands.
  • type_str: Produces a human-readable string for error messages. For example, @i32 for a pointer to i32, Func<i32, i32 -> void> for a function type.

Common Pitfalls

  1. Missing struct/enum placeholders. If you skip the placeholder allocation, circular references (e.g., struct A has a field of type B which has a field of type A) will fail because the TypeId doesn't exist yet.

10   Intermediate Representation

Concept: Decoupling Semantics from Machine Details

A compiler could translate the AST directly into assembly instructions. But this conflates two different concerns: what the program computes (semantics) and how the machine executes it (code generation). An intermediate representation (IR) sits between them, providing a machine-independent description of the program that is easier to analyze and transform than AST or assembly.

The key benefits of an IR layer:

  1. Machine independence. The same IR can target x86-64, ARM, RISC-V, or any backend without changing the IR builder. Only the code generator needs to change.
  2. Optimization. Transformations like constant folding, dead code elimination, and register allocation operate on a flat, uniform instruction sequence — much simpler than walking a tree.
  3. Analysis. Data-flow and control-flow analyses are straightforward on an IR with explicit basic blocks and virtual registers.
  4. Decoupling. The frontend (lexer, parser, type checker) and backend (codegen, register allocator) can be developed and tested independently.

Tradeoffs & Design Space

Tree IR vs. flat IR. A tree IR preserves the AST's hierarchical structure (e.g., C's generic IR trees in GCC). A flat IR linearizes everything into a sequence of instructions grouped into basic blocks. Flat IRs are more natural for optimization and code generation because they expose instruction ordering and data dependencies explicitly. L1S uses a flat, basic-block-based IR similar to LLVM IR.

SSA vs. non-SSA. Static Single Assignment (SSA) form requires that every variable is assigned exactly once. When control flow merges (e.g., after an if/else), phi nodes select the correct value based on which predecessor block executed. SSA simplifies many optimizations (constant propagation, dead code elimination) at the cost of phi-node insertion. L1S uses SSA form with explicit IrPhi nodes.

Register machine vs. stack machine. A register-machine IR (like LLVM or L1S) uses an unlimited set of virtual registers. A stack-machine IR (like JVM bytecode) uses a stack for operands. Register-machine IRs map more directly to physical CPUs and make data-flow analysis straightforward.

Instruction count. The number of IR instructions is a design choice: too few and you can't express the language efficiently; too many and the compiler becomes unwieldy. L1S has 36 instruction variants — a minimal set that covers all language features while mapping cleanly to x86-64 instructions. LLVM, by contrast, has hundreds.

IrInstr: Five Categories, 36 Variants

Every instruction (except Store and MemCopy) produces a result in a destination virtual register. This is the key to SSA form: each vreg is written exactly once.

src/ir.rs — full IrInstr enum by category
pub enum IrInstr {
    // --- Memory ---
    Alloc(VReg, TypeId),       // Reserve stack slot of given type
    Load(VReg, VReg),          // Load value from pointer
    Store(VReg, VReg),         // Store value to pointer (no dest)
    MovImm(VReg, u64, TypeId), // Load integer constant

    // --- Arithmetic (3-address) ---
    Add | Sub | Mul | SDiv | UDiv | SRem | URem,
    And | Or | Xor | Shl | Shr | Ushr,

    // --- Unary ---
    Neg(VReg, VReg),           // Arithmetic negation
    Not(VReg, VReg),           // Logical not (bool)
    BitNot(VReg, VReg),        // Bitwise not

    // --- Comparison (all produce bool) ---
    Eq | Ne |           // Equality / inequality
    Slt | Sgt | Sle | Sge |  // Signed comparisons
    Ult | Ugt | Ule | Uge,     // Unsigned comparisons

    // --- Conversion ---
    Zext(VReg, VReg, TypeId),  // Zero-extend
    Sext(VReg, VReg, TypeId),  // Sign-extend
    Trunc(VReg, VReg, TypeId), // Truncate
    PtrToInt(VReg, VReg, TypeId),
    IntToPtr(VReg, VReg, TypeId),

    // --- Aggregate / Call ---
    Gep(VReg, VReg, VReg, u64), // base[index * stride]
    MemCopy(VReg, VReg, u64),  // memcpy(dst, src, size) (no dest)
    Call(VReg, String, Vec<VReg>), // dst = func(arg0, arg1, ...)
}

The signed/unsigned split for division, remainder, and comparison is necessary because x86-64 has different instructions for signed and unsigned operations. The type checker preserves signedness from the source types, and the IR builder emits the correct variant.

Terminators: How Blocks End

Every block has exactly one terminator. Three options cover all control flow:

src/ir.rs — terminators
pub enum IrTerminator {
    Jump(IrBlockId),                    // Unconditional jump
    Branch(VReg, IrBlockId, IrBlockId), // if (cond) true else false
    Return(VReg),                        // Return value from function
}
  • Jump: Used for unconditional control flow (e.g., after an if-then block, skip the else block).
  • Branch: Conditional branch. If the condition vreg is non-zero (true), go to the first block; otherwise go to the second. Used for if, while, for.
  • Return: Exit the function with the given vreg's value. For void functions, a dummy vreg holding zero is used.

SSA Phi Nodes

When different control-flow paths assign different values to the same variable, a phi node selects the right one:

pub struct IrPhi {
    pub dest: VReg,
    pub type_: TypeId,
    pub sources: Vec<(VReg, IrBlockId)>,
}

Semantically: %3 = phi(bb0: %1, bb1: %2) means "if control arrived from bb0, %3 = %1; if from bb1, %3 = %2." The code generator implements this by ensuring both predecessor values end up in the same physical location.

Phi nodes appear at the start of a block, before any instructions. They are not executable operations — they are a data-flow annotation that tells the optimizer and codegen how values merge at join points.

The dest() Method

Most instructions write to a destination vreg. The dest() method provides a uniform query interface, used extensively by optimization passes and register allocation:

pub fn dest(&self) -> VReg {
    match self {
        IrInstr::Alloc(d, _)
        | IrInstr::Load(d, _)
        | IrInstr::Add(d, _, _) | IrInstr::Sub(d, _, _)
        | /* ... every variant except Store/MemCopy */ => *d,
        IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {
            panic!("Store/MemCopy has no dest")
        }
    }
}

Example: IR for a Simple Expression

Let's trace what the IR looks like for this function:

func main(i32) { return 1 + 2; }

The IR builder produces:

function main(i32) {
// entry: bb0
bb0 (entry):
%0 = Alloc i32 // allocate return slot
%1 = MovImm 1 : i32 // constant 1
%2 = MovImm 2 : i32 // constant 2
%3 = Add %1, %2 // %3 = 1 + 2
Return %3 // return value 3
}

Every detail is explicit:

  • Alloc reserves stack space for the return value (needed for the calling convention)
  • MovImm loads each constant into a virtual register
  • Add performs the addition in 3-address form: %3 = %1 + %2
  • Return exits with %3's value

An optimizer could constant-fold this to a single MovImm %3, 3 : i32 followed by Return %3, eliminating %1 and %2 entirely.

For a conditional program like:

func max(i32, a: i32, b: i32) {
    if (a > b) {
        return a;
    } else {
        return b;
    }
}

The IR would have three blocks (entry, then, else) with a Branch terminator at the end of entry and Return terminators in then and else:

function max(i32, %1: a, %2: b) {
bb0 (entry):
%3 = Sgt %1, %2 // %3 = a > b
Branch %3, bb1, bb2 // if %3 goto bb1 else bb2

bb1 (then):
Return %1

bb2 (else):
Return %2
}

Common Pitfalls

  1. Store/MemCopy have no destination. These are the only instructions without a destination vreg. Passes that iterate all definitions must explicitly skip them, or the dest() method will panic.
  2. Signed vs. unsigned comparisons. Using Slt (signed less-than) for unsigned values gives wrong results. For example, 0xFFFF_FFFF as u32 is 4294967295, but as i32 it's -1. The type checker must preserve signedness through to the IR builder.
  3. Missing phi nodes at merge points. If two branches assign different values to the same variable, a phi node is required where they rejoin. Without it, the variable's value is ambiguous and optimization passes will produce incorrect results.
  4. VReg numbering. The num_vregs field must be strictly greater than the largest vreg index. An off-by-one here causes register allocation to miss the last register.
Why basic blocks? A basic block is a maximal sequence of instructions with a single entry (at the top) and single exit (at the bottom). This property makes data-flow analysis tractable: within a block, all definitions reach all uses. Across blocks, you need data-flow equations or phi nodes. Almost every modern compiler uses basic blocks as the fundamental unit of IR.

11   Lowering to IR

What is Lowering?

Lowering is the translation from a high-level representation (the AST) to a low-level representation (the IR). The AST is a tree that mirrors the grammar of the source language: expressions nest inside expressions, statements contain sub-statements, and control flow is implicit in the nesting. The IR is a flat, explicit instruction stream organized into basic blocks. Every expression becomes a sequence of IR instructions operating on virtual registers. Control flow constructs like if, while, and for get translated into basic blocks with explicit jumps and branches.

This is where the compiler stops thinking like a parser and starts thinking like a machine.

AST (tree) → IrBuilder → IR (flat, blocks + vregs)

The key transformation: a deeply nested AST expression like a + b * 3 becomes a linear sequence of IR instructions, each computing one operation and writing its result to a fresh virtual register:

; AST: a + b * 3
; Lowered IR:
%0 = Mul %b, %3       ; b * 3
%1 = Add %a, %0       ; a + (b * 3)

Virtual Registers: Unlimited Naming

A virtual register (vreg) is a name for a value that doesn't correspond to any physical CPU register. The IR can create as many vregs as it needs. Each instruction produces a value in a new vreg. The later register allocation phase maps these unlimited vregs onto the 16 real x86-64 registers, spilling to the stack when necessary.

In Ignis, a vreg is a small wrapper around a u32 index:

struct VReg(u32);

Displayed as %0, %1, etc. Each IrFunction tracks num_vregs — a running counter incremented every time alloc_vreg_in is called. This gives every value in the function a unique name.

Basic Blocks and Terminators

A basic block is a straight-line sequence of instructions with no branches in and no branches out (except at the end). Every block ends with a terminator — one of:

  • Jump: unconditional branch to another block.
  • Branch: conditional branch to one of two blocks based on a vreg.
  • Return: exit the function, returning a vreg value.

This design means control flow is explicit. Instead of an AST if node that wraps its branches as children, the IR has:

bb0 (entry):
  %0 = ...              ; compute condition
  Branch %0, bb1, bb2   ; if %0, go to bb1, else bb2

bb1 (if.then):
  ...
  Jump bb3              ; merge

bb2 (if.else):
  ...
  Jump bb3

bb3 (if.end):
  ...

Design Space: SSA vs. Linear IR

There are two major families of IR:

  • SSA (Static Single Assignment): Every vreg is assigned exactly once. Phi nodes merge values at control-flow joins. SSA enables powerful optimizations (GVN, sparse conditional constant propagation) because def-use chains are implicit. The cost: phi nodes complicate codegen, and construction requires inserting phi placements.
  • Linear IR (non-SSA): Vregs can be assigned multiple times. No phi nodes. Simpler to build and emit code for. The cost: optimizations require explicit dataflow analysis.

Ignis uses a linear non-SSA IR. Vregs are allocated sequentially (a new one per instruction destination), but they can also be reused by Store/Load pairs to model mutable variables. This is the simplest IR that can still express all programs and be lowered to x86-64.

Why not SSA? SSA would require phi-node insertion and SSA destruction during codegen. For a teaching compiler, the extra complexity is not worth the optimization benefits. Ignis keeps the IR simple and pushes complexity into optional optimization passes.

Ignis: The IrBuilder

Ignis's IrBuilder (in src/ir_build.rs) walks the AST and emits IR instructions into basic blocks. It is the bridge between the type-checked AST and the flat IR module.

Structure

pub struct IrBuilder<'a> {
    module: IrModule,            // the IR module being built
    types: &TypeArena,           // type information from typeck
    annotations: &HashMap<Span, TypeId>,  // type annotations on expressions
    current_func: Option<IrFunction>,
    current_block: Option<IrBlockId>, // where to emit next instruction
    next_vreg: u32,               // counter for fresh vregs
    next_block: u32,
    vars: Vec<HashMap<String, VReg>>,  // lexical scope chain
}

The current_block field is critical: it tracks which block will receive the next emitted instruction. When control flow forks (e.g., an if statement), the builder switches current_block to the new block, emits instructions there, then sets its terminator.

The Entry Point: build()

pub fn build(mut self, prog: &AstProgram) -> IrModule {
    for decl in &prog.decls {
        match decl {
            AstDecl::Func(f) => self.build_func(f),
            AstDecl::StaticVar(sv) => self.build_global(sv),
            _ => {}
        }
    }
    self.module
}

It iterates over declarations, calling build_func for functions and build_global for globals. Struct and enum declarations are type-level only and produce no IR.

Lowering Variables: Alloc + Store + Load

Variables are lowered using a three-instruction pattern that models memory:

  • Alloc %v, T: Reserve a stack slot for variable v of type T.
  • Store %src, %dst: Write a value to a variable's slot.
  • Load %dst, %src: Read a value from a variable's slot.

Every variable declaration is lowered to an Alloc for its slot, then an optional Store for the initializer. Every variable reference becomes a Load from the variable's slot. This mirrors how real machines work: local variables live in stack memory, and reading/writing them requires explicit memory operations.

// Source: let x: i32 = 42;
%0 = Alloc i32        ; reserve slot for x
%1 = MovImm 42        ; materialize constant
Store %1, %0          ; x = 42

// Source: x + 1
%2 = Load %0          ; read x
%3 = MovImm 1
%4 = Add %2, %3       ; x + 1

Lowering Control Flow: If

An if statement introduces three or four blocks:

fn build_if(...) {
    let cond = self.build_expr_in(func, &i.cond);
    let then_bb = self.new_block_in(func, "if.then");
    let merge_bb = self.new_block_in(func, "if.end");
    let else_bb = if has_else {
        self.new_block_in(func, "if.else")
    } else { merge_bb };

    self.set_term_in(func, IrTerminator::Branch(cond, then_bb, else_target));

    self.current_block = Some(then_bb);
    self.build_block_in(func, &i.then_block);
    if self.current_block.is_some() {
        self.set_term_in(func, IrTerminator::Jump(merge_bb));
    }

    if let Some(ref else_branch) = i.else_branch { ... }

    self.current_block = Some(merge_bb);
}

Key detail: after building the then body, we check self.current_block.is_some(). If the then-block ended with a return statement, current_block would have been set to None (since no instructions can follow a return). In that case, we skip the Jump to merge.

Lowering Control Flow: While

A while loop uses three blocks in a cycle:

bb0:                    ; before loop
  Jump bb1              ; jump to header

bb1 (while.header):     ; check condition
  %0 = ...              ; compute condition
  Branch %0, bb2, bb3   ; if true, enter body; else exit

bb2 (while.body):
  ...
  Jump bb1              ; back to header

bb3 (while.exit):

The header block is the key innovation: it both evaluates the condition and serves as the target for the backward jump at the end of the body. This avoids duplicating the condition evaluation.

A for loop expands to five blocks — the most complex control flow pattern:

bb0 (for.init):     ; initializer expression
  ...
  Jump bb1

bb1 (for.header):   ; condition check

bb2 (for.body):     ; loop body

  ...
  Jump bb1          ; back to header

bb4 (for.exit):

Lowering Expressions

Each expression type maps to one or more IR instructions. The pattern is always: build sub-expressions (which produce vregs), allocate a new vreg for the result, emit an instruction that computes into it, and return the vreg.

Integer literals become MovImm:

%0 = MovImm 42

Binary operators build both sides, then emit the op:

; a + b
%0 = Load %a_slot
%1 = Load %b_slot
%2 = Add %0, %1

Function calls emit a Call instruction with the callee name and arguments:

%0 = Call "foo", [%arg1, %arg2]

Array indexing uses Gep (GetElementPointer) to compute the element address, then Load to read:

; arr[idx]
%0 = Gep %arr_ptr, %idx, 4    ; elem_size=4
%1 = Load %0

Scope Chains and Variable Lookup

Ignis models lexical scoping with a stack of HashMap<String, VReg>:

vars: Vec<HashMap<String, VReg>>

When entering a block, a new scope is pushed. Variable declarations insert into the innermost scope. Variable references walk the stack from top to bottom, finding the nearest enclosing definition. When leaving a block, the scope is popped.

fn lookup_var(&self, name: &str) -> VReg {
    for scope in self.vars.iter().rev() {
        if let Some(&vreg) = scope.get(name) {
            return vreg;
        }
    }
    VReg::new(self.next_vreg)  ; default if not found
}

The default fallback (returning a fresh vreg) is a simplification: in practice, an undeclared variable would have been caught by type checking.

What IR Building Does Not Do

The builder is intentionally simple. It does not:

  • Optimize: It emits naive code. Every variable access becomes a Load/Store pair. Every temporary gets its own vreg. Optimization is a separate pass.
  • Allocate registers: Vregs are purely symbolic at this stage.
  • Check types: Type checking has already run. The builder trusts that the AST is well-typed and just uses the annotations to determine sizes.
  • Emit machine code: That's the codegen phase. The IR is still architecture-independent.

12   IR Optimization

What is IR Optimization?

IR optimization transforms the intermediate representation to make it faster, smaller, or both — without changing the program's meaning. The IR builder emits naive code: every operation gets its own vreg, every variable access goes through memory, and unused computations accumulate. Optimization cleans this up.

The key principle: each optimization pass must be sound (it must never change the program's behavior) and idempotent (repeated application converges). Passes are typically organized into a pipeline that runs to a fixed point — keep applying passes until no more changes occur.

Naive IR → Constant Folding → Dead Code Elim → Copy Propagation → Remove Unreachable → Optimized IR

Pass Ordering and Fixed-Point Iteration

Optimizations interact: one pass may expose opportunities for another. Constant folding creates constants that make other expressions foldable. Dead code elimination may make blocks unreachable. Copy propagation may create dead stores.

The standard solution is fixed-point iteration: run all passes in a loop until none of them changes anything. A maximum iteration count prevents infinite loops from buggy passes.

Ignis does exactly this:

pub fn optimize(module: &mut IrModule) {
    for _ in 0..100 {
        let changed = constant_folding(module)
            | dead_code_elimination(module)
            | copy_propagation(module);
        if !changed { break; }
    }
    remove_unreachable_blocks(module);
}

Three passes run in a fixed-point loop (max 100 iterations). Unreachable block removal runs once at the end — it's a cleanup pass that doesn't create new optimization opportunities.

Design Space: Optimization Strategies

IR optimization ranges from simple local peephole passes to sophisticated global analysis:

  • Peephole optimization: Match small patterns in a sliding window of instructions (e.g., Add %x, 0 → %x). Fast but limited.
  • Local optimization: Works within a single basic block. Dead code elimination within a block is local.
  • Global optimization: Works across the entire function. Constant propagation across blocks, global value numbering, loop invariant code motion.
  • Interprocedural optimization: Works across function calls. Inlining, function specialization. Beyond the scope of a teaching compiler.

Ignis uses mostly local and global passes with simple dataflow. Each pass is under 60 lines of code, making them easy to understand and verify.

Pass 1: Constant Folding

Constant folding evaluates arithmetic operations whose operands are known constants at compile time. Instead of emitting Add %0, 2, 3, the pass replaces it with MovImm %0, 5.

fn constant_folding(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut const_values: HashMap<VReg, u64> = HashMap::new();

        for block in &mut func.blocks {
            let mut replacements: Vec<(usize, IrInstr)> = Vec::new();

            for (i, instr) in block.instrs.iter().enumerate() {
                match instr {
                    IrInstr::MovImm(d, v, _) => {
                        const_values.insert(*d, *v);
                    }
                    IrInstr::Add(d, a, b) => {
                        if let (Some(av), Some(bv)) =
                            (const_values.get(a), const_values.get(b))
                        {
                            let val = av.wrapping_add(*bv);
                            replacements.push((i, IrInstr::MovImm(*d, val, ...)));
                            changed = true;
                        }
                    }
                    IrInstr::Sub(d, a, b) => { /* same pattern */ }
                    IrInstr::Mul(d, a, b) => { /* same pattern */ }
                    _ => {}
                }
            }

            for (i, replacement) in replacements {
                block.instrs[i] = replacement;
            }
        }
    }
    changed
}

The pass maintains a HashMap<VReg, u64> tracking which vregs hold known constants. When it encounters a binary op where both operands are in the map, it computes the result and replaces the instruction with a MovImm.

Before:

%0 = MovImm 2
%1 = MovImm 3
%2 = Add %0, %1

After:

%2 = MovImm 5

The %0 = MovImm 2 and %1 = MovImm 3 now become dead code — the next pass will remove them.

Wrapping arithmetic: Ignis uses wrapping_add/wrapping_sub/wrapping_mul. This is correct for a compiler: integer overflow is defined to wrap in the IR, matching typical hardware behavior. A higher-level language might define overflow as undefined behavior, but that's a language semantics choice, not an IR concern.

Pass 2: Dead Code Elimination

Dead code elimination (DCE) removes instructions whose results are never used. An instruction is "dead" if its destination vreg is never referenced by any other instruction, phi node, or terminator.

The algorithm is straightforward:

  1. Collect uses: Scan all instructions, phi nodes, and terminators. Record every vreg that appears as an operand (not a destination).
  2. Filter: Keep only instructions that have side effects (Store, Call, MemCopy) or whose destination appears in the use set.
fn dead_code_elimination(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut uses: HashSet<VReg> = HashSet::new();

        // Phase 1: collect all used vregs
        for block in &func.blocks {
            for instr in &block.instrs {
                for op in instr_uses(instr) { uses.insert(op); }
            }
            for phi in &block.phis {
                for (src, _) in &phi.sources { uses.insert(*src); }
            }
            match &block.terminator {
                IrTerminator::Branch(cond, _, _) => { uses.insert(*cond); }
                IrTerminator::Return(val) => { uses.insert(*val); }
                _ => {}
            }
        }

        // Phase 2: retain only useful instructions
        for block in &mut func.blocks {
            block.instrs.retain(|instr| {
                match instr {
                    IrInstr::Store(_, _) | IrInstr::Call(_, _, _) |
                    IrInstr::MemCopy(_, _, _) => true,  ; side effects
                    _ => uses.contains(&instr.dest()),
                }
            });
        }
    }
    changed
}

Before:

%0 = MovImm 2        ; %0 never used
%1 = MovImm 3
%2 = Add %1, %3      ; %2 never used
%3 = Store %4, %5    ; side effect, must keep
Return %5

After:

%3 = Store %4, %5    ; kept for side effect
Return %5

Pass 3: Copy Propagation

Copy propagation replaces uses of one vreg with another when a Load instruction simply copies a value from a source vreg. If the source vreg was itself a copy from an earlier load, the chain is followed.

Ignis's copy propagation is simple: it maps each Load %d, %s as a copy from %s to %d. Subsequent uses of %d can be replaced with %s (or further back if %s was also a copy).

fn copy_propagation(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut copies: HashMap<VReg, VReg> = HashMap::new();

        for block in &func.blocks {
            for instr in &block.instrs {
                if let IrInstr::Load(d, s) = instr {
                    if let Some(&orig) = copies.get(s) {
                        copies.insert(*d, orig);
                    } else {
                        copies.insert(*d, *s);
                    }
                    changed = true;
                }
            }
        }
    }
    changed
}

Note: Ignis's copy propagation is incomplete — it tracks copy chains but does not actually rewrite uses of the propagated vreg. A more complete implementation would walk all instructions and replace operand vregs with their propagated values. This is a known simplification.

Pass 4: Remove Unreachable Blocks

After optimizations, some basic blocks may become unreachable from the entry block. These blocks waste space and may contain dead code that DCE missed. The fix is a simple graph traversal:

fn remove_unreachable_blocks(module: &mut IrModule) {
    for func in &mut module.functions {
        let entry = func.entry_block;
        let mut reachable: HashSet<IrBlockId> = HashSet::new();
        let mut worklist: Vec<IrBlockId> = vec![entry];

        while let Some(block_id) = worklist.pop() {
            if reachable.insert(block_id) {
                let successors = block_successors(&func.blocks[block_id.index()]);
                worklist.extend(successors);
            }
        }

        func.blocks.retain(|b| reachable.contains(&b.id));
    }
}

The helper block_successors extracts the target block(s) from a block's terminator:

fn block_successors(block: &IrBlock) -> Vec<IrBlockId> {
    match &block.terminator {
        IrTerminator::Jump(target) => vec![*target],
        IrTerminator::Branch(_, t, f) => vec![*t, *f],
        IrTerminator::Return(_) => vec![],
    }
}

After removal, phi nodes that reference removed predecessor blocks are cleaned up, and terminators that jump to removed blocks are redirected to the block itself (a self-loop).

Optimization Debugging Checklist

  1. Did constant folding miss anything? It only handles Add, Sub, Mul. Division, comparisons, and shifts are not folded.
  2. Are side effects properly preserved? DCE never removes Store, Call, or MemCopy even if their destination is unused.
  3. Did copy propagation actually rewrite uses? Ignis's pass tracks copies but doesn't rewrite operands — this is a known gap.
  4. Are self-loop terminators safe? When a Jump target becomes unreachable, it's redirected to the block itself. This could create an infinite loop in unoptimized code.

13   Register Allocation

The Register Allocation Problem

Register allocation is the problem of mapping an unlimited number of virtual registers to a fixed set of physical CPU registers. The IR builder freely creates as many vregs as it needs — a single function might use hundreds. But x86-64 has only 16 general-purpose registers (and many are reserved for specific purposes: stack pointer, frame pointer, etc.).

When there aren't enough physical registers to hold all live values simultaneously, some values must be spilled to the stack. Spilling means storing a value to memory when the register is needed for something else, then loading it back when it's needed again. Every spill adds memory traffic, so good register allocation minimizes spills.

Unlimited vregs
%0, %1, %2, ... %N → Register
Allocator → 16 physical regs
rax, rbx, rcx, ...

↓ Spill slots on stack
[rbp-8], [rbp-16], ...

Design Space: Allocation Strategies

Register allocation is a well-studied NP-complete problem (it's equivalent to graph coloring). Real compilers use a variety of heuristics:

  • Graph coloring: Build an interference graph where edges connect vregs that are simultaneously live. Color the graph with K colors (K = number of registers). If coloring fails, spill a node and retry. Used by GCC, LLVM (-O2). Optimal but complex.
  • Linear scan: Sort live ranges by their start point. Scan forward, assigning registers. When a register is needed and none is free, spill the vreg whose next use is farthest. Used by LuaJIT, many JIT compilers. Fast, good results.
  • Spill everything: Every vreg gets its own stack slot. No interference analysis, no coloring, no scanning. Trivially correct, O(1) per vreg. Maximum memory traffic, but the codegen is straightforward: every instruction reads operands from the stack and writes results to the stack.
Tradeoff: Graph coloring produces the best code but is the hardest to implement and debug. Linear scan is a good middle ground. Spill-everything is the simplest correct allocator — it's what you write when you want a working compiler quickly and performance is secondary.

Ignis: The Spill-Everything StackAllocator

Ignis uses the spill-everything approach. Every vreg — whether it's a local variable, a temporary computation result, or a parameter — gets its own dedicated slot on the stack. The allocator never attempts to keep values in registers across instructions. Instead, it assigns each vreg a fixed offset from the stack base pointer (rbp).

The allocator is called StackAllocator (in src/regalloc.rs, just 92 lines):

pub struct StackAllocator {
    pub slot_map: HashMap<VReg, i64>,  ; vreg ⇒ stack offset
    pub stack_size: i64,                 ; total frame size (negative, growing down)
}

The algorithm is dead simple:

  1. Walk every instruction in the function.
  2. For every vreg that is defined (as a destination), allocate a stack slot.
  3. For every parameter vreg, allocate a stack slot.
  4. For every phi-node destination vreg, allocate a stack slot.
  5. Align the total stack size to 16 bytes (x86-64 ABI requirement).
pub fn allocate(&mut self, func: &IrFunction) {
    // Parameters get slots
    for param in &func.params {
        self.make_slot(param.vreg, param.type_);
    }

    // Instructions: Alloc explicitly requests a slot; other instrs get
    // a slot for their destination vreg
    for block in &func.blocks {
        for instr in &block.instrs {
            match instr {
                IrInstr::Alloc(vreg, type_) => {
                    self.make_slot(*vreg, *type_);
                }
                IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {}
                instr => {
                    let dest = instr.dest();
                    self.make_slot(dest, TypeId::new(11));
                }
            }
        }
        // Phi nodes get slots
        for phi in &block.phis {
            self.make_slot(phi.dest, phi.type_);
        }
    }

    // 16-byte alignment
    self.stack_size = self.stack_size & !15;
}

Slot allocation is just a counter decremented by 8 bytes each time:

fn make_slot(&mut self, vreg: VReg, _type_: TypeId) {
    if !self.slot_map.contains_key(&vreg) {
        self.stack_size -= 8;       ; grow stack downward
        self.slot_map.insert(vreg, self.stack_size);
    }
}

Every slot is 8 bytes regardless of the actual type. This wastes space for i8 or bool values (which need only 1 byte) but keeps the layout simple and aligned.

Stack Frame Layout

After allocation, each vreg maps to a fixed offset from rbp. The stack frame grows downward (toward lower addresses):

        Higher addresses
        +------------------+
rbp ⇒ | saved rbp        |  (pushed by prologue)
        +------------------+
rbp-8   | slot for %0      |  (first allocated)
        +------------------+
rbp-16  | slot for %1      |  (second allocated)
        +------------------+
rbp-24  | slot for %2      |
        +------------------+
        | ...              |
rsp ⇒ +------------------+
        Lower addresses

The codegen phase uses the slot offset to emit memory operand addresses like [rbp-8], [rbp-16], etc.

Why Spill-Everything is (Sometimes) OK

Despite its obvious inefficiency, the spill-everything strategy has real advantages:

  1. Correct by construction: No interference analysis, no liveness tracking, no spilling decisions. Every vreg always has a home. The allocator cannot produce wrong code.
  2. Deterministic: The allocation depends only on the order vregs are encountered, not on heuristics or graph structure. The same IR always produces the same stack layout.
  3. Zero register pressure management: Since every value lives on the stack, there's never a moment where a register is needed but none is free. The codegen never needs to think about spilling during instruction selection.
  4. Simple codegen: Every instruction becomes a "load from stack, operate, store to stack" pattern. The code generator is a straightforward translation without register management.

The cost is performance. Every arithmetic operation requires at least two memory operations (load operands, store result). On modern x86-64 CPUs with good caching, this is often not as bad as it sounds — the stack slots stay in L1 cache, so the memory traffic is fast. But it's still slower than keeping values in registers.

What's Missing: Proper Register Allocation

To move beyond spill-everything, a compiler needs:

  • Liveness analysis: For each vreg, determine the set of program points where its value is still needed. Two vregs whose live ranges don't overlap can share a register.
  • Register coalescing: When a Store is immediately followed by a Load of the same vreg, the store and load can be eliminated if the value stays in a register.
  • Graph coloring or linear scan: An algorithm that assigns physical registers to vregs, spilling only when necessary.

Ignis's StackAllocator is intentionally minimal. It is the simplest correct allocator and serves as a foundation that could be extended with liveness analysis and a proper coloring pass.

Improving the allocator: A straightforward upgrade path is to add linear scan. First add liveness analysis (compute live ranges for each vreg via dataflow). Then sort live ranges by start point and scan forward, assigning registers until you run out, then spill. This would eliminate most memory traffic while keeping the allocator under 200 lines.

Testing the Allocator

Testing a spill-everything allocator is straightforward: allocate a known IR function and verify that every vreg gets a slot and that the stack size is correctly computed:

#[test]
fn test_allocate_allocs() {
    let mut func = make_test_func();
    let mut alloc = StackAllocator::new();
    alloc.allocate(&func);

The key invariants to test: every vreg in the function has a slot, no two vregs share the same slot, and the total stack size is 16-byte aligned.

14   Code Generation

What is Code Generation?

Code generation is the phase where the compiler's intermediate representation is translated into actual machine instructions for a target architecture. This is where the compiler moves from abstract, architecture-neutral data structures to concrete, architecture-specific assembly code. Every decision made in earlier phases — parsing, type checking, optimization — converges here into sequences of bytes that a CPU can execute.

A code generator must solve three interrelated problems:

  1. Instruction selection: Mapping each IR operation to one or more target machine instructions. An IR Add might become a single x86-64 add instruction, while a division might require a sequence involving cdq and idiv.
  2. Register allocation: Deciding which virtual registers map to physical registers versus stack slots. The allocator from Chapter 13 feeds into this.
  3. Calling convention: Following the ABI (Application Binary Interface) rules for how functions pass parameters, return values, and maintain the stack frame.
Pipeline position:

IR Module → Codegen → .asm file → nasm → .o file → cc (linker) → executable

Approaches to Code Generation

There is a broad spectrum of code generation strategies, each with different tradeoffs between implementation effort, code quality, and maintainability:

  • Hand-written AST walkers: The simplest approach. Walk the IR and emit assembly directly for each construct. Easy to implement and debug, but producing optimal code requires enormous effort. This is what L1S, early C compilers (K&R C), and tcc (Tiny C Compiler) use.
  • Template-based codegen: Pre-defined assembly templates with holes for operands. The code generator matches IR patterns to templates. GCC used this approach with RTL (Register Transfer Language) descriptions.
  • Instruction-selection DAGs: Build a Directed Acyclic Graph of target-independent operations, then tile it with target-specific machine instructions via dynamic programming. LLVM's SelectionDAG is the most prominent example. This approach can produce excellent code but is complex to implement.
  • Table-driven codegen: Machine descriptions (like .td files in LLVM) drive automatic code generation. The compiler writer describes the instruction set declaratively, and tooling generates the instruction selector.
  • Backend as a library (LLVM): Instead of writing your own code generator, pass your IR to LLVM and let it handle instruction selection, register allocation, and optimization. This is the most pragmatic choice for production compilers (Rust, Swift, Clang all use LLVM).

L1S takes the simplest possible approach: hand-written code generation where each IrInstr variant maps to a fixed pattern of NASM instructions. The register allocator from Chapter 13 provides a stack slot for every virtual register, so the code generator loads operands from memory and stores results back to memory on every instruction — maximally simple, minimally performant.

Targeting x86-64 and NASM Syntax

L1S targets x86-64 Linux using the System V AMD64 ABI. The code generator emits NASM (Netwide Assembler) syntax, which is a common choice for x86-64 assembly on Linux. Key conventions of the target:

  • Integer arguments: Passed in rdi, rsi, rdx, rcx, r8, r9 (first through sixth). Additional arguments go on the stack.
  • Return value: Placed in rax.
  • Stack alignment: The stack must be 16-byte aligned at the point of a call instruction.
  • Callee-saved registers: rbx, rbp, r12–r15 must be preserved. L1S only uses rbp as the frame pointer.

How Ignis Generates Code

Ignis's Codegen struct is straightforward: it holds a mutable String buffer for the assembly output and a reference to the type arena. The entry point, generate(), produces three NASM sections:

src/codegen.rs — Codegen struct and generate()
pub struct Codegen<'a> {
    pub output: String,
    pub types: &TypeArena,
}

impl<'a> Codegen<'a> {
    pub fn generate(&mut self, module: &IrModule) -> String {
        self.output.clear();
        self.emit("section .text");
        self.emit("global main");
        for func in &module.functions { self.generate_func(func); }
        if !module.strings.is_empty() { /* section .rodata */ }
        if !module.globals.is_empty() { /* section .data */ }
        self.output.clone()
    }
}

The three NASM sections serve distinct purposes:

  • .text: Executable code. Every function becomes a label followed by its instructions.
  • .rodata: Read-only data. String literals emitted as db directives with unique labels (str_0, str_1, ...).
  • .data: Mutable data. Global variables emitted with dd 0 (zero-initialized 4-byte values).

Function Prologue and Epilogue

Every function begins with a prologue that sets up the stack frame and ends with an epilogue that tears it down. This is the standard x86-64 calling convention pattern:

src/codegen.rs — Function code generation
fn generate_func(&mut self, func: &IrFunction) {
    let mut alloc = StackAllocator::new();
    alloc.allocate(func);

    self.emit(&format!("global {}", func.name));
    self.emit(&format!("{}:", func.name));
    self.emit("  push rbp");
    self.emit("  mov rbp, rsp");
    if alloc.stack_size != 0 {
        self.emit(&format!("  sub rsp, {}", -alloc.stack_size));
    }
    // Spill parameter registers to stack slots
    for (i, param) in func.params.iter().enumerate() {
        let slot = alloc.slot(param.vreg);
        let reg = self.param_reg(i);
        if slot != 0 {
            self.emit(&format!("  mov [rbp{}], {}", self.signed(slot), reg));
        }
    }
    // Emit basic blocks
    for block in &func.blocks { /* ... */ }
    // Shared return label (epilogue)
    self.emit(&format!(".return_{}:", func.name));
    if alloc.stack_size != 0 { /* add rsp, N */ }
    self.emit("  pop rbp");
    self.emit("  ret");
}

The prologue sequence deserves careful explanation:

  • push rbp: Save the caller's frame pointer on the stack (8 bytes).
  • mov rbp, rsp: Set our own frame pointer to the current stack pointer. All local variables and parameters are now accessed as offsets from rbp.
  • sub rsp, N: Allocate N bytes on the stack for local variables. N is the (negated) stack size from the StackAllocator, rounded to 16-byte alignment.

The epilogue reverses these steps: add rsp, N deallocates the stack frame, pop rbp restores the caller's frame pointer, and ret returns control to the caller. All return paths converge to a single shared return label rather than inlining the epilogue at every return point — a code-size tradeoff.

Instruction Selection Patterns

The heart of the code generator is the emit_instr() method, which maps each IrInstr variant to NASM instructions. The patterns fall into several categories:

Binary Operations (Add, Sub, Mul, And, Or, Xor)

These share a uniform three-step pattern via the binary() helper:

fn binary(&mut self, op: &str, dest: &VReg, a: &VReg, b: &VReg, alloc: &StackAllocator) {
    self.load_to_rax(*a, alloc);       // rax = a (load from stack)
    self.load_to_reg(*b, "rcx", alloc);   // rcx = b (load from stack)
    self.emit(&format!("  {} eax, ecx", op));  // eax = eax op ecx
    self.store_from_rax(*dest, alloc);  // dest = rax (store to stack)
}

This is a three-address-code style: both operands are loaded into registers, the operation executes, and the result is stored back. The use of rax and rcx as temporary registers is arbitrary but consistent — any caller-saved register would work.

Division and Remainder

x86-64 division uses implicit register pairs, making it more complex:

IrInstr::SDiv(dest, a, b) => {
    self.load_to_rax(*a, alloc);
    self.emit("  cdq");              // sign-extend eax → edx:eax
    self.load_to_reg(*b, "rcx", alloc);
    self.emit("  idiv ecx");          // quotient in eax, remainder in edx
    self.store_from_rax(*dest, alloc);
}

The cdq instruction sign-extends eax into the edx:eax register pair, forming the 64-bit dividend. idiv ecx divides this 64-bit value by ecx. The quotient lands in eax and the remainder in edx. For remainder operations (SRem), the code generator reads from rdx instead of rax.

Comparison Instructions

Comparisons use a two-step pattern: set CPU flags with cmp, then capture a flag with a setcc instruction:

fn cmp(&mut self, setcc: &str, dest: &VReg, a: &VReg, b: &VReg, alloc: &StackAllocator) {
    self.load_to_rax(*a, alloc);
    self.load_to_reg(*b, "rcx", alloc);
    self.emit("  cmp eax, ecx");        // set flags based on a - b
    self.emit(&format!("  {} al", setcc));   // al = 0 or 1 based on flag
    self.emit("  movzx eax, al");      // zero-extend byte to dword
    self.store_from_rax(*dest, alloc);
}

The setcc mnemonic varies by comparison: sete for equality, setne for inequality, setl for signed less-than, setg for signed greater-than, etc. The movzx eax, al zero-extends the 8-bit result to 32 bits, since setcc only writes to the low byte of a register.

Function Calls

Function calls must respect the calling convention: arguments are loaded into the SysV argument registers, then call is emitted, and the return value in rax is stored:

IrInstr::Call(dest, name, args) => {
    for (i, arg) in args.iter().enumerate() {
        let reg = self.arg_reg(i);    // rdi, rsi, rdx, rcx, r8, r9
        self.emit(&format!("  mov {}, [rbp{}]", reg, self.signed(slot)));
    }
    self.emit(&format!("  call {}", name));
    let d_slot = alloc.slot(*dest);
    if d_slot != 0 { self.emit(&format!("  mov [rbp{}], rax", self.signed(d_slot))); }
}

Note that L1S currently only handles up to 6 arguments (the register-passing limit). The 7th+ argument would need to be pushed on the stack per the SysV ABI, but L1S's fallback _ => "rdi" silently reuses rdi, which is a latent bug for functions with more than 6 parameters.

GetElementPtr (GEP)

The GEP instruction computes pointer arithmetic at runtime: base + index * stride. This is used for array indexing and struct field access:

IrInstr::Gep(dest, base, index, stride) => {
    self.load_to_reg(*base, "rax", alloc);
    self.load_to_reg(*index, "rcx", alloc);
    if *stride > 1 { self.emit(&format!("  imul rcx, rcx, {}", stride)); }
    self.emit("  add rax, rcx");
    self.store_from_rax(*dest, alloc);
}

The stride is the element size (e.g., 4 for i32, 8 for a pointer). When stride is 1 (byte addressing), the multiplication is skipped as an optimization.

The Spill-Everything Register Model

A critical design choice visible throughout the code generator is that every instruction loads operands from memory and stores results to memory. There is no attempt to keep values in registers across instructions. This "spill-everything" model has important consequences:

  • Correctness is trivial: Since every value is always in its known stack slot, there is no register allocation interference between instructions.
  • Performance is poor: Each arithmetic instruction generates 3–4 memory accesses (load a, load b, store dest), when a register-only version would need 0. This is 3–4x more memory traffic.
  • Register pressure is zero: The code generator only uses rax and rcx as temporary registers. Even with hundreds of virtual registers, the generated code never spills.
Why spill everything? For a teaching compiler, correctness and simplicity trump performance. The spill-everything model eliminates an entire class of register allocation bugs. A production compiler would use a graph-coloring register allocator (like Chaitin's algorithm) to keep frequently-used values in registers.

Code Generation for Control Flow

Basic block terminators are translated to jump instructions. The pattern is a direct mapping from IR to x86-64:

IR TerminatorNASM Output
Jump(target)jmp .func_target
Branch(cond, t, f)mov rax, [rbp+slot]; test eax, eax; jnz .func_t; jmp .func_f
Return(val)mov rax, [rbp+slot]; jmp .return_func

Branch conditions are evaluated as "non-zero is true": the condition value is loaded, test eax, eax sets the Zero Flag if it is zero, and jnz jumps to the true target if it was non-zero.

Traced Example: Code Generation Walkthrough

Consider this L1S function:

func add(i32, a: i32, b: i32) {
    return a + b;
}

The IR builder produces a function with two parameters (%0 = a, %1 = b), one Alloc for the temporary (%2), one Add instruction, and a Return terminator. The StackAllocator assigns slots: %0 → rbp-8, %1 → rbp-16, %2 → rbp-24. The generated assembly:

global add
add:
  push rbp
  mov rbp, rsp
  sub rsp, 24
  mov [rbp-8], rdi          ; spill %0 (a) from rdi
  mov [rbp-16], rsi         ; spill %1 (b) from rsi
.add_0:
  mov rax, [rbp-8]          ; load %0 (a)
  mov rcx, [rbp-16]         ; load %1 (b)
  add eax, ecx              ; %2 = a + b
  mov [rbp-24], rax         ; spill %2
  mov rax, [rbp-24]         ; load return value
  jmp .return_add
.return_add:
  add rsp, 24
  pop rbp
  ret

Notice the redundant load of %2 immediately after storing it — the result of add is in rax, but the spill-everything model stores it and then immediately reloads it for the return. An optimizing code generator would detect this and keep the value in rax.

Common Pitfalls

  1. Stack offset sign errors: Stack slots are negative offsets from rbp. A positive offset would corrupt the caller's frame. The signed() helper is critical.
  2. Missing size specifiers: NASM requires explicit size specifiers like qword when writing immediates to memory: mov qword [rbp-8], 42, not mov [rbp-8], 42.
  3. Caller-saved register clobbers: The code generator does not track which registers are live across calls. Since every value is on the stack, this is safe, but a register-allocating code generator would need to save and restore caller-saved registers around calls.
  4. Division by zero: x86-64 idiv raises SIGFPE on division by zero. L1S does not check for this.
  5. 6-argument limit: Beyond 6 arguments, the SysV ABI requires stack-based passing, but L1S silently reuses rdi.

15   Assembly & Linking

From Assembly Text to Executable Binary

After code generation, the compiler has produced a string of assembly text — human-readable mnemonics like mov, add, and jmp. But this is not yet machine code. Two more tools are needed to turn this text into a running executable: an assembler and a linker.

The assembler translates assembly mnemonics into machine code bytes, producing an object file (.o). The linker resolves symbolic references between object files, relocates code and data to their final memory addresses, and produces an executable binary. Most compilers do not ship their own assembler or linker — instead, they invoke system tools like nasm (for assembly) and ld or cc (for linking).

The assembly-to-executable pipeline:

program.asm           // NASM assembly text (~100 bytes for a trivial program)
    │
    ▼  nasm -f elf64
    │
program.o              // ELF64 object file (machine code + metadata)
    │
    ▼  cc -no-pie
    │
program                // ELF64 executable (runnable binary)

Assemblers: How Text Becomes Machine Code

An assembler performs a conceptually simple translation: each assembly mnemonic and its operands map to a specific sequence of machine code bytes. For example, on x86-64:

AssemblyMachine Code (hex)Meaning
push rbp55Push base pointer onto stack
mov rbp, rsp48 89 e5Copy stack pointer to base pointer
retc3Return from function

Assemblers differ in their approach to this translation:

  • One-pass assemblers: Read the source once, emitting code as they go. Forward references (jumping to a label defined later) require a fixup table that is patched after the first pass. NASM uses this approach.
  • Two-pass assemblers: First pass builds a symbol table of all labels and their offsets; second pass emits code using the now-known addresses. MASM (Microsoft Assembler) uses this.
  • Multi-pass assemblers: Additional passes enable optimizations like jump-sizing (choosing between short and near jumps).

The assembler also produces an object file in a standard format (ELF64 on Linux, Mach-O on macOS, PE on Windows). An object file contains not just machine code, but also metadata: relocation entries (places where addresses need to be filled in), a symbol table (defined and undefined symbols), and section headers.

Linkers: Resolving References

The linker's job is to combine one or more object files into a single executable. It performs two main tasks:

  1. Symbol resolution: Match undefined symbols (e.g., a call to main from the C runtime startup code) to their definitions in other object files or libraries.
  2. Relocation: Adjust the code and data addresses in each object file to reflect the final memory layout of the executable. A jump to main that was assembled as "jump to offset 0" gets patched to the actual address of main in the final binary.

Modern compilers rarely invoke the linker (ld) directly. Instead, they invoke the C compiler frontend (cc or gcc), which knows how to pass the right flags to the linker, including the necessary C runtime startup files (crt0.o, crti.o, crtn.o) that set up _start and call main.

How Ignis Invokes the Toolchain

Ignis's compiler driver (main.rs) handles assembly and linking with three sequential steps: write the assembly file, invoke NASM, and invoke the C compiler as a linker frontend.

src/main.rs — Assembly, linking, and output

    fs::write(&asm_path, &asm).unwrap_or_else(|e| {
        eprintln!("error writing {}: {}", asm_path, e);
        std::process::exit(1);
    });

    // Step 2: Assemble with NASM
    let status = Command::new("nasm")
        .args(&["-f", "elf64", "-o", &o_path, &asm_path])
        .status()
        .unwrap_or_else(|e| { /* handle error */ });

    let status = Command::new("cc")
        .args(&["-o", &bin_path, &o_path, "-no-pie"])
        .status()
        .unwrap_or_else(|e| { /* handle error */ });

Step 1: Writing the Assembly File

The code generator's output (a String) is written to a .asm file alongside the intended output binary. If the input was program.is, the assembly file is program.asm. The .asm file is deliberately kept after compilation so developers can inspect the generated assembly for debugging.

Step 2: Invoking NASM

NASM (Netwide Assembler) is invoked with three flags:

  • -f elf64: Produce an ELF64 object file, the standard format on Linux x86-64.
  • -o <path>: Where to write the output object file.
  • The input .asm file: The assembly source.

The driver uses Command::status() to run NASM and wait for it to complete. If NASM is not installed (unwrap_or_else catches the io::Error) or if NASM reports assembly errors (status.success() returns false), the driver prints an error and exits with code 1.

NASM is required. Install it with your package manager: 
# Debian/Ubuntu
$ sudo apt install nasm
# Fedora
$ sudo dnf install nasm
# macOS
$ brew install nasm

Step 3: Linking with cc

Using cc (the C compiler, typically gcc or clang) as a linker frontend is a common trick in small compilers. The C compiler knows how to invoke the linker (ld) with all the platform-specific flags needed:

  • The correct C runtime startup files (crt0.o, etc.) are linked automatically. These provide the _start entry point that the kernel calls, which in turn calls our main.
  • Standard library paths are set up correctly.
  • The correct dynamic linker (ld-linux-x86-64.so.2) is specified.

Three flags are passed to cc:

  • -o <binary>: Output executable path.
  • <.o file>: Input object file.
  • -no-pie: Disable position-independent executable generation. L1S's generated assembly uses absolute addresses in some cases (e.g., section .data globals), which are incompatible with PIE. Without this flag, the linker would fail with relocation errors.

Step 4 & 5: Cleanup and Confirmation

After successful linking, the intermediate .o object file is deleted. The .asm file is kept. A success message is printed to stdout:

$ ignis program.is -o myprog
wrote myprog

Output Path Resolution

The driver supports two modes for determining the output binary name:

$ ignis program.is                  # produces ./program
$ ignis program.is -o mybinary     # produces ./mybinary

Without -o, the input filename's extension is stripped using Path::with_extension(""). program.is becomes program. Both .asm and .o paths are derived from this base name.

Common Pitfalls

  1. NASM not installed: The error message is error invoking assembler: No such file or directory. Install NASM via your package manager.
  2. Missing main: If no func main(i32) is defined, the linker will report an undefined reference to main. Every L1S program needs an entry point.

  4. Output path edge cases: Path::with_extension("") works for most cases but can produce surprising results for inputs like .is (hidden file) or filenames with multiple dots.
  5. Temporary file deletion: If the linker succeeds, the .o file is deleted. If you want to inspect it, compile with -o and catch it before deletion, or modify the driver to keep it (e.g., with a --keep flag).

16   Error Diagnostics

Why Error Messages Matter

Error reporting is one of the most important user-facing features of a compiler. A programmer spends far more time debugging than writing code, and the quality of error messages directly determines how quickly they can find and fix problems. A good diagnostic tells the programmer not just what went wrong, but where and why.

Consider two responses to the same input error:

Bad diagnostic:
syntax error on line 5
Good diagnostic (rustc-style):
error[E000]: expected ';' after return expression
  --> test.is:5:12
   |
 5 |     return 42
   |            ^ expected ';'
   |
   = note: every return statement must end with a semicolon

The good diagnostic includes: the severity (error), the file and precise location (test.is:5:12), a source line with an underline caret pointing to the exact position, a descriptive message, and an optional note with additional guidance. This style, popularized by the Rust compiler (and earlier by Elm), has become the gold standard for compiler diagnostics.

Design Space for Diagnostics

Compiler diagnostics can be designed along several axes:

  • Severity levels: At minimum, errors (compilation cannot proceed) and warnings (suspicious but not fatal). Many compilers add notes (auxiliary information), help messages, and lint-level diagnostics.
  • Span attachment: A diagnostic can have zero spans (a global error), one primary span (the main location), or multiple secondary spans (related locations like "see declaration here").
  • Formatting: From bare text (line 5: syntax error) to structured output with source snippets, underlines, color, and annotations. Modern terminals support ANSI color codes, and many compilers (rustc, clang) use them to highlight errors.
  • Error codes: Unique identifiers like E0308 (type mismatch in Rust) that can be looked up in documentation. Some compilers (clang) even suggest fixes.
  • Continuation strategy: Stop-at-first-error vs. recover-and-continue. The latter finds more errors per compilation but requires careful error recovery in the parser.

How Ignis Implements Diagnostics

Ignis implements a simplified version of the rustc-style diagnostic system with three core types: Severity, Diagnostic (individual error/warning/note), and Diagnostics (the collector).

The Severity Enum

Every diagnostic carries a severity level that determines its behavior:

src/diag.rs — Severity
#[derive(Debug, Clone, PartialEq)]
pub enum Severity {
    Error,
    Warning,
    Note,
}

The three levels serve distinct roles:

  • Error: Compilation cannot produce a working binary. The driver exits with a non-zero status code. The has_errors() method checks for any diagnostic with this severity.
  • Warning: The compiler noticed something suspicious but can continue. Warnings alone do not prevent code generation.
  • Note: Supplementary information attached to an error or warning. Notes never appear standalone; they always provide context for another diagnostic.

The Diagnostic Struct (Builder Pattern)

Individual diagnostics are constructed using a builder pattern, where each setter method consumes and returns Self for chaining:

src/diag.rs — Diagnostic struct
#[derive(Debug, Clone)]
pub struct Diagnostic {
    pub severity: Severity,
    pub message: String,
    pub span: Option<Span>,
    pub notes: Vec<(String, Option<Span>)>,
}

impl Diagnostic {
    pub fn new(severity: Severity, message: impl Into<String>) -> Self {
        Diagnostic { severity, message: message.into(), span: None, notes: Vec::new() }
    }
    pub fn at(mut self, span: Span) -> Self {
        self.span = Some(span); self
    }
    pub fn with_note(mut self, msg: impl Into<String>, span: Option<Span>) -> Self {
        self.notes.push((msg.into(), span)); self
    }
}

Usage produces readable, chainable code:

let diag = Diagnostic::new(Severity::Error, "type mismatch")
    .at(some_span)
    .with_note("expected i32, found bool", None)
    .with_note("see definition here", Some(def_span));

The Diagnostics Collector

The Diagnostics struct acts as the central repository. It holds the source text and file path (set once early in the pipeline) and provides convenience methods for emitting diagnostics:

src/diag.rs — Diagnostics collector

    pub diagnostics: Vec<Diagnostic>,

}

impl Diagnostics {
    pub fn set_source(&mut self, source: impl Into<String>, path: impl Into<String>)
    pub fn error(&mut self, msg: impl Into<String>, span: Option<Span>)

The source and file_path fields are set once (typically by the parser) and reused for all emitted diagnostics. Having the source text available allows the formatter to extract and display the offending line with an underline caret. The convenience methods (error, warning, note) use a private at_maybe helper that conditionally attaches a span: if Some(span) is provided, the diagnostic gets a location; if None, it remains location-less.

The Formatter: Producing Rustc-Style Output

The format_diagnostic method is the heart of the system. It produces structured, multiline output with source context:

src/diag.rs — format_diagnostic()
pub fn format_diagnostic(&self, diag: &Diagnostic) -> String {
    let mut out = String::new();
    let severity = match diag.severity { /* "error"/"warning"/"note" */ };

    if let Some(span) = &diag.span {
        // Format: error[E000]: message
        //         --> file:line:col
        //          |
        //        5 | let x: i32 = "hello";
        //          |             ^^^^^^^
        //         = note: expected i32, found @char
    }
    // ... notes ...
    out
}

The formatting logic has several key details:

  • Severity label: Starts with error[E000]:, warning[E000]:, or note[E000]:. The E000 is a placeholder for unique error codes that would allow documentation lookup in a larger compiler.
  • Location line: --> file:line:col mirrors rustc's style. If no file path is available, just line:col is shown.
  • Source line extraction: The offending line is retrieved from source.lines(). Since Span.line is 1-indexed and .lines() is 0-indexed, the conversion (span.line - 1) as usize is critical.
  • Underline caret: A ^ is placed under the span's starting column, repeated for the span width (span.end - span.start, minimum 1). This visually highlights the exact range of the error.
  • Notes: Each note is appended as = note: message, optionally with a location: (at line:col).

Example Diagnostic Output

Given this erroneous L1S program:

The type checker would produce a diagnostic that formats as:

  --> test.is:2:23
   |

   |                   ^^^^^^^
   = note: expected i32, found @char (at 2:23)
   = note: see definition of 'x' (at 2:9)

This output tells the programmer everything they need: the file, the exact line and column, the problematic source text with the offending span highlighted, and two contextual notes explaining what was expected versus what was found.

Common Pitfalls

  1. Missing source context: If set_source is never called, format_diagnostic produces a bare error[E000]: message with no source line or caret. Always call set_source early in the pipeline.
  2. Off-by-one in line numbers: source.lines() returns 0-indexed lines, but Span.line is 1-indexed. Forgetting the -1 conversion produces an off-by-one error or a panic (if span.line is 1, indexing with 0 is correct; indexing with 1 would skip the first line).
  3. Zero-width spans: A span like Span::new(5, 5, ...) has zero width. The .max(1) in the width calculation ensures a single ^ caret is still displayed.
  4. Standalone notes: The note() convenience method on Diagnostics creates a top-level note, which is unusual — notes are meant to annotate errors or warnings. In practice this is harmless but might confuse readers.
  5. Error code placeholders: Every diagnostic uses E000. A production compiler would assign unique codes for documentation, searching, and automated suppression.

What a Compiler Does

Throughout this book, we have traced the path from raw source text to a running executable. Every compiler, from the simplest to the most advanced, follows the same essential stages:

Source Code
  │
  ▼ Lexical Analysis    Characters → Tokens (meaningful lexical units)
  │
  ▼ Syntactic Analysis    Tokens → Abstract Syntax Tree (grammatical structure)
  │
  ▼ Semantic Analysis    AST → Typed AST (type checking, name resolution)
  │
  ▼ IR Construction      AST → Intermediate Representation (machine-independent)
  │
  ▼ Optimization         IR → Optimized IR (dead code elimination, constant folding)
  │
  ▼ Register Allocation  Map virtual registers to stack slots or physical registers
  │
  ▼ Code Generation     IR → Assembly (target-specific machine instructions)
  │
  ▼ Assembly & Linking  Assembly → Object file → Executable binary

Each stage has its own design space with competing tradeoffs. The lexer could be hand-written or generated from regular expressions. The parser could be recursive-descent or LALR. The type checker could use simple annotation checking or full Hindley-Milner inference. The code generator could hand-emit assembly or delegate to LLVM. Ignis makes deliberate, pedagogical choices at every stage: simplicity over performance, clarity over cleverness.

What makes Ignis's approach distinctive:

  • Simplicity-first design: The spill-everything register allocator, the hand-written lexer with save/restore for comments, the straightforward recursive-descent parser — every module prioritizes readability over optimization.
  • Error resilience: Both the lexer and parser recover from errors and continue, maximizing the number of issues found per compilation. The diagnostics system produces rustc-style output with source snippets and caret underlines.
  • Real output: Ignis generates NASM-syntax x86-64 assembly that is assembled and linked into a real ELF64 executable. This is not a toy VM or bytecode interpreter — the generated code runs directly on the CPU.
  • Self-contained: The entire compiler is roughly 2,500 lines of Rust with zero dependencies beyond the standard library, NASM, and the system C compiler.

Future Directions

Ignis is a teaching compiler, but it could grow in many directions:

  • Register allocation: Replace the spill-everything StackAllocator with a graph-coloring allocator (Chaitin's algorithm) to keep values in registers, dramatically reducing memory traffic.
  • Optimization passes: Add constant propagation, common subexpression elimination, loop invariant code motion, and tail call optimization to the ir_opt module.
  • Type inference: Extend the type checker with Hindley-Milner inference and let-polymorphism, reducing the need for explicit type annotations.
  • Standard library: Build a minimal stdlib with I/O, memory allocation, and data structures, leveraging the extern mechanism to call libc functions.
  • Self-hosting: The ultimate milestone — write an L1S compiler in L1S itself. This requires enough language features to implement the compiler's own source code.
  • Additional targets: The IR is largely architecture-neutral. Adding backends for ARM64, RISC-V, or WebAssembly would demonstrate how compilers support multiple platforms.
  • Incremental compilation: A module system with separate compilation and caching would make iteration faster for larger projects.

The Ignis compiler gives you a complete, working system that you can study, modify, and extend.

Ignis does all of this in roughly 2,500 lines of Rust.
