1   Pipeline Overview

The Idea of a Compiler Pipeline

Every compiler, regardless of language or target, is fundamentally a sequence of transformations. Source text enters one end, machine code exits the other, and each intermediate step operates on a progressively lower-level representation. This sequence is called the compiler pipeline.

The key insight is separation of concerns. Rather than one enormous function that goes from source to machine code (a "compiler-monolith"), we decompose the problem into phases. Each phase has a well-defined input, a well-defined output, and a single responsibility. This makes the compiler easier to reason about, test, and extend.

Most compilers divide into two major halves:

  • Frontend — Analyzes the source program: lexing, parsing, semantic analysis. Produces an annotated AST (abstract syntax tree with type information).
  • Backend — Synthesizes target code: instruction selection, register allocation, assembly emission. Produces machine code or assembly.

Between them often sits an Intermediate Representation (IR)—a data structure that is independent of both the source language and the target architecture. IRs are what make retargetable compilers possible: change the frontend to support a new language, change the backend to support a new CPU, keep the IR and its optimization passes unchanged.

Pipeline Design Tradeoffs

How many phases should a compiler have? The answer depends on what you are optimizing for:

  • Single-pass compilers (Pascal, early C) emit code as they parse. They are fast and use little memory, but the language must be designed so that every construct can be compiled without looking ahead. You cannot use a function before declaring it. These are rare today.
  • Two-pass compilers (many toy compilers) separate analysis from code generation. The first pass builds a data structure (often an AST), the second emits code from it. Simple and effective, but optimizations are limited.
  • Multi-pass compilers (LLVM, GCC, Rust, modern C) have many phases: lexing, parsing, name resolution, type checking, several IR transformations, instruction selection, register allocation, peephole optimization. Each pass is small and testable. This is the dominant model.

There is also a stylistic choice around error handling: some compilers abort on the first error, while others collect as many errors as possible in each phase before giving up. Modern compilers favor the latter—it is deeply frustrating to fix one typo only to discover another one on the next compilation.

The Ignis Pipeline at a Glance

Ignis uses an eight-phase multi-pass pipeline. Here is the data flow:

Source → Lexer → Parser → Type Checker → IR Builder → Optimizer → Reg Alloc → Codegen → Assembly

And here is what each phase does with its input:

Phases 1–3 are the frontend. They analyze the program and detect all errors. Phases 4–7 are the backend. They synthesize code and cannot fail (if the frontend produced valid output). The driver running nasm and cc is technically outside the compiler proper, but it is the final step that makes a usable tool.

What Data Flows Between Phases?

Each phase transforms its input into a new representation. These representations form a hierarchy of increasing specificity and decreasing abstraction:

Source text → a flat string of characters. The lexer cannot use this directly for any analysis beyond character grouping.

Tokens → a flat sequence of tagged values. The parser consumes these to build structure. Tokens carry Spans so errors can point back to the original source.

AST → a tree reflecting the grammatical structure of the program. After type checking, every expression node carries its resolved type.

IR → a control-flow graph of basic blocks containing low-level instructions. Virtual registers are infinite (the register allocator later maps them to real registers). This representation makes optimizations straightforward to implement and reason about.

Assembly → a string of x86-64 mnemonics. This is the final output of the compiler; nasm turns it into machine code bytes.

Internal vs External Phases

Some compiler phases are truly internal (they exist only within the compiler binary), while others rely on external tools. In Ignis, phases 1–7 are internal Rust code. Phase 8—assembling and linking—delegates to NASM and the system C compiler. This is a pragmatic choice: writing an assembler and linker from scratch would add thousands of lines without teaching anything new about compilation.

Many real compilers do the same. GCC emits assembly and calls an external assembler (as) and linker (ld). LLVM can emit assembly, object code, or use an integrated assembler. The boundary between "the compiler" and "the toolchain" is fluid and depends on your goals.

The advantage of using NASM is that the generated assembly is human-readable. You can open the .asm file and see exactly what instructions the compiler produced. This is invaluable for debugging and learning.

How the Pipeline is Wired in Ignis

The general pattern of a pipeline driver in Ignis is a straight-line sequence in main.rs. Let's walk through each call:

src/main.rs — the pipeline driver (annotated)
// === PHASE 1: LEXER ===
// The lexer takes a &str and returns (tokens, errors).
// Errors are collected but not fatal - we keep going.
let mut lexer = lexer::Lexer::new(&source);
let (tokens, _lex_errors) = lexer.tokenize();

// === PHASE 2: PARSER ===
// The parser builds an AST from the token stream.
// It owns the Diagnostics object and accumulates errors.
let diag = diag::Diagnostics::new();
let mut parser = parser::Parser::new(tokens, diag);
let mut prog = parser.parse_program();
let mut diag = parser.into_diagnostics();

// Error checkpoint: abort if parsing failed.
if diag.has_errors() {
    diag.print_all();
    std::process::exit(1);
}

// === PHASE 3: TYPE CHECKER ===
// The type checker walks the AST, resolves types, and
// annotates every expression with its type.
let mut types = types::TypeArena::new();
let mut symbols = symbol::SymbolTable::new();
let mut tc = typeck::TypeChecker::new(
    &mut types, &mut symbols, &mut diag);
tc.check_program(&mut prog);
let annotations = tc.annotations.clone();
let has_errors = tc.diag.has_errors();
drop(tc);

// Error checkpoint: type errors abort before codegen.
if has_errors {
    diag.print_all();
    std::process::exit(1);
}

// === PHASE 4: IR BUILDER ===
// Lowers the checked AST into a control-flow-graph IR.
let builder = ir_build::IrBuilder::new(
    &types, &annotations, &mut diag);
let mut ir_module = builder.build(&prog);

// === PHASE 5: OPTIMIZER ===
// Runs constant folding, dead code elimination, and
// copy propagation on the IR.
ir_opt::optimize(&mut ir_module);

// === PHASE 6 (implicit) & 7: REG ALLOC + CODEGEN ===
// The codegen handles register allocation internally.
let mut codegen = codegen::Codegen::new(&types);
let asm = codegen.generate(&ir_module);

// === PHASE 8: ASSEMBLE & LINK ===
// Write .asm file, call nasm, call cc, clean up.
fs::write(&asm_path, &asm);
Command::new("nasm").args(&["-f", "elf64", "-o", &o_path, &asm_path]);
Command::new("cc").args(&["-o", &bin_path, &o_path, "-no-pie"]);

Notice the two error checkpoints: one after parsing and one after type checking. If either produces errors, the compiler prints diagnostics and exits before attempting code generation. This means the backend never has to handle malformed input—a significant simplification.

Why two separate error checkpoints? Parse errors indicate the program is not even syntactically valid. Trying to type-check a syntactically invalid program would produce confusing cascading errors. By checking after parsing, we report only syntax errors. Then we type-check, and report only semantic errors. The user gets clean, separated error output.

Phase Granularity: Combined vs Separate

Some compilers combine adjacent phases. For example, a lexer and parser are often fused into a single pass because they are tightly coupled: the parser calls the lexer on demand (this is called a lexer with parser feedback or a scannerless parser). Ignis keeps them separate, which means the lexer produces a complete token list before the parser starts. This has two advantages:

  • Testability: You can test the lexer in isolation by inspecting its token output.
  • Simplicity: The parser does not need to worry about character-level details.

The cost is memory: the entire token list is stored before parsing begins. For a small language like Ignis this is negligible. For languages with millions of lines of code, some compilers use a two-lexer approach (one fast pre-lexer for the parser's lookahead needs) or re-lex on demand.

Similarly, the register allocator and code generator could be fused. Ignis keeps them separate because register allocation is a well-defined problem (mapping virtual registers to real ones) that benefits from its own dedicated pass and data structures.

Error Handling Philosophy

Different phases handle errors differently, reflecting their position in the pipeline:

  • Lexer: Never crashes. Collects errors in a Vec<LexError> and continues producing tokens. Invalid characters are skipped.
  • Parser: Never crashes. Uses the Diagnostics system to record errors and employs error recovery strategies (e.g., token synchronization) to continue parsing.
  • Type Checker: Never crashes. Records every type error and continues checking other parts of the program.
  • Backend (IR Builder, Codegen): If the frontend produced valid output, the backend should never encounter errors. If it does, it is a compiler bug, not a user error.

This layered approach means a user sees as many errors as possible in a single compilation run. Lexer errors might produce garbled tokens that cause parser errors, so the driver currently stops after parsing if there are lex errors. A production compiler would filter errors to avoid cascading noise, but the infrastructure to collect and report them is in place.

What About the Diagnostics Module?

You may have noticed mod diag in the module list but no dedicated pipeline phase for it. That is because diagnostics is not a phase—it is a library used by multiple phases. The parser, type checker, and IR builder all report errors through the same Diagnostics object. Separating error collection from error reporting means each phase can focus on its primary job and simply call diag.report(...) when something goes wrong.

This is the same pattern used in rustc (which has a Session object for diagnostics) and LLVM (which has DiagnosticHandler). It is another example of separation of concerns within the pipeline architecture.

Why This Design Works for Learning

The eight-phase pipeline is more phases than strictly necessary (a minimal compiler could do it in four: lex, parse, codegen, link). But the extra phases—type checking, IR building, optimization, register allocation—each teach a distinct concept that appears in every production compiler. By separating them, you can study each in isolation:

  • Lexing teaches finite-state machines and character-level scanning.
  • Parsing teaches grammar, recursion, and tree construction.
  • Type checking teaches constraint satisfaction and semantic analysis.
  • IR teaches intermediate representations and control-flow graphs.
  • Optimization teaches program analysis and transformation.
  • Register allocation teaches graph coloring and resource constraints.
  • Code generation teaches instruction selection and calling conventions.

Each phase is small enough (typically 200–500 lines of Rust) to read in one sitting. The total compiler is under 5,000 lines of code—small enough to fit in your head, large enough to be a real compiler.

Summary

The Ignis pipeline is a linear sequence of well-defined transformations, each building on the previous. The frontend (lexer, parser, type checker) analyzes the program and reports errors. The backend (IR builder, optimizer, codegen) synthesizes assembly. The driver (assembly + linking) produces the final executable. This structure is not unique to Ignis—it follows the same architectural principles as GCC, LLVM, and rustc, though at a much smaller scale that fits in a single afternoon of reading.
