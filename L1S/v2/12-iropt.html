12   IR Optimization

What is IR Optimization?

IR optimization transforms the intermediate representation to make it faster, smaller, or both — without changing the program's meaning. The IR builder emits naive code: every operation gets its own vreg, every variable access goes through memory, and unused computations accumulate. Optimization cleans this up.

The key principle: each optimization pass must be sound (it must never change the program's behavior) and idempotent (repeated application converges). Passes are typically organized into a pipeline that runs to a fixed point — keep applying passes until no more changes occur.

Naive IR → Constant Folding → Dead Code Elim → Copy Propagation → Remove Unreachable → Optimized IR

Pass Ordering and Fixed-Point Iteration

Optimizations interact: one pass may expose opportunities for another. Constant folding creates constants that make other expressions foldable. Dead code elimination may make blocks unreachable. Copy propagation may create dead stores.

The standard solution is fixed-point iteration: run all passes in a loop until none of them changes anything. A maximum iteration count prevents infinite loops from buggy passes.

Ignis does exactly this:

pub fn optimize(module: &mut IrModule) {
    for _ in 0..100 {
        let changed = constant_folding(module)
            | dead_code_elimination(module)
            | copy_propagation(module);
        if !changed { break; }
    }
    remove_unreachable_blocks(module);
}

Three passes run in a fixed-point loop (max 100 iterations). Unreachable block removal runs once at the end — it's a cleanup pass that doesn't create new optimization opportunities.

Design Space: Optimization Strategies

IR optimization ranges from simple local peephole passes to sophisticated global analysis:

  • Peephole optimization: Match small patterns in a sliding window of instructions (e.g., Add %x, 0 → %x). Fast but limited.
  • Local optimization: Works within a single basic block. Dead code elimination within a block is local.
  • Global optimization: Works across the entire function. Constant propagation across blocks, global value numbering, loop invariant code motion.
  • Interprocedural optimization: Works across function calls. Inlining, function specialization. Beyond the scope of a teaching compiler.

Ignis uses mostly local and global passes with simple dataflow. Each pass is under 60 lines of code, making them easy to understand and verify.

Pass 1: Constant Folding

Constant folding evaluates arithmetic operations whose operands are known constants at compile time. Instead of emitting Add %0, 2, 3, the pass replaces it with MovImm %0, 5.

fn constant_folding(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut const_values: HashMap<VReg, u64> = HashMap::new();

        for block in &mut func.blocks {
            let mut replacements: Vec<(usize, IrInstr)> = Vec::new();

            for (i, instr) in block.instrs.iter().enumerate() {
                match instr {
                    IrInstr::MovImm(d, v, _) => {
                        const_values.insert(*d, *v);
                    }
                    IrInstr::Add(d, a, b) => {
                        if let (Some(av), Some(bv)) =
                            (const_values.get(a), const_values.get(b))
                        {
                            let val = av.wrapping_add(*bv);
                            replacements.push((i, IrInstr::MovImm(*d, val, ...)));
                            changed = true;
                        }
                    }
                    IrInstr::Sub(d, a, b) => { /* same pattern */ }
                    IrInstr::Mul(d, a, b) => { /* same pattern */ }
                    _ => {}
                }
            }

            for (i, replacement) in replacements {
                block.instrs[i] = replacement;
            }
        }
    }
    changed
}

The pass maintains a HashMap<VReg, u64> tracking which vregs hold known constants. When it encounters a binary op where both operands are in the map, it computes the result and replaces the instruction with a MovImm.

Before:

%0 = MovImm 2
%1 = MovImm 3
%2 = Add %0, %1

After:

%2 = MovImm 5

The %0 = MovImm 2 and %1 = MovImm 3 now become dead code — the next pass will remove them.

Wrapping arithmetic: Ignis uses wrapping_add/wrapping_sub/wrapping_mul. This is correct for a compiler: integer overflow is defined to wrap in the IR, matching typical hardware behavior. A higher-level language might define overflow as undefined behavior, but that's a language semantics choice, not an IR concern.

Pass 2: Dead Code Elimination

Dead code elimination (DCE) removes instructions whose results are never used. An instruction is "dead" if its destination vreg is never referenced by any other instruction, phi node, or terminator.

The algorithm is straightforward:

  1. Collect uses: Scan all instructions, phi nodes, and terminators. Record every vreg that appears as an operand (not a destination).
  2. Filter: Keep only instructions that have side effects (Store, Call, MemCopy) or whose destination appears in the use set.
fn dead_code_elimination(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut uses: HashSet<VReg> = HashSet::new();

        // Phase 1: collect all used vregs
        for block in &func.blocks {
            for instr in &block.instrs {
                for op in instr_uses(instr) { uses.insert(op); }
            }
            for phi in &block.phis {
                for (src, _) in &phi.sources { uses.insert(*src); }
            }
            match &block.terminator {
                IrTerminator::Branch(cond, _, _) => { uses.insert(*cond); }
                IrTerminator::Return(val) => { uses.insert(*val); }
                _ => {}
            }
        }

        // Phase 2: retain only useful instructions
        for block in &mut func.blocks {
            block.instrs.retain(|instr| {
                match instr {
                    IrInstr::Store(_, _) | IrInstr::Call(_, _, _) |
                    IrInstr::MemCopy(_, _, _) => true,  ; side effects
                    _ => uses.contains(&instr.dest()),
                }
            });
        }
    }
    changed
}

Before:

%0 = MovImm 2        ; %0 never used
%1 = MovImm 3
%2 = Add %1, %3      ; %2 never used
%3 = Store %4, %5    ; side effect, must keep
Return %5

After:

%3 = Store %4, %5    ; kept for side effect
Return %5

Pass 3: Copy Propagation

Copy propagation replaces uses of one vreg with another when a Load instruction simply copies a value from a source vreg. If the source vreg was itself a copy from an earlier load, the chain is followed.

Ignis's copy propagation is simple: it maps each Load %d, %s as a copy from %s to %d. Subsequent uses of %d can be replaced with %s (or further back if %s was also a copy).

fn copy_propagation(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut copies: HashMap<VReg, VReg> = HashMap::new();

        for block in &func.blocks {
            for instr in &block.instrs {
                if let IrInstr::Load(d, s) = instr {
                    if let Some(&orig) = copies.get(s) {
                        copies.insert(*d, orig);
                    } else {
                        copies.insert(*d, *s);
                    }
                    changed = true;
                }
            }
        }
    }
    changed
}

Note: Ignis's copy propagation is incomplete — it tracks copy chains but does not actually rewrite uses of the propagated vreg. A more complete implementation would walk all instructions and replace operand vregs with their propagated values. This is a known simplification.

Pass 4: Remove Unreachable Blocks

After optimizations, some basic blocks may become unreachable from the entry block. These blocks waste space and may contain dead code that DCE missed. The fix is a simple graph traversal:

fn remove_unreachable_blocks(module: &mut IrModule) {
    for func in &mut module.functions {
        let entry = func.entry_block;
        let mut reachable: HashSet<IrBlockId> = HashSet::new();
        let mut worklist: Vec<IrBlockId> = vec![entry];

        while let Some(block_id) = worklist.pop() {
            if reachable.insert(block_id) {
                let successors = block_successors(&func.blocks[block_id.index()]);
                worklist.extend(successors);
            }
        }

        func.blocks.retain(|b| reachable.contains(&b.id));
    }
}

The helper block_successors extracts the target block(s) from a block's terminator:

fn block_successors(block: &IrBlock) -> Vec<IrBlockId> {
    match &block.terminator {
        IrTerminator::Jump(target) => vec![*target],
        IrTerminator::Branch(_, t, f) => vec![*t, *f],
        IrTerminator::Return(_) => vec![],
    }
}

After removal, phi nodes that reference removed predecessor blocks are cleaned up, and terminators that jump to removed blocks are redirected to the block itself (a self-loop).

Traced Example: Full Optimization Pipeline

Source:

func example(i32) {
    let x: i32 = 2 + 3;
    let y: i32 = x * 4;
    return y;
}

Before optimization (naive IR):

bb0 (entry):
  %0 = Alloc i32           ; slot for x
  %1 = MovImm 2
  %2 = MovImm 3
  %3 = Add %1, %2          ; 2 + 3
  Store %3, %0             ; x = 2+3
  %4 = Alloc i32           ; slot for y
  %5 = Load %0             ; read x
  %6 = MovImm 4
  %7 = Mul %5, %6          ; x * 4
  Store %7, %4             ; y = x*4
  %8 = Alloc i32           ; slot for return value
  %9 = Load %4             ; read y
  Return %9

After constant folding:

bb0 (entry):
  %0 = Alloc i32
  %1 = MovImm 2            ; unused -> DCE will remove
  %2 = MovImm 3            ; unused -> DCE will remove
  %3 = MovImm 5            ; folded from 2+3
  Store %3, %0
  %4 = Alloc i32
  %5 = Load %0
  %6 = MovImm 4            ; used by Mul, cannot fold yet
  %7 = Mul %5, %6          ; can't fold: %5 not constant
  Store %7, %4
  %8 = Alloc i32           ; unused -> DCE will remove
  %9 = Load %4
  Return %9

After dead code elimination:

bb0 (entry):
  %0 = Alloc i32
  %3 = MovImm 5            ; used by Store
  Store %3, %0
  %4 = Alloc i32           ; used by Load
  %5 = Load %0             ; used by Mul
  %6 = MovImm 4            ; used by Mul
  %7 = Mul %5, %6          ; used by Store
  Store %7, %4
  %9 = Load %4             ; used by Return
  Return %9

After remove unreachable blocks:

Only one block, all reachable. No change.

The optimized IR still has Alloc/Store/Load patterns. A real optimizer would also do store-to-load forwarding and dead store elimination to remove the redundant memory traffic. But even these simple passes already eliminated 3 of the original 11 instructions.

Optimization Debugging Checklist

  1. Did constant folding miss anything? It only handles Add, Sub, Mul. Division, comparisons, and shifts are not folded.
  2. Are side effects properly preserved? DCE never removes Store, Call, or MemCopy even if their destination is unused.
  3. Did copy propagation actually rewrite uses? Ignis's pass tracks copies but doesn't rewrite operands — this is a known gap.
  4. Are self-loop terminators safe? When a Jump target becomes unreachable, it's redirected to the block itself. This could create an infinite loop in unoptimized code.
Testing optimizations: Write tests with specific IR patterns and assert the expected instruction count after optimization. Ignis's test module in ir_opt.rs does exactly this: constructing a minimal function, running a pass, and checking the result.
