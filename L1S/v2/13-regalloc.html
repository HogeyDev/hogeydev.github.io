13   Register Allocation

The Register Allocation Problem

Register allocation is the problem of mapping an unlimited number of virtual registers to a fixed set of physical CPU registers. The IR builder freely creates as many vregs as it needs — a single function might use hundreds. But x86-64 has only 16 general-purpose registers (and many are reserved for specific purposes: stack pointer, frame pointer, etc.).

When there aren't enough physical registers to hold all live values simultaneously, some values must be spilled to the stack. Spilling means storing a value to memory when the register is needed for something else, then loading it back when it's needed again. Every spill adds memory traffic, so good register allocation minimizes spills.

Unlimited vregs
%0, %1, %2, ... %N → Register
Allocator → 16 physical regs
rax, rbx, rcx, ...

↓ Spill slots on stack
[rbp-8], [rbp-16], ...

Design Space: Allocation Strategies

Register allocation is a well-studied NP-complete problem (it's equivalent to graph coloring). Real compilers use a variety of heuristics:

  • Graph coloring: Build an interference graph where edges connect vregs that are simultaneously live. Color the graph with K colors (K = number of registers). If coloring fails, spill a node and retry. Used by GCC, LLVM (-O2). Optimal but complex.
  • Linear scan: Sort live ranges by their start point. Scan forward, assigning registers. When a register is needed and none is free, spill the vreg whose next use is farthest. Used by LuaJIT, many JIT compilers. Fast, good results.
  • Spill everything: Every vreg gets its own stack slot. No interference analysis, no coloring, no scanning. Trivially correct, O(1) per vreg. Maximum memory traffic, but the codegen is straightforward: every instruction reads operands from the stack and writes results to the stack.
Tradeoff: Graph coloring produces the best code but is the hardest to implement and debug. Linear scan is a good middle ground. Spill-everything is the simplest correct allocator — it's what you write when you want a working compiler quickly and performance is secondary.

Ignis: The Spill-Everything StackAllocator

Ignis uses the spill-everything approach. Every vreg — whether it's a local variable, a temporary computation result, or a parameter — gets its own dedicated slot on the stack. The allocator never attempts to keep values in registers across instructions. Instead, it assigns each vreg a fixed offset from the stack base pointer (rbp).

The allocator is called StackAllocator (in src/regalloc.rs, just 92 lines):

pub struct StackAllocator {
    pub slot_map: HashMap<VReg, i64>,  ; vreg ⇒ stack offset
    pub stack_size: i64,                 ; total frame size (negative, growing down)
}

The algorithm is dead simple:

  1. Walk every instruction in the function.
  2. For every vreg that is defined (as a destination), allocate a stack slot.
  3. For every parameter vreg, allocate a stack slot.
  4. For every phi-node destination vreg, allocate a stack slot.
  5. Align the total stack size to 16 bytes (x86-64 ABI requirement).
pub fn allocate(&mut self, func: &IrFunction) {
    // Parameters get slots
    for param in &func.params {
        self.make_slot(param.vreg, param.type_);
    }

    // Instructions: Alloc explicitly requests a slot; other instrs get
    // a slot for their destination vreg
    for block in &func.blocks {
        for instr in &block.instrs {
            match instr {
                IrInstr::Alloc(vreg, type_) => {
                    self.make_slot(*vreg, *type_);
                }
                IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {}
                instr => {
                    let dest = instr.dest();
                    self.make_slot(dest, TypeId::new(11));
                }
            }
        }
        // Phi nodes get slots
        for phi in &block.phis {
            self.make_slot(phi.dest, phi.type_);
        }
    }

    // 16-byte alignment
    self.stack_size = self.stack_size & !15;
}

Slot allocation is just a counter decremented by 8 bytes each time:

fn make_slot(&mut self, vreg: VReg, _type_: TypeId) {
    if !self.slot_map.contains_key(&vreg) {
        self.stack_size -= 8;       ; grow stack downward
        self.slot_map.insert(vreg, self.stack_size);
    }
}

Every slot is 8 bytes regardless of the actual type. This wastes space for i8 or bool values (which need only 1 byte) but keeps the layout simple and aligned.

Stack Frame Layout

After allocation, each vreg maps to a fixed offset from rbp. The stack frame grows downward (toward lower addresses):

        Higher addresses
        +------------------+
rbp ⇒ | saved rbp        |  (pushed by prologue)
        +------------------+
rbp-8   | slot for %0      |  (first allocated)
        +------------------+
rbp-16  | slot for %1      |  (second allocated)
        +------------------+
rbp-24  | slot for %2      |
        +------------------+
        | ...              |
rsp ⇒ +------------------+
        Lower addresses

The codegen phase uses the slot offset to emit memory operand addresses like [rbp-8], [rbp-16], etc.

Traced Example: Before and After Allocation

Consider the optimized IR from Chapter 12:

Total frame size: 64 bytes (56 rounded up to next 16-byte boundary). The function prologue will allocate this much stack space:

; x86-64 prologue
push  rbp
mov   rbp, rsp
sub   rsp, 64       ; allocate 64 bytes for spills

Each IR instruction is then lowered to use memory operands:

; %3 = MovImm 5
mov   qword [rbp-16], 5

; Store %3, %0   (write to variable slot)
mov   rax, [rbp-16]
mov   [rbp-8], rax

; %5 = Load %0  (read from variable slot)
mov   rax, [rbp-8]
mov   [rbp-32], rax

; %6 = MovImm 4
mov   qword [rbp-40], 4

; %7 = Mul %5, %6
mov   rax, [rbp-32]
mov   rbx, [rbp-40]
mul   rbx
mov   [rbp-48], rax

; Store %7, %4
mov   rax, [rbp-48]
mov   [rbp-24], rax

; %9 = Load %4
mov   rax, [rbp-24]
mov   [rbp-56], rax

; Return %9
mov   rax, [rbp-56]
ret

Every value goes through memory. This is correct but slow — a good allocator would keep frequently used values in registers.

Why Spill-Everything is (Sometimes) OK

Despite its obvious inefficiency, the spill-everything strategy has real advantages:

  1. Correct by construction: No interference analysis, no liveness tracking, no spilling decisions. Every vreg always has a home. The allocator cannot produce wrong code.
  2. Deterministic: The allocation depends only on the order vregs are encountered, not on heuristics or graph structure. The same IR always produces the same stack layout.
  3. Zero register pressure management: Since every value lives on the stack, there's never a moment where a register is needed but none is free. The codegen never needs to think about spilling during instruction selection.
  4. Simple codegen: Every instruction becomes a "load from stack, operate, store to stack" pattern. The code generator is a straightforward translation without register management.

The cost is performance. Every arithmetic operation requires at least two memory operations (load operands, store result). On modern x86-64 CPUs with good caching, this is often not as bad as it sounds — the stack slots stay in L1 cache, so the memory traffic is fast. But it's still slower than keeping values in registers.

What's Missing: Proper Register Allocation

To move beyond spill-everything, a compiler needs:

  • Liveness analysis: For each vreg, determine the set of program points where its value is still needed. Two vregs whose live ranges don't overlap can share a register.
  • Register coalescing: When a Store is immediately followed by a Load of the same vreg, the store and load can be eliminated if the value stays in a register.
  • Graph coloring or linear scan: An algorithm that assigns physical registers to vregs, spilling only when necessary.

Ignis's StackAllocator is intentionally minimal. It is the simplest correct allocator and serves as a foundation that could be extended with liveness analysis and a proper coloring pass.

Improving the allocator: A straightforward upgrade path is to add linear scan. First add liveness analysis (compute live ranges for each vreg via dataflow). Then sort live ranges by start point and scan forward, assigning registers until you run out, then spill. This would eliminate most memory traffic while keeping the allocator under 200 lines.

Testing the Allocator

Testing a spill-everything allocator is straightforward: allocate a known IR function and verify that every vreg gets a slot and that the stack size is correctly computed:

#[test]
fn test_allocate_allocs() {
    let mut func = make_test_func();
    let mut alloc = StackAllocator::new();
    alloc.allocate(&func);
    assert_eq!(alloc.slot_map.len(), 2);   ; 2 vregs got slots
    assert!(alloc.stack_size < 0);           ; stack grows down
    assert_eq!(alloc.stack_size % 16, 0);    ; 16-byte aligned
}

The key invariants to test: every vreg in the function has a slot, no two vregs share the same slot, and the total stack size is 16-byte aligned.
