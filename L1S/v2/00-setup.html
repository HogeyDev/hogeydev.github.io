What is a Compiler?

A compiler is a program that translates source code written in one language (the source language) into another language (the target language), typically machine code. The fundamental challenge is that humans think in high-level abstractions—functions, loops, variables, types—while CPUs operate on raw bytes, registers, and jump instructions. A compiler bridges this gap through a series of transformations, each step removing one layer of abstraction.

Not every language implementation is a compiler. The design space includes:

  • Interpreters execute source code directly without translation. They offer fast startup and easy introspection but slower execution. Examples: Python, Bash.
  • Just-In-Time (JIT) compilers translate code at runtime, blending compilation speed with execution speed. Examples: Java HotSpot, V8 (JavaScript).
  • Ahead-Of-Time (AOT) compilers translate entirely before execution. Startup is instant but the compilation itself takes time. Examples: C, Rust, Go.
  • Transpilers translate between high-level languages (e.g., TypeScript to JavaScript).

L1S is an AOT compiler with a traditional multi-phase pipeline. It translates a C-like language into x86-64 assembly, then uses NASM and the system linker to produce a standalone executable.

Why build a compiler? You learn exactly how high-level language features map to machine code. There is no magic—just a series of well-defined transformations. After building one, you will never look at if, while, or function calls the same way again.

Prerequisites

To build and run L1S you need three tools:

ToolVersionPurpose
Rust1.75+Compiler implementation language
NASM2.15+Assembles the generated x86-64 assembly into object files
GCC / ClanganyLinks object files into a final executable

The Rust toolchain handles the entire build process via Cargo. NASM and the C linker are invoked at runtime when compiling an L1S program—they are not needed during development of the compiler itself.

Building L1S

Building the compiler is a standard Cargo workflow:

$ git clone <repo-url> L1S
$ cd L1S
$ cargo build --release
$ ./target/release/ignis --help

The --release flag is important: the compiler itself benefits from optimization, especially on larger input programs.

Running the Compiler

The compiler takes a single source file and an optional output path:

$ cat hello.is
func main(i32) {
    return 42;
}

$ ./target/release/ignis hello.is
wrote hello
$ ./hello
$ echo $?
42

The intermediate hello.asm file is left on disk for inspection. You can open it to see exactly what x86-64 instructions the compiler generated for your program.

If no -o flag is given, the output binary gets the same name as the source file without the .is extension. Under the hood, the compiler produces a .asm file, invokes NASM to create a .o, then invokes cc to link it into an executable.

The Source Language — Ignis

L1S compiles a language called Ignis (Latin for "fire"). It is a minimal, statically-typed, C-like language with:

  • Functions with explicit parameter types and return types
  • i32, bool, char, pointer (@T), and array ([N]T) types
  • let bindings with explicit type annotations
  • if/else, while, return
  • Integer literals in decimal, hex, binary, and octal with underscore separators
  • Integer literals in decimal, hex, binary, and octal with underscore separators
  • Strings, characters, comments

Here is a complete Ignis program:

func gcd(i32, a: i32, b: i32) {
    if b == 0 {
        return a;
    }
    return gcd(b, a % b);
}

func main(i32) {
    let x: i32 = gcd(42, 56);
    return x;
}

The compiler pipeline will transform this into x86-64 assembly, then into an executable.

Project Structure

Every compiler pipeline has roughly the same logical phases. The L1S source tree is organized to mirror this pipeline directly. Each phase lives in its own module:

src/main.rs — module declarations and pipeline orchestration
mod span;       // Source locations (byte offsets, line/col)
mod token;     // Token definitions (keywords, operators, literals)
mod diag;      // Error reporting and diagnostics
mod lexer;     // Lexical analysis (source → tokens)
mod ast;       // Abstract syntax tree definitions
mod parser;    // Syntactic analysis (tokens → AST)
mod types;     // Type system (arena-allocated types)
mod symbol;    // Symbol table (scope-chain name resolution)
mod typeck;    // Semantic analysis / type checking
mod ir;        // Intermediate representation definitions
mod ir_build;  // IR construction (AST → IR)
mod ir_opt;    // IR optimization passes
mod abi;       // Calling convention and ABI details
mod regalloc;  // Register allocation
mod codegen;   // x86-64 code generation (IR → assembly)

Reading top to bottom, the module list tells the story of the compiler. Early modules (span, token, diag) are foundational data types used everywhere. Middle modules (lexer, parser, typeck) perform analysis. Late modules (ir_build, ir_opt, regalloc, codegen) perform synthesis and code generation. The abi module sits at the boundary between IR and codegen, defining how types are laid out in memory and how function calls pass arguments.

The Compiler Driver

The main function in main.rs is the driver—it orchestrates the entire pipeline. Here is how the general pattern of a multi-phase compiler driver plays out in Ignis:

src/main.rs — the pipeline driver
fn main() {
    // 1. Parse CLI arguments
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: ignis <file.is> [-o <output>]");
        std::process::exit(1);
    }

    // 2. Read source file
    let source = fs::read_to_string(&args[1]).unwrap_or_else(|e| { /* ... */ });

    // 3. Lexical analysis: source → tokens
    let mut lexer = lexer::Lexer::new(&source);
    let (tokens, _lex_errors) = lexer.tokenize();

    // 4. Syntactic analysis: tokens → AST
    let diag = diag::Diagnostics::new();
    let mut parser = parser::Parser::new(tokens, diag);
    let mut prog = parser.parse_program();
    let mut diag = parser.into_diagnostics();

    // 5. Error check before proceeding
    if diag.has_errors() { diag.print_all(); std::process::exit(1); }

    // 6. Semantic analysis: type checking
    let mut types = types::TypeArena::new();
    let mut symbols = symbol::SymbolTable::new();
    let mut tc = typeck::TypeChecker::new(&mut types, &mut symbols, &mut diag);
    tc.check_program(&mut prog);
    let annotations = tc.annotations.clone();
    let has_errors = tc.diag.has_errors();
    drop(tc);
    if has_errors { diag.print_all(); std::process::exit(1); }

    // 7. IR construction and optimization
    let builder = ir_build::IrBuilder::new(&types, &annotations, &mut diag);
    let mut ir_module = builder.build(&prog);
    ir_opt::optimize(&mut ir_module);

    // 8. Code generation and linking
    let mut codegen = codegen::Codegen::new(&types);
    let asm = codegen.generate(&ir_module);

    // 9. Write assembly, assemble, link
    // ... (path resolution, nasm, cc invocations)
}

The structure is linear and explicit. Each phase transforms its input into the next phase's input. Early-exit checks after parsing and type checking ensure that malformed programs fail fast with clear error messages rather than producing garbage IR or assembly.

Pipeline design philosophy: The driver does nothing clever. It is a straight-line sequence of function calls with error checks. This simplicity is deliberate—when something goes wrong, you can trace the exact path through the driver to find the bug. Every compiler phase is a pure-ish function: given valid input, it produces deterministic output or a list of errors.

Running Tests

L1S has unit tests embedded in each module and integration tests. Run them with:

$ cargo test

To run tests for a specific module (e.g., the lexer):

$ cargo test -- lexer

The test suite covers both happy paths and error cases. Many modules follow the pattern of defining a test helper that creates the phase, runs it, and returns the output, keeping tests concise and focused.

A Note About the Compiler's Name

The compiler is called Ignis (Latin for "fire"), and its source files use the .is extension. This name was chosen early in development and has stuck. Throughout this book, "Ignis" refers to the compiler itself, and "L1S" is the name of this book and the overall project repository. Think of L1S as the tutorial and Ignis as the tool being built.

Understanding main.rs: The Complete Program

Since main.rs is the entry point and the orchestrator, let's look at its full structure beyond just the pipeline calls. The file also handles:

  • Argument parsing: Reading the input file path and optional -o output flag.
  • File I/O: Reading the source file into a string, writing the assembly output.
  • External tool invocation: Running nasm to assemble and cc to link.
  • Cleanup: Removing the intermediate .o file after linking.

The driver does not parse command-line flags beyond the bare minimum. A production compiler would add --emit=asm, -O2, -g, etc. But for a learning compiler, the simple interface is a feature, not a bug—there is less code to read and fewer paths to test.
