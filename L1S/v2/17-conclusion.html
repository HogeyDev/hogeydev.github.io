17   Conclusion

What a Compiler Does

Throughout this book, we have traced the path from raw source text to a running executable. Every compiler, from the simplest to the most advanced, follows the same essential stages:

Source Code
  │
  ▼ Lexical Analysis    Characters → Tokens (meaningful lexical units)
  │
  ▼ Syntactic Analysis    Tokens → Abstract Syntax Tree (grammatical structure)
  │
  ▼ Semantic Analysis    AST → Typed AST (type checking, name resolution)
  │
  ▼ IR Construction      AST → Intermediate Representation (machine-independent)
  │
  ▼ Optimization         IR → Optimized IR (dead code elimination, constant folding)
  │
  ▼ Register Allocation  Map virtual registers to stack slots or physical registers
  │
  ▼ Code Generation     IR → Assembly (target-specific machine instructions)
  │
  ▼ Assembly & Linking  Assembly → Object file → Executable binary

Each stage has its own design space with competing tradeoffs. The lexer could be hand-written or generated from regular expressions. The parser could be recursive-descent or LALR. The type checker could use simple annotation checking or full Hindley-Milner inference. The code generator could hand-emit assembly or delegate to LLVM. Ignis makes deliberate, pedagogical choices at every stage: simplicity over performance, clarity over cleverness.

What makes Ignis's approach distinctive:

  • Simplicity-first design: The spill-everything register allocator, the hand-written lexer with save/restore for comments, the straightforward recursive-descent parser — every module prioritizes readability over optimization.
  • Error resilience: Both the lexer and parser recover from errors and continue, maximizing the number of issues found per compilation. The diagnostics system produces rustc-style output with source snippets and caret underlines.
  • Real output: Ignis generates NASM-syntax x86-64 assembly that is assembled and linked into a real ELF64 executable. This is not a toy VM or bytecode interpreter — the generated code runs directly on the CPU.
  • Self-contained: The entire compiler is roughly 2,500 lines of Rust with zero dependencies beyond the standard library, NASM, and the system C compiler.

Future Directions

Ignis is a teaching compiler, but it could grow in many directions:

  • Register allocation: Replace the spill-everything StackAllocator with a graph-coloring allocator (Chaitin's algorithm) to keep values in registers, dramatically reducing memory traffic.
  • Optimization passes: Add constant propagation, common subexpression elimination, loop invariant code motion, and tail call optimization to the ir_opt module.
  • Type inference: Extend the type checker with Hindley-Milner inference and let-polymorphism, reducing the need for explicit type annotations.
  • Standard library: Build a minimal stdlib with I/O, memory allocation, and data structures, leveraging the extern mechanism to call libc functions.
  • Self-hosting: The ultimate milestone — write an L1S compiler in L1S itself. This requires enough language features to implement the compiler's own source code.
  • Additional targets: The IR is largely architecture-neutral. Adding backends for ARM64, RISC-V, or WebAssembly would demonstrate how compilers support multiple platforms.
  • Incremental compilation: A module system with separate compilation and caching would make iteration faster for larger projects.

Final Thoughts

Building a compiler is one of the most rewarding projects in software engineering. It touches every corner of computer science: formal languages, data structures, algorithms, computer architecture, and systems programming. The Ignis compiler gives you a complete, working system that you can study, modify, and extend.

The journey from func main(i32) { return 42; } to a running executable involves lexing, parsing, type checking, IR construction, optimization, register allocation, code generation, assembly, and linking. Ignis does all of this in roughly 2,500 lines of Rust. If you have followed along through all 17 chapters, you now understand how a high-level programming language is translated into machine code — one of the deepest and most satisfying topics in computer science.
