4   The Lexer

4.1   What Is Lexical Analysis?

Concept. Every compiler begins by reading a stream of characters — the raw source file. The first transformation is to group these characters into meaningful atoms called tokens. This phase is lexical analysis (lexing or tokenizing).

Think of it as splitting a sentence into words:

Input: func main(i32) { return 42; }

Output:
Func   Ident("main")   OpenParen   I32   CloseParen   OpenBrace   Return   NumLiteral("42")   Semicolon   CloseBrace   Eof

The lexer does not understand semantics — it doesn't know that main is a function name or that 42 is an integer literal. It merely classifies spans of characters into token categories. The parser consumes these tokens to build structure.

The fundamental lexer loop is universal:

  1. Skip whitespace and comments
  2. Look at the next character
  3. Decide what kind of token it starts
  4. Consume characters belonging to that token
  5. Emit the token with its source location
  6. Repeat until EOF

Lexing is a deterministic finite automaton (DFA) in practice: at each step, the current character determines the next state. There is no backtracking and no ambiguity once the character class is known.

4.2   Design Space: How to Build a Lexer

Tradeoffs. There are three main approaches to building a lexer:

ApproachProsCons
Generator (lex/flex) Declarative regex rules, fast generated C code, battle-tested External tool, build dependency, opaque generated code, poor error messages
Hand-written DFA table Fastest possible, small code, data-driven Hard to maintain, table generation is error-prone, worse for simple languages
Hand-written recursive Easy to read and debug, exact error positions, no build deps, flexible Slightly slower per character, more code to write

L1S uses a hand-written recursive lexer. For a language with ~40 token types, this is the sweet spot. The lexer is about 400 lines of straightforward Rust with no external dependencies. Every character is examined exactly once (O(n)), and the performance difference from a generated lexer is negligible for any realistic input file.

Why not flex? Flex-generated lexers excel for languages with many token patterns (C's preprocessor, FORTRAN). For a simple C-like language, the complexity of integrating a code generator into the build pipeline outweighs the benefits. Hand-written lexers are the norm in modern compilers: Rust, Go, Zig, and Clang all use hand-written lexers.

4.3   Core Types

Before the lexer itself, we need the types it produces. The three building blocks are Span, Token, and SpannedToken.

Span encodes a source location: byte offsets plus line/column numbers. Every token carries one, enabling precise error messages downstream.

Token is a flat enum with a variant for every keyword, operator, delimiter, and literal in the language. The full enum is defined in src/token.rs and contains ~50 variants.

LexError pairs an error message with a Span:

src/lexer.rs — LexError
pub struct LexError {
    pub message: String,
    pub span: Span,
}

impl LexError {
    pub fn new(message: impl Into<String>, span: Span) -> Self {
        LexError { message: message.into(), span }
    }
}

The impl Into<String> parameter lets callers pass either &str or String without an explicit conversion. The span always points to the exact character(s) that caused the error, so downstream diagnostics can highlight the offending region.

4.4   The Lexer Struct & Position Model

Concept. A lexer is a stateful transducer. It holds a reference to the source text, a cursor into that text, and accumulates tokens and errors as it advances. The position model must handle two concerns: byte offsets (for slicing the source) and human-readable coordinates (for error messages).

Tradeoff: Byte offsets vs character indices. Rust's str::CharIndices yields (usize, char) pairs, giving us byte offsets automatically. Some lexers store character indices and convert to byte offsets only when needed. Byte offsets are more general — they let us slice the source directly — and Rust's UTF-8 guarantees make them equally convenient.

src/lexer.rs — Lexer struct
pub struct Lexer<'a> {
    source: &str,
    chars: Peekable<CharIndices<'a>>,
    pos: usize,
    line: u32,
    col: u32,
    errors: Vec<LexError>,
}

The fields in detail:

  • source — Borrowed input string. Never modified, only referenced.
  • chars — A Peekable<CharIndices>. CharIndices yields (byte_offset, char) pairs. Peekable gives us one character of lookahead without consuming it, which is essential for multi-character operators like ==.
  • pos — Current byte offset. Monotonically increases as characters are consumed.
  • line, col — 1-based line and column numbers, updated by advance().
  • errors — Accumulated errors. The lexer never panics; it records the error and continues.

Tradeoff: Lifetime parameter. The 'a lifetime ties the lexer to the source string. This means the lexer borrows the input rather than owning it. An alternative would be to copy the source into an owned String, but borrowing avoids an allocation and keeps the API zero-cost.

Position tracking is centralized in a single method:

fn advance(&mut self) -> Option<char> {
    match self.chars.next() {
        Some((_, c)) => {
            let len = c.len_utf8();
            self.pos += len;
            if c == '\n' {
                self.line += 1;
                self.col = 1;
            } else {
                self.col += 1;
            }
            Some(c)
        }
        None => None,
    }
}

Having all position updates in one place is essential for correctness. Every character consumed anywhere in the lexer goes through advance(), so pos, line, and col are always consistent. The complement is peek(), which looks ahead without consuming:

fn peek(&mut self) -> Option<char> {
    self.chars.peek().map(|&(_, c)| c)
}
UTF-8 safety. CharIndices yields complete Unicode characters. The len_utf8() call returns the byte width of the character (1–4 bytes), so pos always lands on a valid UTF-8 boundary. Emoji, CJK characters, and accented letters in comments or string literals are handled correctly without any special logic.

4.5   The Main Loop: tokenize()

Concept. Every lexer's entry point is a loop that drives the state machine: skip ignorable input, lex one token, emit it. The loop must handle three outcomes: a valid token (emit), an unrecognized character (record error, skip), or end of input (emit EOF and stop).

Tradeoff: Error accumulation vs Result. L1S collects errors in a Vec<LexError> and returns them alongside the tokens. The alternative would be to return Result<Vec<SpannedToken>>, stopping at the first error. Error accumulation is friendlier: the user sees all errors in one compilation, not one at a time with recompilation between fixes.

src/lexer.rs — tokenize()
pub fn tokenize(&mut self) -> (Vec<SpannedToken>, Vec<LexError>) {
    let mut tokens = Vec::new();

    loop {
        self.skip_whitespace();
        self.skip_comment();
        self.skip_whitespace();

        if self.chars.peek().is_none() {
            break;
        }

        let start_pos = self.pos;
        let start_line = self.line;
        let start_col = self.col;

        let token = self.lex_token();
        let end_pos = self.pos;
        let span = Span::new(start_pos, end_pos, start_line, start_col);

        match token {
            Some(tok) => {
                tokens.push(SpannedToken::new(tok, span));
                if tok == Token::Eof { break; }
            }
            None => {
                if let Some((_, c)) = self.chars.next() {
                    let len = c.len_utf8();
                    self.pos += len;
                    self.col += 1;
                    let span = Span::new(start_pos, self.pos, start_line, start_col);
                    self.errors.push(LexError::new(
                        format!("unexpected character '{}'", c), span));
                }
            }
        }
    }

    if tokens.last().map(|t| &t.token) != Some(&Token::Eof) {
        let span = Span::new(self.pos, self.pos, self.line, self.col);
        tokens.push(SpannedToken::new(Token::Eof, span));
    }

    (tokens, std::mem::take(&self.errors))
}

Key design decisions in this loop:

  • Whitespace/comments before every token — we skip twice (before and after comments), which handles the edge case of whitespace between // and the following token.
  • Span captured before and after lexing — the start position is saved before lex_token() runs; the end position is read after. This gives accurate bounds for every token.
  • EOF guarantee — the loop ensures at least one Token::Eof is emitted, even for empty input. The parser depends on this sentinel to avoid checking bounds.
  • std::mem::take — swaps self.errors with an empty vec, leaving the lexer in a clean state for potential reuse.

4.6   The Comment Ambiguity

Concept. Many languages use the same starting character for comments and operators: / starts both division (/) and line comments (//). The lexer cannot know which it is from the first character alone — it must look at the next character to decide.

Tradeoff: Save/restore vs two-pass. One approach is to lex / in lex_token() only after comments have been pre-skipped. But comments are supposed to be invisible to later stages, so they need to be consumed before token dispatch. L1S uses a save/restore pattern: tentatively consume the /, peek at the next character, and either consume the comment or restore the saved state so lex_token() sees / as a division operator.

fn skip_comment(&mut self) {
    if self.peek() == Some('/') {
        let saved = self.chars.clone();
        let saved_pos = self.pos;
        let saved_line = self.line;
        let saved_col = self.col;

        self.advance();
        if self.peek() == Some('/') {
            while let Some(c) = self.peek() {
                if c == '\n' { break; }
                self.advance();
            }
        } else {
            self.chars = saved;
            self.pos = saved_pos;
            self.line = saved_line;
            self.col = saved_col;
        }
    }
}

The iterator is cloneable because Peekable<CharIndices> is backed by a slice iterator, which is Clone. When / is followed by something other than /, we restore the entire lexer state, and the / is lexed as Token::Slash during the next call to lex_token().

Performance note. Restoring the iterator means we'll re-examine the / character. In a correct lexer this is unavoidable — you must peek ahead to disambiguate. The cost is minimal because this pattern only fires when the character is / (rare in typical source code).

4.7   Operator Dispatch

Concept. A lexer's dispatch table maps the first character of a potential token to either a single token or a more complex handler. Operators fall into three categories: single-character (+, *, (), two-character (==, !=, &&), and compound starting with a prefix that matches a single operator (- can be ->).

Tradeoff: Character matching vs table-driven. A match expression on the first character is the simplest approach. For a language with ~30 operators it's perfectly readable. An alternative is a lookup table from (char, Option<char>) to Token, but this sacrifices clarity for negligible speed gain.

fn lex_token(&mut self) -> Option<Token> {
    let c = self.peek()?;

    match c {
        '+' => { self.advance(); Some(Token::Plus) }
        '-' => {
            self.advance();
            if self.peek() == Some('>') {
                self.advance();
                Some(Token::Arrow)
            } else {
                Some(Token::Minus)
            }
        }
        '(' => { self.advance(); Some(Token::OpenParen) }
        ')' => { self.advance(); Some(Token::CloseParen) }
        '{' => { self.advance(); Some(Token::OpenBrace) }
        '}' => { self.advance(); Some(Token::CloseBrace) }
        '&' => {
            self.advance();
            if self.peek() == Some('&') {
                self.advance();
                Some(Token::AmpAmp)
            } else {
                Some(Token::Amp)
            }
        }
        '=' => {
            self.advance();
            if self.peek() == Some('=') {
                self.advance();
                Some(Token::EqEq)
            } else {
                Some(Token::Eq)
            }
        }
        '"'  => Some(self.lex_string()),
        '\'' => Some(self.lex_char()),
        '0'..='9' => Some(self.lex_number()),
        c if is_ident_start(c) => Some(self.lex_identifier_or_keyword()),
        _ => None,
    }
}

The pattern for multi-character operators is consistent: advance past the first character, peek at the second, and either return the compound token or fall back to the single-character one. This pattern repeats for !/!=, </<=/<<, >/>=/>>, and |/||.

4.8   Number Lexing

Concept. A numeric literal is a sequence of digits, optionally with a radix prefix. The lexer must distinguish decimal, hexadecimal (0x), binary (0b), and octal (0o) literals, and allow underscores as visual separators.

Tradeoff: Validate now or later? L1S lexes numbers as raw strings without validating their content. 0xGG is happily lexed as NumLiteral("0xGG"). The parser or type checker is responsible for rejecting invalid values. This keeps the lexer simple and decouples concerns. The alternative is to parse and validate the number in the lexer, which means error messages about invalid literals come from the lexer instead of being unified with other diagnostics.

fn lex_number(&mut self) -> Token {
    let mut raw = String::new();

    if self.peek() == Some('0') {
        self.advance();
        raw.push('0');
        match self.peek() {
            Some('x' | 'X') => {
                self.advance(); raw.push('x');
                while let Some(c) = self.peek() {
                    if c.is_ascii_hexdigit() || c == '_' {
                        self.advance(); raw.push(c);
                    } else { break; }
                }
            }
            Some('b' | 'B') => { /* binary: 0-1 and _ */ }
            Some('o' | 'O') => { /* octal: 0-7 and _ */ }
            _ => { /* decimal starting with 0 */ }
        }
    } else {
        while let Some(c) = self.peek() {
            if c.is_ascii_digit() || c == '_' {
                self.advance(); raw.push(c);
            } else { break; }
        }
    }
    Token::NumLiteral(raw)
}
Underscore behavior. Underscores are accepted anywhere in a number: 1_000, 0x_FF_EE, 0b_1010_1111. They are included verbatim in the raw string and must be stripped during parsing. This is the simplest approach — validating underscore placement is left to a later stage.

4.9   String and Character Lexing

Concept. String literals require a stateful sub-loop: enter on ", consume characters until the closing ", handling escape sequences (like \n, \t, \xHH) along the way. Character literals follow the same pattern but produce a single byte.

Tradeoff: Resolve escapes or keep raw? L1S resolves escape sequences during lexing, so "\n" becomes StrLiteral("\n") (one newline character), not StrLiteral("\\n"). This means downstream stages never need to understand escape sequences. The alternative is to keep the raw string and resolve escapes on demand, which is more flexible but pushes complexity into every consumer of string tokens.

fn lex_string(&mut self) -> Token {
    let mut content = String::new();
    self.advance(); // consume opening '"'

    loop {
        match self.peek() {
            None => {
                self.errors.push(LexError::new("unclosed string literal", span));
                break;
            }
            Some('"') => { self.advance(); break; }
            Some('\\') => {
                self.advance();
                match self.advance() {
                    Some('n') => content.push('\n'),
                    Some('t') => content.push('\t'),
                    Some('\\') => content.push('\\'),
                    Some('"') => content.push('"'),
                    Some('0') => content.push('\0'),
                    Some('x') => {
                        let hex = self.lex_hex_escape();
                        content.push(hex as char);
                    }
                    Some(c) => {
                        // invalid escape: record error, push char anyway
                        content.push(c);
                    }
                    None => break,
                }
            }
            Some(_) => {
                self.advance().map(|c| content.push(c));
            }
        }
    }
    Token::StrLiteral(content)
}

Character lexing is the same pattern but constrained to a single character and producing CharLiteral(u8). Both share lex_hex_escape() for \xHH sequences:

fn lex_hex_escape(&mut self) -> u8 {
    let mut hex = String::with_capacity(2);
    for _ in 0..2 {
        match self.peek() {
            Some(c) if c.is_ascii_hexdigit() => {
                self.advance(); hex.push(c);
            }
            _ => break,
        }
    }
    u8::from_str_radix(&hex, 16).unwrap_or(0)
}

4.10   Identifiers and Keywords

Concept. Identifiers and keywords share the same character class: they start with a letter or underscore and continue with letters, digits, and underscores. The lexer consumes the entire word, then checks whether it matches a reserved keyword. This is the simplest and most common approach.

Tradeoff: Keyword-as-identifier vs keyword filtering. Some lexers check for keywords character-by-character during consumption. L1S uses the simpler "consume then classify" approach: read the whole identifier, then look it up in a keyword table. This separates the concerns of "what is an identifier" (a simple regex) from "what is a keyword" (a set membership test).

fn lex_identifier_or_keyword(&mut self) -> Token {
    let mut ident = String::new();
    while let Some(c) = self.peek() {
        if c.is_alphanumeric() || c == '_' {
            self.advance();
            ident.push(c);
        } else { break; }
    }
    keyword_token(&ident).unwrap_or(Token::Ident(ident))
}

The keyword dispatch is a single function in token.rs that maps string names to token variants:

pub fn keyword_token(word: &str) -> Option<Token> {
    match word {
        "func"   => Some(Token::Func),
        "let"    => Some(Token::Let),
        "return" => Some(Token::Return),
        "if"     => Some(Token::If),
        "else"   => Some(Token::Else),
        "true"   => Some(Token::True),
        "false"  => Some(Token::False),
        "i32"    => Some(Token::I32),
        "u32"    => Some(Token::U32),
        // ... 30+ more entries for all keywords and type names
        _ => None,
    }
}

Adding a new keyword requires exactly two changes: add a variant to the Token enum, and add a match arm to keyword_token. The lexer never changes. This separation of concerns is the hallmark of a well-structured lexer.

4.11   Error Recovery Strategy

Concept. A lexer for a production compiler must handle malformed input gracefully. The goal is to report as many errors as possible in a single pass, so the user doesn't have to repeatedly compile to discover each mistake.

Tradeoffs in error recovery:

  • Panic on first error — Simplest, but terrible UX. One missing quote hides every subsequent error.
  • Synchronized recovery — Skip tokens until a synchronization point (like ; or }). Better for the parser; overkill for the lexer.
  • Record-and-continue — Record the error, skip the offending character(s), carry on. This is what L1S does.

L1S never panics. Every error path records a LexError and advances past the bad input. The three error categories are:

ErrorRecovery
Unrecognized character (e.g., $)Consume and skip it
Unclosed string literalClose it at EOF, return partial content
Invalid or unterminated escape sequencePush the raw character and continue
Never crash, never hang. Every loop in the lexer has an exit condition that is guaranteed to be reached: the iterator runs out of characters. There are no recursive descent spirals or unbounded lookahead loops. This is a property worth verifying for any lexer you write.

4.13   Testing the Lexer

Concept. A lexer is one of the easiest compiler components to test because it has a pure functional interface: source text in, tokens out. Every test follows the pattern lex(input) == expected_tokens.

The test module defines two helpers:

fn lex(source: &str) -> Vec<Token> {
    let mut lexer = Lexer::new(source);
    let (tokens, errors) = lexer.tokenize();
    assert!(errors.is_empty(), "Lex errors: {:?}", errors);
    tokens.into_iter().map(|st| st.token).collect()
}

fn lex_with_errors(source: &str) -> (Vec<Token>, Vec<LexError>) {
    let mut lexer = Lexer::new(source);
    lexer.tokenize()
}

Test categories cover:

  • Empty and whitespace-only input — produces only Eof
  • Keywords and identifiers — func vs foo, if vs x123
  • Number formats — decimal, hex, binary, octal, underscores
  • String escapes — \n, \t, \\, \xHH
  • Single and double operators — all operator variants
  • Delimiters — parens, braces, brackets, semicolons, commas, colons
  • Comments — // is skipped
  • Error paths — unclosed strings, invalid escapes, unexpected characters
  • Line tracking — multi-line input produces correct spans

To run the lexer tests:

$ cargo test -- lexer

Example: testing number lexing

#[test]
fn test_numbers() {
    let tokens = lex("42 0x1A 0b1010 0o77 1_000");
    assert_eq!(tokens[0], Token::NumLiteral("42".into()));
    assert_eq!(tokens[1], Token::NumLiteral("0x1A".into()));
    assert_eq!(tokens[4], Token::NumLiteral("1_000".into()));
}

#[test]
fn test_unexpected_character() {
    let (_, errors) = lex_with_errors("foo $ bar");
    assert!(errors.iter().any(|e| e.message.contains("unexpected character")));
}
Debugging tip. When the token stream doesn't match expectations, insert a temporary dump in tokenize(): 
eprintln!("token: {:?} at {}", tok, span);
This prints every token with its source location as it is emitted, which makes mismatches immediately obvious.

4.14   Common Pitfalls

  1. Line/column off by one. Initialize line: 1, col: 1. On newline, increment line and reset col to 1. Test with multi-line input early.
  2. Comment/division ambiguity. Without the save/restore pattern, a lone / is consumed by skip_comment and lost. Always test x / y alongside // comment.
  3. Unclosed strings hanging the lexer. Ensure every sub-loop has an EOF exit. The string lexer must handle "hello (no closing quote) without entering an infinite loop.
  4. Multi-byte UTF-8 confusion. Never index into the source string by character position. Always use CharIndices or equivalent. A byte offset that lands in the middle of a multi-byte character is a crash waiting to happen.
  5. Identifier starting with a digit. 123foo should NOT be an identifier. If lex_number accepts non-digit characters (as L1S currently does), the error is caught later. Decide deliberately where to validate.
  6. Overlapping multi-character sequences. >> and >=> can be ambiguous if you don't handle them left-to-right. L1S always peeks at the immediate next character and greedily forms the longest valid token.
