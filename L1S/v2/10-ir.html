10   Intermediate Representation

Concept: Decoupling Semantics from Machine Details

A compiler could translate the AST directly into assembly instructions. But this conflates two different concerns: what the program computes (semantics) and how the machine executes it (code generation). An intermediate representation (IR) sits between them, providing a machine-independent description of the program that is easier to analyze and transform than AST or assembly.

The key benefits of an IR layer:

  1. Machine independence. The same IR can target x86-64, ARM, RISC-V, or any backend without changing the IR builder. Only the code generator needs to change.
  2. Optimization. Transformations like constant folding, dead code elimination, and register allocation operate on a flat, uniform instruction sequence — much simpler than walking a tree.
  3. Analysis. Data-flow and control-flow analyses are straightforward on an IR with explicit basic blocks and virtual registers.
  4. Decoupling. The frontend (lexer, parser, type checker) and backend (codegen, register allocator) can be developed and tested independently.

Tradeoffs & Design Space

Tree IR vs. flat IR. A tree IR preserves the AST's hierarchical structure (e.g., C's generic IR trees in GCC). A flat IR linearizes everything into a sequence of instructions grouped into basic blocks. Flat IRs are more natural for optimization and code generation because they expose instruction ordering and data dependencies explicitly. L1S uses a flat, basic-block-based IR similar to LLVM IR.

SSA vs. non-SSA. Static Single Assignment (SSA) form requires that every variable is assigned exactly once. When control flow merges (e.g., after an if/else), phi nodes select the correct value based on which predecessor block executed. SSA simplifies many optimizations (constant propagation, dead code elimination) at the cost of phi-node insertion. L1S uses SSA form with explicit IrPhi nodes.

Register machine vs. stack machine. A register-machine IR (like LLVM or L1S) uses an unlimited set of virtual registers. A stack-machine IR (like JVM bytecode) uses a stack for operands. Register-machine IRs map more directly to physical CPUs and make data-flow analysis straightforward.

Instruction count. The number of IR instructions is a design choice: too few and you can't express the language efficiently; too many and the compiler becomes unwieldy. L1S has 36 instruction variants — a minimal set that covers all language features while mapping cleanly to x86-64 instructions. LLVM, by contrast, has hundreds.

Ignis Implementation: Basic-Block IR with Virtual Registers

The IR uses three handle types, each wrapping a u32:

src/ir.rs — handle types
pub struct VReg(pub u32);       // Virtual register: %0, %1, ...
pub struct IrBlockId(pub u32);  // Basic block index: bb0, bb1, ...
pub struct StringId(pub u32);    // String pool index

Each has a Display implementation for readable output: %3, bb1.

Module structure forms a hierarchy:

// Top level: a module contains functions, globals, and strings
pub struct IrModule {
    pub name: String,
    pub functions: Vec<IrFunction>,
    pub globals: Vec<IrGlobal>,
    pub strings: Vec<IrString>,
}

// A function: entry block, parameters as vregs, all blocks
pub struct IrFunction {
    pub name: String,
    pub return_type: TypeId,
    pub params: Vec<IrParam>,
    pub blocks: Vec<IrBlock>,
    pub num_vregs: u32,
    pub entry_block: IrBlockId,
}

// A basic block: phi nodes, instructions, one terminator
pub struct IrBlock {
    pub id: IrBlockId,
    pub name: String,          // e.g., "entry", "loop", "exit"
    pub phis: Vec<IrPhi>,
    pub instrs: Vec<IrInstr>,
    pub terminator: IrTerminator,
}

A function's entry_block is where execution begins. Every IrBlock ends with exactly one terminator — this is what defines a basic block: a maximal straight-line sequence of instructions with a single entry and single exit.

IrInstr: Five Categories, 36 Variants

Every instruction (except Store and MemCopy) produces a result in a destination virtual register. This is the key to SSA form: each vreg is written exactly once.

src/ir.rs — full IrInstr enum by category
pub enum IrInstr {
    // --- Memory ---
    Alloc(VReg, TypeId),       // Reserve stack slot of given type
    Load(VReg, VReg),          // Load value from pointer
    Store(VReg, VReg),         // Store value to pointer (no dest)
    MovImm(VReg, u64, TypeId), // Load integer constant

    // --- Arithmetic (3-address) ---
    Add | Sub | Mul | SDiv | UDiv | SRem | URem,
    And | Or | Xor | Shl | Shr | Ushr,

    // --- Unary ---
    Neg(VReg, VReg),           // Arithmetic negation
    Not(VReg, VReg),           // Logical not (bool)
    BitNot(VReg, VReg),        // Bitwise not

    // --- Comparison (all produce bool) ---
    Eq | Ne |           // Equality / inequality
    Slt | Sgt | Sle | Sge |  // Signed comparisons
    Ult | Ugt | Ule | Uge,     // Unsigned comparisons

    // --- Conversion ---
    Zext(VReg, VReg, TypeId),  // Zero-extend
    Sext(VReg, VReg, TypeId),  // Sign-extend
    Trunc(VReg, VReg, TypeId), // Truncate
    PtrToInt(VReg, VReg, TypeId),
    IntToPtr(VReg, VReg, TypeId),

    // --- Aggregate / Call ---
    Gep(VReg, VReg, VReg, u64), // base[index * stride]
    MemCopy(VReg, VReg, u64),  // memcpy(dst, src, size) (no dest)
    Call(VReg, String, Vec<VReg>), // dst = func(arg0, arg1, ...)
}

The signed/unsigned split for division, remainder, and comparison is necessary because x86-64 has different instructions for signed and unsigned operations. The type checker preserves signedness from the source types, and the IR builder emits the correct variant.

Terminators: How Blocks End

Every block has exactly one terminator. Three options cover all control flow:

src/ir.rs — terminators
pub enum IrTerminator {
    Jump(IrBlockId),                    // Unconditional jump
    Branch(VReg, IrBlockId, IrBlockId), // if (cond) true else false
    Return(VReg),                        // Return value from function
}
  • Jump: Used for unconditional control flow (e.g., after an if-then block, skip the else block).
  • Branch: Conditional branch. If the condition vreg is non-zero (true), go to the first block; otherwise go to the second. Used for if, while, for.
  • Return: Exit the function with the given vreg's value. For void functions, a dummy vreg holding zero is used.

SSA Phi Nodes

When different control-flow paths assign different values to the same variable, a phi node selects the right one:

pub struct IrPhi {
    pub dest: VReg,
    pub type_: TypeId,
    pub sources: Vec<(VReg, IrBlockId)>,
}

Semantically: %3 = phi(bb0: %1, bb1: %2) means "if control arrived from bb0, %3 = %1; if from bb1, %3 = %2." The code generator implements this by ensuring both predecessor values end up in the same physical location.

Phi nodes appear at the start of a block, before any instructions. They are not executable operations — they are a data-flow annotation that tells the optimizer and codegen how values merge at join points.

The dest() Method

Most instructions write to a destination vreg. The dest() method provides a uniform query interface, used extensively by optimization passes and register allocation:

pub fn dest(&self) -> VReg {
    match self {
        IrInstr::Alloc(d, _)
        | IrInstr::Load(d, _)
        | IrInstr::Add(d, _, _) | IrInstr::Sub(d, _, _)
        | /* ... every variant except Store/MemCopy */ => *d,
        IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {
            panic!("Store/MemCopy has no dest")
        }
    }
}

Globals, Strings, and Constants

Global variables are module-level named storage with optional initializers:

pub struct IrGlobal {
    pub name: String,
    pub type_: TypeId,
    pub init: Option<IrConstant>,
}

pub enum IrConstant {
    Int(u64, TypeId),
    String(StringId),
    Zero(TypeId),
}

String literals are pooled in IrModule.strings to avoid duplicates. Each IrString stores its raw byte content. References to strings in IR instructions use StringId indices.

Example: IR for a Simple Expression

Let's trace what the IR looks like for this function:

func main(i32) { return 1 + 2; }

The IR builder produces:

function main(i32) {
// entry: bb0
bb0 (entry):
%0 = Alloc i32 // allocate return slot
%1 = MovImm 1 : i32 // constant 1
%2 = MovImm 2 : i32 // constant 2
%3 = Add %1, %2 // %3 = 1 + 2
Return %3 // return value 3
}

Every detail is explicit:

  • Alloc reserves stack space for the return value (needed for the calling convention)
  • MovImm loads each constant into a virtual register
  • Add performs the addition in 3-address form: %3 = %1 + %2
  • Return exits with %3's value

An optimizer could constant-fold this to a single MovImm %3, 3 : i32 followed by Return %3, eliminating %1 and %2 entirely.

For a conditional program like:

func max(i32, a: i32, b: i32) {
    if (a > b) {
        return a;
    } else {
        return b;
    }
}

The IR would have three blocks (entry, then, else) with a Branch terminator at the end of entry and Return terminators in then and else:

function max(i32, %1: a, %2: b) {
bb0 (entry):
%3 = Sgt %1, %2 // %3 = a > b
Branch %3, bb1, bb2 // if %3 goto bb1 else bb2

bb1 (then):
Return %1

bb2 (else):
Return %2
}

Testing the IR

The IR has no logic of its own — it is purely a data structure — so its tests focus on construction and accessor correctness:

#[test]
fn test_vreg_new() {
    let r = VReg::new(5);
    assert_eq!(r.index(), 5);
}

#[test]
fn test_vreg_display() {
    assert_eq!(format!("{}", VReg::new(3)), "%3");
}

#[test]
fn test_instr_dest() {
    let add = IrInstr::Add(VReg::new(0), VReg::new(1), VReg::new(2));
    assert_eq!(add.dest(), VReg::new(0));

    let call = IrInstr::Call(VReg::new(3), "foo".into(), vec![VReg::new(1)]);
    assert_eq!(call.dest(), VReg::new(3));
}

#[test]
fn test_branch_terminator() {
    let t = IrTerminator::Branch(VReg::new(0), IrBlockId::new(1), IrBlockId::new(2));
    match t {
        IrTerminator::Branch(c, a, b) => {
            assert_eq!(c, VReg::new(0));
            assert_eq!(a, IrBlockId::new(1));
            assert_eq!(b, IrBlockId::new(2));
        }
        _ => panic!("expected Branch"),
    }
}

#[test]
fn test_phi_node() {
    let phi = IrPhi {
        dest: VReg::new(0), type_: TypeId::new(0),
        sources: vec![(VReg::new(1), IrBlockId::new(0)),
                          (VReg::new(2), IrBlockId::new(1))],
    };
    assert_eq!(phi.sources.len(), 2);
}

#[test]
fn test_ir_constant() {
    let c = IrConstant::Int(42, TypeId::new(0));
    match c {
        IrConstant::Int(v, _) => assert_eq!(v, 42),
        _ => panic!("expected Int"),
    }
}

Run IR tests with:

$ cargo test -- ir

Common Pitfalls

  1. Store/MemCopy have no destination. These are the only instructions without a destination vreg. Passes that iterate all definitions must explicitly skip them, or the dest() method will panic.
  2. Signed vs. unsigned comparisons. Using Slt (signed less-than) for unsigned values gives wrong results. For example, 0xFFFF_FFFF as u32 is 4294967295, but as i32 it's -1. The type checker must preserve signedness through to the IR builder.
  3. Missing phi nodes at merge points. If two branches assign different values to the same variable, a phi node is required where they rejoin. Without it, the variable's value is ambiguous and optimization passes will produce incorrect results.
  4. VReg numbering. The num_vregs field must be strictly greater than the largest vreg index. An off-by-one here causes register allocation to miss the last register.
Why basic blocks? A basic block is a maximal sequence of instructions with a single entry (at the top) and single exit (at the bottom). This property makes data-flow analysis tractable: within a block, all definitions reach all uses. Across blocks, you need data-flow equations or phi nodes. Almost every modern compiler uses basic blocks as the fundamental unit of IR.
