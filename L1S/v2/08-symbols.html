8   The Symbol Table

Concept: Name → Definition Mapping

A compiler needs to answer one question more than any other: what does this name refer to? When the parser produces an identifier node like x or main, the compiler must resolve it to a specific declaration — a variable, a function, a type, or an enum variant. The data structure that handles this is the symbol table: a mapping from names to their definitions.

Consider this program:

func main(i32) {
    let x: i32 = 1;
    {
        let x: i32 = 2;  // shadows outer x
    }
}

The identifier x appears three times: two definitions (the outer and inner declarations) and one use (inside the inner block for the initializer 2). The use of x in the initializer 2 refers to the inner definition, not the outer one. This is the essence of lexical scoping: a name refers to the innermost enclosing definition.

Tradeoffs & Design Space

Flat map vs. scope chain. The simplest approach is a single flat HashMap<String, Entry>. It works for languages with no nesting (some assembly dialects) but breaks badly for structured programming. You would need to mangle names (e.g., _main_inner_x) to handle shadowing, which produces confusing error messages and makes semantic analysis impossible.

The standard solution is a scope chain: a linked list of nested maps. Each scope has a parent pointer. Name lookup starts at the innermost scope and walks outward. This directly mirrors how programming languages handle nested blocks, function bodies, and module boundaries.

Persistent vs. ephemeral scopes. Some compilers (like GCC's early C frontend) use a tree of scopes that persists for the entire compilation. Others allocate scopes temporarily during parsing and type checking, discarding them afterward. Persisting scopes enables querying between functions (e.g., for module-level analysis), at the cost of memory. L1S takes a middle path: scopes are stored in a Vec and never deallocated, but only the chain from current to root is active.

What does each symbol entry store? At minimum, the entry needs a type and a declaration span. Some compilers also store mutability, visibility, alignment hints, or linker symbols. L1S keeps it minimal: a SymbolKind enum that discriminates between variables, functions, types, enum variants, specs, modules, and statics.

Ignis Implementation

Ignis implements the classic scope chain pattern. The core types are Scope, ScopeId, and SymbolTable:

src/symbol.rs — scope chain infrastructure
pub struct Scope {
    pub symbols: HashMap<String, SymbolEntry>,
    pub parent: Option<ScopeId>,
}

pub struct SymbolTable {
    scopes: Vec<Scope>,
    current: ScopeId,
}

ScopeId is a newtype over u32 that indexes into the scopes vector. The root scope (index 0) has parent: None and holds all global declarations. push_scope appends a new scope whose parent is the current scope, then updates current. pop_scope follows the parent pointer back:

pub fn push_scope(&mut self) -> ScopeId {
    let id = ScopeId(self.scopes.len() as u32);
    self.scopes.push(Scope {
        symbols: HashMap::new(),
        parent: Some(self.current),
    });
    self.current = id;
    id
}

pub fn pop_scope(&mut self) {
    if let Some(parent) = self.scopes[self.current.0 as usize].parent {
        self.current = parent;
    }
}

The critical insight: scopes are never removed from the vector. This means ScopeId values remain valid for the lifetime of the table, enabling persistent queries. But only the chain from current to root is reachable via lookup.

Lookup walks the chain:

src/symbol.rs — scope-walking lookup
pub fn lookup(&self, name: &str) -> Option<&SymbolEntry> {
    let mut sid = Some(self.current);
    while let Some(id) = sid {
        let scope = &self.scopes[id.0 as usize];
        if let Some(entry) = scope.symbols.get(name) {
            return Some(entry);
        }
        sid = scope.parent;
    }
    None
}

The loop terminates because the root scope has parent: None. This is O(d) where d is the nesting depth — typically a small constant in practice.

Two insertion paths: insert adds to the current scope (used for local variables and parameters). insert_global always adds to scope 0 (used for top-level declarations like functions and structs). The split ensures that a function body's temporary scopes don't accidentally capture global declarations.

Symbol entries carry four fields:

pub struct SymbolEntry {
    pub name: String,
    pub kind: SymbolKind,
    pub span: Span,
    pub is_pub: bool,
}

The span enables "previously defined here" error diagnostics. The kind discriminates via the SymbolKind enum: Variable(TypeId), Function(TypeId), TypeDecl(TypeId), Spec(Vec<(String, TypeId)>), EnumVariant(u64), Module(Vec<String>), and StaticVar(TypeId).

Why an enum and not traits? An enum is closed and exhaustive: the type checker can match on every variant and the compiler ensures no case is missed. A trait-based approach would be more extensible but would require dynamic dispatch or downcasting. For a fixed set of symbol kinds, an enum is simpler and faster.

Example: Tracing Nested Scope Resolution

Let's trace the symbol table state for this program:

func main(i32) {
    let a: i32 = 1;
    {
        let b: i32 = 2;
        let a: i32 = 3;  // shadows outer a
    }
}
StepActionCurrentScopes
1new()S0S0 (root, parent: None)
2insert_global("main")S0S0: {main}
3push_scope()S1S0 (root), S1 (parent: S0)
4insert("a")S1S0: {main}, S1: {a}
5push_scope()S2S0, S1, S2 (parent: S1)
6insert("b")S2S2: {b}
7insert("a")S2S2: {b, a} ← shadows S1's a
8lookup("a")S2Returns S2's a (innermost wins)
9lookup("main")S2Walks S2 → S1 → S0, finds main
10pop_scope()S1S2 still in vector, but current = S1
11lookup("b")S1Not found (b was in S2, no longer on chain)
12lookup("a")S1Returns S1's a (outer binding restored)

Notice step 10: after popping, S2 still exists in the vector. If another part of the compiler holds a ScopeId for S2, it can still inspect it via lookup_in_scope. But the active lookup chain no longer includes S2, so "b" is invisible.

What Does a SymbolEntry Hold?

Each symbol entry stores the name, its kind, the source span of its declaration, and whether it's public. The SymbolKind enum captures every category of named entity in the language:

src/symbol.rs — SymbolKind and SymbolEntry
pub enum SymbolKind {
    Variable(TypeId),              // Local variable or parameter
    Function(TypeId),              // Function (full function type)
    TypeDecl(TypeId),              // Type name (struct, enum, typedef)
    Spec(Vec<(String, TypeId)>), // Spec/trait with method signatures
    EnumVariant(u64),              // Enum variant with its discriminant
    Module(Vec<String>),          // Module path for imports
    StaticVar(TypeId),             // Global/static variable
}

pub struct SymbolEntry {
    pub name: String,
    pub kind: SymbolKind,
    pub span: Span,
    pub is_pub: bool,
}

Each variant carries precisely the data needed for that kind: variables carry a TypeId so the type checker can verify assignments and expressions; functions carry their full function type (parameters + return type) so call sites can be validated; enum variants carry their numeric discriminant so they can be used as integer constants in the IR.

The SymbolTable API

The full public API is minimal but complete:

  • new() — Creates the root scope with ScopeId(0).
  • push_scope() / pop_scope() — Enter and exit a nested scope. Returns the new ScopeId.
  • insert(name, kind, span, is_pub) — Add to the current scope. Returns Err on duplicate.
  • insert_global(name, kind, span, is_pub) — Always add to the root scope, regardless of current nesting.
  • lookup(name) — Walk the scope chain from current to root. Returns the innermost match.
  • lookup_in_scope(name, scope_id) — Check only a specific scope (no walking). Used for targeted queries.
  • current_scope() — Returns the current ScopeId for saving/restoring.

Duplicate Detection & Error Reporting

insert checks for duplicates within the same scope and returns Err(SymbolError::AlreadyDefined) with the previous declaration's span. This is important: shadowing is allowed across scopes (an inner let x hides an outer one), but duplicate names in the same scope is an error. The type checker uses this to produce precise messages like error: redefinition of 'x' pointing at both the new and previous definitions.

Debugging scope chains. When the type checker reports "undefined identifier" for something you know exists, the scope chain is likely wrong. A quick debug helper walks the chain and prints each scope's keys: 
fn dump_scopes(&self) {
    let mut sid = Some(self.current);
    while let Some(id) = sid {
        println!("scope {}: {:?}", id.0, self.scopes[id.0 as usize].symbols.keys());
        sid = self.scopes[id.0 as usize].parent;
    }
}
The most common bugs are forgetting to push_scope at function entry or pop_scope after a block.

Testing the Symbol Table

Testing a symbol table is straightforward: create a table, insert symbols, push/pop scopes, and verify lookups return the expected results. The test module in symbol.rs uses two helpers to reduce boilerplate:

fn dummy_span() -> Span { Span::new(0, 0, 1, 1) }
fn dummy_type() -> TypeId { TypeId::new(0) }

Key test categories:

Basic insert and lookup in the root scope:

#[test]
fn test_insert_and_lookup() {
    let mut st = SymbolTable::new();
    st.insert("x", SymbolKind::Variable(dummy_type()), dummy_span(), false).unwrap();
    assert_eq!(st.lookup("x").unwrap().name, "x");
}

Scope nesting: outer symbols visible from inner, but inner symbols invisible from outer after pop:

#[test]
fn test_scope_nesting() {
    let mut st = SymbolTable::new();
    st.insert("x", SymbolKind::Variable(dummy_type()), dummy_span(), false).unwrap();
    let _inner = st.push_scope();
    st.insert("y", SymbolKind::Variable(dummy_type()), dummy_span(), false).unwrap();
    assert!(st.lookup("x").is_some());   // outer visible from inner
    assert!(st.lookup("y").is_some());   // inner visible
    st.pop_scope();
    assert!(st.lookup("y").is_none());    // inner scope gone
    assert!(st.lookup("x").is_some());   // outer still visible
}

Shadowing: inner x hides outer x temporarily, outer restored on pop:

#[test]
fn test_shadowing() {
    let mut st = SymbolTable::new();
    st.insert("x", SymbolKind::Variable(dummy_type()), dummy_span(), false).unwrap();
    st.push_scope();
    st.insert("x", SymbolKind::Variable(dummy_type()), dummy_span(), false).unwrap();
    assert!(st.lookup("x").is_some());
    st.pop_scope();
    assert!(st.lookup("x").is_some());   // outer x restored
}

Duplicate detection within the same scope:

#[test]
fn test_insert_duplicate_error() {
    let mut st = SymbolTable::new();
    st.insert("x", SymbolKind::Variable(dummy_type()), dummy_span(), false).unwrap();
    match st.insert("x", SymbolKind::Variable(dummy_type()), dummy_span(), false) {
        Err(SymbolError::AlreadyDefined { name, .. }) => assert_eq!(name, "x"),
        _ => panic!("expected AlreadyDefined"),
    }
}

Global insert works from any scope depth:

#[test]
fn test_insert_global() {
    let mut st = SymbolTable::new();
    st.push_scope();
    st.insert_global("g", SymbolKind::Variable(dummy_type()), dummy_span(), true).unwrap();
    st.pop_scope();
    st.push_scope();
    assert!(st.lookup("g").is_some());  // global visible from any scope
}

Run the symbol table tests with:

$ cargo test -- symbol

Common Pitfalls

  1. Scope leak from missing push/pop. If the type checker does not push a scope in check_func, parameters leak into the enclosing scope and can shadow globals. Every block (function body, if/while body, nested braces) must create a new scope.
  2. Mutable borrow conflicts. lookup borrows &self but insert borrows &mut self. You cannot interleave them in the same iteration. The design separates reads and writes cleanly: collect phase inserts everything, check phase only reads.
  3. Using insert where insert_global is needed. If the type checker is inside a function body and calls insert on a top-level declaration, it goes into the function's scope instead of the root. Use insert_global for anything that should be visible everywhere.
Alternatives not taken. Some compilers use a persistent tree of scopes with explicit ScopeId navigation, enabling import resolution across files. L1S's flat vector approach works well for single-file compilation. Multi-file compilation would need a more sophisticated module system.
