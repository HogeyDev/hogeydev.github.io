14   Code Generation

What is Code Generation?

Code generation is the phase where the compiler's intermediate representation is translated into actual machine instructions for a target architecture. This is where the compiler moves from abstract, architecture-neutral data structures to concrete, architecture-specific assembly code. Every decision made in earlier phases — parsing, type checking, optimization — converges here into sequences of bytes that a CPU can execute.

A code generator must solve three interrelated problems:

  1. Instruction selection: Mapping each IR operation to one or more target machine instructions. An IR Add might become a single x86-64 add instruction, while a division might require a sequence involving cdq and idiv.
  2. Register allocation: Deciding which virtual registers map to physical registers versus stack slots. The allocator from Chapter 13 feeds into this.
  3. Calling convention: Following the ABI (Application Binary Interface) rules for how functions pass parameters, return values, and maintain the stack frame.
Pipeline position:

IR Module → Codegen → .asm file → nasm → .o file → cc (linker) → executable

Approaches to Code Generation

There is a broad spectrum of code generation strategies, each with different tradeoffs between implementation effort, code quality, and maintainability:

  • Hand-written AST walkers: The simplest approach. Walk the IR and emit assembly directly for each construct. Easy to implement and debug, but producing optimal code requires enormous effort. This is what L1S, early C compilers (K&R C), and tcc (Tiny C Compiler) use.
  • Template-based codegen: Pre-defined assembly templates with holes for operands. The code generator matches IR patterns to templates. GCC used this approach with RTL (Register Transfer Language) descriptions.
  • Instruction-selection DAGs: Build a Directed Acyclic Graph of target-independent operations, then tile it with target-specific machine instructions via dynamic programming. LLVM's SelectionDAG is the most prominent example. This approach can produce excellent code but is complex to implement.
  • Table-driven codegen: Machine descriptions (like .td files in LLVM) drive automatic code generation. The compiler writer describes the instruction set declaratively, and tooling generates the instruction selector.
  • Backend as a library (LLVM): Instead of writing your own code generator, pass your IR to LLVM and let it handle instruction selection, register allocation, and optimization. This is the most pragmatic choice for production compilers (Rust, Swift, Clang all use LLVM).

L1S takes the simplest possible approach: hand-written code generation where each IrInstr variant maps to a fixed pattern of NASM instructions. The register allocator from Chapter 13 provides a stack slot for every virtual register, so the code generator loads operands from memory and stores results back to memory on every instruction — maximally simple, minimally performant.

Targeting x86-64 and NASM Syntax

L1S targets x86-64 Linux using the System V AMD64 ABI. The code generator emits NASM (Netwide Assembler) syntax, which is a common choice for x86-64 assembly on Linux. Key conventions of the target:

  • Integer arguments: Passed in rdi, rsi, rdx, rcx, r8, r9 (first through sixth). Additional arguments go on the stack.
  • Return value: Placed in rax.
  • Stack alignment: The stack must be 16-byte aligned at the point of a call instruction.
  • Callee-saved registers: rbx, rbp, r12–r15 must be preserved. L1S only uses rbp as the frame pointer.

How Ignis Generates Code

Ignis's Codegen struct is straightforward: it holds a mutable String buffer for the assembly output and a reference to the type arena. The entry point, generate(), produces three NASM sections:

src/codegen.rs — Codegen struct and generate()
pub struct Codegen<'a> {
    pub output: String,
    pub types: &TypeArena,
}

impl<'a> Codegen<'a> {
    pub fn generate(&mut self, module: &IrModule) -> String {
        self.output.clear();
        self.emit("section .text");
        self.emit("global main");
        for func in &module.functions { self.generate_func(func); }
        if !module.strings.is_empty() { /* section .rodata */ }
        if !module.globals.is_empty() { /* section .data */ }
        self.output.clone()
    }
}

The three NASM sections serve distinct purposes:

  • .text: Executable code. Every function becomes a label followed by its instructions.
  • .rodata: Read-only data. String literals emitted as db directives with unique labels (str_0, str_1, ...).
  • .data: Mutable data. Global variables emitted with dd 0 (zero-initialized 4-byte values).

Function Prologue and Epilogue

Every function begins with a prologue that sets up the stack frame and ends with an epilogue that tears it down. This is the standard x86-64 calling convention pattern:

src/codegen.rs — Function code generation
fn generate_func(&mut self, func: &IrFunction) {
    let mut alloc = StackAllocator::new();
    alloc.allocate(func);

    self.emit(&format!("global {}", func.name));
    self.emit(&format!("{}:", func.name));
    self.emit("  push rbp");
    self.emit("  mov rbp, rsp");
    if alloc.stack_size != 0 {
        self.emit(&format!("  sub rsp, {}", -alloc.stack_size));
    }
    // Spill parameter registers to stack slots
    for (i, param) in func.params.iter().enumerate() {
        let slot = alloc.slot(param.vreg);
        let reg = self.param_reg(i);
        if slot != 0 {
            self.emit(&format!("  mov [rbp{}], {}", self.signed(slot), reg));
        }
    }
    // Emit basic blocks
    for block in &func.blocks { /* ... */ }
    // Shared return label (epilogue)
    self.emit(&format!(".return_{}:", func.name));
    if alloc.stack_size != 0 { /* add rsp, N */ }
    self.emit("  pop rbp");
    self.emit("  ret");
}

The prologue sequence deserves careful explanation:

  • push rbp: Save the caller's frame pointer on the stack (8 bytes).
  • mov rbp, rsp: Set our own frame pointer to the current stack pointer. All local variables and parameters are now accessed as offsets from rbp.
  • sub rsp, N: Allocate N bytes on the stack for local variables. N is the (negated) stack size from the StackAllocator, rounded to 16-byte alignment.

The epilogue reverses these steps: add rsp, N deallocates the stack frame, pop rbp restores the caller's frame pointer, and ret returns control to the caller. All return paths converge to a single shared return label rather than inlining the epilogue at every return point — a code-size tradeoff.

Instruction Selection Patterns

The heart of the code generator is the emit_instr() method, which maps each IrInstr variant to NASM instructions. The patterns fall into several categories:

Binary Operations (Add, Sub, Mul, And, Or, Xor)

These share a uniform three-step pattern via the binary() helper:

fn binary(&mut self, op: &str, dest: &VReg, a: &VReg, b: &VReg, alloc: &StackAllocator) {
    self.load_to_rax(*a, alloc);       // rax = a (load from stack)
    self.load_to_reg(*b, "rcx", alloc);   // rcx = b (load from stack)
    self.emit(&format!("  {} eax, ecx", op));  // eax = eax op ecx
    self.store_from_rax(*dest, alloc);  // dest = rax (store to stack)
}

This is a three-address-code style: both operands are loaded into registers, the operation executes, and the result is stored back. The use of rax and rcx as temporary registers is arbitrary but consistent — any caller-saved register would work.

Division and Remainder

x86-64 division uses implicit register pairs, making it more complex:

IrInstr::SDiv(dest, a, b) => {
    self.load_to_rax(*a, alloc);
    self.emit("  cdq");              // sign-extend eax → edx:eax
    self.load_to_reg(*b, "rcx", alloc);
    self.emit("  idiv ecx");          // quotient in eax, remainder in edx
    self.store_from_rax(*dest, alloc);
}

The cdq instruction sign-extends eax into the edx:eax register pair, forming the 64-bit dividend. idiv ecx divides this 64-bit value by ecx. The quotient lands in eax and the remainder in edx. For remainder operations (SRem), the code generator reads from rdx instead of rax.

Comparison Instructions

Comparisons use a two-step pattern: set CPU flags with cmp, then capture a flag with a setcc instruction:

fn cmp(&mut self, setcc: &str, dest: &VReg, a: &VReg, b: &VReg, alloc: &StackAllocator) {
    self.load_to_rax(*a, alloc);
    self.load_to_reg(*b, "rcx", alloc);
    self.emit("  cmp eax, ecx");        // set flags based on a - b
    self.emit(&format!("  {} al", setcc));   // al = 0 or 1 based on flag
    self.emit("  movzx eax, al");      // zero-extend byte to dword
    self.store_from_rax(*dest, alloc);
}

The setcc mnemonic varies by comparison: sete for equality, setne for inequality, setl for signed less-than, setg for signed greater-than, etc. The movzx eax, al zero-extends the 8-bit result to 32 bits, since setcc only writes to the low byte of a register.

Function Calls

Function calls must respect the calling convention: arguments are loaded into the SysV argument registers, then call is emitted, and the return value in rax is stored:

IrInstr::Call(dest, name, args) => {
    for (i, arg) in args.iter().enumerate() {
        let reg = self.arg_reg(i);    // rdi, rsi, rdx, rcx, r8, r9
        self.emit(&format!("  mov {}, [rbp{}]", reg, self.signed(slot)));
    }
    self.emit(&format!("  call {}", name));
    let d_slot = alloc.slot(*dest);
    if d_slot != 0 { self.emit(&format!("  mov [rbp{}], rax", self.signed(d_slot))); }
}

Note that L1S currently only handles up to 6 arguments (the register-passing limit). The 7th+ argument would need to be pushed on the stack per the SysV ABI, but L1S's fallback _ => "rdi" silently reuses rdi, which is a latent bug for functions with more than 6 parameters.

GetElementPtr (GEP)

The GEP instruction computes pointer arithmetic at runtime: base + index * stride. This is used for array indexing and struct field access:

IrInstr::Gep(dest, base, index, stride) => {
    self.load_to_reg(*base, "rax", alloc);
    self.load_to_reg(*index, "rcx", alloc);
    if *stride > 1 { self.emit(&format!("  imul rcx, rcx, {}", stride)); }
    self.emit("  add rax, rcx");
    self.store_from_rax(*dest, alloc);
}

The stride is the element size (e.g., 4 for i32, 8 for a pointer). When stride is 1 (byte addressing), the multiplication is skipped as an optimization.

The Spill-Everything Register Model

A critical design choice visible throughout the code generator is that every instruction loads operands from memory and stores results to memory. There is no attempt to keep values in registers across instructions. This "spill-everything" model has important consequences:

  • Correctness is trivial: Since every value is always in its known stack slot, there is no register allocation interference between instructions.
  • Performance is poor: Each arithmetic instruction generates 3–4 memory accesses (load a, load b, store dest), when a register-only version would need 0. This is 3–4x more memory traffic.
  • Register pressure is zero: The code generator only uses rax and rcx as temporary registers. Even with hundreds of virtual registers, the generated code never spills.
Why spill everything? For a teaching compiler, correctness and simplicity trump performance. The spill-everything model eliminates an entire class of register allocation bugs. A production compiler would use a graph-coloring register allocator (like Chaitin's algorithm) to keep frequently-used values in registers.

Code Generation for Control Flow

Basic block terminators are translated to jump instructions. The pattern is a direct mapping from IR to x86-64:

IR TerminatorNASM Output
Jump(target)jmp .func_target
Branch(cond, t, f)mov rax, [rbp+slot]; test eax, eax; jnz .func_t; jmp .func_f
Return(val)mov rax, [rbp+slot]; jmp .return_func

Branch conditions are evaluated as "non-zero is true": the condition value is loaded, test eax, eax sets the Zero Flag if it is zero, and jnz jumps to the true target if it was non-zero.

Traced Example: Code Generation Walkthrough

Consider this L1S function:

func add(i32, a: i32, b: i32) {
    return a + b;
}

The IR builder produces a function with two parameters (%0 = a, %1 = b), one Alloc for the temporary (%2), one Add instruction, and a Return terminator. The StackAllocator assigns slots: %0 → rbp-8, %1 → rbp-16, %2 → rbp-24. The generated assembly:

global add
add:
  push rbp
  mov rbp, rsp
  sub rsp, 24
  mov [rbp-8], rdi          ; spill %0 (a) from rdi
  mov [rbp-16], rsi         ; spill %1 (b) from rsi
.add_0:
  mov rax, [rbp-8]          ; load %0 (a)
  mov rcx, [rbp-16]         ; load %1 (b)
  add eax, ecx              ; %2 = a + b
  mov [rbp-24], rax         ; spill %2
  mov rax, [rbp-24]         ; load return value
  jmp .return_add
.return_add:
  add rsp, 24
  pop rbp
  ret

Notice the redundant load of %2 immediately after storing it — the result of add is in rax, but the spill-everything model stores it and then immediately reloads it for the return. An optimizing code generator would detect this and keep the value in rax.

Testing Code Generation

The Codegen struct itself has no dedicated unit tests; correctness is verified through end-to-end integration tests that compile L1S programs, assemble and link them, and run the resulting executables. The StackAllocator (on which codegen depends) has unit tests in regalloc.rs that verify slot assignment and stack size computation.

To run all tests including codegen:

$ cargo test

A minimal end-to-end test:

$ echo "func main(i32) { return 42; }" > /tmp/test.is
$ cargo run -- /tmp/test.is
$ /tmp/test; echo $?
42

Common Pitfalls

  1. Stack offset sign errors: Stack slots are negative offsets from rbp. A positive offset would corrupt the caller's frame. The signed() helper is critical.
  2. Missing size specifiers: NASM requires explicit size specifiers like qword when writing immediates to memory: mov qword [rbp-8], 42, not mov [rbp-8], 42.
  3. Caller-saved register clobbers: The code generator does not track which registers are live across calls. Since every value is on the stack, this is safe, but a register-allocating code generator would need to save and restore caller-saved registers around calls.
  4. Division by zero: x86-64 idiv raises SIGFPE on division by zero. L1S does not check for this.
  5. 6-argument limit: Beyond 6 arguments, the SysV ABI requires stack-based passing, but L1S silently reuses rdi.
