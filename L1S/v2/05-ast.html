5   The Abstract Syntax Tree

What Is an Abstract Syntax Tree?

After the lexer converts source text into a flat stream of tokens, the parser's job is to impose structure on that stream. The result is an Abstract Syntax Tree (AST) — a tree data structure where each node represents a grammatical construct in the source program.

The word "abstract" is critical: the AST discards purely syntactic details that are not needed for later compiler stages. Semicolons, parentheses (when used for grouping rather than function calls), braces, and whitespace all vanish in the AST. What remains is the essential structure: declarations contain statements, statements contain expressions, expressions contain sub-expressions.

Source: return (2 + 3) * 4;

Token stream (flat):
Return OpenParen NumLiteral("2") Plus NumLiteral("3") CloseParen Star NumLiteral("4") Semicolon

AST (tree):
Return
  Binary(Mul)
    Binary(Add)
      Int("2")
      Int("3")
    Int("4")

The parentheses, semicolon, and the flat token ordering are gone. What remains is a tree that directly reflects the grammatical structure: a return statement containing a multiplication expression containing an addition expression.

Tradeoffs & Design Space

AST vs. Concrete Syntax Tree (CST)

A Concrete Syntax Tree (also called a parse tree) preserves every detail from the source text — every keyword, every parenthesis, every semicolon gets a node. An AST is a lossy compression of the CST. The tradeoff:

  • CST: Easier to generate (just mirror the grammar rules), and can reproduce exact source text from the tree. But CSTs are large, verbose, and force later stages to navigate through irrelevant syntactic noise.
  • AST: Smaller, more semantic, and easier for analysis passes (type checking, IR generation) to traverse. The parser works harder, but every downstream pass is simpler.

Ignis uses a pure AST — no syntax nodes survive into the tree. The parser discards grouping parens, delimiters, and separators as it builds the tree.

Sum Types vs. Trait Hierarchy

In typed functional languages (Rust, Haskell, OCaml), AST nodes are naturally represented as sum types (enums). Each category of node — expression, statement, declaration — is one enum with variants for each kind. In OOP languages (Java, C++), ASTs typically use an abstract base class with virtual methods (the visitor pattern).

  • Sum types: Exhaustive matching is checked by the compiler. If you add a new expression variant, every match on expressions must handle it (or explicitly use a wildcard). This is safer and more maintainable.
  • Visitor pattern: More extensible (you can add new operations without modifying existing classes) but lacks exhaustiveness guarantees.

Ignis uses Rust enums for all AST node categories.

Span on Every Node

Every AST node must carry its source location so that errors in later stages (type checking, code generation) can point back to the relevant source code. The design options:

  • Span field on every variant: Every variant of an enum carries its own Span. This is verbose but explicit.
  • Common struct wrapper: struct SpannedNode { node: AstExpr, span: Span }. This factors out the span but adds an indirection layer.
  • Span on a subset of nodes: Only nodes that can produce errors carry spans. This saves memory but risks missing error locations.

Ignis puts a Span field in every variant of every AST enum. Every expression, every statement, every type node knows its source location.

Box vs. Arena Allocation

AST nodes reference each other recursively — an expression can contain sub-expressions. In Rust, this requires indirection:

  • Box<AstExpr>: Heap-allocated, simple, each node is independently owned. This is what Ignis uses.
  • Arena allocation: All nodes are allocated in a contiguous arena with &'arena references between them. More memory-efficient and cache-friendly, but requires a lifetime parameter on the entire AST and makes mutation harder.

Ignis uses Box for recursive nodes — simple and correct, at the cost of individual heap allocations per node.

Ignis Implementation: The AST Hierarchy

The AST is defined in src/ast.rs. It is organized into four levels: types, expressions, statements, and declarations. A program is a vector of top-level declarations.

Level 1: Types (AstType)

A type in Ignis is either a primitive, a named type, or a compound type built from other types:

src/ast.rs — BinaryOp, PrimType, AstType
pub enum BinaryOp {
    Add, Sub, Mul, Div, Mod,
    And, Or, Xor,
    Shl, Shr,
    Eq, Ne, Lt, Gt, Le, Ge,
    LogicAnd, LogicOr,
}

pub enum PrimType {
    Void, Bool, Char, Usize, Isize,
    U64, I64, U32, I32, U16, I16, U8, I8,
}

pub enum AstType {
    Prim(PrimType, Span),
    Named(String, Span),
    Ptr(Box<AstType>, Span),
    Array(Box<AstExpr>, Box<AstType>, Span),
    Func(Vec<AstType>, Box<AstType>, Span),
    Spec(String, Span),
}

BinaryOp and PrimType are standalone enums shared across the AST. AstType represents type annotations in the tree:

  • Prim: A built-in type like i32 or bool. The token Token::I32 becomes AstType::Prim(PrimType::I32, ...).
  • Named: A user-defined type referenced by name, like MyStruct. The parser cannot resolve names — that happens during type checking.
  • Ptr: A pointer type like *i32. Boxed because it wraps another type recursively.
  • Array: A fixed-size array type like [42]i32. Contains an expression for the size and a type for the element type.
  • Func: A function type like (i32, bool) -> void. Contains parameter types and a return type.
  • Spec: A trait/spec type reference, used in impl declarations.

The span() method extracts the span from any type variant:

impl AstType {
    pub fn span(&self) -> Span {
        match self {
            AstType::Prim(_, s)
            | AstType::Named(_, s)
            | AstType::Ptr(_, s)
            | AstType::Spec(_, s) => *s,
            AstType::Array(_, _, s)
            | AstType::Func(_, _, s) => *s,
        }
    }
}

This method is used throughout the compiler whenever an error needs to point to a type annotation.

Level 2: Expressions (AstExpr)

Expressions are the heart of the AST — they represent computations that produce values. Ignis defines 17 expression variants:

src/ast.rs — AstExpr
pub enum AstExpr {
    Int(String, Span),
    Str(String, Span),
    Char(u8, Span),
    Bool(bool, Span),
    Ident(String, Span),
    Deref(Box<AstExpr>, Span),
    AddrOf(Box<AstExpr>, Span),
    Neg(Box<AstExpr>, Span),
    Not(Box<AstExpr>, Span),
    BitNot(Box<AstExpr>, Span),
    Binary(BinaryOp, Box<AstExpr>, Box<AstExpr>, Span),
    Assign(Box<AstExpr>, Box<AstExpr>, Span),
    Call(Box<AstExpr>, Vec<AstExpr>, Span),
    Index(Box<AstExpr>, Box<AstExpr>, Span),
    StructInit(Box<AstExpr>, Vec<AstMemberInit>, Span),
    Field(Box<AstExpr>, String, Span),
    Arrow(Box<AstExpr>, String, Span),
    Cast(Box<AstType>, Box<AstExpr>, Span),
    Paren(Box<AstExpr>, Span),
}

Let's examine the categories:

Literals (Int, Str, Char, Bool): These are leaf nodes — they have no children. The integer literal stores its raw string (not a parsed number) for the same reason the token does: parsing the numeric value is deferred until it's needed. The boolean literal is resolved from true/false keywords into an actual bool.

Identifiers (Ident): References to variables, functions, or types by name. Name resolution happens during type checking, not parsing.

Unary operators (Deref, AddrOf, Neg, Not, BitNot): Each unary operator gets its own variant rather than a generic Unary(UnaryOp, ...). This makes pattern matching in later passes more explicit.

Binary operators (Binary): A single variant covers all binary operations, parameterized by the BinaryOp enum. This is more compact than one variant per operator and makes it easy to iterate over all binary expressions. The operator semantics (associativity, precedence) are handled by the parser and do not appear in the AST.

Compound expressions (Call, Index, StructInit, Field, Arrow): These represent operations with multiple sub-expressions. A function call has a callee and argument list. A struct initializer has a type name and member initializers. Field access has a base expression and a field name.

Type operations (Cast): A cast expression like cast(i32, x) carries both the target type and the source expression.

Grouping (Paren): Parenthesized expressions are preserved in the AST. This is intentional: while (2 + 3) and 2 + 3 produce the same tree after evaluation, preserving parentheses allows the compiler to produce better error messages (“unnecessary parentheses” warnings, for example).

Level 3: Statements (AstStmt)

Statements are the executable units of a program. Unlike expressions, statements do not produce values:

src/ast.rs — AstStmt and supporting structs
pub enum AstStmt {
    VarDecl(AstVarDecl),
    If(AstIf),
    While(AstWhile),
    For(AstFor),
    Return(AstReturn),
    Asm(AstAsm),
    Block(AstBlock),
    Expr(AstExpr),
    Import(AstImport),
}

Each statement variant wraps a dedicated struct rather than inlining fields directly into the enum. This keeps variant payloads named and makes the code self-documenting:

pub struct AstVarDecl {
    pub span: Span,
    pub name: String,
    pub type_: Option<AstType>,
    pub init: Option<AstExpr>,
}

pub struct AstIf {
    pub span: Span,
    pub cond: AstExpr,
    pub then_block: AstBlock,
    pub else_branch: Option<Box<AstStmt>>,
}

pub struct AstBlock {
    pub span: Span,
    pub stmts: Vec<AstStmt>,
}

Key design decisions in the statement hierarchy:

  • Variable declarations (AstVarDecl) have an optional initializer in the AST struct (Option), but the type annotation is required by the parser — Ignis does not support type inference. let x: i32 = 42 (both) and let x: i32 (no init) are the valid forms; let x = 42 without a colon is a parse error.
  • If statements have an optional else branch stored as Option<Box<AstStmt>>. The box is necessary because AstStmt is a recursive type. The else branch can itself be another If (producing an else if chain) or any other statement type.
  • Expression statements (AstStmt::Expr) wrap an expression in a statement. This is how function calls used as statements (foo();) are represented — the expression Call is wrapped in an Expr statement.
  • Blocks are first-class statement nodes. A block has its own span and contains a vector of statements. Blocks appear in function bodies, if/while/for bodies, and can be nested arbitrarily.

Level 4: Declarations (AstDecl)

At the top level of a program, declarations introduce named entities — functions, structs, enums, traits, implementations, type aliases, static variables, and imports:

src/ast.rs — AstDecl and declaration structs
pub enum AstDecl {
    Struct(AstStructDecl),
    Enum(AstEnumDecl),
    Spec(AstSpecDecl),
    Impl(AstImplDecl),
    Func(AstFuncDecl),
    Typedef(AstTypedefDecl),
    Import(AstImport),
    StaticVar(AstStaticVarDecl),
}

pub struct AstFuncDecl {
    pub span: Span,
    pub pub_: bool,       // visibility: pub or private
    pub name: String,
    pub return_type: AstType,
    pub params: Vec<AstParam>,
    pub body: AstBlock,
}

pub struct AstStructDecl {
    pub span: Span,
    pub pub_: bool,
    pub name: String,
    pub fields: Vec<AstField>,
}

Notable design points:

  • Visibility (pub_: bool): Each declaration carries its visibility. The trailing underscore on pub_ is the Rust convention for avoiding the reserved keyword pub.
  • Function declarations always have a body block, even if empty. Function signatures (for traits) use AstFuncSig instead, which has no body.
  • Enum variants are simple strings with optional explicit values (variant_values: Option<Vec<u64>>). The noprefix flag controls whether variant names are prefixed with the enum name in code generation.
  • Import paths are represented as Vec<String> — e.g., import std.io becomes AstImport { path: ["std", "io"] }.

The Program Root

pub struct AstProgram {
    pub decls: Vec<AstDecl>,
}

A program is just a list of declarations. The order matters — later compiler stages process declarations in order, top-to-bottom. There is no explicit module node or file node; the program struct is the root.

Why no root span on AstProgram? The program's span would be the entire source file, which is not particularly useful for error reporting. Errors are always reported at specific declarations, statements, or expressions inside the program. The root struct exists only to hold the declaration list.

Example: Building the AST for a Function

Consider this Ignis program:

func add(i32, a: i32, b: i32) {
    return a + b;
}

The parser produces this AST:

AstProgram {
    decls: [
        AstDecl::Func(AstFuncDecl {
            name: "add",
            pub_: false,
            return_type: AstType::Prim(PrimType::I32, ...),
            params: [
                AstParam { name: "a", type_: AstType::Prim(PrimType::I32, ...) },
                AstParam { name: "b", type_: AstType::Prim(PrimType::I32, ...) },
            ],
            body: AstBlock {
                stmts: [
                    AstStmt::Return(AstReturn {
                        value: AstExpr::Binary(
                            BinaryOp::Add,
                            AstExpr::Ident("a", ...),
                            AstExpr::Ident("b", ...),
                            ...
                        ),
                    }),
                ],
            },
        }),
    ],
}

Notice what the AST has done:

  • Discarded: The func keyword, the parentheses around parameters, the comma between parameters, the braces around the body, the semicolon after the return statement, the whitespace.
  • Resolved: i32 tokens became AstType::Prim(PrimType::I32). The + operator became BinaryOp::Add. The return keyword became an AstStmt::Return wrapper.
  • Structured: The flat token stream Func Ident("add") OpenParen I32 Ident("a") ... became a tree with explicit parent-child relationships.

Pattern: The span() Method Across Node Types

Every AST node type that can appear in an error message implements a span() method. The pattern is consistent:

impl AstExpr {
    pub fn span(&self) -> Span {
        match self {
            AstExpr::Int(_, s)
            | AstExpr::Str(_, s)
            // ... leaf nodes
            | AstExpr::Paren(_, s) => *s,
            AstExpr::Binary(_, _, _, s)
            | AstExpr::Call(_, _, s)
            // ... compound nodes
            | AstExpr::Cast(_, _, s) => *s,
        }
    }
}

The span is always the last field in every variant. This is a convention, not a Rust requirement, but it makes the pattern easy to follow: every variant's span is in the same position.

Testing the AST Structure

The AST module does not have its own test module — the AST types are tested implicitly through the parser and downstream passes. However, the design ensures correctness at the type level:

  • Recursive types use Box, which prevents unbounded stack allocation.
  • Optional fields use Option, making missing data explicit at the type level rather than using sentinel values.
  • Enum variants with the wrong field types are caught at compile time — you cannot accidentally put an expression where a statement is expected.

Key Takeaways

  1. The AST is a semantic tree — it discards syntactic sugar (semicolons, grouping parens, braces) and preserves only the grammatical structure needed for analysis and code generation.
  2. Every node carries its span via a span() method. No AST node is ever created without source location information.
  3. Sum types (enums) provide exhaustiveness checking. Adding a new expression variant forces the compiler to check every match on expressions.
  4. Boxed recursion (Box<AstExpr>) enables tree-shaped data with deterministic ownership and no garbage collection.
  5. The hierarchy is four levels deep: types → expressions → statements → declarations. Each level builds on the previous, and the root is a flat list of declarations.
