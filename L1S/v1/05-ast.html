5   The Abstract Syntax Tree

What is an AST?

An Abstract Syntax Tree (AST) is a tree representation of the syntactic structure of source code. Where the lexer produces a flat sequence of tokens, the parser arranges those tokens into a tree that captures the grammatical structure and nesting of the program.

The word "abstract" is key: the AST omits details that are present in the concrete syntax (parens, semicolons, braces) and preserves only what is semantically meaningful. For example, the parser discards the opening and closing braces of a block — the AstBlock struct stores only the statements inside.

Where Span Lives in Every Node

Every AST node carries a Span field. This is the most consistent pattern in the entire design:

  • Enums like AstType, AstExpr store the span alongside other data in each variant.
  • Structs like AstFuncDecl, AstBlock, AstParam have a dedicated pub span: Span field.
  • The span() accessor method on AstType and AstExpr extracts the span regardless of which variant is active.
Why thread Span through every node? Because error messages need to point to specific locations. When the type checker says "type mismatch in return statement," it needs to point to the exact expression. Without Span on every node, the best you can do is "error somewhere on line 5." With Span, every sub-expression carries its source location.

Supporting Types

BinaryOp

src/ast.rs — BinaryOp
pub enum BinaryOp {
    // Arithmetic
    Add, Sub, Mul, Div, Mod,
    // Bitwise
    And, Or, Xor,
    // Shift
    Shl, Shr,
    // Comparison
    Eq, Ne, Lt, Gt, Le, Ge,
    // Logical
    LogicAnd, LogicOr,
}

This enum exists because AstExpr::Binary needs to distinguish a + b from a == b without having 17 separate binary expression variants. The variants map 1:1 to token operators, except that = (assignment) is AstExpr::Assign, not a BinaryOp.

PrimType

pub enum PrimType {
    Void, Bool, Char, Usize, Isize,
    U64, I64, U32, I32, U16, I16, U8, I8,
}

This mirrors the primitive type tokens from Token but at the AST level. It's separated from AstType because AstType::Prim(PrimType, Span) encapsulates the idea of "a type that is a primitive" as a single variant, rather than having 13 variants like AstType::I32(Span).

AstType — The Type System Nodes

Types in L1S can be primitives, named (user-defined), pointers, arrays, function signatures, or spec qualifications:

src/ast.rs — AstType
pub enum AstType {
    Prim(PrimType, Span),
    Named(String, Span),
    Ptr(Box<AstType>, Span),
    Array(Box<AstExpr>, Box<AstType>, Span),
    Func(Vec<AstType>, Box<AstType>, Span),
    Spec(String, Span),
}
VariantSource ExampleMeaning
Prim(PrimType, Span)i32, boolBuilt-in primitive type
Named(String, Span)MyStruct, FooUser-defined type by name
Ptr(Box<AstType>, Span)*i32, *MyStructPointer to another type
Array(AstExpr, AstType, Span)[42]i32Array with size expression and element type
Func(Vec<AstType>, AstType, Span)(i32, bool) -> voidFunction type (params, return)
Spec(String, Span)spec MyTraitSpec/trait constraint reference
Why Box for recursive types? AstType::Ptr contains an AstType, which is a recursive definition. Without Box, Rust would not know the size of the enum at compile time (infinite recursion). Box puts the inner type on the heap and stores a pointer in the enum, making the enum's size finite. The same pattern applies to AstExpr variants that contain other AstExpr values.

The span() accessor extracts the span from any variant:

impl AstType {
    pub fn span(&self) -> Span {
        match self {
            AstType::Prim(_, s)
            | AstType::Named(_, s)
            | AstType::Ptr(_, s)
            | AstType::Spec(_, s) => *s,
            AstType::Array(_, _, s)
            | AstType::Func(_, _, s) => *s,
        }
    }
}

The span is always the last field in every variant, making it easy to extract via a single match.

AstExpr — The Expression Nodes (28 variants)

Expressions are the richest part of the AST. Every expression variant holds its value data and a Span:

src/ast.rs — AstExpr
pub enum AstExpr {
    /// Literals
    Int(String, Span),
    Str(String, Span),
    Char(u8, Span),
    Bool(bool, Span),

    /// Named reference
    Ident(String, Span),

    /// Unary operators
    Deref(Box<AstExpr>, Span),
    AddrOf(Box<AstExpr>, Span),
    Neg(Box<AstExpr>, Span),
    Not(Box<AstExpr>, Span),
    BitNot(Box<AstExpr>, Span),

    /// Binary operations
    Binary(BinaryOp, Box<AstExpr>, Box<AstExpr>, Span),

    /// Assignment
    Assign(Box<AstExpr>, Box<AstExpr>, Span),

    /// Calls and indexing
    Call(Box<AstExpr>, Vec<AstExpr>, Span),
    Index(Box<AstExpr>, Box<AstExpr>, Span),

    /// Struct operations
    StructInit(Box<AstExpr>, Vec<AstMemberInit>, Span),
    Field(Box<AstExpr>, String, Span),
    Arrow(Box<AstExpr>, String, Span),

    /// Cast and grouping
    Cast(Box<AstType>, Box<AstExpr>, Span),
    Paren(Box<AstExpr>, Span),
}

The parsing rule of thumb: each token or token pattern that represents an expression maps to exactly one AstExpr variant:

Why 13 expression variants instead of fewer, more general ones? Each variant corresponds to a distinct syntactic form that the parser can recognize immediately. A more general representation (e.g., a single Unary(UnaryOp, Box<AstExpr>) for all unary operators) would work but require an additional enum and make pattern matching in the type checker less readable. The tradeoff is more variants but clearer code at the usage sites.

AstMemberInit

This small struct supports struct initialization syntax like MyStruct{.x = 1, .y = 2}:

pub struct AstMemberInit {
    pub span: Span,
    pub name: String,
    pub value: AstExpr,
}

The name identifies which field to initialize, and value is the expression assigned to it. The span covers the entire .name = value range.

AstStmt — The Statement Types

Statements are the executable units of a program. They do not produce values:

src/ast.rs — AstStmt
pub enum AstStmt {
    VarDecl(AstVarDecl),
    If(AstIf),
    While(AstWhile),
    For(AstFor),
    Return(AstReturn),
    Asm(AstAsm),
    Block(AstBlock),
    Expr(AstExpr),
    Import(AstImport),
}

Each variant wraps a dedicated struct. Let's examine each one:

AstBlock

pub struct AstBlock {
    pub span: Span,
    pub stmts: Vec<AstStmt>,
}

A block is simply a list of statements wrapped in { }. The span covers from the opening brace to the closing brace. The braces themselves are not stored — they are syntactic decoration that the parser strips away.

AstVarDecl

pub struct AstVarDecl {
    pub span: Span,
    pub name: String,
    pub type_: Option<AstType>,
    pub init: Option<AstExpr>,
}

Both type_ and init are Option in the struct definition — the type is optional at the AST level to support potential extensions. In practice, the Ignis parser always requires a colon and explicit type annotation, so type_ is never None in a valid parse. The init may be omitted (leaving a variable uninitialized, which the type checker rejects). Two valid declaration forms exist:

  • let x: i32 = 42; → name = "x", type_ = Some(I32), init = Some(42)
  • let x: i32; → name = "x", type_ = Some(I32), init = None (error at runtime if used uninitialized)

Ignis does not support type inference — let x = 42; without a colon produces a parse error.

AstIf

pub struct AstIf {
    pub span: Span,
    pub cond: AstExpr,
    pub then_block: AstBlock,
    pub else_branch: Option<Box<AstStmt>>,
}

The else_branch is Option<Box<AstStmt>> because else is optional. When present, it can be either another AstStmt::If (for else if) or an AstStmt::Block (for else { }). Boxing the AstStmt avoids infinite size recursion (an AstIf can contain another AstIf via the else branch).

AstWhile

pub struct AstWhile {
    pub span: Span,
    pub cond: AstExpr,
    pub body: AstBlock,
}

Simple: loop while cond is true.

AstFor

pub struct AstFor {
    pub span: Span,
    pub init: AstExpr,
    pub cond: AstExpr,
    pub update: AstExpr,
    pub body: AstBlock,
}

All three parts of the for loop (init, cond, update) are required to be present in the AST. The parser fills in defaults (e.g., true for omitted condition).

AstReturn

pub struct AstReturn {
    pub span: Span,
    pub value: AstExpr,
}

Note: the return value is not optional in this struct. Even return; in a void function produces AstExpr::Ident("void", ...) or is handled by the parser inserting a void expression. The return statement always has a value expression in the AST.

AstAsm

pub struct AstAsm {
    pub span: Span,
    pub template: String,
}

Inline assembly template strings are stored as-is. The assembler backend handles parsing the template.

AstImport (statement form)

pub struct AstImport {
    pub span: Span,
    pub path: Vec<String>,
}

A path like import std.io produces path = ["std", "io"]. The Vec representation makes it easy to walk the module hierarchy.

Note that AstImport appears in two places: as AstDecl::Import (top-level) and AstStmt::Import (inside functions). The struct is shared between both.

AstDecl — The Declaration Types (7 variants)

Declarations are top-level constructs that introduce named entities into the program:

src/ast.rs — AstDecl
pub enum AstDecl {
    Struct(AstStructDecl),
    Enum(AstEnumDecl),
    Spec(AstSpecDecl),
    Impl(AstImplDecl),
    Func(AstFuncDecl),
    Typedef(AstTypedefDecl),
    Import(AstImport),
    StaticVar(AstStaticVarDecl),
}

Every declaration variant has a pub_ field (visibility), a name (the declared name), and declaration-specific fields:

AstStructDecl

pub struct AstStructDecl {
    pub span: Span,
    pub pub_: bool,
    pub name: String,
    pub fields: Vec<AstField>,
}

Each field is:

pub struct AstField {
    pub span: Span,
    pub name: String,
    pub type_: AstType,
}

AstEnumDecl

pub struct AstEnumDecl {
    pub span: Span,
    pub pub_: bool,
    pub name: String,
    pub noprefix: bool,
    pub variants: Vec<String>,
    pub variant_values: Option<Vec<u64>>,
}

The noprefix flag corresponds to the noprefix attribute on enums, which means variants are referenced without the enum name prefix. The variant_values are optional explicit discriminant values.

AstSpecDecl and AstImplDecl

pub struct AstSpecDecl {
    pub span: Span,
    pub pub_: bool,
    pub name: String,
    pub items: Vec<AstSpecItem>,
}

pub enum AstSpecItem {
    Func(AstFuncSig),
    Typedef(AstTypedefDecl),
}

pub struct AstImplDecl {
    pub span: Span,
    pub impl_type: AstType,
    pub spec_name: String,
    pub methods: Vec<AstFuncDecl>,
}

AstSpecDecl defines a trait: a named collection of function signatures and typedefs. AstImplDecl implements a spec for a specific type, providing concrete method bodies. The spec_name field names which spec is being implemented.

AstFuncDecl

pub struct AstFuncDecl {
    pub span: Span,
    pub pub_: bool,
    pub name: String,
    pub return_type: AstType,
    pub params: Vec<AstParam>,
    pub body: AstBlock,
}

Parameters are modeled as AstParam:

pub struct AstParam {
    pub span: Span,
    pub name: String,
    pub type_: AstType,
}

AstTypedefDecl

pub struct AstTypedefDecl {
    pub span: Span,
    pub pub_: bool,
    pub name: String,
    pub type_: AstType,
}

Creates an alias: typedef MyInt = i32;.

AstStaticVarDecl

pub struct AstStaticVarDecl {
    pub span: Span,
    pub name: String,
    pub type_: AstType,
    pub init: Option<AstExpr>,
}

AstProgram — The Root

The entire program is a list of declarations:

pub struct AstProgram {
    pub decls: Vec<AstDecl>,
}

That's it. A program is nothing more than an ordered sequence of top-level declarations. The parser produces this, and the semantic analyzer walks it.

Note the implicit return type: When a function declaration omits the return type (as in func main(i32) { ... }), the parser inserts Void. This is a parser-level decision that happens before the AST is built, so the AST always has an explicit return type.

Testing the AST

The ast.rs file defines the AST types but does not have its own test module — the AST types are tested indirectly through parser tests. However, the types themselves are designed to be testable:

// Example: manually constructing an AST node is straightforward:
let span = Span::new(0, 2, 1, 1);
let expr = AstExpr::Int("42".into(), span);
assert_eq!(expr.span(), span);

// Matching on the expression:
match expr {
    AstExpr::Int(val, _) => assert_eq!(val, "42"),
    _ => panic!("expected Int"),
}

// Constructing a full function declaration:
let func = AstFuncDecl {
    span: Span::new(0, 20, 1, 1),
    pub_: false,
    name: "main".into(),
    return_type: AstType::Prim(PrimType::Void, Span::new(0, 0, 1, 1)),
    params: vec![],
    body: AstBlock {
        span: Span::new(10, 20, 1, 11),
        stmts: vec![],
    },
};

To run parser tests (which exercise the AST construction):

$ cargo test -- parser

Design Choices Explained

Why separate PrimType from AstType?

Without PrimType, AstType would need 13 separate variants just for primitives:

// Without PrimType (worse):
enum AstType { Void, Bool, I32, ... }

With PrimType, we have a single AstType::Prim(PrimType, Span) variant. This collapses 13 into 1. Any code that handles all primitives can match on AstType::Prim(_, _) rather than listing all 13 variants.

Why AstExpr::Cast rather than a function call?

The cast keyword is a built-in language construct, not a library function. It gets its own AST node so the type checker can verify the cast is valid (e.g., you can't cast a pointer to bool). If cast were modeled as a regular function call, the type checker would need special-casing anyway.

Why does AstField store type_: AstType rather than String?

At parse time, the field type is resolved to an AstType. A field declared as x: i32 produces AstType::Prim(PrimType::I32, ...). A field declared as y: MyStruct produces AstType::Named("MyStruct", ...). The semantic analyzer later resolves Named to the actual struct definition.

Why AstStmt::Expr(AstExpr)?

Expression-statements allow expressions to be used as statements. This is what makes f(); and a = b + c; valid statements. The expression is evaluated for its side effects; its result is discarded.

Common Pitfalls & Debugging Tips

  1. Missing Span on a new node: Every new AST node must carry a Span. If you add a new variant to AstExpr without a Span field, the span() accessor will fail to compile. Add it to both the variant and the match arms.
  2. Box placement: When adding a recursive variant (e.g., a new unary expression), wrap the inner AstExpr in Box. Without it, Rust will error with "recursive type has infinite size."
  3. Cloning the AST: If you need to clone an AstExpr tree, every node must implement Clone. The #[derive(Clone)] on all types ensures this, but if you add a field that is not Clone, the derives will fail.
  4. Vec vs. Box: Use Vec for variable-length lists (function params, struct fields, block statements). Use Box for single recursive children (loop body, unary operand).
  5. Span consistency: The span should cover the entire syntactic construct from the first token to the last. For AstExpr::Binary, the span covers from left's start to right's end. For AstFuncDecl, from func to the closing }. Off-by-one span errors manifest as confusing error messages downstream.
Debugging tip: Implement a Display or debug-print function for your AST to visualize parse results. A simple recursive function that prints the tree with indentation is invaluable for verifying that the parser produced the correct structure. Something like println!("{:#?}", ast_program) with Rust's debug formatting can get you surprisingly far.
