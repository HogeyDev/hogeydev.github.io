10   Intermediate Representation

Why an Intermediate Representation?

A compiler could translate AST directly to assembly. But that conflates two concerns: what the program does and how the machine does it. An Intermediate Representation (IR) sits between the AST and codegen, providing:

  1. Machine independence: The same IR can be compiled to x86-64, ARM, or any backend.
  2. Clear semantics: Every IR instruction has a single, well-defined meaning, unlike AST nodes which bundle multiple concerns.
  3. Optimization: Transformations like constant folding and dead code elimination are easier on a flat instruction list than on a tree.
  4. SSA form: The IR uses Static Single Assignment (via phis), which makes data-flow analysis straightforward.

L1S uses a basic-block IR similar to LLVM's: a function is a graph of basic blocks, each containing a sequence of instructions terminated by a jump, branch, or return.

Pipeline position of IR:

AST → Type Checker → IR Builder → IR Opt → Codegen

Core Types: VReg, IrBlockId, StringId

The IR uses three index-like handle types, each wrapping a u32:

src/ir.rs — Handle types
pub struct VReg(pub u32);       // Virtual register: %0, %1, %2, ...
pub struct IrBlockId(pub u32);  // Basic block: bb0, bb1, bb2, ...
pub struct StringId(pub u32);    // String pool index

Each has a Display implementation for readable IR dump output:

// VReg displays as %0, %1, ...
impl fmt::Display for VReg {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "%{}", self.0)   // e.g., %3
    }
}

// IrBlockId displays as bb0, bb1, ...
impl fmt::Display for IrBlockId {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "bb{}", self.0)  // e.g., bb1
    }
}

Module, Function, and Block Structure

The top-level container is IrModule, holding functions, globals, and a string pool:

pub struct IrModule {
    pub name: String,
    pub functions: Vec<IrFunction>,
    pub globals: Vec<IrGlobal>,
    pub strings: Vec<IrString>,
}

An IrFunction contains a vector of basic blocks, with an explicit entry block pointer:

pub struct IrFunction {
    pub name: String,
    pub return_type: TypeId,
    pub params: Vec<IrParam>,
    pub blocks: Vec<IrBlock>,
    pub num_vregs: u32,
    pub entry_block: IrBlockId,
}

Each parameter is assigned a virtual register:

pub struct IrParam {
    pub name: String,
    pub type_: TypeId,
    pub vreg: VReg,
}

An IrBlock is the core unit: a sequence of instructions, optionally with phi nodes, ending with a terminator:

pub struct IrBlock {
    pub id: IrBlockId,
    pub name: String,
    pub phis: Vec<IrPhi>,
    pub instrs: Vec<IrInstr>,
    pub terminator: IrTerminator,
}

IrInstr — Every Instruction Variant

There are 36 instruction variants, organized into five categories. The first operand is always the destination virtual register (except for Store and MemCopy):

src/ir.rs — Full IrInstr enum
pub enum IrInstr {
    // === Memory ===
    Alloc(VReg, TypeId),       // Allocate stack slot
    Load(VReg, VReg),          // Load value from pointer
    Store(VReg, VReg),         // Store value to pointer
    MovImm(VReg, u64, TypeId), // Move immediate constant into register

    // === Arithmetic (3-address, dst = lhs op rhs) ===
    Add(VReg, VReg, VReg),    // dst = lhs + rhs
    Sub(VReg, VReg, VReg),    // dst = lhs - rhs
    Mul(VReg, VReg, VReg),    // dst = lhs * rhs
    SDiv(VReg, VReg, VReg),   // dst = lhs / rhs (signed)
    UDiv(VReg, VReg, VReg),   // dst = lhs / rhs (unsigned)
    SRem(VReg, VReg, VReg),   // dst = lhs % rhs (signed)
    URem(VReg, VReg, VReg),   // dst = lhs % rhs (unsigned)
    And(VReg, VReg, VReg),     // dst = lhs & rhs (bitwise)
    Or(VReg, VReg, VReg),      // dst = lhs | rhs (bitwise)
    Xor(VReg, VReg, VReg),     // dst = lhs ^ rhs (bitwise)
    Shl(VReg, VReg, VReg),     // dst = lhs << rhs
    Shr(VReg, VReg, VReg),     // dst = lhs >> rhs (arithmetic/signed)
    Ushr(VReg, VReg, VReg),    // dst = lhs >> rhs (logical/unsigned)

    // === Unary ===
    Neg(VReg, VReg),           // dst = -src (arithmetic negation)
    Not(VReg, VReg),           // dst = !src (logical not, bool only)
    BitNot(VReg, VReg),        // dst = ~src (bitwise not)

    // === Comparison (all produce a bool) ===
    Eq(VReg, VReg, VReg),      // dst = lhs == rhs
    Ne(VReg, VReg, VReg),      // dst = lhs != rhs
    Slt(VReg, VReg, VReg),     // dst = lhs < rhs (signed)
    Sgt(VReg, VReg, VReg),     // dst = lhs > rhs (signed)
    Sle(VReg, VReg, VReg),     // dst = lhs <= rhs (signed)
    Sge(VReg, VReg, VReg),     // dst = lhs >= rhs (signed)
    Ult(VReg, VReg, VReg),     // dst = lhs < rhs (unsigned)
    Ugt(VReg, VReg, VReg),     // dst = lhs > rhs (unsigned)
    Ule(VReg, VReg, VReg),     // dst = lhs <= rhs (unsigned)
    Uge(VReg, VReg, VReg),     // dst = lhs >= rhs (unsigned)

    // === Conversion ===
    Zext(VReg, VReg, TypeId),  // Zero-extend to wider type
    Sext(VReg, VReg, TypeId),  // Sign-extend to wider type
    Trunc(VReg, VReg, TypeId), // Truncate to narrower type
    PtrToInt(VReg, VReg, TypeId), // Bitcast pointer to integer
    IntToPtr(VReg, VReg, TypeId), // Bitcast integer to pointer

    // === Aggregate / Bulk Memory ===
    Gep(VReg, VReg, VReg, u64), // Get element pointer (base, index, stride)
    MemCopy(VReg, VReg, u64),  // Copy memory (dst ptr, src ptr, size)

    // === Call ===
    Call(VReg, String, Vec<VReg>), // Call function (dest, name, args)
}

Semantic Categories Explained

Memory instructions (Alloc, Load, Store, MovImm): These manage the interaction between values and memory. Alloc reserves stack space; Load/Store transfer values between registers and memory; MovImm loads a compile-time constant.

Arithmetic (Add, Sub, Mul, etc.): Three-address operations where the first vreg is the destination and the remaining two are sources. The signed/unsigned split for division and remainder matters because x86-64 has different instructions (idiv vs. div).

Comparison (Eq, Slt, Ult, etc.): Each comparison produces a boolean (0 or 1) in the destination register. The signed/unsigned variants reflect different x86-64 condition codes (setl vs. setb).

Conversion (Zext, Sext, Trunc): These change the bit width of a value. Zext zero-extends (for unsigned types), Sext sign-extends (for signed types), Trunc truncates to a narrower type. PtrToInt/IntToPtr are reinterpretation casts.

Gep and MemCopy: Gep computes a pointer to an array element: dest = base + index * stride. It is a pointer arithmetic instruction, not a load/store. MemCopy copies a fixed-size block of memory (for struct assignment).

IrTerminator — How Blocks End

Every basic block ends with exactly one terminator. There are only three:

pub enum IrTerminator {
    Jump(IrBlockId),                    // Unconditional jump to target block
    Branch(VReg, IrBlockId, IrBlockId), // Branch on condition
    Return(VReg),                        // Return value from function
}
  • Jump: Always goes to the target block. Used for straight-line control flow (e.g., skip over else branch).
  • Branch: If condition vreg is non-zero, go to the first (true) block; otherwise go to the second (false) block. Used for if and while.
  • Return: Exit the function with the given vreg as the return value. A void function returns %0 (a dummy zero).

SSA and Phi Nodes

The IR is in SSA form: every virtual register is defined exactly once. When different control flow paths produce different values for the same variable, an IrPhi node selects the correct one based on which predecessor block we arrived from:

pub struct IrPhi {
    pub dest: VReg,
    pub type_: TypeId,
    pub sources: Vec<(VReg, IrBlockId)>,
}

Semantically: %3 = phi(bb0: %1, bb1: %2) means "if the previous block was bb0, %3 = %1; if from bb1, %3 = %2." The codegen pass implements this by ensuring both predecessors place their value in the same physical location (register or stack slot).

The dest() Method

Most instructions produce a result in a destination vreg. The dest() method provides a uniform way to query this for optimization passes:

src/ir.rs — dest() method
impl IrInstr {
    pub fn dest(&self) -> VReg {
        match self {
            IrInstr::Alloc(d, _)
            | IrInstr::Load(d, _)
            | IrInstr::Add(d, _, _) | IrInstr::Sub(d, _, _)
            | IrInstr::Mul(d, _, _) | IrInstr::SDiv(d, _, _)
            | IrInstr::UDiv(d, _, _) | IrInstr::SRem(d, _, _)
            | IrInstr::URem(d, _, _) | IrInstr::And(d, _, _)
            | IrInstr::Or(d, _, _) | IrInstr::Xor(d, _, _)
            | IrInstr::Shl(d, _, _) | IrInstr::Shr(d, _, _)
            | IrInstr::Ushr(d, _, _) | IrInstr::Neg(d, _)
            | IrInstr::Not(d, _) | IrInstr::BitNot(d, _)
            | IrInstr::Eq(d, _, _) | IrInstr::Ne(d, _, _)
            | IrInstr::Slt(d, _, _) | IrInstr::Sgt(d, _, _)
            | IrInstr::Sle(d, _, _) | IrInstr::Sge(d, _, _)
            | IrInstr::Ult(d, _, _) | IrInstr::Ugt(d, _, _)
            | IrInstr::Ule(d, _, _) | IrInstr::Uge(d, _, _)
            | IrInstr::Zext(d, _, _) | IrInstr::Sext(d, _, _)
            | IrInstr::Trunc(d, _, _) | IrInstr::PtrToInt(d, _, _)
            | IrInstr::IntToPtr(d, _, _) | IrInstr::Gep(d, _, _, _)
            | IrInstr::Call(d, _, _) | IrInstr::MovImm(d, _, _) => *d,
            IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {
                panic!("Store/MemCopy has no dest")
            }
        }
    }
}

This is used by optimization passes (e.g., dead code elimination) and register allocation to find all definitions without matching on every variant.

IrGlobal, IrString, IrConstant

Globals are named static variables at the module level:

pub struct IrGlobal {
    pub name: String,
    pub type_: TypeId,
    pub init: Option<IrConstant>,
}

Strings are pooled to avoid duplicates and stored as raw byte arrays:

pub struct IrString {
    pub id: StringId,
    pub value: Vec<u8>,
}

pub enum IrConstant {
    Int(u64, TypeId),   // Integer constant with type
    String(StringId),   // Reference to pooled string
    Zero(TypeId),       // Zero-initialized value of given type
}

Example IR Dump

Here is what the IR looks like for a simple program. The IR builder produces this structure, and a hypothetical printer would output:

Source: func main(i32) { return 1+2; }

IR output:

function main(i32) {
// entry: bb0
bb0 (entry):
%0 = Alloc i32 // allocate return slot (unused)
%1 = MovImm 1 : i32 // load constant 1
%2 = MovImm 2 : i32 // load constant 2
%3 = Add %1, %2 // %3 = 1 + 2
Return %3 // return %3 (value 3)
}

Even this trivial example shows the IR at work: the expression 1+2 is decomposed into three instructions, each with a single, clear meaning. An optimizer could constant-fold this to MovImm %3, 3 and eliminate %1 and %2.

Testing the IR

The IR test suite covers construction of every data structure:

#[test]
fn test_vreg_new() {
    let r = VReg::new(5);
    assert_eq!(r.index(), 5);
}

#[test]
fn test_vreg_display() {
    assert_eq!(format!("{}", VReg::new(3)), "%3");
}

#[test]
fn test_ir_module() {
    let m = IrModule {
        name: "test".into(), functions: Vec::new(),
        globals: Vec::new(), strings: Vec::new(),
    };
    assert_eq!(m.name, "test");
}

#[test]
fn test_ir_function() {
    let f = IrFunction {
        name: "main".into(), return_type: TypeId::new(0),
        params: Vec::new(), blocks: Vec::new(),
        num_vregs: 0, entry_block: IrBlockId::new(0),
    };
    assert_eq!(f.name, "main");
}

#[test]
fn test_instr_dest() {
    let add = IrInstr::Add(VReg::new(0), VReg::new(1), VReg::new(2));
    assert_eq!(add.dest(), VReg::new(0));

    let call = IrInstr::Call(VReg::new(3), "foo".into(), vec![VReg::new(1)]);
    assert_eq!(call.dest(), VReg::new(3));
}

#[test]
fn test_branch_terminator() {
    let t = IrTerminator::Branch(VReg::new(0), IrBlockId::new(1), IrBlockId::new(2));
    match t {
        IrTerminator::Branch(c, a, b) => {
            assert_eq!(c, VReg::new(0));
            assert_eq!(a, IrBlockId::new(1));
            assert_eq!(b, IrBlockId::new(2));
        }
        _ => panic!("expected Branch"),
    }
}

#[test]
fn test_phi_node() {
    let phi = IrPhi {
        dest: VReg::new(0), type_: TypeId::new(0),
        sources: vec![(VReg::new(1), IrBlockId::new(0)),
                          (VReg::new(2), IrBlockId::new(1))],
    };
    assert_eq!(phi.sources.len(), 2);
}

#[test]
fn test_ir_constant() {
    let c = IrConstant::Int(42, TypeId::new(0));
    match c {
        IrConstant::Int(v, _) => assert_eq!(v, 42),
        _ => panic!("expected Int"),
    }
}

To run IR tests:

$ cargo test -- ir

Common Pitfalls

  1. Store/MemCopy have no destination: Unlike every other instruction, Store and MemCopy do not produce a value. The dest() method panics on these. Optimization passes must handle this case explicitly.
  2. Signed vs. unsigned comparisons: Using Slt when Ult is intended will give wrong results for negative numbers (e.g., -1 as u32 > 1 as u32 in unsigned, but -1 < 1 in signed). The type checker must preserve signedness through to the IR.
  3. Missing phis for SSA correctness: If multiple control flow paths assign to the same variable, a phi node is required. The IR builder must insert phis at join points (e.g., after if/else).
  4. VReg numbering mismatch: num_vregs in IrFunction must be greater than the highest vreg index. Off-by-one errors here cause register allocation to miss the last register.
Why 36 instruction variants? This is a minimal set that can express all of L1S's language features while mapping cleanly to x86-64 instructions. In contrast, LLVM has hundreds of instruction opcodes. The simplicity of L1S's IR makes it easy to reason about and transform, at the cost of some expressiveness.
