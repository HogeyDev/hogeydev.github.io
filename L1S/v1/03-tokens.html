3   The Token System

What is a Token?

A token is the smallest meaningful unit of a program. Where the source file is a flat sequence of characters, a token is a structured piece of data that says "this is a keyword," "this is an identifier," "this is an operator." The lexer's job is to convert characters to tokens; the parser's job is to convert tokens to an AST.

Every compiler has a token type. L1S defines its tokens in a single file src/token.rs — the single source of truth shared by the lexer, parser, and all downstream tools.

Why a dedicated token file? Both the lexer (which produces tokens) and the parser (which consumes them) need to reference the same token types. Putting them in token.rs avoids circular dependencies and keeps the token definitions independent of both the lexing and parsing implementations.

The Token Enum

The core of the system is a Token enum with variants organized into five categories:

src/token.rs — Token enum
pub enum Token {
    // Keywords
    Func, Let, Return, If, Else,
    While, For, Struct, Enum, Spec,
    Impl, Import, Asm, Cast, Pub,
    Static, Typedef, Noprefix, True, False,

    // Primitive types
    Void, Bool, Char, Usize, Isize,
    U64, I64, U32, I32, U16, I16, U8, I8,

    // Literals
    NumLiteral(String),
    StrLiteral(String),
    CharLiteral(u8),

    // Identifiers
    Ident(String),

    // Operators
    Plus, Minus, Star, Slash, Percent,
    Amp, Pipe, Caret, Tilde, Bang,
    Eq, EqEq, BangEq, Lt, Gt,
    LtEq, GtEq, AmpAmp, PipePipe,
    LtLt, GtGt, Arrow, Dot, At,

    // Delimiters
    OpenParen, CloseParen,
    OpenBrace, CloseBrace,
    OpenBracket, CloseBracket,
    Semicolon, Comma, Colon,

    // Special
    Eof,
}

Keywords are reserved words that have special meaning in the language. They are not identifiers — you cannot name a variable func or let:

Why are true and false keywords, not identifiers? In L1S, boolean literals are keywords so the parser can distinguish them from regular identifiers without symbol table lookups. This is a common design — Rust, Java, and Python all treat boolean literals as reserved. The alternative (lexing them as Ident("true") and relying on the parser to classify them) adds complexity for no benefit.

Category 2: Primitive Types (13 variants)

Every built-in type has its own token variant. This is a deliberate design choice: primitive types are not lexed as identifiers and then resolved later. They are first-class tokens from the start:

BoolboolBoolean (1 byte)

U8 / I8u8 / i88-bit unsigned/signed int
Keywords vs. identifiers: the critical tradeoff A typical two-phase design (lex everything as Ident, then classify in the parser) is simpler for the lexer but pushes complexity downstream. L1S's approach — making keywords and primitives distinct tokens — means the lexer itself handles classification, so the parser gets unambiguous tokens. The tradeoff is that adding a new keyword requires a new enum variant and a match arm in keyword_token. However, this is minor: both changes happen in the same file.

Category 3: Literals

Literals carry embedded data. They are the only token variants that hold values:

Token VariantDataSource ExampleStored As
NumLiteralString42, 0xFFRaw source text (not parsed integer)
StrLiteralString"hello"Content with escapes resolved
CharLiteralu8'x'Single byte value

Why store NumLiteral as a String rather than an i64 or similar? The lexer does not know the expected type of the literal. 42 could be i32, u64, or even a pointer. The raw string is preserved (including radix prefixes like 0x and underscores) and parsed into a numeric value during semantic analysis when the target type is known.

Why store StrLiteral with escapes resolved? Escape sequences (\n, \x48, \t) are lexer concerns — they describe how characters are encoded in source, not what the string content actually is. Resolving them in the lexer means downstream passes see "\n" as a single newline byte, not a backslash followed by n.

Why u8 for CharLiteral? L1S does not support multi-byte character literals. A character literal is always exactly one byte.

Category 4: Operators (24 variants)

Every operator is a distinct token. Multi-character operators (==, !=, &&, ||, <<, >>, ->, <=, >=) each get their own variant rather than being sequences of single-character tokens:

Why separate multi-character operators? If == were lexed as Eq Eq (two = tokens), the parser would need unbounded lookahead to distinguish if a = b (assignment in condition, likely an error) from if a == b (comparison). By making == a single token, the parser can decide immediately upon seeing it. This is a classic tradeoff: more token variants vs. simpler parsing.

Category 6: Eof

A single Eof variant signals the end of input. The lexer guarantees exactly one Eof token at the end of the token stream, even for empty input.

The SpannedToken Wrapper

Tokens rarely travel alone. Every token needs its source location so errors can point back to the right place. The SpannedToken struct pairs a token with its Span:

src/token.rs — SpannedToken
pub struct SpannedToken {
    pub token: Token,
    pub span: Span,
}

impl SpannedToken {
    pub fn new(token: Token, span: Span) -> Self {
        SpannedToken { token, span }
    }
}
Why a separate struct instead of embedding Span in Token? Two reasons. First, not every context needs a span — for example, tests that check token sequences don't care about positions. Second, Span is a property of the token's occurrence in source, not of the token itself. A Token::Func variant doesn't inherently have a position; a SpannedToken with Token::Func does. This separation models the domain correctly.

The keyword_token() Function

This is the bridge function that maps identifier-like strings to their keyword counterparts:

src/token.rs — keyword_token()
pub fn keyword_token(word: &str) -> Option<Token> {
    match word {
        "func" | "Func" => Some(Token::Func),
        "let" => Some(Token::Let),
        "return" => Some(Token::Return),
        "if" => Some(Token::If),
        "else" => Some(Token::Else),
        "while" => Some(Token::While),
        "for" => Some(Token::For),
        "struct" => Some(Token::Struct),
        "enum" => Some(Token::Enum),
        "spec" => Some(Token::Spec),
        "impl" => Some(Token::Impl),
        "import" => Some(Token::Import),
        "asm" => Some(Token::Asm),
        "cast" => Some(Token::Cast),
        "pub" => Some(Token::Pub),
        "static" => Some(Token::Static),
        "typedef" => Some(Token::Typedef),
        "noprefix" => Some(Token::Noprefix),
        "true" => Some(Token::True),
        "false" => Some(Token::False),
        "void" => Some(Token::Void),
        "bool" => Some(Token::Bool),
        "char" => Some(Token::Char),
        "usize" => Some(Token::Usize),
        "isize" => Some(Token::Isize),
        "u64" => Some(Token::U64),
        "i64" => Some(Token::I64),
        "u32" => Some(Token::U32),
        "i32" => Some(Token::I32),
        "u16" => Some(Token::U16),
        "i16" => Some(Token::I16),
        "u8" => Some(Token::U8),
        "i8" => Some(Token::I8),
        _ => None,
    }
}

This is a pure function: given a string, it returns either the matching keyword token or None. It has no side effects and no dependencies. The lexer calls it like this:

fn lex_identifier_or_keyword(&mut self) -> Token {
    let mut ident = String::new();
    while let Some(c) = self.peek() {
        if c.is_alphanumeric() || c == '_' {
            self.advance();
            ident.push(c);
        } else { break; }
    }
    keyword_token(&ident).unwrap_or(Token::Ident(ident))
}
Note that Func matches both "func" and "Func". L1S supports capitalized function names for type constructor functions (like StructName(val)), while func remains the keyword form. Both map to the same Token::Func variant; the parser distinguishes them by context.

Why keyword_token Is the Right Abstraction

The design of having a centralized keyword_token function is a key architectural decision. Consider the alternatives:

  • No keyword classification in the lexer: Lex every alphanumeric sequence as Ident and let the parser decide. This forces the parser to carry a keyword table, complicates error messages ("expected keyword 'func', found identifier 'func'"?), and makes the token stream less informative.
  • Hardcode keyword detection in the lexer: Put a giant match directly inside lex_identifier_or_keyword. This works but couples the lexer to the keyword list, making it harder to reuse the token definitions in other tools (formatters, linters, syntax highlighters).
  • Use a hashmap: Populate a HashMap<&str, Token> at startup. This is more flexible but adds allocation and initialization overhead, and the match-based approach is faster (the compiler optimizes string matches into a perfect hash or jump table).

The keyword_token approach is the sweet spot: it's a standalone, testable, pure function that the lexer delegates to, and that any other tool can also use.

Testing the Token System

The #[cfg(test)] module at the bottom of token.rs tests every aspect of the type system:

src/token.rs — tests module
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_keywords() {
        assert_eq!(keyword_token("func"), Some(Token::Func));
        assert_eq!(keyword_token("let"), Some(Token::Let));
        assert_eq!(keyword_token("return"), Some(Token::Return));
        assert_eq!(keyword_token("if"), Some(Token::If));
        assert_eq!(keyword_token("else"), Some(Token::Else));
        assert_eq!(keyword_token("while"), Some(Token::While));
        assert_eq!(keyword_token("for"), Some(Token::For));
        assert_eq!(keyword_token("struct"), Some(Token::Struct));
        assert_eq!(keyword_token("enum"), Some(Token::Enum));
        assert_eq!(keyword_token("spec"), Some(Token::Spec));
        assert_eq!(keyword_token("impl"), Some(Token::Impl));
        assert_eq!(keyword_token("import"), Some(Token::Import));
        assert_eq!(keyword_token("asm"), Some(Token::Asm));
        assert_eq!(keyword_token("cast"), Some(Token::Cast));
        assert_eq!(keyword_token("pub"), Some(Token::Pub));
        assert_eq!(keyword_token("static"), Some(Token::Static));
        assert_eq!(keyword_token("typedef"), Some(Token::Typedef));
        assert_eq!(keyword_token("noprefix"), Some(Token::Noprefix));
        assert_eq!(keyword_token("true"), Some(Token::True));
        assert_eq!(keyword_token("false"), Some(Token::False));
    }

    #[test]
    fn test_primitive_keywords() {
        assert_eq!(keyword_token("void"), Some(Token::Void));
        assert_eq!(keyword_token("i32"), Some(Token::I32));
        assert_eq!(keyword_token("u64"), Some(Token::U64));
    }

    #[test]
    fn test_non_keyword() {
        assert_eq!(keyword_token("foo"), None);
        assert_eq!(keyword_token("bar"), None);
        assert_eq!(keyword_token("x123"), None);
    }

    #[test]
    fn test_spanned_token() {
        let span = Span::new(0, 4, 1, 1);
        let st = SpannedToken::new(Token::Func, span);
        assert_eq!(st.token, Token::Func);
        assert_eq!(st.span, span);
    }

    #[test]
    fn test_token_equality() {
        assert_eq!(Token::Plus, Token::Plus);
        assert_eq!(Token::NumLiteral("42".into()),
                   Token::NumLiteral("42".into()));
        assert_ne!(Token::Plus, Token::Minus);
    }

    #[test]
    fn test_ident_token() {
        let ident = Token::Ident("hello".into());
        match ident {
            Token::Ident(s) => assert_eq!(s, "hello"),
            _ => panic!("expected Ident"),
        }
    }

    #[test]
    fn test_literal_tokens() {
        let n = Token::NumLiteral("123".into());
        let s = Token::StrLiteral("abc".into());
        let c = Token::CharLiteral(b'x');
        match n { Token::NumLiteral(ref v) => assert_eq!(v, "123"), _ => panic!() }
        match s { Token::StrLiteral(ref v) => assert_eq!(v, "abc"), _ => panic!() }
        match c { Token::CharLiteral(v) => assert_eq!(v, b'x'), _ => panic!() }
    }

    #[test]
    fn test_eof_token() {
        assert_eq!(format!("{:?}", Token::Eof), "Eof");
    }
}

There are 8 test functions covering:

  • test_keywords: Every keyword maps correctly via keyword_token().
  • test_primitive_keywords: Same for the primitive type names.
  • test_non_keyword: Ordinary identifiers return None.
  • test_spanned_token: The SpannedToken wrapper correctly stores both a token and a span.
  • test_token_equality: The PartialEq derive works correctly (including data comparison for NumLiteral).
  • test_ident_token: Pattern matching on the Ident variant extracts the string.
  • test_literal_tokens: All three literal variants store the right data types.
  • test_eof_token: Debug formatting for the Eof variant.

To run the token tests in isolation:

$ cargo test -- token

Derive Macros: Why #[derive(Debug, Clone, PartialEq)]?

The Token enum derives three traits:

  • Debug: Required for assert_eq! error messages, which print both sides on failure. Also useful for debugging the lexer output.
  • Clone: The parser often needs to inspect a token without consuming it (lookahead). Clone lets us keep a copy.
  • PartialEq: Required for test assertions and for the parser's token matching. Note that Eq is NOT derived — NumLiteral(String) can never be compared reflexively in a meaningful way for the compiler's purposes, but PartialEq with no Eq is sufficient for all our use cases.
Why not derive Eq? Rust's Eq trait requires reflexivity: a == a must always be true. For all of our token variants this holds, so we could derive it. However, by convention, we only derive what we need. Adding Eq later is a non-breaking change. This is a minor design choice, but it's worth noting for awareness.

The Big Picture: How Tokens Flow

Data flow through the compiler pipeline:

Source text (.l1s file)
↓
Lexer.tokenize()
↓
Vec<SpannedToken>   (each token knows its source location)
↓
Parser.parse()   (consumes tokens, produces AstProgram)
↓
Semantic analysis / Codegen

Every stage after the lexer works with Token variants through the SpannedToken wrapper. The parser never sees raw characters — it only sees the structured token stream. This separation is what makes the compiler modular: you could replace the lexer entirely (e.g., with a hand-written scanner or a lexer generator like logos) without changing anything downstream, as long as the same Token enum is produced.

Common Pitfalls & Debugging Tips

  1. Forgetting to add a new keyword in two places: Adding Loop to the Token enum but not to keyword_token() means the lexer will produce Ident("loop") instead of Token::Loop. Always update both the variant and the match arm.
  2. Case sensitivity: keyword_token matches both "func" and "Func" for Token::Func, but "Struct" is NOT a keyword (only "struct" is). If you change this, update the tests too.
  3. Derive mismatch: If you add a field to a token variant that doesn't implement PartialEq or Debug, the compiler will error. Make sure new fields satisfy the trait bounds.
  4. Span is not Clone: Span derives Clone + Copy. If you change this, SpannedToken stops being clonable and the parser breaks.
  5. Testing empty input: Always test that keyword_token("") returns None. Empty strings should never match a keyword.
Debugging tip: If the parser is producing unexpected errors, dump the token stream first. Add a quick dbg! to the parser's token consumption loop to see what tokens the lexer actually produced. Often the bug is in the lexer (wrong token for a keyword) rather than in the parser.
