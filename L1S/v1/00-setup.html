What is a Compiler?

A compiler is a program that translates source code written in one language (the source language) into another language (the target language), typically machine code or assembly. L1S translates a C-like language into x86-64 assembly, then uses nasm and cc to produce an executable binary.

At its core, a compiler has no magic. It's a pipeline of well-defined transformations, each taking structured data from the previous phase and producing different structured data for the next phase:

Source Text → Tokens → AST → Typed AST → IR → Opt IR → Assembly → Executable

Each box represents a data structure. Each arrow is a compiler phase. By the end of this book, you'll understand every arrow and every box in the diagram above.

Required Tools

Before you can build and run L1S, you need three tools installed:

ToolPurposeInstallation Check
Rust (rustc & cargo) The compiler itself is written in Rust. You need the Rust toolchain to build L1S. rustc --version && cargo --version
NASM The Netwide Assembler. L1S emits x86-64 NASM syntax assembly; nasm turns it into an object file. nasm --version
cc (GCC or Clang) The C compiler driver. Used as a linker to produce the final executable from the object file. cc --version
Platform note: L1S targets x86-64 Linux (ELF64). The generated assembly uses Linux syscalls. Cross-platform support is not currently implemented.

Install missing tools using your system package manager:

# Debian/Ubuntu
sudo apt install rustc cargo nasm build-essential

# Arch Linux
sudo pacman -S rust nasm base-devel

# Fedora
sudo dnf install rust cargo nasm gcc

Building L1S

Clone or navigate to the L1S project root (the directory containing Cargo.toml) and build:

$ cargo build

This compiles all Rust source files in src/ and produces a debug binary at target/debug/l1s (or the crate name defined in Cargo.toml).

For a release build (optimized, runs faster):

$ cargo build --release

To verify your build was successful:

$ cargo build 2>&1
   Compiling l1s v0.1.0 (/path/to/L1S)
    Finished dev [unoptimized + debuginfo] target(s) in 0.31s
First build: The first cargo build will download dependencies and compile them, which may take a while. Subsequent builds are incremental and fast.

Running L1S

Once built, run L1S on a source file:

$ cargo run -- examples/hello.is

The -- separates cargo arguments from the program arguments. Everything after -- is passed to the compiled binary.

This will:

  1. Read examples/hello.is
  2. Lex, parse, type-check, and compile it
  3. Emit examples/hello.asm (the NASM assembly file)
  4. Invoke nasm to assemble into examples/hello.o
  5. Invoke cc to link into examples/hello
  6. Print wrote examples/hello
  7. Clean up the .o file

Run the resulting executable:

$ ./examples/hello

If your program prints output, you will see it here.

The -o Flag

By default, L1S derives the output binary name from the input filename: hello.is produces hello. Use the -o flag to specify a different output path:

$ cargo run -- examples/hello.is -o my_program

This writes:

  • my_program.asm — the assembly output
  • my_program — the final executable

The logic lives in main.rs:68-73:

let bin_path = if args.len() > 2 && args[2] == "-o" {
    args[3].clone()
} else {
    let input = Path::new(&args[1]);
    input.with_extension("").to_string_lossy().to_string()
};

If -o is present, the next argument is used as the output path. Otherwise, the input filename has its extension stripped.

CLI Entry Point

All L1S compilation begins in src/main.rs. The main function (lines 22–32) handles argument parsing and file reading:

src/main.rs — Entry point (lines 22-32)
fn main() {
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: ignis <file.is> [-o <output>]");
        std::process::exit(1);
    }

    let source = fs::read_to_string(&args[1]).unwrap_or_else(|e| {
        eprintln!("error reading {}: {}", args[1], e);
        std::process::exit(1);
    });
}

Step by step:

  1. Collect arguments: env::args().collect() gives a Vec<String> of all command-line arguments. The first is the program path; the second should be the input file.
  2. Validate usage: If fewer than 2 arguments are provided, print usage and exit with code 1. Early and clear error messages are a compiler virtue.
  3. Read source file: fs::read_to_string reads the entire file into a String. On failure, print the OS error and exit. After this, source contains the full program text.

The remaining 80 lines of main.rs connect the compiler phases together, which we explore in Chapter 1.

Project Structure

The L1S compiler consists of 15 Rust source files plus the entry point. Each module handles one part of the compilation pipeline:

FileModulePurpose
src/main.rs—Compiler driver: orchestrates all phases, CLI argument handling
src/span.rsspanSource location tracking (byte offsets + line/column)
src/token.rstokenToken type definitions and keyword classification
src/lexer.rslexerLexical analysis: source text → token stream
src/ast.rsastAbstract syntax tree data structures
src/parser.rsparserRecursive-descent parser: tokens → AST
src/types.rstypesType system (interned type arena, structural equality)
src/symbol.rssymbolSymbol table for name resolution
src/diag.rsdiagError diagnostics and reporting
src/typeck.rstypeckType checker: semantic analysis
src/ir.rsirIntermediate representation data structures
src/ir_build.rsir_buildAST → IR lowering
src/ir_opt.rsir_optIR optimization passes
src/abi.rsabiCalling convention and ABI details
src/regalloc.rsregallocRegister allocation (virtual → physical registers)
src/codegen.rscodegenx86-64 NASM code generation

The mod declarations in main.rs register each module with the compiler. The Rust compiler compiles them in the order they are declared, which follows the pipeline order.

Development Workflow

During compiler development, you will frequently iterate on one phase at a time. These cargo commands will be your daily tools:

CommandUse Case
cargo buildFull compilation, produces the binary
cargo checkFast type-check only (no binary produced)
cargo run -- test.isBuild and run the compiler on a test file
cargo testRun all unit and integration tests
cargo test -- lexerRun tests matching "lexer" in their name
cargo clippyRun Rust lints to catch common mistakes

cargo check is particularly useful during development: it verifies type correctness without running the full code generation pipeline, making it significantly faster than cargo build.

The .is File Extension

L1S source files use the .is extension (short for "Ignis Source," the historical name of the project). The compiler does not enforce this extension — it will happily read .txt or any other extension — but following the convention helps tools and team members identify L1S source files at a glance.

Example program structure:

// A complete L1S program
func add(i32, a: i32, b: i32) {
    return a + b;
}

func main(i32) {
    let result: i32 = add(2, 3);
    return result;
}

Your First L1S Program

Create a file called hello.is:

func main(i32) {
    return 42;
}

Build and run:

$ cargo run -- hello.is
$ ./hello
$ echo $?
42
Exit code: The value returned from main becomes the program's exit code. return 42; means the shell sees exit code 42 after the program runs.

Common Pitfalls

  1. Missing NASM: If nasm is not installed, the build succeeds but compilation fails with error invoking assembler. Install NASM with your package manager.
  2. Wrong working directory: The source file path is relative to where you run cargo run, not relative to the source file. Run the command from the project root.
  3. cargo run vs cargo build: cargo run builds if needed and runs the binary. cargo build just compiles. Use cargo build to check for compile errors without running.
  4. Unclosed strings at EOF: The lexer handles these gracefully (records an error and continues), so a typo won't crash the compiler.
Quick sanity check: Run cargo build 2>&1 | head -5 to verify the compiler itself compiles cleanly. If you see errors, check your Rust toolchain version with rustc --version.
