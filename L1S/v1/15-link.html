15   Assembly & Linking

The Compiler Driver

With the assembly string produced by Codegen, we have human-readable x86-64 NASM source code. But a string is not an executable — we need to assemble it into machine code and link it into a runnable binary. This is the job of the compiler driver in main.rs.

The driver is a single main() function that orchestrates the entire compilation pipeline, from reading source files to producing executables. Let's trace the complete flow.

The Full Pipeline

Build pipeline from source to executable:

/home/user/project/
    │
    ├── main.is          // L1S source (you write this)
    │
    │   [ignis compiles main.is]
    │
    ├── main.asm         // NASM assembly (generated by Codegen)
    │
    │   [nasm -f elf64 main.asm]
    │
    ├── main.o           // ELF64 object file (generated by nasm)
    │
    │   [cc -o main main.o -no-pie]
    │
    ├── main             // Executable binary (generated by linker)
    │
    │   [./main]
    │
    └── output: 42      // Program runs!

Notice that main.o is a temporary file — the driver deletes it after linking successfully.

The Driver Code (lines 68–109)

Here is the complete assembly and linking section of the driver:

src/main.rs — Assembly, linking, and output
    // --- Determine output binary path ---
    let bin_path = if args.len() > 2 && args[2] == "-o" {
        args[3].clone()
    } else {
        let input = Path::new(&args[1]);
        input.with_extension("").to_string_lossy().to_string()
    };

    let asm_path = format!("{}.asm", bin_path);
    let o_path = format!("{}.o", bin_path);

    // --- Step 1: Write .asm file ---
    let _ = fs::write(&asm_path, &asm).unwrap_or_else(|e| {
        eprintln!("error writing {}: {}", asm_path, e);
        std::process::exit(1);
    });

    --- Step 2: Assemble with nasm ---
    let status = Command::new("nasm")
        .args(&["-f", "elf64", "-o", &o_path, &asm_path])
        .status()
        .unwrap_or_else(|e| {
            eprintln!("error invoking assembler: {}", e);
            std::process::exit(1);
        });

    if !status.success() {
        std::process::exit(1);
    }

    --- Step 3: Link with cc ---
    let status = Command::new("cc")
        .args(&["-o", &bin_path, &o_path, "-no-pie"])
        .status()
        .unwrap_or_else(|e| {
            eprintln!("error invoking linker: {}", e);
            std::process::exit(1);
        });

    if !status.success() {
        std::process::exit(1);
    }

    --- Step 4: Cleanup .o file ---
    let _ = fs::remove_file(&o_path);

    println!("wrote {}", bin_path);
}

Step-by-Step Walkthrough

1. Output Path Resolution (lines 68–73)

The driver supports an optional -o <name> flag:

$ ignis program.is                  # produces ./program
$ ignis program.is -o mybinary     # produces ./mybinary

The path is parsed by checking args[2] == "-o". If the flag is present, args[3] is used as the output name. Otherwise, the input filename's extension is stripped: program.is becomes program. This uses Path::with_extension("") which handles edge cases like foo.is.is correctly (strips only the last extension).

2. Write Assembly File (lines 78–81)

fs::write(&asm_path, &asm).unwrap_or_else(|e| {
    eprintln!("error writing {}: {}", asm_path, e);
    std::process::exit(1);
});

The assembly string from Codegen::generate() is written to a .asm file in the same directory as the intended output binary. If the write fails (e.g., permission denied, disk full), the error message is printed to stderr and the process exits with code 1.

Why keep the .asm file? The .asm file is intentionally not deleted. This lets developers inspect the generated assembly for debugging purposes. If you want to see what code L1S produced for your program, just look at program.asm.

3. Assemble with NASM (lines 83–93)

let status = Command::new("nasm")
    .args(&["-f", "elf64", "-o", &o_path, &asm_path])
    .status()
    .unwrap_or_else(|e| { /* ... */ });

if !status.success() { std::process::exit(1); }

NASM (Netwide Assembler) is invoked with these flags:

  • -f elf64: Output format is ELF64 (Executable and Linkable Format, 64-bit), the standard object file format on Linux x86-64.
  • -o <.o path>: Output object file path.
  • <.asm path>: Input assembly source file.

The Command::status() method runs the process and returns its exit status. If NASM isn't installed, unwrap_or_else catches the std::io::Error (e.g., "No such file or directory") and prints a helpful message. If NASM finds errors in the assembly (syntax errors, undefined symbols), status.success() returns false and the driver exits.

NASM must be installed! L1S requires NASM to be available on your PATH. Install it with your package manager: 
# Debian/Ubuntu
$ sudo apt install nasm

# Fedora
$ sudo dnf install nasm

# macOS
$ brew install nasm

4. Link with cc (lines 95–105)

let status = Command::new("cc")
    .args(&["-o", &bin_path, &o_path, "-no-pie"])
    .status()
    .unwrap_or_else(|e| { /* ... */ });

if !status.success() { std::process::exit(1); }

The C compiler (cc, typically gcc or clang) is used as the linker. This is a common trick in small compilers — cc knows how to invoke the linker (ld) with all the right flags to set up the C runtime (_start, libc crt files, etc.). We don't need to link against libc itself, but the crt startup files are necessary for a working executable. The flags:

  • -o <binary>: Output executable path.
  • <.o file>: Input object file.
  • -no-pie: Produce a non-position-independent executable. L1S doesn't generate PIC-compliant code, so we disable PIE to avoid linker errors. Without this flag, the linker would try to make a PIE binary and fail because the assembly uses absolute addresses in some cases.

5. Cleanup (line 107)

let _ = fs::remove_file(&o_path);

The intermediate .o object file is deleted. The let _ = suppresses the "unused result" warning — if deletion fails (e.g., file already removed), we don't care. The .asm file is kept for debugging.

6. Success Message (line 109)

println!("wrote {}", bin_path);

A simple confirmation message tells the user the binary is ready.

The Complete main() Function

For context, here is the full driver including the compilation stages that precede assembly and linking:

src/main.rs — Complete compiler driver
fn main() {
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: ignis <file.is> [-o <output>]");
        std::process::exit(1);
    }

    // 1. Read source file
    let source = fs::read_to_string(&args[1]).unwrap_or_else(|e| { /* ... */ });

    // 2. Lex
    let mut lexer = lexer::Lexer::new(&source);
    let (tokens, _lex_errors) = lexer.tokenize();

    // 3. Parse
    let diag = diag::Diagnostics::new();
    let mut parser = parser::Parser::new(tokens, diag);
    let mut prog = parser.parse_program();
    let mut diag = parser.into_diagnostics();
    if diag.has_errors() { diag.print_all(); std::process::exit(1); }

    // 4. Type check
    let mut types = types::TypeArena::new();
    let mut symbols = symbol::SymbolTable::new();
    let mut tc = typeck::TypeChecker::new(&mut types, &mut symbols, &mut diag);
    tc.check_program(&mut prog);
    let annotations = tc.annotations.clone();
    let has_errors = tc.diag.has_errors();
    if has_errors { diag.print_all(); std::process::exit(1); }

    // 5. Build IR
    let builder = ir_build::IrBuilder::new(&types, &annotations, &mut diag);
    let mut ir_module = builder.build(&prog);

    // 6. Optimize IR
    ir_opt::optimize(&mut ir_module);

    // 7. Generate assembly
    let mut codegen = codegen::Codegen::new(&types);
    let asm = codegen.generate(&ir_module);

    // 8. Write .asm, assemble, link, cleanup (lines 68-109, shown above)
    // ...
}

Each stage is fallible. The driver either exits early on error (with a non-zero exit code) or proceeds to the next stage. The lexer errors are currently ignored (the _lex_errors variable is unused), but parser and type-checker errors abort compilation.

Error Handling Strategy

The driver uses a straightforward error handling pattern throughout:

StageError CheckExit Code
Usageargs.len() < 21
File readunwrap_or_else1
Parsediag.has_errors()1
Type checkdiag.has_errors()1
File writeunwrap_or_else1
NASM invocationunwrap_or_else + status.success()1
CC invocationunwrap_or_else + status.success()1

All errors are printed to stderr with descriptive messages. The process exits with code 1 on any failure, which is the standard Unix convention for non-zero exit on error.

The ABI Module

The abi.rs module is a small utility that delegates to TypeArena for size and alignment queries, plus a stack alignment helper:

src/abi.rs — ABI helpers
pub fn type_size(types: &TypeArena, t: TypeId) -> u64 {
    types.size_of(t)
}

pub fn type_align(types: &TypeArena, t: TypeId) -> u64 {
    types.align_of(t)
}

pub fn stack_align(size: u64) -> u64 {
    (size + 15) & !15
}

type_size and type_align are thin wrappers that provide a clean interface for other modules (like ir_build during struct layout). The stack_align function rounds up to the next 16-byte boundary, matching the x86-64 SysV ABI requirement. The tests verify this:

#[test]
fn test_stack_align() {
    assert_eq!(stack_align(4), 16);
    assert_eq!(stack_align(16), 16);
    assert_eq!(stack_align(17), 32);
}

#[test]
fn test_type_size() {
    let mut types = TypeArena::new();
    let i32 = types.intern(TypeNode::I32);
    assert_eq!(type_size(&types, i32), 4);
}

Testing the Build Pipeline

The build pipeline is best tested with end-to-end integration tests. A typical test would:

  1. Write a small L1S program to a temp file.
  2. Run ignis temp.is -o temp_bin.
  3. Check exit code is 0.
  4. Run ./temp_bin and check its exit code or stdout.
  5. Clean up temp files.

At the time of writing, the driver's external tool dependencies (NASM, cc) would require integration tests to be run separately:

$ cargo build --release
$ echo "func main(i32) { return 42; }" > /tmp/test.is
$ ./target/release/ignis /tmp/test.is -o /tmp/test_bin
wrote /tmp/test_bin
$ /tmp/test_bin; echo $?
42
Quick debugging: If linking fails, examine the generated .asm file. Try assembling it manually with nasm -f elf64 file.asm to see NASM's error messages. Then try linking manually with cc -o file file.o -no-pie. This isolates whether the error is in assembly generation or linking.

Common Pitfalls & Debugging Tips

  1. NASM not found: The error message is "error invoking assembler: No such file or directory". Install NASM via your package manager.
  2. Linker errors about main: The assembly declares global main, but if your program doesn't define a main function, the linker will fail with an undefined reference. Every L1S program needs a func main(i32) entry point.
  3. Position-independent code: If you omit -no-pie, the linker may emit warnings or errors about relocations against read-only sections. Always use -no-pie until L1S supports PIC.
  4. File path edge cases: Path::with_extension("") handles foo.is → foo, but foo (no extension) becomes foo (unchanged). Inputs like .is (hidden file) become empty string — test your output paths.
  5. Permission denied: If fs::write fails, check that the output directory is writable. The .asm file is written alongside the binary, so consider the directory permissions.
