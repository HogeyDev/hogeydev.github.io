What is a Compiler?

A compiler is a program that translates source code written in one language (the source language) into another language (the target language), typically machine code or assembly. L1S translates a C-like language into x86-64 assembly, then uses nasm and cc to produce an executable binary.

At its core, a compiler has no magic. It's a pipeline of well-defined transformations, each taking structured data from the previous phase and producing different structured data for the next phase:

Source Text → Tokens → AST → Typed AST → IR → Opt IR → Assembly → Executable

Each box represents a data structure. Each arrow is a compiler phase. By the end of this book, you'll understand every arrow and every box in the diagram above.

Building L1S

Clone or navigate to the L1S project root (the directory containing Cargo.toml) and build:

$ cargo build

This compiles all Rust source files in src/ and produces a debug binary at target/debug/l1s (or the crate name defined in Cargo.toml).

For a release build (optimized, runs faster):

$ cargo build --release

To verify your build was successful:

$ cargo build 2>&1
   Compiling l1s v0.1.0 (/path/to/L1S)
    Finished dev [unoptimized + debuginfo] target(s) in 0.31s
First build: The first cargo build will download dependencies and compile them, which may take a while. Subsequent builds are incremental and fast.

Running L1S

Once built, run L1S on a source file:

$ cargo run -- examples/hello.is

The -- separates cargo arguments from the program arguments. Everything after -- is passed to the compiled binary.

This will:

  1. Read examples/hello.is
  2. Lex, parse, type-check, and compile it
  3. Emit examples/hello.asm (the NASM assembly file)
  4. Invoke nasm to assemble into examples/hello.o
  5. Invoke cc to link into examples/hello
  6. Print wrote examples/hello
  7. Clean up the .o file

Run the resulting executable:

$ ./examples/hello

If your program prints output, you will see it here.

The -o Flag

By default, L1S derives the output binary name from the input filename: hello.is produces hello. Use the -o flag to specify a different output path:

$ cargo run -- examples/hello.is -o my_program

This writes:

  • my_program.asm — the assembly output
  • my_program — the final executable

The logic lives in main.rs:68-73:

let bin_path = if args.len() > 2 && args[2] == "-o" {
    args[3].clone()
} else {
    let input = Path::new(&args[1]);
    input.with_extension("").to_string_lossy().to_string()
};

If -o is present, the next argument is used as the output path. Otherwise, the input filename has its extension stripped.

CLI Entry Point

All L1S compilation begins in src/main.rs. The main function (lines 22–32) handles argument parsing and file reading:

src/main.rs — Entry point (lines 22-32)
fn main() {
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: ignis <file.is> [-o <output>]");
        std::process::exit(1);
    }

    let source = fs::read_to_string(&args[1]).unwrap_or_else(|e| {
        eprintln!("error reading {}: {}", args[1], e);
        std::process::exit(1);
    });
}

Step by step:

  1. Collect arguments: env::args().collect() gives a Vec<String> of all command-line arguments. The first is the program path; the second should be the input file.
  2. Validate usage: If fewer than 2 arguments are provided, print usage and exit with code 1. Early and clear error messages are a compiler virtue.
  3. Read source file: fs::read_to_string reads the entire file into a String. On failure, print the OS error and exit. After this, source contains the full program text.

The remaining 80 lines of main.rs connect the compiler phases together, which we explore in Chapter 1.

Common Pitfalls

  1. Missing NASM: If nasm is not installed, the build succeeds but compilation fails with error invoking assembler. Install NASM with your package manager.
  2. Wrong working directory: The source file path is relative to where you run cargo run, not relative to the source file. Run the command from the project root.

1   Pipeline Overview

The Big Picture

A compiler is not a monolithic blob. It is a pipeline — a series of discrete phases, each consuming the output of the previous phase and producing input for the next. This modular design means each phase can be developed, tested, and understood in isolation.

The L1S compiler has eight phases that transform source text into a runnable executable:

Source → 1. Lexer → 2. Parser → 3. Type Checker → 4. IR Builder → 5. Optimizer → 6. Code Generator → 7. Assembler → 8. Linker → Executable

Phases 1–6 are implemented in Rust (inside the compiler binary). Phases 7–8 delegate to external tools (nasm and cc).

Phase 1: Lexer — Source → Tokens

src/main.rs — Lexer phase
let mut lexer = lexer::Lexer::new(&source);
let (tokens, _lex_errors) = lexer.tokenize();

The lexer reads the raw source string character by character and groups them into tokens. Comments and whitespace are discarded. Each token carries a Span (byte offset, line, column) so error messages can reference precise locations.

Input: A &str containing the program source text.

Output: Vec<SpannedToken> — a flat list of all tokens in file order, ending with Token::Eof.

Errors: Vec<LexError> — currently collected but not reported. The lexer never panics on bad input.

Lex errors are silently collected. The _lex_errors variable is currently ignored. Future versions of the compiler will integrate them into the diagnostics system. Even so, the lexer still records them — error recovery is built into its design.

Phase 2: Parser — Tokens → AST

src/main.rs — Parser phase
let diag = diag::Diagnostics::new();
let mut parser = parser::Parser::new(tokens, diag);
let mut prog = parser.parse_program();
let mut diag = parser.into_diagnostics();

if diag.has_errors() {
    diag.print_all();
    std::process::exit(1);
}

The parser consumes the token stream and builds an Abstract Syntax Tree (AST). It uses recursive-descent parsing with a Pratt parser for expressions. If the input is syntactically invalid (e.g., missing semicolon, unbalanced braces), the parser records errors in the diagnostics system and prints them before aborting.

Input: Vec<SpannedToken> from the lexer.

Output: A Program AST node (the root of the tree).

Errors: If the parser finds syntax errors, the program exits with code 1 after printing all error messages. No further phases run.

Fatal vs. non-fatal: Unlike the lexer, parser errors are fatal. If the source has a syntax error, there is no meaningful AST to pass to later phases. The compiler stops here.

Phase 3: Type Checker — Raw AST → Typed AST

src/main.rs — Type checker phase
let mut types = types::TypeArena::new();
let mut symbols = symbol::SymbolTable::new();
let mut tc = typeck::TypeChecker::new(&mut types, &mut symbols, &mut diag);
tc.check_program(&mut prog);

let annotations = tc.annotations.clone();
let has_errors = tc.diag.has_errors();
drop(tc);

if has_errors {
    diag.print_all();
    std::process::exit(1);
}

The type checker walks the AST and verifies semantic correctness. It resolves names using the SymbolTable, infers and checks types using the TypeArena, and annotates every expression node with its resolved type. This is where errors like "type mismatch" and "undefined variable" are caught.

Input: A Program AST with unresolved names and types.

Output: The same AST is mutated in-place, with type annotations attached. The annotations map (cloned out of the type checker before it is dropped) is used by later phases.

Errors: Type errors are fatal; the compiler exits before code generation.

Ownership handoff: The type checker owns the TypeArena, SymbolTable, and Diagnostics during its lifetime. After drop(tc), the annotations are extracted for the IR builder, and the remaining diagnostics object is reused for later phases.

Phase 4: IR Builder — AST → IR

src/main.rs — IR builder phase
let builder = ir_build::IrBuilder::new(&types, &annotations, &mut diag);
let mut ir_module = builder.build(&prog);

The IR builder lowers the AST into a lower-level Intermediate Representation (IR). The IR uses basic blocks, virtual registers, and a small set of instruction types. This is where structured control flow (if, while) becomes explicit jumps and branches, and where expressions are decomposed into sequences of simple operations.

Input: A typed Program AST + type annotations.

Output: An ir::Module containing functions, each made of basic blocks with IR instructions.

Phase 5: Optimizer — IR → Optimized IR

src/main.rs — Optimizer phase
ir_opt::optimize(&mut ir_module);

The optimizer runs a series of simple but effective optimization passes on the IR:

  • Constant folding: Evaluate compile-time constant expressions (e.g., 2 + 3 → 5)
  • Dead code elimination: Remove instructions whose results are never used
  • Copy propagation: Replace uses of a copy with the original value
  • Unreachable block removal: Delete basic blocks that cannot be reached from any entry point

Input: An ir::Module with unoptimized IR.

Output: The same ir::Module mutated in-place, now with cleaner/faster IR.

Phase 6: Code Generator — IR → Assembly

src/main.rs — Codegen phase
let mut codegen = codegen::Codegen::new(&types);
let asm = codegen.generate(&ir_module);

This phase performs two crucial tasks internally. First, the register allocator (regalloc.rs) maps the virtual registers in the IR to physical x86-64 registers (or spills them to the stack). Then, the code generator (codegen.rs) emits NASM-syntax x86-64 assembly text, including function prologues/epilogues, instruction selection, and the ABI-defined calling convention.

Input: An optimized ir::Module with virtual registers.

Output: A String containing complete NASM assembly source code.

Phase 7: Assembler — Assembly → Object File

src/main.rs — Assembler (nasm)
let status = Command::new("nasm")
    .args(&["-f", "elf64", "-o", &o_path, &asm_path])
    .status()
    .unwrap_or_else(|e| {
        eprintln!("error invoking assembler: {}", e);
        std::process::exit(1);
    });

if !status.success() {
    std::process::exit(1);
}

The compiler shells out to nasm with the -f elf64 flag to produce a 64-bit ELF object file. If nasm fails (bad assembly syntax, missing instructions), the compiler exits with code 1.

Input: A .asm file on disk (NASM syntax).

Output: A .o file (ELF64 relocatable object file).

Phase 8: Linker — Object → Executable

src/main.rs — Linker (cc)
let status = Command::new("cc")
    .args(&["-o", &bin_path, &o_path, "-no-pie"])
    .status()
    .unwrap_or_else(|e| {
        eprintln!("error invoking linker: {}", e);
        std::process::exit(1);
    });

if !status.success() {
    std::process::exit(1);
}

let _ = fs::remove_file(&o_path);

println!("wrote {}", bin_path);

The compiler uses cc (typically GCC or Clang) as a linker. The -no-pie flag produces a position-dependent executable, which avoids complications with position-independent code in the generated assembly. After linking, the intermediate .o file is deleted, and the executable path is printed.

Input: A .o ELF object file.

Output: A statically-linked ELF executable ready to run.

Error Handling Throughout the Pipeline

Each phase handles errors differently:

PhaseError StrategyStops Pipeline?
LexerRecords errors, continues lexingNo (but errors are currently ignored)
ParserRecords diagnostic, aborts with exit(1)Yes
Type CheckerRecords diagnostic, aborts with exit(1)Yes
IR BuilderPanics on internal errors (should not happen after type check)Yes
OptimizerNo errors (sound transformations)N/A
Code GeneratorPanics on internal errorsYes
AssemblerExits with code 1 if nasm failsYes
LinkerExits with code 1 if cc failsYes
Defensive design: Earlier phases should catch all user-facing errors (syntax, type). Later phases (IR builder, codegen) assume well-formed input and use unwrap/panic for internal invariants. If a user hits a panic in a later phase, it is a compiler bug, not a user error.

Common Pitfalls & Debugging Tips

  1. One phase at a time: When debugging, isolate which phase is failing. A parse error prevents type checking. A type error prevents codegen. Check errors at the earliest phase first.
  2. Missing tool in PATH: If nasm or cc is missing, the compiler exits with error invoking assembler/linker. Verify with which nasm and which cc.
  3. Incremental testing: Each phase can be tested independently. The lexer tests run without the parser, the parser tests run on hand-written token streams, and so on.
  4. Examine intermediate output: To see what a phase produces, add a temporary println! or write the output to a file. The assembly file (*.asm) is always written to disk — inspect it to understand codegen's output.
Passes vs. phases: A "phase" is a major stage of the pipeline (e.g., lexer, parser). A "pass" is a single traversal of the IR or AST. The type checker may make multiple passes over the AST. The optimizer runs multiple passes over the IR. Don't confuse the two concepts.

2   Source Locations — Span

Why Source Locations Matter

When a compiler reports an error, it must tell the programmer exactly where the error occurred. "Type mismatch on line 42" is useful. "Type mismatch" alone is useless.

To enable precise error reporting, every token, AST node, and IR instruction in L1S carries a Span — a small data structure that records the source location of the corresponding piece of source code.

Consider this error message:

error[E002]: type mismatch
  --> test.is:3:12
   |
 3 |     let x: i32 = "hello";
   |            ^^^^^^^^^^^ expected i32, found string

The 3:12 and the underlined region come directly from a Span. Without Span, the compiler would be flying blind.

The Span Struct

Every source location in L1S is represented by the Span struct in src/span.rs. It is just 16 bytes — four u32/usize fields — so it is Copy and cheap to pass around:

src/span.rs — The Span struct
use std::fmt;

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct Span {
    pub start: usize,  // byte offset of the first character
    pub end:   usize,  // byte offset one past the last character
    pub line:  u32,    // 1-based line number
    pub col:   u32,    // 1-based column number (of the start position)
}

Field by field:

  • start: usize — The byte offset from the beginning of the source file to the first byte of this span. For a token like func at the very beginning of the file, start = 0.
  • end: usize — The byte offset one past the last byte of this span. The length of the spanned text is end - start. For func, end = 4.
  • line: u32 — The 1-based line number of the start position. Line 1 is the first line. This is what editors display and what users understand.
  • col: u32 — The 1-based column number of the start position. Column 1 is the first character on the line. Tabs count as one column.
Why byte offsets AND line/col? Byte offsets (start, end) are precise and machine-friendly — you can slice the source string with &source[span.start..span.end]. Line/col numbers are human-friendly — they appear in error messages and editor integrations. Having both means the compiler never needs to recalculate line/col from byte offsets (which is O(n) for large files).

Span::new() — The Constructor

pub fn new(start: usize, end: usize, line: u32, col: u32) -> Self {
    Span { start, end, line, col }
}

The constructor simply wraps the four values into a Span. There is no validation — it is the caller's responsibility to ensure end >= start and that the line/col values are consistent with the byte offsets.

This is called from the lexer every time a token is produced:

let span = Span::new(start_pos, end_pos, start_line, start_col);

The lexer records start_pos/start_line/start_col before consuming the token, then records end_pos afterward. The span covers exactly the characters of the token.

Span::dummy() — The "Nowhere" Span

pub fn dummy() -> Self {
    Span { start: 0, end: 0, line: 0, col: 0 }
}

dummy() returns a span with all-zero fields. This is used as a placeholder when no real source location exists:

  • Synthetic AST nodes: The parser may insert implicit nodes (e.g., a default return value) that don't correspond to any source text.
  • Compiler-generated IR: Temporary variables and basic blocks created during optimization have no original source location.
  • Error recovery placeholders: When the parser skips bad tokens, it may insert dummy spans for missing constructs.

Note that line: 0 and col: 0 are invalid for real source locations (which are 1-based), making it easy to identify dummy spans in debug output. A helper like span.is_dummy() would check line == 0.

Span::merge() — Combining Spans

pub fn merge(a: Span, b: Span) -> Span {
    let start = a.start.min(b.start);
    let end = a.end.max(b.end);
    let line = a.line.min(b.line);
    let col = if a.line < b.line { a.col } else { a.col.min(b.col) };
    Span { start, end, line, col }
}

merge() produces a new Span that covers both input spans. This is the workhorse for compound constructs. When the parser builds an AST node like a + b, it merges the spans of the left operand, the operator, and the right operand to produce a span covering the entire expression.

The merge logic:

  • Byte offsets: start takes the minimum, end takes the maximum. The merged span covers from the earliest byte to the latest byte.
  • Line number: Uses the minimum (earliest) line number.
  • Column number: If the spans are on different lines (a.line < b.line), use a.col (the column of the earlier line). If they are on the same line, use the minimum column.

This column logic means the merged span always points to the beginning of the construct, which is the most intuitive location for error messages.

Merge is transitive and associative: You can chain merges to cover arbitrarily large regions: Span::merge(Span::merge(a, b), c) covers everything from the start of a to the end of c. This is exactly how the parser builds spans for deeply nested expressions.

The Display Impl

impl fmt::Display for Span {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}:{}", self.line, self.col)
    }
}

Formatting a Span produces the familiar line:col string (e.g., "3:14"). This is what appears in error messages and debug output. The byte offsets are not shown in the Display impl — they are for internal use — but can be inspected via the struct fields or the Debug derive ({:?}).

        Span { start, end, line, col }
    }

    /// Create a dummy/invalid span for synthetic constructs.
    /// All fields are 0, which is invalid for real source locations.
    pub fn dummy() -> Self {
        Span { start: 0, end: 0, line: 0, col: 0 }
    }

    /// Merge two spans into one that covers both.
    /// Used for compound constructs like binary expressions.
    pub fn merge(a: Span, b: Span) -> Span {
        let start = a.start.min(b.start);
        let end = a.end.max(b.end);
        let line = a.line.min(b.line);
        let col = if a.line < b.line { a.col }
                         else { a.col.min(b.col) };
        Span { start, end, line, col }
    }
}

Why Both Byte Offsets AND Line/Col?

New compiler authors often ask: why store both (start, end) and (line, col)? Couldn't we just store byte offsets and compute line/col on demand?

The short answer: Performance and convenience.

Byte offsets are essential for slicing into the source string. When the error formatter needs to extract the offending line and underline the error, it does:

let text = &source[span.start..span.end];  // O(1) slice

This is how the diagnostics system retrieves the exact text of a token or expression.

Line/col are essential for error messages. Computing line/col from byte offsets requires scanning the source from the beginning of the file to find all newlines before the given offset — an O(n) operation. For a large file with many errors, this would be repeated hundreds of times.

RepresentationGood ForBad For
Byte offsets (start, end)Source slicing, range comparison, mergeDisplaying to humans
Line/colError messages, editor integrationsComputationally expensive to produce from offsets
BothBest of both worlds — computed once by the lexer, used everywhere

The lexer tracks both simultaneously as it scans, so the cost is paid exactly once per token.

How Span Merging Works for Compound Constructs

The parser uses Span::merge whenever it builds a composite AST node. Consider parsing the expression 1 + 2 * 3:

  1. Lexer produces spans for each token:
    TokenSpan (byte range)Span (line:col)
    1[0, 1)1:1
    +[2, 3)1:3
    2[4, 5)1:5
    *[6, 7)1:7
    3[8, 9)1:9
  2. The parser recognizes 2 * 3 as a BinaryExpr(Mul) and merges: 
    mul_span = Span::merge(2_span, Span::merge(*_span, 3_span))
// Result: [4, 9), line: 1, col: 5
  3. The parser recognizes 1 + (2 * 3) as a BinaryExpr(Add) and merges: 
    add_span = Span::merge(1_span, Span::merge(+_span, mul_span))
// Result: [0, 9), line: 1, col: 1

The merged span for the entire expression 1 + 2 * 3 is 1:1, pointing to the first character. If the type checker later reports an error on this expression, the span points to the beginning of the entire construct.

Common Pitfalls & Debugging Tips

  1. Spans pointing to wrong locations: The lexer must record start_pos/start_line/start_col before consuming a token, and end_pos after. If you record the start position after consuming the first character, all spans will be off by at least one character.
  2. Zero vs one-based: Line/col are 1-based. A common bug is initializing line: 0, col: 0 for the first character, which makes error messages show 0:0 instead of 1:1. Only Span::dummy() should have 0 values.
  3. UTF-8 confusion: Byte offsets are byte positions, not character positions. source[span.start..span.end] works correctly only if both indices fall on UTF-8 character boundaries. The lexer uses CharIndices and len_utf8() to guarantee this.
  4. Merging dummy spans: If you merge a real span with a dummy span (line: 0), the result will have line: 0, which is invalid. Guard against this with if a.line == 0 { return b; } if b.line == 0 { return a; }.
  5. End-exclusive convention: end is the byte offset one past the last character of the span. Text extracted via &source[s.start..s.end] is correct. If you accidentally treat end as inclusive, every span will be one byte too long.
Debugging spans: Add this helper to your debugging arsenal: 
fn dump_span(source: &str, s: &Span) {
    println!("Span: {}:{}..{}:{} (bytes {}..{}) text={:?}",
             s.line, s.col, s.end, s.col + (s.end - s.start),
             s.start, s.end, &source[s.start..s.end]);
}
This prints the location, byte range, and the actual source text covered by the span, making it easy to verify correctness.

3   The Token System

What is a Token?

A token is the smallest meaningful unit of a program. Where the source file is a flat sequence of characters, a token is a structured piece of data that says "this is a keyword," "this is an identifier," "this is an operator." The lexer's job is to convert characters to tokens; the parser's job is to convert tokens to an AST.

Every compiler has a token type. L1S defines its tokens in a single file src/token.rs — the single source of truth shared by the lexer, parser, and all downstream tools.

Why a dedicated token file? Both the lexer (which produces tokens) and the parser (which consumes them) need to reference the same token types. Putting them in token.rs avoids circular dependencies and keeps the token definitions independent of both the lexing and parsing implementations.

The SpannedToken Wrapper

Tokens rarely travel alone. Every token needs its source location so errors can point back to the right place. The SpannedToken struct pairs a token with its Span:

src/token.rs — SpannedToken
pub struct SpannedToken {
    pub token: Token,
    pub span: Span,
}

impl SpannedToken {
    pub fn new(token: Token, span: Span) -> Self {
        SpannedToken { token, span }

}
Why a separate struct instead of embedding Span in Token? Two reasons. First, not every context needs a span — for example, tests that check token sequences don't care about positions. Second, Span is a property of the token's occurrence in source, not of the token itself. A Token::Func variant doesn't inherently have a position; a SpannedToken with Token::Func does. This separation models the domain correctly.

fn lex_identifier_or_keyword(&mut self) -> Token {
    let mut ident = String::new();
    while let Some(c) = self.peek() {
        if c.is_alphanumeric() || c == '_' {
            self.advance();
            ident.push(c);
        } else { break; }
    }
    keyword_token(&ident).unwrap_or(Token::Ident(ident))
}
Note that Func matches both "func" and "Func". L1S supports capitalized function names for type constructor functions (like StructName(val)), while func remains the keyword form. Both map to the same Token::Func variant; the parser distinguishes them by context.

Why keyword_token Is the Right Abstraction

The design of having a centralized keyword_token function is a key architectural decision. Consider the alternatives:

  • No keyword classification in the lexer: Lex every alphanumeric sequence as Ident and let the parser decide. This forces the parser to carry a keyword table, complicates error messages ("expected keyword 'func', found identifier 'func'"?), and makes the token stream less informative.
  • Hardcode keyword detection in the lexer: Put a giant match directly inside lex_identifier_or_keyword. This works but couples the lexer to the keyword list, making it harder to reuse the token definitions in other tools (formatters, linters, syntax highlighters).
  • Use a hashmap: Populate a HashMap<&str, Token> at startup. This is more flexible but adds allocation and initialization overhead, and the match-based approach is faster (the compiler optimizes string matches into a perfect hash or jump table).

The keyword_token approach is the sweet spot: it's a standalone, testable, pure function that the lexer delegates to, and that any other tool can also use.

Derive Macros: Why #[derive(Debug, Clone, PartialEq)]?

The Token enum derives three traits:

  • Debug: Required for assert_eq! error messages, which print both sides on failure. Also useful for debugging the lexer output.
  • Clone: The parser often needs to inspect a token without consuming it (lookahead). Clone lets us keep a copy.
  • PartialEq: Required for test assertions and for the parser's token matching. Note that Eq is NOT derived — NumLiteral(String) can never be compared reflexively in a meaningful way for the compiler's purposes, but PartialEq with no Eq is sufficient for all our use cases.
Why not derive Eq? Rust's Eq trait requires reflexivity: a == a must always be true. For all of our token variants this holds, so we could derive it. However, by convention, we only derive what we need. Adding Eq later is a non-breaking change. This is a minor design choice, but it's worth noting for awareness.

The Big Picture: How Tokens Flow

Data flow through the compiler pipeline:

↓

↓

↓

Every stage after the lexer works with Token variants through the SpannedToken wrapper. The parser never sees raw characters — it only sees the structured token stream. This separation is what makes the compiler modular: you could replace the lexer entirely (e.g., with a hand-written scanner or a lexer generator like logos) without changing anything downstream, as long as the same Token enum is produced.

Common Pitfalls & Debugging Tips

  1. Forgetting to add a new keyword in two places: Adding Loop to the Token enum but not to keyword_token() means the lexer will produce Ident("loop") instead of Token::Loop. Always update both the variant and the match arm.
  2. Case sensitivity: keyword_token matches both "func" and "Func" for Token::Func, but "Struct" is NOT a keyword (only "struct" is). If you change this, update the tests too.
  3. Derive mismatch: If you add a field to a token variant that doesn't implement PartialEq or Debug, the compiler will error. Make sure new fields satisfy the trait bounds.
  4. Span is not Clone: Span derives Clone + Copy. If you change this, SpannedToken stops being clonable and the parser breaks.
  5. Testing empty input: Always test that keyword_token("") returns None. Empty strings should never match a keyword.
Debugging tip: If the parser is producing unexpected errors, dump the token stream first. Add a quick dbg! to the parser's token consumption loop to see what tokens the lexer actually produced. Often the bug is in the lexer (wrong token for a keyword) rather than in the parser.

4   The Lexer

What is Lexing?

Before a compiler can understand the structure of a program, it needs to break the raw source text into meaningful chunks called tokens. This process is lexical analysis (or lexing/tokenizing).

Think of lexing like splitting a sentence into words and punctuation:

Input: func main(i32) { return 42; }

Output tokens:
Func Ident("main") OpenParen I32 CloseParen OpenBrace Return NumLiteral("42") Semicolon CloseBrace Eof

The lexer does not understand whether 42 is used correctly, whether main is a function or a variable, or whether the braces match — that's the parser's job. The lexer's job is simply to group characters into the predefined token types from token.rs.

What Makes a Good Lexer?

  1. Error recovery: Never crash on bad input. Record the error and keep going.
  2. Accurate positions: Every token must carry its exact source location (byte offset, line, column) so errors can point back to the right place.
  3. Deterministic: Given the same input, always produce the same tokens.
  4. Fast: Lexing is O(n) — every character is examined exactly once.

The LexError Type

Before we look at the lexer itself, we define what a lex error looks like. It carries a human-readable message and a Span pointing to the offending character:

src/lexer.rs — LexError
use crate::span::Span;
use crate::token::{keyword_token, SpannedToken, Token};

/// An error produced during lexical analysis.
/// The span points to the exact character(s) that caused the error.
pub struct LexError {
    pub message: String,
    pub span: Span,
}

impl LexError {
    pub fn new(message: impl Into<String>, span: Span) -> Self {
        LexError { message: message.into(), span }
    }
}

Simple but effective. The error format impl Into<String> means callers can pass either a &str or a String without explicit conversion.

Why not use Diagnostics directly? The lexer precedes the diagnostics system in the pipeline. Lex errors are collected in a Vec<LexError> and converted/reported during later stages. This keeps the lexer self-contained and testable.

The Lexer Struct

The core of the lexer is a stateful struct that wraps the source string and tracks position:

src/lexer.rs — Lexer struct
pub struct Lexer<'a> {
    // The original source text. We keep this so we can slice it for error messages.
    source: &str,
    // A peekable iterator over (byte_offset, char) pairs. Peekable gives
    // us single-character lookahead without consuming the character.
    chars: Peekable<CharIndices<'a>>,
    // Current byte offset in the source
    pos: usize,
    // Current line number (1-based)
    line: u32,
    // Current column number (1-based)
    col: u32,
    // Accumulated errors. We keep going even after finding errors.
    errors: Vec<LexError>,
}

Let's examine each field:

  • source: The full source text. We never modify it; we just read from it through the chars iterator.
  • chars: std::str::CharIndices wrapped in Peekable. CharIndices yields (usize, char) pairs — the byte offset and the Unicode character. Wrapping it in Peekable lets us look at the next character without consuming it, which is essential for recognizing multi-character operators like == and !=.
  • pos: Current byte position. Updated by advance(). Used for Span::start.
  • line, col: Human-readable position tracking. Updated by advance(). Line and column are 1-based (first line is line 1, first column is column 1).
  • errors: Errors collected during tokenization. Instead of panicking or returning Result, the lexer records errors and continues, allowing the compilation to report all errors in one pass.

Construction

The constructor initializes all fields:

impl<'a> Lexer<'a> {
    pub fn new(source: &str) -> Self {
        Lexer {
            source,
            chars: source.char_indices().peekable(),
            pos: 0,
            line: 1,
            col: 1,
            errors: Vec::new(),
        }
    }

Note: we start at line: 1, col: 1, not 0, 0. This matches what editors and compilers conventionally display.

The Main Entry Point: tokenize()

The tokenize method is the public interface. It loops, skipping whitespace and comments, then lexing one token at a time until the source is exhausted:

src/lexer.rs — tokenize()
pub fn tokenize(&mut self) -> (Vec<SpannedToken>, Vec<LexError>) {
    let mut tokens = Vec::new();

    loop {
        self.skip_whitespace();
        self.skip_comment();
        self.skip_whitespace();

        if self.chars.peek().is_none() {
            break;
        }

        let start_pos = self.pos;
        let start_line = self.line;
        let start_col = self.col;

        let token = self.lex_token();

        let end_pos = self.pos;
        let span = Span::new(start_pos, end_pos, start_line, start_col);

        match token {
            Some(tok) => {
                if tok == Token::Eof {
                    tokens.push(SpannedToken::new(tok, span));
                    break;
                }
                tokens.push(SpannedToken::new(tok, span));
            }
            None => {
                // Unrecognized character: record error, advance past it
                if let Some((_, c)) = self.chars.next() {
                    let len = c.len_utf8();
                    self.pos += len;
                    self.col += 1;
                    let span = Span::new(start_pos, self.pos, start_line, start_col);
                    self.errors.push(LexError::new(
                        format!("unexpected character '{}'", c), span));
                }
            }
        }
    }

    // Ensure we always emit an Eof token
    if tokens.last().map(|t| &t.token) != Some(&Token::Eof) {
        let span = Span::new(self.pos, self.pos, self.line, self.col);
        tokens.push(SpannedToken::new(Token::Eof, span));
    }

    (tokens, std::mem::take(&self.errors))
}

Step-by-step walkthrough:

  1. Skip whitespace and comments: Before we try to read a token, skip over any whitespace ( , \t, \n) and comments (// ...). Comments are treated as whitespace — they produce no tokens.
  2. Check for EOF: If no characters remain, break out of the loop.
  3. Record start position: Save the current pos, line, col before consuming the token.
  4. Lex one token: Call lex_token(), which returns Option<Token>.
  5. Create Span: The span covers from the start position to wherever we ended up after lexing.
  6. Handle the result:
    • Some(tok) → Push the token with its span. If it's EOF, stop.
    • None → Unrecognized character. Advance past it, record error.
  7. Guarantee EOF: After the loop, ensure we emitted at least one Token::Eof. An empty source still produces [Eof].
  8. Return: Take the errors out of self (using std::mem::take) and return them alongside the tokens.
Why skip_comment after skip_whitespace? Consider // comment\nfunc. We skip whitespace (nothing before /), then skip_comment detects // and skips to the newline. Then we skip whitespace again (the newline itself). This two-pass approach correctly handles edge cases like whitespace between // and the following token.

Position Tracking: advance()

The advance() method consumes one character from the iterator and updates all position fields. This is the only place where position tracking happens, which keeps it correct and consistent:

fn advance(&mut self) -> Option<char> {
    match self.chars.next() {
        Some((_, c)) => {
            let len = c.len_utf8();
            self.pos += len;           // byte offset advances by char's UTF-8 length
            if c == '\n' {
                self.line += 1;             // new line
                self.col = 1;               // reset column
            } else {
                self.col += 1;               // same line, next column
            }
            Some(c)
        }
        None => None,
    }
}

peek() is the complement — it looks at the next character without consuming it:

fn peek(&mut self) -> Option<char> {
    self.chars.peek().map(|&(_, c)| c)
}
UTF-8 safety: CharIndices yields complete Unicode characters, not raw bytes. The len_utf8() method tells us how many bytes the character occupies, so pos always points to the correct byte boundary. This means emoji, accented characters, and CJK characters in comments or strings are handled correctly.

Skipping Whitespace and Comments

Whitespace is straightforward — consume any ASCII whitespace character:

fn skip_whitespace(&mut self) {
    loop {
        match self.peek() {
            Some(c) if c.is_ascii_whitespace() => { self.advance(); }
            _ => break,
        }
    }
}

Comments use a "peek and decide" pattern. Since comments start with /, which is also the division operator, we need lookahead:

fn skip_comment(&mut self) {
    if self.peek() == Some('/') {
        // Save state in case this is NOT a comment
        let saved = self.chars.clone();
        let saved_pos = self.pos;
        let saved_line = self.line;
        let saved_col = self.col;

        self.advance();
        if self.peek() == Some('/') {
            // Line comment: consume until newline or EOF
            while let Some(c) = self.peek() {
                if c == '\n' { break; }
                self.advance();
            }
        } else {
            // Not a comment! Restore state so '/' can be lexed as Slash
            self.chars = saved;
            self.pos = saved_pos;
            self.line = saved_line;
            self.col = saved_col;
        }
    }
}

The save/restore pattern is the key insight here. When we see /, we don't know yet if it's a comment or the division operator. We tentatively advance past the /, and then look at the next character. If it's another /, we consume the rest of the line. If not, we restore the saved state, and the / will be picked up by lex_token() as Token::Slash.

The Token Dispatch: lex_token()

This method examines the current character and dispatches to the appropriate specialized lexer function. It's a giant match on the first character:

src/lexer.rs — lex_token() dispatch
fn lex_token(&mut self) -> Option<Token> {
    let c = self.peek()?;

    match c {
        // Single-character operators
        '+' => { self.advance(); Some(Token::Plus) }
        '-' => {
            self.advance();
            if self.peek() == Some('>') {
                self.advance();
                Some(Token::Arrow)       // -> arrow operator
            } else {
                Some(Token::Minus)
            }
        }
        '*' => { self.advance(); Some(Token::Star) }
        '/' => { self.advance(); Some(Token::Slash) }
        '%' => { self.advance(); Some(Token::Percent) }

        // Multi-character operators with &
        '&' => {
            self.advance();
            if self.peek() == Some('&') {
                self.advance();
                Some(Token::AmpAmp)     // && logical and
            } else {
                Some(Token::Amp)           // & bitwise and / address-of
            }
        }

        // Equality and comparison (and their multi-char variants)
        '=' => {
            self.advance();
            if self.peek() == Some('=') {
                self.advance();
                Some(Token::EqEq)       // == equality
            } else {
                Some(Token::Eq)            // = assignment
            }
        }

        // ... '!', '<', '>', and other operators follow the same pattern

        // Strings, chars, numbers, identifiers
        '"' => Some(self.lex_string()),
        '\'' => Some(self.lex_char()),
        '0'..='9' => Some(self.lex_number()),
        c if is_ident_start(c) => Some(self.lex_identifier_or_keyword()),

        // Unknown character: return None so tokenize() records an error
        _ => None,
    }
}

The dispatch logic:

  • Single characters like +, *, (, ) → advance and return the corresponding token variant.
  • Ambiguous operators like - (could be ->), = (could be ==): advance, peek at the next char, then decide.
  • Complex literals (", ', digits, letters): delegate to a specialized method.
  • Fallthrough (_ => None): the character is unrecognized. Returns None so tokenize() records an error and advances.

Number Lexing: lex_number()

L1S supports decimal, hexadecimal, binary, and octal integer literals, plus underscores for readability:

fn lex_number(&mut self) -> Token {
    let mut raw = String::new();

    // Check for radix prefix (0x, 0b, 0o)
    if self.peek() == Some('0') {
        self.advance();
        raw.push('0');
        match self.peek() {
            Some('x' | 'X') => {
                self.advance(); raw.push('x');
                while let Some(c) = self.peek() {
                    if c.is_ascii_hexdigit() || c == '_' {
                        self.advance(); raw.push(c);
                    } else { break; }
                }
            }
            Some('b' | 'B') => {
                self.advance(); raw.push('b');
                while let Some(c) = self.peek() {
                    if c == '0' || c == '1' || c == '_' {
                        self.advance(); raw.push(c);
                    } else { break; }
                }
            }
            Some('o' | 'O') => {
                self.advance(); raw.push('o');
                while let Some(c) = self.peek() {
                    if c.is_ascii_digit() && c < '8' || c == '_' {
                        self.advance(); raw.push(c);
                    } else { break; }
                }
            }
            _ => {
                // Plain 0 followed by digits: decimal
                while let Some(c) = self.peek() {
                    if c.is_ascii_digit() || c == '_' {
                        self.advance(); raw.push(c);
                    } else { break; }
                }
            }
        }
    } else {
        // Non-zero decimal number
        while let Some(c) = self.peek() {
            if c.is_ascii_digit() || c == '_' {
                self.advance(); raw.push(c);
            } else { break; }
        }
    }

    Token::NumLiteral(raw)
}
Note: The lexer does not validate the number format. It just collects the raw string. A token like 0xGG would be lexed as a hex literal even though GG is not valid hex. The parser or a later stage should validate the actual value. This keeps the lexer simple and fast.

String Lexing: lex_string()

Strings are delimited by double quotes and support common escape sequences:

fn lex_string(&mut self) -> Token {
    let mut content = String::new();
    self.advance(); // consume opening '"'

    loop {
        match self.peek() {
            None => {
                // Premature EOF: unclosed string
                let span = Span::new(self.pos, self.pos, self.line, self.col);
                self.errors.push(LexError::new("unclosed string literal", span));
                break;
            }
            Some('"') => {
                self.advance(); // consume closing '"'
                break;
            }
            Some('\\') => {
                self.advance(); // consume backslash
                match self.advance() {
                    Some('n') => content.push('\n'),
                    Some('t') => content.push('\t'),
                    Some('\\') => content.push('\\'),
                    Some('"') => content.push('"'),
                    Some('0') => content.push('\0'),
                    Some('x') => {
                        let hex = self.lex_hex_escape();
                        content.push(hex as char);
                    }
                    Some(c) => {
                        // Invalid escape sequence
                        let span = Span::new(self.pos, self.pos, self.line, self.col);
                        self.errors.push(LexError::new(
                            format!("invalid escape sequence '\\{}'", c), span));
                        content.push(c);
                    }
                    None => {
                        // Unterminated escape at EOF
                        let span = Span::new(self.pos, self.pos, self.line, self.col);
                        self.errors.push(LexError::new("unterminated escape sequence", span));
                        break;
                    }
                }
            }
            Some(_) => {
                // Normal character: push it
                self.advance().map(|c| content.push(c));
            }
        }
    }

    Token::StrLiteral(content)
}

Note that string content is stored with escapes resolved. "\n" becomes Token::StrLiteral("\n") (a single newline character), not Token::StrLiteral("\\n"). This means the parser and later stages never need to handle escape sequences.

Character Lexing: lex_char()

Character literals follow the same pattern as strings but produce a single byte:

fn lex_char(&mut self) -> Token {
    self.advance(); // consume opening '\''
    let c = match self.peek() {
        None => { /* error: unclosed */ return Token::CharLiteral(0); }
        Some('\\') => { /* escape sequence */ }
        Some(c) => { self.advance(); c }
    };
    // Expect closing '\''
    match self.peek() {
        Some('\'') => { self.advance(); Token::CharLiteral(c as u8) }
        _ => { /* error: unclosed */ Token::CharLiteral(c as u8) }
    }
}

Note the as u8 cast: character literals are single bytes. Multi-byte characters are not supported (the lexer explicitly errors on them).

The Hex Escape Helper

Both strings and characters use \xHH hex escapes. The helper reads at most two hex digits:

fn lex_hex_escape(&mut self) -> u8 {
    let mut hex = String::with_capacity(2);
    for _ in 0..2 {
        match self.peek() {
            Some(c) if c.is_ascii_hexdigit() => {
                self.advance(); hex.push(c);
            }
            _ => break,
        }
    }
    u8::from_str_radix(&hex, 16).unwrap_or(0)
}

Identifier and Keyword Lexing

The most elegant part of the lexer is how it handles identifiers and keywords:

fn lex_identifier_or_keyword(&mut self) -> Token {
    let mut ident = String::new();
    while let Some(c) = self.peek() {
        if c.is_alphanumeric() || c == '_' {
            self.advance();
            ident.push(c);
        } else { break; }
    }
    keyword_token(&ident).unwrap_or(Token::Ident(ident))
}

The critical call is keyword_token(&ident). This single line is the bridge between the lexer and the token definitions:

  • If the string matches a keyword (like "func", "let", "i32", "bool"), it returns the corresponding token variant (Token::Func, Token::Let, etc.).
  • If it doesn't match any keyword, unwrap_or falls back to Token::Ident(ident).

The helper is_ident_start defines what characters can begin an identifier:

fn is_ident_start(c: char) -> bool {
    c.is_ascii_alphabetic() || c == '_'
}

Note: digits cannot start an identifier. 123foo is a number (123) followed by an identifier (foo). Well, actually the lexer will lex 123foo as NumLiteral("123foo") because lex_number consumes any alphanumeric characters after the digits. This is a known simplification — the parser or type checker should catch invalid numeric literals.

Single-function keyword dispatch: The keyword_token function in token.rs is the single source of truth for the keyword list. Adding a new keyword requires exactly two changes: add a variant to the Token enum, and add a match arm to keyword_token. The lexer code never needs to change.

Error Recovery Philosophy

The L1S lexer employs a best-effort error recovery strategy:

  1. Never panic: The lexer should never crash, no matter how malformed the input.
  2. Record and continue: When an error is encountered (invalid character, unclosed string, bad escape), the lexer records the error, advances past the bad input, and continues lexing. This maximizes the number of errors reported per compilation.
  3. The caller decides: Errors are returned as Vec<LexError> alongside the tokens. The caller (the compiler driver) decides whether to abort or continue. Currently the driver ignores lex errors for simplicity, but they could be reported alongside parse errors.

Common Pitfalls & Debugging Tips

  1. Position off by one: Always initialize line: 1, col: 1 (not 0). Check your advance() method: newlines should increment line and reset col to 1.
  2. Comments confused with division: The skip_comment save/restore pattern is critical. Without it, / would be consumed before lex_token could see it.
  3. Unclosed strings: Always handle the case where EOF is reached while inside a string. Record an error, return the partial content, and continue.
  4. Multi-byte characters: Use CharIndices rather than raw byte indexing to correctly handle UTF-8. Never assume one byte = one character.
  5. Test error paths: It's easy to only test happy paths. Make sure to test invalid input, empty input, and edge cases like "\x" (incomplete hex escape).
Debugging the lexer: Add a println! to tokenize() that dumps each token as it's produced: 
println!("token: {:?} at {}", tok, span);
This is the single most useful debugging tool for the lexer. Run it on your test input and verify the token stream matches your expectations.

What is an AST?

An Abstract Syntax Tree (AST) is a tree representation of the syntactic structure of source code. Where the lexer produces a flat sequence of tokens, the parser arranges those tokens into a tree that captures the grammatical structure and nesting of the program.

The word "abstract" is key: the AST omits details that are present in the concrete syntax (parens, semicolons, braces) and preserves only what is semantically meaningful. For example, the parser discards the opening and closing braces of a block — the AstBlock struct stores only the statements inside.

Where Span Lives in Every Node

Every AST node carries a Span field. This is the most consistent pattern in the entire design:

  • Enums like AstType, AstExpr store the span alongside other data in each variant.
  • Structs like AstFuncDecl, AstBlock, AstParam have a dedicated pub span: Span field.
  • The span() accessor method on AstType and AstExpr extracts the span regardless of which variant is active.
Why thread Span through every node? Because error messages need to point to specific locations. When the type checker says "type mismatch in return statement," it needs to point to the exact expression. Without Span on every node, the best you can do is "error somewhere on line 5." With Span, every sub-expression carries its source location.

Design Choices Explained

Why separate PrimType from AstType?

Without PrimType, AstType would need 13 separate variants just for primitives:

// Without PrimType (worse):
enum AstType { Void, Bool, I32, ... }

With PrimType, we have a single AstType::Prim(PrimType, Span) variant. This collapses 13 into 1. Any code that handles all primitives can match on AstType::Prim(_, _) rather than listing all 13 variants.

Why AstExpr::Cast rather than a function call?

The cast keyword is a built-in language construct, not a library function. It gets its own AST node so the type checker can verify the cast is valid (e.g., you can't cast a pointer to bool). If cast were modeled as a regular function call, the type checker would need special-casing anyway.

Why does AstField store type_: AstType rather than String?

At parse time, the field type is resolved to an AstType. A field declared as x: i32 produces AstType::Prim(PrimType::I32, ...). A field declared as y: MyStruct produces AstType::Named("MyStruct", ...). The semantic analyzer later resolves Named to the actual struct definition.

Why AstStmt::Expr(AstExpr)?

Expression-statements allow expressions to be used as statements. This is what makes f(); and a = b + c; valid statements. The expression is evaluated for its side effects; its result is discarded.

Common Pitfalls & Debugging Tips

  1. Missing Span on a new node: Every new AST node must carry a Span. If you add a new variant to AstExpr without a Span field, the span() accessor will fail to compile. Add it to both the variant and the match arms.
  2. Box placement: When adding a recursive variant (e.g., a new unary expression), wrap the inner AstExpr in Box. Without it, Rust will error with "recursive type has infinite size."
  3. Cloning the AST: If you need to clone an AstExpr tree, every node must implement Clone. The #[derive(Clone)] on all types ensures this, but if you add a field that is not Clone, the derives will fail.
  4. Vec vs. Box: Use Vec for variable-length lists (function params, struct fields, block statements). Use Box for single recursive children (loop body, unary operand).
  5. Span consistency: The span should cover the entire syntactic construct from the first token to the last. For AstExpr::Binary, the span covers from left's start to right's end. For AstFuncDecl, from func to the closing }. Off-by-one span errors manifest as confusing error messages downstream.
Debugging tip: Implement a Display or debug-print function for your AST to visualize parse results. A simple recursive function that prints the tree with indentation is invaluable for verifying that the parser produced the correct structure. Something like println!("{:#?}", ast_program) with Rust's debug formatting can get you surprisingly far.

6   The Parser

From Tokens to Syntax Trees

The lexer gives us a flat list of tokens. The parser's job is to impose structure on that list — to recognize that func main(i32) { return 42; } is a function declaration containing a return statement, not just 11 tokens in a row.

L1S uses a hand-written recursive descent parser with a Pratt parser for expressions. This is a classic combination: recursive descent handles the irregular grammar of declarations and statements, while Pratt's algorithm elegantly handles operator precedence without a separate operator grammar.

Parser pipeline:

[Tokens] → parse_program() → parse_declaration() → parse_func_decl() / parse_struct_decl() / ...
        → parse_statement() → parse_if() / parse_while() / ...
        → parse_expr() → Pratt (prefix + infix) → AstProgram

The Parser Struct

The parser wraps the token stream and a diagnostics accumulator:

src/parser.rs — Parser struct
pub struct Parser {
    tokens: Vec<SpannedToken>,
    pos: usize,
    diag: Diagnostics,
}

impl Parser {
    pub fn new(tokens: Vec<SpannedToken>, diag: Diagnostics) -> Self {
        Parser { tokens, pos: 0, diag }
    }

    pub fn into_diagnostics(self) -> Diagnostics {
        self.diag
    }
}

Fields:

  • tokens: The full token stream from the lexer. Owned as Vec<SpannedToken> so the parser has full control and can move data out.
  • pos: Current index into the token stream. The "cursor."
  • diag: Accumulated errors and warnings. Consumed at the end via into_diagnostics().

Cursor Helpers

These tiny methods are the building blocks of the entire parser. Understanding them is understanding how every parse method works:

src/parser.rs — cursor operations
fn is_at_end(&self) -> bool {
    self.peek_token() == Token::Eof
}

fn peek(&self) -> &SpannedToken {
    &self.tokens[self.pos]
}

fn peek_token(&self) -> Token {
    if self.pos >= self.tokens.len() {
        return Token::Eof;
    }
    self.tokens[self.pos].token.clone()
}

fn peek_nth(&self, n: usize) -> Token {
    let idx = self.pos + n;
    if idx >= self.tokens.len() { return Token::Eof; }
    self.tokens[idx].token.clone()
}

fn advance(&mut self) -> &SpannedToken {
    let t = &self.tokens[self.pos];
    self.pos += 1;
    t
}

Check, match, expect — the three modes of token consumption:

fn check(&self, tok: &Token) -> bool {
    self.peek_token() == *tok
}

fn match_token(&mut self, tok: &Token) -> bool {
    if self.check(tok) { self.advance(); true } else { false }
}

fn expect(&mut self, tok: &Token) -> Result<Span, Span> {
    if self.check(tok) { let span = self.peek().span; self.advance(); Ok(span) }
    else { Err(self.current_span()) }
}

fn expect_ident(&mut self) -> Result<String, Span> {
    if matches!(self.peek_token(), Token::Ident(_)) {
        let token = self.advance();
        if let Token::Ident(name) = &token.token { Ok(name.clone()) }
        else { unreachable!() }
    } else { Err(self.current_span()) }
}
check vs match_token vs expect: check is a pure predicate (no mutation). match_token consumes the token if it matches and returns bool — useful for optional syntax like the comma between parameters. expect consumes the token if it matches, but signals failure via Result — the caller can decide whether to emit an error or insert a synthetic token.

Error Recovery Pattern

The most important design pattern in the parser is how expect_ident and expect integrate with error recovery. Consider parse_func_decl:

fn parse_func_decl(&mut self, pub_: bool) -> AstFuncDecl {
    let span = self.expect(&Token::Func).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    // ... continue parsing with synthetic name "_"
}

The unwrap_or_else calls are deliberate: they let the parser continue after an error by inserting a synthetic value. A missing identifier becomes "_", a missing closing brace gets a dummy span. The error is already recorded in the diagnostics, but the parser can keep going to find more errors. This maximizes the number of errors reported per compilation.

Infinite Loop Guards

Every loop that consumes input has a panic guard. If the parser ever gets stuck (e.g., a bug causes it to loop without advancing), the guard fires after 10,000 iterations:

fn parse_program(&mut self) -> AstProgram {
    let mut decls = Vec::new();
    let mut guard = 0u32;
    while !self.is_at_end() {
        guard += 1;
        if guard > 10000 {
            panic!("INFINITE LOOP in parse_program at pos={}", self.pos);
        }
        if let Some(decl) = self.parse_declaration() {
            decls.push(decl);
        } else {
            self.advance(); // skip on parse failure
        }
    }
    AstProgram { decls }
}

The guard + self.advance() on failure is the double lock: even if parse_declaration returns None without advancing, we force-advance past the current token to break the loop.

Declaration Parsing: parse_declaration()

This is the top-level dispatch for the program's declarations. It looks at the current token and routes to the appropriate sub-parser:

src/parser.rs — declaration dispatch
fn parse_declaration(&mut self) -> Option<AstDecl> {
    let is_pub = self.match_token(&Token::Pub);

    match self.peek_token() {
        Token::Struct => Some(AstDecl::Struct(self.parse_struct_decl(is_pub))),
        Token::Enum   => Some(AstDecl::Enum(self.parse_enum_decl(is_pub))),
        Token::Spec   => Some(AstDecl::Spec(self.parse_spec_decl(is_pub))),
        Token::Impl   => { /* impl cannot be pub */ Some(AstDecl::Impl(self.parse_impl_decl())) }
        Token::Func   => Some(AstDecl::Func(self.parse_func_decl(is_pub))),
        Token::Typedef => Some(AstDecl::Typedef(self.parse_typedef_decl(is_pub))),
        Token::Import => Some(AstDecl::Import(self.parse_import())),
        Token::Static => Some(AstDecl::StaticVar(self.parse_static_var_decl())),
        _ => { if is_pub { self.error("expected declaration after 'pub'", self.current_span()); } None }
    }
}

The pub keyword is treated as a prefix modifier: match_token(&Token::Pub) consumes it if present, and the is_pub flag is forwarded to the sub-parser. Some declarations (impl, import, static) reject pub and record an error if it appears.

parse_func_decl — Function Declarations

fn parse_func_decl(&mut self, pub_: bool) -> AstFuncDecl {
    let span = self.expect(&Token::Func).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenParen).unwrap_or_else(|_| Span::dummy());
    let return_type = self.parse_type();
    let params = if self.match_token(&Token::Comma) {
        self.parse_params()
    } else { Vec::new() };
    self.expect(&Token::CloseParen).unwrap_or_else(|_| Span::dummy());
    let body = self.parse_block();
    AstFuncDecl { span, pub_, name, return_type, params, body }
}

Walkthrough: func main(i32, x: i32, y: i32) { ... }

  1. Consume func keyword, get the span.
  2. Read the name via expect_ident → "main".
  3. Open paren → the return type is parsed first (i32).
  4. Comma? If a comma follows the return type, there are named parameters. parse_params reads x: i32, y: i32.
  5. Close paren, then parse the body block.
Why return type before parameters? This is unusual. Most C-like languages put the return type before the function name. L1S places it first so that func main(i32, argc: i32, argv: @char) reads naturally: the return type is the first type after the opening paren, then named parameters follow. A bare type like i32 with no comma/name means "zero parameters, return type i32."

parse_struct_decl

fn parse_struct_decl(&mut self, pub_: bool) -> AstStructDecl {
    let span = self.expect(&Token::Struct).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenBrace).unwrap_or_else(|_| Span::dummy());
    let fields = self.parse_fields();
    self.expect(&Token::CloseBrace).unwrap_or_else(|_| Span::dummy());
    AstStructDecl { span, pub_, name, fields }
}

Fields are parsed as name: type; pairs. The parse_fields method breaks when it sees CloseBrace or EOF, so missing semicolons on the last field are tolerated.

parse_enum_decl

fn parse_enum_decl(&mut self, pub_: bool) -> AstEnumDecl {
    let span = self.expect(&Token::Enum).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    // Optional [noprefix] attribute
    let mut noprefix = false;
    if self.match_token(&Token::OpenBracket) {
        loop {
            if self.match_token(&Token::Noprefix) { noprefix = true; }
            else if self.peek_token() == Token::CloseBracket { break; }
            else { self.error("expected enum modifier", self.current_span()); self.advance(); }
            if !self.match_token(&Token::Comma) { break; }
        }
        self.expect(&Token::CloseBracket).unwrap_or_else(|_| Span::dummy());
    }
    self.expect(&Token::OpenBrace).unwrap_or_else(|_| Span::dummy());
    let variants = self.parse_variants();
    self.expect(&Token::CloseBrace).unwrap_or_else(|_| Span::dummy());
    AstEnumDecl { span, pub_, name, noprefix, variants, variant_values: None }
}

The [noprefix] modifier is parsed as an attribute inside brackets. This pattern (bracket-delimited attributes before the body) is extensible — new modifiers can be added without changing the grammar structure.

Spec and Impl Declarations

Spec declarations define interfaces (like Rust traits):

fn parse_spec_decl(&mut self, pub_: bool) -> AstSpecDecl {
    let span = self.expect(&Token::Spec).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenBrace).unwrap_or_else(|_| Span::dummy());
    let mut items = Vec::new();
    while !self.check(&Token::CloseBrace) && !self.is_at_end() {
        if let Some(item) = self.parse_spec_item() { items.push(item); }
        else { self.advance(); }
    }
    self.expect(&Token::CloseBrace).unwrap_or_else(|_| Span::dummy());
    AstSpecDecl { span, pub_, name, items }
}

Impl declarations provide implementations of specs for a type:

fn parse_impl_decl(&mut self) -> AstImplDecl {
    let span = self.expect(&Token::Impl).unwrap();
    let impl_type = self.parse_type();
    self.expect(&Token::For).unwrap_or_else(|_| Span::dummy());
    let spec_name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::OpenBrace).unwrap_or_else(|_| Span::dummy());
    let mut methods = Vec::new();
    while !self.check(&Token::CloseBrace) && !self.is_at_end() {
        methods.push(self.parse_func_decl(false));
    }
    self.expect(&Token::CloseBrace).unwrap_or_else(|_| Span::dummy());
    AstImplDecl { span, impl_type, spec_name, methods }
}

The impl keyword is followed by a type (which can be any valid type expression), then for, then the spec name, then a block of function declarations.

Statement Parsing: parse_statement()

Statements are the executable units inside function bodies. The dispatch is clean because each statement starts with a unique keyword:

src/parser.rs — statement dispatch
fn parse_statement(&mut self) -> Option<AstStmt> {
    match self.peek_token() {
        Token::Return    => Some(AstStmt::Return(self.parse_return())),
        Token::Let       => Some(AstStmt::VarDecl(self.parse_var_decl())),
        Token::If        => Some(AstStmt::If(self.parse_if())),
        Token::While     => Some(AstStmt::While(self.parse_while())),
        Token::For       => Some(AstStmt::For(self.parse_for())),
        Token::Asm       => Some(AstStmt::Asm(self.parse_asm())),
        Token::OpenBrace => Some(AstStmt::Block(self.parse_block())),
        Token::Import    => Some(AstStmt::Import(self.parse_import())),
        _ => { // Expression statement
            if self.is_at_end() { return None; }
            let expr = self.parse_expr();
            let span = self.previous_span();
            self.expect(&Token::Semicolon).unwrap_or_else(|_| Span::dummy());
            Some(AstStmt::Expr(expr))
        }
    }
}

The fallthrough case is the expression statement — any expression followed by a semicolon. This is how foo(); (function call as statement) and x = 42; (assignment as statement) work.

parse_if

fn parse_if(&mut self) -> AstIf {
    let span = self.expect(&Token::If).unwrap();
    self.expect(&Token::OpenParen).unwrap_or_else(|_| Span::dummy());
    let cond = self.parse_expr();
    self.expect(&Token::CloseParen).unwrap_or_else(|_| Span::dummy());
    let then_block = self.parse_block();
    let else_branch = if self.match_token(&Token::Else) {
        match self.peek_token() {
            Token::If => Some(Box::new(AstStmt::If(self.parse_if()))),
            _ => Some(Box::new(AstStmt::Block(self.parse_block()))),
        }
    } else { None };
    AstIf { span, cond, then_block, else_branch }
}

Note the else if handling: when else is followed by if, the parser recursively calls parse_if, creating a nested chain: if (a) {} else if (b) {} else {} becomes If(cond=a, else=Some(If(cond=b, else=Some(Block)))).

parse_var_decl

fn parse_var_decl(&mut self) -> AstVarDecl {
    let span = self.expect(&Token::Let).unwrap();
    let name = self.expect_ident().unwrap_or_else(|_| "_".into());
    self.expect(&Token::Colon).unwrap_or_else(|_| Span::dummy());
    let type_ = Some(self.parse_type());
    let init = if self.match_token(&Token::Eq) { Some(self.parse_expr()) } else { None };
    self.expect(&Token::Semicolon).unwrap_or_else(|_| Span::dummy());
    AstVarDecl { span, name, type_, init }
}

Variable declarations follow the pattern let name: type = init; where both the type annotation and initializer are required.

Pratt Parsing: Expressions

The elegant part of the parser. Instead of encoding precedence in nested functions (a la parse_add_expr calling parse_mul_expr calling parse_primary), Pratt parsing uses binding powers:

Operator binding powers (lower = looser, higher = tighter):

= (assignment) ........... 10
|| (logical or) .......... 20
&& (logical and) ......... 30
== != (equality) ........... 40
< > <= >= (comparison) ..... 50
| (bitwise or) .......... 60
^ (bitwise xor) ......... 70
& (bitwise and) ......... 80
<< >> (shift) .............. 90
+ - (addition) ........... 100
* / % (multiplication) .... 110
( [ { . -> (postfix) ...... 130

The core algorithm is just three methods:

src/parser.rs — Pratt expression parsing
fn parse_expr(&mut self) -> AstExpr {
    self.parse_expr_bp(0)
}

fn parse_expr_bp(&mut self, min_bp: u8) -> AstExpr {
    let mut lhs = self.parse_prefix();
    loop {
        let bp = self.infix_bp();
        if bp == 0 || bp < min_bp { break; }
        lhs = self.parse_infix(lhs, bp);
    }
    lhs
}

How it works step by step:

  1. Parse a prefix expression (literal, identifier, unary operator, parenthesized expression).
  2. Look at the next token — what's its binding power? Zero means "not an infix operator."
  3. If the next operator binds tighter than the current minimum, parse it as an infix (which recursively calls parse_expr_bp for the right-hand side with a higher minimum).
  4. If the next operator is looser (or zero), stop and return what we have.

infix_bp — The Binding Power Table

fn infix_bp(&self) -> u8 {
    match self.peek_token() {
        Token::Eq => 10,
        Token::PipePipe => 20,
        Token::AmpAmp => 30,
        Token::EqEq | Token::BangEq => 40,
        Token::Lt | Token::Gt | Token::LtEq | Token::GtEq => 50,
        Token::Pipe => 60,
        Token::Caret => 70,
        Token::Amp => 80,
        Token::LtLt | Token::GtGt => 90,
        Token::Plus | Token::Minus => 100,
        Token::Star | Token::Slash | Token::Percent => 110,
        Token::OpenParen | Token::OpenBracket | Token::OpenBrace
            | Token::Dot | Token::Arrow => 130,
        _ => 0,
    }
}

parse_prefix — Atomic and Unary Expressions

fn parse_prefix(&mut self) -> AstExpr {
    let tok = self.peek_token();
    let span = self.current_span();
    match tok {
        Token::NumLiteral(ref val) => { self.advance(); AstExpr::Int(val.clone(), span) }
        Token::StrLiteral(ref val) => { self.advance(); AstExpr::Str(val.clone(), span) }
        Token::CharLiteral(val) => { self.advance(); AstExpr::Char(val, span) }
        Token::True  => { self.advance(); AstExpr::Bool(true, span) }
        Token::False => { self.advance(); AstExpr::Bool(false, span) }
        Token::Ident(ref name) => { self.advance(); AstExpr::Ident(name.clone(), span) }
        Token::OpenParen => { /* parenthesized expression */ }
        Token::Minus  => { self.advance(); AstExpr::Neg(Box::new(self.parse_expr_bp(120)), span) }
        Token::Bang   => { self.advance(); AstExpr::Not(Box::new(self.parse_expr_bp(120)), span) }
        Token::Tilde  => { self.advance(); AstExpr::BitNot(Box::new(self.parse_expr_bp(120)), span) }
        Token::Amp    => { self.advance(); AstExpr::AddrOf(Box::new(self.parse_expr_bp(120)), span) }
        Token::At     => { self.advance(); AstExpr::Deref(Box::new(self.parse_expr_bp(120)), span) }
        Token::Cast   => { /* cast(type, expr) */ }
        _ => { self.error("expected expression", self.current_span()); self.advance(); AstExpr::Int("0".into(), span) }
    }
}

Unary operators (-, !, ~, &, @) all use binding power 120 for their operand — this is higher than all binary operators, so -x * y parses as (-x) * y, not -(x * y).

parse_infix — Binary Operators and Postfix Syntax

This method receives the already-parsed left-hand side and the binding power, then dispatches to the appropriate handler:

fn parse_infix(&mut self, lhs: AstExpr, bp: u8) -> AstExpr {
    let op_span = self.current_span();
    let tok = self.peek_token();
    match tok {
        // Postfix: call, index, field access, struct init
        Token::OpenParen   => { /* call: foo(args) */ }
        Token::OpenBracket => { /* index: a[i] */ }
        Token::OpenBrace   => { /* struct init: Point { x: 1, y: 2 } */ }
        Token::Dot         => { /* field: p.x */ }
        Token::Arrow       => { /* arrow: p->x */ }
        // Binary operators: all follow the same pattern
        Token::Plus  => { self.advance(); let rhs = self.parse_expr_bp(bp + 1);
                          AstExpr::Binary(BinaryOp::Add, Box::new(lhs), Box::new(rhs), op_span) }
        // ... all other binary ops follow the same pattern with bp + 1
        Token::Eq   => { // Assignment is RIGHT-associative: use raw bp, not bp + 1
            self.advance();
            let rhs = self.parse_expr_bp(bp); // no +1
            AstExpr::Assign(Box::new(lhs), Box::new(rhs), op_span)
        }
        _ => lhs,
    }
}
Left vs Right Associativity: For left-associative operators (+, *, ==), the right-hand operand is parsed with bp + 1, meaning 1 + 2 + 3 becomes ((1 + 2) + 3). For right-associative =, the RHS is parsed with the same bp, so x = y = 0 becomes x = (y = 0).

Type Parsing: parse_type()

Types in L1S form their own mini-language. The parser handles array, pointer, function, spec, and named types in one recursive method:

src/parser.rs — parse_type
fn parse_type(&mut self) -> AstType {
    let span = self.current_span();
    // [expr] type  →  array type
    if self.match_token(&Token::OpenBracket) {
        let size = self.parse_expr();
        self.expect(&Token::CloseBracket).unwrap_or_else(|_| Span::dummy());
        let inner = self.parse_type();
        return AstType::Array(Box::new(size), Box::new(inner), span);
    }
    // @type  →  pointer type
    if self.match_token(&Token::At) {
        let inner = self.parse_type();
        return AstType::Ptr(Box::new(inner), span);
    }
    // Func<ret, args...>
    if self.match_token(&Token::Func) {
        self.expect(&Token::Lt).unwrap_or_else(|_| Span::dummy());
        let mut types = Vec::new();
        types.push(self.parse_type());
        while self.match_token(&Token::Comma) { types.push(self.parse_type()); }
        self.expect(&Token::Gt).unwrap_or_else(|_| Span::dummy());
        let ret = Box::new(types.remove(0));
        return AstType::Func(types, ret, span);
    }
    // spec Name  →  spec type
    if self.match_token(&Token::Spec) {
        let name = self.expect_ident().unwrap_or_else(|_| "_".into());
        return AstType::Spec(name, span);
    }
    // Primitive or named type
    match self.peek_token() {
        Token::Void => { self.advance(); AstType::Prim(PrimType::Void, span) }
        Token::Bool => { self.advance(); AstType::Prim(PrimType::Bool, span) }
        Token::I32  => { self.advance(); AstType::Prim(PrimType::I32, span) }
        // ... all other primitives
        Token::Ident(ref name) => { let name = name.clone(); self.advance(); AstType::Named(name, span) }
        _ => { self.error("expected type", span); AstType::Prim(PrimType::Void, span) }
    }
}

The key design: parse_type is a single recursive method, not a hierarchy. This is possible because L1S types have simple prefix syntax (@i32, [10]i32, Func<i32, void>). The recursive calls handle nesting: @[10]@i32 is a pointer to an array of 10 pointers to i32.

Common Pitfalls & Debugging Tips

  1. Infinite loops: Every while loop in the parser must guarantee progress. Always pair a guard counter with a force-advance on failure. Test with cargo test -- --nocapture to see panic messages.
  2. Forgetting to advance: check doesn't consume. match_token does. Mixing them up causes infinite loops. The rule: use match_token for optional syntax, check for lookahead decisions, expect for required syntax.
  3. Pratt binding power off-by-one: Left-associative ops use bp + 1 for the RHS. Right-associative ops use bp. Getting this wrong silently produces wrong parse trees. Always test with 1 + 2 + 3 (should left-associate) and x = y = 0 (should right-associate).
  4. Error recovery swallowing tokens: When unwrap_or_else uses a synthetic value, the parser continues. But the next parse call might be confused by the missing token. The guard counter catches this. If a test hangs, check that the guard fires.
  5. Debug with --nocapture: Add temporary eprintln!("parsing stmt: {:?}", self.peek_token()); calls and run with cargo test -- --nocapture to trace parser state.
Adding a new statement or operator: You need to touch exactly these places: add a Token variant (if new syntax), add an AstStmt or AstExpr variant, add the parsing method, add an infix_bp entry (for operators), add dispatch in parse_statement or parse_infix.

7   The Type System

Why an Arena-Based Type System?

After parsing, we have an AST sprinkled with type annotations. But the AST types are just syntax — AstType::Prim(PrimType::I32, ...), AstType::Named("Point", ...). During semantic analysis, we need to resolve these to an internal representation that supports:

  • O(1) equality checking: Is @i32 the same type as @i32? Yes. Is Func<i32, i32> the same as Func<i32, i32>? Yes — we should not have to recursively compare nodes every time.
  • Structural sharing: @Point appears in many places in the program. Each occurrence should point to the same Ptr(Point) node.
  • Late resolution: Type Point might be defined after it's used. We need forward-reference placeholders.
  • Size and alignment queries: Every type must be able to answer "how many bytes does this occupy?"

L1S uses a TypeArena — an intern pool that stores each unique TypeNode exactly once. Types are referenced by TypeId, a lightweight u32 wrapper. Equality is just id1 == id2 (a u32 comparison).

Memory layout of the arena:

TypeArena {
 types: [Void, I32, Ptr(%1), Func([%0,%0], %1), ...]
 structs: [StructInfo { name: "Point", fields: [...], size: 8, align: 4 }]
 enums: [EnumInfo { name: "Color", variants: [...], noprefix: false }]
}

Each index is a TypeId, shown here as %0, %1, etc.

TypeId — The Handle

A TypeId is just a newtype over u32, but it has display formatting for debugging:

src/types.rs — TypeId
use std::fmt;

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct TypeId(u32);

impl TypeId {
    pub fn new(id: u32) -> Self { TypeId(id) }
    pub fn index(self) -> usize { self.0 as usize }
}

impl fmt::Display for TypeId {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "%{}", self.0)
    }
}

Deriving Copy and Clone means every use of a type ID is a cheap u32 copy. No reference counting, no lifetime tracking, no garbage.

TypeNode — The Type Representation

Every type that can exist in L1S is represented by exactly one variant of this enum:

src/types.rs — TypeNode enum
#[derive(Debug, Clone)]
pub enum TypeNode {
    // Primitives (13 variants)
    Void,
    Bool,
    Char,
    U8, I8, U16, I16, U32, I32, U64, I64,
    Usize, Isize,

    // Compound types
    Ptr(TypeId),               // @T  — pointer to T
    Array(u64, TypeId),         // [N]T — array of N x T
    Func(Vec<TypeId>, TypeId), // Func<ret, arg1, arg2, ...>

    // Named and aliased types
    Named(String, TypeId),     // Forward reference: name + resolved ID
    Alias(String, TypeId),     // typedef alias

    // User-defined types (backed by StructInfo/EnumInfo)
    Struct(TypeId),             // struct type, ID indexes into structs[]
    Enum(TypeId),               // enum type, ID indexes into enums[]

    // Spec (interface) type
    Spec(String, Vec<(String, TypeId)>),
}

TypeNode layout rationale:

  • Primitives are standalone variants, not a nested Prim wrapper. This eliminates an indirection when matching.
  • Ptr and Array hold TypeIds, not boxed TypeNodes. This is the key to structural sharing — the i32 inside Ptr(i32) is the same TypeId(%1) as every other reference to i32.
  • Named carries a TypeId alongside the name. Initially this is a placeholder; it gets resolved as the compiler encounters the type's definition.
  • Struct and Enum store a separate TypeId that indexes into the arena's parallel structs/enums vectors. This avoids making TypeNode carry heavy field/variant data.

TypeArena — The Intern Pool

The arena stores all types in a Vec<TypeNode>, plus separate vectors for struct and enum metadata:

src/types.rs — TypeArena
#[derive(Debug)]
pub struct TypeArena {
    types: Vec<TypeNode>,
    structs: Vec<StructInfo>,
    enums: Vec<EnumInfo>,
}

impl TypeArena {
    pub fn new() -> Self {
        TypeArena { types: Vec::new(), structs: Vec::new(), enums: Vec::new() }
    }

    pub fn intern(&mut self, node: TypeNode) -> TypeId {
        let id = self.lookup(&node);
        match id {
            Some(tid) => tid,
            None => {
                let id = TypeId(self.types.len() as u32);
                self.types.push(node);
                id
            }
        }
    }

    pub fn intern_no_dedup(&mut self, node: TypeNode) -> TypeId {
        let id = TypeId(self.types.len() as u32);
        self.types.push(node);
        id
    }

    pub fn get(&self, id: TypeId) -> &TypeNode {
        &self.types[id.index()]
    }
}

intern is the main insertion path. It looks up the node first; if it already exists, it returns the existing TypeId. This is structural deduplication.

intern_no_dedup skips the lookup. This is used for Named types ("this is Point even if another type has the same shape") and for types that will be mutated later (placeholder structs).

Structural Equality: nodes_equal

The dedup logic depends on nodes_equal, a recursive function that compares two TypeNodes structurally:

src/types.rs — nodes_equal
fn nodes_equal(a: &TypeNode, b: &TypeNode, arena: &TypeArena) -> bool {
    match (a, b) {
        // Primitives: same variant = equal (trivial, no data to compare)
        (TypeNode::Void, TypeNode::Void)
        | (TypeNode::Bool, TypeNode::Bool)
        | (TypeNode::Char, TypeNode::Char)
        | (TypeNode::U8, TypeNode::U8)
        ...
        | (TypeNode::Isize, TypeNode::Isize) => true,

        // Compounds: compare inner TypeIds (which are themselves deduplicated)
        (TypeNode::Ptr(a), TypeNode::Ptr(b)) => a == b,
        (TypeNode::Array(na, ta), TypeNode::Array(nb, tb)) => na == nb && ta == tb,
        (TypeNode::Func(a_args, a_ret), TypeNode::Func(b_args, b_ret)) => {
            a_ret == b_ret
            && a_args.len() == b_args.len()
            && a_args.iter().zip(b_args).all(|(x, y)| x == y)
        }

        // Named/Alias: compare by name (the inner TypeId may differ during
        // resolution, but the name is the identity)
        (TypeNode::Named(a, _), TypeNode::Named(b, _)) => a == b,
        (TypeNode::Alias(a, _), TypeNode::Alias(b, _)) => a == b,

        // Struct/Enum: compare the struct/enum ID (not the fields)
        (TypeNode::Struct(a), TypeNode::Struct(b)) => a == b,
        (TypeNode::Enum(a), TypeNode::Enum(b)) => a == b,

        // Spec: compare by name
        (TypeNode::Spec(a, _), TypeNode::Spec(b, _)) => a == b,

        // Different variants: never equal
        _ => false,
    }
}

The crucial insight: The outer TypeId comparison for compound types (Ptr, Array, Func) works because their inner types have already been deduplicated. When you build Ptr(i32), the i32 TypeId is already unique across the arena. So a == b for two TypeIds is a single u32 comparison. The arena ensures structural equality transitively.

O(n) intern, O(1) equality: The intern method is O(n) in the number of types (linear scan). For small compilers this is fine. If performance becomes an issue, the scan can be replaced with a hash map keyed on a discriminant + inner IDs. But the O(1) equality (TypeId == TypeId) is a permanent benefit.

Size and Alignment Computation

Every type must know its size (in bytes) and alignment requirement. These are computed by recursive functions on TypeNode. No layout is computed for Named, Alias, Func, Spec, or Enum (enums use the ABI size).

src/types.rs — size_of and align_of
pub fn size_of(&self, id: TypeId) -> u64 {
    match self.get(id) {
        TypeNode::Void => 0,
        TypeNode::Bool | TypeNode::Char | TypeNode::U8 | TypeNode::I8 => 1,
        TypeNode::U16 | TypeNode::I16 => 2,
        TypeNode::U32 | TypeNode::I32 => 4,
        TypeNode::U64 | TypeNode::I64 | TypeNode::Usize | TypeNode::Isize => 8,
        TypeNode::Ptr(_) => 8,
        TypeNode::Array(n, elem) => n * self.size_of(*elem),
        TypeNode::Struct(sid) => {
            let idx = sid.index();
            if idx < self.structs.len() { self.structs[idx].size } else { 0 }
        }
        _ => 0,
    }
}

pub fn align_of(&self, id: TypeId) -> u64 {
    match self.get(id) {
        TypeNode::Void => 1,
        TypeNode::Bool | TypeNode::Char | TypeNode::U8 | TypeNode::I8 => 1,
        TypeNode::U16 | TypeNode::I16 => 2,
        TypeNode::U32 | TypeNode::I32 => 4,
        TypeNode::U64 | TypeNode::I64 | TypeNode::Usize | TypeNode::Isize => 8,
        TypeNode::Ptr(_) => 8,
        TypeNode::Array(_, elem) => self.align_of(*elem),
        TypeNode::Struct(sid) => {
            let idx = sid.index();
            if idx < self.structs.len() { self.structs[idx].align } else { 1 }
        }
        _ => 1,
    }
}
Note: These are ABI sizes without padding tracking. For structs, the size and alignment are computed externally by the struct layout pass and stored in StructInfo. The size_of/align_of methods just read those precomputed values. For arrays, size is count * element_size with no trailing padding.

pub fn add_struct(&mut self, info: StructInfo) -> TypeId {

pub fn get_struct(&self, id: TypeId) -> Option<&StructInfo> {
    if let TypeNode::Struct(sid) = self.get(id) {

    } else { None }

The add_struct method assigns the struct a TypeId first, interns the Struct(id) node, then pushes the metadata. This means any TypeNode::Struct(id) that appears before the info is pushed will return a placeholder until the struct is fully defined.

Forward References and Placeholders

L1S allows types to reference other types that are defined later in the source. For example, a linked list node:

struct Node {
    value: i32;
    next: @Node;  // forward reference to Node itself
}

To support this, the type resolution phase first creates placeholder struct and enum entries, then fills in their fields:

pub fn add_struct_placeholder(&mut self) -> TypeId {
    let id = TypeId(self.structs.len() as u32);
    let info = StructInfo {
        name: String::new(), fields: Vec::new(), size: 0, align: 1,
    };
    let tid = self.intern(TypeNode::Struct(id));
    self.structs.push(info);
    tid
}

pub fn add_enum_placeholder(&mut self) -> TypeId {
    let id = TypeId(self.enums.len() as u32);
    let info = EnumInfo {
        name: String::new(), variants: Vec::new(), noprefix: false,
    };
    let tid = self.intern(TypeNode::Enum(id));
    self.enums.push(info);
    tid
}

The placeholder creates a struct/enum entry with empty fields. Before code generation, the compiler fills in the name, fields, size, and alignment using get_struct_mut or get_enum_mut. Any type that references a placeholder struct gets the correct TypeId immediately; only the field/offset data is deferred.

  1. Missing dedup for compound types: The nodes_equal function must handle every compound variant. 8   The Symbol Table

Why a Symbol Table?

After the parser builds an AST, we have a tree of expressions and declarations. But to understand what a name like x or main refers to, we need name resolution. The symbol table maps identifiers to their definitions, respecting lexical scoping rules.

Consider this program:

func main(i32) {
    let x: i32 = 1;
    {
        let x: i32 = 2;  // shadows outer x
    }
}

The x inside the inner block refers to a different variable than the outer x. A flat global map cannot handle this — we need a scope chain.

Scope Chain vs. Flat Map

A flat map (single HashMap<String, Entry>) stores every name at the same level. It works for languages like C with file-scoped static names, but fails for:

  • Lexical scoping: inner blocks see outer variables but not vice versa.
  • Shadowing: an inner declaration with the same name hides the outer one.
  • Nested functions: function parameters create a new scope.
  • Correct resolution: lookup("x") should return the innermost binding.

The scope chain solves this by maintaining a linked list of scopes. Each scope has a parent pointer. lookup walks from the current scope upward until it finds the name.

Scope chain for the nested block example:

Scope 0 (global)
main: Function, add: Function
↑ parent
Scope 1 (function body)
x: Variable(i32)
↑ parent
Scope 2 (inner block)
x: Variable(i32) ← shadows scope 1's x

lookup("x") at scope 2 returns scope 2's x.
lookup("x") at scope 1 returns scope 1's x (if scope 2 doesn't have it).

SymbolKind and SymbolEntry

Every named entity in the program is a symbol. The SymbolKind enum distinguishes them:

src/symbol.rs — SymbolKind and SymbolEntry
pub enum SymbolKind {
    /// A local or parameter variable, carrying its type.
    Variable(TypeId),
    /// A function declaration, with its full function type.
    Function(TypeId),
    /// A type name (struct, enum, typedef), carrying the TypeId.
    TypeDecl(TypeId),
    /// A spec (trait/interface) with its method signatures.
    Spec(Vec<(String, TypeId)>),
    /// An enum variant, carrying its numeric value.
    EnumVariant(u64),
    /// A module path (for imports).
    Module(Vec<String>),
    /// A static/global variable.
    StaticVar(TypeId),
}

Each variant carries exactly the data needed. Variable(TypeId) holds the variable's type. Function(TypeId) holds the full function type (parameters + return). EnumVariant(u64) holds the variant's numeric discriminant.

pub struct SymbolEntry {
    pub name: String,
    pub kind: SymbolKind,
    pub span: Span,
    pub is_pub: bool,
}

The span field points to the declaration site, enabling "previously defined here" error messages. The is_pub flag controls visibility (currently reserved for future module-level access control).

Scope and SymbolTable

A scope is just a map of names to entries, with an optional parent:

    pub symbols: HashMap<String, SymbolEntry>,

}

The SymbolTable owns a Vec<Scope> and tracks which scope is current:

ScopeId(u32) is an index into the scopes vector. The root scope (index 0) has parent: None and holds all global declarations.

Construction

Pushing and Popping Scopes

push_scope creates a new scope with the current scope as its parent, then sets it as current:

pub fn push_scope(&mut self) -> ScopeId {
    let id = ScopeId(self.scopes.len() as u32);
    self.scopes.push(Scope {
        symbols: HashMap::new(),
        parent: Some(self.current),
    });
    self.current = id;
    id
}

pop_scope restores the parent as current. Note: the scope is not removed from the vector — it persists for lookup by ScopeId if needed:

pub fn pop_scope(&mut self) {
    if let Some(parent) = self.scopes[self.current.0 as usize].parent {
        self.current = parent;
    }
}

Insertion

insert adds a symbol to the current scope. It returns an error if a symbol with the same name already exists there:

pub fn insert(
    &mut self, name: &str, kind: SymbolKind,
    span: Span, is_pub: bool,
) -> Result<(), SymbolError> {
    let scope = &mut self.scopes[self.current.0 as usize];
    if let Some(existing) = scope.symbols.get(name) {
        return Err(SymbolError::AlreadyDefined {
            name: name.to_string(),
            previous_span: existing.span,
        });
    }
    scope.symbols.insert(name.to_string(), SymbolEntry { .. });
    Ok(())
}

Lookup: Walking the Scope Chain

The lookup method is the heart of name resolution. It starts at the current scope and walks up the parent chain:

src/symbol.rs — lookup() walks the scope chain
pub fn lookup(&self, name: &str) -> Option<&SymbolEntry> {
    let mut sid = Some(self.current);
    while let Some(id) = sid {
        let scope = &self.scopes[id.0 as usize];
        if let Some(entry) = scope.symbols.get(name) {
            return Some(entry);
        }
        sid = scope.parent;
    }
    None
}

Step-by-step:

  1. Start at self.current (the innermost scope).
  2. Try to find name in the current scope's symbols map.
  3. If found, return the entry immediately — this is the innermost binding.
  4. If not found, follow the parent pointer to the enclosing scope.
  5. If we reach a scope with parent: None (the root) and still don't find it, return None.

lookup_in_scope is a restricted variant that only checks a specific scope (by ScopeId), without walking the chain:

pub fn lookup_in_scope(&self, name: &str, scope: ScopeId) -> Option<&SymbolEntry> {
    self.scopes[scope.0 as usize].symbols.get(name)
}

This is useful for checking whether a name exists "directly" in a scope (e.g., for error messages about shadowing across specific boundaries).

SymbolError Enum

Error cases are explicitly modeled:

pub enum SymbolError {
    /// Name already exists in the current scope.

    /// Name not found in any scope.
    NotFound { name: String },
    /// Name exists but is not a type.

Common Pitfalls

  1. Forgetting to push/pop scopes in blocks: Every block (function body, if/while body, nested braces) must push a new scope. If you forget, variables leak upward.
  2. Using insert instead of insert_global for top-level decls: If you're inside a function and call insert("main", ...), it goes into the function's scope, not the global scope. Always use insert_global for top-level declarations.
  3. Pop not restoring the right parent: If pop_scope doesn't correctly follow the parent pointer, the scope chain breaks. Verify by checking current before and after.
  4. Mutable borrow conflicts: lookup borrows &self immutably but insert borrows &mut self. You cannot interleave them in the same scope iteration. The symbol table design avoids this by separating read and write operations clearly.
Why not a single HashMap? Some toy compilers use a single flat map and mangle names (e.g., _main_x for inner x). This works for code generation but breaks error messages ("undefined variable _main_x" is confusing) and makes shadowing analysis impossible. The scope chain is the principled approach used by almost all production compilers.

9   Type Checking

What is Type Checking?

After parsing and building the symbol table, the compiler needs to verify that the program is type-safe. Type checking ensures that: Every expression has a known type. Operations are applied to compatible operands. Function arguments match parameter types. Return statements match the declared return type.

The L1S type checker is a multi-phase semantic analysis pass. It runs after parsing and symbol table construction but before IR generation.

Why Multi-Phase?

L1S supports forward references — a function can call another function defined later in the file, and structs can contain pointers to structs not yet fully defined. A single linear pass would fail because definitions are encountered after their uses.

The solution is four phases:

Phase 1: Collect → Register all function signatures, struct/enum placeholders, typedefs, specs, and globals in the symbol table.
Phase 2: Check Structs & Enums → Compute field types, offsets, sizes, and variant values.
Phase 3: Check Functions → Type-check all function bodies with full type information available.
Phase 4: Check Impls → Verify spec implementations against spec declarations.

This separation means that when checking func foo() { bar(); }, bar is already in the symbol table even if defined later. Similarly, struct Node can have a field @Node because its TypeId is created (as a placeholder) in phase 1.

The TypeChecker Struct

src/typeck.rs — TypeChecker struct
pub struct TypeChecker<'a> {
    pub types: &mut TypeArena,
    pub symbols: &mut SymbolTable,
    pub diag: &mut Diagnostics,
    pub annotations: HashMap<Span, TypeId>,
    current_func_return: Option<TypeId>,
    spec_impls: HashMap<String, Vec<(TypeId, Vec<(String, TypeId)>)>>,
}

Key fields:

  • types: The TypeArena (see Chapter 7), used to intern types and query size/alignment.
  • symbols: The SymbolTable (see Chapter 8), used to look up and insert declarations.
  • diag: The diagnostics system, used to report type errors with source locations.
  • annotations: A map from Span to TypeId that records the inferred type of every expression. This is consumed by the IR builder to know what type each value has.
  • current_func_return: Tracks the return type of the function currently being checked, so return statements can validate their value.
  • spec_impls: Accumulated impl blocks, keyed by spec name, checked in phase 4.

Phase 1: check_program and collect_decl

check_program orchestrates all four phases:

pub fn check_program(&mut self, prog: &mut AstProgram) {
    let decls = prog.decls.clone();
    for decl in &decls { self.collect_decl(decl); }   // Phase 1
    for decl in &decls {
        match decl {
            AstDecl::Struct(s) => self.check_struct(s),
            AstDecl::Enum(e) => self.check_enum(e),     // Phase 2
            _ => {}
        }
    }
    for decl in &decls {                                  // Phase 3
        if let AstDecl::Func(f) = decl { self.check_func(f); }
    }
    for decl in &decls {                                  // Phase 4
        if let AstDecl::Impl(imp) = decl { self.check_impl(imp); }
    }
}

Note the prog.decls.clone() at the top. We clone the declarations list so that we can iterate it multiple times without borrowing conflicts, since check_struct and check_func may mutate the program through the type arena and symbol table.

collect_decl registers each declaration in the symbol table. For functions it computes the full TypeNode::Func type; for structs and enums it creates placeholders via add_struct_placeholder / add_enum_placeholder:

src/typeck.rs — collect_decl for structs and functions
fn collect_decl(&mut self, decl: &AstDecl) {
    match decl {
        AstDecl::Func(f) => {
            let func_type = self.ast_type_to_id(&f.return_type);
            let mut param_types = Vec::new();
            for p in &f.params {
                param_types.push(self.ast_type_to_id(&p.type_));
            }
            let ft = self.types.intern(TypeNode::Func(param_types, func_type));
            let _ = self.symbols.insert_global(&f.name, SymbolKind::Function(ft), f.span, f.pub_);
        }
        AstDecl::Struct(s) => {
            let tid = self.types.add_struct_placeholder();
            let _ = self.symbols.insert_global(&s.name, SymbolKind::TypeDecl(tid), s.span, s.pub_);
        }
        // ... enum, spec, typedef, import, static_var, impl follow the same pattern
    }
}

The placeholder is crucial: it creates a TypeId that other declarations can reference before the struct body is checked. Without it, a @Node pointer in a struct field would fail because Node has no type yet.

Phase 2a: check_struct — Alignment and Padding

Struct layout computes the offset of each field, respecting alignment requirements. This mirrors how C compilers lay out structs:

src/typeck.rs — check_struct with alignment padding
fn check_struct(&mut self, s: &AstStructDecl) {
    let entry = self.symbols.lookup(&s.name).unwrap();
    let tid = match &entry.kind {
        SymbolKind::TypeDecl(tid) => *tid, _ => return,
    };
    let mut fields = Vec::new();
    let mut offset: u64 = 0;
    let mut max_align: u64 = 1;
    for f in &s.fields {
        let ft = self.ast_type_to_id(&f.type_);
        let align = self.types.align_of(ft);
        let padded = (offset + align - 1) / align * align;  // round up to alignment
        offset = padded;
        fields.push(FieldInfo { name: f.name.clone(), type_: ft, offset });
        offset += self.types.size_of(ft);
        if align > max_align { max_align = align; }
    }
    let final_size = (offset + max_align - 1) / max_align * max_align;
    // Store computed layout back into the struct info
    if let Some(sinfo) = self.types.get_struct_mut(tid) {
        sinfo.name = s.name.clone();
        sinfo.fields = fields;
        sinfo.size = final_size;
        sinfo.align = max_align;
    }
}

The padding formula (offset + align - 1) / align * align rounds offset up to the next multiple of align. For example, if offset = 5 and align = 4, the result is (5 + 3) / 4 * 4 = 8 / 4 * 4 = 8 — 3 bytes of padding before the field.

The final struct size is also rounded up to the alignment of the most-aligned field, matching C's struct layout rules.

Phase 2b: check_enum — Variant Numbering

Enums are simpler: variants are numbered sequentially starting from 0, or from explicit values:

fn check_enum(&mut self, e: &AstEnumDecl) {
    let entry = self.symbols.lookup(&e.name).unwrap();
    let tid = match &entry.kind {
        SymbolKind::TypeDecl(tid) => *tid, _ => return,
    };
    let mut variants = Vec::new();
    let mut val: u64 = 0;
    if let Some(ref values) = e.variant_values {
        for (i, v) in e.variants.iter().enumerate() {
            let actual = if i < values.len() { values[i] } else { val };
            variants.push(VariantInfo { name: v.clone(), value: actual });
            val = actual + 1;
        }
    } else { /* auto-number from 0 */ }
    if let Some(einfo) = self.types.get_enum_mut(tid) {
        einfo.name = e.name.clone(); einfo.variants = variants; einfo.noprefix = e.noprefix;
    }
    // Register each variant as a global symbol so they can be used as values
    if let Some(einfo) = self.types.get_enum(tid) {
        for v in &einfo.variants {
            let _ = self.symbols.insert_global(&v.name, SymbolKind::EnumVariant(v.value), e.span, e.pub_);
        }
    }
}

Phase 3: check_func — Function Body Checking

Function checking pushes a new scope for parameters, then checks the body:

fn check_func(&mut self, f: &AstFuncDecl) {
    self.current_func_return = Some(self.ast_type_to_id(&f.return_type));
    self.symbols.push_scope();             // new scope for params + locals
    for p in &f.params {
        let pt = self.ast_type_to_id(&p.type_);
        let _ = self.symbols.insert(&p.name, SymbolKind::Variable(pt), p.span, false);
    }
    self.check_block(&f.body);
    self.symbols.pop_scope();              // restore outer scope
    self.current_func_return = None;
}

check_block, check_stmt, check_expr

Blocks push/pop scopes:

fn check_block(&mut self, block: &AstBlock) {
    self.symbols.push_scope();
    for stmt in &block.stmts { self.check_stmt(stmt); }
    self.symbols.pop_scope();
}

Statement checking dispatches on the variant. Variable declarations check optional type annotations and optional initializers:

AstStmt::VarDecl(v) => {
    let t = if let Some(ref type_) = v.type_ {
        self.ast_type_to_id(type_)
    } else {
        self.diag.error("type inference not yet supported", Some(v.span));
        self.types.intern(TypeNode::Void)
    };
    if let Some(ref init) = v.init {
        let init_type = self.check_expr(init);
        if !self.types_equal(init_type, t) {
            self.diag.error(
                format!("type mismatch: expected {}, got {}",
                    self.type_str(t), self.type_str(init_type)),
                Some(v.span));
        }
    }
    let _ = self.symbols.insert(&v.name, SymbolKind::Variable(t), v.span, false);
}

Return statements check against current_func_return:

AstStmt::Return(r) => {
    let val_type = self.check_expr(&r.value);
    if let Some(ret) = self.current_func_return {
        if !self.types_equal(val_type, ret) {
            self.diag.error(
                format!("return type mismatch: expected {}, got {}",
                    self.type_str(ret), self.type_str(val_type)),
                Some(r.span));
        }
    }
}

Control flow statements validate their conditions:

AstStmt::If(i) => {
    let cond_type = self.check_expr(&i.cond);
    let bool_id = self.types.intern(TypeNode::Bool);
    if cond_type != bool_id {
        self.diag.error("if condition must be bool", Some(i.cond.span()));
    }
    self.check_block(&i.then_block);
    if let Some(ref else_branch) = i.else_branch { self.check_stmt(else_branch); }
}

Expression Type Inference

check_expr is the workhorse. It calls check_expr_inner and records the result in annotations:

fn check_expr(&mut self, expr: &AstExpr) -> TypeId {
    let t = self.check_expr_inner(expr);
    self.annotations.insert(expr.span(), t);
    t
}

Inside check_expr_inner, expression types are determined recursively. Literals get their obvious types; identifiers are looked up in the symbol table; operators validate their operands numerically:

src/typeck.rs — key expression cases
fn check_expr_inner(&mut self, expr: &AstExpr) -> TypeId {
    match expr {
        AstExpr::Int(_, _) => self.types.intern(TypeNode::I32),
        AstExpr::Str(_, _) => {
            let char_t = self.types.intern(TypeNode::Char);
            self.types.intern(TypeNode::Ptr(char_t))
        }
        AstExpr::Bool(_, _) => self.types.intern(TypeNode::Bool),
        AstExpr::Ident(name, span) => {
            match self.symbols.lookup(name) {
                Some(entry) => match &entry.kind {
                    SymbolKind::Variable(t) | SymbolKind::StaticVar(t) => *t,
                    SymbolKind::EnumVariant(_) => self.types.intern(TypeNode::I32),
                    SymbolKind::Function(ft) => *ft,
                    _ => { /* error */ self.types.intern(TypeNode::Void) }
                },
                None => { /* error */ self.types.intern(TypeNode::Void) }
            }
        }
        // ... Deref, AddrOf, Neg, Not, BitNot, Binary, Call, Index, etc.
    }
}

Binary operations validate both operands and produce the appropriate result type:

AstExpr::Binary(op, lhs, rhs, _) => {
    let lt = self.check_expr(lhs); let rt = self.check_expr(rhs);
    match op {
        BinaryOp::Eq | BinaryOp::Ne => {
            if !self.types_equal(lt, rt) { /* error */ }
            self.types.intern(TypeNode::Bool)
        }
        _ => {
            if !self.types_equal(lt, rt) { /* error */ }
            if !self.is_numeric(lt) { /* error */ }
            lt  // result type matches operands
        }
    }
}

ast_type_to_id — From AST Types to Interned Types

This method converts an AstType (the parser's representation) into a TypeId (the type arena's interned representation). It handles primitives, named types (by symbol lookup), pointers, arrays, function types, and specs:

src/typeck.rs — ast_type_to_id
pub fn ast_type_to_id(&mut self, t: &AstType) -> TypeId {
    match t {
        AstType::Prim(p, _) => match p {
            PrimType::Void => self.types.intern(TypeNode::Void),
            PrimType::I32 => self.types.intern(TypeNode::I32),
            // ... other primitives
        },
        AstType::Named(name, span) => {
            match self.symbols.lookup(name) {
                Some(entry) => match &entry.kind {
                    SymbolKind::TypeDecl(tid) => *tid,
                    _ => { /* error: not a type */ self.types.intern(TypeNode::Void) }
                },
                None => { /* error: undefined type */ self.types.intern(TypeNode::Void) }
            }
        }
        AstType::Ptr(inner, _) => {
            let inner_id = self.ast_type_to_id(inner);
            self.types.intern(TypeNode::Ptr(inner_id))
        }
        AstType::Array(size_expr, inner, _) => {
            let size = self.eval_const_expr(size_expr);
            let inner_id = self.ast_type_to_id(inner);
            self.types.intern(TypeNode::Array(size, inner_id))
        }
        // ... Func, Spec
    }
}

Helper Methods: types_equal, resolve_type, is_numeric, type_str

types_equal respects type aliases by following the Alias chain:

fn types_equal(&self, a: TypeId, b: TypeId) -> bool {
    self.resolve_type(a) == self.resolve_type(b)
}

fn resolve_type(&self, t: TypeId) -> TypeId {
    match self.types.get(t) {
        TypeNode::Alias(_, inner) => self.resolve_type(*inner),
        _ => t,
    }
}

This means MyInt (a typedef MyInt = i32) is equal to i32 for type checking purposes, but type_str still displays the alias name for error messages.

is_numeric checks if a type is any integer type (including char):

fn is_numeric(&self, t: TypeId) -> bool {
    matches!(self.types.get(t),
        TypeNode::U8 | TypeNode::I8
        | TypeNode::U16 | TypeNode::I16
        | TypeNode::U32 | TypeNode::I32
        | TypeNode::U64 | TypeNode::I64
        | TypeNode::Usize | TypeNode::Isize
        | TypeNode::Char
    )
}

type_str produces human-readable type names for error messages:

fn type_str(&self, t: TypeId) -> String {
    match self.types.get(t) {
        TypeNode::Void => "void".into(),
        TypeNode::I32 => "i32".into(),

            let ps: Vec<String> = params.iter().map(|p| self.type_str(*p)).collect();

        TypeNode::Alias(name, _) => name.clone(),
        // ... remaining variants
    }

Common Pitfalls

  1. Missing placeholders: If you don't create a placeholder for structs/enums before phase 2, circular references (struct A has a pointer to struct B which has a pointer back to A) will fail.
  2. Scope leaks from function params: If you don't push a scope in check_func, parameters leak into the surrounding scope and can shadow globals incorrectly.
  3. Alias equivalence in types_equal: Always call resolve_type before comparing, or typedef A = i32 and i32 will appear different.
  4. Forgetting the annotation map: The IR builder needs every expression's type. If check_expr doesn't insert into annotations, IR generation will fail on that expression.
Why not type inference? L1S currently requires explicit type annotations for all variable declarations. Full type inference (like Rust's let x = 42 inferring i32) would require unification-based inference (Hindley-Milner), which is a significant complexity increase. The explicit annotation approach keeps the type checker simple and the error messages clear.

Why an Intermediate Representation?

A compiler could translate AST directly to assembly. But that conflates two concerns: what the program does and how the machine does it. An Intermediate Representation (IR) sits between the AST and codegen, providing:

  1. Machine independence: The same IR can be compiled to x86-64, ARM, or any backend.
  2. Clear semantics: Every IR instruction has a single, well-defined meaning, unlike AST nodes which bundle multiple concerns.
  3. Optimization: Transformations like constant folding and dead code elimination are easier on a flat instruction list than on a tree.
  4. SSA form: The IR uses Static Single Assignment (via phis), which makes data-flow analysis straightforward.

L1S uses a basic-block IR similar to LLVM's: a function is a graph of basic blocks, each containing a sequence of instructions terminated by a jump, branch, or return.

Pipeline position of IR:

AST → Type Checker → IR Builder → IR Opt → Codegen

IrInstr — Every Instruction Variant

There are 36 instruction variants, organized into five categories. The first operand is always the destination virtual register (except for Store and MemCopy):

src/ir.rs — Full IrInstr enum
pub enum IrInstr {
    // === Memory ===
    Alloc(VReg, TypeId),       // Allocate stack slot
    Load(VReg, VReg),          // Load value from pointer
    Store(VReg, VReg),         // Store value to pointer
    MovImm(VReg, u64, TypeId), // Move immediate constant into register

    // === Arithmetic (3-address, dst = lhs op rhs) ===
    Add(VReg, VReg, VReg),    // dst = lhs + rhs
    Sub(VReg, VReg, VReg),    // dst = lhs - rhs
    Mul(VReg, VReg, VReg),    // dst = lhs * rhs
    SDiv(VReg, VReg, VReg),   // dst = lhs / rhs (signed)
    UDiv(VReg, VReg, VReg),   // dst = lhs / rhs (unsigned)
    SRem(VReg, VReg, VReg),   // dst = lhs % rhs (signed)
    URem(VReg, VReg, VReg),   // dst = lhs % rhs (unsigned)
    And(VReg, VReg, VReg),     // dst = lhs & rhs (bitwise)
    Or(VReg, VReg, VReg),      // dst = lhs | rhs (bitwise)
    Xor(VReg, VReg, VReg),     // dst = lhs ^ rhs (bitwise)
    Shl(VReg, VReg, VReg),     // dst = lhs << rhs
    Shr(VReg, VReg, VReg),     // dst = lhs >> rhs (arithmetic/signed)
    Ushr(VReg, VReg, VReg),    // dst = lhs >> rhs (logical/unsigned)

    // === Unary ===
    Neg(VReg, VReg),           // dst = -src (arithmetic negation)
    Not(VReg, VReg),           // dst = !src (logical not, bool only)
    BitNot(VReg, VReg),        // dst = ~src (bitwise not)

    // === Comparison (all produce a bool) ===
    Eq(VReg, VReg, VReg),      // dst = lhs == rhs
    Ne(VReg, VReg, VReg),      // dst = lhs != rhs
    Slt(VReg, VReg, VReg),     // dst = lhs < rhs (signed)
    Sgt(VReg, VReg, VReg),     // dst = lhs > rhs (signed)
    Sle(VReg, VReg, VReg),     // dst = lhs <= rhs (signed)
    Sge(VReg, VReg, VReg),     // dst = lhs >= rhs (signed)
    Ult(VReg, VReg, VReg),     // dst = lhs < rhs (unsigned)
    Ugt(VReg, VReg, VReg),     // dst = lhs > rhs (unsigned)
    Ule(VReg, VReg, VReg),     // dst = lhs <= rhs (unsigned)
    Uge(VReg, VReg, VReg),     // dst = lhs >= rhs (unsigned)

    // === Conversion ===
    Zext(VReg, VReg, TypeId),  // Zero-extend to wider type
    Sext(VReg, VReg, TypeId),  // Sign-extend to wider type
    Trunc(VReg, VReg, TypeId), // Truncate to narrower type
    PtrToInt(VReg, VReg, TypeId), // Bitcast pointer to integer
    IntToPtr(VReg, VReg, TypeId), // Bitcast integer to pointer

    // === Aggregate / Bulk Memory ===
    Gep(VReg, VReg, VReg, u64), // Get element pointer (base, index, stride)
    MemCopy(VReg, VReg, u64),  // Copy memory (dst ptr, src ptr, size)

    // === Call ===
    Call(VReg, String, Vec<VReg>), // Call function (dest, name, args)
}

Semantic Categories Explained

Memory instructions (Alloc, Load, Store, MovImm): These manage the interaction between values and memory. Alloc reserves stack space; Load/Store transfer values between registers and memory; MovImm loads a compile-time constant.

Arithmetic (Add, Sub, Mul, etc.): Three-address operations where the first vreg is the destination and the remaining two are sources. The signed/unsigned split for division and remainder matters because x86-64 has different instructions (idiv vs. div).

Comparison (Eq, Slt, Ult, etc.): Each comparison produces a boolean (0 or 1) in the destination register. The signed/unsigned variants reflect different x86-64 condition codes (setl vs. setb).

Conversion (Zext, Sext, Trunc): These change the bit width of a value. Zext zero-extends (for unsigned types), Sext sign-extends (for signed types), Trunc truncates to a narrower type. PtrToInt/IntToPtr are reinterpretation casts.

Gep and MemCopy: Gep computes a pointer to an array element: dest = base + index * stride. It is a pointer arithmetic instruction, not a load/store. MemCopy copies a fixed-size block of memory (for struct assignment).

IrTerminator — How Blocks End

Every basic block ends with exactly one terminator. There are only three:

pub enum IrTerminator {
    Jump(IrBlockId),                    // Unconditional jump to target block
    Branch(VReg, IrBlockId, IrBlockId), // Branch on condition
    Return(VReg),                        // Return value from function
}
  • Jump: Always goes to the target block. Used for straight-line control flow (e.g., skip over else branch).
  • Branch: If condition vreg is non-zero, go to the first (true) block; otherwise go to the second (false) block. Used for if and while.
  • Return: Exit the function with the given vreg as the return value. A void function returns %0 (a dummy zero).

SSA and Phi Nodes

The IR is in SSA form: every virtual register is defined exactly once. When different control flow paths produce different values for the same variable, an IrPhi node selects the correct one based on which predecessor block we arrived from:

pub struct IrPhi {
    pub dest: VReg,
    pub type_: TypeId,
    pub sources: Vec<(VReg, IrBlockId)>,
}

Semantically: %3 = phi(bb0: %1, bb1: %2) means "if the previous block was bb0, %3 = %1; if from bb1, %3 = %2." The codegen pass implements this by ensuring both predecessors place their value in the same physical location (register or stack slot).

The dest() Method

Most instructions produce a result in a destination vreg. The dest() method provides a uniform way to query this for optimization passes:

src/ir.rs — dest() method
impl IrInstr {
    pub fn dest(&self) -> VReg {
        match self {
            IrInstr::Alloc(d, _)
            | IrInstr::Load(d, _)
            | IrInstr::Add(d, _, _) | IrInstr::Sub(d, _, _)
            | IrInstr::Mul(d, _, _) | IrInstr::SDiv(d, _, _)
            | IrInstr::UDiv(d, _, _) | IrInstr::SRem(d, _, _)
            | IrInstr::URem(d, _, _) | IrInstr::And(d, _, _)
            | IrInstr::Or(d, _, _) | IrInstr::Xor(d, _, _)
            | IrInstr::Shl(d, _, _) | IrInstr::Shr(d, _, _)
            | IrInstr::Ushr(d, _, _) | IrInstr::Neg(d, _)
            | IrInstr::Not(d, _) | IrInstr::BitNot(d, _)
            | IrInstr::Eq(d, _, _) | IrInstr::Ne(d, _, _)
            | IrInstr::Slt(d, _, _) | IrInstr::Sgt(d, _, _)
            | IrInstr::Sle(d, _, _) | IrInstr::Sge(d, _, _)
            | IrInstr::Ult(d, _, _) | IrInstr::Ugt(d, _, _)
            | IrInstr::Ule(d, _, _) | IrInstr::Uge(d, _, _)
            | IrInstr::Zext(d, _, _) | IrInstr::Sext(d, _, _)
            | IrInstr::Trunc(d, _, _) | IrInstr::PtrToInt(d, _, _)
            | IrInstr::IntToPtr(d, _, _) | IrInstr::Gep(d, _, _, _)
            | IrInstr::Call(d, _, _) | IrInstr::MovImm(d, _, _) => *d,
            IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {
                panic!("Store/MemCopy has no dest")
            }
        }
    }
}

This is used by optimization passes (e.g., dead code elimination) and register allocation to find all definitions without matching on every variant.

    assert_eq!(phi.sources.len(), 2);
}

Common Pitfalls

  1. Store/MemCopy have no destination: Unlike every other instruction, Store and MemCopy do not produce a value. The dest() method panics on these. Optimization passes must handle this case explicitly.
  2. Signed vs. unsigned comparisons: Using Slt when Ult is intended will give wrong results for negative numbers (e.g., -1 as u32 > 1 as u32 in unsigned, but -1 < 1 in signed). The type checker must preserve signedness through to the IR.
  3. Missing phis for SSA correctness: If multiple control flow paths assign to the same variable, a phi node is required. The IR builder must insert phis at join points (e.g., after if/else).
  4. VReg numbering mismatch: num_vregs in IrFunction must be greater than the highest vreg index. Off-by-one errors here cause register allocation to miss the last register.
Why 36 instruction variants? This is a minimal set that can express all of L1S's language features while mapping cleanly to x86-64 instructions. In contrast, LLVM has hundreds of instruction opcodes. The simplicity of L1S's IR makes it easy to reason about and transform, at the cost of some expressiveness.

11   IR Building

From AST to IR

The IR Builder is the bridge between the high-level AST and the low-level intermediate representation. It walks the AST and emits IrInstrs into basic blocks, transforming structured control flow (if, while, for) into unstructured jumps and branches.

This phase is called lowering — we take a high-level, tree-shaped representation and produce a low-level, graph-shaped one. Every variable declaration becomes an Alloc + Store pair. Every expression becomes a sequence of instructions that compute into a virtual register. Every control flow construct becomes a pattern of basic blocks connected by Jump and Branch terminators.

The IrBuilder Struct

The builder is stateful, accumulating an IrModule as it traverses the AST:

src/ir_build.rs — IrBuilder struct
pub struct IrBuilder<'a> {
    pub module: IrModule,
    pub types: &TypeArena,
    pub annotations: &HashMap<Span, TypeId>,
    pub diag: &mut Diagnostics,
    prim_void: TypeId,
    prim_bool: TypeId,
    prim_char: TypeId,
    prim_i32: TypeId,
    current_func: Option<IrFunction>,
    current_block: Option<IrBlockId>,
    next_vreg: u32,
    next_block: u32,
    next_string: u32,
    vars: Vec<HashMap<String, VReg>>,
}

The key fields:

  • module: The IrModule being built. Functions, globals, and strings accumulate here.
  • current_func / current_block: Which function and block we're currently emitting into. The builder operates in "insertion-point" mode.
  • next_vreg: Counter for virtual register allocation. Each call to alloc_vreg_in increments func.num_vregs.
  • vars: A stack of scope maps. Each scope maps variable names to their VReg (the Alloc'd slot, not the loaded value).

The Build Driver: build()

The entry point iterates over top-level declarations:

src/ir_build.rs — entry point
pub fn build(mut self, prog: &AstProgram) -> IrModule {
    for decl in &prog.decls {
        match decl {
            AstDecl::Func(f) => self.build_func(f),
            AstDecl::StaticVar(sv) => self.build_global(sv),
            _ => {}
        }
    }
    self.module
}

We skip struct, enum, and spec declarations — those are type-level constructs already handled by the type checker. Only functions and static variables produce IR.

Function Setup: build_func()

Every function gets an entry block, parameter setup via Alloc + Store, and a default void return if none exists:

src/ir_build.rs — build_func
fn build_func(&mut self, func: &AstFuncDecl) {
    let return_type = self.annotations.get(&func.return_type.span()).copied()
        .unwrap_or_else(|| self.prim_i32);

    let mut ir_func = IrFunction {
        name: func.name.clone(),
        return_type,
        params: Vec::new(),
        blocks: Vec::new(),
        num_vregs: 0,
        entry_block: IrBlockId::new(0),
    };

    let entry = self.new_block_in(&mut ir_func, "entry");
    ir_func.entry_block = entry;
    self.current_block = Some(entry);
    self.vars.push(HashMap::new());

    for p in &func.params {
        let pt = self.type_from_ast(&p.type_);
        let vreg = self.alloc_vreg_in(&mut ir_func);
        let param_vreg = self.alloc_vreg_in(&mut ir_func);
        ir_func.params.push(IrParam {
            name: p.name.clone(), type_: pt, vreg: param_vreg,
        });
        self.emit_in(&mut ir_func, IrInstr::Alloc(vreg, pt));
        self.emit_in(&mut ir_func, IrInstr::Store(param_vreg, vreg));
        self.vars.last_mut().unwrap().insert(p.name.clone(), vreg);
    }

    self.build_block_in(&mut ir_func, &func.body);
    self.vars.pop();

    if self.current_block.is_some() {
        let ret_vreg = self.alloc_vreg_in(&mut ir_func);
        self.emit_in(&mut ir_func, IrInstr::Alloc(ret_vreg, self.prim_void));
        self.set_term_in(&mut ir_func, IrTerminator::Return(ret_vreg));
    }

    self.module.functions.push(ir_func);
}

Step-by-step:

  1. Look up the return type from the annotation map (set by the type checker).
  2. Create an empty IrFunction with num_vregs: 0.
  3. Create the entry block — every function starts here.
  4. Push a new scope onto self.vars.
  5. For each parameter: allocate two vregs — one for the parameter slot (Alloc) and one for the parameter value itself. The parameter value is Stored into the slot. The slot vreg is recorded in the scope so identifier references find it.
  6. Build the function body, then pop the scope.
  7. If no terminator was set (e.g. missing return), emit a default void return.
Why Alloc + Store for parameters? This ensures every variable access follows the same pattern: Load from a slot. Parameters are just pre-initialized variables. The code generator handles them identically to local variables — both live on the stack.

Global Variables: build_global()

fn build_global(&mut self, sv: &AstStaticVarDecl) {
    let type_ = self.type_from_ast(&sv.type_);
    self.module.globals.push(IrGlobal {
        name: sv.name.clone(),

Globals are simple — register their name and type. Initialization is left as None for now. The codegen phase emits them in the .data section.

Statement Lowering: build_stmt_in()

This is the heart of the lowering process. Every AST statement becomes zero or more IR instructions inside basic blocks:

src/ir_build.rs — build_stmt_in dispatch
fn build_stmt_in(&mut self, func: &mut IrFunction, stmt: &AstStmt) {
    match stmt {
        AstStmt::VarDecl(v) => {
            let t = self.type_from_ast_opt(&v.type_);
            let vreg = self.alloc_vreg_in(func);
            self.emit_in(func, IrInstr::Alloc(vreg, t));
            if let Some(ref init) = v.init {
                let init_vreg = self.build_expr_in(func, init);
                self.emit_in(func, IrInstr::Store(init_vreg, vreg));
            }
            if let Some(scope) = self.vars.last_mut() {
                scope.insert(v.name.clone(), vreg);
            }
        }
        AstStmt::Return(r) => {
            let val = self.build_expr_in(func, &r.value);
            self.set_term_in(func, IrTerminator::Return(val));
            self.current_block = None;
        }
        AstStmt::If(i) => { /* see below */ }
        AstStmt::While(w) => { /* see below */ }
        AstStmt::For(f) => { /* see below */ }
        AstStmt::Block(b) => self.build_block_in(func, b),
        AstStmt::Expr(e) => { self.build_expr_in(func, e); }
        _ => {}
    }
}

Variable Declaration

Every let x: i32 = expr; produces three things:

  1. Alloc %vreg, type — reserve a stack slot
  2. Evaluate the initializer (returns a vreg containing the value)
  3. Store %init, %vreg — write into the slot

The slot vreg is recorded in the current scope, so later Ident references can load from it.

Return Statement

A return expr; evaluates the expression, emits Return(val) as the block terminator, then sets current_block = None. The None signal is critical — it tells subsequent statements that the block was terminated. If we didn't clear it, we'd emit dead code after the return.

Control Flow Lowering

Structured control flow (if, while, for) is lowered to basic block graphs. Each construct follows a fixed pattern.

If / Else

src/ir_build.rs — If lowering
AstStmt::If(i) => {
    let cond = self.build_expr_in(func, &i.cond);
    let then_bb = self.new_block_in(func, "if.then");
    let merge_bb = self.new_block_in(func, "if.end");
    let has_else = i.else_branch.is_some();
    let else_bb = if has_else {
        self.new_block_in(func, "if.else")
    } else {
        merge_bb
    };

    let else_target = if has_else { else_bb } else { merge_bb };
    self.set_term_in(func, IrTerminator::Branch(cond, then_bb, else_target));

    self.current_block = Some(then_bb);
    self.build_block_in(func, &i.then_block);
    if self.current_block.is_some() {
        self.set_term_in(func, IrTerminator::Jump(merge_bb));
    }

    if let Some(ref else_branch) = i.else_branch {
        self.current_block = Some(else_bb);
        self.build_stmt_in(func, else_branch);
        if self.current_block.is_some() {
            self.set_term_in(func, IrTerminator::Jump(merge_bb));
        }
    }

    self.current_block = Some(merge_bb);
}

For Loop

The for loop is the most complex pattern — it decomposes into five blocks:

src/ir_build.rs — For lowering
AstStmt::For(f) => {
    let init_bb = self.new_block_in(func, "for.init");
    let header_bb = self.new_block_in(func, "for.header");
    let body_bb = self.new_block_in(func, "for.body");
    let update_bb = self.new_block_in(func, "for.update");
    let exit_bb = self.new_block_in(func, "for.exit");

    if self.current_block.is_some() {
        self.set_term_in(func, IrTerminator::Jump(init_bb));
    }

    self.current_block = Some(init_bb);
    self.build_expr_in(func, &f.init);
    self.set_term_in(func, IrTerminator::Jump(header_bb));

    self.current_block = Some(header_bb);
    let cond = self.build_expr_in(func, &f.cond);
    self.set_term_in(func, IrTerminator::Branch(cond, body_bb, exit_bb));

    self.current_block = Some(body_bb);
    self.build_block_in(func, &f.body);
    if self.current_block.is_some() {
        self.set_term_in(func, IrTerminator::Jump(update_bb));
    }

    self.current_block = Some(update_bb);
    self.build_expr_in(func, &f.update);
    self.set_term_in(func, IrTerminator::Jump(header_bb));

    self.current_block = Some(exit_bb);
}
For loop block structure:

prev → for.init → for.header ↓ for.body → for.update → for.exit ↵ update → header

The for loop pattern generalizes to: for.init → for.header → (branch body/exit) → for.body → for.update → for.header. The back-edge from for.update to for.header is what makes it a loop.

Expression Lowering: build_expr_in()

Every expression produces IR that computes into a virtual register:

src/ir_build.rs — build_expr_in (selected cases)
fn build_expr_in(&mut self, func: &mut IrFunction, expr: &AstExpr) -> VReg {
    match expr {
        AstExpr::Int(s, _) => {
            let t = self.prim_i32;
            let vreg = self.alloc_vreg_in(func);
            self.emit_in(func, IrInstr::MovImm(vreg, s.parse::<u64>().unwrap_or(0), t));
            vreg
        }
        AstExpr::Ident(name, _) => {
            let vreg = self.lookup_var(name);
            let load = self.alloc_vreg_in(func);
            self.emit_in(func, IrInstr::Load(load, vreg));
            load
        }
        AstExpr::Binary(op, lhs, rhs, _) => {
            let l = self.build_expr_in(func, lhs);
            let r = self.build_expr_in(func, rhs);
            let vreg = self.alloc_vreg_in(func);
            let instr = match op {
                BinaryOp::Add => IrInstr::Add(vreg, l, r),
                BinaryOp::Sub => IrInstr::Sub(vreg, l, r),
                BinaryOp::Mul => IrInstr::Mul(vreg, l, r),
                BinaryOp::Eq  => IrInstr::Eq(vreg, l, r),
                // ... other operators follow the same pattern
            };
            self.emit_in(func, instr);
            vreg
        }
        AstExpr::Call(callee, args, _) => {
            // ... builds args, emits a Call vreg
        }
        // ... Index, Field, Cast, etc.
    }
}

The expression lowering creates a new vreg for each intermediate result. This is SSA-like — each vreg is written exactly once (except for the Alloc+Store+Load pattern for variables, which breaks strict SSA). The vreg numbering is dense: %0, %1, %2, ... within each function.

Variable Lookup

The lookup_var method walks the scope stack from innermost to outermost, returning the Alloc vreg for the variable's stack slot:

fn lookup_var(&self, name: &str) -> VReg {
    for scope in self.vars.iter().rev() {
        if let Some(&vreg) = scope.get(name) {
            return vreg;
        }
    }
    let v = VReg::new(self.next_vreg);
    v
}

Note: the return value is the slot vreg, not the loaded value. The caller (build_expr_in for Ident) wraps the result in Load(v, slot) to produce the value. This separation means the slot holds an address and Load dereferences it — the slot is effectively a pointer to the variable's stack location.

Helper Methods

Four small helpers make the builder ergonomic:

fn alloc_vreg_in(&mut self, func: &mut IrFunction) -> VReg {
    let v = VReg::new(func.num_vregs);
    func.num_vregs += 1;
    v
}

fn new_block_in(&mut self, func: &mut IrFunction, name: &str) -> IrBlockId {
    let id = IrBlockId::new(func.blocks.len() as u32);
    func.blocks.push(IrBlock {
        id,
        name: name.to_string(),
        phis: Vec::new(),
        instrs: Vec::new(),
        terminator: IrTerminator::Jump(id),
    });
    id
}

fn emit_in(&mut self, func: &mut IrFunction, instr: IrInstr) {
    if let Some(block_id) = self.current_block {
        let idx = block_id.index();
        if idx < func.blocks.len() {
            func.blocks[idx].instrs.push(instr);
        }
    }
}

fn set_term_in(&mut self, func: &mut IrFunction, term: IrTerminator) {
    if let Some(block_id) = self.current_block {
        let idx = block_id.index();
        if idx < func.blocks.len() {
            func.blocks[idx].terminator = term;
        }
    }
}

VReg allocation scheme: VRegs are allocated densely per function, starting at 0. The total num_vregs tells the register allocator how many virtual registers the function uses. VRegs are never reused — each is written by exactly one instruction (its "definition" site). This is SSA-like but not strict SSA because variables use Alloc+Store+Load which implicitly creates multiple definitions per vreg.

Block naming convention: Blocks are named by their control flow origin:

  • entry — function entry
  • if.then, if.else, if.end — if/else structure
  • while.header, while.body, while.exit — while loop
  • for.init, for.header, for.body, for.update, for.exit — for loop

These names are purely human-readable — blocks are identified by their IrBlockId (which is just a dense u32 index). But the names appear in debug output and IR dumps, making them invaluable for debugging.

Traced Example: func main(i32) { let x: i32 = 1+2; return x; }

Let's trace the builder step by step on this function:

Step 1: Function setup

  • Create IrFunction named "main", return type = i32.
  • Create bb0 named "entry". Set as entry block and current block.
  • Push a new scope.

Step 2: Parameter

  • Parameter p0: i32 → %0 = Alloc i32, %1 = (param value), then Store %1, %0.
  • Scope: {"p0": %0}

Step 3: Variable declaration

  • let x: i32 = 1+2;
  • %2 = Alloc i32 (slot for x)
  • Evaluate 1+2:
    • %3 = MovImm 1, i32
    • %4 = MovImm 2, i32
    • %5 = Add %3, %4
  • Store %5, %2 (store result into x's slot)
  • Scope: {"p0": %0, "x": %2}

Step 4: Return statement

  • return x;
  • Look up x → %0... wait, x is %2. Load %6, %2
  • Return %6
  • current_block = None

Final IR:

main:
  ; Parameters: p0 = %1
  ; vregs: %0, %1, %2, %3, %4, %5, %6

bb0 (entry):
  %0 = Alloc i32           ; param slot
  Store %1, %0                ; store param into slot
  %2 = Alloc i32           ; x's slot
  %3 = MovImm 1, i32
  %4 = MovImm 2, i32
  %5 = Add %3, %4          ; 1 + 2

  %6 = Load %2              ; load x for return
  Return %6

This is the complete IR for the function. The Alloc instructions tell the register allocator "this vreg needs a stack slot." The Store and Load instructions manage data movement between slots and values. The Add computes in virtual registers.

Common Pitfalls & Debugging Tips

  1. Forgetting to set current_block = None after a terminator: If you emit a Return or Branch but don't clear current_block, subsequent statements emit dead code into the terminated block. Always set current_block = None after setting a terminator that ends the block's lifetime.
  2. Not checking current_block.is_some() before adding Jump: After building then/else bodies, always check if the block still exists. A return inside the then-block would have set current_block = None, and we must not emit a Jump into a dead block.
  3. Variable shadowing: The scope stack correctly handles shadowing — lookup_var returns the innermost binding. But if you forget to push/pop scopes in build_block_in, variables leak between blocks.
  4. Dangling Alloc without Store: If a variable is declared but never assigned, its slot is allocated but never written. This is fine for uninitialized variables, but the optimizer can't remove the Alloc because it might be used by a later Load.
Debugging the IR builder: Add a custom Display impl or debug printer for IrFunction that dumps blocks in order with their instructions and terminators. Call it after each function is built. This catches 90% of builder bugs.

12   Optimization

Why Optimize?

The IR builder emits straightforward, unoptimized code. Every expression gets new vregs, every variable gets a stack slot, and every intermediate result is fully computed. This is correct but wasteful. Optimization refines the IR to eliminate dead computation, fold constants, and remove unnecessary operations.

In L1S, optimization is optional in principle but essential in practice — without it, even trivial programs like return 1+2; would emit three instructions instead of one. The optimizer runs as a fixed-point loop: repeatedly apply passes until no change occurs or we hit a limit.

The Optimizer Driver: optimize()

src/ir_opt.rs — main driver
pub fn optimize(module: &mut IrModule) {
    for _ in 0..100 {
        let changed = constant_folding(module)
            | dead_code_elimination(module)
            | copy_propagation(module);
        if !changed {
            break;
        }
    }
    remove_unreachable_blocks(module);
}

The driver runs up to 100 iterations. Each iteration applies three passes in sequence. If none of them changed anything, we've reached a fixed point and stop. After the loop, we run one cleanup pass (removing unreachable blocks) that doesn't count toward the iteration limit.

Why a fixed-point loop? Passes interact: constant folding may create new dead code (a folded expression's operands become unused). DCE removes it, which may enable more copy propagation. Running them repeatedly until stablization catches all these cascading effects. The 100-iteration cap prevents infinite loops from buggy passes.

Pass 1: Constant Folding

What it does: When both operands of Add, Sub, or Mul are known constants, replace the instruction with a MovImm of the computed result.

Why it matters: Computations like 1 + 2, x * 0, and 42 - 7 appear frequently in source code. Without folding, every constant expression survives into the generated assembly as runtime arithmetic. Folding eliminates them entirely.

src/ir_opt.rs — constant_folding
fn constant_folding(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut const_values: HashMap<VReg, u64> = HashMap::new();

        for block in &mut func.blocks {
            let mut replacements: Vec<(usize, IrInstr)> = Vec::new();

            for (i, instr) in block.instrs.iter().enumerate() {
                match instr {
                    IrInstr::MovImm(d, v, _) => {
                        const_values.insert(*d, *v);
                    }
                    IrInstr::Add(d, a, b) => {
                        if let (Some(av), Some(bv)) =
                                (const_values.get(a), const_values.get(b)) {
                            let val = av.wrapping_add(*bv);
                            replacements.push((i, IrInstr::MovImm(*d, val, ...)));
                            const_values.insert(*d, val);
                            changed = true;
                        }
                    }
                    IrInstr::Sub(d, a, b) => { /* same pattern */ }
                    IrInstr::Mul(d, a, b) => { /* same pattern */ }
                    _ => {}
                }
            }

            for (i, replacement) in replacements {
                block.instrs[i] = replacement;
            }
        }
    }
    changed
}

The pass maintains a HashMap<VReg, u64> mapping vregs to their known constant values. As it scans instructions left to right:

  1. MovImm: Record the vreg and its value.
  2. Add/Sub/Mul: If both source operands are in the map, compute the result and schedule a replacement.
  3. Other instructions: Clear the constant map (in a more aggressive version, but here we just skip them — the entries remain, which is a simplification).
Simplification: The current implementation doesn't clear the constant map when it encounters a non-foldable instruction. This means it can incorrectly fold across blocks in some edge cases. A production optimizer would kill entries when encountering stores or calls that might invalidate them.

Pass 2: Dead Code Elimination (DCE)

What it does: Removes instructions whose destination vreg is never used by any other instruction, terminator, or phi node.

Why it matters: The IR builder emits instructions for every expression, even if the result is never used (e.g., a discarded expression statement like 1+2;). DCE cleans these up. It's especially important after constant folding — when Add gets replaced by MovImm, the original operands' definitions may become dead.

src/ir_opt.rs — dead_code_elimination
fn dead_code_elimination(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut uses: HashSet<VReg> = HashSet::new();

        // Collect all used vregs
        for block in &func.blocks {
            for instr in &block.instrs {
                for op in instr_uses(instr) {
                    uses.insert(op);
                }
            }
            for phi in &block.phis {
                for (src, _) in &phi.sources {
                    uses.insert(*src);
                }
            }
            match &block.terminator {
                IrTerminator::Branch(cond, _, _) => { uses.insert(*cond); }
                IrTerminator::Return(val) => { uses.insert(*val); }
                _ => {}
            }
        }

        // Remove unused instructions
        for block in &mut func.blocks {
            block.instrs.retain(|instr| {
                let keep = match instr {
                    // Side-effecting instructions are always kept
                    IrInstr::Store(_, _)
                    | IrInstr::Call(_, _, _)
                    | IrInstr::MemCopy(_, _, _) => true,
                    _ => {
                        let dest = instr.dest();
                        uses.contains(&dest)
                    }
                };
                if !keep {
                    changed = true;
                }
                keep
            });
        }
    }
    changed
}

Two-phase design:

  1. Collect uses: Walk every instruction, phi, and terminator. For each operand they reference, add the operand's vreg to a HashSet. This tells us which vregs are needed.
  2. Retain or remove: For each instruction, check if its destination vreg is in the uses set. If not, it's dead and gets removed. Instructions with side effects (Store, Call, MemCopy) are always kept, because they affect memory even if their dest vreg is unused.

Pass 3: Copy Propagation

What it does: When a Load %d, %s is seen, subsequent loads from %d can be replaced with a direct reference to %s. More precisely, it tracks copies: Load %d, %s means "the value in %d came from %s." If %s was itself a copy of another vreg, we forward the original.

Why it matters: The IR builder emits Load for every variable reference. If a variable is loaded multiple times, copy propagation can eliminate redundant loads. In L1S's simple implementation, this pass is lightweight but lays groundwork for more aggressive optimization.

src/ir_opt.rs — copy_propagation
fn copy_propagation(module: &mut IrModule) -> bool {
    let mut changed = false;
    for func in &mut module.functions {
        let mut copies: HashMap<VReg, VReg> = HashMap::new();

        for block_idx in 0..func.blocks.len() {
            let block = &func.blocks[block_idx];
            for instr in &block.instrs {
                if let IrInstr::Load(d, s) = instr {
                    if let Some(&orig) = copies.get(s) {
                        copies.insert(*d, orig);
                        changed = true;
                    } else {
                        copies.insert(*d, *s);
                    }
                }
            }
        }
    }
    changed
}

The current implementation only records copies but doesn't rewrite uses. It's a skeleton: it detects that %d is a copy of %s (or transitively of orig), but doesn't yet rename operands. A full implementation would walk all uses of %d and replace them with %s.

Copy propagation vs. SSA: In a proper SSA-based compiler, copy propagation is trivial — you just rename. L1S doesn't use SSA, so copy propagation requires a dataflow analysis to know where copies are valid. The current implementation is deliberately conservative.

Pass 4: Remove Unreachable Blocks

What it does: After all other optimizations, some blocks may become unreachable (e.g., the false-branch target of a branch whose condition is now constant-true). This pass removes them.

Why it matters: Dead blocks waste space and confuse debugging. They can also cause problems for the register allocator (which iterates all blocks).

src/ir_opt.rs — remove_unreachable_blocks
fn remove_unreachable_blocks(module: &mut IrModule) {
    for func in &mut module.functions {
        let entry = func.entry_block;
        let mut reachable: HashSet<IrBlockId> = HashSet::new();
        let mut worklist: Vec<IrBlockId> = vec![entry];
        reachable.insert(entry);

        while let Some(block_id) = worklist.pop() {
            let idx = block_id.index();
            if idx >= func.blocks.len() { continue; }
            let successors = block_successors(&func.blocks[idx]);
            for succ in successors {
                if reachable.insert(succ) {
                    worklist.push(succ);
                }
            }
        }

        func.blocks.retain(|b| reachable.contains(&b.id));

        // Clean up phis and terminators that reference removed blocks
        for block in &mut func.blocks {
            block.phis.iter_mut().for_each(|phi| {
                phi.sources.retain(|(_, pred)| reachable.contains(pred));
            });
            if let IrTerminator::Jump(target) = &block.terminator {
                if !reachable.contains(target) {
                    block.terminator = IrTerminator::Jump(block.id);
                }
            }
        }
    }
}

Algorithm: BFS from the entry block. Every block reachable via block_successors is marked. Then we filter the block list, keeping only reachable ones. Finally, we clean up phi source entries that reference dead predecessors, and redirect Jump terminators that point to removed blocks.

Helper Functions

fn instr_uses(instr: &IrInstr) -> Vec<VReg> {
    match instr {
        IrInstr::Alloc(_, _)
        | IrInstr::Neg(_, _)
        | IrInstr::Not(_, _)
        | IrInstr::BitNot(_, _)
        | IrInstr::MovImm(_, _, _) => vec![],
        IrInstr::Load(_, s) => vec![*s],
        IrInstr::Store(v, p) => vec![*v, *p],
        IrInstr::Add(_, a, b)
        | IrInstr::Sub(_, a, b)
        | IrInstr::Mul(_, a, b)
        | IrInstr::SDiv(_, a, b)
        // ... all binary ops: returns [*a, *b]
        | IrInstr::Gep(_, a, b, _) => vec![*a, *b],
        IrInstr::Call(_, _, args) => args.clone(),
        // ... unary ops: returns [*s]
    }
}

fn block_successors(block: &IrBlock) -> Vec<IrBlockId> {
    match &block.terminator {
        IrTerminator::Jump(target) => vec![*target],
        IrTerminator::Branch(_, t, f) => vec![*t, *f],
        IrTerminator::Return(_) => vec![],
    }
}

instr_uses returns the set of vregs that an instruction reads from. Alloc and MovImm read nothing. Load reads its source. Binary ops read both operands. Call reads all argument vregs.

block_successors returns the blocks that control flow can reach from this block's terminator — essential for BFS reachability analysis.

Common Pitfalls & Debugging Tips

  1. Forgetting to mark side-effectful instructions: If DCE removes a Store or Call, the program becomes incorrect. Always check: "does this instruction affect memory, I/O, or global state?" If yes, keep it regardless of whether its dest vreg is used.
  2. Incorrect constant map management: The constant folding pass doesn't clear entries when it encounters a Store or Call that could modify a vreg's value. This is a known limitation — in a real compiler, the constant map should be invalidated at side-effecting operations.
  3. Fixed-point termination: Ensure the loop always terminates. Each pass should return false when it makes no changes. The 100-iteration cap is a safety net, but you should design passes to converge naturally.
  4. Unreachable block cleanup: After removing blocks, always clean up references. A dangling Jump to a removed block will crash the code generator. Phi node source entries referencing removed predecessors should be filtered.
Debugging optimizer output: The best debugging strategy is to print the IR before and after each pass. Add a dump_ir() function that prints each function's blocks, instructions, and terminators. Run it between passes to see exactly what changed.

13   Register Allocation

The Problem

The IR builder creates virtual registers (VReg) — infinite, numbered temporaries. Real x86-64 hardware has only 16 general-purpose registers (and some are reserved for the stack pointer, base pointer, etc.). Register allocation maps the unbounded set of virtual registers onto the bounded set of physical registers.

When there aren't enough physical registers, values must spill to the stack — stored in a stack slot and loaded back when needed. This is the fundamental tension in register allocation: keeping values in registers (fast) vs. memory (slow but abundant).

The L1S Strategy: Spill Everything

L1S takes the simplest possible approach: every vreg gets its own stack slot. No register allocation at all — values live exclusively on the stack, and the code generator emits memory-to-memory operations via load-store sequences.

Why spill everything? Register allocation is a classic NP-hard problem (graph coloring). Production compilers use heuristics like linear scan or Chaitin-Briggs. For a teaching compiler, the complexity isn't justified. Spill-everything is trivially correct, trivially simple, and still produces working code. The performance cost is significant (every operation involves memory traffic), but the compiler is functional.

The StackAllocator Struct

src/regalloc.rs — StackAllocator
pub struct StackAllocator {
    pub slot_map: HashMap<VReg, i64>,
    pub stack_size: i64,
}

impl StackAllocator {
    pub fn new() -> Self {
        StackAllocator {
            slot_map: HashMap::new(),
            stack_size: 0,
        }
    }

The allocator maintains two things:

  • slot_map: Maps each VReg to an i64 stack offset (negative, relative to rbp).
  • stack_size: Total stack frame size (grows negative as slots are added).

Making a Slot: make_slot()

fn make_slot(&mut self, vreg: VReg, _type_: TypeId) {
    if !self.slot_map.contains_key(&vreg) {
        self.stack_size -= 8;
        self.slot_map.insert(vreg, self.stack_size);
    }
}

Every slot is exactly 8 bytes, regardless of the actual type. The slot is allocated by decrementing stack_size by 8 and assigning the new value as the offset. Since we start at 0 and decrement, the first slot is at offset -8, the second at -16, etc.

8 bytes per slot: This wastes space for small types (a bool or i8 gets 8 bytes) but simplifies the code generator enormously — all stack slots are uniformly sized and aligned. The x86-64 ABI requires 8-byte stack alignment anyway. The waste is (8 - actual_size) bytes per slot, which is negligible for a teaching compiler.

The Allocation Driver: allocate()

pub fn allocate(&mut self, func: &IrFunction) {
    // 1. Parameters
    for param in &func.params {
        self.make_slot(param.vreg, param.type_);
    }

    // 2. Instructions
    for block in &func.blocks {
        for instr in &block.instrs {
            match instr {
                IrInstr::Alloc(vreg, type_) => {
                    self.make_slot(*vreg, *type_);
                }
                // Store and MemCopy don't define a new vreg
                IrInstr::Store(_, _) | IrInstr::MemCopy(_, _, _) => {}
                // Everything else: allocate a slot for the destination
                instr => {
                    let dest = instr.dest();
                    self.make_slot(dest, TypeId::new(11));
                }
            }
        }
        // 3. Phi nodes
        for phi in &block.phis {
            self.make_slot(phi.dest, phi.type_);
        }
    }

    // 4. Align to 16-byte boundary
    self.stack_size = self.stack_size & !15;
}

The allocation order matters:

  1. Parameters first: Their slots get the highest addresses (closest to rbp).
  2. Instruction destinations: Every IrInstr variant that writes a vreg gets a slot. We check for Alloc specially (it already carries a type), but for everything else we just allocate a generic 8-byte slot. Store and MemCopy are skipped because they don't define a new vreg.
  3. Phi destinations: Each phi node gets a slot too.
  4. Alignment: After all slots are allocated, we round stack_size down to the nearest 16-byte boundary. This ensures the stack frame stays 16-byte aligned as required by the SysV x86-64 ABI.
Why align to 16 bytes? The x86-64 ABI requires the stack to be 16-byte aligned before a call instruction. The function prologue pushes rbp (subtracting 8 from rsp), so the frame must have an additional 8 bytes of padding to maintain alignment. The alignment step stack_size & !15 rounds down to the nearest multiple of 16, which may add up to 15 bytes of padding.

Slot Lookup: slot()

pub fn slot(&self, vreg: VReg) -> i64 {
    self.slot_map.get(&vreg).copied().unwrap_or(0)
}

The code generator calls slot(vreg) to get the rbp-relative offset for any virtual register. If the vreg wasn't allocated a slot (a bug), it returns 0 (pointing to the saved rbp on the stack — also a bug, but at least it doesn't crash).

Let's trace the calculation. With 3 vregs:

  • Start: stack_size = 0
  • Slot for %0: stack_size = -8
  • Slot for %1: stack_size = -16
  • Slot for %2: stack_size = -24
  • Align: -24 & !15 = -32

The final stack_size is -32, meaning the function's frame is 32 bytes. The prologue will subtract 32 from rsp to allocate space for all slots, then the epilogue will add 32 back before returning.

Why Spill Everything Works

The spill-everything strategy is correct because of the IR's design:

  1. Alloc reserves a slot. The code generator maps this to a stack offset immediately.
  2. Load %d, %s becomes mov rax, [rbp + slot(s)]; mov [rbp + slot(d)], rax — both source and dest are stack slots.
  3. Add %d, %a, %b becomes mov rax, [rbp + slot(a)]; add rax, [rbp + slot(b)]; mov [rbp + slot(d)], rax — load both operands, compute, store result.
  4. Store %v, %p becomes mov rax, [rbp + slot(v)]; mov [rbp + slot(p)], rax.

Every instruction degenerates to load-store-load-store. The CPU's registers are used only as temporary scratch (typically using rax and rcx), never for persistent values. This is slow but trivially correct.

The tradeoff: Spill-everything means every IR instruction compiles to 2-3 machine instructions (load, operate, store). A good register allocator might keep half the vregs in registers, reducing memory traffic by 2-3x. For L1S, the simplicity win outweighs the performance loss — the compiler is for learning, not production.

Common Pitfalls & Debugging Tips

  1. Forgetting to allocate slots for phi nodes: Phis are often overlooked because they live in the IrBlock.phis list, not in the instruction stream. If you skip them, the code generator will crash trying to find a slot for a phi destination.
  2. Not handling Store and MemCopy: These instructions don't define a destination vreg. Calling instr.dest() on them panics. Always skip them explicitly.
  3. Stack alignment: The x86-64 ABI requires 16-byte stack alignment at the point of a call. After the prologue pushes rbp (8 bytes), the frame size must be adjusted to restore 16-byte alignment. The & !15 trick works because we're working with negative offsets — but verify with actual numbers to be sure.
  4. Slot offset sign: Stack slots are at negative offsets from rbp (since the stack grows downward). The code generator uses [rbp + offset] with negative offsets. If a slot returns 0, it points to the saved rbp, which will cause subtle corruption.
Performance warning: The spill-everything strategy produces extremely memory-heavy code. A single Add becomes three memory operations: load lhs, load rhs, store result. For a toy compiler this is fine, but be aware that the generated programs will run 10-50x slower than an optimized compiler's output.

14   Code Generation

From IR to Assembly

Code generation is the point where theory meets reality. After all the parsing, type checking, IR construction, and register allocation, we finally produce actual x86-64 assembly. The Codegen struct is responsible for translating an IrModule into a NASM-syntax assembly string that can be assembled and linked into a working executable.

Pipeline position:

IR Module → Codegen → .asm file → nasm → .o file → cc (linker) → executable

The Codegen Struct

The struct is simple: it holds a mutable String buffer for the assembly output and a reference to the TypeArena for type queries:

src/codegen.rs — Codegen struct
pub struct Codegen<'a> {
    pub output: String,
    pub types: &TypeArena,
}

impl<'a> Codegen<'a> {
    pub fn new(types: &TypeArena) -> Self {
        Codegen { output: String::new(), types }
    }
}

The output buffer is cleared on each call to generate(), so a single Codegen instance can be reused. The types reference is only needed for string escape handling (via TypeArena size queries by escape_string).

The Main Entry Point: generate()

generate() is the public API. It produces an assembly file with three sections:

src/codegen.rs — generate()
pub fn generate(&mut self, module: &IrModule) -> String {
    self.output.clear();
    self.emit("section .text");
    self.emit("global main");
    self.emit("");

    for func in &module.functions {
        self.generate_func(func);
    }

    if !module.strings.is_empty() {
        self.emit("section .rodata");
        for s in &module.strings {
            self.emit(&format!("str_{}:", s.id.0));
            self.emit(&format!("  db \"{}\", 0", self.escape_string(&s.value)));
        }
    }

    if !module.globals.is_empty() {
        self.emit("section .data");
        for g in &module.globals {
            self.emit(&format!("global {}", g.name));
            self.emit(&format!("{}:", g.name));
            self.emit("  dd 0");
        }
    }

    self.output.clone()
}

The three sections:

  • section .text: Executable code. Every function becomes a global label followed by its instructions. The entry point main is declared global so the linker can find it.
  • section .rodata: Read-only data. String literals are emitted as db directives with NASM escape sequences. Each string gets a unique label like str_0, str_1, etc., matching the StringId used in the IR.
  • section .data: Mutable data. Global variables are emitted with dd 0 (4 bytes initialized to zero) and declared global for linker visibility.
NASM syntax: L1S uses NASM's db (declare byte) for string data and dd (declare dword) for 32-bit integer globals. The global directive makes symbols visible to the linker, equivalent to .globl in AT&T syntax.

Function Code Generation: generate_func()

Each function is translated with a consistent prologue, per-block code, and epilogue:

src/codegen.rs — generate_func()
fn generate_func(&mut self, func: &IrFunction) {
    let mut alloc = StackAllocator::new();
    alloc.allocate(func);

    self.emit(&format!("; --- {} ---", func.name));
    self.emit(&format!("global {}", func.name));
    self.emit(&format!("{}:", func.name));

    self.emit("  push rbp");
    self.emit("  mov rbp, rsp");
    if alloc.stack_size != 0 {
        self.emit(&format!("  sub rsp, {}", -alloc.stack_size));
    }

    for (i, param) in func.params.iter().enumerate() {
        let slot = alloc.slot(param.vreg);
        let reg = self.param_reg(i);
        if slot != 0 {
            self.emit(&format!("  mov [rbp{}], {}", self.signed(slot), reg));
        }
    }

    for block in &func.blocks {
        self.emit(&format!(".{}_{}:", func.name, block.id.0));
        for instr in &block.instrs {
            self.emit_instr(instr, &alloc);
        }
        self.emit_terminator(&block.terminator, &alloc, func);
    }

    self.emit(&format!(".return_{}:", func.name));
    if alloc.stack_size != 0 {
        self.emit(&format!("  add rsp, {}", -alloc.stack_size));
    }
    self.emit("  pop rbp");
    self.emit("  ret");
    self.emit("");
}

Step-by-step walkthrough:

  1. StackAllocator: A fresh StackAllocator is created and run on the function. It assigns each virtual register (VReg) a negative offset from rbp. The stack_size is the total space needed, rounded down to 16-byte alignment.
  2. Prologue (push rbp; mov rbp, rsp; sub rsp, N): Saves the caller's frame pointer, sets up our own frame, and allocates stack space. This is the standard x86-64 function prologue.
  3. Parameter stubs: Parameters arrive in registers (rdi, rsi, etc. per the System V ABI). Each is spilled to its stack slot so that all subsequent code accesses parameters via memory (consistent with the StackAllocator's model).
  4. Per-block label: Each basic block gets a label like .main_0: (function name + block ID). These are local labels (prefixed with . in NASM), scoped to the enclosing global label.
  5. Emit instructions: Every IrInstr in the block is translated to one or more NASM instructions via emit_instr.
  6. Emit terminator: The block terminator (Jump, Branch, or Return) is emitted. Return jumps to the shared return label rather than inlining the epilogue.
  7. Return label: A single .return_<name>: label handles add rsp, N (restore stack), pop rbp, and ret. All return paths converge here.
Why a single return label? If we inlined the epilogue at every Return terminator, the code would be larger but faster (no extra jump). L1S uses a shared return label for smaller code size. An optimizing compiler could choose either strategy.

Instruction Emission: emit_instr()

This is the heart of the code generator — a massive match on every IrInstr variant. Let's examine the key patterns:

Binary Operations (Add, Sub, Mul, And, Or, Xor)

Most binary operations use the binary helper:

fn binary(&mut self, op: &str, dest: &VReg, a: &VReg, b: &VReg, alloc: &StackAllocator) {
    self.load_to_rax(*a, alloc);
    self.load_to_reg(*b, "rcx", alloc);
    self.emit(&format!("  {} eax, ecx", op));
    self.store_from_rax(*dest, alloc);
}

// Usage in emit_instr:
IrInstr::Add(dest, a, b) => { self.binary("add", dest, a, b, alloc); }
IrInstr::Sub(dest, a, b) => self.binary("sub", dest, a, b, alloc),
IrInstr::Mul(dest, a, b) => self.binary("imul", dest, a, b, alloc),

The pattern is: rax = a; rcx = b; op eax, ecx; dest = rax. This is a three-address code style: both operands are loaded into registers, the operation is performed, then the result is stored back.

Division and Remainder

Division is special because x86-64 uses implicit register pairs:

IrInstr::SDiv(dest, a, b) => {
    self.load_to_rax(*a, alloc);
    self.emit("  cdq");          // sign-extend eax into edx:eax
    self.load_to_reg(*b, "rcx", alloc);
    self.emit("  idiv ecx");     // quotient in eax, remainder in edx
    self.store_from_rax(*dest, alloc);
}

IrInstr::SRem(dest, a, b) => {
    self.load_to_rax(*a, alloc);
    self.emit("  cdq");
    self.load_to_reg(*b, "rcx", alloc);
    self.emit("  idiv ecx");
    let d_slot = alloc.slot(*dest);  // remainder is in edx, not eax!
    if d_slot != 0 {
        self.emit(&format!("  mov [rbp{}], rdx", self.signed(d_slot)));
    }
}

cdq sign-extends eax into edx:eax (the dividend), and idiv ecx divides the 64-bit edx:eax pair by ecx. The quotient lands in eax, remainder in edx. This is why SRem reads from rdx rather than rax.

Immediate Moves

IrInstr::MovImm(dest, val, _) => {
    let d_slot = alloc.slot(*dest);
    if d_slot != 0 {
        self.emit(&format!("  mov qword [rbp{}], {}", self.signed(d_slot), val));
    }
}

Note the qword specifier: NASM needs to know the operand size when writing an immediate to memory.

Function Calls

IrInstr::Call(dest, name, args) => {
    for (i, arg) in args.iter().enumerate() {
        let reg = self.arg_reg(i);
        let slot = alloc.slot(*arg);
        if slot != 0 {
            self.emit(&format!("  mov {}, [rbp{}]", reg, self.signed(slot)));
        } else {
            self.emit(&format!("  xor {}, {}", reg, reg));
        }
    }
    self.emit(&format!("  call {}", name));
    let d_slot = alloc.slot(*dest);
    if d_slot != 0 {
        self.emit(&format!("  mov [rbp{}], rax", self.signed(d_slot)));
    }
}

Arguments are loaded into the System V argument registers (see below), then call is emitted. The return value in rax is immediately stored to the destination's stack slot.

GetElementPtr (GEP)

IrInstr::Gep(dest, base, index, stride) => {
    self.load_to_reg(*base, "rax", alloc);
    self.load_to_reg(*index, "rcx", alloc);
    if *stride > 1 {
        self.emit(&format!("  imul rcx, rcx, {}", stride));
    }
    self.emit("  add rax, rcx");
    let d_slot = alloc.slot(*dest);
    if d_slot != 0 {
        self.emit(&format!("  mov [rbp{}], rax", self.signed(d_slot)));
    }
}

GEP computes base + index * stride at runtime, producing a pointer. The stride is the element size (e.g., 4 for i32 arrays).

Terminator Emission: emit_terminator()

Three kinds of terminators control control flow between basic blocks:

fn emit_terminator(&mut self, term: &IrTerminator, alloc: &StackAllocator, func: &IrFunction) {
    match term {
        IrTerminator::Jump(target) => {
            self.emit(&format!("  jmp .{}_{}", func.name, target.0));
        }
        IrTerminator::Branch(cond, t, f) => {
            let slot = alloc.slot(*cond);
            if slot != 0 {
                self.emit(&format!("  mov rax, [rbp{}]", self.signed(slot)));
            } else {
                self.emit("  xor eax, eax");
            }
            self.emit("  test eax, eax");
            self.emit(&format!("  jnz .{}_{}", func.name, t.0));
            self.emit(&format!("  jmp .{}_{}", func.name, f.0));
        }
        IrTerminator::Return(val) => {
            let slot = alloc.slot(*val);
            if slot != 0 {
                self.emit(&format!("  mov rax, [rbp{}]", self.signed(slot)));
            } else {
                self.emit("  xor eax, eax");
            }
            self.emit(&format!("  jmp .return_{}", func.name));
        }
    }
}
  • Jump: Unconditional jump to a block label.
  • Branch: Conditional branch. Load the condition into rax, test eax, eax sets ZF if zero, jnz jumps to the true target if non-zero, otherwise falls through to the jmp to the false target.
  • Return: Load the return value into rax (the return value register per SysV ABI), then jump to the shared return label.

Comparison Instructions (cmp pattern)

Comparisons (Eq, Ne, Slt, Sgt, Sle, Sge) use a two-step pattern: compare, then set a byte based on the flags:

fn cmp(&mut self, setcc: &str, dest: &VReg, a: &VReg, b: &VReg, alloc: &StackAllocator) {
    self.load_to_rax(*a, alloc);      // rax = a
    self.load_to_reg(*b, "rcx", alloc);  // rcx = b
    self.emit("  cmp eax, ecx");       // compare
    self.emit(&format!("  {} al", setcc));  // sete/setne/setl/setg/...
    self.emit("  movzx eax, al");     // zero-extend byte to 32-bit
    self.store_from_rax(*dest, alloc);
}

cmp sets CPU flags. sete al writes 1 to al if equal (ZF set), 0 otherwise. movzx eax, al zero-extends the byte result to a full 32-bit register. The specific setcc mnemonic varies by comparison: sete for Eq, setne for Ne, setl for signed less-than, etc.

The x86-64 System V ABI Calling Convention

L1S targets Linux x86-64, which follows the System V AMD64 ABI. The key conventions are:

Argument registers (integer/pointer):
PositionRegisterPurpose in L1S
1strdiFirst parameter / first call argument
2ndrsiSecond parameter / second call argument
3rdrdxThird parameter / third call argument
4thrcxFourth parameter / fourth call argument
5thr8Fifth parameter / fifth call argument
6thr9Sixth parameter / sixth call argument

Return value: rax
Stack alignment: 16-byte alignment at call instruction
Caller-saved: rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11
Callee-saved: rbx, rbp, r12–r15

L1S implements this with two helper methods:

fn param_reg(&self, index: usize) -> &'static str {
    match index {
        0 => "rdi",
        1 => "rsi",
        2 => "rdx",
        3 => "rcx",
        4 => "r8",
        5 => "r9",
        _ => "rdi",
    }
}

fn arg_reg(&self, index: usize) -> &'static str {
    match index {
        0 => "rdi",
        1 => "rsi",
        2 => "rdx",
        3 => "rcx",
        4 => "r8",
        5 => "r9",
        _ => "rdi",
    }
}

They are intentionally identical — both parameters and call arguments use the same register assignment. The fallback _ => "rdi" is a bug/quirk: beyond 6 args, the ABI requires stack-based passing, but L1S silently reuses rdi. This works for L1S's typical programs which rarely exceed 6 parameters.

6-argument limit: The current code silently mishandles more than 6 arguments. A production compiler would either detect this at compile time and emit an error, or implement stack-based argument passing for the 7th+ argument per the SysV ABI.

Helper Methods

Several small helpers keep emit_instr code clean:

signed() — Since stack slots are negative offsets, we need to format them for NASM syntax which uses + or -:

fn signed(&self, n: i64) -> String {
    if n >= 0 { format!("+{}", n) }
    else { format!("-{}", -n) }
}

This produces rbp-24 for negative slots (the common case) and rbp+16 for the return address area.

load_to_rax() and load_to_reg() — Load a virtual register's value into a physical register:

fn load_to_rax(&mut self, vreg: VReg, alloc: &StackAllocator) {
    let slot = alloc.slot(vreg);
    if slot != 0 {
        self.emit(&format!("  mov rax, [rbp{}]", self.signed(slot)));
    } else {
        self.emit("  xor eax, eax");
    }
}

fn load_to_reg(&mut self, vreg: VReg, reg: &str, alloc: &StackAllocator) {
    let slot = alloc.slot(vreg);
    if slot != 0 {
        self.emit(&format!("  mov {}, [rbp{}]", reg, self.signed(slot)));
    } else {
        self.emit(&format!("  xor {}, {}", reg, reg));
    }
}

If the slot() returns 0 (the vreg wasn't allocated a slot), we xor reg, reg to zero it rather than crashing. This can happen if an unused vreg snuck through.

store_from_rax() — The reverse: write rax to a vreg's stack slot:

fn store_from_rax(&mut self, vreg: VReg, alloc: &StackAllocator) {
    let slot = alloc.slot(vreg);
    if slot != 0 {
        self.emit(&format!("  mov [rbp{}], rax", self.signed(slot)));
    }
}

escape_string() — Converts a byte slice into a NASM-compatible string literal:

fn escape_string(&self, bytes: &[u8]) -> String {
    let mut s = String::new();
    for &b in bytes {
        match b {
            b'\\' => s.push_str("\\\\"),
            b'"' => s.push_str("\\\""),
            b'\n' => s.push_str("\\n"),
            b'\t' => s.push_str("\\t"),
            0 => s.push_str("\\0"),
            c if c.is_ascii_graphic() || c == b' ' => s.push(c as char),
            _ => write!(s, "\\x{:02x}", b).unwrap(),
        }
    }
    s
}

Note: non-printable, non-escape bytes are output as \xHH hex escapes, which NASM understands natively.

Common Pitfalls & Debugging Tips

  1. Stack offset sign errors: Stack slots are negative offsets from rbp (e.g., rbp-8). The signed helper must produce -8, not +8. A positive offset would corrupt the caller's stack frame.
  2. Missing qword specifier: NASM requires size specifiers for memory operands with immediates. mov [rbp-8], 42 is ambiguous; it must be mov qword [rbp-8], 42.
  3. Caller-saved registers: The codegen doesn't track register liveness — every value is loaded from and stored to memory. This is correct but slow. A real register allocator would keep values in registers across multiple operations.
  4. Stack alignment: The SysV ABI requires 16-byte stack alignment at the call instruction. The prologue's push rbp (8 bytes) plus sub rsp, N where N is a multiple of 8 maintains 16-byte alignment. The StackAllocator's stack_size & !15 alignment helps ensure this.
  5. Division by zero: x86-64 idiv raises a hardware exception on division by zero. L1S does not check for this; the program will crash with SIGFPE.

The Compiler Driver

With the assembly string produced by Codegen, we have human-readable x86-64 NASM source code. But a string is not an executable — we need to assemble it into machine code and link it into a runnable binary. This is the job of the compiler driver in main.rs.

The driver is a single main() function that orchestrates the entire compilation pipeline, from reading source files to producing executables. Let's trace the complete flow.

The Full Pipeline

Build pipeline from source to executable:

/home/user/project/
    │
    ├── main.is          // L1S source (you write this)
    │
    │   [ignis compiles main.is]
    │
    ├── main.asm         // NASM assembly (generated by Codegen)
    │
    │   [nasm -f elf64 main.asm]
    │
    ├── main.o           // ELF64 object file (generated by nasm)

    │

1. Output Path Resolution (lines 68–73)

The path is parsed by checking args[2] == "-o". If the flag is present, args[3] is used as the output name. Otherwise, the input filename's extension is stripped: program.is becomes program. src/main.rs — Complete compiler driver
fn main() {
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: ignis <file.is> [-o <output>]");
        std::process::exit(1);
    }

    // 1. Read source file
    let source = fs::read_to_string(&args[1]).unwrap_or_else(|e| { /* ... */ });

    // 2. Lex
    let mut lexer = lexer::Lexer::new(&source);
    let (tokens, _lex_errors) = lexer.tokenize();

    // 3. Parse
    let diag = diag::Diagnostics::new();
    let mut parser = parser::Parser::new(tokens, diag);
    let mut prog = parser.parse_program();
    let mut diag = parser.into_diagnostics();
    if diag.has_errors() { diag.print_all(); std::process::exit(1); }

    // 4. Type check
    let mut types = types::TypeArena::new();
    let mut symbols = symbol::SymbolTable::new();
    let mut tc = typeck::TypeChecker::new(&mut types, &mut symbols, &mut diag);
    tc.check_program(&mut prog);
    let annotations = tc.annotations.clone();
    let has_errors = tc.diag.has_errors();
    if has_errors { diag.print_all(); std::process::exit(1); }

    // 5. Build IR
    let builder = ir_build::IrBuilder::new(&types, &annotations, &mut diag);
    let mut ir_module = builder.build(&prog);

    // 6. Optimize IR
    ir_opt::optimize(&mut ir_module);

    // 7. Generate assembly
    let mut codegen = codegen::Codegen::new(&types);
    let asm = codegen.generate(&ir_module);

    // 8. Write .asm, assemble, link, cleanup (lines 68-109, shown above)
    // ...
}

Each stage is fallible. The driver either exits early on error (with a non-zero exit code) or proceeds to the next stage. The lexer errors are currently ignored (the _lex_errors variable is unused), but parser and type-checker errors abort compilation.

Common Pitfalls & Debugging Tips

  1. NASM not found: The error message is "error invoking assembler: No such file or directory". Install NASM via your package manager.

  5. Permission denied: If fs::write fails, check that the output directory is writable. The .asm file is written alongside the binary, so consider the directory permissions.

16   Error Diagnostics

The Need for Good Diagnostics

When a compiler encounters an error, its response determines the quality of the developer experience. A good diagnostic tells you what went wrong, where it went wrong, and why. A bad diagnostic prints "syntax error" with no location. L1S's diagnostics system is modeled after rustc's output, with source snippets, underline carets, and optional notes.

The Severity Enum

Every diagnostic has a severity level that determines how it's displayed and whether compilation should continue:

src/diag.rs — Severity enum
#[derive(Debug, Clone, PartialEq)]
pub enum Severity {
    Error,
    Warning,
    Note,
}

impl fmt::Display for Severity {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            Severity::Error => write!(f, "error"),
            Severity::Warning => write!(f, "warning"),
            Severity::Note => write!(f, "note"),
        }
    }
}

The three levels serve distinct purposes:

  • Error: Compilation cannot proceed. The binary is not produced. The driver exits with code 1.
  • Warning: Something suspicious but not fatal. The compiler may continue, but warnings should be investigated.
  • Note: Additional context attached to an error or warning. A note never appears alone; it always annotates another diagnostic.

The Diagnostic Struct (Builder Pattern)

Individual diagnostics are constructed using a builder pattern — each setter method consumes and returns Self, allowing chaining:

src/diag.rs — Diagnostic struct
#[derive(Debug, Clone)]
pub struct Diagnostic {
    pub severity: Severity,
    pub message: String,
    pub span: Option<Span>,
    pub notes: Vec<(String, Option<Span>)>,
}

impl Diagnostic {
    pub fn new(severity: Severity, message: impl Into<String>) -> Self {
        Diagnostic { severity, message: message.into(), span: None, notes: Vec::new() }
    }

    pub fn at(mut self, span: Span) -> Self {
        self.span = Some(span);
        self
    }

    pub fn with_note(mut self, msg: impl Into<String>, span: Option<Span>) -> Self {
        self.notes.push((msg.into(), span));
        self
    }
}

Usage looks like this:

let diag = Diagnostic::new(Severity::Error, "type mismatch")
    .at(some_span)
    .with_note("expected i32, found bool", None)
    .with_note("see definition here", Some(def_span));

        self.file_path = Some(path.into());
    }

    pub fn error(&mut self, msg: impl Into<String>, span: Option<Span>) {
        self.diagnostics.push(Diagnostic::new(Severity::Error, msg).at_maybe(span));
    }

    pub fn warning(&mut self, msg: impl Into<String>, span: Option<Span>) {
        self.diagnostics.push(Diagnostic::new(Severity::Warning, msg).at_maybe(span));
    }

    pub fn note(&mut self, msg: impl Into<String>, span: Option<Span>) {
        self.diagnostics.push(Diagnostic::new(Severity::Note, msg).at_maybe(span));
    }

    pub fn has_errors(&self) -> bool {
        self.diagnostics.iter().any(|d| d.severity == Severity::Error)
    }

    pub fn print_all(&self) {
        for diag in &self.diagnostics {
            self.print_diagnostic(diag);
        }
    }
}

The source and file_path fields are set once (typically by the parser or driver) and reused for all diagnostics. Having the source text available allows format_diagnostic to extract and display the offending line with an underline caret.

The at_maybe Helper

impl Diagnostic {
    fn at_maybe(self, span: Option<Span>) -> Self {
        if let Some(s) = span { self.at(s) }
        else { self }
    }
}

This is a private helper (no pub) — it is only used internally by the convenience methods. If a Some(span) is provided, the diagnostic gets a location; if None, it remains location-less.

The Formatter: format_diagnostic()

This is the heart of the diagnostics system. It produces rustc-style output with source location, source line, underline caret, and optional notes:

src/diag.rs — format_diagnostic()
pub fn format_diagnostic(&self, diag: &Diagnostic) -> String {
    let mut out = String::new();
    let severity = match diag.severity {
        Severity::Error => "error",
        Severity::Warning => "warning",
        Severity::Note => "note",
    };

    if let Some(span) = &diag.span {
        let loc = if let Some(path) = &self.file_path {
            format!("{}:{}:{}", path, span.line, span.col)
        } else {
            format!("{}:{}", span.line, span.col)
        };

        out.push_str(&format!("{}[E000]: {}
", severity, diag.message));
        out.push_str(&format!("  --> {}
", loc));

        if let Some(source) = &self.source {
            let lines: Vec<&str> = source.lines().collect();
            if let Some(line) = lines.get((span.line - 1) as usize) {
                out.push_str(&format!("   |
"));
                out.push_str(&format!("{:>4} | {}
", span.line, line));

                let col = span.col as usize;
                let width = (span.end - span.start).max(1);
                let underline = format!("{:>width$}", "^", width = col + width);
                let underline = &underline[..underline.len().min(col + width)];
                out.push_str(&format!("   | {}
", underline));
            }
        }
    } else {
        out.push_str(&format!("{}[E000]: {}
", severity, diag.message));
    }

    for (note, note_span) in &diag.notes {
        if let Some(ns) = note_span {
            out.push_str(&format!("
   = note: {} (at {}:{})
", note, ns.line, ns.col));
        } else {
            out.push_str(&format!("
   = note: {}
", note));
        }
    }

    out
}

How it works:

  1. Severity label: Each diagnostic starts with error[E000]:, warning[E000]:, or note[E000]:. The E000 is a placeholder for error codes, which L1S doesn't yet use but reserves for future categorization.
  2. Location: If the diagnostic has a span, a --> file:line:col line is printed, rustc-style. If no file path is available, just line:col.
  3. Source line: The offending line is extracted from the source text using source.lines(). Line numbers are 1-based, so we subtract 1 for the index.
  4. Underline caret: A ^ is placed under the span, repeating for multi-character spans (^^^). The width is span.end - span.start, with a minimum of 1.
  5. Notes: Each note is appended as = note: message, optionally with a (at line:col) location.
rustc-style output: The format deliberately mimics rustc's error messages. Compare: 
error[E000]: type mismatch
  --> test.gn:1:1
   |
 1 | func main() { return 42; }
   | ^^^^^
   = note: expected i32, found bool
This consistency helps developers who are familiar with Rust's compiler output.

Common Pitfalls & Debugging Tips

  1. Forgot to set source: If set_source is never called, format_diagnostic will produce a bare error[E000]: message with no source line or caret. Always call set_source early in the pipeline.
  2. Line number off by one: source.lines() returns 0-indexed lines, but Span.line is 1-indexed. The span.line - 1 conversion is critical.
  3. Zero-width spans: The .max(1) on the width calculation ensures that a zero-length span (e.g., Span::new(5, 5, ...)) still gets a single ^ caret rather than an empty underline.
  4. Notes without diagnostic context: Notes are only meaningful when attached to an error or warning. Using the note() convenience method standalone creates a top-level note, which will be printed but may confuse readers.
  5. Error code placeholders: All diagnostics currently use E000. In a larger compiler, each error would have a unique code (like E0001, E0002) for easier documentation and searching.

Pipeline Recap

The compilation pipeline flows through 9 distinct stages:

L1S Compilation Pipeline:

Source (.is)
  │
  ▼ 1. Lexer (src/lexer.rs)
  │   Characters → Tokens
  │   Error recovery: records bad chars, keeps going
  ▼ 2. Parser (src/parser.rs)
  │   Tokens → AST
  │   Recursive descent + Pratt for expressions
  ▼ 3. Type Checker (src/typeck.rs)
  │   AST → Typed AST
  │   Bidirectional inference, unification
  ▼ 4. IR Builder (src/ir_build.rs)
  │   Typed AST → IR (basic blocks, vregs)
  │   Lowers structured control flow to branches
  ▼ 5. IR Optimizer (src/ir_opt.rs)
  │   IR → Optimized IR
  │   Dead code elimination, constant folding
  ▼ 6. Register Allocator (src/regalloc.rs)
  │   Assigns stack slots to virtual registers
  │   Computes total stack frame size
  ▼ 7. Code Generator (src/codegen.rs)
  │   IR → NASM Assembly (.asm)
  │   x86-64, SysV ABI
  ▼ 8. Assembler (nasm -f elf64)
  │   .asm → ELF64 Object File (.o)
  ▼ 9. Linker (cc -no-pie)
      .o → Executable binary
      Links C runtime, produces runnable program

Each stage consumes the output of the previous stage and produces input for the next. The pipeline is linear — there are no back-edges for optimization feedback, though the IR optimizer does perform multiple passes over the IR in place.

What Makes L1S Special

  • Simplicity over performance: The compiler prioritizes clear, readable code over optimization. The register allocator is a naive stack-based approach, yet it produces working code.
  • Error resilience: Both the lexer and parser continue after errors, maximizing the number of issues reported per compilation.
  • rustc-style diagnostics: The error messages include source snippets with underline carets and notes, providing a developer-friendly experience.
  • Real x86-64 code: This is not a toy VM or bytecode interpreter — L1S generates real assembly that runs directly on the CPU.
  • Self-contained: The entire compiler is under 2,500 lines of Rust, with zero dependencies beyond the standard library and external tools (NASM, cc).

Future Directions

L1S is a teaching compiler, but there are many directions you could take it:

1. Register Allocation Improvements

The current StackAllocator spills every virtual register to the stack. A graph-coloring register allocator (like Chaitin's algorithm) would keep values in physical registers, dramatically reducing memory traffic. The IR already has the framework for liveness analysis via VReg and basic blocks.

2. Optimization Passes

The ir_opt module is minimal. You could add:

  • Constant propagation: Replace MovImm + arithmetic with folded results.
  • Common subexpression elimination: Reuse computation results.
  • Loop invariant code motion: Hoist computations out of loops.
  • Tail call optimization: Convert return func() to a jump.

3. More Type Inference

The current type checker uses bidirectional inference but requires explicit type annotations in many places. Adding Hindley-Milner type inference (or even full unification with let-polymorphism) would make the language much more expressive.

4. Standard Library

L1S currently has no standard library. Building one with I/O, memory allocation, string manipulation, and data structures would make it practical for real programs. Since L1S can call external C functions via extern, you can wrap libc functions.

5. Self-Hosting

The ultimate milestone: write an L1S compiler in L1S itself. This requires adding enough features (strings, file I/O, arrays, structs with methods) to make the compiler source code translatable. Self-hosting is the definitive proof that a language is "real."

6. Other Targets

The code generator is the only architecture-specific component. With a different backend, L1S could target ARM64 (Apple Silicon, Raspberry Pi), RISC-V, or WebAssembly. The IR is largely architecture-neutral.

7. Incremental Compilation

For large projects, recompiling everything on every change is wasteful. A module system with separate compilation and caching would make iteration faster.

Building a compiler is one of the most rewarding projects in software engineering. It touches every part of computer science: formal languages, data structures, algorithms, architecture, and systems programming. L1S gives you a complete, working compiler that you can study, modify, and extend.

The code is deliberately kept small and readable. Every module is independent enough to be understood in isolation, yet interconnected enough to form a real pipeline.
