Building a Compiler Rust Edition

Chapter 02: Lexing

Raw text is just characters. How do we find the meaningful pieces?

What you'll learn in this chapter: How to write a lexer (tokenizer) from scratch in Rust — scanning characters, grouping them into tokens, handling keywords and operators, and producing a stream the parser can consume.

The Problem

Open a source file and you see structure: keywords like func and return, numbers like 42, operators like +, punctuation like braces and semicolons. But to the computer, it's just a flat sequence of bytes. The lexer's job is structural: group those bytes into meaningful chunks called tokens.

Consider this program fragment:

func main(i32) { return 42; }

A human sees five pieces: a keyword, a name, parentheses with a type, braces, and a return statement. The lexer needs to produce exactly that:

Input:  func main(i32) { return 42; }
Tokens:
  [0] Keyword(Func)   "func"
  [1] Ident("main")   "main"
  [2] OpenParen       "("
  [3] Ident("i32")    "i32"
  [4] CloseParen      ")"
  [5] OpenBrace       "{"
  [6] Keyword(Return) "return"
  [7] IntLit(42)      "42"
  [8] Semicolon       ";"
  [9] CloseBrace      "}"

Each token has two parts: a kind (what it is) and a span (where it came from in the source). The parser will consume these tokens to build the AST, but the parser never needs to think about individual characters again — the lexer abstracts that away.

The Token Enum

In Rust, every kind of token is a variant of an enum. This is the core data structure of the lexer:

src/token.rs
use crate::span::Span;

#[derive(Debug, Clone, PartialEq)]
pub struct Token {
    pub kind: TokenKind,
    pub span: Span,
}

#[derive(Debug, Clone, PartialEq)]
pub enum TokenKind {
    // Keywords
    Keyword(Keyword),

    // Literals
    Ident(String),
    IntLit(i64),

    // Single-character operators / punctuation
    Plus,       // +
    Minus,      // -
    Star,       // *
    Slash,      // /
    Percent,    // %
    Equals,     // =
    Bang,       // !
    Lt,         // <
    Gt,         // >
    Ampersand,  // &
    Pipe,       // |
    OpenParen,  // (
    CloseParen, // )
    OpenBrace,  // {
    CloseBrace, // }
    Comma,      // ,
    Semicolon,  // ;
    Colon,      // :
    Arrow,      // ->

    // Multi-character operators
    EqEq,       // ==
    BangEq,     // !=
    LtEq,       // <=
    GtEq,       // >=
    AndAnd,     // &&
    OrOr,       // ||
}

#[derive(Debug, Clone, PartialEq)]
pub enum Keyword {
    Func,
    Let,
    If,
    Else,
    While,
    Return,
    True,
    False,
    Cast,
}
token.ml
type token = {
  kind: token_kind;
  span: Span.span;
}

type token_kind =
  | Keyword of keyword
  | Ident of string
  | IntLit of int64
  | Plus | Minus | Star | Slash
  | Percent | Equals | Bang
  | Lt | Gt | Ampersand | Pipe
  | OpenParen | CloseParen
  | OpenBrace | CloseBrace
  | Comma | Semicolon | Colon | Arrow
  | EqEq | BangEq | LtEq | GtEq

type keyword =
  | Kw_Func | Kw_Let | Kw_If
  | Kw_Else | Kw_While | Kw_Return
  | Kw_True | Kw_False | Kw_Cast
Design note: Notice that Ident and IntLit carry data (String and i64), while operators like Plus and Semicolon carry nothing — they are self-evident. This is a classic Rust enum pattern: variants with payloads for things that need data, unit variants for things that don't. The compiler will warn you if you try to match non-exhaustively.

Keywords vs Identifiers

Our language has reserved keywords like func, return, if, and let. Unreserved words like main, add, or counter are identifiers. The lexer handles both with the same initial logic — it scans a word starting with a letter — and then checks whether the word is a keyword:

src/keyword.rs
use crate::token::Keyword;

pub fn keyword_token(word: &str) -> Option<Keyword> {
    match word {
        "func"   => Some(Keyword::Func),
        "let"    => Some(Keyword::Let),
        "if"     => Some(Keyword::If),
        "else"   => Some(Keyword::Else),
        "while"  => Some(Keyword::While),
        "return" => Some(Keyword::Return),
        "true"   => Some(Keyword::True),
        "false"  => Some(Keyword::False),
        "cast"   => Some(Keyword::Cast),
        _         => None,
    }
}
keyword.ml
let keyword_token word =
  match word with
  | "func"   -> Some Kw_Func
  | "let"    -> Some Kw_Let
  | "if"     -> Some Kw_If
  | "else"   -> Some Kw_Else
  | "while"  -> Some Kw_While
  | "return" -> Some Kw_Return
  | "true"   -> Some Kw_True
  | "false"  -> Some Kw_False
  | "cast"   -> Some Kw_Cast
  | _         -> None

This function is pure: a String goes in, an Option<Keyword> comes out. No state, no side effects. The Option return type models the two cases perfectly: Some(kw) for recognized keywords, None for identifiers and everything else. The caller decides what to do.

The Lexer Struct

The lexer holds the source text and a cursor position. It exposes a next_token method that advances through the text and returns one token at a time:

src/lexer.rs
use crate::keyword::keyword_token;
use crate::span::Span;
use crate::token::{Keyword, Token, TokenKind};

pub struct Lexer {
    chars: Vec<char>,
    pos: usize,
}

impl Lexer {
    pub fn new(src: &str) -> Self {
        Self {
            chars: src.chars().collect(),
            pos: 0,
        }
    }

    fn peek(&self) -> Option<char> {
        self.chars.get(self.pos).copied()
    }

    fn advance(&mut self) -> Option<char> {
        let ch = self.peek()?;
        self.pos += 1;
        Some(ch)
    }

    fn is_at_end(&self) -> bool {
        self.pos >= self.chars.len()
    }
}
lexer.ml
type lexer = {
  chars: char array;
  mutable pos: int;
}

let make_lexer src =
  { chars = Array.of_list (String.to_seq src |> List.of_seq);
    pos = 0 }

let peek lex =
  if lex.pos < Array.length lex.chars
  then Some lex.chars.(lex.pos)
  else None

let advance lex =
  let ch = peek lex in
  lex.pos <- lex.pos + 1;
  ch

let is_at_end lex =
  lex.pos >= Array.length lex.chars
Why Vec<char> instead of indexing into &str? Rust strings are UTF-8 encoded. Indexing by byte source[0..5] might slice in the middle of a multi-byte character. Using .chars().collect::<Vec<char>>() gives us O(1) access to each Unicode code point. For ASCII-only source code (which our language is), we could use bytes directly, but using char is more robust and the cost is negligible for compilation-scale inputs.

Lexing Methods

The main entry point is next_token, which dispatches to specialized methods based on the current character:

src/lexer.rs (continued)
impl Lexer {
    pub fn next_token(&mut self) -> Option<Token> {
        self.skip_whitespace();
        self.skip_comment();

        if self.is_at_end() {
            return None;
        }

        let start = self.pos;
        let ch = self.peek().unwrap();

        let kind = if ch.is_ascii_alphabetic() || ch == '_' {
            self.lex_word()
        } else if ch.is_ascii_digit() {
            self.lex_number()
        } else {
            self.lex_operator()
        };

        let span = Span::new(start, self.pos);
        Some(Token { kind, span })
    }
}
lexer.ml (continued)
let next_token lex =
  skip_whitespace lex;
  skip_comment lex;

  if is_at_end lex then None
  else
    let start = lex.pos in
    let ch = match peek lex with Some c -> c | None -> failwith "unreachable" in
    let kind =
      if Char.alpha ch || ch = '_'
      then lex_word lex
      else if Char.digit ch
      then lex_number lex
      else lex_operator lex
    in
    Some { kind; span = Span.make_span start lex.pos }

skip_whitespace

Whitespace separates tokens but carries no semantic meaning. We skip it entirely:

src/lexer.rs
fn skip_whitespace(&mut self) {
    while let Some(ch) = self.peek() {
        if ch.is_ascii_whitespace() {
            self.advance();
        } else {
            break;
        }
    }
}
lexer.ml
let skip_whitespace lex =
  let rec loop () =
    match peek lex with
    | Some ch when Char.white ch -> advance lex; loop ()
    | _ -> ()
  in
  loop ()

skip_comment

Comments start with // and extend to the end of the line. We also call this after skip_whitespace so that whitespace-before-comment is consumed first:

src/lexer.rs
fn skip_comment(&mut self) {
    if self.peek() == Some('/') && self.peek_next() == Some('/') {
        while let Some(ch) = self.advance() {
            if ch == '\n' {
                break;
            }
        }
    }
}
lexer.ml
let skip_comment lex =
  match peek lex, peek_next lex with
  | Some '/', Some '/' ->
      let rec loop () =
        match advance lex with
        | Some '\n' | None -> ()
        | Some _ -> loop ()
      in
      loop ()
  | _ -> ()
Note on peek_next: We need a peek_next method to look at the character after the current one without advancing. This is essential for detecting two-character sequences like //, ==, !=, etc.: fn peek_next(&self) -> Option<char> { self.chars.get(self.pos + 1).copied() }.

lex_word

Words start with a letter or underscore and contain letters, digits, and underscores. After scanning the whole word, we check if it's a keyword:

src/lexer.rs
fn lex_word(&mut self) -> TokenKind {
    let start = self.pos;
    while let Some(ch) = self.peek() {
        if ch.is_ascii_alphanumeric() || ch == '_' {
            self.advance();
        } else {
            break;
        }
    }
    let word: String = self.chars[start..self.pos]
        .iter()
        .collect();
    match keyword_token(&word) {
        Some(kw) => TokenKind::Keyword(kw),
        None => TokenKind::Ident(word),
    }
}
lexer.ml
let lex_word lex =
  let start = lex.pos in
  let rec loop () =
    match peek lex with
    | Some ch when Char.alpha ch || Char.digit ch || ch = '_' ->
        advance lex; loop ()
    | _ -> ()
  in
  loop ();
  let word = String.of_seq (Array.to_seq lex.chars |> Seq.drop start |> Seq.take (lex.pos - start)) in
  match keyword_token word with
  | Some kw -> Keyword kw
  | None -> Ident word

lex_number

Numbers are sequences of digits, optionally preceded by a - for negative literals. We parse them into i64:

src/lexer.rs
fn lex_number(&mut self) -> TokenKind {
    let start = self.pos;
    while let Some(ch) = self.peek() {
        if ch.is_ascii_digit() || ch == '_' {
            self.advance();
        } else {
            break;
        }
    }
    let s: String = self.chars[start..self.pos]
        .iter()
        .filter(|&c| c != '_')
        .collect();
    let n: i64 = s.parse().expect("invalid integer literal");
    TokenKind::IntLit(n)
}
lexer.ml
let lex_number lex =
  let start = lex.pos in
  let rec loop () =
    match peek lex with
    | Some ch when Char.digit ch || ch = '_' ->
        advance lex; loop ()
    | _ -> ()
  in
  loop ();
  let raw = String.of_seq (Array.to_seq lex.chars |> Seq.drop start |> Seq.take (lex.pos - start)) in
  let s = String.filter (fun c -> c <> '_') raw in
  IntLit (Int64.of_string s)

We support _ as a digit separator (like Rust's 1_000_000). The filter call strips them before parsing. This is purely a lexer concern — the parser never sees the underscores.

lex_operator

The operator lexer needs to distinguish single-character operators from multi-character sequences. This is where we handle the "maximal munch" principle: always match the longest possible token:

src/lexer.rs
fn lex_operator(&mut self) -> TokenKind {
    let ch = self.advance().expect("lex_operator called at end");
    match ch {
        '+' => TokenKind::Plus,
        '-' => {
            if self.peek() == Some('>') {
                self.advance();
                TokenKind::Arrow
            } else {
                TokenKind::Minus
            }
        }
        '*' => TokenKind::Star,
        '/' => { // Note: comments handled before lex_operator
            TokenKind::Slash
        }
        '%' => TokenKind::Percent,
        '=' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::EqEq
            } else {
                TokenKind::Equals
            }
        }
        '!' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::BangEq
            } else {
                TokenKind::Bang
            }
        }
        '<' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::LtEq
            } else {
                TokenKind::Lt
            }
        }
        '>' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::GtEq
            } else {
                TokenKind::Gt
            }
        }
        '&' => {
            if self.peek() == Some('&') {
                self.advance();
                TokenKind::AndAnd
            } else {
                TokenKind::Ampersand
            }
        }
        '|' => {
            if self.peek() == Some('|') {
                self.advance();
                TokenKind::OrOr
            } else {
                TokenKind::Pipe
            }
        }
        '(' => TokenKind::OpenParen,
        ')' => TokenKind::CloseParen,
        '{' => TokenKind::OpenBrace,
        '}' => TokenKind::CloseBrace,
        ',' => TokenKind::Comma,
        ';' => TokenKind::Semicolon,
        ':' => TokenKind::Colon,
        _ => panic!("unexpected character: {:?}", ch),
    }
}
lexer.ml
let lex_operator lex =

  match ch with
  | '+' -> Plus
  | '-' -> begin
      match peek lex with
      | Some '>' -> advance lex; Arrow
      | _ -> Minus
    end
  | '=' -> begin
      match peek lex with
      | Some '=' -> advance lex; EqEq
      | _ -> Equals
    end
  | '!' -> begin
      match peek lex with
      | Some '=' -> advance lex; BangEq
      | _ -> Bang
    end
  | '<' -> begin
      match peek lex with
      | Some '=' -> advance lex; LtEq
      | _ -> Lt
    end
  | '>' -> begin
      match peek lex with
      | Some '=' -> advance lex; GtEq
      | _ -> Gt
    end
  | '&' -> begin
      match peek lex with
      | Some '&' -> advance lex; AndAnd
      | _ -> Ampersand
    end
  | '|' -> begin
      match peek lex with
      | Some '|' -> advance lex; OrOr
      | _ -> Pipe
    end
  | '(' -> OpenParen
  | ')' -> CloseParen
  | '{' -> OpenBrace
  | '}' -> CloseBrace
  | ',' -> Comma
  | ';' -> Semicolon
  | ':' -> Colon
  | _ -> failwith (Printf.sprintf "unexpected character: %c" ch)
Maximal munch: The lexer always matches the longest possible token. When it sees ==, it produces one EqEq token rather than two Equals tokens. This is the standard rule across virtually all programming languages. Without it, a == b would parse as a = = b, which is nonsense.

Error Handling: Invalid Characters

Our current implementation panic!s on unexpected characters. In a production compiler, this is unacceptable — a single stray @ in a 10,000-line file should not crash the compiler. Instead, we should emit an error and continue:

src/lexer.rs — error recovery
/// Result type for lexer operations.
pub type LexResult<T> = Result<T, LexError>;

#[derive(Debug, Clone)]
pub struct LexError {
    pub message: String,
    pub span: Span,
}

impl Lexer {
    pub fn next_token(&mut self) -> Option<Result<Token, LexError>> {
        self.skip_whitespace();
        self.skip_comment();

        if self.is_at_end() {
            return None;
        }

        let start = self.pos;
        let ch = self.peek().unwrap();

        if ch.is_ascii_alphabetic() || ch == '_' {
            Some(Ok(self.make_token(start, self.lex_word())))
        } else if ch.is_ascii_digit() {
            Some(Ok(self.make_token(start, self.lex_number())))
        } else {
            match self.try_lex_operator() {
                Ok(kind) => Some(Ok(self.make_token(start, kind))),
                Err(msg) => {
                    self.advance(); // skip the invalid character
                    Some(Err(LexError {
                        message: msg,
                        span: Span::new(start, self.pos),
                    }))
                }
            }
        }
    }
}
lexer.ml
type lex_error = { message: string; span: Span.span }

type lex_result = (token, lex_error) result

let next_token lex =
  skip_whitespace lex;
  skip_comment lex;

  if is_at_end lex then None
  else
    let start = lex.pos in
    let ch = match peek lex with Some c -> c | None -> failwith "unreachable" in
    let result =
      if Char.alpha ch || ch = '_'
      then Ok (lex_word lex)
      else if Char.digit ch
      then Ok (lex_number lex)
      else try_lex_operator lex
    in
    match result with
    | Ok kind ->
        Some (Ok { kind; span = Span.make_span start lex.pos })
    | Error msg ->
        advance lex;
        Some (Error { message = msg; span = Span.make_span start lex.pos })

With this approach, the lexer never panics. Invalid characters produce an Err token that the parser can collect and report. The compiler keeps going, finding more errors in the same pass.

Testing the lexer: The best way to verify your lexer is to write tests that print the token stream for a given input. Create a fn lex_all(src: &str) -> Vec<Token> helper and compare the output against expected token lists in your tests. This catches subtle bugs like off-by-one spans or misclassified keywords.

What's Next

The lexer transforms source text into a flat token stream. But programs aren't flat — they're trees. Functions contain statements, statements contain expressions, expressions contain sub-expressions. The next chapter tackles the parser: the phase that takes our flat token stream and reconstructs the tree structure that the grammar describes.
