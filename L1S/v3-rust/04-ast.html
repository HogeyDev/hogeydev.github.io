Building a Compiler Rust Edition

Chapter 04: The AST

The parser produces structure. What does that structure look like?

What you'll learn in this chapter: The complete abstract syntax tree definition for our language, how Rust enums model tree nodes, why we discard certain syntactic details, and how to visualize and debug ASTs.

The Problem

The parser consumes tokens and produces a tree. But what kind of tree? Every programming language has two potential tree representations:

Concrete Syntax Tree (CST) Abstract Syntax Tree (AST)
Contains Every token: keywords, punctuation, parentheses, semicolons Only semantic nodes: expressions, statements, declarations
Preserves Syntactic details: which kind of parenthesis, exact operator spelling Only the structure and meaning
Used for Formatting, refactoring tools, syntax highlighting Semantic analysis, code generation, optimization

Our compiler uses an AST — we discard details that don't affect meaning. The semicolons, parentheses, and braces are structural hints during parsing, but they don't appear in the AST. What remains is the pure semantics of the program.

Source:         func add(i32, a: i32, b: i32) { return a + b; }

CST (contains everything):
  Keyword(Func) Ident("add") OpenParen Ident("i32") Comma ...

AST (only the meaning):
  FuncDecl("add", return=i32, params=[("a",i32),("b",i32)],
    body=[Return(Binary(+, Ident("a"), Ident("b")))])

The AST in Rust: Enums All the Way Down

In Rust, an AST is a collection of enums. Each enum represents a category of node, and each variant represents a specific kind of node. Data that belongs to a node (like a name or a child node) is carried by the variant's fields.

src/ast.rs
use crate::span::Span;
use crate::token::{BinaryOp, UnaryOp};

#[derive(Debug, Clone, PartialEq)]
pub struct AstProgram {
    pub funcs: Vec<AstFuncDecl>,
}

#[derive(Debug, Clone, PartialEq)]
pub struct AstFuncDecl {
    pub name: String,
    pub ret_type: AstType,
    pub params: Vec<AstParam>,
    pub body: Vec<AstStmt>,
}

#[derive(Debug, Clone, PartialEq)]
pub struct AstParam {
    pub name: String,
    pub ty: AstType,
}

#[derive(Debug, Clone, PartialEq)]
pub enum AstType {
    Ident(String),          // i32, bool, void, user-defined
    Ptr(Box<AstType>),     // @T — pointer type
    Array(Box<AstType>, usize), // [N]T
}

#[derive(Debug, Clone, PartialEq)]
pub enum AstStmt {
    Return(AstExpr),
    VarDecl {
        name: String,
        ty: AstType,
        init: AstExpr,
    },
    If(Box<AstExpr>, Box<AstStmt>, Option<Box<AstStmt>>),
    While(Box<AstExpr>, Box<AstStmt>),
    Block(Vec<AstStmt>),
    Expr(AstExpr),
    Error,  // placeholder for parse errors — keeps the compiler going
}

#[derive(Debug, Clone, PartialEq)]
pub enum AstExpr {
    Int(i64),
    Bool(bool),
    Ident(String),
    Unary {
        op: UnaryOp,
        expr: Box<AstExpr>,
    },
    Binary {
        op: BinaryOp,
        left: Box<AstExpr>,
        right: Box<AstExpr>,
    },
    Call {
        callee: Box<AstExpr>,
        args: Vec<AstExpr>,
    },
    Cast {
        target_type: AstType,
        expr: Box<AstExpr>,
    },
    Error,  // placeholder for parse errors
}

  | AstType_Ptr of ast_type
  | AstType_Array of ast_type * int

  | AstStmt_Return of ast_expr
  | AstStmt_VarDecl of string * ast_type * ast_expr
  | AstStmt_If of ast_expr * ast_stmt * ast_stmt option
  | AstStmt_While of ast_expr * ast_stmt
  | AstStmt_Block of ast_stmt list
  | AstStmt_Expr of ast_expr
  | AstStmt_Error

type ast_expr =
  | AstExpr_Int of int64
  | AstExpr_Bool of bool
  | AstExpr_Ident of string
  | AstExpr_Unary of unary_op * ast_expr
  | AstExpr_Binary of ast_expr * binary_op * ast_expr
  | AstExpr_Call of ast_expr * ast_expr list
  | AstExpr_Cast of ast_type * ast_expr
  | AstExpr_Error
Reading the OCaml: If you know Rust enums, you know OCaml variants. The main difference is naming: Rust uses AstExpr::Int(i64) while OCaml uses AstExpr_Int of int64. Rust carries fields with { } naming, OCaml uses positional arguments. But the structure is identical — same variants, same nesting patterns, same recursive references through Box (Rust) or implicit boxing (OCaml).

Why This Design?

Every AST design choice has a rationale. Let's walk through the key decisions:

AstType is an Enum

Types can be simple names (i32, bool), pointers (@T), or arrays ([N]T). By making AstType an enum, we can add more type constructors later (tuples, generics, function types) without changing any code that pattern-matches on types. The compiler will tell us everywhere we need to update.

Box for Recursive Types

Rust needs to know the size of every type at compile time. An enum is as large as its largest variant. If AstExpr contained an AstExpr directly (without boxing), the size would be infinite. Box<AstExpr> stores the child on the heap and keeps the enum size fixed at a pointer width. This is the standard Rust pattern for recursive data structures — the same reason List uses Box in Rust but not in OCaml (OCaml uses uniform representation and implicit boxing).

AstStmt::Error and AstExpr::Error

These sentinel variants are our error recovery mechanism. When the parser encounters an invalid construct, it produces an Error node instead of panicking. Later phases handle Error by propagating it — type checking an error expression yields an error type, code generation for an error statement emits nothing. This keeps the pipeline running even with invalid input.

Choice of Children

Note that AstStmt::Return contains an AstExpr (the value to return), while AstStmt::If contains Box<AstExpr> and Box<AstStmt>. The Box is used when the variant would otherwise contain a recursive type directly (making the enum too large). Return doesn't need Box because Vec<AstStmt> and String in sibling variants are already large enough that the single AstExpr doesn't make the enum bigger.

Enum sizing trick: You can check enum sizes with std::mem::size_of::<AstExpr>(). In practice, Rust's enum layout optimization means that variants with a single non-nullable pointer field (like Box) may be smaller than you expect. But the rule is simple: wrap recursive variants in Box to be safe and clear.

The Display Trait for AST Debugging

During development, you'll often want to print an AST to see what the parser produced. Implementing Display for debugging is invaluable:

src/ast.rs — Display impl
use std::fmt;

impl fmt::Display for AstProgram {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        for func in &self.funcs {
            write!(f, "{}\n\n", func)?;
        }
        Ok(())
    }
}

impl fmt::Display for AstFuncDecl {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "func {}({}", self.name, self.ret_type)?;
        for param in &self.params {
            write!(f, ", {}", param)?;
        }
        write!(f, ") {{\n")?;
        for stmt in &self.body {
            write!(f, "  {};\n", stmt)?;
        }
        write!(f, "}}")
    }
}

impl fmt::Display for AstStmt {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AstStmt::Return(expr) => write!(f, "return {}", expr),
            AstStmt::VarDecl { name, ty, init } => {
                write!(f, "let {}: {} = {}", name, ty, init)
            }
            AstStmt::If(cond, then_branch, else_branch) => {
                write!(f, "if ({}) {{\n{}\n  }}", cond, then_branch)?;
                if let Some(else_) = else_branch {
                    write!(f, " else {{\n{}\n  }}", else_)?;
                }
                Ok(())
            }
            AstStmt::While(cond, body) => {
                write!(f, "while ({}) {{\n{}\n  }}", cond, body)
            }
            AstStmt::Block(stmts) => {
                write!(f, "{{\n")?;
                for s in stmts {
                    write!(f, "  {};\n", s)?;
                }
                write!(f, "}}")
            }
            AstStmt::Expr(expr) => write!(f, "{}", expr),
            AstStmt::Error => write!(f, "<error>"),
        }
    }
}

impl fmt::Display for AstExpr {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AstExpr::Int(n) => write!(f, "{}", n),
            AstExpr::Bool(b) => write!(f, "{}", b),
            AstExpr::Ident(name) => write!(f, "{}", name),
            AstExpr::Unary { op, expr } => write!(f, "({}{})", op, expr),
            AstExpr::Binary { op, left, right } => {
                write!(f, "({} {} {})", left, op, right)
            }
            AstExpr::Call { callee, args } => {
                write!(f, "{}({})", callee,
                    args.iter()
                        .map(|a| format!("{}", a))
                        .collect::<Vec<_>>()
                        .join(", "))?;
                Ok(())
            }
            AstExpr::Cast { target_type, expr } => {
                write!(f, "cast({}, {})", target_type, expr)
            }
            AstExpr::Error => write!(f, "<error>"),
        }
    }
}
ast.ml — show functions
let rec show_expr = function
  | AstExpr_Int n -> string_of_int n
  | AstExpr_Bool b -> string_of_bool b
  | AstExpr_Ident name -> name
  | AstExpr_Unary (op, e) ->
      Printf.sprintf "(%s%s)" (show_unary_op op) (show_expr e)
  | AstExpr_Binary (l, op, r) ->
      Printf.sprintf "(%s %s %s)" (show_expr l) (show_binary_op op) (show_expr r)
  | AstExpr_Call (callee, args) ->
      let args_str = String.concat ", " (List.map show_expr args) in
      Printf.sprintf "%s(%s)" (show_expr callee) args_str
  | AstExpr_Cast (ty, e) ->
      Printf.sprintf "cast(%s, %s)" (show_type ty) (show_expr e)
  | AstExpr_Error -> "<error>"

For a more structural view during debugging, we implement a tree renderer:

src/ast.rs — tree debug
impl AstProgram {
    pub fn debug_tree(&self) {
        println!("Program");
        for func in &self.funcs {
            func.debug_tree("  └── ");
        }
    }
}

impl AstExpr {
    pub fn debug_tree(&self, prefix: &str) {
        match self {
            AstExpr::Int(n) => println!("{}Int({})", prefix, n),
            AstExpr::Bool(b) => println!("{}Bool({})", prefix, b),
            AstExpr::Ident(name) => println!("{}Ident(\"{}\")", prefix, name),
            AstExpr::Binary { op, left, right } => {
                println!("{}Binary({})", prefix, op);
                left.debug_tree(&format!("{}  ├── ", prefix));
                right.debug_tree(&format!("{}  └── ", prefix));
            }
            // ... other variants similarly
        }
    }
}
ast.ml — tree debug
let rec debug_expr prefix = function
  | AstExpr_Int n -> Printf.printf "%sInt(%Ld)\n" prefix n
  | AstExpr_Bool b -> Printf.printf "%sBool(%b)\n" prefix b
  | AstExpr_Ident name -> Printf.printf "%sIdent(%S)\n" prefix name
  | AstExpr_Binary (l, op, r) ->
      Printf.printf "%sBinary(%s)\n" prefix (show_binary_op op);
      debug_expr (prefix ^ "  ├── ") l;
      debug_expr (prefix ^ "  └── ") r

Which produces:

Program
  └── FuncDecl("add")
        ├── ReturnType: Ident("i32")
        ├── Params:
        │   ├── Param("a", Ident("i32"))

Debug-driven development: When the parser produces unexpected output, the first thing to do is print the debug tree. Is the tree missing a node? Does a Binary have its left and right swapped? The tree view makes structural issues immediately visible. We keep debug_tree around for the entire compiler development process — it's invaluable for diagnosing phase bugs.

Complete Walkthrough

Let's trace the full pipeline from source to AST for our running example:

func add(i32, a: i32, b: i32) {
  return a + b;
}

Step 1: Lexing produces 21 tokens (we traced this in Chapter 2).

Step 2: Parsing organizes tokens into tree nodes.

Step 3: The AST — here's the complete structure in memory:

AstProgram {
  funcs: [
    AstFuncDecl {
      name: "add",
      ret_type: AstType::Ident("i32"),
      params: [
        AstParam { name: "a", ty: AstType::Ident("i32") },
        AstParam { name: "b", ty: AstType::Ident("i32") },
      ],
      body: [
        AstStmt::Return(
          AstExpr::Binary {
            op: Plus,
            left: Box(AstExpr::Ident("a")),
            right: Box(AstExpr::Ident("b")),
          }
        ),
      ],
    },
  ],
}

Let's walk through the memory layout. The AstProgram owns a Vec of AstFuncDecl. The Vec is on the stack (pointer, length, capacity), and the function declarations are on the heap. Inside AstFuncDecl, the body field is another Vec of AstStmt. The Return variant holds an AstExpr inline (not boxed, because it fits in the enum size). Inside Binary, the left and right are Box pointers to heap-allocated sub-expressions.

Stack (Parser::parse_program returns):
  AstProgram { funcs: (ptr, len, cap) }
                              │
Heap:                         ▼
                    ┌─────────────────────┐
                    │ AstFuncDecl          │
                    │   name: "add"        │
                    │   ret_type: Ident    │
                    │   params: Vec        │
                    │   body: Vec          │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │ Vec<AstStmt>         │
                    │   [0]: Return(expr)  │
                    └─────────────────────┘
                              │
                    ┌─────────▼───────────┐
                    │ AstExpr::Binary      │
                    │   op: Plus           │
                    │   left: Box ─────┐   │
                    │   right: Box ──┐ │   │
                    └─────────┬───────┼─┼───┘
                              │       │ │
                    ┌─────────▼───┐ ┌─▼─▼─────┐
                    │ Ident("a")  │ │ Ident("b")│
                    └─────────────┘ └──────────┘

This is the heart of the compiler. Everything from here on — semantic analysis, lowering, code generation — operates on this tree. The tree is the bridge between the programmer's intentions (expressed as source text) and the machine's instructions (expressed as assembly or bytecode).

The AST and the Borrow Checker

One of the most elegant properties of the AST in Rust is that it enforces tree shape through ownership. Every node has exactly one parent. The Box pointer ensures exclusive ownership. There are no cycles, no shared references, no dangling pointers.

This is exactly what the borrow checker verifies. When the semantic analyzer walks the AST, it can traverse with references:

fn analyze_expr(&self, expr: &AstExpr) -> Type {
    match expr {
        AstExpr::Ident(name) => self.lookup_type(name),
        AstExpr::Binary { left, right, .. } => {
            let lt = self.analyze_expr(left);
            let rt = self.analyze_expr(right);
            // ... unify lt and rt
        }
    }
}

No mutex, no GC, no reference counting. Just immutable references into a tree with guaranteed lifetime. If the tree is alive, the references are valid. The borrow checker proves it.

When you lower the AST to an IR, you consume the tree by value, moving nodes out:

fn lower_func(self, func: &mut AstFuncDecl) -> IrFunc {
    let body = std::mem::take(&mut func.body);
    // body is now empty Vec, we own the original statements
}

The pipeline is linear: lex → parse → analyze → lower → codegen. Data moves forward. Old representations are dropped and freed. There's no tracing through old token streams or half-analyzed ASTs. Rust's ownership model makes the pipeline's structure explicit in the type system.
