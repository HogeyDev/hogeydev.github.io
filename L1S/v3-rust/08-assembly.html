Building a Compiler Rust Edition

8. The Final Mile

We have assembly text — a string of x86-64 instructions. How do we get from that text to a running executable?

This chapter is about the driver: the code that ties all the pieces together, invokes the assembler and linker, handles errors, manages file I/O, and presents everything to the user. It is the least glamorous part of the compiler, but it is the part the user interacts with most.

The Full Pipeline

Here is every step, from source file to executable:

Source file (.l1s) │ ▼ read_file() — read UTF-8 text from disk Source text │ ▼ lexer::tokenize() — character-by-character → tokens Vec<Token> │ ▼ parser::parse() — recursive descent → AST Program (AST) │ ▼ symbol::build() — collect declarations → symbol table SymbolTable │ ▼ check::typeck() — type check every expression Vec<Diag> (0 errors → OK) │ ▼ ir_build::build() — AST → SSA IR (Braun et al.) Function (IR) │ ▼ iropt::optimize() — constant fold, DCE, algebraic Function (optimized IR) │ ▼ regalloc::assign() — linear scan → register mapping HashMap<VReg, Alloc> │ ▼ codegen::generate() — IR → x86-64 assembly text Assembly text (.asm) │ ▼ write_file() — write .asm to disk output.asm │ ▼ nasm -f elf64 — assemble → object file output.o │ ▼ cc -o output output.o -no-pie — link → executable output (executable)

This is the complete compiler pipeline. Every step is a function call. The interfaces are clean: the output of each step is the input of the next. If any step fails, the pipeline stops (or collects errors and continues, in the case of the type checker).

Error Handling

Our error handling philosophy is the same as Rust's: collect all errors, present them together, and let the user fix them in one batch. Like cargo, we never stop at the first error. We keep going to find as many as possible.

Here is the driver code:

src/main.rs
use std::process;

fn main() {
    let args: Vec<String> = std::env::args().collect();

    if args.len() < 3 || args[1] != "-o" {
        eprintln!("Usage: l1s input.is -o output");
        process::exit(1);
    }

    let input_path = &args[2];
    let output_path = &args[3];

    // Phase 1: Read and lex
    let source = match std::fs::read_to_string(input_path) {
        Ok(s) => s,
        Err(e) => {
            eprintln!("error: could not read '{}': {}", input_path, e);
            process::exit(1);
        }
    };

    // Phase 2: Lex
    let (tokens, lex_errors) = lexer::tokenize(&source);
    report(&lex_errors);
    if has_errors(&lex_errors) { process::exit(1); }

    // Phase 3: Parse
    let (ast, parse_errors) = parser::parse(&tokens);
    report(&parse_errors);
    if has_errors(&parse_errors) { process::exit(1); }

    // Phase 4: Symbol table + Type check
    let sym = symbol::build(&ast);
    let type_errors = check::typeck(&ast, &sym);
    report(&type_errors);
    if has_errors(&type_errors) { process::exit(1); }

    // Phase 5: IR build
    let mut ir = ir_build::build(&ast, &sym);

    // Phase 6: Optimize
    iropt::optimize(&mut ir);

    // Phase 7: Register allocation
    let alloc = regalloc::assign(&ir);

    // Phase 8: Code generation
    let asm_path = format!("{}.asm", output_path);
    let asm = codegen::generate(&ir, &alloc);
    std::fs::write(&asm_path, &asm).unwrap();

    // Phase 9: Assemble and link
    let status = process::Command::new("nasm")
        .args(&["-f", "elf64", &asm_path, "-o", &format!("{}.o", output_path)])
        .status()
        .expect("failed to run nasm");

    if !status.success() {
        eprintln!("error: assembler failed");
        process::exit(1);
    }

    let status = process::Command::new("cc")
        .args(&["-o", output_path, &format!("{}.o", output_path), "-no-pie"])
        .status()
        .expect("failed to run cc");

    if !status.success() {
        eprintln!("error: linker failed");
        process::exit(1);
    }
}
main.ml
let () =
  let args = Array.to_list Sys.argv in
  let input, output = match args with
    | _ :: input :: "-o" :: output :: [] -> input, output
    | _ ->
        Printf.eprintf "Usage: l1s input.is -o output\n";
        exit 1
  in

  (* Phase 1: Read *)
  let source = match In_channel.read_all input with
    | s -> s
    | exception exn ->
        Printf.eprintf "error: could not read '%s': %s\n" input (Printexc.to_string exn);
        exit 1
  in

  (* Phase 2: Lex *)
  let tokens, lex_errs = Lexer.tokenize source in
  report lex_errs;
  if has_errors lex_errs then exit 1;

  (* Phase 3: Parse *)
  let ast, parse_errs = Parser.parse tokens in
  report parse_errs;
  if has_errors parse_errs then exit 1;

  (* Phase 4: Type check *)
  let sym = Symbol.build ast in
  let tc_errs = Check.typeck ast sym in
  report tc_errs;
  if has_errors tc_errs then exit 1;

  (* Phase 5-7: IR, opt, regalloc *)
  let ir = Ir_build.build ast sym in
  let ir = Iropt.optimize ir in
  let alloc = Regalloc.assign ir in

  (* Phase 8: Codegen *)
  let asm_path = Printf.sprintf "%s.asm" output in
  let asm = Codegen.generate ir alloc in
  Out_channel.write_all asm_path asm;

  (* Phase 9: Assemble and link *)
  let _ = Sys.command (Printf.sprintf
    "nasm -f elf64 %s -o %s.o && cc -o %s %s.o -no-pie"
    asm_path output output output)
  in
  ()
Why -no-pie? By default, modern GCC produces Position Independent Executables (PIE). Our compiler generates absolute addresses in the assembly (not PIC), so we need -no-pie to tell the linker not to assume position-independence. If we wanted PIC, we would need to generate RIP-relative addressing — a good exercise.

Debugging Flags

When something goes wrong, we need to see what each stage produces. We add debugging flags that dump the intermediate representations:

FlagWhat it dumpsUse case
--dump-astPretty-printed ASTParser bugs — see what the parser actually builds
--dump-irFormatted IR with basic blocksIR builder or optimizer bugs — see the SSA form
--dump-asmGenerated assemblyCodegen bugs — see the raw output before assembling
--dump-allAll of the aboveFull trace — every stage visible

These flags are invaluable for debugging. When a compiled program produces the wrong result, dump the IR to see if the optimizer introduced a bug. Dump the assembly to see if the wrong instruction was selected. Build the discipline of always checking the intermediate output — it saves hours of staring at source code.

Build System

Our Rust compiler is built with Cargo, the standard Rust build system. Our OCaml compiler uses Dune (formerly jbuilder). Here is what each build file looks like:

Cargo.toml
[package]
name = "l1s"
version = "0.1.0"
edition = "2021"

[dependencies]
# No external dependencies for the core compiler
# std::process for invoking nasm and cc
dune-project
(lang dune 3.0)
(name l1s)

Both build systems handle the same thing: compiling the source files, linking them into a single binary, and providing a command-line interface. The Rust version produces a single l1s binary. The OCaml version produces a l1s.exe native binary (or l1s.bc for bytecode).

Testing

Every phase of the compiler should be tested independently. We use a three-tier testing strategy:

Unit tests live alongside each module and test individual functions. The lexer tests verify that "42" produces Token::Num(42). The parser tests verify that "return 1 + 2;" produces the correct AST. The type checker tests verify that type mismatches are caught.

Integration tests compile complete programs and run them, checking the exit code. This is the most important test category — it verifies the entire pipeline works end-to-end.

tests/integration.rs
use std::process::Command;

fn compile_and_run(source: &str, expected: i32) {
    std::fs::write("test_input.l1s", source).unwrap();
    Command::new("l1s")
        .args(&["test_input.l1s", "-o", "test_output"])
        .status()
        .unwrap();
    let output = Command::new("./test_output")
        .output()
        .unwrap();
    assert_eq!(output.status.code(), Some(expected));
}

#[test]
fn test_return_42() {
    compile_and_run("func main(i32) { return 42; }", 42);
}

#[test]
fn test_add() {
    compile_and_run(
        "func main(i32) { let x: i32 = 10; let y: i32 = 32; return x + y; }",
        42,
    );
}

#[test]
fn test_if_else() {
    compile_and_run(
        "func main(i32) { if (1 > 0) { return 1; } else { return 0; } }",
        1,
    );
}
tests/test_runner.ml
let compile_and_run source expected =
  In_channel.write_all "test_input.l1s" source;
  Sys.command "l1s test_input.l1s -o test_output" |> ignore;
  let code = Sys.command "./test_output" in
  assert (code = expected)

let () =
  compile_and_run "func main(i32) { return 42; }" 42;
  compile_and_run
    "func main(i32) { let x: i32 = 10; let y: i32 = 32; return x + y; }"
    42;
  Printf.printf "All tests passed!\n"
Test-driven compiler development The best way to build a compiler is to write the test first, then make it pass. Want to add & (bitwise AND)? Write a test that compiles return 5 & 3; and expects 1. Then implement it in the lexer, parser, IR builder, and codegen. The test tells you when you are done. This is the same discipline as Rust's cargo test.

Full Compilation Session

Let us walk through a complete session from source to execution:

$ cat > example.l1s << EOF
func main(i32) {
    return 42;
}
EOF

$ cargo run -- example.l1s -o example
   Compiling l1s v0.1.0
    Finished dev [unoptimized + debuginfo] target(s) in 0.12s
     Running `target/debug/l1s example.l1s -o example`

$ nasm -f elf64 example.asm -o example.o

$ cc -o example example.o -no-pie

$ ./example; echo $?
42

That is our compiler, producing a real executable, running on real hardware. Every step is visible. Every step is something we built. The entire toolchain — from nasm to cc — is standard Linux infrastructure. Our compiler plugs right in.

We can also compile and run in a single shell pipeline:

$ echo "func main(i32) { return 42; }" > /tmp/test.l1s
$ l1s /tmp/test.l1s -o /tmp/test && /tmp/test; echo $?
42
Speed Our compiler compiles this program in under 5 milliseconds (excluding the Cargo compile time of the compiler itself). The compilation is dominated by NASM and cc invocation overhead, not by our code. For a simple compiler, that is fast enough. As the language grows, we may want to add an internal assembler to avoid the NASM dependency — but that is a project for another day.

What the Driver Enables

The driver is the user's interface to the compiler. It is the part that makes the compiler usable. With it, we can:

  • Compile programs from files by running a single command.
  • See detailed errors with source locations and descriptions.
  • Debug the compiler by dumping intermediate representations.
  • Run a test suite automatically with integration tests.
  • Integrate with build systems like Make or Cargo.

The driver is also the place where we handle command-line arguments, file paths, and system calls. It is the only part of the compiler that interacts with the operating system directly. Everything else is pure data transformation — functions that take input and produce output with no side effects. This separation of concerns is what makes the compiler testable and maintainable.

Organizing the Test Suite

Our test suite has three tiers, mirroring Rust's own testing philosophy:

Tier 1: Unit tests live alongside each module and test individual functions. These are fast (microseconds each) and catch bugs at the smallest scope. Every module should have at least basic unit tests for its core functions.

src/check.rs — unit tests
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn check_literal_types() {
        let sym = SymbolTable::new();
        assert_eq!(
            check_expr(&Expr::Int(42), &sym, &Type::Prim("i32")),
            TypeResult::Type(Type::Prim("i32"))
        );
    }

    #[test]
    fn type_mismatch_detected() {
        let prog = parse("func main(i32) { let x: bool = 42; }");
        let sym = build_top_level(&prog).unwrap();
        let diags = typeck(&prog, &sym);
        assert!(diags.len() > 0);
        assert!(diags[0].msg.contains("type mismatch"));
    }
}
check.ml — unit tests
let () =
  (* check_literal_types *)
  let sym = Symbol.create () in
  assert (check_expr (Int 42) sym (Prim "i32") =
            Type (Prim "i32"));

  (* type_mismatch_detected *)
  let prog = Parser.parse (Lexer.tokenize
    "func main(i32) { let x: bool = 42; }") in
  let sym = Symbol.build_top_level prog in
  let diags = Check.typeck prog sym in
  assert (List.length diags > 0);
  assert (String.contains ~sub:"type mismatch" (List.hd diags).msg)

Tier 2: Integration tests compile complete programs and run them. These are our most valuable tests because they exercise every layer of the compiler simultaneously. A single integration test can catch bugs in the lexer, parser, type checker, IR builder, optimizer, register allocator, code generator, assembler, and linker.

Tier 3: Golden file tests are the most thorough. We compile a program, capture the generated assembly, and compare it against a saved "golden" file. If the assembly changes, we see the diff. These tests catch regressions in code generation quality — even when the output is still correct, we can see if it became worse.

# Golden file test structure
tests/
  golden/
    return_42.l1s          # source
    return_42.asm          # expected assembly output
    add.l1s
    add.asm
    factorial.l1s
    factorial.asm

Running golden file tests is simple: compile each .l1s file, diff the output against the expected .asm, and report mismatches. This catches subtle codegen bugs that functional tests might miss.

Test-driven compiler development workflow The fastest way to build a compiler is TDD: (1) Write a test for the feature you want. (2) Run it — it fails because the feature does not exist yet. (3) Implement the feature in the smallest possible change. (4) Run the test — it passes. (5) Commit. Every test is a specification. Every passing test is proof of progress. This is how rustc itself is developed — the Rust test suite has over 100,000 tests.

Exercises

1. Add --dump-ast flag Implement the --dump-ast flag that prints the pretty-printed AST to stdout before compiling. This is useful for debugging the parser. The flag should be optional — if not provided, the compiler proceeds normally.
2. Add a test runner script Write a test runner that scans a directory for .l1s files, compiles each one, runs the resulting executable, and checks that the exit code matches a comment at the top of the file (e.g., // exit: 42). This makes it easy to add test cases without modifying the test source code.
3. Add --keep-asm flag By default, clean up the .asm and .o files after linking. Add a --keep-asm flag that leaves them on disk for inspection. This is helpful when debugging the code generator.
4. Parallel compilation If the source file contains multiple functions, they could theoretically be compiled in parallel. Explore adding threaded compilation using Rust's std::thread or OCaml's Thread module. What synchronization is needed? Is the speedup worth the complexity?
5. Incremental compilation Cache the IR for each function. On recompilation, only recompute functions whose source has changed (compare file modification times or content hashes). This is what rustc does with its incremental compilation — it dramatically speeds up edit-compile-test cycles.
