Building a Compiler Rust Edition

7. From IR to Machine Code

We have SSA IR. Clean, well-formed, optimized. But the CPU does not understand virtual registers or basic blocks. The CPU understands x86-64 instructions: mov, add, cmp, jmp, call, ret. Our job in this chapter is to turn IR into real x86-64 assembly.

This is the most satisfying phase of the compiler. Everything before was setup. Now we produce something that actually runs on hardware.

There are three problems to solve:

  1. Register allocation: We have infinite virtual registers. The CPU has 16 (or fewer). Map virtual registers to real registers or stack slots.
  2. Instruction selection: Each IR instruction maps to one or more x86-64 instructions.
  3. Phi resolution: Phi nodes are conceptual — in machine code, they become moves at the end of predecessor blocks.

Register Allocation: Linear Scan

Think of register allocation like Rust's allocator: we have a fixed resource (registers) and competing demands (virtual registers). Every virtual register has a live interval — the range of instructions where its value is needed. If two virtual registers have overlapping live intervals, they cannot share a register. If they do not overlap, they can.

Linear scan is the simplest practical register allocation algorithm. It is not optimal (graph coloring gives better results), but it is fast, simple, and good enough for our compiler. LLVM uses linear scan as a fast path and graph coloring for optimization.

The algorithm:

  1. Compute live intervals for every virtual register.
  2. Sort by start point.
  3. Scan: maintain a set of active intervals (those that are live at the current point). When an interval ends, free its register. When a new interval starts, assign it a register. If none is free, spill — store the value to the stack and reload it when needed.
Live intervals: %0: [1, 3] ──── %1: [2, 5] ────── %2: [4, 6] ──── %3: [7, 8] ── Registers available: r0, r1, r2 Scan: instr 1: %0 starts → assign r0 instr 2: %1 starts → assign r1 instr 4: %2 starts → assign r2 (all regs used, but one ends soon) instr 6: %2 ends → free r2 instr 7: %3 starts → assign r2 (reuse!)

Here is the core allocator in Rust:

src/regalloc.rs
pub struct LiveInterval {
    pub vreg: VReg,
    pub start: usize,
    pub end: usize,
}

pub fn linear_scan(
    intervals: &mut [LiveInterval],
    regs: usize,
) -> HashMap<VReg, Alloc> {
    intervals.sort_by_key(|iv| iv.start);

    let mut active: Vec<usize> = Vec::new();      // indices into intervals
    let mut free_regs: Vec<usize> = (0..regs).collect(); // available registers
    let mut result = HashMap::new();
    let mut stack_slot = regs;  // next stack slot for spilled values

    for i in 0..intervals.len() {
        // Expire dead intervals
        active.retain(|&j| {
            if intervals[j].end <= intervals[i].start {
                free_regs.push(j);   // free the register
                false
            } else {
                true
            }
        });

        if let Some(reg) = free_regs.pop() {
            result.insert(intervals[i].vreg, Alloc::Reg(reg));
            active.push(i);
        } else {
            // No free register — spill to stack
            let slot = stack_slot;
            stack_slot += 1;
            result.insert(intervals[i].vreg, Alloc::Stack(slot));
        }
    }

    result
}
regalloc.ml
type live_interval = {
  vreg:  int;
  start: int;
  end_:   int;
}

type alloc = Reg of int | Stack of int

let linear_scan intervals num_regs =
  let intervals = List.sort (fun a b -> Int.compare a.start b.start) intervals in
  let active = ref [] in
  let free_regs = ref (List.init num_regs (fun i -> i)) in
  let result = Hashtbl.create 64 in
  let next_slot = ref num_regs in

  List.iter (fun iv ->
    (* Expire dead intervals *)
    active := List.filter (fun j ->
      let j_iv = List.nth intervals j in
      if j_iv.end_ <= iv.start then (
        free_regs := j :: !free_regs;
        false
      ) else
        true
    ) !active;

    match !free_regs with
    | reg :: rest ->
        free_regs := rest;
        Hashtbl.replace result iv.vreg (Reg reg);
        active := (List.length active) :: !active
    | [] ->
        let slot = !next_slot in
        next_slot := slot + 1;
        Hashtbl.replace result iv.vreg (Stack slot)
  ) intervals;
  result
Linear scan is not perfect Our implementation is simplified. A real linear scan allocator also handles spill costs — when no register is free, you can choose to spill the interval that ends latest (farthest in the future) rather than the current one. This makes better decisions. Adding spill cost heuristics is exercise 3.

The Stack Frame

Every function that uses local variables or spills registers needs a stack frame. The frame is a region of memory on the stack that holds local data. It is set up in the prologue and torn down in the epilogue:

High addresses ┌──────────────────┐ │ argument N │ ← caller's stack │ ... │ │ 1st argument │ ├──────────────────┤ │ return address │ ← pushed by `call` ├──────────────────┤ │ saved rbp │ ← pushed by `push rbp` ├──────────────────┤ │ local variable 0 │ │ ... │ ← accessed via [rbp - offset] │ spilled vregs │ │ local variable N │ └──────────────────┘ Low addresses

The prologue and epilogue in assembly:

src/codegen.rs
fn emit_prologue(asm: &mut Assembly, frame_size: usize) {
    asm.emit("    push rbp");
    asm.emit("    mov rbp, rsp");
    if frame_size > 0 {
        // Align frame size to 16 bytes (x86-64 ABI requirement)
        let aligned = (frame_size + 15) & !15;
        asm.emit(&format!("    sub rsp, {}", aligned));
    }
}

fn emit_epilogue(asm: &mut Assembly) {
    asm.emit("    mov rsp, rbp");
    asm.emit("    pop rbp");
    asm.emit("    ret");
}

The code generator walks each basic block, emitting instructions. Spilled registers become loads from and stores to [rbp - offset]. Non-spilled registers become direct register references.

The Assembly Buffer

We collect the generated assembly in a simple string buffer. Here is the type:

src/codegen.rs
pub struct Assembly {
    lines: Vec<String>,
    indent: usize,
}

impl Assembly {
    pub fn new() -> Self {
        Self { lines: Vec::new(), indent: 0 }
    }

    pub fn emit(&mut self, line: &str) {
        self.lines.push(line.to_string());
    }

    pub fn emit_label(&mut self, label: &str) {
        self.lines.push(format!("{}:", label));
    }

    pub fn finish(&self) -> String {
        self.lines.join("\n")
    }
}
codegen.ml
type assembly = {
  mutable lines: string list;
}

let create () = { lines = [] }

let emit asm line =
  asm.lines <- line :: asm.lines

let emit_label asm label =
  asm.lines <- (Printf.sprintf "%s:" label) :: asm.lines

let finish asm =
  String.concat "\n" (List.rev asm.lines)

We emit lines in order as we traverse the IR. Labels for basic blocks, instructions for operations. The output is valid NASM syntax — NASM is the assembler we invoke to turn this text into machine code.

Generating Code for a Basic Block

Here is how we generate code for a complete basic block, including its terminator:

src/codegen.rs
fn emit_block(
    block: &BasicBlock,
    alloc: &HashMap<VReg, Alloc>,
    asm: &mut Assembly,
) {
    asm.emit_label(&format!(".L{}", block.id));

    for instr in &block.instrs {
        emit_instr(instr, alloc, asm);
    }

    match &block.terminator {
        Terminator::Return(val) => {
            if let Some(v) = val {
                let rv = reg_or_stack(*v, alloc);
                asm.emit(&format!("    mov rax, {}", rv));
            }
            emit_epilogue(asm);
        }
        Terminator::Jump(target) => {
            asm.emit(&format!("    jmp .L{}", target));
        }
        Terminator::Branch(cond, t, f) => {
            let rc = reg_or_stack(*cond, alloc);
            asm.emit(&format!("    cmp {}, 1", rc));
            asm.emit(&format!("    je .L{}", t));
            asm.emit(&format!("    jmp .L{}", f));
        }
    }
}
codegen.ml

src/codegen.rs
fn emit_instr(
    instr: &Instr,
    alloc: &HashMap<VReg, Alloc>,
    asm: &mut Assembly,
) {
    match *instr {
        Instr::Const(dst, n) => {
            let d = reg_or_stack(dst, alloc);
            asm.emit(&format!("    mov {}, {}", d, n));
        }
        Instr::Add(dst, a, b) => {
            let d = reg_or_stack(dst, alloc);
            let ra = reg_or_stack(a, alloc);
            let rb = reg_or_stack(b, alloc);
            asm.emit(&format!("    mov {}, {}", d, ra));
            asm.emit(&format!("    add {}, {}", d, rb));
        }
        Instr::Eq(dst, a, b) => {
            let d = reg_or_stack(dst, alloc);
            let ra = reg_or_stack(a, alloc);
            let rb = reg_or_stack(b, alloc);
            asm.emit(&format!("    cmp {}, {}", ra, rb));
            asm.emit("    sete al");
            asm.emit(&format!("    movzx {}, al", d));
        }
        Instr::Ret(val) => {
            let rv = reg_or_stack(val, alloc);
            asm.emit(&format!("    mov rax, {}", rv));
        }
        // ... other instructions
        _ => {}
    }
}
codegen.ml
let emit_instr instr alloc asm =
  match instr with
  | Const (dst, n) ->
      let d = reg_or_stack dst alloc in
      emit asm (Printf.sprintf "    mov %s, %d" d n)
  | Add (dst, a, b) ->
      let d  = reg_or_stack dst alloc in
      let ra = reg_or_stack a   alloc in
      let rb = reg_or_stack b   alloc in
      emit asm (Printf.sprintf "    mov %s, %s" d ra);
      emit asm (Printf.sprintf "    add %s, %s" d rb)
  | Eq (dst, a, b) ->
      let d  = reg_or_stack dst alloc in
      let ra = reg_or_stack a   alloc in
      let rb = reg_or_stack b   alloc in
      emit asm (Printf.sprintf "    cmp %s, %s" ra rb);
      emit asm "    sete al";
      emit asm (Printf.sprintf "    movzx %s, al" d)
  | Ret (Some val) ->
      let rv = reg_or_stack val alloc in
      emit asm (Printf.sprintf "    mov rax, %s" rv)
  | _ -> ()

Phi Resolution

Phi nodes do not exist in x86-64. They are a conceptual tool for the optimizer. In machine code, we resolve them by inserting move instructions at the end of each predecessor block:

IR with Phi: b2 (then): jmp b4 b3 (else): jmp b4 b4 (merge): %3 = phi [b2: %0, b3: %1] ret %3 Generated assembly: b2: mov rax, r0 ; move %0 into rax (return value) jmp b4 b3: mov rax, r1 ; move %1 into rax jmp b4 b4: ; %3 is now in rax (no Phi needed) ret

The key insight: a Phi node is not a real instruction. It is a statement about data flow. When we generate code, we resolve it by copying the incoming value at the end of each predecessor into the register or stack slot that holds the Phi's result. If multiple Phis exist, we may need temporary registers to avoid clobbering (the "swap problem"), but for our simple compiler, sequential moves work fine.

Complete Assembly Output

Let us see what our compiler produces for a complete program. Here is the simplest possible:

func main(i32) {
    return 42;
}

The output assembly:

    global main
    section .text

main:
    ; prologue
    push rbp
    mov rbp, rsp

    ; return 42
    mov rax, 42

    ; epilogue
    mov rsp, rbp
    pop rbp
    ret

That is real x86-64 assembly. Generated by our compiler. Running on real hardware. Seven instructions. It pushes the base pointer, saves the frame pointer, loads 42 into rax (the return value register), restores the frame, pops the base pointer, and returns.

Now something more complex — a function with parameters and arithmetic:

func add(i32, a: i32, b: i32) {
    return a + b;
}

The output assembly:

    global add
    section .text

add:
    ; prologue
    push rbp
    mov rbp, rsp

    ; parameters arrive in rdi, rsi
    ; a is in rdi, b is in rsi (SysV ABI)

    ; a + b
    mov rax, rdi
    add rax, rsi

    ; epilogue
    mov rsp, rbp
    pop rbp
    ret

The SysV calling convention passes the first two integer arguments in rdi and rsi. Our compiler maps parameter VRegs to these registers. The addition becomes a mov and an add. The result is already in rax, ready for the caller to read.

That's real hardware You can take this assembly, assemble it with NASM, link it with cc, and run it on any x86-64 Linux machine. Our compiler produces actual executables. Not bytecode. Not a VM. Real native code.

What Code Generation Enables

With code generation, our pipeline is complete:

  • Lexer → tokens
  • Parser → AST
  • Type checker → validated AST
  • IR builder → SSA IR
  • Optimizer → optimized SSA IR
  • Register allocator → mapped VRegs
  • Code generator → x86-64 assembly

Every stage is a well-defined transformation with a clean interface. The output of the last stage is a string of assembly text that NASM can assemble into machine code.

This is the point where the compiler stops being a thought experiment and starts being a tool. Every other program we write in our language — from "hello world" to a self-hosting compiler — goes through this pipeline. And now you understand every step.

Calling Convention in Detail

The System V AMD64 ABI specifies exactly how functions call each other. Here are the rules our code generator must follow:

Argument passing: Integer and pointer arguments are passed in registers, left to right: rdi, rsi, rdx, rcx, r8, r9. Additional arguments are pushed onto the stack in reverse order.

Return value: Integer and pointer return values go in rax. Our function main(i32) returns an i32 — the generated assembly loads the return value into rax before ret.

Callee-saved registers: rbx, rbp, and r12–r15 must be preserved across function calls. If our code generator uses these registers, it must save and restore them. Our simple allocator avoids this by using scratch registers and stack slots.

Stack alignment: The stack must be 16-byte aligned before a call instruction. Our prologue push rbp (8 bytes) + sub rsp, N (with N aligned to 16) maintains this invariant.

Here is how a function call looks in the generated assembly:

;; Call: result = add(3, 4);
    mov rdi, 3           ; first argument
    mov rsi, 4           ; second argument
    call add             ; push return addr, jump to add
    mov [rbp-8], rax    ; store return value to result

The callee (add) receives rdi=3 and rsi=4. It computes the result, leaves it in rax, and returns. The caller reads rax and stores it. This is the entire calling convention — everything else is mechanical.

Exercises

1. Graph coloring register allocation Replace linear scan with a graph coloring allocator. Build an interference graph where nodes are VRegs and edges connect overlapping live intervals. Color the graph with k colors (k = number of physical registers). Spill if coloring fails. This produces better code than linear scan and is the algorithm used by GCC and LLVM's -O2.
2. Floating point support Add f32 and f64 types. The IR needs new instructions for floating-point arithmetic. The code generator needs SSE/AVX instructions: movss, addss, mulss, ucomiss, etc. New registers: xmm0–xmm15. This is a significant extension that touches every part of the compiler.
3. Spill cost heuristics Improve linear scan with spill cost: when no register is free, spill the active interval whose end is farthest from the current point (instead of spilling the newly arriving VReg). This requires tracking which interval to spill and adding spill/reload instructions to the IR.
4. Peephole optimization on assembly After the code generator emits instructions, run a peephole optimizer over the assembly text. Replace patterns like mov rax, [rbp-8]; mov [rbp-8], rax (a redundant load-store pair) with nothing. Or replace mov rax, 0 with xor rax, rax (smaller encoding).
5. Tail call optimization When a function's last action is calling another function, and the callee's return value is directly returned, optimize the call into a jump (jmp instead of call; ret). This avoids growing the stack for recursive functions. Requires recognizing the tail position in the IR.
