The Magic Trick

You write code. The computer runs it. How?

At first this seems like magic. You type letters into a file — func main(i32) { return 42; } — and somehow the electrons in your CPU arrange themselves to return 42. But there's no magic, just a series of well-defined transformations. Each transformation takes one representation and produces another, moving from text toward machine code.

That chain of transformations is a compiler. And you're going to build one.

By the end of this book you'll have built a working compiler for a small but real language called L1. It will have functions, variables, arithmetic, control flow, and first-class types. You'll write the lexer, parser, semantic analyzer, code generator, and enough runtime to execute real programs.

example/main.l1

example/main.l1

Every piece here means something:

Token Meaning
func Declare a function. Like fn in Rust.
main The function name (entry point, like fn main() in Rust).
(i32) Return type: this function returns a 32-bit signed integer.
{ return 42; } Function body: evaluates 42 and returns it to the caller.

The compiler's job is to take this text, understand its structure (a function named main returning i32 with a body that returns 42), verify it's well-formed, and generate code that a computer can execute.

The Pipeline at 30,000 Feet

A modern compiler is organized as a series of phases. Each phase transforms the program from one representation to the next, gradually moving from source text toward machine code:

Source text
    │
    ▼
┌─────────────┐
│   LEXER     │  → Tokens: func, main, (, i32, ), {, return, 42, ;, }
└─────────────┘
    │
    ▼
┌─────────────┐
│   PARSER    │  → AST: FuncDecl "main" (body: Return(Int(42)))
└─────────────┘
    │
    ▼
┌─────────────┐
│  SEMANTICS  │  → Type-checked AST, symbol table
└─────────────┘
    │
    ▼
┌─────────────┐
│  LOWERING   │  → IR: basic blocks, instructions
└─────────────┘
    │
    ▼
┌─────────────┐
│   CODEGEN   │  → Assembly / machine code
└─────────────┘
    │
    ▼
   Output

Each phase:

  1. Lexing — Groups characters into tokens (words, numbers, operators, punctuation). Turns return 42 into [Keyword("return"), Int(42)].
  2. Parsing — Organizes tokens into a tree according to grammar rules. Discovers that return 42 is a return statement with an integer expression.
  3. Semantic Analysis — Checks types, resolves names, ensures rules are followed. Verifies that 42 is a valid i32 and that main returns the right type.
  4. Lowering / IR Generation — Transforms the high-level AST into a low-level intermediate representation suitable for optimization.
  5. Code Generation — Emits actual machine instructions or assembly.

This book covers all five phases, building up from nothing to a complete, working compiler.

Why Rust for a Compiler?

Rust and compiler construction are a perfect match. The language features that make Rust a joy for systems programming are the same features that make it exceptional for building compilers.

Enums Are Algebraic Data Types

An AST is a tree of nodes. In Rust, every kind of node is a variant of an enum:

ast.rs
pub enum AstExpr {
    Int(i64),
    Bool(bool),
    Ident(String),
    Binary {
        left: Box<AstExpr>,
        op: BinaryOp,
        right: Box<AstExpr>,
    },
    Call {
        callee: Box<AstExpr>,
        args: Vec<AstExpr>,
    },
}
ast.ml
type ast_expr =
  | Int of int64
  | Bool of bool
  | Ident of string
  | Binary of ast_expr * binary_op * ast_expr
  | Call of ast_expr * ast_expr list

If you know OCaml, this pattern is familiar — it's an algebraic data type. If you only know Rust, you already understand ADTs: every enum in Rust is an algebraic data type with pattern matching. This means ASTs are not just representable in Rust; they're idiomatic.

Option and Result for Errors

Compilers are error-handling machines. Every phase can fail:

  • The lexer hits an invalid character.
  • The parser sees an unexpected token.
  • The semantic analyzer finds a type mismatch.

Rust's Option and Result types model these failures exactly. The ? operator threads errors through the pipeline naturally:

parser.rs
fn parse_statement(&mut self) -> Result<AstStmt, ParseError> {
    match self.peek() {
        Some(Token::Keyword(Kw::Return)) => {
            self.advance();
            let expr = self.parse_expr()?;
            self.expect(Token::Semicolon)?;
            Ok(AstStmt::Return(expr))
        }
        _ => self.parse_expr_stmt(),
    }
}
parser.ml
let parse_statement parser =
  match peek parser with
  | Some (Token_Keyword Kw_Return) ->
      advance parser;
      let expr = parse_expr parser in
      expect parser Token_Semicolon;
      AstStmt_Return expr
  | _ -> parse_expr_stmt parser

Rust forces you to handle every case — exactly what a compiler needs. No null pointers, no uncaught exceptions, no surprises at runtime.

Pattern Matching Walks Trees

Once you have an AST as an enum, match is your tree-walking superpower:

codegen.rs
fn gen_expr(&self, expr: &AstExpr) -> Value {
    match expr {
        AstExpr::Int(n) => self.emit_const(*n),
        AstExpr::Bool(b) => self.emit_const(*b as i64),
        AstExpr::Ident(name) => self.emit_load_var(name),
        AstExpr::Binary { left, op, right } => {
            let l = self.gen_expr(left);
            let r = self.gen_expr(right);
            self.emit_binary(op, l, r)
        }
        AstExpr::Call { callee, args } => {
            let compiled_args: Vec<_> =
                args.iter().map(|a| self.gen_expr(a)).collect();
            self.emit_call(self.gen_expr(callee), &compiled_args)
        }
    }
}
codegen.ml
let rec gen_expr env = function
  | AstExpr_Int n -> emit_const env n
  | AstExpr_Bool b -> emit_const env (if b then 1L else 0L)
  | AstExpr_Ident name -> emit_load_var env name
  | AstExpr_Binary (l, op, r) ->
      let lv = gen_expr env l in
      let rv = gen_expr env r in
      emit_binary env op lv rv
  | AstExpr_Call (callee, args) ->
      let args = List.map (gen_expr env) args in
      emit_call env (gen_expr env callee) args

Pattern matching is exhaustive — if you add a variant to AstExpr, the compiler will tell you every place you need to handle it. In a compiler with dozens of node types across multiple phases, this is invaluable.

Ownership and the Pipeline

Compilation is a pipeline: data flows forward and is consumed. The lexer produces tokens, the parser consumes them to produce an AST, the semantic analyzer consumes the AST, and so on. This is a linear flow, and Rust's ownership model captures it perfectly.

Each phase moves its input, guaranteeing that no phase accidentally mutates shared state. There's no garbage collector tracing through old token streams — once a phase is done, its data is freed. The borrow checker statically prevents the kinds of aliasing bugs that plague compilers written in languages with shared mutable state.

How Computers Execute Code

It's worth pausing to understand what a compiler actually does at the hardware level. A CPU understands one thing: machine code — sequences of bytes that encode instructions like "load this value into a register," "add two registers," "compare and branch if equal."

Machine code looks like this (x86-64):

; return 42 on x86-64
b8 2a 00 00 00    mov eax, 42
c3                ret

No one writes this by hand. We write:

func main(i32) { return 42; }

The compiler bridges this gap. It takes the human-friendly representation and systematically transforms it into the machine-friendly one. Every phase of the compiler is a step on this journey from abstract (text, trees, types) to concrete (registers, instructions, bytes).

What's remarkable is that this isn't magic — it's a series of completely mechanical transformations. Given the same input, a correct compiler always produces the same output. Every choice the compiler makes — how to lay out a struct in memory, which register to use for a variable, how to order instructions — is determined by rules. Our job is to encode those rules in code.

What Makes a Good Compiler?

As you build your compiler, keep these properties in mind:

Property Why It Matters
Correctness The compiled program must behave exactly as the source specifies. A compiler bug can corrupt data, crash systems, or worse.
Good errors The compiler must tell you what went wrong and where. Vague error messages waste developer time.
Speed Compilation should be fast. Developers compile frequently; every second of compile time adds up.
Generated code quality The output should be efficient. A slow compiled program is a frustrating program.

In this book, we prioritize correctness and good errors above all else. Speed and code quality come from good architecture, not premature optimization. Our compiler will be clean, correct, and reasonably fast — and we'll have a clear path to optimization in later chapters.

Chapter 01: Text and Spans

Where in the source did this error happen?

What you'll learn in this chapter: How to represent positions in source code, why byte offsets are the right internal representation, how to convert to human-readable line:column pairs, and how to render compiler error messages with underlines.

The Problem

Consider this program:

func main(i32) {
  return 42 + "hello";
}

There's a type error: you can't add an integer and a string. The compiler needs to tell the programmer where the error is. Vague messages like "type error at line 2" are frustrating. A good compiler pinpoints the exact location:

error[E0001]: type mismatch: cannot add `i32` and `[i8]`
  → src/main.l1:2:10
   |
 2 |   return 42 + "hello";
   |          ^^^^^^^^^^^^ i32 + [i8]

Every character in that error message — the filename, the line number, the column, the underlines — comes from position tracking. Before we write a single line of lexer or parser code, we need to build the infrastructure for answering "where is this?"

Representing Positions

There are two common ways to identify a location in a source file:

Approach Example Pros Cons
Line:Column 2:10 Human-readable, familiar from every editor Expensive to compute with, ambiguous with tabs
Byte offset 27 Cheap, unambiguous, O(1) operations Needs conversion for human display

Internally, compilers almost always use byte offsets. A byte offset is just a usize: the number of bytes from the start of the source file to a given character. Operations like "does this span contain that position?" become a single integer comparison. Walking forward or backward is pointer-like arithmetic.

Line:column pairs are computed on demand for error messages. This keeps the critical path fast and the internal representation simple.

Span

A Span represents a region of source text from a start offset to an end offset:

src/span.rs
/// A region of source text identified by byte offsets.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct Span {
    /// Byte offset of the first character (0-based).
    pub start: usize,
    /// Byte offset of the first character after the span.
    pub end: usize,
}

impl Span {
    pub fn new(start: usize, end: usize) -> Self {
        debug_assert!(start <= end);
        Self { start, end }
    }

    pub fn len(&self) -> usize {
        self.end - self.start
    }

    pub fn contains(&self, offset: usize) -> bool {
        self.start <= offset && offset < self.end
    }

    pub fn merge(&self, other: &Span) -> Span {
        Span::new(
            self.start.min(other.start),
            self.end.max(other.end),
        )
    }
}
span.ml
type span = {
  start: int;  (* byte offset, 0-based *)
  end_:   int;  (* first byte after the span *)
}

let make_span start end_ = { start; end_ }

let span_len s = s.end_ - s.start

let span_contains s offset =
  s.start <= offset && offset < s.end_

let merge_spans a b =
  { start = min a.start b.start;
    end_  = max a.end_ b.end_ }

The end field uses the "one past the end" convention, which is standard across Rust (slices use it, iterators use it, the standard library uses it). This means end - start is the length, and you can represent empty spans (useful for inserting at a point or marking locations of synthetic nodes).

We derive Copy because Span is a tiny value type — two usizes, 16 bytes on 64-bit. Passing it around by value is cheaper than by reference.

SourceFile

A Span alone isn't enough. Given an offset like 27, we need to know: which file? What's on that line? The SourceFile struct holds the original source text and the line-start index:

src/source_file.rs
use crate::span::Span;

#[derive(Debug, Clone)]
pub struct SourceFile {
    pub name: String,
    pub src: String,
    /// Byte offset of the start of each line.
    line_starts: Vec<usize>,
}

impl SourceFile {
    pub fn new(name: String, src: String) -> Self {
        let line_starts = Self::compute_line_starts(&src);
        Self { name, src, line_starts }
    }

    fn compute_line_starts(src: &str) -> Vec<usize> {
        let mut starts = vec![0];
        for (i, ch) in src.char_indices() {
            if ch == '\n' {
                starts.push(i + ch.len_utf8());
            }
        }
        starts
    }

    pub fn line_col(&self, offset: usize) -> (usize, usize) {
        let line = match self.line_starts.binary_search(&offset) {
            Ok(i) => i + 1,
            Err(i) => i,
        };
        let col = offset - self.line_starts[line - 1];
        (line, col)
    }

    pub fn get_line(&self, line: usize) -> &str {
        let start = self.line_starts[line - 1];
        let end = if line < self.line_starts.len() {
            self.line_starts[line]
        } else {
            self.src.len()
        };
        &self.src[start..end.saturating_sub(1)]
    }
}
source_file.ml
type source_file = {
  name: string;
  src: string;
  line_starts: int array;
}

let compute_line_starts src =
  let starts = ref [0] in
  let i = ref 0 in
  String.iter (fun c ->
    if c = '\n' then starts := (!i + 1) :: !starts;
    incr i
  ) src;
  Array.of_list (List.rev !starts)

let make_source_file name src =
  { name; src; line_starts = compute_line_starts src }

let line_col sf offset =
  let i = Array.binary_search sf.line_starts offset in
  let line = i + 1 in
  let col = offset - sf.line_starts.(i) in
  (line, col)

The line_starts array is computed once when the source file is loaded. We use binary search (binary_search on Vec) to find which line contains a given offset. This is O(log L) where L is the number of lines — negligible for any realistic source file.

Why binary search and not a linear scan? In most compilers, you'll convert far more offsets to line:column during error reporting than there are lines in the source. A linear scan from the start would be O(L) per lookup. Binary search makes it O(log L). For a 1000-line file, that's 10 comparisons instead of up to 1000.

Why Byte Offsets?

You might wonder: why not just store line and column everywhere? It seems more convenient. Here's why compilers use offsets internally:

  1. Efficiency. Incrementing a usize when advancing through source text is one instruction. Updating line:column requires checking for newlines, tracking column width, handling tabs — that's branching logic on every character.
  2. Unambiguous slicing. A Span { start: 12, end: 17 } clearly identifies a slice of text: &source[12..17]. Line:column ranges require recomputing offsets to extract text.
  3. Arithmetic. "Does this span contain that position?" is start <= offset && offset < end. With line:column, you need a lexicographic comparison.
  4. Span merging. Given two spans, the merged span is min(start)..max(end). Clean and fast.
  5. Interchange. Many tools (debug info, source maps, DWARF) use byte offsets. Converting early avoids repeated conversions.

The rule of thumb: store offsets, display lines.

Error Rendering

Now we have everything we need to render beautiful error messages. The core function takes a SourceFile, a Span, and a message, and produces the annotated output:

src/diagnostics.rs
use crate::source_file::SourceFile;
use crate::span::Span;

pub struct Diagnostic {
    pub message: String,
    pub span: Span,
}

impl Diagnostic {
    pub fn render(&self, source: &SourceFile) -> String {
        let (line, col) = source.line_col(self.span.start);
        let source_line = source.get_line(line);
        let underline = " ".repeat(col)
            + &"^".repeat(self.span.len().max(1));

        format!(
            "error: {}\n  → {}:{}:{}\n   |\n {} | {}\n   | {}\n",
            self.message,
            source.name,
            line,
            col,
            line,
            source_line,
            underline,
        )
    }
}
diagnostics.ml
type diagnostic = {
  message: string;
  span: Span.span;
}

let render_diagnostic diag source =
  let line, col = SourceFile.line_col source diag.span.start in
  let source_line = SourceFile.get_line source line in
  let underline = String.make col ' ' ^
                  String.make (max (diag.span.end_ - diag.span.start) 1) '^' in
  Printf.sprintf {|error: %s
  → %s:%d:%d
   |
 %d | %s
   | %s
|} diag.message source.name line col line source_line underline

Applied to our type error example, this produces:

error: type mismatch: cannot add `i32` and `[i8]`
  → src/main.l1:2:10
   |
 2 |   return 42 + "hello";
   |          ^^^^^^^^^^^^ i32 + [i8]

We compute the underline length from the span's byte length. For a single-character token like +, the underline is one ^ character. For a multi-token expression, it spans the full range. The max(1) ensures even a zero-width span produces a visible marker.

Tab warning: Our line_col function uses byte offset directly as the column number. This works great for ASCII text but gives wrong column numbers for tabs (which occupy 1 byte but visually take multiple columns) and multi-byte Unicode characters. A production compiler would track visual column width or expand tabs. For our language, we'll assume spaces-only indentation.

Exercise: Tracing Through an Error

Let's trace through a complete error scenario. Consider this program saved in src/main.l1:

func main(i32) {
  return 1 + true;
}

When the semantic analyzer encounters 1 + true, it finds the expression's span (let's say 21..28). It calls source.line_col(21), which binary-searches line_starts:

line_starts = [0, 20, 37, 39]
offset = 21

binary_search(21):
  idx 2: line_starts[2] = 37 > 21  (too high)
  idx 1: line_starts[1] = 20 <= 21 && 21 < 37  (found!)
  line = 1 + 1 = 2
  col = 21 - 20 = 1

Result: line 2, column 1

Wait — column 1? That's pointing at the space before return. But we want column 10 (the start of 1 + true). Ah — our offset 21 is off. Let's recount:

func main(i32) {\n  return 1 + true;\n}
0         1         2         3
0123456789012345678901234567890123456789

1 + true starts at offset 28 (0-indexed). So source.line_col(28) gives:

binary_search(28):
  line_starts = [0, 20, 37]
  28 is between 20 and 37, so Err(2) → line = 2
  col = 28 - 20 = 8

Result: line 2, column 8

Column 8 is where 1 is. Now the error message works correctly.

This is why we test. Off-by-one errors in position tracking produce confusing error messages that point at the wrong place. Always verify your line_col logic with a few test cases.

What Spans Enable

Good span infrastructure opens up powerful features:

  • Multi-span errors: Point at both the use of an undefined variable and a similar name that is defined: "did you mean 'foobar'?" with an arrow pointing at the definition.
  • Note attachments: "note: 'foo' defined here" with a secondary span.
  • Warnings with suggestions: "warning: unused variable 'x'" pointing at the declaration with a "help: prefix with _" suggestion.
  • IDE integration: Language servers use spans to implement go-to-definition, find-references, and inline diagnostics.

We won't implement all of these, but the foundation is the same: Span and SourceFile.

Chapter 02: Lexing

Raw text is just characters. How do we find the meaningful pieces?

What you'll learn in this chapter: How to write a lexer (tokenizer) from scratch in Rust — scanning characters, grouping them into tokens, handling keywords and operators, and producing a stream the parser can consume.

The Problem

Open a source file and you see structure: keywords like func and return, numbers like 42, operators like +, punctuation like braces and semicolons. But to the computer, it's just a flat sequence of bytes. The lexer's job is structural: group those bytes into meaningful chunks called tokens.

Consider this program fragment:

func main(i32) { return 42; }

A human sees five pieces: a keyword, a name, parentheses with a type, braces, and a return statement. The lexer needs to produce exactly that:

Input:  func main(i32) { return 42; }
Tokens:
  [0] Keyword(Func)   "func"
  [1] Ident("main")   "main"
  [2] OpenParen       "("
  [3] Ident("i32")    "i32"
  [4] CloseParen      ")"
  [5] OpenBrace       "{"
  [6] Keyword(Return) "return"
  [7] IntLit(42)      "42"
  [8] Semicolon       ";"
  [9] CloseBrace      "}"

Each token has two parts: a kind (what it is) and a span (where it came from in the source). The parser will consume these tokens to build the AST, but the parser never needs to think about individual characters again — the lexer abstracts that away.

Keywords vs Identifiers

Our language has reserved keywords like func, return, if, and let. Unreserved words like main, add, or counter are identifiers. The lexer handles both with the same initial logic — it scans a word starting with a letter — and then checks whether the word is a keyword:

        "else"   => Some(Keyword::Else),

        "true"   => Some(Keyword::True),

The Lexer Struct

The lexer holds the source text and a cursor position. It exposes a next_token method that advances through the text and returns one token at a time:

src/lexer.rs
use crate::keyword::keyword_token;
use crate::span::Span;
use crate::token::{Keyword, Token, TokenKind};

pub struct Lexer {
    chars: Vec<char>,
    pos: usize,
}

impl Lexer {
    pub fn new(src: &str) -> Self {
        Self {
            chars: src.chars().collect(),
            pos: 0,
        }
    }

    fn peek(&self) -> Option<char> {
        self.chars.get(self.pos).copied()
    }

    fn advance(&mut self) -> Option<char> {
        let ch = self.peek()?;
        self.pos += 1;
        Some(ch)
    }

    fn is_at_end(&self) -> bool {
        self.pos >= self.chars.len()
    }
}

Why Vec<char> instead of indexing into &str? Rust strings are UTF-8 encoded. Indexing by byte source[0..5] might slice in the middle of a multi-byte character. Using .chars().collect::<Vec<char>>() gives us O(1) access to each Unicode code point. For ASCII-only source code (which our language is), we could use bytes directly, but using char is more robust and the cost is negligible for compilation-scale inputs.

Lexing Methods

The main entry point is next_token, which dispatches to specialized methods based on the current character:

        self.skip_whitespace();
        self.skip_comment();

        if self.is_at_end() {
            return None;
        }

            self.lex_operator()

skip_whitespace

Whitespace separates tokens but carries no semantic meaning. We skip it entirely:

fn skip_whitespace(&mut self) {
    while let Some(ch) = self.peek() {
        if ch.is_ascii_whitespace() {
            self.advance();
        } else {
            break;
        }
    }
}

lex_word

Words start with a letter or underscore and contain letters, digits, and underscores. After scanning the whole word, we check if it's a keyword:

src/lexer.rs
fn lex_word(&mut self) -> TokenKind {
    let start = self.pos;
    while let Some(ch) = self.peek() {
        if ch.is_ascii_alphanumeric() || ch == '_' {
            self.advance();
        } else {
            break;
        }
    }
    let word: String = self.chars[start..self.pos]
        .iter()
        .collect();
    match keyword_token(&word) {
        Some(kw) => TokenKind::Keyword(kw),
        None => TokenKind::Ident(word),
    }
}

lex_number

Numbers are sequences of digits, optionally preceded by a - for negative literals. We parse them into i64:

src/lexer.rs
fn lex_number(&mut self) -> TokenKind {
    let start = self.pos;
    while let Some(ch) = self.peek() {
        if ch.is_ascii_digit() || ch == '_' {
            self.advance();
        } else {
            break;
        }
    }
    let s: String = self.chars[start..self.pos]
        .iter()
        .filter(|&c| c != '_')
        .collect();
    let n: i64 = s.parse().expect("invalid integer literal");
    TokenKind::IntLit(n)
}

lex_operator

The operator lexer needs to distinguish single-character operators from multi-character sequences. This is where we handle the "maximal munch" principle: always match the longest possible token:

fn lex_operator(&mut self) -> TokenKind {
    let ch = self.advance().expect("lex_operator called at end");
    match ch {
        '+' => TokenKind::Plus,
        '-' => {
            if self.peek() == Some('>') {
                self.advance();
                TokenKind::Arrow
            } else {
                TokenKind::Minus
            }
        }
        '*' => TokenKind::Star,
        '/' => { // Note: comments handled before lex_operator
            TokenKind::Slash

        '%' => TokenKind::Percent,
        '=' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::EqEq
            } else {
                TokenKind::Equals
            }
        }
        '!' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::BangEq
            } else {
                TokenKind::Bang
            }
        }
        '<' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::LtEq
            } else {
                TokenKind::Lt
            }
        }
        '>' => {
            if self.peek() == Some('=') {
                self.advance();
                TokenKind::GtEq
            } else {
                TokenKind::Gt
            }
        }
        '&' => {
            if self.peek() == Some('&') {
                self.advance();
                TokenKind::AndAnd
            } else {
                TokenKind::Ampersand
            }
        }
        '|' => {
            if self.peek() == Some('|') {

Error Handling: Invalid Characters

Our current implementation panic!s on unexpected characters. In a production compiler, this is unacceptable — a single stray @ in a 10,000-line file should not crash the compiler. Instead, we should emit an error and continue:

src/lexer.rs — error recovery
/// Result type for lexer operations.
pub type LexResult<T> = Result<T, LexError>;

#[derive(Debug, Clone)]
pub struct LexError {
    pub message: String,
    pub span: Span,
}

impl Lexer {
    pub fn next_token(&mut self) -> Option<Result<Token, LexError>> {
        self.skip_whitespace();
        self.skip_comment();

        if self.is_at_end() {
            return None;
        }

        let start = self.pos;
        let ch = self.peek().unwrap();

        if ch.is_ascii_alphabetic() || ch == '_' {
            Some(Ok(self.make_token(start, self.lex_word())))
        } else if ch.is_ascii_digit() {
            Some(Ok(self.make_token(start, self.lex_number())))
        } else {
            match self.try_lex_operator() {
                Ok(kind) => Some(Ok(self.make_token(start, kind))),
                Err(msg) => {
                    self.advance(); // skip the invalid character
                    Some(Err(LexError {
                        message: msg,
                        span: Span::new(start, self.pos),
                    }))
                }
            }
        }
    }
}
lexer.ml

  skip_comment lex;

Chapter 03: Parsing

Tokens are flat. Programs have structure.

What you'll learn in this chapter: How to build a recursive descent parser in Rust, how Pratt parsing makes expressions elegant, how to handle error recovery, and how to transform a token stream into an abstract syntax tree.

The Problem

The lexer gives us a flat list of tokens: [Keyword(Func), Ident("add"), OpenParen, ...]. But a program isn't a list — it's a tree. A function contains statements. A statement contains expressions. An expression contains sub-expressions.

The parser's job is to discover this tree structure from the flat token stream, guided by the grammar. It answers questions like:

  • Is this if keyword the start of an if-statement? Check for (, an expression, ), and a body.
  • Does this + bind tighter than that ==? Check operator precedence.
  • Is this } the end of a function body or an if-else? Check the nesting context.

The output is an abstract syntax tree (AST) — a tree representation of the program's structure that discards syntactic details (like parentheses and semicolons) while preserving the semantic structure.

Tokens (flat):
  func add ( i32 , a : i32 , b : i32 ) { return a + b ; }

AST (tree):
  Program
    └── FuncDecl "add" → i32
          ├── Param("a", i32)
          ├── Param("b", i32)
          └── Body
                └── Return
                      └── Binary(+, "a", "b")

Recursive Descent Parsing

Recursive descent is the simplest and most widely used parsing technique for hand-written compilers. The idea: each grammar rule becomes a function. The function consumes tokens from the input and returns an AST node.

Here's our grammar rules in informal form:

Rule Grammar Function
Program func_decl* parse_program
FuncDecl func ident(type, params) "{" stmt* "}" parse_func_decl
Statement return expr ; | let ident : type = expr ; | if (expr) block else block | while (expr) block | expr ; parse_statement
Expression ... (Pratt parsing) parse_expr
Type ident | [N]type | @type parse_type

Because functions call each other recursively (a function body contains statements, which contain expressions, which can contain function calls), the parser is naturally recursive. Hence the name.

The Parser Struct

The parser holds the token stream, a cursor, and a list of diagnostics. Errors are collected, not thrown — we keep going after each error to find more:

src/parser.rs
use crate::ast::*;
use crate::diagnostics::Diagnostic;
use crate::token::{Keyword, Token, TokenKind};

pub struct Parser {
    tokens: Vec<Token>,
    pos: usize,
    pub diagnostics: Vec<Diagnostic>,
}

impl Parser {
    pub fn new(tokens: Vec<Token>) -> Self {
        Self {
            tokens,
            pos: 0,
            diagnostics: Vec::new(),
        }
    }

    fn peek(&self) -> Option<&TokenKind> {
        self.tokens.get(self.pos).map(|t| &t.kind)
    }

    fn advance(&mut self) -> &Token {
        let tok = &self.tokens[self.pos];
        self.pos += 1;
        tok
    }

    fn expect(&mut self, kind: TokenKind) -> bool {
        if self.peek() == Some(&kind) {
            self.advance();
            true
        } else {
            let found = self.peek().cloned().unwrap_or(TokenKind::Semicolon);
            self.error(format!("expected {:?}, found {:?}", kind, found));
            false
        }
    }

    fn error(&mut self, msg: String) {
        let span = self.tokens[self.pos.min(self.tokens.len() - 1)].span;
        self.diagnostics.push(Diagnostic { message: msg, span });
    }
}

Why expect returns a bool: In a recursive descent parser, expect both consumes the expected token and reports the error. Returning bool lets the caller decide whether to abort parsing the current construct or try to recover. Most callers will continue — we want maximal error reporting, not early exit.

Parsing a Program

A program is a sequence of function declarations:

src/parser.rs — program & func decl
pub fn parse_program(&mut self) -> AstProgram {
    let mut funcs = Vec::new();
    while self.peek().is_some() {
        let decl = self.parse_func_decl();
        funcs.push(decl);
    }
    AstProgram { funcs }
}

fn parse_func_decl(&mut self) -> AstFuncDecl {
    self.expect(TokenKind::Keyword(Keyword::Func));
    let name = self.expect_ident();

    self.expect(TokenKind::OpenParen);
    let ret_type = self.parse_type();
    let mut params = Vec::new();

    while self.peek() == Some(&TokenKind::Comma) {
        self.advance(); // consume comma
        params.push(self.parse_param());
    }

    self.expect(TokenKind::CloseParen);
    self.expect(TokenKind::OpenBrace);

    let mut body = Vec::new();
    while self.peek() != Some(&TokenKind::CloseBrace)
        && self.peek().is_some()
    {
        body.push(self.parse_statement());
    }

    self.expect(TokenKind::CloseBrace);
    AstFuncDecl { name, ret_type, params, body }
}

fn parse_param(&mut self) -> AstParam {
    let name = self.expect_ident();
    self.expect(TokenKind::Colon);
    let ty = self.parse_type();
    AstParam { name, ty }
}

Parsing Statements

Statements come in several forms. The parser peeks at the current token to decide which kind to parse:

src/parser.rs — statements
fn parse_statement(&mut self) -> AstStmt {
    match self.peek() {
        Some(TokenKind::Keyword(Keyword::Return)) => {
            self.advance();
            let expr = self.parse_expr();
            self.expect(TokenKind::Semicolon);
            AstStmt::Return(expr)
        }
        Some(TokenKind::Keyword(Keyword::Let)) => {
            self.advance();
            let name = self.expect_ident();
            self.expect(TokenKind::Colon);
            let ty = self.parse_type();
            self.expect(TokenKind::Equals);
            let init = self.parse_expr();
            self.expect(TokenKind::Semicolon);
            AstStmt::VarDecl { name, ty, init }
        }
        Some(TokenKind::Keyword(Keyword::If)) => {
            self.parse_if_stmt()
        }
        Some(TokenKind::Keyword(Keyword::While)) => {
            self.parse_while_stmt()
        }
        Some(TokenKind::OpenBrace) => {
            AstStmt::Block(self.parse_block())
        }
        _ => {
            let expr = self.parse_expr();
            self.expect(TokenKind::Semicolon);
            AstStmt::Expr(expr)
        }
    }
}

fn parse_if_stmt(&mut self) -> AstStmt {
    self.advance(); // consume 'if'
    self.expect(TokenKind::OpenParen);
    let cond = self.parse_expr();
    self.expect(TokenKind::CloseParen);
    let then_branch = self.parse_block_or_stmt();

    let else_branch = if self.peek() == Some(&TokenKind::Keyword(Keyword::Else)) {
        self.advance();
        Some(self.parse_block_or_stmt())
    } else {
        None
    };

    AstStmt::If(Box::new(cond), then_branch, else_branch.map(Box::new))
}

Each match arm is clean and self-contained. The compiler enforces exhaustiveness — if we add a new statement kind to the AST, the parse_statement match will fail to compile until we add the corresponding arm.

Pratt Parsing for Expressions

Expressions are where parsing gets interesting. Consider a + b * c: the parser must produce a + (b * c), not (a + b) * c. This is operator precedence. Then there's associativity: a - b - c means (a - b) - c, not a - (b - c).

The classic recursive descent approach uses one function per precedence level (parse_expr, parse_term, parse_factor). This works but is verbose and rigid. Pratt parsing is more elegant: each operator has a binding power (a number), and a single function handles all operators by comparing binding powers.

src/parser.rs — Pratt binding powers
const prefix_bp(kind: &TokenKind) -> Option<u8> {
    match kind {
        TokenKind::Minus => Some(9),   // unary negation
        TokenKind::Bang  => Some(9),   // logical not
        TokenKind::IntLit(_) | TokenKind::Ident(_) | TokenKind::OpenParen => {
            Some(0)  // atoms — consume and return
        }
        _ => None,
    }
}

const infix_bp(kind: &TokenKind) -> Option<(u8, u8)> {
    match kind {
        TokenKind::OrOr    => Some((1, 2)),
        TokenKind::AndAnd  => Some((3, 4)),
        TokenKind::EqEq
        | TokenKind::BangEq => Some((5, 6)),
        TokenKind::Lt
        | TokenKind::Gt
        | TokenKind::LtEq
        | TokenKind::GtEq   => Some((7, 8)),
        TokenKind::Plus
        | TokenKind::Minus  => Some((9, 10)),
        TokenKind::Star
        | TokenKind::Slash
        | TokenKind::Percent => Some((11, 12)),
        TokenKind::OpenParen => Some((13, 14)), // function call
        _ => None,
    }
}

The tuple (left_bp, right_bp) encodes both precedence and associativity. For left-associative operators like +, left_bp < right_bp (9 and 10). For right-associative operators (if we had any, like exponentiation), we'd have left_bp > right_bp. The parse_expr_bp function uses these to decide when to stop recursing:

src/parser.rs — Pratt expression parser
pub fn parse_expr(&mut self) -> AstExpr {
    self.parse_expr_bp(0)
}

fn parse_expr_bp(&mut self, min_bp: u8) -> AstExpr {
    // Parse a prefix expression (atom or unary operator).
    let mut lhs = match self.peek() {
        Some(kind) if prefix_bp(kind).is_some() => {
            let (_, r_bp) = self.parse_prefix()
        }
        _ => {
            self.error("expected expression".to_string());
            AstExpr::Error
        }
    };

    // Parse infix operators while their binding power is high enough.
    loop {
        let kind = match self.peek() {
            Some(k) => k.clone(),
            None => break,
        };

        let (l_bp, r_bp) = match infix_bp(&kind) {
            Some(bp) => bp,
            None => break,
        };

        if l_bp < min_bp {
            break;
        }

        self.advance(); // consume the operator
        let rhs = self.parse_expr_bp(r_bp);
        lhs = self.make_binary(kind, lhs, rhs);
    }

    lhs
}

fn parse_prefix(&mut self) -> (u8, AstExpr) {
    let kind = self.peek().unwrap().clone();
    let r_bp = prefix_bp(&kind).unwrap();
    match kind {
        TokenKind::Minus | TokenKind::Bang => {
            self.advance();
            let expr = self.parse_expr_bp(r_bp);
            (r_bp, AstExpr::Unary(kind, Box::new(expr)))
        }
        TokenKind::IntLit(n) => {
            self.advance();
            (r_bp, AstExpr::Int(n))
        }
        TokenKind::Ident(name) => {
            self.advance();
            (r_bp, AstExpr::Ident(name))
        }
        TokenKind::OpenParen => {
            self.advance();
            let expr = self.parse_expr_bp(0);
            self.expect(TokenKind::CloseParen);
            (r_bp, expr)
        }
        _ => unreachable!(),
    }
}

    | Some kind when prefix_bp kind <> None ->
        let (_, expr) = parse_prefix p in
        expr
    | _ ->
        error p "expected expression";

  in

    | None -> lhs
    | Some kind ->

         | Some (l_bp, r_bp) when l_bp >= min_bp ->
             advance p;
             let rhs = parse_expr_bp p r_bp in
             loop (make_binary kind lhs rhs)
         | _ -> lhs)
  in
  loop lhs

and parse_prefix p =
  let kind = match peek p with Some k -> k | None -> failwith "unreachable" in
  let r_bp = match prefix_bp kind with Some bp -> bp | None -> 0 in
  match kind with
  | Token_Minus | Token_Bang ->
      advance p;
      let expr = parse_expr_bp p r_bp in
      (r_bp, AstExpr_Unary (kind, expr))
  | Token_IntLit n ->
      advance p; (r_bp, AstExpr_Int n)
  | Token_Ident name ->
      advance p; (r_bp, AstExpr_Ident name)
  | Token_OpenParen ->
      advance p;
      let expr = parse_expr_bp p 0 in
      expect p Token_CloseParen;
      (r_bp, expr)
  | _ -> failwith "unreachable"

The algorithm is beautiful in its simplicity:

  1. Parse a prefix expression (atom, parenthesized expression, or unary operator).
  2. Look at the next token. If it's an infix operator with binding power high enough, consume it and parse the right-hand side with the operator's right binding power.
  3. Repeat until no operator has high enough binding power.
Why this is elegant in Rust: The match on TokenKind drives the entire parser. Each arm is short and focused. Adding a new operator means adding a variant to TokenKind, adding an entry to the binding power tables, and (if it's a new prefix form) adding an arm to parse_prefix. The compiler tells you if you miss anything.

Error Recovery

When the parser encounters an error, the worst thing it can do is stop. The programmer wants to see all errors in their file, not just the first one. Error recovery is the art of getting back to a known state and continuing.

Our strategy is straightforward:

  • Sentinel nodes: When parsing fails, produce AstExpr::Error or AstStmt::Error. These are real AST nodes that downstream phases can handle gracefully (e.g., by propagating the error or treating them as the unit type).
  • Synchronization: After an error in a statement, skip tokens until we find a statement boundary (}, ;, keyword). This prevents cascading errors where one mistake causes fifty spurious errors.
  • Collect don't throw: All errors go into diagnostics. The parser never panics.
src/parser.rs — error recovery
fn synchronize(&mut self) {
    while let Some(kind) = self.peek() {
        match kind {
            TokenKind::Semicolon => {
                self.advance();
                return;
            }
            TokenKind::CloseBrace
            | TokenKind::Keyword(Keyword::Func)
            | TokenKind::Keyword(Keyword::Let)
            | TokenKind::Keyword(Keyword::If)
            | TokenKind::Keyword(Keyword::While)
            | TokenKind::Keyword(Keyword::Return) => {
                return;
            }
            _ => {
                self.advance();
            }
        }
    }
}
parser.ml — error recovery
let synchronize p =
  let rec loop () =
    match peek p with
    | None -> ()
    | Some Token_Semicolon -> advance p; ()
    | Some Token_CloseBrace -> ()
    | Some (Token_Keyword (Kw_Func | Kw_Let | Kw_If | Kw_While | Kw_Return)) -> ()
    | Some _ -> advance p; loop ()
  in
  loop ()

When parse_statement encounters an error, it calls synchronize and returns AstStmt::Error. The loop in parse_func_decl continues to the next statement. The loop in parse_program continues to the next function. The programmer gets error messages for every problem in their file, not just the first one.

Chapter 04: The AST

The parser produces structure. What does that structure look like?

What you'll learn in this chapter: The complete abstract syntax tree definition for our language, how Rust enums model tree nodes, why we discard certain syntactic details, and how to visualize and debug ASTs.

The Problem

The parser consumes tokens and produces a tree. But what kind of tree? Every programming language has two potential tree representations:

Concrete Syntax Tree (CST) Abstract Syntax Tree (AST)
Contains Every token: keywords, punctuation, parentheses, semicolons Only semantic nodes: expressions, statements, declarations
Preserves Syntactic details: which kind of parenthesis, exact operator spelling Only the structure and meaning
Used for Formatting, refactoring tools, syntax highlighting Semantic analysis, code generation, optimization

Our compiler uses an AST — we discard details that don't affect meaning. The semicolons, parentheses, and braces are structural hints during parsing, but they don't appear in the AST. What remains is the pure semantics of the program.

Source:         func add(i32, a: i32, b: i32) { return a + b; }

CST (contains everything):
  Keyword(Func) Ident("add") OpenParen Ident("i32") Comma ...

AST (only the meaning):
  FuncDecl("add", return=i32, params=[("a",i32),("b",i32)],
    body=[Return(Binary(+, Ident("a"), Ident("b")))])

The AST in Rust: Enums All the Way Down

In Rust, an AST is a collection of enums. Each enum represents a category of node, and each variant represents a specific kind of node. Data that belongs to a node (like a name or a child node) is carried by the variant's fields.

Why This Design?

Every AST design choice has a rationale. Let's walk through the key decisions:

AstType is an Enum

Types can be simple names (i32, bool), pointers (@T), or arrays ([N]T). By making AstType an enum, we can add more type constructors later (tuples, generics, function types) without changing any code that pattern-matches on types. The compiler will tell us everywhere we need to update.

Box for Recursive Types

Rust needs to know the size of every type at compile time. An enum is as large as its largest variant. If AstExpr contained an AstExpr directly (without boxing), the size would be infinite. Box<AstExpr> stores the child on the heap and keeps the enum size fixed at a pointer width. This is the standard Rust pattern for recursive data structures — the same reason List uses Box in Rust but not in OCaml (OCaml uses uniform representation and implicit boxing).

AstStmt::Error and AstExpr::Error

These sentinel variants are our error recovery mechanism. When the parser encounters an invalid construct, it produces an Error node instead of panicking. Later phases handle Error by propagating it — type checking an error expression yields an error type, code generation for an error statement emits nothing. This keeps the pipeline running even with invalid input.

Choice of Children

Note that AstStmt::Return contains an AstExpr (the value to return), while AstStmt::If contains Box<AstExpr> and Box<AstStmt>. The Box is used when the variant would otherwise contain a recursive type directly (making the enum too large). Return doesn't need Box because Vec<AstStmt> and String in sibling variants are already large enough that the single AstExpr doesn't make the enum bigger.

Enum sizing trick: You can check enum sizes with std::mem::size_of::<AstExpr>(). In practice, Rust's enum layout optimization means that variants with a single non-nullable pointer field (like Box) may be smaller than you expect. But the rule is simple: wrap recursive variants in Box to be safe and clear.

The AST and the Borrow Checker

One of the most elegant properties of the AST in Rust is that it enforces tree shape through ownership. Every node has exactly one parent. The Box pointer ensures exclusive ownership. There are no cycles, no shared references, no dangling pointers.

This is exactly what the borrow checker verifies. When the semantic analyzer walks the AST, it can traverse with references:

fn analyze_expr(&self, expr: &AstExpr) -> Type {
    match expr {
        AstExpr::Ident(name) => self.lookup_type(name),
        AstExpr::Binary { left, right, .. } => {
            let lt = self.analyze_expr(left);
            let rt = self.analyze_expr(right);
            // ... unify lt and rt
        }
    }
}

No mutex, no GC, no reference counting. Just immutable references into a tree with guaranteed lifetime. If the tree is alive, the references are valid. The borrow checker proves it.

When you lower the AST to an IR, you consume the tree by value, moving nodes out:

fn lower_func(self, func: &mut AstFuncDecl) -> IrFunc {
    let body = std::mem::take(&mut func.body);
    // body is now empty Vec, we own the original statements
}

The pipeline is linear: lex → parse → analyze → lower → codegen. Data moves forward. Old representations are dropped and freed. There's no tracing through old token streams or half-analyzed ASTs. Rust's ownership model makes the pipeline's structure explicit in the type system.

5. Does This Make Sense?

The parser gave us a tree. A beautiful, recursive, well-typed AST. But is it meaningful?

Consider this program:

func main(i32) {
    let x: bool = 42;
    return x;
}

The parser accepts this without complaint. It is syntactically valid. But it is semantically meaningless — we are storing an integer in a boolean variable, and returning a boolean from a function declared to return i32. The tree is well-formed, but the program is nonsense.

This chapter is about catching that nonsense. We will build two passes over the AST:

  • Phase 1 — Name resolution (symbol table): Walk all declarations, register every name, determine what each identifier refers to.
  • Phase 2 — Type checking: Walk every expression, compute its type, verify it matches the expected type.

Think of it like Rust's own name resolution. Before Rust checks whether x.foo() is a valid method call, it needs to know what x is, what type it has, and what foo means in that context. We do the same: resolve names first, then check types.

The Symbol Table

A symbol table maps names to their definitions. In a language with nested scopes, we need a stack of scopes. When we enter a block, we push a new scope. When we leave, we pop it. Looking up a name searches from innermost to outermost scope — exactly like Rust's path resolution.

Our symbol table supports three kinds of entries:

  • Global variables — declared at the top level of the program.
  • Functions — also at the top level, with parameter names in a nested scope.
  • Local variables — declared inside function bodies with let.

Here is the data structure:

src/symbol.rs
use std::collections::HashMap;

pub enum SymbolKind {
    GlobalVar(Type),
    Func {
        ret_type: Type,
        param_types: Vec<Type>,
    },
    LocalVar(Type),
}

pub struct SymbolTable {
    scopes: Vec<HashMap<String, SymbolKind>>,
}

impl SymbolTable {
    pub fn new() -> Self {
        Self { scopes: vec![HashMap::new()] }
    }

    pub fn push_scope(&mut self) {
        self.scopes.push(HashMap::new());
    }

    pub fn pop_scope(&mut self) {
        self.scopes.pop();
    }

    pub fn insert(&mut self, name: &str, kind: SymbolKind) {
        self.scopes.last_mut().unwrap().insert(name.to_string(), kind);
    }

    pub fn lookup(&self, name: &str) -> Option<&SymbolKind> {
        for scope in self.scopes.iter().rev() {
            if let Some(entry) = scope.get(name) {
                return Some(entry);
            }
        }
        None
    }
}

Populating the Symbol Table

We walk the AST in two passes. The first pass registers all top-level declarations (functions and global variables). The second pass visits each function body, pushing a new scope for the function's parameters, then another for each nested block.

                }

            }
        }

Type Checking

The type checker is the heart of semantic analysis. It walks the AST and assigns a type to every expression, verifying consistency at each step. Every expression has a type — the type checker discovers it. This is like Rust's type inference, but simpler: we do not infer types from usage, we verify that the declared and actual types match.

The checker returns either a type (for expressions) or unit (for statements), or an error. Here is the core function signature:

src/check.rs
pub enum TypeResult {
    Type(Type),
    Error,
}

fn check_expr(
    expr: &Expr,
    sym: &SymbolTable,
    ret_type: &Type,
) -> TypeResult {
    match expr {
        Expr::Int(_) => TypeResult::Type(Type::Prim("i32".into())),
        Expr::Bool(_) => TypeResult::Type(Type::Prim("bool".into())),
        Expr::Ident(name) => check_ident(name, sym),
        Expr::Binary { op, lhs, rhs } => check_binary(op, lhs, rhs, sym, ret_type),
        Expr::Unary { op, rhs } => check_unary(op, rhs, sym, ret_type),
        Expr::Call { callee, args } => check_call(callee, args, sym, ret_type),
        Expr::Cast { expr, target } => check_cast(expr, target, sym, ret_type),
        Expr::Assign { target, value } => check_assign(target, value, sym, ret_type),
    }
}

Checking Binary Operators

fn check_binary(
    op: &BinaryOp,
    lhs: &Expr,
    rhs: &Expr,
    sym: &SymbolTable,
    ret_type: &Type,
) -> TypeResult {
    let lt = check_expr(lhs, sym, ret_type);
    let rt = check_expr(rhs, sym, ret_type);

    let (lt, rt) = match (lt, rt) {
        (TypeResult::Type(l), TypeResult::Type(r)) => (l, r),
        _ => return TypeResult::Error,
    };

    match op {
        BinaryOp::Add | BinaryOp::Sub | BinaryOp::Mul | BinaryOp::Div => {
            expect_type(&lt, &Type::Prim("i32"), "arithmetic operand");
            expect_type(&rt, &Type::Prim("i32"), "arithmetic operand");
            TypeResult::Type(Type::Prim("i32"))
        }
        BinaryOp::Eq | BinaryOp::Neq => {
            expect_same_type(&lt, &rt);
            TypeResult::Type(Type::Prim("bool"))
        }
        BinaryOp::Lt | BinaryOp::Gt | BinaryOp::Le | BinaryOp::Ge => {
            expect_type(&lt, &Type::Prim("i32"), "comparison operand");
            expect_type(&rt, &Type::Prim("i32"), "comparison operand");
            TypeResult::Type(Type::Prim("bool"))
        }
    }

Notice the structure: we recurse into both sub-expressions, collect their types, then check consistency against the operator's rules. If either sub-expression already had an error, we propagate Error without additional noise. This is the same pattern Rust's compiler uses — collect errors eagerly, report them gracefully.

6. The Bridge: Intermediate Representation

The AST is source-language-shaped. It has if nodes, while loops, return statements — all the structure of our source language. The CPU does not know about any of these. The CPU knows about registers, memory addresses, arithmetic operations, and jumps.

We need something in between. An Intermediate Representation (IR) that is lower-level than the AST but higher-level than assembly. Rust has HIR (High-level IR) and MIR (Mid-level IR) for exactly this purpose. The Rust compiler desugars your Rust code into HIR, then simplifies further into MIR, which is used for borrow checking and optimization before code generation.

Our IR is similar in spirit to MIR: a flat list of instructions organized into basic blocks, operating on virtual registers in Static Single Assignment (SSA) form.

What is SSA?

Static Single Assignment means every variable is assigned exactly once. If you assign to the same variable twice in the source, the IR creates two different virtual registers — one for each assignment. This is like Rust's immutability-by-default, but taken to the extreme:

Source: SSA IR: x = 1 %0 = const 1 x = x + 2 %1 = const 2 %2 = add %0, %1 return x ret %2 (x is reassigned, (every %reg is defined once, not the same x!) no reassignment!)

Why SSA? Because it makes optimizations simpler — like Rust's borrow checker makes memory safety simpler. In SSA form, every use of a value points to exactly one definition. There is no need for reaching-definition analysis because the definition is baked into the name. Constant folding, dead code elimination, and many other optimizations become trivial traversals.

SSA and Rust's ownership There is a deep connection between SSA and Rust's ownership model. In both systems, every value has a single "home." In Rust, ownership ensures one binding at a time. In SSA, the IR ensures one definition per register. Both eliminate classes of bugs at the representation level rather than through analysis passes.

IR Design

Our IR has three core concepts: virtual registers (VRegs), instructions, and basic blocks.

A virtual register is an abstract temporary — like a Rust local variable but unbounded in number. We give them names like %0, %1, etc. During code generation, the register allocator maps them to real x86-64 registers or stack slots.

A basic block is a straight-line sequence of instructions with exactly one entry point (the top) and one exit point (the bottom, which is a terminator like jmp, br, or ret).

An instruction is a single operation: add, sub, load, store, const, call, etc. Each instruction defines (at most) one virtual register.

Building SSA IR Directly

The classic approach to SSA is: build a non-SSA IR first, then run a separate pass to convert it to SSA form. That is wasteful. Instead, we use the algorithm from Braun et al. (2013) — "Simple and Efficient Construction of Static Single Assignment Form" — which builds SSA directly during IR construction.

The algorithm works like this:

  1. Each AST variable maps to a stack of VRegs. Every time we see an assignment to a variable, we push a new VReg onto its stack. This gives us the "single assignment" property — each assignment creates a new virtual register.
  2. When we use a variable, look up its current VReg. Pop the most recent VReg from the variable's stack. That is the definition that reaches this use.
  3. If the variable is defined in multiple predecessors, insert a Phi node. When we reach a join point in the control flow (after an if statement), the variable might have different values depending on which branch we came from. The Phi node selects the correct value based on the incoming edge.
  4. Blocks start unsealed. A block is "sealed" when all its predecessors are known. Before sealing, we do not know every path that might reach the block. We defer Phi insertion until the block is sealed and all incoming values are known.

Here is the core data structure for the SSA builder:

src/ir_build.rs
use std::collections::HashMap;

struct SSABuilder {
    // Every AST variable has a stack of VRegs
    // Each assignment pushes a new VReg
    stacks: HashMap<String, Vec<VReg>>,

    // Phi instructions we still need to fill in
    incomplete_phis: HashMap<(BlockId, String), VReg>,

    // Map from BlockId → whether all predecessors are known
    sealed: Vec<bool>,

    // Next VReg ID to allocate
    next_vreg: usize,
}

impl SSABuilder {
    fn new() -> Self {
        Self {
            stacks: HashMap::new(),
            incomplete_phis: HashMap::new(),
            sealed: Vec::new(),
            next_vreg: 0,
        }
    }

    fn fresh_vreg(&mut self) -> VReg {
        let v = self.next_vreg;
        self.next_vreg += 1;
        v
    }

    /// Read the current value of a variable.
    /// Returns the VReg that holds its most recent definition.
    fn read_var(&mut self, name: &str, block: BlockId) -> VReg {
        if let Some(stack) = self.stacks.get(name) {
            if let Some(&vreg) = stack.last() {
                return vreg;
            }
        }
        // Variable not defined yet in this block —
        // try to add a Phi or read from a predecessor
        self.read_var_rec(name, block)
    }

    /// Write a new value for a variable.
    /// Pushes a fresh VReg onto its stack.
    fn write_var(&mut self, name: &str) -> VReg {
        let vreg = self.fresh_vreg();
        self.stacks.entry(name.to_string())
            .or_default()
            .push(vreg);
        vreg
    }
}

func max: b1 (entry): %0 = const 1 (type constant — size hint) %1 = const 0 (type constant) %2 = gt %0, %1 (a > b) br %2, b2, b3 b2 (then): ret %0 (return a) b3 (else): ret %1 (return b)

Wait — we need to load a and b from somewhere. In our IR, function parameters are VRegs that enter the entry block. The gt instruction compares the two parameter VRegs. If the condition is true, we branch to b2 and return a's VReg. Otherwise, we branch to b3 and return b's VReg.

The Phi nodes appear when a variable is defined in multiple predecessors. Consider this variant where we compute a result and return it at the end:

func max(i32, a: i32, b: i32) {
    let result: i32;
    if (a > b) {
        result = a;
    } else {
        result = b;
    }
    return result;
}
func max: b1 (entry): %0 = const 1 (a as VReg) %1 = const 0 (b as VReg) %2 = gt %0, %1 br %2, b2, b3 b2 (then): store %0, result_slot jmp b4 b3 (else): store %1, result_slot jmp b4 b4 (merge): %3 = phi [b2: %0, b3: %1] ret %3

The Phi node %3 = phi [b2: %0, b3: %1] says: if we came from block b2, %3 has the value of %0 (which is a); if we came from block b3, %3 has the value of %1 (which is b). The Phi is purely conceptual — during code generation, it will be resolved into moves at the end of each predecessor block.

Why not just use memory? We could store result to a stack slot in each branch and load it after the join. That works, but it defeats optimization. The Phi node tells the optimizer: "this value comes from one of two places." The optimizer can propagate both possibilities and potentially eliminate the merge entirely.

Optimization Passes

The SSA IR is not just a bridge to code generation — it is a platform for optimization. Because every VReg has exactly one definition, we can analyze and transform the IR with simple passes.

Constant Folding

Constant folding evaluates constant expressions at compile time. If we have const %0, 3; const %1, 4; add %2, %0, %1, we replace it with const %2, 7. This is like Rust's const evaluation — the computation moves from runtime to compile time.

Here is the constant folding pass — a 30-line traversal:

src/iropt.rs
use std::collections::HashMap;

pub fn constant_fold(func: &mut Function) {
    let mut values: HashMap<VReg, i64> = HashMap::new();

    for block in &mut func.blocks {
        for instr in &mut block.instrs {
            match instr {
                Instr::Const(vreg, n) => {
                    values.insert(*vreg, *n);
                }
                Instr::Add(dst, l, r) => {
                    if let (Some(a), Some(b)) = (values.get(l), values.get(r)) {
                        let result = a.wrapping_add(*b);
                        values.insert(*dst, result);
                        *instr = Instr::Const(*dst, result);
                    }
                }
                Instr::Sub(dst, l, r) => {
                    if let (Some(a), Some(b)) = (values.get(l), values.get(r)) {
                        let result = a.wrapping_sub(*b);
                        values.insert(*dst, result);
                        *instr = Instr::Const(*dst, result);
                    }
                }
                // ... Mul, Div, comparisons, etc.
                _ => {}
            }
        }
    }
}

Dead Code Elimination

An instruction is dead if its result VReg is never used. In SSA form, dead code elimination is trivial: count the uses of each VReg. If a definition has zero uses, remove it. This is like Rust's dead code warnings — the compiler tells you about values you compute but never use.

src/iropt.rs
pub fn dead_code_elimination(func: &mut Function) {
    // Count uses of every VReg
    let mut use_count: HashMap<VReg, usize> = HashMap::new();

    for block in &func.blocks {
        for instr in &block.instrs {
            for &vreg in used_vregs(instr) {
                *use_count.entry(vreg).or_insert(0) += 1;
            }
        }
    }

    // Remove instructions whose result is never used
    for block in &mut func.blocks {
        block.instrs.retain(|instr| {
            if let Some(&dst) = def_vreg(instr) {
                use_count.get(&dst).unwrap_or(&0) > &0
                    || has_side_effect(instr)
            } else {
                true  // instructions without defs (e.g., Store) are never dead
            }
        });
    }
}

Algebraic Simplification

Algebraic simplification replaces expensive operations with cheaper ones:

PatternReplacementRationale
add %x, 0%xAdding zero is identity
sub %x, 0%xSubtracting zero is identity
mul %x, 1%xMultiplying by one is identity
mul %x, 0const 0Multiplying by zero is zero

These are all simple pattern-matches on the instruction. In Rust, we can implement them as a match inside a traversal. The elegance of SSA is that these patterns are purely local — they depend only on the current instruction and the instructions that define its operands, never on distant code.

Optimization pipelines Like Rust's rustc, our compiler should run optimizations in a pipeline. Run constant folding first (it creates more constants), then algebraic simplification (clean up the leftovers), then dead code elimination (remove instructions made redundant by the previous passes). Run the pipeline multiple times until no further changes occur — this is called "fixpoint iteration."

What the IR Enables

The IR is the turning point of the compiler. Everything before it is frontend: source-language-specific. Everything after is backend: target-machine-specific. The IR is the interface between them. Changing the frontend language does not change the backend. Changing the target architecture does not change the frontend.

This is the same architectural insight that powers LLVM. LLVM IR is the common representation shared by all LLVM frontends (C, C++, Rust, Swift, ...) and all LLVM backends (x86, ARM, RISC-V, ...). Our IR is simpler, but the idea is the same.

Let us look at the IR for a program with multiple functions, nested control flow, and variable reassignment:

func factorial(i32, n: i32) {
    let result: i32 = 1;
    while (n > 0) {
        result = result * n;
        n = n - 1;
    }
    return result;
}

This program has a loop, variable reassignment (n and result are mutated), and a return at the end. Here is the SSA IR our compiler generates:

func factorial: b1 (entry): %0 = const 1 ; result = 1 %1 = gt %n, 0 ; n > 0? br %1, b2, b4 ; while condition check b2 (loop body): %r1 = phi [b1: %0, b3: %r3] ; result — comes from entry or loop back edge %n1 = phi [b1: %n, b3: %n2] ; n — same %r2 = mul %r1, %n1 ; result * n %n2 = sub %n1, 1 ; n - 1 %c2 = gt %n2, 0 ; n > 0 again br %c2, b3, b4 ; continue? b3 (loop continue): jmp b2 ; back to loop header b4 (exit): %r_final = phi [b1: %0, b2: %r2] ; final result value ret %r_final

Observe the Phi nodes:

  • %r1 merges %0 (initial result = 1 from the entry block) with %r3 (the updated result * n from the loop back edge).
  • %n1 merges the initial parameter %n with the decremented %n2 from the back edge.
  • %r_final merges the initial %0 (if the loop never executes) with %r2 (if the loop executed at least once).

This is the power of SSA: every use of a variable points to exactly one definition. The Phi nodes make the data flow explicit at join points. And because each reassignment creates a new VReg, the optimizer can track each value independently.

7. From IR to Machine Code

We have SSA IR. Clean, well-formed, optimized. But the CPU does not understand virtual registers or basic blocks. The CPU understands x86-64 instructions: mov, add, cmp, jmp, call, ret. Our job in this chapter is to turn IR into real x86-64 assembly.

This is the most satisfying phase of the compiler. Everything before was setup. Now we produce something that actually runs on hardware.

There are three problems to solve:

  1. Register allocation: We have infinite virtual registers. The CPU has 16 (or fewer). Map virtual registers to real registers or stack slots.
  2. Instruction selection: Each IR instruction maps to one or more x86-64 instructions.
  3. Phi resolution: Phi nodes are conceptual — in machine code, they become moves at the end of predecessor blocks.

Register Allocation: Linear Scan

Think of register allocation like Rust's allocator: we have a fixed resource (registers) and competing demands (virtual registers). Every virtual register has a live interval — the range of instructions where its value is needed. If two virtual registers have overlapping live intervals, they cannot share a register. If they do not overlap, they can.

Linear scan is the simplest practical register allocation algorithm. It is not optimal (graph coloring gives better results), but it is fast, simple, and good enough for our compiler. LLVM uses linear scan as a fast path and graph coloring for optimization.

The algorithm:

  1. Compute live intervals for every virtual register.
  2. Sort by start point.
  3. Scan: maintain a set of active intervals (those that are live at the current point). When an interval ends, free its register. When a new interval starts, assign it a register. If none is free, spill — store the value to the stack and reload it when needed.
Live intervals: %0: [1, 3] ──── %1: [2, 5] ────── %2: [4, 6] ──── %3: [7, 8] ── Registers available: r0, r1, r2 Scan: instr 1: %0 starts → assign r0 instr 2: %1 starts → assign r1 instr 4: %2 starts → assign r2 (all regs used, but one ends soon) instr 6: %2 ends → free r2 instr 7: %3 starts → assign r2 (reuse!)

Here is the core allocator in Rust:

src/regalloc.rs
pub struct LiveInterval {
    pub vreg: VReg,
    pub start: usize,
    pub end: usize,
}

pub fn linear_scan(
    intervals: &mut [LiveInterval],
    regs: usize,
) -> HashMap<VReg, Alloc> {
    intervals.sort_by_key(|iv| iv.start);

    let mut active: Vec<usize> = Vec::new();      // indices into intervals
    let mut free_regs: Vec<usize> = (0..regs).collect(); // available registers
    let mut result = HashMap::new();
    let mut stack_slot = regs;  // next stack slot for spilled values

    for i in 0..intervals.len() {
        // Expire dead intervals
        active.retain(|&j| {
            if intervals[j].end <= intervals[i].start {
                free_regs.push(j);   // free the register
                false
            } else {
                true
            }
        });

        if let Some(reg) = free_regs.pop() {
            result.insert(intervals[i].vreg, Alloc::Reg(reg));
            active.push(i);
        } else {
            // No free register — spill to stack
            let slot = stack_slot;
            stack_slot += 1;
            result.insert(intervals[i].vreg, Alloc::Stack(slot));
        }
    }

    result
}

    block: &BasicBlock,
    alloc: &HashMap<VReg, Alloc>,

    asm.emit_label(&format!(".L{}", block.id));

    for instr in &block.instrs {

                asm.emit(&format!("    mov rax, {}", rv));

            emit_epilogue(asm);

        Terminator::Jump(target) => {
            asm.emit(&format!("    jmp .L{}", target));

            asm.emit(&format!("    cmp {}, 1", rc));
            asm.emit(&format!("    je .L{}", t));
            asm.emit(&format!("    jmp .L{}", f));
        }
    }

Phi Resolution

Phi nodes do not exist in x86-64. They are a conceptual tool for the optimizer. In machine code, we resolve them by inserting move instructions at the end of each predecessor block:

IR with Phi: b2 (then): jmp b4 b3 (else): jmp b4 b4 (merge): %3 = phi [b2: %0, b3: %1] ret %3 Generated assembly: b2: mov rax, r0 ; move %0 into rax (return value) jmp b4 b3: mov rax, r1 ; move %1 into rax jmp b4 b4: ; %3 is now in rax (no Phi needed) ret

The key insight: a Phi node is not a real instruction. It is a statement about data flow. When we generate code, we resolve it by copying the incoming value at the end of each predecessor into the register or stack slot that holds the Phi's result. If multiple Phis exist, we may need temporary registers to avoid clobbering (the "swap problem"), but for our simple compiler, sequential moves work fine.

Calling Convention in Detail

The System V AMD64 ABI specifies exactly how functions call each other. Here are the rules our code generator must follow:

Argument passing: Integer and pointer arguments are passed in registers, left to right: rdi, rsi, rdx, rcx, r8, r9. Additional arguments are pushed onto the stack in reverse order.

Return value: Integer and pointer return values go in rax. Our function main(i32) returns an i32 — the generated assembly loads the return value into rax before ret.

Callee-saved registers: rbx, rbp, and r12–r15 must be preserved across function calls. If our code generator uses these registers, it must save and restore them. Our simple allocator avoids this by using scratch registers and stack slots.

Stack alignment: The stack must be 16-byte aligned before a call instruction. Our prologue push rbp (8 bytes) + sub rsp, N (with N aligned to 16) maintains this invariant.

Here is how a function call looks in the generated assembly:

;; Call: result = add(3, 4);
    mov rdi, 3           ; first argument
    mov rsi, 4           ; second argument
    call add             ; push return addr, jump to add
    mov [rbp-8], rax    ; store return value to result

The callee (add) receives rdi=3 and rsi=4. It computes the result, leaves it in rax, and returns. The caller reads rax and stores it. This is the entire calling convention — everything else is mechanical.

Error Handling

Our error handling philosophy is the same as Rust's: collect all errors, present them together, and let the user fix them in one batch. Like cargo, we never stop at the first error. We keep going to find as many as possible.

Here is the driver code:

src/main.rs
use std::process;

fn main() {
    let args: Vec<String> = std::env::args().collect();

    if args.len() < 3 || args[1] != "-o" {
        eprintln!("Usage: l1s input.is -o output");
        process::exit(1);
    }

    let input_path = &args[2];
    let output_path = &args[3];

    // Phase 1: Read and lex
    let source = match std::fs::read_to_string(input_path) {
        Ok(s) => s,
        Err(e) => {
            eprintln!("error: could not read '{}': {}", input_path, e);
            process::exit(1);
        }
    };

    // Phase 2: Lex
    let (tokens, lex_errors) = lexer::tokenize(&source);
    report(&lex_errors);
    if has_errors(&lex_errors) { process::exit(1); }

    // Phase 3: Parse
    let (ast, parse_errors) = parser::parse(&tokens);
    report(&parse_errors);
    if has_errors(&parse_errors) { process::exit(1); }

    // Phase 4: Symbol table + Type check
    let sym = symbol::build(&ast);
    let type_errors = check::typeck(&ast, &sym);
    report(&type_errors);
    if has_errors(&type_errors) { process::exit(1); }

    // Phase 5: IR build
    let mut ir = ir_build::build(&ast, &sym);

    // Phase 6: Optimize
    iropt::optimize(&mut ir);

    // Phase 7: Register allocation
    let alloc = regalloc::assign(&ir);

    // Phase 8: Code generation
    let asm_path = format!("{}.asm", output_path);
    let asm = codegen::generate(&ir, &alloc);
    std::fs::write(&asm_path, &asm).unwrap();

    // Phase 9: Assemble and link
    let status = process::Command::new("nasm")
        .args(&["-f", "elf64", &asm_path, "-o", &format!("{}.o", output_path)])
        .status()
        .expect("failed to run nasm");

    if !status.success() {
        eprintln!("error: assembler failed");
        process::exit(1);
    }

    let status = process::Command::new("cc")
        .args(&["-o", output_path, &format!("{}.o", output_path), "-no-pie"])
        .status()
        .expect("failed to run cc");

    if !status.success() {
        eprintln!("error: linker failed");
        process::exit(1);
    }
}
main.ml
let () =
  let args = Array.to_list Sys.argv in
  let input, output = match args with
    | _ :: input :: "-o" :: output :: [] -> input, output
    | _ ->
        Printf.eprintf "Usage: l1s input.is -o output\n";
        exit 1
  in

  (* Phase 1: Read *)
  let source = match In_channel.read_all input with
    | s -> s
    | exception exn ->
        Printf.eprintf "error: could not read '%s': %s\n" input (Printexc.to_string exn);
        exit 1
  in

  (* Phase 2: Lex *)
  let tokens, lex_errs = Lexer.tokenize source in
  report lex_errs;
  if has_errors lex_errs then exit 1;

  (* Phase 3: Parse *)
  let ast, parse_errs = Parser.parse tokens in
  report parse_errs;
  if has_errors parse_errs then exit 1;

  (* Phase 4: Type check *)
  let sym = Symbol.build ast in
  let tc_errs = Check.typeck ast sym in
  report tc_errs;
  if has_errors tc_errs then exit 1;

  (* Phase 5-7: IR, opt, regalloc *)
  let ir = Ir_build.build ast sym in
  let ir = Iropt.optimize ir in
  let alloc = Regalloc.assign ir in

  (* Phase 8: Codegen *)
  let asm_path = Printf.sprintf "%s.asm" output in
  let asm = Codegen.generate ir alloc in
  Out_channel.write_all asm_path asm;

  (* Phase 9: Assemble and link *)
  let _ = Sys.command (Printf.sprintf
    "nasm -f elf64 %s -o %s.o && cc -o %s %s.o -no-pie"
    asm_path output output output)
  in
  ()
Why -no-pie? By default, modern GCC produces Position Independent Executables (PIE). Our compiler generates absolute addresses in the assembly (not PIC), so we need -no-pie to tell the linker not to assume position-independence. If we wanted PIC, we would need to generate RIP-relative addressing — a good exercise.

Testing

Every phase of the compiler should be tested independently. We use a three-tier testing strategy:

Unit tests live alongside each module and test individual functions. The lexer tests verify that "42" produces Token::Num(42). The parser tests verify that "return 1 + 2;" produces the correct AST. The type checker tests verify that type mismatches are caught.

Integration tests compile complete programs and run them, checking the exit code. This is the most important test category — it verifies the entire pipeline works end-to-end.

What the Driver Enables

The driver is the user's interface to the compiler. It is the part that makes the compiler usable. With it, we can:

  • Compile programs from files by running a single command.
  • See detailed errors with source locations and descriptions.
  • Debug the compiler by dumping intermediate representations.
  • Run a test suite automatically with integration tests.
  • Integrate with build systems like Make or Cargo.

The driver is also the place where we handle command-line arguments, file paths, and system calls. It is the only part of the compiler that interacts with the operating system directly. Everything else is pure data transformation — functions that take input and produce output with no side effects. This separation of concerns is what makes the compiler testable and maintainable.

Test-driven compiler development workflow The fastest way to build a compiler is TDD: (1) Write a test for the feature you want. (2) Run it — it fails because the feature does not exist yet. (3) Implement the feature in the smallest possible change. (4) Run the test — it passes. (5) Commit. Every test is a specification. Every passing test is proof of progress. This is how rustc itself is developed — the Rust test suite has over 100,000 tests.

Ideas for the Reader

So where do you go from here? Here are concrete projects, ordered from easiest to most ambitious:

Add one feature at a time. The best advice: add the thing you miss most. Is it frustrating that you cannot print output? Add a print function that calls C's printf via an extern declaration. Do you want to write recursive functions? Make sure the calling convention handles it correctly. Does the one-pass compilation annoy you? Add a REPL.

Port to a new backend. Our compiler targets x86-64, but the concepts transfer directly. ARM64 uses different register names but the same patterns. RISC-V is the cleanest ISA ever designed — only 47 instructions. WebAssembly is a stack machine — instead of registers, push and pop values. LLVM IR lets us piggyback on LLVM's optimizers and backends, giving us every architecture LLVM supports.

Write an optimization pass. Constant folding is the easiest and most immediately rewarding. Watch your generated code shrink from dozens of instructions to a single mov eax, const. Then add dead code elimination, then copy propagation, then algebraic simplification. Each pass is 20–50 lines and each one makes your code measurably better.

Add proper register allocation. Replace spill-everything with linear scan — or go further and implement graph coloring. The difference in generated code quality is dramatic. Programs that spill everything run 10x slower than programs that keep values in registers.

Self-host. Write the compiler in your language. The compiler we have is simple enough that a determined programmer could translate the Rust or OCaml source into our language in a few weeks. Then compile that source with our existing compiler, and the new compiler can compile itself. This is the final boss of compiler projects.

A Concrete Roadmap: 10 Projects

If you want a structured path forward, here is a sequence of 10 projects that will take you from our minimal compiler to something substantially more capable:

  1. Add a standard library. Implement print_i32, print_bool, read_i32 as extern functions that call C's printf and scanf. This makes programs interactive and testable beyond exit codes.
  2. Add string literals. Define a string type (@u8 — pointer to immutable byte array), store string data in the read-only data section (.rodata), and teach the type checker about pointer types. Strings unlock meaningful output.
  3. Add fixed-size arrays. Support [N]Type syntax. Stack-allocate them. Add bounds checks in the IR or codegen. Arrays unlock real data structures.
  4. Add compound assignment operators. +=, -=, *=, etc. Desugar to x = x + y in a pre-IR pass. Small change, big ergonomic improvement.
  5. Add for loops. Syntactic sugar over while. Desugar in the parser: for (i = 0; i < n; i = i + 1) { body } becomes a while with initialization, condition, and increment.
  6. Write an optimization pipeline. Start with constant folding (easiest), then dead code elimination, then copy propagation, then algebraic simplification. Run them in a loop until fixpoint.
  7. Replace spill-everything with linear scan. Compute live intervals, allocate registers, spill only when necessary. This is the single biggest performance improvement you can make.
  8. Add a REPL. Read-eval-print loop with dynamic code generation. Read a line, compile it to a shared object, dlopen it, call the function. This is the most fun project on this list.
  9. Port to ARM64. Rewrite just the codegen module (~300 lines). ARM64 has 32 registers, a load-store architecture, and no x86-64 legacy baggage. You will deeply understand architecture independence.
  10. Self-host. Write the compiler in your language. Bootstrap it through your existing compiler. Fix the bugs. Iterate. When your new compiler can compile itself, you have achieved compiler nirvana.

The Architecture of the Compiler: A Retrospective

Looking back at the pipeline, one pattern stands out: every stage is a function. Each stage takes input of one type and produces output of another. There are no side effects, no global state, no hidden dependencies between stages. This is not an accident — it is a design principle we followed from the start, and it is the same principle that makes Rust's compiler (and most production compilers) modular and testable.

Source ──► Tokens ──► AST ──► IR ──► Assembly ──► Executable lexer parser typeck codegen nasm + cc

Each arrow is a pure transformation. The interfaces are:

  • fn lex(source: &str) -> (Vec<Token>, Vec<Diag>)
  • fn parse(tokens: &[Token]) -> (Program, Vec<Diag>)
  • fn typeck(ast: &Program, sym: &SymbolTable) -> Vec<Diag>
  • fn build_ir(ast: &Program, sym: &SymbolTable) -> Function
  • fn codegen(ir: &Function, alloc: &Allocation) -> String

If you ever build a larger compiler, preserve this property. Test each stage independently. Compose them in the driver. Never let one stage reach into another stage's internals. The day you want to replace the code generator (to target ARM64, for example), you will be grateful that the IR cleanly separates frontend from backend.

Why Two Implementations?

Throughout this book, we presented every piece of code in Rust and OCaml. Why double the work?

First, because seeing the same algorithm in two languages clarifies what is essential versus what is incidental. The pattern-matching structure of a type checker is the same in Rust and OCaml — that is essential. The exact syntax for declaring a recursive enum — that is incidental. When you understand both, you understand the concept, not just the syntax.

Second, because Rust and OCaml represent two different points in the design space of systems programming. Rust gives you control over memory layout, zero-cost abstractions, and compile-time safety guarantees via the borrow checker. OCaml gives you rapid prototyping, garbage-collected ease, and the most elegant pattern matching in any industrial language. Both are excellent for compiler construction — but for different reasons.

Third, because building the same thing twice is the best way to learn. The first time, you are figuring out the problem. The second time, you are refining your understanding. The Rust version taught us about ownership and lifetimes in tree-structured data — every Box<Expr> is a reminder that the compiler has finite memory. The OCaml version taught us about recursive algebraic types and functional transformations — the elegance of expressing tree rewrites as pure functions.

If you only read one implementation, read the one in the language you are less familiar with. If you know Rust, read the OCaml code — the pattern matching will feel liberating. If you know OCaml, read the Rust code — the explicitness about memory management will feel enlightening.
