Building a Compiler Rust Edition

5. Does This Make Sense?

The parser gave us a tree. A beautiful, recursive, well-typed AST. But is it meaningful?

Consider this program:

func main(i32) {
    let x: bool = 42;
    return x;
}

The parser accepts this without complaint. It is syntactically valid. But it is semantically meaningless — we are storing an integer in a boolean variable, and returning a boolean from a function declared to return i32. The tree is well-formed, but the program is nonsense.

This chapter is about catching that nonsense. We will build two passes over the AST:

  • Phase 1 — Name resolution (symbol table): Walk all declarations, register every name, determine what each identifier refers to.
  • Phase 2 — Type checking: Walk every expression, compute its type, verify it matches the expected type.

Think of it like Rust's own name resolution. Before Rust checks whether x.foo() is a valid method call, it needs to know what x is, what type it has, and what foo means in that context. We do the same: resolve names first, then check types.

The Symbol Table

A symbol table maps names to their definitions. In a language with nested scopes, we need a stack of scopes. When we enter a block, we push a new scope. When we leave, we pop it. Looking up a name searches from innermost to outermost scope — exactly like Rust's path resolution.

Our symbol table supports three kinds of entries:

  • Global variables — declared at the top level of the program.
  • Functions — also at the top level, with parameter names in a nested scope.
  • Local variables — declared inside function bodies with let.

Here is the data structure:

src/symbol.rs
use std::collections::HashMap;

pub enum SymbolKind {
    GlobalVar(Type),
    Func {
        ret_type: Type,
        param_types: Vec<Type>,
    },
    LocalVar(Type),
}

pub struct SymbolTable {
    scopes: Vec<HashMap<String, SymbolKind>>,
}

impl SymbolTable {
    pub fn new() -> Self {
        Self { scopes: vec![HashMap::new()] }
    }

    pub fn push_scope(&mut self) {
        self.scopes.push(HashMap::new());
    }

    pub fn pop_scope(&mut self) {
        self.scopes.pop();
    }

    pub fn insert(&mut self, name: &str, kind: SymbolKind) {
        self.scopes.last_mut().unwrap().insert(name.to_string(), kind);
    }

    pub fn lookup(&self, name: &str) -> Option<&SymbolKind> {
        for scope in self.scopes.iter().rev() {
            if let Some(entry) = scope.get(name) {
                return Some(entry);
            }
        }
        None
    }
}
symbol.ml
type symbol_kind =
  | GlobalVar of typ
  | Func of { ret_type: typ; param_types: typ list }
  | LocalVar of typ

type symbol_table = {
  mutable scopes: (string, symbol_kind) Hashtbl.t list;
}

let create () = {
  scopes = [Hashtbl.create 16];
}

let push_scope tbl =
  tbl.scopes <- Hashtbl.create 16 :: tbl.scopes

let pop_scope tbl =
  match tbl.scopes with
  | _ :: rest -> tbl.scopes <- rest
  | [] -> failwith "scope underflow"

let insert tbl name kind =
  match tbl.scopes with
  | h :: _ -> Hashtbl.replace h name kind
  | [] -> failwith "no scope"

let lookup tbl name =
  let rec go = function
    | [] -> None
    | h :: rest ->
        match Hashtbl.find_opt h name with
        | Some _ as r -> r
        | None -> go rest
  in
  go tbl.scopes
Scopes in Rust and OCaml In Rust, we use Vec<HashMap<String, SymbolKind>> — a stack of hash maps. Pushing adds a new map to the end. Popping removes it. Lookup iterates in reverse. In OCaml, we use a list of hash tables — cons for push, pattern match for pop, recursive traversal for lookup. Same concept, different idioms.

Populating the Symbol Table

We walk the AST in two passes. The first pass registers all top-level declarations (functions and global variables). The second pass visits each function body, pushing a new scope for the function's parameters, then another for each nested block.

Here is the first pass — collecting declarations:

src/symbol.rs
pub fn build_top_level(program: &Program) -> Result<SymbolTable, Vec<String>> {
    let mut sym = SymbolTable::new();
    let mut errors = Vec::new();

    for decl in &program.decls {
        match decl {
            Decl::Func(f) => {
                let param_types = f.params.iter().map(|p| p.ty.clone()).collect();
                if sym.lookup(&f.name).is_some() {
                    errors.push(format!("duplicate definition: {}", f.name));
                } else {
                    sym.insert(&f.name, SymbolKind::Func {
                        ret_type: f.ret_type.clone(),
                        param_types,
                    });
                }
            }
            Decl::Var(v) => {
                todo!()
            }
        }
    }

    if errors.is_empty() { Ok(sym) } else { Err(errors) }
}
symbol.ml
let build_top_level program =
  let sym = create () in
  let errors = ref [] in
  List.iter (function
    | Func f ->
        let param_types = List.map (fun p -> p.ty) f.params in
        (match lookup sym f.name with
         | Some _ ->
             errors := Printf.sprintf "duplicate definition: %s" f.name :: !errors
         | None ->
             insert sym f.name (Func { ret_type = f.ret_type; param_types }))
    | Var v -> failwith "todo"
  ) program.decls;
  if !errors = [] then Ok sym else Error !errors
Duplicate detection We check lookup before insert in the top-level pass to catch duplicate function names. In Rust, this is a compile error. In our language, it should be too. A duplicate definition is almost certainly a bug.

Type Checking

The type checker is the heart of semantic analysis. It walks the AST and assigns a type to every expression, verifying consistency at each step. Every expression has a type — the type checker discovers it. This is like Rust's type inference, but simpler: we do not infer types from usage, we verify that the declared and actual types match.

The checker returns either a type (for expressions) or unit (for statements), or an error. Here is the core function signature:

src/check.rs
pub enum TypeResult {
    Type(Type),
    Error,
}

fn check_expr(
    expr: &Expr,
    sym: &SymbolTable,
    ret_type: &Type,
) -> TypeResult {
    match expr {
        Expr::Int(_) => TypeResult::Type(Type::Prim("i32".into())),
        Expr::Bool(_) => TypeResult::Type(Type::Prim("bool".into())),
        Expr::Ident(name) => check_ident(name, sym),
        Expr::Binary { op, lhs, rhs } => check_binary(op, lhs, rhs, sym, ret_type),
        Expr::Unary { op, rhs } => check_unary(op, rhs, sym, ret_type),
        Expr::Call { callee, args } => check_call(callee, args, sym, ret_type),
        Expr::Cast { expr, target } => check_cast(expr, target, sym, ret_type),
        Expr::Assign { target, value } => check_assign(target, value, sym, ret_type),
    }
}
check.ml
type type_result =
  | Type of typ
  | Error

let rec check_expr expr sym ret_type =
  match expr with
  | Int _      -> Type (Prim "i32")
  | Bool _     -> Type (Prim "bool")
  | Ident name -> check_ident name sym
  | Binary { op; lhs; rhs } -> check_binary op lhs rhs sym ret_type
  | Unary { op; rhs } -> check_unary op rhs sym ret_type
  | Call { callee; args } -> check_call callee args sym ret_type
  | Cast { expr; target } -> check_cast expr target sym ret_type
  | Assign { target; value } -> check_assign target value sym ret_type

Type Rules

Each AST node has a type rule — a specification of what types its sub-expressions must have and what type the node itself produces. Here is the full table:

NodeChild types requiredResult type
Int(n)—i32
Bool(b)—bool
Ident(x)—type of x from symbol table
Add / Sub / Mul / Divboth operands i32i32
Eq / Neqboth operands same typebool
Lt / Gt / Le / Geboth operands i32bool
Neg (unary minus)operand i32i32
Not (boolean not)operand boolbool
Call(f, args)args match parameter types of freturn type of f

In Rust, this is a massive match statement — and it is beautiful. Each arm corresponds to exactly one type rule. The compiler checks exhaustiveness: if we forget to handle a variant, Rust tells us at compile time. OCaml's pattern matching gives the same guarantee.

Checking Binary Operators

Let us look at how we check binary expressions. This is where the type rules become concrete code:

src/check.rs
fn check_binary(
    op: &BinaryOp,
    lhs: &Expr,
    rhs: &Expr,
    sym: &SymbolTable,
    ret_type: &Type,
) -> TypeResult {
    let lt = check_expr(lhs, sym, ret_type);
    let rt = check_expr(rhs, sym, ret_type);

    let (lt, rt) = match (lt, rt) {
        (TypeResult::Type(l), TypeResult::Type(r)) => (l, r),
        _ => return TypeResult::Error,
    };

    match op {
        BinaryOp::Add | BinaryOp::Sub | BinaryOp::Mul | BinaryOp::Div => {
            expect_type(&lt, &Type::Prim("i32"), "arithmetic operand");
            expect_type(&rt, &Type::Prim("i32"), "arithmetic operand");
            TypeResult::Type(Type::Prim("i32"))
        }
        BinaryOp::Eq | BinaryOp::Neq => {
            expect_same_type(&lt, &rt);
            TypeResult::Type(Type::Prim("bool"))
        }
        BinaryOp::Lt | BinaryOp::Gt | BinaryOp::Le | BinaryOp::Ge => {
            expect_type(&lt, &Type::Prim("i32"), "comparison operand");
            expect_type(&rt, &Type::Prim("i32"), "comparison operand");
            TypeResult::Type(Type::Prim("bool"))
        }
    }
}
check.ml
let rec check_binary op lhs rhs sym ret_type =
  let lt = check_expr lhs sym ret_type in
  let rt = check_expr rhs sym ret_type in
  match lt, rt with
  | Type l, Type r ->
      (match op with
       | Add | Sub | Mul | Div ->
           expect_type l (Prim "i32") "arithmetic operand";
           expect_type r (Prim "i32") "arithmetic operand";
           Type (Prim "i32")
       | Eq | Neq ->
           expect_same_type l r;
           Type (Prim "bool")
       | Lt | Gt | Le | Ge ->
           expect_type l (Prim "i32") "comparison operand";
           expect_type r (Prim "i32") "comparison operand";
           Type (Prim "bool"))
  | _ -> Error

Notice the structure: we recurse into both sub-expressions, collect their types, then check consistency against the operator's rules. If either sub-expression already had an error, we propagate Error without additional noise. This is the same pattern Rust's compiler uses — collect errors eagerly, report them gracefully.

Error Messages

A good type error tells you three things: what went wrong, where it went wrong, and what was expected. Let us look at the error for our original nonsense program:

// test.l1s
func main(i32) {
    let x: bool = 42;
    return x;
}

Our type checker produces:

error: type mismatch: expected bool, found i32
  ┌─> test.l1s:2:20
  │
2 │     let x: bool = 42;
  │                    ^^ this expression has type i32
  │
  = note: variable 'x' was declared with type bool

Here is how we produce this:

src/diag.rs
pub struct Diag {
    pub level: String,    // "error" | "warning"
    pub msg: String,
    pub file: String,
    pub line: usize,
    pub col: usize,
    pub source: String,
    pub note: Option<String>,
}

impl std::fmt::Display for Diag {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        writeln!(f, "{}: {}", self.level, self.msg)?;
        writeln!(f, "  ┌─> {}:{}:{}", self.file, self.line, self.col)?;
        writeln!(f, "  │")?;
        write!(f, "{:>4} │ {}\n  │ {:padding$}^^",
            self.line, self.source, "",
            padding = self.col - 1)?;
        writeln!(f, "  │ {}", self.msg)?;
        if let Some(ref note) = self.note {
            writeln!(f, "  │")?;
            writeln!(f, "  = note: {}", note)?;
        }
        Ok(())
    }
}
diag.ml
type diag = {
  level:  string;
  msg:    string;
  file:   string;
  line:   int;
  col:    int;
  source: string;
  note:   string option;
}

let format d =
  Printf.sprintf
    "%s: %s\n\
       ┌─> %s:%d:%d\n\
      \n\
      %4d │ %s\n\
        │ %s^^\n\
        │ %s\n%s"
    d.level d.msg
    d.file d.line d.col
    d.line d.source
    (String.make (d.col - 1) ' ')
    d.msg
    (match d.note with
     | Some n -> Printf.sprintf "\n  = note: %s" n
     | None  -> "")
Error recovery A good type checker does not stop at the first error. It continues checking the rest of the program, collecting all errors, and presents them together. This is how rustc works — and how our compiler should work too. We use a Vec<Diag> accumulator threaded through the checker. When we encounter an error, we push to the accumulator and return TypeResult::Error, allowing the checker to keep going.

Checking a Complete Function

Let us walk through type-checking the full main function step by step. We will trace the execution of the checker on this program:

func main(i32) {
    let x: bool = 42;
    return x;
}
Enter check_func(main, ret_type=i32, params=[]) | ├─ push_scope() | (register no parameters — main takes none) | ├─ Enter check_block(body) | | | ├─ push_scope() | | | ├─ Check stmt[0]: VarDecl(x: bool = 42) | | ├─ check_expr(Int(42)) | | | └─ return Type(i32) | | ├─ Expected type: bool | | ├─ Actual type: i32 | | ├─ MISMATCH → emit error | | └─ (still register x as bool in symbol table) | | | ├─ Check stmt[1]: Return(Ident("x")) | | ├─ check_expr(Ident("x")) | | | └─ lookup "x" → bool | | | └─ return Type(bool) | | ├─ Expected return type: i32 | | ├─ Actual type: bool | | ├─ MISMATCH → emit error | | └─ return Error(propagated) | | | └─ pop_scope() | └─ pop_scope()

Two errors are emitted, both pointing to the exact source location. The user sees both at once:

error: type mismatch: expected bool, found i32
  ┌─> test.l1s:2:20
  │
2 │     let x: bool = 42;
  │                    ^^ this expression has type i32

error: type mismatch: expected i32, found bool
  ┌─> test.l1s:3:5
  │
3 │     return x;
  │     ^^^^^^^^^^ function returns i32

This is the same error-reporting philosophy that rustc uses: structured diagnostics with source underlines, multi-line formatting, and helpful notes. Our format is inspired by Rust's, and for the same reason — it makes the error immediately understandable.

What the Type Checker Enables

With a symbol table and type checker in place, we can now:

  • Reject invalid programs before they reach code generation.
  • Provide rich error messages with source locations.
  • Annotate the AST with computed types for later passes.
  • Verify function calls match their declarations (arity and types).
  • Check control flow — every path through a non-void function must return.
  • Verify conditions in if and while are boolean.

The type checker is the gatekeeper. Nothing gets past it that does not make sense.

The type checker is also a documentation generator Once every expression is annotated with its type, we know the precise type of every sub-expression in the program. This information is invaluable for code generation: we know how many bytes each value occupies, which registers to use, and how to call functions with the right calling convention.

Putting It Together

Here is how the pipeline looks with semantic analysis added:

Source text │ ▼ Lexer Tokens │ ▼ Parser AST │ ▼ Symbol table (Phase 1) Annotated AST + SymbolTable │ ▼ Type checker (Phase 2) Validated AST + SymbolTable + TypeAnnotations │ ▼ IR Builder Intermediate Representation

Two passes over the AST. The first collects names. The second checks types. Both produce errors if something is wrong. If both pass, the program is semantically valid and ready for code generation.

Exercises

1. Add struct type checking Extend the type checker to support struct types. A struct declaration struct Point { x: i32, y: i32 } defines a new named type. Field access p.x retrieves the field's type. What changes are needed in the symbol table? In the type checker? How do you handle recursive structs?
2. Function overload resolution Our language currently forbids duplicate function names. Change it to support overloading: two functions with the same name but different parameter types. The type checker must select the correct overload based on argument types. What changes in the symbol table? How do you handle ambiguity?
3. Control flow analysis Add a pass that verifies every path through a non-void function ends with a return statement. This is a simple reachability analysis on the AST: after an if with both branches, control resumes after the if; after a return, control stops. A function is valid if all paths through its body end in return. Rust calls this "reachability checking."
4. Type aliases Add type Int = i32; declarations. The type checker should resolve aliases to their underlying type. Store alias mappings in the symbol table. A value of type Int is interchangeable with i32.
5. Integer literal type inference Currently all integer literals are i32. Add support for explicit suffixes: 42u64, 7i8. The type checker should read the suffix and produce the correct type. If no suffix is present, default to i32 (like Rust's default integer type).
