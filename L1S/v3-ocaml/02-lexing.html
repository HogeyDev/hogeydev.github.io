Building a Compiler OCaml Edition

2 · Lexing

The Problem

Raw text is a string of characters. But a compiler does not think in characters — it thinks in tokens: keywords like func and let, numbers like 42, operators like + and -, punctuation like ( and }. The lexer (also called the tokenizer or scanner) is the phase that converts a string of characters into a stream of tokens, ready for the parser.

Source: "let x: i32 = 42;" Tokens: [Let, Ident("x"), Colon, TyI32, Eq, Int(42), Semicol]

The lexer is where we first encounter the power of OCaml variant types. A token is the textbook example of a discriminated union — each token kind is a constructor, and some carry data (an identifier's name, an integer's value).

pub enum Token {
    Int(i64),
    Float(f64),
    Ident(String),
    Str(String),
    Let, Func, If, Else, While, Return,
    True, False, Void, Bool, Cast,
    TyI32, TyI64, TyU32, TyU64, TyI8, TyU8,
    Arrow,
    Eq, EqEq, Bang, Neq,
    Lt, Gt, Leq, Geq,
    Plus, Minus, Star, Slash,
    LParen, RParen, LBrace, RBrace,

    Colon, Semicol, Comma, Dot, At,

}

This is the entire vocabulary of our language expressed as a single OCaml type. Reading the type definition is equivalent to reading the language's token-level specification. Each constructor corresponds to a token kind. Data-carrying constructors like Int of int and Ident of string attach the lexeme's value directly to the token. Pattern matching is exhaustive — if we later add a Loop token, every match on tokens will remind us where we need to handle it.

Mapping Keywords to Tokens

Identifiers and keywords share the same character class (letters and digits, starting with a letter). The lexer reads an identifier, then checks whether it is actually a keyword. This is the keyword_token function:

lexer.ml — keyword dispatch
let keyword_token s =
  match s with
  | "let"    -> Some Let
  | "func"   -> Some Func
  | "if"     -> Some If
  | "else"   -> Some Else
  | "while"  -> Some While
  | "return" -> Some Return
  | "true"   -> Some True
  | "false"  -> Some False
  | "void"   -> Some Void
  | "bool"   -> Some Bool
  | "cast"   -> Some Cast
  | "i32"    -> Some TyI32
  | "i64"    -> Some TyI64
  | "u32"    -> Some TyU32
  | "u64"    -> Some TyU64
  | "i8"     -> Some TyI8
  | "u8"     -> Some TyU8
  | _        -> None

fn keyword_token(s: &str) -> Option<Token> {
    match s {
        "let"    => Some(Token::Let),
        "func"   => Some(Token::Func),
        "if"     => Some(Token::If),
        "else"   => Some(Token::Else),
        "while"  => Some(Token::While),
        "return" => Some(Token::Return),
        "true"   => Some(Token::True),
        "false"  => Some(Token::False),
        "void"   => Some(Token::Void),
        "bool"   => Some(Token::Bool),
        "cast"   => Some(Token::Cast),
        "i32"    => Some(Token::TyI32),
        "i64"    => Some(Token::TyI64),
        "u32"    => Some(Token::TyU32),
        "u64"    => Some(Token::TyU64),
        "i8"     => Some(Token::TyI8),
        "u8"     => Some(Token::TyU8),
        _        => None,
    }
}

The pattern is simple: match a string, return a token option. None means the identifier is a regular name, not a keyword. The use of option here is classic OCaml — the type system forces the caller to handle both cases. OCaml's match with its concise | pattern -> expr syntax makes the mapping read like a table.

Pattern matching elegance

This is one of those moments where OCaml's design shines. In C, this function would be a chain of strcmp calls or a giant if-else ladder. In Python, it would be a dictionary lookup or another if-elif chain. In OCaml, it is the most natural thing in the world — a pattern match on strings. The compiler even checks for exhaustiveness: if you add a keyword to the language but forget to add it to keyword_token, OCaml won't warn you here (since strings are not a closed type), but the parser will fail to compile when it tries to produce a token that does not exist.

The Lexer Module

The lexer itself is a module that wraps a position, the source text, and produces tokens on demand. We structure it as a function that takes the source text and returns a list of tokens (each annotated with its span):

lexer.ml — the full lexer
type spanned = Token.t * Span.t

type state = {
  source: string;
  pos:    int ref;
}

let make source = { source; pos = ref 0 }

let peek s =
  if !(s.pos) >= String.length s.source then None
  else Some s.source.[!(s.pos)]

let advance s =
  let c = s.source.[!(s.pos)] in
  s.pos := !(s.pos) + 1;
  c

let skip_whitespace s =
  while match peek s with
        | Some (' ' | '\t' | '\n' | '\r') -> advance s; true
        | _ -> false
  do done

let lex_number s start =
  while match peek s with
        | Some ('0'..'9' | '.') -> advance s; true
        | _ -> false
  do done;
  let text = String.sub s.source start (!(s.pos) - start) in
  if String.contains text '.' then
    Float (float_of_string text)
  else
    Int (int_of_string text)

let lex_ident s start =
  while match peek s with
        | Some ('a'..'z' | 'A'..'Z' | '0'..'9' | '_') -> advance s; true
        | _ -> false
  do done;
  let text = String.sub s.source start (!(s.pos) - start) in
  match keyword_token text with
  | Some tok -> tok
  | None    -> Ident text

let lex_string s =
  let buf = Buffer.create 16 in
  while match peek s with
        | Some '"' -> advance s; false
        | Some c     -> Buffer.add_char buf c; advance s; true
        | None       -> failwith "unterminated string"
  do done;
  Str (Buffer.contents buf)

let next s =
  skip_whitespace s;
  let start = !(s.pos) in
  match peek s with
  | None -> (Eof, Span.make start start)
  | Some c ->
    let tok =
      match c with
      | '0'..'9'       -> lex_number s start
      | 'a'..'z' | 'A'..'Z' | '_' -> lex_ident s start
      | '"'             -> advance s; lex_string s
      | '+'             -> advance s; Plus
      | '-'             -> advance s;
          if peek s = Some '>' then (advance s; Arrow)
          else Minus
      | '*'             -> advance s; Star
      | '/'             -> advance s; Slash
      | '('             -> advance s; LParen
      | ')'             -> advance s; RParen
      | '{'             -> advance s; LBrace
      | '}'             -> advance s; RBrace
      | '['             -> advance s; LBrack
      | ']'             -> advance s; RBrack
      | ':'             -> advance s; Colon
      | ';'             -> advance s; Semicol
      | ','             -> advance s; Comma
      | '.'             -> advance s; Dot
      | '@'             -> advance s; At
      | '='             -> advance s;
          if peek s = Some '=' then (advance s; EqEq)
          else Eq
      | '!'             -> advance s;
          if peek s = Some '=' then (advance s; Neq)
          else Bang
      | '<'             -> advance s;
          if peek s = Some '=' then (advance s; Leq)
          else Lt
      | '>'             -> advance s;
          if peek s = Some '=' then (advance s; Geq)
          else Gt
      | _               -> failwith (Printf.sprintf "unexpected char: %c" c)
    in
    (tok, Span.make start !(s.pos))

let tokenize source =
  let s = make source in
  let rec loop acc =
    let tok, span = next s in
    match tok with
    | Eof -> List.rev ((tok, span) :: acc)
    | _   -> loop ((tok, span) :: acc)
  in
  loop []
src/lexer.rs — the full lexer
pub struct Lexer {
    source: Vec<u8>,
    pos: usize,
}

pub type Spanned = (Token, Span);

impl Lexer {
    pub fn new(source: &str) -> Self {
        Self { source: source.bytes().collect(), pos: 0 }
    }

    fn peek(&self) -> Option<u8> {
        if self.pos >= self.source.len() { None }
        else { Some(self.source[self.pos]) }
    }

    fn advance(&mut self) -> u8 {
        let c = self.source[self.pos];
        self.pos += 1;
        c
    }

    fn skip_whitespace(&mut self) {
        while let Some(b) = self.peek() {
            if b.is_ascii_whitespace() { self.advance(); }
            else { break; }
        }
    }

    fn lex_number(&mut self, start: usize) -> Token {
        while let Some(b) = self.peek() {
            if b.is_ascii_digit() || b == b'.' { self.advance(); }
            else { break; }
        }
        let text = std::str::from_utf8(&self.source[start..self.pos]).unwrap();
        if text.contains('.') {
            Token::Float(text.parse().unwrap())
        } else {
            Token::Int(text.parse().unwrap())
        }
    }

    fn lex_ident(&mut self, start: usize) -> Token {
        while let Some(b) = self.peek() {
            if b.is_ascii_alphanumeric() || b == b'_' { self.advance(); }
            else { break; }
        }
        let text = std::str::from_utf8(&self.source[start..self.pos]).unwrap();
        keyword_token(text).unwrap_or_else(|| Token::Ident(text.to_string()))
    }

    fn next(&mut self) -> Spanned {
        self.skip_whitespace();
        let start = self.pos;
        let tok = match self.peek() {
            None => Token::Eof,
            Some(c) => match c {
                b'0'..=b'9'          => self.lex_number(start),
                b'a'..=b'z' | b'A'..=b'Z' | b'_' => self.lex_ident(start),
                b'"' => { self.advance(); self.lex_string() }
                b'+' => { self.advance(); Token::Plus }
                b'-' => { self.advance();
                    if self.peek() == Some(b'>') { self.advance(); Token::Arrow }
                    else { Token::Minus } }
                b'=' => { self.advance();
                    if self.peek() == Some(b'=') { self.advance(); Token::EqEq }
                    else { Token::Eq } }
                b'!' => { self.advance();
                    if self.peek() == Some(b'=') { self.advance(); Token::Neq }
                    else { Token::Bang } }
                // ... remaining single- and multi-char tokens
                _ => panic!("unexpected char: {}", c as char),
            },
        };
        (tok, Span::new(start, self.pos))
    }

    pub fn tokenize(&mut self) -> Vec<Spanned> {
        let mut tokens = Vec::new();
        loop {
            let (tok, span) = self.next();
            tokens.push((tok.clone(), span));
            if tok == Token::Eof { break; }
        }
        tokens
    }
}
State as a record with a ref

Notice that the OCaml lexer state uses a record with a pos ref rather than a mutable field. OCaml records are immutable by default, so we use a ref for the single mutable cell. This is a deliberate choice — it makes the state explicit and avoids hidden mutation. Rust uses &mut self everywhere, which is equally explicit but differently structured.

How the Lexer Works

The lexer is a character-at-a-time state machine. The core loop:

  1. Skip whitespace — advance past spaces, tabs, newlines, carriage returns.
  2. Peek at the next character — if none, emit Eof.
  3. Dispatch on the character class:
    • Digits → call lex_number, which reads a sequence of digits and an optional decimal point.
    • Letters or underscore → call lex_ident, which reads an identifier, then checks via keyword_token whether it is a keyword.
    • Quote → call lex_string to read a string literal.
    • Operators and punctuation → match the specific character, with lookahead for two-character operators like == and ->.
    • Anything else → error: unexpected character.
  4. Return the token with its span — the start position was recorded before reading, the end position is the current position after reading.

The tokenize function wraps this in a loop that accumulates tokens until Eof is reached. Note the accumulator pattern: we prepend each token to a list (O(1)), then reverse at the end. This is idiomatic OCaml — building lists forward requires appending, which is O(n). Building backward and reversing is O(n) total.

let source = "let x: i32 = 42;" in

Every token knows exactly where it came from. If the parser later finds a syntax error, it can report the precise span of the offending token.

Extending the Lexer: Hex, Octal, and Binary Literals

A practical lexer must handle more than decimal integers. Let us extend lex_number to support hexadecimal (0xFF), octal (0o77), and binary (0b1010) literals. The key insight is to look for the prefix before falling into the digit-reading loop:

lexer.ml — extended lex_number
let lex_number s start =
  let base, is_float =
    if peek s = Some '0' then
      advance s;
      match peek s with
      | Some ('x' | 'X') -> advance s; 16, false
      | Some ('o' | 'O') -> advance s; 8,  false
      | Some ('b' | 'B') -> advance s; 2,  false
      | _ -> 10, true
    else
      10, true
  in
  while match peek s with
        | Some ('0'..'9' | 'a'..'f' | 'A'..'F') -> advance s; true
        | Some '.' when is_float -> advance s; true
        | _ -> false
  do done;
  let text = String.sub s.source start (!(s.pos) - start) in
  if base <> 10 then
    Int (int_of_string text)
  else if String.contains text '.' then
    Float (float_of_string text)
  else
    Int (int_of_string text)
src/lexer.rs — extended lex_number
fn lex_number(&mut self, start: usize) -> Token {
    let (base, is_float) =
        if self.peek() == Some(b'0') {
            self.advance();
            match self.peek() {
                Some(b'x' | b'X') => { self.advance(); (16, false) }
                Some(b'o' | b'O') => { self.advance(); (8,  false) }
                Some(b'b' | b'B') => { self.advance(); (2,  false) }
                _ => (10, true),
            }
        } else {
            (10, true)
        };
    while let Some(b) = self.peek() {
        let ok = b.is_ascii_digit()
             || (base == 16 && (b == b'a'..=b'f' || b == b'A'..=b'F'))
             || (is_float && b == b'.');
        if ok { self.advance(); } else { break; }
    }
    let text = std::str::from_utf8(&self.source[start..self.pos]).unwrap();
    if base != 10 || !text.contains('.') {
        Token::Int(i64::from_str_radix(&text[2..], base).unwrap())
    } else {
        Token::Float(text.parse().unwrap())
    }
}
Pattern matching with when guards

The OCaml version uses a when guard: Some '.' when is_float. This is an OCaml pattern guard — a boolean condition that must be true for the pattern to match. Guards let you express conditions that cannot be captured by patterns alone. Here, we only treat . as a float character when we are in decimal mode (after 0 we temporarily set is_float to false because 0xDEAD.BEEF is not valid).

Lexer Design Decisions

Several design decisions in this lexer are worth discussing explicitly:

No separate token list type. We use (Token.t * Span.t) list directly. OCaml's type system makes this readable — there is no need for a wrapper type. The tuple expresses everything: a token and where it came from.

No lexer generator. We are writing the lexer by hand rather than using ocamllex (OCaml's lexer generator). Why? Because for a small language, a hand-written lexer is simpler, more transparent, and easier to debug. The regular expressions that power lexer generators are powerful, but the character-at-a-time approach gives us complete control. When our language grows, we can switch to ocamllex — the token type and the interface do not change.

Error handling via exceptions. The lexer uses failwith for unrecoverable errors (unterminated strings, unexpected characters). This is a pragmatic choice for a pedagogical compiler. A production compiler would use proper error types, but for now, exceptions are sufficient. We will improve error handling in later chapters.

failwith is not the end

Using failwith means the lexer crashes on invalid input. For a real compiler, you would want recovery — skip the bad character and continue to find more errors. We leave this as an exercise. The lexer is the first place where the compiler interacts with imperfect human input, and it is worth spending time on robust error recovery.

Exercises

Exercise 2.1 — Add a keyword

Add a loop keyword to the language. Add the constructor to Token.t, add the mapping to keyword_token, and add the printing case to to_string. Verify that the lexer produces Loop tokens for loop in source code.

Exercise 2.2 — Line comments

Add support for line comments (// ...). The lexer should skip from // to the end of the line. Hint: this belongs in skip_whitespace or as a separate skip_comment call.

Exercise 2.3 — Block comments

Add support for block comments (/* ... */). This is harder than line comments because they can nest (and should they?). Decide whether your block comments nest and implement accordingly.

Exercise 2.4 — Error recovery

Modify the lexer so that when it encounters an unexpected character, it records the error but continues lexing (by skipping the bad character). The tokenize function should return both a token list and an error list. This is the first step toward a resilient compiler.

Exercise 2.5 — Character classes with pattern matching

The OCaml lexer uses '0'..'9' range patterns. Read about OCaml's pattern-matching features: range patterns, or-patterns (already used here), and pattern guards. Write a short summary of which pattern features the lexer uses and where each one appears. This deepens your understanding of how OCaml's pattern matching directly expresses the lexer's decision logic.

Exercise 2.6 — Benchmark against ocamllex

Implement the same lexer using ocamllex (the OCaml lexer generator). Compare the line count, readability, and performance of the two approaches. This exercise teaches you both the power and the limitations of regex-based lexing.

Exercise 2.7 — Unicode identifiers

Modify the lexer to accept Unicode letters in identifiers (not just ASCII). OCaml's String module operates on bytes, so you will need to handle UTF-8 multi-byte sequences. Use the Uutf library or implement a simple UTF-8 decoder. This exercise reveals why many compilers restrict themselves to ASCII identifiers — the complexity is real.

Exercise 2.8 — Raw string literals

Add raw string literals similar to Rust's r#"..."# or Python's """...""". In raw strings, escape sequences like \n are not processed — the string content is taken literally. Design the delimiter syntax carefully to avoid ambiguity with content that contains the delimiter itself.

Exercise 2.9 — Lexer module signature

Create an lexer.mli file that exposes only spanned, make, and tokenize. Hide the internal state type, peek, advance, and the helper functions. This exercise demonstrates OCaml's module abstraction — the implementation becomes a black box that callers cannot depend on, letting you refactor freely.
