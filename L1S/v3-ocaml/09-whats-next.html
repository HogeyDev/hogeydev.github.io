The OCaml version is remarkably concise. Pattern matching and algebraic data types make tree transformations compact and correct. The module system provides clean encapsulation without boilerplate. Functors offer a path to reusable optimization passes. This is why OCaml has been the language of choice for compiler research for over two decades.

Not a weakness — a starting point Every production compiler started where we are now. LLVM began as a research project. GCC began as a small C compiler. Rust began as a personal project. OCaml's own compiler started as a PhD thesis. Every feature above was added one at a time.

Ideas for the Reader

Here are concrete projects, ordered from easiest to hardest:

Add One Feature at a Time

The best advice: add the thing you miss most. Is it frustrating that you can't print output? Add a print function that calls printf via the C FFI. Do you want to write recursive functions? Make sure the calling convention handles it correctly (it should already).

Each feature teaches you something new. Adding print teaches you about FFI. Adding arrays teaches you about memory layout. Adding closures teaches you about environment capture and heap allocation.

Port to a New Backend

Our compiler targets x86-64, but the IR is architecture-independent. Port the codegen module to a new target:

  • ARM64 (AArch64): Different registers (x0-x30), same concepts. Used in Apple Silicon, most phones.
  • RISC-V: The cleanest ISA. Only 47 base instructions. Registers x0-x31.
  • WebAssembly: A stack-based virtual ISA that runs in browsers. Completely different codegen model.
  • LLVM IR: Generate LLVM IR instead of assembly. LLVM handles register allocation and machine code for you. This is what Rust, Swift, and Clang do.

Porting requires rewriting only the codegen module. The frontend — lexer, parser, type checker, IR builder — stays exactly the same. This is the power of a well-designed IR.

Add an Optimization Pass

Our generated code is full of redundant loads and stores. Try these:

Constant folding: If the IR has Const(%0, 3); Const(%1, 4); Add(%2, %0, %1), replace it with Const(%2, 7). A single pass over the IR.

Dead store elimination: If a value is stored to a stack slot and never loaded before being overwritten, the first store is dead. Requires liveness analysis.

Peephole optimization: After codegen, scan the assembly for inefficient patterns. mov rax, [rbp-8]; mov [rbp-8], rax is a no-op — remove it.

In OCaml, each optimization is a functor: module Folded = ConstantFolding(BaseIR). Compose them: module OptIR = DCE(ConstantFolding(BaseIR)).

References and Further Reading

If this book has sparked a deeper interest in compilers — and we hope it has — here are the resources that will take you further:

Textbooks

  • Modern Compiler Implementation in ML by Andrew W. Appel. This is the direct spiritual successor to what we've built here. Appel walks through the same pipeline (lexing, parsing, type checking, IR generation, code generation) using ML — the language that OCaml descends from. His treatment of register allocation via graph coloring is a classic. If you read one compiler book, read this one.
  • Compilers: Principles, Techniques, and Tools (the "Dragon Book") by Aho, Lam, Sethi, and Ullman. The canonical reference for parsing theory, data-flow analysis, and code optimization. The dragon on the cover is legendary for a reason.
  • Types and Programming Languages by Benjamin C. Pierce. The definitive introduction to type systems. If you want to understand what our type checker was doing at a deeper theoretical level — inference, polymorphism, subtyping — start here.
  • Practical Foundations for Programming Languages by Robert Harper. A more advanced treatment of type theory and semantics. Harper was one of the designers of Standard ML, and his perspective is deeply informed by ML-family languages.
  • Linkers and Loaders by John R. Levine. Everything about what happens after assembly: relocation, symbol resolution, dynamic linking, position-independent code.

OCaml-Specific Resources

  • Real World OCaml by Yaron Minsky, Anil Madhavapeddy, and Jason Hickey. The modern guide to OCaml programming, published by O'Reilly. Covers the module system, functors, and the ecosystem. Available free online at dev.realworldocaml.org.
  • Jane Street — the largest industrial user of OCaml. Their tech blog and open-source contributions (Core, Incremental, Async) are invaluable for seeing how OCaml is used at scale in production systems, including their internal compiler tools.
  • OCaml Labs and Tarides — research and development groups that drive the evolution of the OCaml compiler and tooling. Their work on the OCaml compiler itself — the type checker, pattern match compilation, and backend — is directly relevant to what we've built.
  • The OCaml compiler source code itself. The real typing/ directory implements Hindley-Milner type inference with side-effectful unification. The middle_end/ and backend/ directories implement the actual compiler pipeline for OCaml programs. It's hundreds of thousands of lines, but the structure will look familiar: AST, type checking, IR, code generation.
  • OCaml Discuss (discuss.ocaml.org) and OCaml Discord — active communities where you can ask questions about OCaml compiler development.

Specific Papers and Projects

  • "SSA-based Compiler Design" — a tutorial by Braun, Buchwald, and Hack on the SSA construction algorithm we referenced in Chapter 7. The paper presents the algorithm in a language-agnostic way, but the OCaml implementation is only about 80 lines.
  • "Linear Scan Register Allocation" by Poletto and Sarkar. The paper that introduced the algorithm we sketched in Chapter 8. It's short, readable, and the results are immediately applicable.
  • LLVM (llvm.org) — the industry-standard compiler infrastructure. LLVM's IR is more sophisticated than ours, but the concepts are the same. The LLVM documentation is excellent, and the source is written in modern C++ with clear abstractions.
  • Cranelift — a code generator written in Rust. Designed for use cases like WebAssembly and JIT compilation. Its codegen pipeline is remarkably similar to what we built, but production-quality.

The Journey Ahead

Building a compiler is one of those rare projects that changes how you see every other program. After this, when you open a source file, you don't just see code — you see the pipeline: the lexer carving out tokens, the parser building trees, the type checker walking nodes, the IR builder linearizing control flow, the code generator emitting instructions, the assembler and linker producing bytes.

In OCaml, this pipeline felt natural because the language was designed for it. Variant types mirror the structure of syntax trees. Pattern matching makes tree traversal declarative. The module system scales with complexity. If you started this book knowing Rust or another language and learned OCaml along the way, you've gained not just a compiler, but a new way of thinking about programs.

What you build next is up to you. Add a feature. Port to ARM64. Implement SSA. Write a garbage collector. Make the compiler self-hosting. The framework is solid, the IR is clean, and the tests pass. Every direction is open.

The code is yours now The companion code in code/ocaml/ is your starting point. Modify it. Break it. Fix it. Extend it. The best way to learn compilers is to change a working compiler and see what happens. We've given you a working compiler. Now make it yours.

Thank you for reading. Thank you for building. And most of all — thank you for staying curious. The machine speaks because you took the time to learn its language.

Exercises

1. Your own language feature

Design and implement a language feature that matters to you. It could be for loops, break/continue, compound assignment operators (+=, -=), or something entirely new. Trace it through every phase of the compiler.

2. Benchmark your compiler

Write a benchmark: a program that computes something computationally intensive (e.g., Fibonacci, matrix multiply, prime sieve). Compile it with our compiler and with gcc/clang. Compare the generated assembly and the execution times. Where does our compiler fall short? Which optimizations would close the gap?

3. Read the real OCaml compiler

Download the OCaml compiler source code. Find the equivalent files to the ones we built: parsing/parser.mly, typing/typecore.ml, middle_end/, asmcomp/. Read one file. Compare it to what we built. What's the same? What's different? What do you learn from the production version?

4. Teach someone else

The best way to cement your understanding is to explain it. Write a blog post, give a talk, or pair-program with a friend who hasn't built a compiler. You'll discover which parts you truly understand and which parts need more work. The compiler community grows when we share what we've learned.
