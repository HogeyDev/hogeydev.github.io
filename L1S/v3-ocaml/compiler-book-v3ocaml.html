What Is a Compiler?

A compiler is a program that translates source code from one language to another. Usually, the source is a high-level programming language and the target is a lower-level one — assembly language, bytecode, or machine code. The key insight is that this translation preserves meaning: the compiled program does exactly what the source program says, just in a different language.

This is harder than it sounds. The source language is designed for human comprehension: it has named variables, structured control flow, and rich type annotations. The target language is designed for machine execution: registers, memory addresses, jump instructions. Bridging this gap is the compiler's job, and it requires a deep understanding of both the source language's semantics and the target architecture's capabilities.

There is a famous result from Strachey: "A compiler is a function from source programs to target programs that preserves meaning." This is the mathematical ideal. Real compilers approximate it — they have bugs, they miss optimizations, they reject valid programs due to limitations in their analysis. But the ideal guides everything we do. Every phase of the compiler can be seen as a transformation that preserves meaning while changing representation.

What We're Building

We are building a compiler for a small but real systems language. It has functions, variables, types, arithmetic, control flow, and enough machinery to compile meaningful programs. Here is what a program looks like:

func main(i32) {
  let x: i32 = 42;
  let y: i32 = x + 1;
  return y;
}

Every piece of that program means something. func declares a function. main is its name. i32 after the name is the return type. Inside the body, let introduces variables, : gives them types, = assigns values, and return sends a value back to the caller. The language looks familiar because it borrows from C, Rust, and modern systems languages — but we keep it small so a single person can implement it end to end.

Language Design Philosophy

The language we compile is deliberately minimal. It is a statically typed, imperative systems language with a C-family syntax. Every feature we include has a pedagogical purpose:

  • Static types — teach semantic analysis and type checking. Our type system includes integers of various widths, booleans, pointers, and arrays. This is rich enough to be interesting but simple enough to implement in one chapter.
  • Functions — teach calling conventions, stack frames, and parameter passing. Functions are the fundamental unit of abstraction in most languages, and compiling them correctly is the cornerstone of a working compiler.
  • Control flow (if, while) — teach branching and code generation for conditional execution. Our control flow constructs map directly to x86-64 conditional jumps, making the code generation phase straightforward.
  • Mutable variables with let — teach stack allocation and memory operations. Unlike functional languages where variables are immutable by default, our language is imperative. This means we need to think about memory from day one.
  • Pointers and arrays — teach memory addressing, pointer arithmetic, and bounds considerations. These are the features that make a systems language useful for low-level programming.

What we deliberately exclude: objects, closures, generics, pattern matching (ironically, for a compiler written in OCaml), exceptions, concurrency, modules, inheritance, operator overloading. Each of these is a fascinating compilation challenge, but each would double the size of the compiler. The best way to learn compiler construction is to build a complete compiler for a small language, not a half-finished compiler for a large one.

The Compiler Pipeline (30,000 Feet)

Every compiler, from the simplest toy to clang or gcc, follows the same basic structure. Source text goes in one end; machine code (or assembly) comes out the other. Between them, the compiler performs a series of transformations, each one converting the program from one representation to another, getting progressively closer to machine language.

Source ──▶ Lexer ──▶ Parser ──▶ Semantic Analysis ──▶ Codegen ──▶ Output tokens AST IR asm

There are four major phases, and we will dedicate a chapter (or more) to each one:

  1. Lexing (Chapter 2) — The lexer reads the raw source text and breaks it into tokens: keywords, identifiers, numbers, operators, punctuation. It discards whitespace and comments. The output is a stream of tokens, each annotated with its source position. The lexer is the simplest phase — it is essentially a character-by-character state machine with a few helper functions for numbers and identifiers.
  2. Parsing (Chapter 3) — The parser takes the token stream and builds a tree structure called an abstract syntax tree (AST). The AST captures the grammatical structure of the program: which expressions are nested inside which, which statements belong to which function, which declarations form the program. We use recursive descent with Pratt-style expression parsing, which gives us clean, readable code that mirrors the grammar directly.
  3. Semantic Analysis (Chapter 5) — The semantic analyzer walks the AST and checks things that the parser cannot: type correctness, variable scoping, function call arity, return path coverage. This phase produces a typed AST or intermediate representation (IR) with all semantic information resolved. In many compilers, this is where the majority of error messages come from.
  4. Code Generation (Chapter 7) — The code generator takes the analyzed IR and emits x86-64 assembly. This is where the abstract becomes concrete — variables become registers or stack slots, control flow becomes jumps and branches, function calls follow the System V AMD64 ABI. The output is assembly text that the system assembler (nasm or gas) can turn into an object file.

Between parsing and code generation, we insert an intermediate representation (IR) phase (Chapter 6). The IR is a simplified, low-level representation that bridges the gap between the AST and assembly. It uses three-address code, has an infinite set of temporaries, and makes control flow explicit through basic blocks and jumps. The IR simplifies code generation dramatically — instead of generating assembly directly from the AST (which is complex), we first lower the AST to IR (which is straightforward), then generate assembly from IR (which is also straightforward).

AST ──▶ Lowering ──▶ IR ──▶ Codegen ──▶ Assembly (ch6) (ch7)

This multi-stage pipeline is the defining architectural pattern of compilers. Each stage has a well-defined input and output representation. Each stage is independent — you can modify the parser without touching the code generator, as long as the AST stays the same. This modularity is what makes a large compiler project manageable.

Not just four phases

Real compilers have many more phases: preprocessing, optimization passes, register allocation, instruction scheduling. Our compiler will have all of these, but we start simple and add passes as we go. The pipeline architecture — a series of transformations between representations — is the key insight.

Why OCaml for a Compiler?

This is the question we hope to answer on every page, but let us give the short version upfront.

OCaml is an extraordinary language for writing compilers. It is not an accident. The ML family of languages (of which OCaml is the most prominent modern descendant) was literally invented for implementing programming languages. Robin Milner's work on LCF and the ML language in the 1970s set the stage, and the tradition continues today. The OCaml compiler itself is written in OCaml. The Coq proof assistant's extraction mechanism targets OCaml for producing verified compilers. When you write a compiler in OCaml, you stand on the shoulders of giants.

Algebraic data types (variants). An AST is an algebraic data type. Every kind of node in the tree — every expression, every statement, every type — is a variant constructor. OCaml lets you write this directly:

type expr =
  | Int of int
  | Var of string
  | BinOp of expr * binop * expr
  | Call of string * expr list

This is not a simulation of a tree — this is the tree. Every constructor tag is a node kind. The of clause says what children it carries. Pattern matching deconstructs it:

let rec string_of_expr e =
  match e with
  | Int n       -> string_of_int n
  | Var x       -> x
  | BinOp (l,op,r) -> Printf.sprintf "(%s %s %s)"
      (string_of_expr l) (string_of_binop op) (string_of_expr r)
  | Call (f,args) -> Printf.sprintf "%s(%s)" f
      (String.concat ", " (List.map string_of_expr args))

The compiler exhaustively checks every case. If you add a new node kind and forget to handle it somewhere, OCaml warns you. This is not a nice-to-have. It is a compiler correctness guarantee baked into the language.

Modules and functors. Compilers are large programs that need careful organization. OCaml's module system — with signatures, functors (parameterized modules), and first-class modules — lets you structure passes, register allocators, and optimization pipelines with genuine abstraction. No header files, no fragile include guards, no preprocessor. Just clean, compositional interfaces.

Consider how a code generator pass might work. In OCaml, we can parameterize a pass module over its configuration:

module type CONFIG = sig
  val target_arch : string
  val optimization_level : int
end

module Codegen(C : CONFIG) = struct
  let arch = C.target_arch
  let level = C.optimization_level
  (* ... implementation uses arch and level ... *)
end

This is a functor — a module parameterized by another module. In Rust, you would use generics with trait bounds. Both approaches work, but OCaml's module system scales differently: it handles complex dependencies (multiple modules, nested signatures, recursive modules) with a uniformity that trait systems struggle to match.

The canonical textbook uses OCaml. Andrew Appel's Modern Compiler Implementation in ML is the classic. You are in good company. The ML tradition runs deep in compiler engineering — from the FLINT project at Yale to the OCaml compiler itself (written in OCaml, of course).

What This Compiler Does Not Do

It is worth being explicit about what we are not building. This compiler does not have a linker — we produce assembly files and assume the system assembler and linker handle the rest. It does not have a standard library — the language is minimal enough that all programs are self-contained. It does not have separate compilation — every compilation unit is built monolithically. These are real concerns for production compilers, but they are orthogonal to the core pipeline we study.

What the compiler does have is a complete pipeline from text to working assembly for x86-64 Linux. Every phase is implemented, every data structure is real, and every test file produces an executable that actually runs. You will write code, compile it with your compiler, and run it. The Problem

A compiler cannot just say "error." It has to say where the error happened. If your program has a type mismatch on line 17, the compiler must tell you: here, at this exact position in the source. Getting this right is the difference between a compiler you love and a compiler you want to throw out a window.

Every error message in our compiler needs three pieces of information:

  • What went wrong — the error message itself
  • Where it went wrong — a location in source code
  • Why it matters — context from the compiler phase

This chapter solves the "where." We define types and data structures for representing source positions and source files. These are the foundation on which all error reporting is built.

The Span

A span is a range in the source text. It has a start index and an end index, each measured in bytes from the beginning of the source file. A single character is a span where start + 1 = end. An entire function definition is a span from its first character to its last.

In OCaml, a span is a record. Records with labeled fields are one of OCaml's great pleasures — every field has a name, every access is checked, and the compiler will warn you if you forget a field:

span.ml
type t = {
  start: int;
  stop:  int;
}

let make start stop = { start; stop }

let empty = { start = 0; stop = 0 }

let contains span pos =
  pos >= span.start && pos < span.stop

let merge a b =
  { start = min a.start b.start;
    stop  = max a.stop  b.stop
  }
src/span.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Span {
    pub start: usize,
    pub stop: usize,
}

impl Span {
    pub fn new(start: usize, stop: usize) -> Self {
        Self { start, stop }
    }

    pub fn contains(&self, pos: usize) -> bool {
        pos >= self.start && pos < self.stop
    }

    pub fn merge(&self, other: &Self) -> Self {
        Self {
            start: self.start.min(other.start),
            stop:  self.stop.max(other.stop),
        }
    }
}
Records in OCaml

OCaml records use curly braces with field: type; syntax. Construction uses named fields — { start; stop } is a shorthand for { start = start; stop = stop }. Field access uses dot notation: span.start. The compiler tracks exactly which fields are present and will reject incomplete record expressions. This is a small thing that saves enormous debugging time.

The Span.t type is deliberately minimal. It stores byte offsets into the source text — not line and column numbers. Why byte offsets? Because they are the natural unit that the lexer works with. The lexer advances through the source one character (one byte for ASCII, one to four for UTF-8) at a time and records the current position. Converting a byte offset to a line number is a separate operation, and we keep it separate.

Converting Offsets to Lines

To produce human-readable error messages, we need to map a byte offset to a line number and column. This is the job of the SourceFile module. It stores the source text and a precomputed list of line start positions.

source_file.ml
type t = {
  name:  string;
  text:  string;
  lines: int array;
}

let of_string name text =
  let lines = Array.of_list (0 :: line_starts text) in
  { name; text; lines }

let line_and_column file offset =
  let lines = file.lines in
  let rec search lo hi =
    if hi - lo <= 1 then lo
    else
      let mid = (lo + hi) / 2 in
      if lines.(mid) <= offset then search mid hi
      else search lo mid
  in
  let line = search 0 (Array.length lines - 1) in
  let col  = offset - lines.(line) in
  (line + 1, col + 1)
src/source_file.rs
pub struct SourceFile {
    pub name:  String,
    pub text:  String,
    lines: Vec<usize>,
}

impl SourceFile {
    pub fn new(name: &str, text: &str) -> Self {
        let lines = std::iter::once(0)
            .chain(text.char_indices()
                .filter_map(|(i, c)| if c == '\n' { Some(i + 1) } else { None }))
            .collect();
        Self { name: name.to_string(), text: text.to_string(), lines }
    }

    pub fn line_and_column(&self, offset: usize) -> (usize, usize) {
        let line = match self.lines.binary_search(&offset) {
            Ok(i) => i,
            Err(i) => i.saturating_sub(1),
        };
        let col = offset - self.lines[line];
        (line + 1, col + 1)
    }
}

The line_starts helper scans the text and records the byte offset of every newline character, producing positions like [0, 47, 103, ...] — each entry is the start of a new line. To find the line for a given offset, we binary-search the array. Binary search is logarithmic; for a file with thousands of lines, the lookup is essentially instantaneous.

Here is the helper function itself — a simple fold over the string:

source_file.ml — line_starts helper
let line_starts text =
  let rec scan pos acc =
    match String.index_from text pos '\n' with
    | n -> scan (n + 1) ((n + 1) :: acc)
    | exception Not_found -> List.rev acc
  in
  scan 0 []

String.index_from and exceptions

The OCaml version uses String.index_from, which raises Not_found if no newline is found. We catch this exception to terminate the recursion. This is a clean use of OCaml's exception mechanism as a control flow device — it avoids an extra bounds check on every iteration. The alternative would be to manually scan character by character, which is equally valid but more verbose.

Error Reporting with Spans

With spans and source files, we can build an error reporter that shows the user exactly what went wrong, with context:

error.ml
type t = {
  msg:     string;
  span:    Span.t;
  phase:   string;
}

let report file err =
  let line, col = SourceFile.line_and_column file err.span.start in
  Printf.eprintf "%s:%d:%d: %s\n"
    file.name line col err.msg

let report_with_context file err =
  let line, col = SourceFile.line_and_column file err.span.start in
  let line_start = file.lines.(line - 1) in
  let line_end =
    if line < Array.length file.lines
    then file.lines.(line) - 1
    else String.length file.text
  in
  let line_text = String.sub file.text line_start (line_end - line_start) in
  Printf.eprintf "%s:%d:%d: %s\n" file.name line col err.msg;
  Printf.eprintf "  %s\n" line_text;
  Printf.eprintf "  %s^ %s\n"
    (String.make (col - 1) ' ')
    err.msg
src/error.rs
pub struct Error {
    pub msg:   String,
    pub span:  Span,
    pub phase: String,
}

pub fn report(file: &SourceFile, err: &Error) {
    let (line, col) = file.line_and_column(err.span.start);
    eprintln!("{}:{}:{}: {}", file.name, line, col, err.msg);
}

pub fn report_with_context(file: &SourceFile, err: &Error) {
    let (line, col) = file.line_and_column(err.span.start);
    let line_start = file.lines[line - 1];
    let line_end = if line < file.lines.len() { file.lines[line] - 1 }
                            else { file.text.len() };
    let line_text = &file.text[line_start..line_end];
    eprintln!("{}:{}:{}: {}", file.name, line, col, err.msg);
    eprintln!("  {}", line_text);
    eprintln!("  {:width$}^ {}", "", err.msg, width = col - 1);
}
Span merging

When reporting errors involving multiple locations — for example, a type mismatch between two expressions — you can use Span.merge to create a combined span that covers both locations. This produces error messages like: error: expected i32, found i64 (here and here).

Span Arithmetic

Spans support a small algebra of operations that become surprisingly useful as the compiler grows. Beyond merge and contains, we might want:

  • Intersection — does one span overlap with another? Useful for checking if two AST nodes share source text.
  • Span difference — "the part of this span before that point." Useful for error messages that need to point at a specific sub-expression within a larger expression.
  • Span ordering — does span A start before span B? The lexer produces tokens in order, and we sometimes need to verify that property.

Notice that all of these are pure functions on the Span.t record. No mutable state, no IO, no side effects. This is OCaml at its best: small, composable, verifiable operations on simple data.

Why Not Just Use Line Numbers?

An obvious question: why do we work with byte offsets instead of line-and-column pairs directly? Because line-and-column information is fragile — it changes when the source file is edited, and it is ambiguous when the same span appears in different contexts. Byte offsets are absolute. The lexer can advance a position counter without knowing or caring which line it is on. Line-and-column computation is pushed to the error reporting phase, which is the only place it matters.

This separation of concerns — data structures that are convenient for the machine versus those that are convenient for the human — is a recurring theme in compiler design. The span is for the compiler. The error message is for the human.

The Problem

Raw text is a string of characters. But a compiler does not think in characters — it thinks in tokens: keywords like func and let, numbers like 42, operators like + and -, punctuation like ( and }. The lexer (also called the tokenizer or scanner) is the phase that converts a string of characters into a stream of tokens, ready for the parser.

Source: "let x: i32 = 42;" Tokens: [Let, Ident("x"), Colon, TyI32, Eq, Int(42), Semicol]

The lexer is where we first encounter the power of OCaml variant types. A token is the textbook example of a discriminated union — each token kind is a constructor, and some carry data (an identifier's name, an integer's value).

Pattern matching elegance

This is one of those moments where OCaml's design shines. In C, this function would be a chain of strcmp calls or a giant if-else ladder. In Python, it would be a dictionary lookup or another if-elif chain. In OCaml, it is the most natural thing in the world — a pattern match on strings. The compiler even checks for exhaustiveness: if you add a keyword to the language but forget to add it to keyword_token, OCaml won't warn you here (since strings are not a closed type), but the parser will fail to compile when it tries to produce a token that does not exist.

The Lexer Module

The lexer itself is a module that wraps a position, the source text, and produces tokens on demand. We structure it as a function that takes the source text and returns a list of tokens (each annotated with its span):

lexer.ml — the full lexer
type spanned = Token.t * Span.t

type state = {
  source: string;
  pos:    int ref;
}

let make source = { source; pos = ref 0 }

let peek s =
  if !(s.pos) >= String.length s.source then None
  else Some s.source.[!(s.pos)]

let advance s =
  let c = s.source.[!(s.pos)] in
  s.pos := !(s.pos) + 1;
  c

let skip_whitespace s =
  while match peek s with
        | Some (' ' | '\t' | '\n' | '\r') -> advance s; true
        | _ -> false
  do done

let lex_number s start =
  while match peek s with
        | Some ('0'..'9' | '.') -> advance s; true
        | _ -> false
  do done;
  let text = String.sub s.source start (!(s.pos) - start) in
  if String.contains text '.' then
    Float (float_of_string text)
  else
    Int (int_of_string text)

let lex_ident s start =
  while match peek s with
        | Some ('a'..'z' | 'A'..'Z' | '0'..'9' | '_') -> advance s; true
        | _ -> false
  do done;
  let text = String.sub s.source start (!(s.pos) - start) in
  match keyword_token text with
  | Some tok -> tok
  | None    -> Ident text

let lex_string s =
  let buf = Buffer.create 16 in
  while match peek s with
        | Some '"' -> advance s; false
        | Some c     -> Buffer.add_char buf c; advance s; true
        | None       -> failwith "unterminated string"
  do done;
  Str (Buffer.contents buf)

let next s =
  skip_whitespace s;
  let start = !(s.pos) in
  match peek s with
  | None -> (Eof, Span.make start start)
  | Some c ->
    let tok =
      match c with
      | '0'..'9'       -> lex_number s start
      | 'a'..'z' | 'A'..'Z' | '_' -> lex_ident s start
      | '"'             -> advance s; lex_string s
      | '+'             -> advance s; Plus
      | '-'             -> advance s;
          if peek s = Some '>' then (advance s; Arrow)
          else Minus
      | '*'             -> advance s; Star
      | '/'             -> advance s; Slash
      | '('             -> advance s; LParen
      | ')'             -> advance s; RParen
      | '{'             -> advance s; LBrace
      | '}'             -> advance s; RBrace
      | '['             -> advance s; LBrack
      | ']'             -> advance s; RBrack
      | ':'             -> advance s; Colon
      | ';'             -> advance s; Semicol
      | ','             -> advance s; Comma
      | '.'             -> advance s; Dot
      | '@'             -> advance s; At

          else Eq
      | '!'             -> advance s;
          if peek s = Some '=' then (advance s; Neq)
          else Bang
      | '<'             -> advance s;
          if peek s = Some '=' then (advance s; Leq)
          else Lt
      | '>'             -> advance s;

          else Gt
      | _               -> failwith (Printf.sprintf "unexpected char: %c" c)
    in
    (tok, Span.make start !(s.pos))

    let tok, span = next s in
    match tok with
    | Eof -> List.rev ((tok, span) :: acc)

src/lexer.rs — the full lexer
pub struct Lexer {
    source: Vec<u8>,
    pos: usize,
}

pub type Spanned = (Token, Span);

impl Lexer {
    pub fn new(source: &str) -> Self {
        Self { source: source.bytes().collect(), pos: 0 }
    }

    fn peek(&self) -> Option<u8> {
        if self.pos >= self.source.len() { None }
        else { Some(self.source[self.pos]) }
    }

    fn advance(&mut self) -> u8 {
        let c = self.source[self.pos];
        self.pos += 1;
        c
    }

    fn skip_whitespace(&mut self) {
        while let Some(b) = self.peek() {
            if b.is_ascii_whitespace() { self.advance(); }
            else { break; }
        }
    }

    fn lex_number(&mut self, start: usize) -> Token {
        while let Some(b) = self.peek() {
            if b.is_ascii_digit() || b == b'.' { self.advance(); }
            else { break; }
        }
        let text = std::str::from_utf8(&self.source[start..self.pos]).unwrap();
        if text.contains('.') {
            Token::Float(text.parse().unwrap())
        } else {
            Token::Int(text.parse().unwrap())
        }
    }

    fn lex_ident(&mut self, start: usize) -> Token {
        while let Some(b) = self.peek() {
            if b.is_ascii_alphanumeric() || b == b'_' { self.advance(); }
            else { break; }
        }
        let text = std::str::from_utf8(&self.source[start..self.pos]).unwrap();
        keyword_token(text).unwrap_or_else(|| Token::Ident(text.to_string()))
    }

    fn next(&mut self) -> Spanned {
        self.skip_whitespace();
        let start = self.pos;
        let tok = match self.peek() {
            None => Token::Eof,
            Some(c) => match c {
                b'0'..=b'9'          => self.lex_number(start),
                b'a'..=b'z' | b'A'..=b'Z' | b'_' => self.lex_ident(start),
                b'"' => { self.advance(); self.lex_string() }
                b'+' => { self.advance(); Token::Plus }
                b'-' => { self.advance();
                    if self.peek() == Some(b'>') { self.advance(); Token::Arrow }
                    else { Token::Minus } }
                b'=' => { self.advance();
                    if self.peek() == Some(b'=') { self.advance(); Token::EqEq }
                    else { Token::Eq } }
                b'!' => { self.advance();
                    if self.peek() == Some(b'=') { self.advance(); Token::Neq }
                    else { Token::Bang } }
                // ... remaining single- and multi-char tokens
                _ => panic!("unexpected char: {}", c as char),
            },
        };
        (tok, Span::new(start, self.pos))
    }

    pub fn tokenize(&mut self) -> Vec<Spanned> {
        let mut tokens = Vec::new();
        loop {
            let (tok, span) = self.next();
            tokens.push((tok.clone(), span));
            if tok == Token::Eof { break; }
        }
        tokens
    }
}
State as a record with a ref

Notice that the OCaml lexer state uses a record with a pos ref rather than a mutable field. OCaml records are immutable by default, so we use a ref for the single mutable cell. This is a deliberate choice — it makes the state explicit and avoids hidden mutation. Rust uses &mut self everywhere, which is equally explicit but differently structured.

How the Lexer Works

The lexer is a character-at-a-time state machine. The core loop:

  1. Skip whitespace — advance past spaces, tabs, newlines, carriage returns.
  2. Peek at the next character — if none, emit Eof.
  3. Dispatch on the character class:
    • Digits → call lex_number, which reads a sequence of digits and an optional decimal point.
    • Letters or underscore → call lex_ident, which reads an identifier, then checks via keyword_token whether it is a keyword.
    • Quote → call lex_string to read a string literal.
    • Operators and punctuation → match the specific character, with lookahead for two-character operators like == and ->.
    • Anything else → error: unexpected character.
  4. Return the token with its span — the start position was recorded before reading, the end position is the current position after reading.

The tokenize function wraps this in a loop that accumulates tokens until Eof is reached. Note the accumulator pattern: we prepend each token to a list (O(1)), then reverse at the end. This is idiomatic OCaml — building lists forward requires appending, which is O(n). Building backward and reversing is O(n) total.

Extending the Lexer: Hex, Octal, and Binary Literals

A practical lexer must handle more than decimal integers. Let us extend lex_number to support hexadecimal (0xFF), octal (0o77), and binary (0b1010) literals. The key insight is to look for the prefix before falling into the digit-reading loop:

lexer.ml — extended lex_number
let lex_number s start =
  let base, is_float =
    if peek s = Some '0' then
      advance s;
      match peek s with
      | Some ('x' | 'X') -> advance s; 16, false
      | Some ('o' | 'O') -> advance s; 8,  false
      | Some ('b' | 'B') -> advance s; 2,  false
      | _ -> 10, true
    else
      10, true
  in
  while match peek s with
        | Some ('0'..'9' | 'a'..'f' | 'A'..'F') -> advance s; true
        | Some '.' when is_float -> advance s; true
        | _ -> false
  do done;
  let text = String.sub s.source start (!(s.pos) - start) in
  if base <> 10 then
    Int (int_of_string text)
  else if String.contains text '.' then
    Float (float_of_string text)
  else
    Int (int_of_string text)

fn lex_number(&mut self, start: usize) -> Token {
    let (base, is_float) =
        if self.peek() == Some(b'0') {
            self.advance();
            match self.peek() {
                Some(b'x' | b'X') => { self.advance(); (16, false) }
                Some(b'o' | b'O') => { self.advance(); (8,  false) }
                Some(b'b' | b'B') => { self.advance(); (2,  false) }
                _ => (10, true),
            }
        } else {
            (10, true)
        };
    while let Some(b) = self.peek() {
        let ok = b.is_ascii_digit()
             || (base == 16 && (b == b'a'..=b'f' || b == b'A'..=b'F'))
             || (is_float && b == b'.');
        if ok { self.advance(); } else { break; }
    }
    let text = std::str::from_utf8(&self.source[start..self.pos]).unwrap();
    if base != 10 || !text.contains('.') {
        Token::Int(i64::from_str_radix(&text[2..], base).unwrap())
    } else {
        Token::Float(text.parse().unwrap())
    }
}
Pattern matching with when guards

The OCaml version uses a when guard: Some '.' when is_float. This is an OCaml pattern guard — a boolean condition that must be true for the pattern to match. Guards let you express conditions that cannot be captured by patterns alone. Here, we only treat . as a float character when we are in decimal mode (after 0 we temporarily set is_float to false because 0xDEAD.BEEF is not valid).

Lexer Design Decisions

Several design decisions in this lexer are worth discussing explicitly:

No separate token list type. We use (Token.t * Span.t) list directly. OCaml's type system makes this readable — there is no need for a wrapper type. The tuple expresses everything: a token and where it came from.

No lexer generator. We are writing the lexer by hand rather than using ocamllex (OCaml's lexer generator). Why? Because for a small language, a hand-written lexer is simpler, more transparent, and easier to debug. The regular expressions that power lexer generators are powerful, but the character-at-a-time approach gives us complete control. When our language grows, we can switch to ocamllex — the token type and the interface do not change.

Error handling via exceptions. The lexer uses failwith for unrecoverable errors (unterminated strings, unexpected characters). This is a pragmatic choice for a pedagogical compiler. A production compiler would use proper error types, but for now, exceptions are sufficient. We will improve error handling in later chapters.

failwith is not the end

Using failwith means the lexer crashes on invalid input. For a real compiler, you would want recovery — skip the bad character and continue to find more errors. We leave this as an exercise. The lexer is the first place where the compiler interacts with imperfect human input, and it is worth spending time on robust error recovery.

The Problem

The lexer turned raw text into tokens. Now we need to turn those tokens into a tree — an abstract syntax tree (AST) — that captures the grammatical structure of the program. The parser is where the flat token stream becomes a hierarchical representation.

Tokens: [Func, Ident("main"), LParen, TyI32, RParen, LBrace, Let, Ident("x"), Colon, TyI32, Eq, Int(42), Semicol, Return, Ident("x"), Semicol, RBrace, Eof] AST: FuncDecl "main" ├─ RetType: TyI32 └─ Body: ├─ Let "x" TyI32 (Int 42) └─ Return (Var "x")

We use a recursive descent parser — one function per grammar rule, each consuming tokens and building AST nodes. For expressions, we use Pratt parsing, which handles operator precedence through a table-driven approach that is remarkably elegant in OCaml.

The Grammar

Before writing code, we define the grammar. Here is our language in informal BNF:

program       ::= decl*

decl          ::= func_decl
func_decl     ::= "func" ident "(" param_list ")" type? block
param_list    ::= (ident ":" type ("," ident ":" type)*)?
block         ::= "{" stmt* "}"
stmt          ::= let_stmt | if_stmt | while_stmt | return_stmt | expr_stmt
let_stmt      ::= "let" ident ":" type "=" expr ";"
if_stmt       ::= "if" "(" expr ")" block ("else" block)?
while_stmt    ::= "while" "(" expr ")" block
return_stmt   ::= "return" expr ";"
expr_stmt     ::= expr ";"
expr          ::= assignment
assignment    ::= ident "=" expr | logical_or
logical_or    ::= logical_and ("||" logical_and)*
logical_and   ::= equality ("&&" equality)*
equality      ::= comparison (("==" | "!=") comparison)*
comparison    ::= term (("<" | ">" | "<=" | ">=") term)*
term          ::= factor (("+" | "-") factor)*
factor        ::= unary (("*" | "/") unary)*
unary         ::= ("!" | "-") unary | call
call          ::= primary ("(" arg_list ")")*
arg_list      ::= expr ("," expr)*
primary       ::= int | float | string | "true" | "false"
                | ident | "(" expr ")" | "cast" "(" type "," expr ")"
type          ::= "i32" | "i64" | "u32" | "u64" | "i8" | "u8"
                | "bool" | "void" | "[" int "]" type | "@" type

This grammar encodes precedence through its layers. The deeper you go, the tighter the binding. primary is the tightest (parentheses, literals, identifiers), and assignment is the loosest. Each layer calls the next, so 1 + 2 * 3 parses as 1 + (2 * 3) because factor (which handles * and /) is called from within term (which handles + and -).

Grammar layers and precedence

In traditional BNF, higher precedence means tighter binding, and you express it by nesting grammar rules. Multiplication binds tighter than addition, so factor is called from within term. This is the standard technique, and it works well for languages with a small number of precedence levels. For languages with many levels (like C with ~15 levels), Pratt parsing is cleaner — we use it here.

Recursive Descent: The Top Level

The parser is a module with an inner state that tracks position in the token stream. Each function consumes as many tokens as its grammar rule requires and returns an AST node.

pub struct Parser {
    tokens: Vec<Spanned>,
    pos: usize,
}

impl Parser {
    pub fn new(tokens: Vec<Spanned>) -> Self {
        Self { tokens, pos: 0 }
    }

    fn peek(&self) -> Option<&Token> {
        if self.pos >= self.tokens.len() { None }
        else { Some(&self.tokens[self.pos].0) }
    }

    fn advance(&mut self) -> Spanned {
        let tok = self.tokens[self.pos].clone();
        self.pos += 1;
        tok
    }

    fn expect(&mut self, expected: &Token) -> Spanned {
        let (tok, span) = self.advance();
        if &tok != expected {
            panic!("expected {:?} at {}", expected, span.start);
        }
        (tok, span)
    }

    fn check(&self, tok: &Token) -> bool {
        self.peek() == Some(tok)
    }
}

The state is minimal: an array of tokens and a mutable position. The core functions — peek, advance, expect, check — are the parser's primitives. The entire parser is built on these four operations, much like the lexer was built on peek and advance at the character level.

Mutually recursive parsing functions

OCaml's let rec ... and ... syntax is essential here — parsing functions are naturally mutually recursive. parse_program calls parse_func_decl, which calls parse_param_list, parse_type_opt, and parse_block, which in turn calls parse_stmt, which calls parse_expr, and so on. The and keyword lets us define them all in a single mutually recursive group. This is far cleaner than forward declarations in C or arranging definitions in dependency order.

Pratt Expression Parsing

Expressions are where parsing gets interesting. The classic approach — one function per precedence level — scales poorly. Pratt parsing (also called "top-down operator precedence") uses a table that maps each token to its binding power and a handler function. The result is compact, extensible, and beautiful.

The idea is simple: every token has a binding power (precedence). A token with higher binding power "attracts" the expression on its right more strongly. The core Pratt loop reads a "prefix" expression, then repeatedly attempts to read more with an "infix" handler, stopping when the next token's binding power is too low.

parser.ml — Pratt expression parsing
let binding_power = function
  | Eq       -> 10
  | EqEq | Neq  -> 20
  | Lt | Gt | Leq | Geq -> 30
  | Plus | Minus  -> 40
  | Star | Slash  -> 50
  | _            -> 0

let rec parse_expr s min_bp =
  let lhs = parse_prefix s in
  parse_postfix s min_bp lhs

and parse_prefix s =
  match peek s with
  | Some (Int _) ->
      let Int n, span = advance s in (IntLit n, span)
  | Some (Float _) ->
      let Float f, span = advance s in (FloatLit f, span)
  | Some (Str _) ->
      let Str sval, span = advance s in (StrLit sval, span)
  | Some True  -> let _, span = advance s in (BoolLit true,  span)
  | Some False -> let _, span = advance s in (BoolLit false, span)
  | Some (Ident _) ->
      let Ident name, span = advance s in (Var name, span)
  | Some LParen ->
      let _, lspan = advance s in
      let expr = parse_expr s 0 in
      let _, rspan = expect s RParen in
      (expr, Span.merge lspan rspan)
  | Some Minus ->
      let _, span = advance s in
      let rhs, rspan = parse_expr s 60 in
      (Unary (Neg, rhs), Span.merge span rspan)
  | Some Bang ->
      let _, span = advance s in
      let rhs, rspan = parse_expr s 60 in
      (Unary (Not, rhs), Span.merge span rspan)
  | Some Cast ->
      parse_cast s
  | _ ->
      failwith "unexpected token in expression"

and parse_postfix s min_bp lhs =
  match peek s with
  | Some tok when binding_power tok > min_bp ->
      let (_, span) = advance s in
      let rhs, rspan = parse_expr s (binding_power tok) in
      let op = tok_to_binop tok in
      let new_lhs = (BinOp (fst lhs, op, rhs),
                     Span.merge (snd lhs) rspan) in
      parse_postfix s min_bp new_lhs
  | Some LParen ->
      parse_call s lhs
  | _ -> lhs

fn binding_power(tok: &Token) -> u32 {
    match tok {
        Token::Eq             => 10,
        Token::EqEq | Token::Neq => 20,
        Token::Lt  | Token::Gt
      | Token::Leq | Token::Geq => 30,
        Token::Plus  | Token::Minus => 40,
        Token::Star  | Token::Slash => 50,
        _ => 0,
    }
}

fn parse_expr(parser: &mut Parser, min_bp: u32) -> SpannedExpr {
    let mut lhs = parse_prefix(parser);
    loop {
        let tok = match parser.peek() {
            Some(t) => t.clone(),
            None => break,
        };
        if binding_power(&tok) <= min_bp { break; }
        match tok {
            Token::Eq => { parser.advance();
                let (rhs, rspan) = parse_expr(parser, binding_power(&tok) - 1);
                lhs = (Expr::Assign(Box::new(lhs.0), Box::new(rhs)),
                       Span::merge(&lhs.1, &rspan)); }
            Token::Plus | Token::Minus
          | Token::Star | Token::Slash
          | Token::EqEq | Token::Neq
          | Token::Lt | Token::Gt
          | Token::Leq | Token::Geq => { parser.advance();
                let (rhs, rspan) = parse_expr(parser, binding_power(&tok));
                lhs = (Expr::BinOp(Box::new(lhs.0), tok_to_binop(&tok), Box::new(rhs)),
                       Span::merge(&lhs.1, &rspan)); }
            Token::LParen => { lhs = parse_call(parser, lhs); }
            _ => break,
        }
    }
    lhs
}

The key insight of Pratt parsing is in parse_postfix (OCaml) and the loop in parse_expr (Rust): after reading a prefix expression, we look at the next token. If it is an infix operator with binding power greater than the current minimum, we consume it, parse the right-hand side with the operator's binding power as the new minimum, and loop. This creates the correct left-to-right, precedence-aware parse tree.

Notice how binding_power is a pure function mapping tokens to integers. In OCaml, the function keyword creates an anonymous function with pattern matching on the argument. The result reads like a table — clean, concise, and exhaustive. Adding a new operator means adding a token constructor, a binding power entry, and a case in tok_to_binop.

        Token::Minus => BinOp::Sub,
        Token::Star  => BinOp::Mul,

The separation is clean — the token type is a parser concern, and the binop type is an AST concern. The tok_to_binop function is the bridge.

Parsing Blocks and Statements

Statements follow a straightforward pattern: peek at the next token, dispatch to the appropriate handler, or fall back to expression-statement:

    | Some RBrace -> List.rev acc

  | Some Let   -> parse_let s

pub fn parse_block(parser: &mut Parser) -> SpannedBlock {
    let (_, lspan) = parser.expect(&Token::LBrace);
    let mut stmts = Vec::new();
    loop {
        match parser.peek() {
            Some(Token::RBrace) => break,
            Some(Token::Eof) => panic!("unterminated block"),
            _ => stmts.push(parse_stmt(parser)),
        }
    }
    let (_, rspan) = parser.expect(&Token::RBrace);
    (stmts, Span::merge(&lspan, &rspan))
}

pub fn parse_stmt(parser: &mut Parser) -> SpannedStmt {
    match parser.peek() {
        Some(Token::Let)   => parse_let(parser),
        Some(Token::If)    => parse_if(parser),
        Some(Token::While) => parse_while(parser),
        Some(Token::Return) => parse_return(parser),
        _ => {
            let expr = parse_expr(parser, 0);
            parser.expect(&Token::Semicol);
            (Stmt::Expr(Box::new(expr.0)), expr.1)
        }
    }
}
Pattern matching as dispatch

The parse_stmt function is a textbook use of pattern matching as control flow. Each branch handles one statement kind. The default branch (expression statement) falls through to the expression parser. This is clean, linear, and extensible: adding a new statement kind means adding a new branch. The compiler's exhaustiveness checking means that if you ever refactor the statement type, the parser will tell you exactly which branches need updating.

Every parser function follows the same pattern: consume tokens in sequence, assemble the AST node, attach a merged span. The result is a tree where every node knows its source location.

The Parser Works Like a State Monad

If you squint, the parser functions look like they are threading state through a computation. Each function takes a state and returns a result plus a (potentially modified) state. The modification is via the ref cell, which makes it mutable rather than purely functional, but the structure is the same.

In a purely functional setting, parse_expr would return state -> expr * state. This is the state monad pattern. We elide the full monadic structure for simplicity — mutable reference cells are pragmatic and idiomatic in OCaml when the state is simple and local. But the conceptual link is there: the parser is a stateful computation over the token stream, and each function transforms the state.

Parser combinators

If this monadic structure appeals to you, investigate OCaml's parser combinator libraries (like angstrom or mparse). They make the state threading explicit and compose parsers from smaller parsers using operators. For this compiler, a hand-written recursive descent parser is the right balance of clarity and power, but combinators are a rich area to explore.

pub fn parse_if(parser: &mut Parser) -> SpannedStmt {
    let (_, if_span) = parser.expect(&Token::If);
    parser.expect(&Token::LParen);
    let (cond, _) = parse_expr(parser, 0);
    parser.expect(&Token::RParen);
    let (then_block, tspan) = parse_block(parser);
    let (else_block, end_span) = match parser.peek() {
        Some(Token::Else) => {
            parser.expect(&Token::Else);
            let (block, espan) = parse_block(parser);
            (Some(block), espan)
        }
        _ => (None, tspan),
    };
    (Stmt::If { cond: Box::new(cond), then_block, else_block },
     Span::merge(&if_span, &end_span))
}

pub fn parse_while(parser: &mut Parser) -> SpannedStmt {
    let (_, wspan) = parser.expect(&Token::While);
    parser.expect(&Token::LParen);
    let (cond, _) = parse_expr(parser, 0);
    parser.expect(&Token::RParen);
    let (body, bspan) = parse_block(parser);
    (Stmt::While { cond: Box::new(cond), body },
     Span::merge(&wspan, &bspan))
}

Notice the pattern: consume a keyword, consume a parenthesized expression, consume a block, optionally consume an else clause. The option type for else_block captures the grammatical optionality directly. In Rust, this is Option<Vec<Stmt>>; in OCaml, it is stmt list option. Both express the same idea.

Parsing Error Recovery

The current parser uses failwith on any error, which crashes the compiler on the first invalid token. A robust parser needs error recovery — skip tokens until a synchronization point (like ; or }) and continue to find more errors.

We leave this as an exercise, but the key insight is: at synchronization points, the parser knows what tokens it expects. If it encounters an unexpected token, it skips forward until it finds something it recognizes, reports the error, and resumes. The result is a parser that produces multiple errors per invocation, which is dramatically more useful than a parser that stops at the first mistake.

The Central Data Structure

The abstract syntax tree (AST) is the spine of the compiler. Every phase before parsing produces input for it; every phase after parsing consumes it. The lexer produces tokens, the parser builds the AST, and the semantic analyzer, optimizer, and code generator all walk it.

An AST is a tree representation of the program's grammatical structure. It is "abstract" because it omits syntactic details that are not meaningful for analysis — semicolons, parentheses, braces. The concrete syntax let x: i32 = 1 + 2; becomes something like:

LetStmt ├─ name: "x" ├─ ty: TyI32 └─ init: BinOp ├─ lhs: IntLit(1) ├─ op: Add └─ rhs: IntLit(2)

Notice what is missing: the let keyword, the colon, the equals sign, the semicolon. Those are concrete syntax — they guide the parser but carry no semantic content. The AST captures only what matters: what the program does, not how it was written.

Read the OCaml type definition slowly. Each constructor is a node kind. The arguments to each constructor are the children of that node. BinOp of expr * binop * expr means a binary operation node has three children: left expression, operator, right expression. stmt list means a block is a list of statements. ty option means a return type is optional (functions can omit it, defaulting to void).

The clarity is not accidental. ML's type system was designed for exactly this kind of tree representation. Every other representation — abstract base classes in C++, interfaces in Java, traits in Rust — is a workaround. OCaml gives you the real thing.

Recursive Types and Mutability

Notice that expr and stmt are defined independently but refer to each other: stmt has an Expr of expr variant, and in a richer language, expr might contain inline blocks. OCaml handles this naturally because type definitions can be mutually recursive using type ... and ....

Also notice that Let in stmt uses a inline record syntax — Let of { name: ident; ty: ty; init: expr }. This is OCaml 4.03+ feature that gives us named fields for variant constructors. Before this feature, you had to define a separate record type. Now the constructor carries its own labeled fields. This is perfect for AST nodes with multiple children — you do not have to remember positional argument order.

Rust cannot do inline records in enums and uses struct fields or tuple variants instead. OCaml's approach is cleaner for ASTs because every node is self-documenting.

Walking the AST

The AST is useless if we cannot traverse it. Consider a function that collects all variable references in an expression:

ast.ml — collecting variable references
let rec vars_of_expr acc = function
  | Var name     -> name :: acc
  | IntLit _ | FloatLit _ | StrLit _ | BoolLit _ -> acc
  | BinOp (l, _, r) ->
      acc |> vars_of_expr l |> vars_of_expr r
  | Unary (_, e)   -> vars_of_expr acc e
  | Assign (l, r)  -> acc |> vars_of_expr l |> vars_of_expr r
  | Call (_, args) ->

  | Cast (_, e)    -> vars_of_expr acc e
  | ArrayIdx (a, i) ->

  | Let { init; _ }        -> vars_of_expr acc init
  | If { cond; then_block; else_block } ->
      let acc = vars_of_expr acc cond in
      let acc = List.fold_left vars_of_stmt acc then_block in
      (match else_block with
       | Some b -> List.fold_left vars_of_stmt acc b
       | None   -> acc)
  | While { cond; body }    ->
      let acc = vars_of_expr acc cond in
      List.fold_left vars_of_stmt acc body
  | Return e -> vars_of_expr acc e
  | Expr e   -> vars_of_expr acc e

pub fn vars_of_expr(expr: &Expr) -> Vec<Ident> {
    match expr {
        Expr::Var(name) => vec![name.clone()],
        Expr::IntLit(_) | Expr::FloatLit(_)
      | Expr::StrLit(_) | Expr::BoolLit(_) => vec![],
        Expr::BinOp(l, _, r) => {
            let mut v = vars_of_expr(l);
            v.extend(vars_of_expr(r));
            v
        }
        Expr::Unary(_, e) => vars_of_expr(e),
        Expr::Assign(l, r) => {
            let mut v = vars_of_expr(l);
            v.extend(vars_of_expr(r));
            v
        }
        Expr::Call(_, args) => {
            args.iter().flat_map(vars_of_expr).collect()
        }
        Expr::Cast(_, e) => vars_of_expr(e),
        Expr::ArrayIdx(a, i) => {
            let mut v = vars_of_expr(a);
            v.extend(vars_of_expr(i));
            v
        }
    }
}

This is the canonical pattern for working with ASTs: a recursive function that pattern-matches on each variant, processes the children recursively, and combines the results. The OCaml version uses an accumulator parameter passed via the |> operator (forward pipe), which reads naturally left-to-right. Each branch is concise, exhaustive, and obviously correct.

The forward pipe |> is worth noting: acc |> vars_of_expr l |> vars_of_expr r means "take acc, pass it to vars_of_expr l, pass the result to vars_of_expr r". This reads much more naturally than the nested function call version: vars_of_expr r (vars_of_expr l acc).

Exhaustiveness is a correctness guarantee

When you add a new variant to expr — say, StructAccess — the OCaml compiler will warn you about every match that does not handle it. The warnings are not optional; they are the compiler telling you that your semantic analysis, your pretty-printer, your code generator, and every other AST-walking function is now incomplete. Fix them all, and you can be confident that the new feature is properly integrated. No runtime bugs from forgotten dispatch cases. This is the power of algebraic data types in a compiler.

The Type Type

The ty type is itself an algebraic data type, separate from the parser's token set. This is important: types in the AST are abstract. They are not "the token i32" — they are the concept of a 32-bit integer. The parser converts I32 tokens to TyI32 constructors; the semantic analyzer works with the ty type directly.

This separation means the type system is uniform regardless of how types are written in source code. Whether the source says i32, int32_t, or Int32, the AST internally uses TyI32. The concrete syntax is a parser concern; the AST is concerned only with meaning.

The Program and Declarations

A program is simply a list of declarations. Currently only FuncDecl exists, but the decl type is a variant precisely so that we can add GlobalDecl, StructDecl, ImportDecl, or anything else without changing the top-level structure. Every function that processes a program now only needs to handle the cases that exist, and the compiler will warn us when we add new ones.

Designing the AST: Philosophy and Tradeoffs

AST design is a balancing act. Every design decision has downstream consequences for every compiler phase that walks the tree. Here are the key axes of the design space:

Fat vs. thin nodes. A "fat" AST node carries all information that any phase might need — spans, inferred types, symbol table references, optimization flags. A "thin" AST node carries only what the parser produces. Our AST is deliberately thin: each node carries its syntactic children and nothing more. Type information, scoping data, and other analysis results are stored separately. This keeps the AST definition simple and makes it easy to reason about. The tradeoff is that analysis phases must carry their own auxiliary data structures (symbol tables, type environments) rather than piggybacking on the AST.

Desugaring in the AST. Should for loops exist as a first-class AST node, or should the parser desugar them into while loops? We chose to keep the AST minimal — if a construct can be expressed in terms of simpler constructs, we desugar it early. This reduces the number of cases in every traversal. The cost is that error messages refer to the desugared form, which may not match the source code the user wrote.

Position storage. We store spans alongside AST nodes using tuples: (decl * Span.t). An alternative is to store the span directly inside each node. We chose the tuple approach because it keeps the AST type clean — the span is metadata, not part of the tree. When we compare ASTs for equality (in tests), we can ignore spans by comparing only the first element.

These three axes — fat vs. thin, desugaring strategy, position storage — are the fundamental design decisions that shape every AST. There are no universally correct answers, only tradeoffs. Our choices favor clarity and maintainability over performance. As the compiler grows, you may find reasons to move in a different direction.

The AST as a Contract

One way to think about the AST is as a contract between the frontend (lexer + parser) and the backend (analysis + code generation). The frontend promises to produce a well-formed AST. The backend promises to handle every node kind. The contract is enforced by OCaml's type system — the frontend can only produce nodes that exist, and the backend must handle every variant (or explicitly decide not to, catching the inevitable runtime error).

This contract is more valuable than any documentation. Documentation rots. Type definitions do not. When you read an OCaml AST type definition, you are reading the authoritative, always-up-to-date specification of what the compiler's intermediate representation looks like. Every other description — comments, design documents, blog posts — is secondary.

5 · Does This Make Sense?

The parser gave us a beautiful AST. Every if has its branches, every while has its body, every expression nests just right. The tree is syntactically valid. But is it meaningful?

Consider this program — and notice how naturally we reach for a variant type: 

func main(i32) {
  let x: bool = 42;
}

The parser will happily build an AST for this. It doesn't know that 42 isn't a bool. It doesn't care. The parser's job is just to check that the tokens form a valid tree according to our grammar.

Now it's our job — the semantic analyzer's job — to determine whether the tree actually means something. In OCaml terms, we're about to write a recursive function over the AST that returns either a typed tree or an error, making the proof of well-typedness explicit in the types.

Two phases, one pass? Some textbooks separate "semantic analysis" into distinct phases: binding (connecting identifiers to declarations) and type checking (checking type constraints). We'll do them together in a single pass over the AST, because they interleave naturally: to check the type of an identifier, we need to look it up in the symbol table.

Phase 1: Building a Symbol Table

Before we can check types, we need to know what every name refers to. A symbol table is a data structure that maps names to their declarations. Think of it as a dictionary the compiler carries around: "What is x? Oh, it's a variable of type i32, declared on line 12."

Our language has three kinds of scopes:

  • Global scope — function names and global variables.
  • Function scope — parameter names.
  • Block scope — local variables declared with let inside a block.

Scopes nest. A variable declared in an outer block is visible in inner blocks, but a variable in an inner block shadows the outer one. When we enter a new block, we push a new scope. When we leave it, we pop it off.

OCaml's module system lets us encapsulate the symbol table cleanly. Unlike languages where a symbol table is just a hash map, in OCaml we can wrap it in a module that enforces scope discipline at the type level. The SymbolTable module hides the representation; you interact with it only through enter_scope, exit_scope, insert, and lookup. This is the same pattern Appel uses in his ML textbook, and it's idiomatic OCaml.

Here's what our symbol table looks like:

symbol.ml
type symbol =
  | Variable of typ
  | Function of {
      params: typ list;
      ret: typ;
    }

module SymbolTable : sig
  type t
  val create : unit -> t
  val enter_scope : t -> unit
  val exit_scope : t -> unit
  val insert : t -> string -> symbol -> unit
  val lookup : t -> string -> symbol option
end = struct
  type scope = (string, symbol) Hashtbl.t
  type t = { mutable scopes: scope list }

  let create () = { scopes = [Hashtbl.create 16] }

  let enter_scope tbl =
    tbl.scopes <- Hashtbl.create 16 :: tbl.scopes

  let exit_scope tbl =
    match tbl.scopes with
    | [] -> failwith "scope imbalance"
    | _ :: rest -> tbl.scopes <- rest

  let insert tbl name sym =
    match tbl.scopes with
    | [] -> failwith "no active scope"
    | scope :: _ -> Hashtbl.replace scope name sym

  let lookup tbl name =
    let rec go = function
      | [] -> None
      | scope :: rest ->
        match Hashtbl.find_opt scope name with
        | Some _ as s -> s
        | None -> go rest
    in
    go tbl.scopes
end

use std::collections::HashMap;
use crate::ast::*;

pub enum Symbol {
  Variable(Type),
  Function { params: Vec<Type>, ret: Type },
}

pub struct SymbolTable {
  scopes: Vec<HashMap<String, Symbol>>,
}

impl SymbolTable {
  pub fn new() -> Self {
    SymbolTable { scopes: vec![HashMap::new()] }
  }

  pub fn enter_scope(&mut self) {
    self.scopes.push(HashMap::new());
  }

  pub fn exit_scope(&mut self) {
    self.scopes.pop();
  }

  pub fn insert(&mut self, name: &str, sym: Symbol) {
    if let Some(scope) = self.scopes.last_mut() {
      scope.insert(name.to_string(), sym);
    }
  }

  pub fn lookup(&self, name: &str) -> Option<&Symbol> {
    for scope in self.scopes.iter().rev() {
      if let Some(sym) = scope.get(name) {
        return Some(sym);
      }
    }
    None
  }
}

The key insight here is the lookup function: it walks scopes from innermost to outermost, returning the first match. This gives us lexical (static) scoping — a name always refers to the innermost declaration in scope. In OCaml, the go inner function is a perfect example of using a recursive helper to traverse a list until we find what we need.

Populating the Symbol Table

We populate the symbol table with a top-level walk. For each function declaration, we insert the function name and its signature. Then we walk the body to find local declarations. We do this before type checking so that when we encounter a function call or variable reference, we already know about it.

This two-phase approach — collect first, check second — is a pattern that appears throughout compiler construction. In OCaml, we implement it with two separate recursive functions: collect_decls and check_func. The types keep us honest: collect_decls returns a symbol_table, which check_func reads but does not modify.

pub fn build(ast: &Program) -> SymbolTable {
  let mut table = SymbolTable::new();

  for decl in &ast.decls {
    match decl {
      Decl::Func(func) => {
        let params =
          func.params.iter()
            .map(|p| p.ty.clone())
            .collect();
        table.insert(&func.name,
          Symbol::Func { params, ret: func.ret_ty.clone() });
        table.enter_scope();
        for param in &func.params {
          table.insert(&param.name,
            Symbol::Var(param.ty.clone()));
        }
        walk_let_decls(&mut table, &func.body);
        table.exit_scope();
      }
      Decl::Var(v) => {
        table.insert(&v.name,
          Symbol::Var(v.ty.clone()));
      }
    }
  }
  table
}

fn walk_let_decls(table: &mut SymbolTable, block: &Block) {
  table.enter_scope();
  for stmt in &block.stmts {
    match stmt {
      Stmt::Let(_, name, ty, _) =>
        table.insert(name, Symbol::Var(ty.clone())),
      Stmt::Block(b) => walk_let_decls(table, b),
      _ => {}
    }
  }
  table.exit_scope();
}

Phase 2: Type Checking

With a populated symbol table, we can now walk the AST a second time and check every expression and statement for type correctness. This is the heart of semantic analysis — and it's where OCaml's pattern matching really shines.

Every expression in our language has a type. The type checker's job is to determine that type and verify it's used correctly. The rules are straightforward:

Type Checking Rules (informal): Integer literal → declared type (i32 by default) Bool literal → bool Identifier → type from symbol table Binary op (+, -, etc) → both operands must be same numeric type Comparison (==, <) → both operands same type, result is bool Function call → args match param types, result is return type Assignment → RHS type matches declared LHS type Cast → any numeric type to any numeric type If/while condition → must be bool Return → type must match function's return type

Here's the type checker. Notice how each match case reads like a specification rule — that's the beauty of pattern matching on variant types:

  | Bool _   -> Type.Bool

    | Some (Function _) ->

pub struct Checker {
  symbols: SymbolTable,
  pub errors: Vec<Error>,
  current_ret: Type,
}

impl Checker {
  pub fn new(symbols: SymbolTable) -> Self {
    Checker { symbols, errors: vec![], current_ret: Type::Void }
  }

  pub fn check_expr(&mut self, expr: &Expr) -> Type {
    match expr {
      Expr::Int(_) => Type::I32,
      Expr::Bool(_) => Type::Bool,

      Expr::Ident(span, name) => {
        match self.symbols.lookup(name) {
          Some(Symbol::Var(ty)) => ty.clone(),
          Some(Symbol::Func(_)) => {
            self.error(span, "expected variable, found function");
            Type::I32
          }
          None => {
            self.error(span,
              &format("undefined variable `{}`", name));
            Type::I32
          }
        }
      }

      Expr::Binary(span, op, lhs, rhs) => {
        let lt = self.check_expr(lhs);
        let rt = self.check_expr(rhs);
        if lt != rt {
          self.error(span,
            &format("type mismatch: expected {}, found {}",
              lt, rt));
        }
        match op {
          BinOp::Add | BinOp::Sub
          | BinOp::Mul | BinOp::Div => lt,
          BinOp::Eq | BinOp::Lt | BinOp::Gt => Type::Bool,
          BinOp::And | BinOp::Or => {
            if lt != Type::Bool {
              self.error(span, "logical op needs bool operands");
            }
            Type::Bool
          }
        }
      }

      Expr::Call(span, name, args) => {
        match self.symbols.lookup(name) {
          Some(Symbol::Func{params, ret}) => {
            if args.len() != params.len() {
              self.error(span,
                &format("function `{}` expects {} args, got {}",
                  name, params.len(), args.len()));
            }
            for (i, (arg, param_ty))
                in args.iter().zip(&params).enumerate() {
              let arg_ty = self.check_expr(arg);
              if &arg_ty != param_ty {
                self.error(&arg.span,
                  &format("arg {}: expected {}, found {}",
                    i+1, param_ty, arg_ty));
              }
            }
            ret.clone()
          }
          _ => {
            self.error(span,
              &format("undefined function `{}`", name));
            Type::I32
          }
        }
      }

      Expr::Cast(_, to_ty, inner) => {
        let inner_ty = self.check_expr(inner);
        if !is_numeric(&inner_ty) || !is_numeric(to_ty) {
          self.error(&inner.span,
            "cast only between numeric types");
        }
        to_ty.clone()
      }
    }
  }
}

      let expr_ty = self.check_expr(expr);
      if expr_ty != self.current_ret {
        self.error(span,
          &format("return type mismatch: expected {}, found {}",
            self.current_ret, expr_ty));
      }
    }

    Stmt::If(span, cond, then_blk, else_blk) => {
      let cond_ty = self.check_expr(cond);
      if cond_ty != Type::Bool {
        self.error(span, "if condition must be bool");
      }
      self.symbols.enter_scope();
      for s in &then_blk.stmts { self.check_stmt(s); }
      self.symbols.exit_scope();
      if let Some(else_b) = else_blk {
        self.symbols.enter_scope();
        for s in &else_b.stmts { self.check_stmt(s); }
        self.symbols.exit_scope();
      }
    }

    Stmt::While(span, cond, body) => {
      let cond_ty = self.check_expr(cond);
      if cond_ty != Type::Bool {
        self.error(span, "while condition must be bool");
      }
      self.symbols.enter_scope();
      for s in &body.stmts { self.check_stmt(s); }
      self.symbols.exit_scope();
    }

    Stmt::Expr(_, expr) => { self.check_expr(expr); }
    Stmt::Block(b) => {
      self.symbols.enter_scope();
      for s in &b.stmts { self.check_stmt(s); }
      self.symbols.exit_scope();
    }
  }
}

The "Aha" Moment

Here's what we've realized: the type checker is where most of the "compile error" user experience lives. The parser catches syntax errors. The lexer catches token errors. But the type checker catches the errors that real programmers make every day.

In OCaml, there's a deeper insight here. The check_expr function we wrote — returning a typ — is essentially an algorithmic implementation of a type system. Appel calls this "syntax-directed translation": the structure of the type rules mirrors the structure of the AST. Every match case is one typing rule.

The symbol table + type checker pattern is universal. Every statically typed language has one. Rust has its typeck pass. OCaml has its typing module (a masterpiece of algorithmic type checking with side-effectful unification). Our tiny checker is the same idea, scaled down but structurally identical.

What's Next

We have a semantically valid program. Every name is bound. Every type checks out. The AST is correct and meaningful. Now we face a new problem: the AST is tied to the source language. It has If nodes and While loops and function declarations. But the machine doesn't know about any of that. We need something in between — an intermediate representation that bridges the gap.

The Shape of a Program Bridging the Gap

From chapter 06

Building a Compiler OCaml Edition

6 · Bridging the Gap

Think about what we have: a beautiful, type-checked AST. Every node represents some concept from our source language — If, While, FuncDecl, Let. These are perfect for analysis. We can ask "what type does this expression have?" or "what variables does this function reference?"

Now think about what the machine wants: flat sequences of instructions. Load a value into a register. Add two registers. Compare. Jump if equal. The machine doesn't have if. It has cmp and jne. It doesn't have while. It has jmp and conditional branches.

We need a translator. Someone who speaks both "high-level source" and "low-level machine" fluently. That translator is an intermediate representation (IR).

Why not go directly from AST to assembly? We could. Early compilers did. But separating IR generation from assembly generation gives us enormous flexibility: we can optimize the IR, target multiple architectures from the same IR, and reason about programs at a level simpler than the full AST. Think of IR as the compiler's lingua franca.

The Problem with the AST

Let's look at a concrete example: 

func max(i32, a: i32, b: i32) {
  if (a > b) {
    return a;
  } else {
    return b;
  }
}

The AST for this is a tree:

FuncDecl "max" ret=i32 params=[a:i32, b:i32] └── Block └── If ├── cond: Binary Gt │ ├── Ident "a" │ └── Ident "b" ├── then: Block │ └── Return ─── Ident "a" └── else: Block └── Return ─── Ident "b"

This tree is great for analysis. But to generate code, we need to linearize it. We need to decide: what order do we evaluate things? Where do we store intermediate results? How do we encode the branch?

The IR answers all three questions. It's a linear representation of the computation. Every expression becomes a sequence of instructions that compute into virtual registers. Every control flow construct becomes basic blocks connected by branches.

Designing Our IR

OCaml's variant types make IR design a joy. Each IR instruction is a constructor of a variant type. The entire instruction set fits in one clean type definition, and the compiler's exhaustiveness checker ensures we never miss a case when processing the IR.

The key insight: virtual registers are an unlimited resource. Unlike real CPU registers (which number 16 on x86-64), we can create as many virtual registers as we want. The register allocator (later) will map them to real registers or spill them to the stack. For now, they're just integers. In OCaml, type vreg = int is all we need.

The IR Builder

The IR builder is a stateful object that carries a few things:

  • A counter for generating fresh virtual registers (%0, %1, ...)
  • A counter for generating fresh block labels
  • The current block we're emitting instructions into
  • A map from variable names to stack slot indices
  • The list of completed blocks (for the function we're building)
OCaml's recursive modules make the SSA builder elegant. If we wanted to build SSA form directly (where every variable is assigned exactly once), we'd need mutually recursive functions: the builder calls the expression translator, which calls the block builder. OCaml's module rec and and let us express this naturally. The Braun et al. algorithm for SSA construction translates directly to mutually recursive OCaml functions.

Here's how we build IR for expressions. Each expression returns the virtual register that holds its result:

pub struct IrBuilder {
  vreg_counter: usize,
  block_counter: usize,
  slots: HashMap<String, usize>,
  pub current_block: IrBlock,
  pub blocks: Vec<IrBlock>,
}

impl IrBuilder {
  pub fn new() -> Self {
    IrBuilder {
      vreg_counter: 0,
      block_counter: 0,
      slots: HashMap::new(),
      current_block: IrBlock::new("entry".into()),
      blocks: vec![],
    }
  }

  fn fresh_vreg(&mut self) -> VReg {
    let r = self.vreg_counter;
    self.vreg_counter += 1;
    r
  }

  fn fresh_block(&mut self) -> String {
    let b = format!("b{}", self.block_counter);
    self.block_counter += 1;
    b
  }

  fn emit(&mut self, instr: IrInstr) {
    self.current_block.instrs.push(instr);
  }

  fn finish_block(&mut self, term: IrTerm) -> String {
    self.current_block.term = term;
    let label = self.current_block.label.clone();
    let block = std::mem::replace(
      &mut self.current_block,
      IrBlock::new(self.fresh_block()),
    );
    self.blocks.push(block);
    label
  }
}

Expressions to IR

Translating an expression is straightforward: we emit instructions that compute the value into a fresh virtual register. The pattern match is exhaustive — if we add a new expr variant, OCaml's compiler will warn us about the missing case:

ir_builder.ml — expressions
let rec expr_to_ir b = function
  | Int (_, n) ->
    let r = fresh_vreg b in
    emit b (Const (r, n)); r

  | Bool (_, v) ->
    let r = fresh_vreg b in
    emit b (Const (r, if v then 1L else 0L)); r

  | Ident (_, name) ->
    let slot = Hashtbl.find b.slots name in
    let r = fresh_vreg b in
    emit b (Load (r, slot)); r

  | Binary (_, op, lhs, rhs) ->
    let l = expr_to_ir b lhs in
    let r = expr_to_ir b rhs in
    let d = fresh_vreg b in
    let instr = match op with
      | Add -> Add (d, l, r) | Sub -> Sub (d, l, r)
      | Mul -> Mul (d, l, r) | Div -> Div (d, l, r)
      | Eq  -> Eq (d, l, r)  | Lt -> Lt (d, l, r)
      | Gt  -> Gt (d, l, r)  | And -> And (d, l, r)
      | Or  -> Or (d, l, r)
    in
    emit b instr; d

  | Call (_, name, args) ->
    let arg_regs = List.map (expr_to_ir b) args in
    let r = fresh_vreg b in
    emit b (Call (r, name, arg_regs)); r

  | Cast (_, _, inner) ->
    (* Cast is a no-op in IR — same bits *)
    expr_to_ir b inner

  | Assign (_, target, value) ->
    let val_reg = expr_to_ir b value in
    (match target with
    | Ident (_, name) ->
      let slot = Hashtbl.find b.slots name in
      emit b (Store (slot, val_reg))
    | _ -> failwith "assign target must be identifier");
    val_reg

fn expr_to_ir(&mut self, expr: &Expr) -> VReg {
  match expr {
    Expr::Int(_, val) => {
      let r = self.fresh_vreg();
      self.emit(IrInstr::Const(r, *val)); r
    }
    Expr::Bool(_, val) => {
      let r = self.fresh_vreg();
      self.emit(IrInstr::Const(r, if *val { 1 } else { 0 })); r
    }
    Expr::Ident(_, name) => {
      let slot = self.slots[name];
      let r = self.fresh_vreg();
      self.emit(IrInstr::Load(r, slot)); r
    }
    Expr::Binary(_, op, lhs, rhs) => {
      let l = self.expr_to_ir(lhs);
      let r = self.expr_to_ir(rhs);
      let d = self.fresh_vreg();
      let instr = match op {
        BinOp::Add => IrInstr::Add(d, l, r),
        BinOp::Sub => IrInstr::Sub(d, l, r),
        BinOp::Mul => IrInstr::Mul(d, l, r),
        BinOp::Div => IrInstr::Div(d, l, r),
        BinOp::Eq  => IrInstr::Eq(d, l, r),
        BinOp::Lt  => IrInstr::Lt(d, l, r),
        BinOp::Gt  => IrInstr::Gt(d, l, r),
        BinOp::And => IrInstr::And(d, l, r),
        BinOp::Or  => IrInstr::Or(d, l, r),
      };
      self.emit(instr); d
    }
    Expr::Call(_, name, args) => {
      let arg_regs = args.iter()
        .map(|a| self.expr_to_ir(a)).collect();
      let r = self.fresh_vreg();
      self.emit(IrInstr::Call(r, name.clone(), arg_regs)); r
    }
    Expr::Cast(.., inner) => self.expr_to_ir(inner),
    Expr::Assign(_, target, value) => {
      let val_reg = self.expr_to_ir(value);
      if let Expr::Ident(_, name) = target.as_ref() {
        let slot = self.slots[name];
        self.emit(IrInstr::Store(slot, val_reg));
      }
      val_reg
    }
  }
}

Statements and Control Flow

Statements are where things get interesting. A statement doesn't produce a value — it emits instructions and may split control flow into multiple blocks. The If and While cases are where the IR builder earns its keep:

ir_builder.ml — statements
let rec stmt_to_ir b = function
  | Let (_, name, _, init) -> (
    match init with
    | None -> ()
    | Some init_expr ->
      let val = expr_to_ir b init_expr in
      let slot = Hashtbl.length b.slots in
      Hashtbl.replace b.slots name slot;
      emit b (Store (slot, val))
  )

  | Return (_, expr) ->
    let val = expr_to_ir b expr in
    finish_block b (Ret val)

  | If (_, cond, then_blk, else_blk) ->
    let cond_reg = expr_to_ir b cond in
    let then_lab = fresh_block b in
    let else_lab = fresh_block b in
    let after_lab = fresh_block b in

    finish_block b (BrCond (cond_reg, then_lab, else_lab));

    b.current_block <- { b.current_block with label = then_lab };
    List.iter (stmt_to_ir b) then_blk.stmts;
    if not (ended_with_return then_blk) then
      finish_block b (Br after_lab);

    | None ->

      finish_block b (Br after_lab)

    b.current_block <- { b.current_block with label = after_lab }

  | While (_, cond, body) ->

    let body_lab = fresh_block b in
    let after_lab = fresh_block b in

    b.current_block <- { b.current_block with label = header_lab };
    let cond_reg = expr_to_ir b cond in

src/ir_builder.rs — statements
fn stmt_to_ir(&mut self, stmt: &Stmt) {
  match stmt {
    Stmt::Let(_, name, _, init) => {
      if let Some(init) = init {
        let val = self.expr_to_ir(init);
        let slot = self.slots.len();
        self.slots.insert(name.clone(), slot);
        self.emit(IrInstr::Store(slot, val));
      }
    }

    Stmt::Return(_, expr) => {
      let val = self.expr_to_ir(expr);
      self.finish_block(IrTerm::Ret(val));
    }

    Stmt::If(_, cond, then_blk, else_blk) => {
      let cond_reg = self.expr_to_ir(cond);
      let then_lab = self.fresh_block();
      let else_lab = self.fresh_block();
      let after_lab = self.fresh_block();

      self.finish_block(IrTerm::BrCond(cond_reg, then_lab, else_lab));

      self.current_block.label = then_lab;
      for s in &then_blk.stmts { self.stmt_to_ir(s); }
      if !ended_with_return(then_blk) {
        self.finish_block(IrTerm::Br(after_lab));
      }

      if let Some(blk) = else_blk {
        self.current_block.label = else_lab;
        for s in &blk.stmts { self.stmt_to_ir(s); }
        if !ended_with_return(blk) {
          self.finish_block(IrTerm::Br(after_lab));
        }
      } else {
        self.current_block.label = else_lab;
        self.finish_block(IrTerm::Br(after_lab));
      }

      self.current_block.label = after_lab;
    }

    Stmt::While(_, cond, body) => {
      let header_lab = self.fresh_block();
      let body_lab = self.fresh_block();
      let after_lab = self.fresh_block();

      self.finish_block(IrTerm::Br(header_lab));

      self.current_block.label = header_lab;
      let cond_reg = self.expr_to_ir(cond);
      self.finish_block(IrTerm::BrCond(cond_reg, body_lab, after_lab));

      self.current_block.label = body_lab;
      for s in &body.stmts { self.stmt_to_ir(s); }
      if !ended_with_return(body) {
        self.finish_block(IrTerm::Br(header_lab));
      }

      self.current_block.label = after_lab;
    }

    Stmt::Expr(_, expr) => { self.expr_to_ir(expr); }
    Stmt::Block(b) => {
      for s in &b.stmts { self.stmt_to_ir(s); }
    }
  }
}

Walkthrough: func main(i32) { return 1 + 2; }

Let's trace the IR builder on a tiny function. The builder starts in b0 (entry). It processes the Return statement:

  1. Evaluate 1: emit %0 = const 1
  2. Evaluate 2: emit %1 = const 2
  3. Evaluate 1 + 2: emit %2 = add %0, %1
  4. Finish block with ret %2

The tree has been flattened into a linear sequence of instructions. Every intermediate value has a name (a virtual register). The control flow is trivial — just one block that returns.

The Problem

Here's a piece of IR our compiler might produce:

  entry:

The Stack Frame

When a function runs, it sets up a stack frame. The convention we follow (System V AMD64 ABI): 

; Prologue
push rbp          ; save caller's base pointer
mov  rbp, rsp     ; our frame starts here
sub  rsp, <N>     ; allocate N bytes for locals

; ... function body ...

; Epilogue
mov  rsp, rbp     ; restore stack pointer
pop  rbp          ; restore caller's base pointer
ret               ; return to caller

After the prologue, every local variable lives at a negative offset from rbp: [rbp-8], [rbp-16], and so on. The frame size N must be a multiple of 16 to maintain stack alignment.

Strategy: Spill Everything

The simplest possible register allocation strategy is: don't use registers at all. Put every virtual register on the stack. Access them through rbp-relative loads and stores.

This is called "spilling everything." It's naive — every operation becomes a load-compute-store sequence — but it's correct, and it's the foundation every real register allocator improves upon.

Linear Scan Register Allocation

"Spill everything" produces terrible code. A better approach is linear scan register allocation, which assigns virtual registers to physical registers when possible and only spills when necessary.

The algorithm is surprisingly simple:

  1. Compute the live interval for each virtual register (first use to last use).
  2. Walk intervals in order of start point.
  3. Maintain a set of free physical registers.
  4. When an interval starts, assign a free register. If none is free, spill the interval that ends latest.
  5. When an interval ends, free its register.

In OCaml, this maps naturally to operations on lists and sets. The Set.Make(Int) functor gives us type-safe register sets:

codegen.ml — linear scan (simplified)
module RegSet = Set.Make(Int)

type interval = {
  vreg: int;
  start_: int;
  end_: int;
}

let linear_scan intervals num_phys_regs =
  let sorted = List.sort (fun a b -> Int.compare a.start_ b.start_) intervals in
  let free = ref (RegSet.of_list (List.init num_phys_regs Fun.id)) in
  let active = ref [] in
  let assignment = ref VRegMap.empty in

  List.iter (fun interval ->
    (* Expire old intervals *)
    active := List.filter (fun (_, end_) -> end_ > interval.start_)
      (!active);
    List.iter (fun (reg, _) -> free := RegSet.add reg (!free))
      (List.filter (fun (_, end_) -> end_ <= interval.start_)
        (!active));

    (* Assign or spill *)
    if not (RegSet.is_empty (!free)) then begin
      let reg = RegSet.min_elt (!free) in
      free := RegSet.remove reg (!free);
      active := (reg, interval.end_) :: !active;
      assignment := VRegMap.add interval.vreg reg (!assignment)
    end else begin
      (* Spill: no free register *)
      spill interval
    end
  ) sorted;
  !assignment

use std::collections::{HashMap, HashSet};

struct Interval { vreg: u32, start: usize, end: usize }

fn linear_scan(
  intervals: Vec<Interval>,
  num_phys: usize,
) -> HashMap<u32, usize> {
  let mut sorted = intervals;
  sorted.sort_by_key(|i| i.start);

  let mut free: HashSet<usize> = (0..num_phys).collect();
  let mut active: Vec<(usize, usize)> = vec![];
  let mut assignment = HashMap::new();

  for interval in &sorted {
    // Expire old intervals
    active.retain(|&(reg, end)| {
      if end <= interval.start {
        free.insert(reg);
        false
      } else { true }
    });

    if let Some(&reg) = free.iter().next() {
      free.remove(&reg);
      active.push((reg, interval.end));
      assignment.insert(interval.vreg, reg);
    } else {
      spill(interval); // spill to stack
    }
  }
  assignment
}

  mov  [rbp-24], rax ; store comparison result

  ; conditional branch
  mov  rax, [rbp-24]
  cmp  rax, 0
  jne  .b1
  jmp  .b2

  ; then block: return a
  .b1:
  mov  rax, [rbp-8]
  mov  rsp, rbp
  pop  rbp
  ret

  ; else block: return b
  .b2:
  mov  rax, [rbp-16]
  mov  rsp, rbp
  pop  rbp
  ret
This is terrible code — and that's okay! Every unnecessary load and store is an opportunity for optimization. The register allocator will eliminate many of them. The key is that the generated code is correct. We can make it fast incrementally.

Calling Convention Details

The System V AMD64 ABI calling convention governs how functions pass arguments and return values. When our code generator emits a call, it must follow these rules:

  • Integer arguments 1-6 go in rdi, rsi, rdx, rcx, r8, r9 (in that order).
  • Additional arguments (7+) are pushed onto the stack in right-to-left order.
  • Return value goes in rax.
  • Stack alignment: RSP must be 16-byte aligned before the call instruction.
  • Caller-saved registers: rax, rcx, rdx, rsi, rdi, r8-r11 — the caller must save them if needed after the call.
  • Callee-saved registers: rbx, rbp, r12-r15 — the callee must save and restore them if used.

Our current "spill everything" strategy handles this trivially: everything is on the stack, so no registers need saving. But when we implement proper register allocation, we'll need to track which registers are live across calls and spill them around call sites. This is where liveness analysis becomes essential.

Complete Walkthrough: while Loop Codegen

Let's trace the full codegen for a function with a loop: 

func sum_to_n(i32, n: i32) {
  let i: i32 = 0;
  let total: i32 = 0;
  while (i < n) {
    total = total + i;
    i = i + 1;
  }
  return total;
}

The IR builder produces a control flow graph with 4 blocks: entry, header, body, and after. The code generator then emits: 

global sum_to_n
sum_to_n:
  push rbp
  mov  rbp, rsp
  sub  rsp, 32       ; 4 vregs × 8 bytes

  ; entry: i = 0, total = 0
  mov  rax, 0
  mov  [rbp-8], rax   ; store i
  mov  rax, 0
  mov  [rbp-16], rax  ; store total
  jmp  .header

  ; header: check i < n
  .header:
  mov  rax, [rbp-8]   ; load i
  mov  rcx, [rbp-24]  ; load n
  cmp  rax, rcx
  setl al
  movzx rax, al
  mov  [rbp-32], rax  ; store comparison
  cmp  rax, 0
  jne  .body
  jmp  .after

  ; body: total += i; i++
  .body:
  mov  rax, [rbp-16]  ; load total
  add  rax, [rbp-8]   ; add i
  mov  [rbp-16], rax  ; store total
  mov  rax, [rbp-8]   ; load i
  add  rax, 1
  mov  [rbp-8], rax   ; store i
  jmp  .header

  ; after: return total
  .after:
  mov  rax, [rbp-16]  ; load total
  mov  rsp, rbp
  pop  rbp
  ret

The pattern is clear: every variable access becomes a load-modify-store sequence through rax. A real register allocator would keep i and total in registers r12 and r13, eliminating all those memory operations. But the code is correct — and correctness is our foundation.

The "Aha" Moment

Here's what we've done: we've crossed the final abstraction boundary. The IR — with its virtual registers and basic blocks — has become real assembly code that a CPU can execute. Every IR instruction maps to one or more x86-64 instructions. Every virtual register becomes a stack slot or a physical register. Every basic block becomes a labeled section of assembly.

In OCaml, the instruction selection function is a testament to the power of pattern matching. The match expression reads like a spec: "For Add(d, a, b), emit mov, add, mov." There's no ceremony, no visitor pattern, no method dispatch. Just a direct, exhaustive mapping from IR to assembly.

Exercises

1. Improve the stack allocator

Our stack allocator assigns 8 bytes to every virtual register. But booleans only need 1 byte. Modify the allocator to track the size of each value and allocate only as many bytes as needed. This reduces frame size and memory traffic.

2. Full linear scan register allocation

Replace the "spill everything" strategy with full linear scan register allocation. Implement live interval computation, register assignment, and spill code insertion. Test it on functions with many virtual registers.

3. Peephole optimization on the assembly

Add a peephole pass that scans the emitted assembly for redundant patterns. For example, mov rax, [rbp-8]; mov [rbp-8], rax is a no-op. Remove it. Other patterns: consecutive mov instructions to the same register, or cmp rax, 0; jne (can be jne with rax test).

4. Argument passing via registers

The System V ABI passes the first 6 integer arguments in registers (rdi, rsi, rdx, rcx, r8, r9). Modify the code generator to use registers for function arguments instead of pushing everything onto the stack. This is a major performance win.

5. Add floating-point support

Add f64 type support to the compiler. This requires new IR instructions (FAdd, FSub, etc.), new codegen patterns using SSE registers (xmm0-xmm15), and new type checking rules. The OCaml match makes adding new instruction variants straightforward — just add a new constructor and a new case.

We can generate assembly text from source code. But assembly text isn't an executable. It's a string — characters in a file. How do we cross that final gap from .asm to a program that actually runs?

The answer involves two tools we don't build ourselves: an assembler (NASM) and a linker (cc/ld). Our job is to orchestrate them. We become the conductor of a pipeline that starts with a .spark source file and ends with a binary that the OS can load and execute.

The Driver Pipeline

The driver is the top-level orchestrator. It connects every piece of the compiler into a single coherent flow:

Source file (.spark) │ ▼ 1. Read source into string │ ▼ 2. Lex → token list │ ▼ 3. Parse → AST │ ▼ 4. Type check → annotated AST │ ▼ 5. Build IR → ir_module │ ▼ 6. Codegen → assembly string │ ▼ 7. Write assembly to .asm file │ ▼ 8. Invoke NASM: nasm -f elf64 file.asm -o file.o │ ▼ 9. Invoke linker: cc -o file file.o -no-pie │ ▼ Executable (file)

Error Handling Strategy

A good compiler doesn't stop at the first error. It collects as many errors as possible, then presents them to the user all at once. Our OCaml compiler uses a simple pattern: each phase collects errors in a mutable list, and the driver checks the error count before proceeding to the next phase.

Read source → string │ ▼ Lex → tokens (collect errors) │ succ │ fail ▼ ▼ Parse → AST Print errors & exit │ succ │ fail ▼ ▼ Type check → AST Print errors & exit │ succ │ fail ▼ ▼ Build IR → module Print errors & exit │ succ ▼ Codegen → assembly string │ ▼ Write .asm, assemble, link
Error-aware types in OCaml In OCaml, we use ('a, error list) Result.t for operations that can fail, or simply maintain a mutable errors: error list field in each phase object. Multi-error accumulation is crucial: finding all bugs in one compile pass is much faster than fixing one at a time.

Here's the main function of the compiler, connecting every piece. OCaml's Unix module provides process management, and Printf / eprintf handle output:

let read_file path =
  let ch = open_in path in
  let n = in_channel_length ch in
  let s = really_input_string ch n in
  close_in ch;
  s

let report_errors file errors =
  List.iter (fun e ->
    Errors.report file e.span e.msg

let () =
  let args = Sys.argv in
  if Array.length args < 2 then begin
    eprintf "usage: %s <file.spark>\n" args.(0);
    exit 1
  end;

  let path = args.(1) in

  (* 1. Read source *)
  let source = try read_file path
    with e -> eprintf "error reading %s: %s\n" path
                  (Printexc.to_string e); exit 1
  in
  let file = Span.make_source_file path source in

  (* 2. Lex *)
  let lexer = Lexer.create file in
  let tokens = Lexer.lex lexer in
  if lexer.errors <> [] then
    report_errors file lexer.errors;

  (* 3. Parse *)
  let parser = Parser.create tokens file in
  let ast = Parser.parse parser in
  if parser.errors <> [] then
    report_errors file parser.errors;

  (* 4. Symbol table + Type check *)
  let symbols = Symbol.collect_decls ast in
  let checker = Check.create symbols in
  let typed_ast = Check.check_program checker ast in
  if checker.errors <> [] then
    report_errors file checker.errors;

  (* 5. Build IR *)
  let ir_builder = IrBuilder.create () in
  let ir_module = IrBuilder.build_module ir_builder typed_ast in
  if ir_builder.errors <> [] then
    report_errors file ir_builder.errors;

  (* 6. Codegen *)
  let emitter = Emitter.create () in
  Codegen.codegen_module emitter ir_module;

  (* 7. Write assembly *)
  let asm_path = Filename.chop_extension path ^ ".asm" in
  Emitter.write_to_file emitter asm_path;
  printf "wrote %s\n" asm_path;

  (* 8. Assemble with NASM *)
  let obj_path = Filename.chop_extension path ^ ".o" in
  let nasm_status = Sys.command
    (Printf.sprintf "nasm -f elf64 %s -o %s" asm_path obj_path)
  in
  if nasm_status <> 0 then exit 1;

  (* 9. Link with cc *)
  let bin_path = Filename.chop_extension path in
  let link_status = Sys.command
    (Printf.sprintf "cc -o %s %s -no-pie" bin_path obj_path)
  in
  if link_status <> 0 then exit 1;

  printf "built %s\n" bin_path

fn main() {
  let args: Vec<String> = env::args().collect();
  if args.len() < 2 {
    eprintln!("usage: {} <file.spark>", args[0]);
    process::exit(1);
  }

  let path = &args[1];
  let source = fs::read_to_string(path)
    .unwrap_or_else(|e| { eprintln!("error: {}", e); process::exit(1); });

  let file = span::SourceFile::new(path, &source);
  let mut errors: Vec<span::Error> = vec![];

  let mut lexer = lexer::Lexer::new(&file);
  let tokens = lexer.lex();
  if !lexer.errors.is_empty() { report_and_exit(&file, &lexer.errors); }

  let mut parser = parser::Parser::new(&tokens, &file);
  let ast = parser.parse();
  if !parser.errors.is_empty() { report_and_exit(&file, &parser.errors); }

  let symbols = symbol::build(&ast);
  let mut checker = check::Checker::new(symbols);
  let ast = checker.check(ast);
  if !checker.errors.is_empty() { report_and_exit(&file, &checker.errors); }

  let mut ir_builder = ir::Builder::new();
  let ir_module = ir_builder.build(&ast);
  if !ir_builder.errors.is_empty() { report_and_exit(&file, &ir_builder.errors); }

  let mut emitter = emitter::Emitter::new();
  codegen::codegen_module(&ir_module, &mut emitter);

  let asm_path = path.replace(".spark", ".asm");
  emitter.write(&asm_path).unwrap();
  println!("wrote {}", asm_path);

  let obj_path = path.replace(".spark", ".o");
  let status = process::Command::new("nasm")
    .args(&["-f", "elf64", &asm_path, "-o", &obj_path])
    .status().expect("nasm not found");
  if !status.success() { process::exit(1); }

  let bin = path.replace(".spark", "");
  let status = process::Command::new("cc")
    .args(&["-o", &bin, &obj_path, "-no-pie"])
    .status().expect("cc not found");
  if !status.success() { process::exit(1); }

  println!("built {}", bin);
}

Ideas for the Reader

Here are concrete projects, ordered from easiest to hardest:

Add One Feature at a Time

The best advice: add the thing you miss most. Is it frustrating that you can't print output? Add a print function that calls printf via the C FFI. Do you want to write recursive functions? Make sure the calling convention handles it correctly (it should already).

Each feature teaches you something new. Adding print teaches you about FFI. Adding arrays teaches you about memory layout. Adding closures teaches you about environment capture and heap allocation.

Port to a New Backend

Our compiler targets x86-64, but the IR is architecture-independent. Port the codegen module to a new target:

  • ARM64 (AArch64): Different registers (x0-x30), same concepts. Used in Apple Silicon, most phones.
  • RISC-V: The cleanest ISA. Only 47 base instructions. Registers x0-x31.
  • WebAssembly: A stack-based virtual ISA that runs in browsers. Completely different codegen model.
  • LLVM IR: Generate LLVM IR instead of assembly. LLVM handles register allocation and machine code for you. This is what Rust, Swift, and Clang do.

Porting requires rewriting only the codegen module. The frontend — lexer, parser, type checker, IR builder — stays exactly the same. This is the power of a well-designed IR.

Add an Optimization Pass

Our generated code is full of redundant loads and stores. Try these:

Constant folding: If the IR has Const(%0, 3); Const(%1, 4); Add(%2, %0, %1), replace it with Const(%2, 7). A single pass over the IR.

Dead store elimination: If a value is stored to a stack slot and never loaded before being overwritten, the first store is dead. Requires liveness analysis.

Peephole optimization: After codegen, scan the assembly for inefficient patterns. mov rax, [rbp-8]; mov [rbp-8], rax is a no-op — remove it.

In OCaml, each optimization is a functor: module Folded = ConstantFolding(BaseIR). Compose them: module OptIR = DCE(ConstantFolding(BaseIR)).

  • "SSA-based Compiler Design" — a tutorial by Braun, Buchwald, and Hack on the SSA construction algorithm we referenced in Chapter 7. The paper presents the algorithm in a language-agnostic way, but the OCaml implementation is only about 80 lines.
  • "Linear Scan Register Allocation" by Poletto and Sarkar. The paper that introduced the algorithm we sketched in Chapter 8. It's short, readable, and the results are immediately applicable.
  • LLVM (llvm.org) — the industry-standard compiler infrastructure. LLVM's IR is more sophisticated than ours, but the concepts are the same. The LLVM documentation is excellent, and the source is written in modern C++ with clear abstractions.
  • Cranelift — a code generator written in Rust. Designed for use cases like WebAssembly and JIT compilation. Its codegen pipeline is remarkably similar to what we built, but production-quality.

The Journey Ahead

Building a compiler is one of those rare projects that changes how you see every other program. After this, when you open a source file, you don't just see code — you see the pipeline: the lexer carving out tokens, the parser building trees, the type checker walking nodes, the IR builder linearizing control flow, the code generator emitting instructions, the assembler and linker producing bytes.

In OCaml, this pipeline felt natural because the language was designed for it. Variant types mirror the structure of syntax trees. Pattern matching makes tree traversal declarative. The module system scales with complexity. If you started this book knowing Rust or another language and learned OCaml along the way, you've gained not just a compiler, but a new way of thinking about programs.

What you build next is up to you. Add a feature. Port to ARM64. Implement SSA. Write a garbage collector. Make the compiler self-hosting. The framework is solid, the IR is clean, and the tests pass. Every direction is open.

The code is yours now The companion code in code/ocaml/ is your starting point. Modify it. Break it. Fix it. Extend it. The best way to learn compilers is to change a working compiler and see what happens. We've given you a working compiler. Now make it yours.

Thank you for reading. Thank you for building. And most of all — thank you for staying curious. The machine speaks because you took the time to learn its language.
