Building a Compiler OCaml Edition

3 · Parsing

The Problem

The lexer turned raw text into tokens. Now we need to turn those tokens into a tree — an abstract syntax tree (AST) — that captures the grammatical structure of the program. The parser is where the flat token stream becomes a hierarchical representation.

Tokens: [Func, Ident("main"), LParen, TyI32, RParen, LBrace, Let, Ident("x"), Colon, TyI32, Eq, Int(42), Semicol, Return, Ident("x"), Semicol, RBrace, Eof] AST: FuncDecl "main" ├─ RetType: TyI32 └─ Body: ├─ Let "x" TyI32 (Int 42) └─ Return (Var "x")

We use a recursive descent parser — one function per grammar rule, each consuming tokens and building AST nodes. For expressions, we use Pratt parsing, which handles operator precedence through a table-driven approach that is remarkably elegant in OCaml.

The Grammar

Before writing code, we define the grammar. Here is our language in informal BNF:

program       ::= decl*

decl          ::= func_decl
func_decl     ::= "func" ident "(" param_list ")" type? block
param_list    ::= (ident ":" type ("," ident ":" type)*)?
block         ::= "{" stmt* "}"
stmt          ::= let_stmt | if_stmt | while_stmt | return_stmt | expr_stmt
let_stmt      ::= "let" ident ":" type "=" expr ";"
if_stmt       ::= "if" "(" expr ")" block ("else" block)?
while_stmt    ::= "while" "(" expr ")" block
return_stmt   ::= "return" expr ";"
expr_stmt     ::= expr ";"
expr          ::= assignment
assignment    ::= ident "=" expr | logical_or
logical_or    ::= logical_and ("||" logical_and)*
logical_and   ::= equality ("&&" equality)*
equality      ::= comparison (("==" | "!=") comparison)*
comparison    ::= term (("<" | ">" | "<=" | ">=") term)*
term          ::= factor (("+" | "-") factor)*
factor        ::= unary (("*" | "/") unary)*
unary         ::= ("!" | "-") unary | call
call          ::= primary ("(" arg_list ")")*
arg_list      ::= expr ("," expr)*
primary       ::= int | float | string | "true" | "false"
                | ident | "(" expr ")" | "cast" "(" type "," expr ")"
type          ::= "i32" | "i64" | "u32" | "u64" | "i8" | "u8"
                | "bool" | "void" | "[" int "]" type | "@" type

This grammar encodes precedence through its layers. The deeper you go, the tighter the binding. primary is the tightest (parentheses, literals, identifiers), and assignment is the loosest. Each layer calls the next, so 1 + 2 * 3 parses as 1 + (2 * 3) because factor (which handles * and /) is called from within term (which handles + and -).

Grammar layers and precedence

In traditional BNF, higher precedence means tighter binding, and you express it by nesting grammar rules. Multiplication binds tighter than addition, so factor is called from within term. This is the standard technique, and it works well for languages with a small number of precedence levels. For languages with many levels (like C with ~15 levels), Pratt parsing is cleaner — we use it here.

Recursive Descent: The Top Level

The parser is a module with an inner state that tracks position in the token stream. Each function consumes as many tokens as its grammar rule requires and returns an AST node.

parser.ml — state and core utilities
type state = {
  tokens: (Token.t * Span.t) array;
  pos:    int ref;
}

let make tokens = { tokens; pos = ref 0 }

let peek s =
  if !(s.pos) >= Array.length s.tokens then None
  else Some (fst s.tokens.(!(s.pos)))

let advance s =
  let tok = s.tokens.(!(s.pos)) in
  s.pos := !(s.pos) + 1;
  tok

let expect s expected =
  let tok, span = advance s in
  if tok <> expected then
    failwith (Printf.sprintf "expected %s at %d"
      (Token.to_string expected) span.start);
  (tok, span)

let check s tok =
  match peek s with
  | Some t when t = tok -> true
  | _ -> false
src/parser.rs — state and core utilities
pub struct Parser {
    tokens: Vec<Spanned>,
    pos: usize,
}

impl Parser {
    pub fn new(tokens: Vec<Spanned>) -> Self {
        Self { tokens, pos: 0 }
    }

    fn peek(&self) -> Option<&Token> {
        if self.pos >= self.tokens.len() { None }
        else { Some(&self.tokens[self.pos].0) }
    }

    fn advance(&mut self) -> Spanned {
        let tok = self.tokens[self.pos].clone();
        self.pos += 1;
        tok
    }

    fn expect(&mut self, expected: &Token) -> Spanned {
        let (tok, span) = self.advance();
        if &tok != expected {
            panic!("expected {:?} at {}", expected, span.start);
        }
        (tok, span)
    }

    fn check(&self, tok: &Token) -> bool {
        self.peek() == Some(tok)
    }
}

The state is minimal: an array of tokens and a mutable position. The core functions — peek, advance, expect, check — are the parser's primitives. The entire parser is built on these four operations, much like the lexer was built on peek and advance at the character level.

Parsing Declarations

At the top level, a program is a sequence of declarations. Currently only function declarations exist, but the architecture is ready for more:

parser.ml — top-level parsing
let rec parse_program s =
  let rec loop acc =
    match peek s with
    | Some Func ->
        let decl = parse_func_decl s in
        loop (decl :: acc)
    | Some Eof ->
        List.rev acc
    | Some tok ->
        failwith ("unexpected token at top level")
    | None ->
        List.rev acc
  in
  loop []

and parse_func_decl s =
  let start_span = snd (expect s Func) in
  let (_, name_span) = advance s in
  let name = match fst (advance s) with
    | Ident n -> n
    | _ -> failwith "expected identifier"
  in
  expect s LParen;
  let params = parse_param_list s in
  expect s RParen;
  let ret_type = parse_type_opt s in
  let body = parse_block s in
  let end_span = snd body in
  (FuncDecl { name; params; ret_type; body = fst body },
   Span.merge start_span end_span)
src/parser.rs — top-level parsing
pub fn parse_program(parser: &mut Parser) -> Vec<SpannedDecl> {
    let mut decls = Vec::new();
    loop {
        match parser.peek() {
            Some(Token::Func) => decls.push(parse_func_decl(parser)),
            Some(Token::Eof) => break,
            Some(tok) => panic!("unexpected token at top level: {:?}", tok),
            None => break,
        }
    }
    decls
}

pub fn parse_func_decl(parser: &mut Parser) -> SpannedDecl {
    let start_span = parser.expect(&Token::Func).1;
    let name = match parser.advance().0 {
        Token::Ident(n) => n,
        _ => panic!("expected identifier"),
    };
    parser.expect(&Token::LParen);
    let params = parse_param_list(parser);
    parser.expect(&Token::RParen);
    let ret_type = parse_type_opt(parser);
    let (body, end_span) = parse_block(parser);
    (ast::Decl::FuncDecl { name, params, ret_type, body },
     Span::merge(&start_span, &end_span))
}
Mutually recursive parsing functions

OCaml's let rec ... and ... syntax is essential here — parsing functions are naturally mutually recursive. parse_program calls parse_func_decl, which calls parse_param_list, parse_type_opt, and parse_block, which in turn calls parse_stmt, which calls parse_expr, and so on. The and keyword lets us define them all in a single mutually recursive group. This is far cleaner than forward declarations in C or arranging definitions in dependency order.

Pratt Expression Parsing

Expressions are where parsing gets interesting. The classic approach — one function per precedence level — scales poorly. Pratt parsing (also called "top-down operator precedence") uses a table that maps each token to its binding power and a handler function. The result is compact, extensible, and beautiful.

The idea is simple: every token has a binding power (precedence). A token with higher binding power "attracts" the expression on its right more strongly. The core Pratt loop reads a "prefix" expression, then repeatedly attempts to read more with an "infix" handler, stopping when the next token's binding power is too low.

parser.ml — Pratt expression parsing
let binding_power = function
  | Eq       -> 10
  | EqEq | Neq  -> 20
  | Lt | Gt | Leq | Geq -> 30
  | Plus | Minus  -> 40
  | Star | Slash  -> 50
  | _            -> 0

let rec parse_expr s min_bp =
  let lhs = parse_prefix s in
  parse_postfix s min_bp lhs

and parse_prefix s =
  match peek s with
  | Some (Int _) ->
      let Int n, span = advance s in (IntLit n, span)
  | Some (Float _) ->
      let Float f, span = advance s in (FloatLit f, span)
  | Some (Str _) ->
      let Str sval, span = advance s in (StrLit sval, span)
  | Some True  -> let _, span = advance s in (BoolLit true,  span)
  | Some False -> let _, span = advance s in (BoolLit false, span)
  | Some (Ident _) ->
      let Ident name, span = advance s in (Var name, span)
  | Some LParen ->
      let _, lspan = advance s in
      let expr = parse_expr s 0 in
      let _, rspan = expect s RParen in
      (expr, Span.merge lspan rspan)
  | Some Minus ->
      let _, span = advance s in
      let rhs, rspan = parse_expr s 60 in
      (Unary (Neg, rhs), Span.merge span rspan)
  | Some Bang ->
      let _, span = advance s in
      let rhs, rspan = parse_expr s 60 in
      (Unary (Not, rhs), Span.merge span rspan)
  | Some Cast ->
      parse_cast s
  | _ ->
      failwith "unexpected token in expression"

and parse_postfix s min_bp lhs =
  match peek s with
  | Some tok when binding_power tok > min_bp ->
      let (_, span) = advance s in
      let rhs, rspan = parse_expr s (binding_power tok) in
      let op = tok_to_binop tok in
      let new_lhs = (BinOp (fst lhs, op, rhs),
                     Span.merge (snd lhs) rspan) in
      parse_postfix s min_bp new_lhs
  | Some LParen ->
      parse_call s lhs
  | _ -> lhs
src/parser.rs — Pratt expression parsing
fn binding_power(tok: &Token) -> u32 {
    match tok {
        Token::Eq             => 10,
        Token::EqEq | Token::Neq => 20,
        Token::Lt  | Token::Gt
      | Token::Leq | Token::Geq => 30,
        Token::Plus  | Token::Minus => 40,
        Token::Star  | Token::Slash => 50,
        _ => 0,
    }
}

fn parse_expr(parser: &mut Parser, min_bp: u32) -> SpannedExpr {
    let mut lhs = parse_prefix(parser);
    loop {
        let tok = match parser.peek() {
            Some(t) => t.clone(),
            None => break,
        };
        if binding_power(&tok) <= min_bp { break; }
        match tok {
            Token::Eq => { parser.advance();
                let (rhs, rspan) = parse_expr(parser, binding_power(&tok) - 1);
                lhs = (Expr::Assign(Box::new(lhs.0), Box::new(rhs)),
                       Span::merge(&lhs.1, &rspan)); }
            Token::Plus | Token::Minus
          | Token::Star | Token::Slash
          | Token::EqEq | Token::Neq
          | Token::Lt | Token::Gt
          | Token::Leq | Token::Geq => { parser.advance();
                let (rhs, rspan) = parse_expr(parser, binding_power(&tok));
                lhs = (Expr::BinOp(Box::new(lhs.0), tok_to_binop(&tok), Box::new(rhs)),
                       Span::merge(&lhs.1, &rspan)); }
            Token::LParen => { lhs = parse_call(parser, lhs); }
            _ => break,
        }
    }
    lhs
}

The key insight of Pratt parsing is in parse_postfix (OCaml) and the loop in parse_expr (Rust): after reading a prefix expression, we look at the next token. If it is an infix operator with binding power greater than the current minimum, we consume it, parse the right-hand side with the operator's binding power as the new minimum, and loop. This creates the correct left-to-right, precedence-aware parse tree.

Notice how binding_power is a pure function mapping tokens to integers. In OCaml, the function keyword creates an anonymous function with pattern matching on the argument. The result reads like a table — clean, concise, and exhaustive. Adding a new operator means adding a token constructor, a binding power entry, and a case in tok_to_binop.

Here is the companion tok_to_binop function that completes the picture:

parser.ml — tok_to_binop
let tok_to_binop = function
  | Plus  -> Add
  | Minus -> Sub
  | Star  -> Mul
  | Slash -> Div
  | EqEq  -> Eq
  | Neq   -> Neq
  | Lt    -> Lt
  | Gt    -> Gt
  | Leq   -> Leq
  | Geq   -> Geq
  | _     -> failwith "unexpected token as operator"
src/parser.rs — tok_to_binop
fn tok_to_binop(tok: &Token) -> BinOp {
    match tok {
        Token::Plus  => BinOp::Add,
        Token::Minus => BinOp::Sub,
        Token::Star  => BinOp::Mul,
        Token::Slash => BinOp::Div,
        Token::EqEq  => BinOp::Eq,
        Token::Neq   => BinOp::Neq,
        Token::Lt    => BinOp::Lt,
        Token::Gt    => BinOp::Gt,
        Token::Leq   => BinOp::Leq,
        Token::Geq   => BinOp::Geq,
        _ => panic!("unexpected token as operator: {:?}", tok),
    }
}

This function converts between two levels of representation: the token-level operators (what the lexer produced) and the AST-level operators (what the semantic analyzer will use). The separation is clean — the token type is a parser concern, and the binop type is an AST concern. The tok_to_binop function is the bridge.

Parsing Blocks and Statements

Statements follow a straightforward pattern: peek at the next token, dispatch to the appropriate handler, or fall back to expression-statement:

parser.ml — statements
and parse_block s =
  let _, lspan = expect s LBrace in
  let rec loop acc =
    match peek s with
    | Some RBrace -> List.rev acc
    | Some Eof    -> failwith "unterminated block"
    | _ ->
        let stmt = parse_stmt s in
        loop (stmt :: acc)
  in
  let stmts = loop [] in
  let _, rspan = expect s RBrace in
  (stmts, Span.merge lspan rspan)

and parse_stmt s =
  match peek s with
  | Some Let   -> parse_let s
  | Some If    -> parse_if s
  | Some While -> parse_while s
  | Some Return -> parse_return s
  | _ ->
      let expr = parse_expr s 0 in
      expect s Semicol;
      expr
src/parser.rs — statements
pub fn parse_block(parser: &mut Parser) -> SpannedBlock {
    let (_, lspan) = parser.expect(&Token::LBrace);
    let mut stmts = Vec::new();
    loop {
        match parser.peek() {
            Some(Token::RBrace) => break,
            Some(Token::Eof) => panic!("unterminated block"),
            _ => stmts.push(parse_stmt(parser)),
        }
    }
    let (_, rspan) = parser.expect(&Token::RBrace);
    (stmts, Span::merge(&lspan, &rspan))
}

pub fn parse_stmt(parser: &mut Parser) -> SpannedStmt {
    match parser.peek() {
        Some(Token::Let)   => parse_let(parser),
        Some(Token::If)    => parse_if(parser),
        Some(Token::While) => parse_while(parser),
        Some(Token::Return) => parse_return(parser),
        _ => {
            let expr = parse_expr(parser, 0);
            parser.expect(&Token::Semicol);
            (Stmt::Expr(Box::new(expr.0)), expr.1)
        }
    }
}
Pattern matching as dispatch

The parse_stmt function is a textbook use of pattern matching as control flow. Each branch handles one statement kind. The default branch (expression statement) falls through to the expression parser. This is clean, linear, and extensible: adding a new statement kind means adding a new branch. The compiler's exhaustiveness checking means that if you ever refactor the statement type, the parser will tell you exactly which branches need updating.

The Let Statement

As a concrete example, here is how we parse let x: i32 = 42;:

parser.ml — let statement
and parse_let s =
  let _, let_span = expect s Let in
  let name = match peek s with
    | Some (Ident n) -> let _, _ = advance s in n
    | _ -> failwith "expected identifier after let"
  in
  expect s Colon;
  let ty = parse_type s in
  expect s Eq;
  let init, init_span = parse_expr s 0 in
  let _, semi_span = expect s Semicol in
  (Let { name; ty; init }, Span.merge let_span semi_span)
src/parser.rs — let statement
pub fn parse_let(parser: &mut Parser) -> SpannedStmt {
    let (_, let_span) = parser.expect(&Token::Let);
    let name = match parser.peek() {
        Some(Token::Ident(n)) => { parser.advance(); n.clone() }
        _ => panic!("expected identifier after let"),
    };
    parser.expect(&Token::Colon);
    let ty = parse_type(parser);
    parser.expect(&Token::Eq);
    let (init, init_span) = parse_expr(parser, 0);
    let (_, semi_span) = parser.expect(&Token::Semicol);
    (Stmt::Let { name, ty, init: Box::new(init) },
     Span::merge(&let_span, &semi_span))
}

Every parser function follows the same pattern: consume tokens in sequence, assemble the AST node, attach a merged span. The result is a tree where every node knows its source location.

The Parser Works Like a State Monad

If you squint, the parser functions look like they are threading state through a computation. Each function takes a state and returns a result plus a (potentially modified) state. The modification is via the ref cell, which makes it mutable rather than purely functional, but the structure is the same.

In a purely functional setting, parse_expr would return state -> expr * state. This is the state monad pattern. We elide the full monadic structure for simplicity — mutable reference cells are pragmatic and idiomatic in OCaml when the state is simple and local. But the conceptual link is there: the parser is a stateful computation over the token stream, and each function transforms the state.

Parser combinators

If this monadic structure appeals to you, investigate OCaml's parser combinator libraries (like angstrom or mparse). They make the state threading explicit and compose parsers from smaller parsers using operators. For this compiler, a hand-written recursive descent parser is the right balance of clarity and power, but combinators are a rich area to explore.

Parsing If and While Statements

Control flow statements follow the same pattern. Let us look at parse_if and parse_while:

parser.ml — if and while
and parse_if s =
  let _, if_span = expect s If in
  expect s LParen;
  let cond, cspan = parse_expr s 0 in
  expect s RParen;
  let then_block, tspan = parse_block s in
  let else_block, end_span =
    match peek s with
    | Some Else ->
        let _, _ = expect s Else in
        let block, espan = parse_block s in
        (Some block, espan)
    | _ -> (None, tspan)
  in
  (If { cond; then_block; else_block },
   Span.merge if_span end_span)

and parse_while s =
  let _, wspan = expect s While in
  expect s LParen;
  let cond, cspan = parse_expr s 0 in
  expect s RParen;
  let body, bspan = parse_block s in
  (While { cond; body },
   Span.merge wspan bspan)
src/parser.rs — if and while
pub fn parse_if(parser: &mut Parser) -> SpannedStmt {
    let (_, if_span) = parser.expect(&Token::If);
    parser.expect(&Token::LParen);
    let (cond, _) = parse_expr(parser, 0);
    parser.expect(&Token::RParen);
    let (then_block, tspan) = parse_block(parser);
    let (else_block, end_span) = match parser.peek() {
        Some(Token::Else) => {
            parser.expect(&Token::Else);
            let (block, espan) = parse_block(parser);
            (Some(block), espan)
        }
        _ => (None, tspan),
    };
    (Stmt::If { cond: Box::new(cond), then_block, else_block },
     Span::merge(&if_span, &end_span))
}

pub fn parse_while(parser: &mut Parser) -> SpannedStmt {
    let (_, wspan) = parser.expect(&Token::While);
    parser.expect(&Token::LParen);
    let (cond, _) = parse_expr(parser, 0);
    parser.expect(&Token::RParen);
    let (body, bspan) = parse_block(parser);
    (Stmt::While { cond: Box::new(cond), body },
     Span::merge(&wspan, &bspan))
}

Notice the pattern: consume a keyword, consume a parenthesized expression, consume a block, optionally consume an else clause. The option type for else_block captures the grammatical optionality directly. In Rust, this is Option<Vec<Stmt>>; in OCaml, it is stmt list option. Both express the same idea.

Parsing Error Recovery

The current parser uses failwith on any error, which crashes the compiler on the first invalid token. A robust parser needs error recovery — skip tokens until a synchronization point (like ; or }) and continue to find more errors.

We leave this as an exercise, but the key insight is: at synchronization points, the parser knows what tokens it expects. If it encounters an unexpected token, it skips forward until it finds something it recognizes, reports the error, and resumes. The result is a parser that produces multiple errors per invocation, which is dramatically more useful than a parser that stops at the first mistake.

Exercises

Exercise 3.1 — Add a ternary expression

Add support for the ternary expression cond ? then_expr : else_expr. You will need to add a token for ?, add it to the binding power table, and implement the infix handler. The binding power should be lower than assignment. This exercises every layer of the Pratt parser.

Exercise 3.2 — Parse for loops

Add for (init; cond; update) { body } to the language. This requires a new statement kind, a new keyword, and a new parser function. Desugar the for loop into a while loop in the AST, or keep it as a first-class node — both are valid choices.

Exercise 3.3 — Error recovery

Implement error recovery in the parser. Define synchronization points (;, }, ), keywords). When the parser encounters an error, skip tokens until a synchronization point is found, report the error, and continue. The parser should return both a (potentially partial) AST and a list of errors.

Exercise 3.4 — Expression pretty-printer

Write a function Ast.string_of_expr that converts an expression AST node back into source text. This is called a "pretty-printer" and is useful for debugging — you can parse some code, then print it, and check that the output matches the input (modulo whitespace). Test it on expressions with various precedence levels to verify that your output adds parentheses correctly.

Exercise 3.5 — Compare with Appel's parser

Appel's Modern Compiler Implementation in ML uses a recursive descent parser with a slightly different approach to expressions. Compare his technique with Pratt parsing. What are the tradeoffs? Which is easier to extend? Which produces better error messages? There is no single answer — this is about understanding design space.

Exercise 3.6 — Parser as a state monad

Refactor the parser to be purely functional: instead of a mutable pos ref, have each function return result * state. Create a bind operator >>= to compose parser functions. This exercise connects the practical parser to the theoretical state monad and deepens your understanding of both.

Exercise 3.7 — Left recursion in Pratt parsing

Our Pratt parser handles left-associative operators correctly (the loop in parse_postfix produces a left-associative tree). Modify the parser to support right-associative operators like exponentiation (use ** as the token). Right-associative operators need a binding power that is lower on the right side. Implement this by subtracting 1 from the binding power when recursing into the right operand.

Exercise 3.8 — Parse type annotations

Implement parse_type and parse_type_opt functions. parse_type should handle all the type forms: simple keywords (i32, bool), pointer types (@Type), and array types ([N]Type). This is a smaller recursive descent parser nested inside the main parser.

Exercise 3.9 — Dangling else problem

Our if-statement grammar has the classic "dangling else" ambiguity: in if (a) if (b) { ... } else { ... }, which if does the else belong to? Our parser resolves this by matching else with the nearest unmatched if (the standard resolution). Verify this by writing a test case. Then implement an alternative: require braces around if-bodies in certain cases to disambiguate.
