Building a Compiler OCaml Edition

1 · Text and Spans

The Problem

A compiler cannot just say "error." It has to say where the error happened. If your program has a type mismatch on line 17, the compiler must tell you: here, at this exact position in the source. Getting this right is the difference between a compiler you love and a compiler you want to throw out a window.

Every error message in our compiler needs three pieces of information:

  • What went wrong — the error message itself
  • Where it went wrong — a location in source code
  • Why it matters — context from the compiler phase

This chapter solves the "where." We define types and data structures for representing source positions and source files. These are the foundation on which all error reporting is built.

The Span

A span is a range in the source text. It has a start index and an end index, each measured in bytes from the beginning of the source file. A single character is a span where start + 1 = end. An entire function definition is a span from its first character to its last.

In OCaml, a span is a record. Records with labeled fields are one of OCaml's great pleasures — every field has a name, every access is checked, and the compiler will warn you if you forget a field:

span.ml
type t = {
  start: int;
  stop:  int;
}

let make start stop = { start; stop }

let empty = { start = 0; stop = 0 }

let contains span pos =
  pos >= span.start && pos < span.stop

let merge a b =
  { start = min a.start b.start;
    stop  = max a.stop  b.stop
  }
src/span.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Span {
    pub start: usize,
    pub stop: usize,
}

impl Span {
    pub fn new(start: usize, stop: usize) -> Self {
        Self { start, stop }
    }

    pub fn contains(&self, pos: usize) -> bool {
        pos >= self.start && pos < self.stop
    }

    pub fn merge(&self, other: &Self) -> Self {
        Self {
            start: self.start.min(other.start),
            stop:  self.stop.max(other.stop),
        }
    }
}
Records in OCaml

OCaml records use curly braces with field: type; syntax. Construction uses named fields — { start; stop } is a shorthand for { start = start; stop = stop }. Field access uses dot notation: span.start. The compiler tracks exactly which fields are present and will reject incomplete record expressions. This is a small thing that saves enormous debugging time.

The Span.t type is deliberately minimal. It stores byte offsets into the source text — not line and column numbers. Why byte offsets? Because they are the natural unit that the lexer works with. The lexer advances through the source one character (one byte for ASCII, one to four for UTF-8) at a time and records the current position. Converting a byte offset to a line number is a separate operation, and we keep it separate.

Converting Offsets to Lines

To produce human-readable error messages, we need to map a byte offset to a line number and column. This is the job of the SourceFile module. It stores the source text and a precomputed list of line start positions.

source_file.ml
type t = {
  name:  string;
  text:  string;
  lines: int array;
}

let of_string name text =
  let lines = Array.of_list (0 :: line_starts text) in
  { name; text; lines }

let line_and_column file offset =
  let lines = file.lines in
  let rec search lo hi =
    if hi - lo <= 1 then lo
    else
      let mid = (lo + hi) / 2 in
      if lines.(mid) <= offset then search mid hi
      else search lo mid
  in
  let line = search 0 (Array.length lines - 1) in
  let col  = offset - lines.(line) in
  (line + 1, col + 1)
src/source_file.rs
pub struct SourceFile {
    pub name:  String,
    pub text:  String,
    lines: Vec<usize>,
}

impl SourceFile {
    pub fn new(name: &str, text: &str) -> Self {
        let lines = std::iter::once(0)
            .chain(text.char_indices()
                .filter_map(|(i, c)| if c == '\n' { Some(i + 1) } else { None }))
            .collect();
        Self { name: name.to_string(), text: text.to_string(), lines }
    }

    pub fn line_and_column(&self, offset: usize) -> (usize, usize) {
        let line = match self.lines.binary_search(&offset) {
            Ok(i) => i,
            Err(i) => i.saturating_sub(1),
        };
        let col = offset - self.lines[line];
        (line + 1, col + 1)
    }
}

The line_starts helper scans the text and records the byte offset of every newline character, producing positions like [0, 47, 103, ...] — each entry is the start of a new line. To find the line for a given offset, we binary-search the array. Binary search is logarithmic; for a file with thousands of lines, the lookup is essentially instantaneous.

Here is the helper function itself — a simple fold over the string:

source_file.ml — line_starts helper
let line_starts text =
  let rec scan pos acc =
    match String.index_from text pos '\n' with
    | n -> scan (n + 1) ((n + 1) :: acc)
    | exception Not_found -> List.rev acc
  in
  scan 0 []
src/source_file.rs — line_starts (inline in new)
// Already shown inline in SourceFile::new() above.
// The iterator chain is idiomatic Rust for this pattern:
//   iter::once(0).chain(text.char_indices()
//     .filter_map(|(i,c)| if c == '\n' { Some(i+1) } else { None }))
String.index_from and exceptions

The OCaml version uses String.index_from, which raises Not_found if no newline is found. We catch this exception to terminate the recursion. This is a clean use of OCaml's exception mechanism as a control flow device — it avoids an extra bounds check on every iteration. The alternative would be to manually scan character by character, which is equally valid but more verbose.

Error Reporting with Spans

With spans and source files, we can build an error reporter that shows the user exactly what went wrong, with context:

error.ml
type t = {
  msg:     string;
  span:    Span.t;
  phase:   string;
}

let report file err =
  let line, col = SourceFile.line_and_column file err.span.start in
  Printf.eprintf "%s:%d:%d: %s\n"
    file.name line col err.msg

let report_with_context file err =
  let line, col = SourceFile.line_and_column file err.span.start in
  let line_start = file.lines.(line - 1) in
  let line_end =
    if line < Array.length file.lines
    then file.lines.(line) - 1
    else String.length file.text
  in
  let line_text = String.sub file.text line_start (line_end - line_start) in
  Printf.eprintf "%s:%d:%d: %s\n" file.name line col err.msg;
  Printf.eprintf "  %s\n" line_text;
  Printf.eprintf "  %s^ %s\n"
    (String.make (col - 1) ' ')
    err.msg
src/error.rs
pub struct Error {
    pub msg:   String,
    pub span:  Span,
    pub phase: String,
}

pub fn report(file: &SourceFile, err: &Error) {
    let (line, col) = file.line_and_column(err.span.start);
    eprintln!("{}:{}:{}: {}", file.name, line, col, err.msg);
}

pub fn report_with_context(file: &SourceFile, err: &Error) {
    let (line, col) = file.line_and_column(err.span.start);
    let line_start = file.lines[line - 1];
    let line_end = if line < file.lines.len() { file.lines[line] - 1 }
                            else { file.text.len() };
    let line_text = &file.text[line_start..line_end];
    eprintln!("{}:{}:{}: {}", file.name, line, col, err.msg);
    eprintln!("  {}", line_text);
    eprintln!("  {:width$}^ {}", "", err.msg, width = col - 1);
}
Span merging

When reporting errors involving multiple locations — for example, a type mismatch between two expressions — you can use Span.merge to create a combined span that covers both locations. This produces error messages like: error: expected i32, found i64 (here and here).

Span Arithmetic

Spans support a small algebra of operations that become surprisingly useful as the compiler grows. Beyond merge and contains, we might want:

  • Intersection — does one span overlap with another? Useful for checking if two AST nodes share source text.
  • Span difference — "the part of this span before that point." Useful for error messages that need to point at a specific sub-expression within a larger expression.
  • Span ordering — does span A start before span B? The lexer produces tokens in order, and we sometimes need to verify that property.

Notice that all of these are pure functions on the Span.t record. No mutable state, no IO, no side effects. This is OCaml at its best: small, composable, verifiable operations on simple data.

Why Not Just Use Line Numbers?

An obvious question: why do we work with byte offsets instead of line-and-column pairs directly? Because line-and-column information is fragile — it changes when the source file is edited, and it is ambiguous when the same span appears in different contexts. Byte offsets are absolute. The lexer can advance a position counter without knowing or caring which line it is on. Line-and-column computation is pushed to the error reporting phase, which is the only place it matters.

This separation of concerns — data structures that are convenient for the machine versus those that are convenient for the human — is a recurring theme in compiler design. The span is for the compiler. The error message is for the human.

Exercises

Exercise 1.1 — Extend Span

Add a to_string function to Span that formats a span as "(start,end)". Add a sub function that returns the substring of the source that a span covers. Hint: this requires passing the source text to the function, which is the first hint that spans and source files are tightly coupled.

Exercise 1.2 — Multi-line spans

Modify report_with_context to handle spans that cross multiple lines. When a span covers more than one line, show all the lines involved with appropriate caret markers.

Exercise 1.3 — UTF-8 aware columns

Our column numbers are byte offsets, which works for ASCII but produces wrong results for multi-byte UTF-8 characters. Modify line_and_column to count actual Unicode characters (use Stringext or the Uutf library, or compute column using String.length on the substring up to the offset).

Exercise 1.4 — OCaml module interface

Create an .mli file for Span that exposes only t, make, empty, contains, and merge. Make sure the implementation still compiles. This exercise demonstrates OCaml's module abstraction — the implementation of t becomes opaque to the rest of the compiler.

Exercise 1.5 — Compare with Appel

Open Appel's Modern Compiler Implementation in ML and find how he handles source positions. How does his approach compare with ours? What did he simplify, and what did we add? There is no right answer — this is about understanding design tradeoffs.

Exercise 1.6 — Positionless span

Add a Builtin function to Span that returns a special span used for compiler-generated code (synthetic nodes that do not correspond to any source location). The span should have a flag or a sentinel value indicating it is "builtin." Error reporting code should skip displaying source context for builtin spans. This is essential for macros and desugared code.

Exercise 1.7 — OCaml records with mutable fields

Rewrite the Span type to use a mutable record field instead of a ref cell for the position. In OCaml, you can declare record fields as mutable: { mutable pos: int }. Compare the readability and safety of the two approaches. When would you use each?

Exercise 1.8 — Source file cache

If the compiler processes many files, recomputing line_starts for each one is wasteful. Add a cache (using OCaml's Hashtbl or a functor-based map) that stores already-computed SourceFile.t values keyed by filename. This introduces the module system's Map.Make functor and is good practice with OCaml's standard library.
