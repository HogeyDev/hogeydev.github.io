Building a Compiler

5 · Does This Make Sense?

We have a tree. But does the program make sense? Can you add a bool to an i32? Call a function that doesn't exist?

The parser gave us a beautiful AST. Every if has its branches, every while has its body, every expression nests just right. The tree is syntactically valid. But is it meaningful?

Consider this program: 

func main(i32) {
  let x: bool = 42;
}

The parser will happily build an AST for this. It will create a variable declaration with an initializer — no problem. It doesn't know that 42 isn't a bool. It doesn't care. The parser's job is just to check that the tokens form a valid tree according to our grammar.

Now it's our job — the semantic analyzer's job — to determine whether the tree actually means something. This phase is where most of the "compile error" user experience lives. Good error messages here are what separate a helpful compiler from a frustrating one.

Two phases, one pass? Some textbooks separate "semantic analysis" into distinct phases: binding (connecting identifiers to declarations) and type checking (checking type constraints). We'll do them together in a single pass over the AST, because they interleave naturally: to check the type of an identifier, we need to look it up in the symbol table.

Phase 1: Building a Symbol Table

Before we can check types, we need to know what every name refers to. A symbol table is a data structure that maps names to their declarations. Think of it as a dictionary the compiler carries around: "What is x? Oh, it's a variable of type i32, declared on line 12."

Our language has three kinds of scopes:

  • Global scope — function names and global variables.
  • Function scope — parameter names.
  • Block scope — local variables declared with let inside a block.

Scopes nest. A variable declared in an outer block is visible in inner blocks, but a variable in an inner block shadows the outer one. When we enter a new block, we push a new scope. When we leave it, we pop it off.

Here's the key insight: we don't need to execute the program to build the symbol table. We just walk the AST and record declarations. For each function, we add its name, parameter types, and return type to the table. For each let, we add the variable name and its declared type.

Let's see what this looks like in code:

src/symbols.rs
use std::collections::HashMap;
use crate::ast::*;

pub struct SymbolTable {
  // Stack of scopes. Each scope maps names to symbols.
  scopes: Vec<HashMap<String, Symbol>>,
}

pub enum Symbol {
  Variable(Type),
  Function {
    params: Vec<Type>,
    ret: Type,
  },
}

impl SymbolTable {
  pub fn new() -> Self {
    SymbolTable { scopes: vec![HashMap::new()] }
  }

  pub fn enter_scope(&mut self) {
    self.scopes.push(HashMap::new());
  }

  pub fn exit_scope(&mut self) {
    self.scopes.pop();
  }

  pub fn insert(&mut self, name: &str, sym: Symbol) {
    if let Some(scope) = self.scopes.last_mut() {
      scope.insert(name.to_string(), sym);
    }
  }

  pub fn lookup(&self, name: &str) -> Option<&Symbol> {
    for scope in self.scopes.iter().rev() {
      if let Some(sym) = scope.get(name) {
        return Some(sym);
      }
    }
    None
  }
}

let enter_scope tbl =
  tbl.scopes <- Hashtbl.create 16 :: tbl.scopes

let exit_scope tbl =
  match tbl.scopes with
  | [] -> failwith "scope imbalance"
  | _ :: rest -> tbl.scopes <- rest

let insert tbl name sym =
  match tbl.scopes with
  | [] -> failwith "no active scope"
  | scope :: _ -> Hashtbl.replace scope name sym

let lookup tbl name =
  let rec go = function
    | [] -> None
    | scope :: rest ->
      match Hashtbl.find_opt scope name with
      | Some _ as s -> s
      | None -> go rest
  in
  go tbl.scopes

The key method is lookup: it walks scopes from innermost to outermost, returning the first match. This gives us lexical (static) scoping — a name always refers to the innermost declaration in scope.

Populating the Symbol Table

We populate the symbol table with a top-level walk. For each function declaration, we insert the function name and its signature. Then we walk the body to find local declarations. We do this before type checking so that when we encounter a function call or variable reference, we already know about it.

src/symbols.rs (continued)
pub fn build(ast: &Module) -> SymbolTable {
  let mut table = SymbolTable::new();

  for decl in &ast.declarations {
    match decl {
      Decl::Function(func) => {
        let params = func.params
          .iter()
          .map(|p| p.ty.clone())
          .collect();
        table.insert(&func.name, Symbol::Function {
          params,
          ret: func.ret_ty.clone(),
        });

        // Now walk the body to find local variables
        table.enter_scope();
        for param in &func.params {
          table.insert(&param.name,
            Symbol::Variable(param.ty.clone()));
        }
        walk_block(&mut table, &func.body);
        table.exit_scope();
      }
    }
  }
  table
}

fn walk_block(table: &mut SymbolTable, block: &Block) {
  table.enter_scope();
  for stmt in &block.stmts {
    if let Stmt::Let(name, ty, _) = stmt {
      table.insert(name, Symbol::Variable(ty.clone()));
    }
  }
  table.exit_scope();
}

let build ast =
  let tbl = create () in
  let add_decl decl =
    match decl with
    | Decl.Function f ->
      let params = List.map (fun p -> p.ty) f.params in
      insert tbl f.name (Function { params; ret = f.ret_ty });
      enter_scope tbl;

      walk_block tbl f.body;
      exit_scope tbl
  in
  List.iter add_decl ast.declarations;

Forward references? Our language supports calling functions defined later in the file. The symbol table phase handles this: we register all function names before type checking, so a function can call any other function regardless of definition order.

Phase 2: Type Checking

With a populated symbol table, we can now walk the AST a second time and check every expression and statement for type correctness. This is the heart of semantic analysis.

Every expression in our language has a type. The type checker's job is to determine that type and verify it's used correctly:

Type Checking Rules (informal): Integer literal → declared type (i32 by default) Bool literal → bool Identifier → type from symbol table Binary op (+, -, etc) → both operands must be same numeric type Comparison (==, <, etc) → both operands same type, result is bool Function call → args match param types, result is return type Assignment → RHS type matches declared LHS type Cast → any numeric type to any numeric type If/while condition → must be bool Return → type must match function's return type

Let's be precise about our rules:

  • Well-formedness: Every type annotation must be valid. [5]bool is valid (array of 5 bools). @Foo is valid if Foo is a declared type.
  • No implicit coercion: We require explicit casts. let x: i32 = 42u8; is a type error. You must write let x: i32 = cast(i32, 42u8);.
  • Function arity: The number of arguments must match the number of parameters.

The Type Checker in Action

Here's the type checker. It walks expressions and returns their types, reporting errors along the way:

src/typeck.rs
pub struct TypeChecker {
  symbols: SymbolTable,
  errors: Vec<CompileError>,
  current_ret: Type,
}

impl TypeChecker {
  pub fn new(symbols: SymbolTable) -> Self {
    TypeChecker {
      symbols,
      errors: vec![],
      current_ret: Type::Void,
    }
  }

  pub fn check_expr(&mut self,
      expr: &Expr) -> Type {
    match expr {
      Expr::IntLit(_, _) => Type::I32,
      Expr::BoolLit(_, _) => Type::Bool,

      Expr::Ident(span, name) => {
        match self.symbols.lookup(name) {
          Some(Symbol::Variable(ty)) => ty.clone(),
          Some(Symbol::Function(_)) => {
            self.error(span,
              "expected a variable, found function");
            Type::I32 // error recovery
          }
          None => {
            self.error(span,
              &format("undefined variable `{}`", name));
            Type::I32
          }
        }
      }

      Expr::Binary(span, op, lhs, rhs) => {
        let lt = self.check_expr(lhs);
        let rt = self.check_expr(rhs);
        if lt != rt {
          self.error(span,
            &format("type mismatch: expected {}, found {}",
              lt, rt));
        }
        match op {
          BinOp::Add | BinOp::Sub
          | BinOp::Mul | BinOp::Div => lt,
          BinOp::Eq | BinOp::Lt
          | BinOp::Gt => Type::Bool,
          BinOp::And | BinOp::Or => {
            if lt != Type::Bool {
              self.error(span,
                "logical op requires bool operands");
            }
            Type::Bool
          }
        }
      }

      Expr::Call(span, name, args) => {
        match self.symbols.lookup(name) {
          Some(Symbol::Function{params, ret}) => {
            if args.len() != params.len() {
              self.error(span,
                &format("function `{}` expects {} \
                    arguments, got {}",
                  name, params.len(), args.len()));
            }
            for (i, (arg, param_ty))
                in args.iter().zip(&params).enumerate() {
              let arg_ty = self.check_expr(arg);
              if &arg_ty != param_ty {
                self.error(&arg.span(),
                  &format("argument {}: expected {}, \
                      found {}",
                    i + 1, param_ty, arg_ty));
              }
            }
            ret.clone()
          }
          _ => {
            self.error(span,
              &format("undefined function `{}`", name));
            Type::I32
          }
        }
      }

      Expr::Cast(_, to_ty, inner) => {
        let inner_ty = self.check_expr(inner);
        if !is_numeric(&inner_ty)
            || !is_numeric(to_ty) {
          self.error(&inner.span(),
            "cast only supported between numeric types");
        }
        to_ty.clone()
      }
    }
  }
}

type type_checker = {
  symbols: symbol_table;
  mutable errors: compile_error list;
  mutable current_ret: typ;
}

let create symbols = {
  symbols;
  errors = [];
  current_ret = Type.Void;
}

let rec check_expr tc expr =
  match expr with
  | Expr.IntLit _ -> Type.I32
  | Expr.BoolLit _ -> Type.Bool

  | Expr.Ident (span, name) -> (
    match lookup tc.symbols name with
    | Some (Variable ty) -> ty
    | Some (Function _) ->
      error tc span
        "expected a variable, found function";
      Type.I32
    | None ->
      error tc span
        ("undefined variable `" ^ name ^ "`");
      Type.I32
  )

  | Expr.Binary (span, op, lhs, rhs) ->
    let lt = check_expr tc lhs in
    let rt = check_expr tc rhs in
    if lt <> rt then
      error tc span
        ("type mismatch: expected " ^ string_of_type lt
         ^ ", found " ^ string_of_type rt);
    (match op with
    | Add | Sub | Mul | Div -> lt
    | Eq | Lt | Gt -> Type.Bool
    | And | Or ->
      if lt <> Type.Bool then
        error tc span
          "logical op requires bool operands";
      Type.Bool)

  | Expr.Call (span, name, args) -> (
    match lookup tc.symbols name with
    | Some (Function { params; ret }) ->
      let n_params = List.length params in
      let n_args = List.length args in
      if n_params <> n_args then
        error tc span
          ("function `" ^ name
           ^ "` expects " ^ string_of_int n_params
           ^ " arguments, got "
           ^ string_of_int n_args);
      List.iteri (fun i (arg, param_ty) ->
        let arg_ty = check_expr tc arg in
        if arg_ty <> param_ty then
          error tc arg.span
            ("argument " ^ string_of_int (i+1)
             ^ ": expected " ^ string_of_type param_ty
             ^ ", found " ^ string_of_type arg_ty)
      ) (List.combine args params);
      ret
    | _ ->
      error tc span
        ("undefined function `" ^ name ^ "`");
      Type.I32
  )

  | Expr.Cast (_, to_ty, inner) ->
    let inner_ty = check_expr tc inner in
    if not (is_numeric inner_ty
            && is_numeric to_ty) then
      error tc inner.span
        "cast only supported between numeric types";
    to_ty

Checking Statements

Statements don't have types — they have effects. But they still need checking. A return statement's expression must match the function's return type. An if condition must be bool. A let initializer must match the declared type.

src/typeck.rs (continued)
fn check_stmt(&mut self, stmt: &Stmt) {
  match stmt {
    Stmt::Let(span, name, ty, init) => {
      if let Some(init) = init {
        let init_ty = self.check_expr(init);
        if &init_ty != ty {
          self.error(span,
            &format("type mismatch: expected {}, \
                found {}", ty, init_ty));
        }
      }
    }

    Stmt::Return(span, expr) => {
      let expr_ty = self.check_expr(expr);
      if expr_ty != self.current_ret {
        self.error(span,
          &format("return type mismatch: expected \
              {}, found {}",
            self.current_ret, expr_ty));
      }
    }

    Stmt::If(span, cond, then, otherwise) => {
      let cond_ty = self.check_expr(cond);
      if cond_ty != Type::Bool {
        self.error(span,
          "if condition must be bool");
      }
      self.symbols.enter_scope();
      for s in then { self.check_stmt(s); }
      self.symbols.exit_scope();
      if let Some(else_block) = otherwise {
        self.symbols.enter_scope();
        for s in else_block {
          self.check_stmt(s); }
        self.symbols.exit_scope();
      }
    }

    Stmt::While(span, cond, body) => {
      let cond_ty = self.check_expr(cond);
      if cond_ty != Type::Bool {
        self.error(span,
          "while condition must be bool");
      }
      self.symbols.enter_scope();
      for s in body {
        self.check_stmt(s); }
      self.symbols.exit_scope();
    }

    Stmt::Expr(_, expr) => {
      self.check_expr(expr);
    }
  }
}
typeck.ml (continued)
let rec check_stmt tc stmt =
  match stmt with
  | Stmt.Let (span, name, ty, init) -> (
    match init with
    | None -> ()
    | Some init_expr ->
      let init_ty = check_expr tc init_expr in
      if init_ty <> ty then
        error tc span
          ("type mismatch: expected "
           ^ string_of_type ty
           ^ ", found "
           ^ string_of_type init_ty)
  )

  | Stmt.Return (span, expr) ->
    let expr_ty = check_expr tc expr in
    if expr_ty <> tc.current_ret then
      error tc span
        ("return type mismatch: expected "
         ^ string_of_type tc.current_ret
         ^ ", found "
         ^ string_of_type expr_ty)

  | Stmt.If (span, cond, then_blk, else_blk) ->
    let cond_ty = check_expr tc cond in
    if cond_ty <> Type.Bool then
      error tc span "if condition must be bool";
    enter_scope tc.symbols;
    List.iter (check_stmt tc) then_blk;
    exit_scope tc.symbols;
    (match else_blk with
    | None -> ()
    | Some blk ->
      enter_scope tc.symbols;
      List.iter (check_stmt tc) blk;
      exit_scope tc.symbols)

  | Stmt.While (span, cond, body) ->
    let cond_ty = check_expr tc cond in
    if cond_ty <> Type.Bool then
      error tc span "while condition must be bool";
    enter_scope tc.symbols;
    List.iter (check_stmt tc) body;
    exit_scope tc.symbols

  | Stmt.Expr (_, expr) ->
    check_expr tc expr

Error Messages with Spans

Remember those spans from Chapter 1? This is where they earn their keep. Instead of just saying "type error", we can point the user to the exact character that caused the problem.

Here's how our error method produces a message with source underlines: 

// Given source: `let x: bool = 42;`
// span for the type annotation spans chars 8-12 ("bool")
// span for the literal spans chars 15-17 ("42")

error[E0001]: type mismatch: expected bool, found i32
  ┌─ test.l1s:1:15
  │
1 │ let x: bool = 42;
  │       ────   ──
  │       │       │
  │       │       └── expected i32, found bool literal
  │       └── type annotation: bool

The span gives us the file, line, and column. We can underline the relevant tokens and show a hint. This is the difference between a compiler that frustrates and one that teaches.

Our error reporting function:

src/errors.rs
pub fn report_error(
  source: &str,
  span: &Span,
  message: &str,
) {
  let (line, col) = span.start_line_col(source);

  eprintln!(
    "error: {}\n  ┌─ {}:{}:{}\n  │\n{}  │ {}",
    message,
    span.file,
    line, col,
    underline(source, span),
    " ",
  );
}

fn underline(source: &str, span: &Span) -> String {
  let lines: Vec<&str> = source.lines().collect();
  let (line, _) = span.start_line_col(source);
  let line_str = lines[line - 1];

  format!(
    "{} │ {}\n  │ {}",
    line, line_str,
    " ".repeat(span.start - span.line_start)
      + &"~".repeat(span.end - span.start),
  )
}

Walking Through a Type Error

Let's trace through what happens when we check: 

func main(i32) {
  let x: bool = 42;
}

Step 1: Build the symbol table. We register main as a function returning i32, taking no parameters (wait — the return type i32 here refers to main's return type, and the parameter type is declared before the name). Actually, in our language, func main(i32) means a function named main that returns i32 with no parameters. Actually, let's clarify: in our language syntax, func name(ret_type, param: Type) { ... }, the first type is the return type. So func main(i32) means "function main returns i32, takes no parameters." We register this in the symbol table.

Step 2: Type-check the function body. We enter the function scope. We encounter the let statement:

  1. Look up the type annotation: bool — this is well-formed.
  2. Check the initializer: 42 is an integer literal, type i32.
  3. Compare: bool vs i32 — mismatch!
  4. Report: type mismatch: expected bool, found i32 at the span of the let statement.
error[E0001]: type mismatch: expected bool, found i32
  ┌─ test.l1s:2:3
  │
2 │   let x: bool = 42;
  │   ───────────────────
  │   │
  │   └── type mismatch in variable declaration
Error recovery: Notice that our type checker doesn't stop at the first error. It continues checking the rest of the function, using a "default type" (usually i32) for expressions that failed. This way, the user sees all errors in one compilation — no "fix error 1 to see error 2" frustration.

The Full Pipeline

After parsing, the semantic analysis phase runs: 

// Pseudocode for the compiler driver
fn compile(source: &str) -> Result<String, Vec<CompileError>> {
  let tokens = lex(source);
  let ast = parse(tokens);
  let symbols = build_symbol_table(&ast);
  let mut checker = TypeChecker::new(symbols);
  checker.check_module(&ast);
  if !checker.errors.is_empty() {
    return Err(checker.errors);
  }
  // ... next phase: code generation
}
What about type inference? Our language requires explicit type annotations on let declarations. This is a deliberate design choice for v3 — it keeps the type checker simple and gives us clear error messages. Type inference (where the compiler figures out types for you) is a fascinating extension that we'll hint at in the exercises.

The "Aha" Moment

Here's what we've just realized: the type checker is where most of the "compile error" user experience lives. The parser catches syntax errors. The lexer catches token errors. But the type checker catches the errors that real programmers make every day: passing the wrong types, misspelling variable names, returning the wrong value.

A good type checker with clear error messages is the difference between a language that feels like a partner and one that feels like an adversary. Every error message is a teaching moment. The span-based underlines we built in Chapter 1 — that was the foundation. This chapter is where we actually use them to help people.

The symbol table + type checker pattern is universal. Every statically typed language has one. Rust has its typeck pass. OCaml has its typing module. GCC has its c-typeck.c. The details differ, but the fundamental idea is the same: walk the AST, build a map of names, check that types are used consistently.

Exercises

1. Dead code warning

Add a warning for unreachable code after a return statement. In a block like { return 1; let x = 2; }, the let statement is dead code. How would you detect this in the type checker? Should it be an error or a warning? Implement it.

Hint: The type checker walks statements sequentially. Keep track of whether you've seen an unconditional return.

2. Implicit widening for u8 to i32

Our language requires explicit casts between types. But in many languages, widening conversions (e.g., u8 to i32) are implicit because they cannot lose information. Modify the type checker to allow implicit widening from smaller unsigned types to larger signed types. What rule would you use for u8 + i32?

3. Constant folding in the type checker

When we see let x: i32 = 2 + 3;, the type checker could evaluate 2 + 3 at compile time and store it as 5. This is called constant folding. Add a pass that evaluates constant expressions during type checking and replaces them with their results. This is a small optimization that has big effects on generated code.

4. Better error recovery

Our type checker uses Type::I32 as a default when it encounters an error. This can cause cascading errors. Implement a proper Type::Error variant that suppresses further errors derived from that expression. A Type::Error should equal-check as compatible with any type (so it doesn't trigger more mismatches).

5. Array bounds checking

Our language supports array types like [5]i32. Add a check that array access expressions use an index expression of type i32 (or another integer type). Bonus: warn if the index is a literal that is out of bounds for the array.

What's Next

We have a semantically valid program. Every name is bound. Every type checks out. The AST is correct and meaningful. Now we face a new problem: the AST is tied to the source language. It has if nodes and while loops and function declarations. But the machine doesn't know about any of that. It knows about registers, memory addresses, jumps, and labels. We need something in between — an intermediate representation that bridges the gap between our high-level AST and low-level machine code. That's the next chapter.
