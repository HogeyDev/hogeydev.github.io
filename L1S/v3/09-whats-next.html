Building a Compiler

Chapter 09: Where Next?

We built a compiler.

Not a toy. Not a simulation. A real compiler that takes source code in a real (if small) language and produces x86-64 assembly that runs on real hardware. The pipeline is complete. The executable works. You understand every line of it.

Take a breath. This is an accomplishment. Most programmers never build a compiler. You did.

What We Built

Let's look back at the full pipeline, from end to end:

Source Code │ ▼ Lexer ─────────────── Characters → Tokens │ ▼ Parser ─────────────── Tokens → AST (Abstract Syntax Tree) │ ▼ Semantic Analysis ─────────────── Names resolved, types checked │ ▼ IR Builder ─────────────── AST → IR (virtual registers, basic blocks) │ ▼ Code Generator ─────────────── IR → x86-64 Assembly │ ▼ Assembler + Linker ─────────────── Assembly → Executable

Each stage is a well-defined transformation. Each transformation is something you could explain to another programmer. The whole pipeline, in Rust and OCaml, is about 1500 lines of code.

Here's a rough inventory of what we built in each compiler:

Two complete compilers, under 1500 lines each. The OCaml version is slightly shorter because pattern matching and algebraic data types make tree transformations more concise. The Rust version is slightly longer but gives you fine-grained control over memory and zero-cost error handling.

What We Didn't Build

Our compiler works, but it's minimal. Here's what production compilers have that we left out:

FeatureWhy It MattersDifficulty
Optimizations Our code is terrible: redundant loads and stores everywhere. Constant folding, dead code elimination, and common subexpression elimination would make it 10x faster. Medium
Register allocation We spill everything to the stack. Graph coloring or linear scan register allocation would keep values in registers, which is orders of magnitude faster. Hard
Garbage collection Our language has no heap allocation, so we don't need GC. But adding pointers means you need to decide: manual memory management, reference counting, or tracing GC. Hard
Standard library We have no built-in functions for I/O, string manipulation, math, or data structures. Every program must return an integer — no printing, no file access. Easy
Modules / imports All code must be in one file. No way to split a program across multiple files or reuse code from other projects. Medium
Generics / templates Functions can only work with concrete types. No List<T> or fn map<A, B>. Polymorphism is a major extension. Hard
Closures No anonymous functions, no capturing variables from enclosing scopes. Implementing closures requires closure conversion and trampolines or function pointers with environments. Hard
String types We have only integers and booleans. Strings require a character encoding decision (ASCII? UTF-8?), memory allocation, and runtime support. Medium
Floating point No f32 or f64 types. Floating point requires SSE/AVX instructions and careful handling of NaN, infinity, and rounding modes. Medium
Self-hosting The compiler is written in Rust and OCaml, not in our language. Could we write our compiler in our own language and compile it with itself? That's the ultimate milestone. Very Hard
Not a weakness — a starting point Don't read this list as "our compiler is incomplete." Read it as "our compiler is the foundation for any of these." Every production compiler started where we are now. LLVM began as a research project. GCC began as a small C compiler. Rust began as a personal project. Every single feature above was added one at a time.

Ideas for the Reader

So where do you go from here? Here are concrete projects, ordered from easiest to hardest:

Add One Feature at a Time

The best advice: add the thing you miss most. Is it frustrating that you can't print output? Add a print function that calls the C standard library's printf. Do you want to write recursive functions? Make sure the calling convention handles it correctly (it should already). Does the one-pass compilation annoy you? Add a REPL.

Each feature teaches you something new about the compiler. Adding print teaches you about foreign function interfaces. Adding arrays teaches you about memory layout and bounds checking. Adding closures teaches you about environment capture and heap allocation.

Port to a New Backend

Our compiler targets x86-64, but the concepts transfer directly to other architectures. Pick one:

  • ARM64 (AArch64): Used in every modern phone, many laptops, and Apple Silicon. Different register names (x0-x30), different instruction encoding, but the same concepts: mov, add, sub, cmp, b, bl, ret. The "spill everything" strategy works identically.
  • RISC-V: The cleanest instruction set ever designed. Only 47 instructions in the base ISA. Registers x0-x31. Everything is simple and regular. A beautiful target for learning.
  • WebAssembly: A virtual ISA that runs in every modern web browser. Stack-based, not register-based. The codegen is completely different — instead of registers, you push and pop values on an implicit stack.
  • LLVM IR: Instead of generating assembly directly, generate LLVM intermediate representation. LLVM handles register allocation and machine code generation for any architecture it supports. This is what Rust, Swift, and Clang do.

Porting requires rewriting the codegen module (and only the codegen module). The frontend — lexer, parser, type checker, IR builder — stays exactly the same. This is the power of a well-designed IR.

Add an Optimization Pass

Our generated code is full of redundant loads and stores. Here's the easiest optimization to add:

Constant folding: If the IR has const %0, 3; const %1, 4; add %2, %0, %1, replace it with const %2, 7. This is a single pass over the IR that evaluates constant expressions at compile time.

Dead store elimination: If a value is stored to a stack slot and never loaded before being overwritten, the first store is dead. Remove it. This requires live variable analysis.

Peephole optimization: After codegen, scan the assembly for common inefficient patterns and replace them. For example: mov rax, [rbp-8]; mov [rbp-8], rax (a load followed by a store to the same location) is a no-op and can be removed.

Each optimization is a small, independent pass. Add them one at a time, test after each, and watch your generated code get faster.

Here's a concrete example. Before constant folding, return 2 + 3 * 4 generates IR operations for every computation and loads/stores for every intermediate. After folding, the IR becomes simply const %0, 14; ret %0 — no runtime arithmetic at all.

;; Before constant folding
const %0, 2
const %1, 3
const %2, 4
mul %3, %1, %2
add %4, %0, %3
ret %4

;; After constant folding
const %0, 14
ret %0

That's fewer instructions, fewer stack slots, and faster execution — all from a 20-line optimization pass.

Write a Debug Info Generator

Our compiler produces working executables, but GDB can't give you source-level debugging. To fix this, you'd need to generate DWARF debug information — a standardized format that maps machine addresses back to source positions, variable names, and types.

This is mostly plumbing: emit DWARF sections in the assembly file along with the code. NASM supports the directives you'd need (.loc, .cfi_*, etc.). The payoff: gdb ./test would show you source lines, let you set breakpoints by function name, and inspect variables by name.

Further Reading

The books and resources that will take you further than we could in these nine chapters:

ResourceWhy Read It
Compilers: Principles, Techniques, and Tools (the "Dragon Book")
— Aho, Lam, Sethi, Ullman		 The classic textbook. Covers the full breadth of compiler theory: finite automata for lexing, LR parsing, type systems, data-flow analysis, and register allocation. Dense but definitive.
Crafting Interpreters
— Robert Nystrom		 The most approachable book on language implementation. Builds a tree-walking interpreter and then a bytecode VM, all in clean Java and C. Excelent on the "why" behind design choices. Free online.
Engineering a Compiler
— Cooper and Torczon		 Pragmatic and well-structured. Especially strong on instruction selection, register allocation, and optimization. More approachable than the Dragon Book while still being thorough.
Modern Compiler Implementation in ML
— Andrew Appel		 Uses OCaml (well, SML) for all examples. Builds a complete compiler for the Tiger language. If you liked our OCaml compiler, you'll love this — it's the same approach, scaled up to a real university course.
System V AMD64 ABI The official specification for calling conventions, stack layout, and data types on x86-64 Linux. Required reading if you want to interoperate with C or understand every detail of our codegen.
NASM Manual The assembler we use. Its manual documents every directive, every instruction, and every output format. Keep it open when you're working on the codegen.
LLVM Language Reference If you decide to target LLVM IR, this is your spec. LLVM's type system (with opaque pointers, SSA form, and metadata) influenced the design of our IR heavily.

The Road Ahead

Building a compiler changes how you think about code. You start seeing the hidden structure: the boundary between compile-time and run-time, the difference between syntax and semantics, the layering of abstractions that makes modern computing possible.

Here's what I hope you take away from this journey:

Compilers are not magic. They are mechanical transformations, each one simple enough to hold in your head. The complexity comes from the number of stages, not from any single stage.

The pipeline scales. Every optimization pass, every language feature, every backend target we discussed follows the same pattern: define a representation, write a transformation, compose it with the rest. The skills you learned here apply directly to production compilers.

You can build anything. If you can build a compiler, you can build almost any software system. The discipline of defining precise interfaces between stages, of designing data structures that make transformations natural, of testing each piece independently — these are universal software engineering skills.

The companion code is yours Both the Rust and OCaml compilers are fully functional. Clone them, modify them, break them, fix them. Add your feature. Port them to a new architecture. Use them as the starting point for a language of your own design. They're under 1500 lines each — you can hold the entire compiler in your head and your editor at the same time.

Thank You

If you've read this far, you've done something most programmers never attempt. You've seen behind the curtain. You understand what happens between return 42 and mov eax, 42; ret. You've built both the theory and the practice of a complete compiler pipeline.

The next step is yours. The compiler you build from here — with your features, your optimizations, your design decisions — will teach you more than any book could. The tools are in your hands. The concepts are in your head.

A Concrete Roadmap

If you want a structured path forward, here's a sequence of 10 projects that will take you from our toy compiler to something substantially more capable:

  1. Add a standard library. Implement print_i32, print_bool, read_i32 as extern functions that call C's printf and scanf. This makes programs interactive.
  2. Add string literals. Define a string type, store string data in the read-only data section (.rodata), and pass pointers around. Teach the type checker about @u8 (pointer to byte).
  3. Add arrays. Fixed-size arrays with [N]Type syntax. Stack-allocate them. Add bounds checks in the IR or codegen.
  4. Add a for loop. Syntactic sugar over while. Desugar in the parser or in a separate pass before IR building.
  5. Add compound assignment. +=, -=, etc. Desugar to x = x + y.
  6. Add an optimization pass. Start with constant folding (easiest), then dead store elimination, then common subexpression elimination.
  7. Add proper register allocation. Replace spill-everything with linear scan. Keep frequently-used values in registers.
  8. Add a REPL. Read-eval-print loop with dynamic code generation. This is the most fun project on this list.
  9. Port to ARM64. Rewrite just the codegen. You'll learn a new instruction set and appreciate the architecture-independence of your IR.
  10. Self-host. Write the compiler in your language. Bootstrap it through your existing compiler. This is the final boss of compiler projects.

Each project builds on the previous one. Each one teaches something new. And each one brings you closer to building a compiler that competes with production systems.

Go build something amazing.
