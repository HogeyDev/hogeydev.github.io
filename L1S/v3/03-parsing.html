Building a Compiler

3. Making Sense of Structure

In the last chapter, we built a lexer. It chews through raw source text and spits out tokens — tidy little labeled chunks like KwFunc, Ident("main"), Int(42). A flat list. One token after another, left to right.

Here is the token stream for func main(i32) { return 42; }:

Token stream (flat): KwFunc Ident("main") LParen Ident("i32") RParen LBrace KwReturn Int(42) Semicolon RBrace

A flat list is a terrible way to represent a program. Where does the function body start? Which tokens belong to the return statement? Is the 42 inside the parens or inside the braces? The answer is obvious to us — we can see the nesting — but the computer only sees a sequence. We need to reconstruct the structure.

What we want is a tree:

Tree structure (what we want): Program FuncDecl "main" ReturnType: Ident("i32") Body: Return Int(42)

This chapter is about that transformation. Flat tokens in, structured tree out. We will build a parser.

A Grammar, Discovered

Before we write code, we need a clear picture of what our language looks like. Not in prose — that is too fuzzy — but in a precise, recursive description. A grammar.

Let us write down the shape of each construct in our language:

program = decl* decl = func_decl | var_decl func_decl = "func" ident "(" type ("," ident ":" type)* ")" block block = "{" stmt* "}" stmt = return_stmt | var_decl | if_stmt | while_stmt | expr_stmt return_stmt = "return" expr ";" var_decl = "let" ident ":" type "=" expr ";" if_stmt = "if" "(" expr ")" block ("else" block)? while_stmt = "while" "(" expr ")" block expr_stmt = expr ";" expr = ... (we'll get to this)

Each line is a production rule. The thing on the left is made of the things on the right. The vertical bar | means "or". The star * means "zero or more". The question mark ? means "optional".

This grammar is recursive: a block contains statements, and some statements (like if) contain blocks. That recursion is what lets us represent arbitrarily nested programs.

Recursive Descent: Grammar Rules Become Functions

Here is the key insight of recursive descent parsing: every production rule in the grammar becomes a function in the parser.

Think about what a rule like return_stmt = "return" expr ";" means. It says: "if you are trying to parse a return statement, first look for the keyword return, then parse an expression, then expect a semicolon." That is literally what our function will do:

src/parser.rs (skeleton)
fn parse_return_stmt(&mut self) -> Result<Stmt, Error> {
    self.expect(TokenKind::KwReturn)?;
    let expr = self.parse_expression()?;
    self.expect(TokenKind::Semicolon)?;
    Ok(Stmt::Return(expr))
}
parser.ml (skeleton)
let parse_return_stmt parser =
  expect parser KwReturn;
  let expr = parse_expression parser in
  expect parser Semicolon;
  Stmt::Return expr
Look ma, no parser generator We are writing the parser by hand. This gives us complete control over error messages, recovery, and performance. Most production compilers do the same — gcc, clang, rustc, and ocamlc all use hand-written recursive descent parsers.

Building the Parser, Layer by Layer

Step 1: parse_program

A program is a sequence of declarations. We keep parsing until we run out of tokens:

src/parser.rs
fn parse_program(tokens: &[Token]) -> Result<Program, Error> {
    let mut p = Parser::new(tokens);
    let mut decls = Vec::new();
    while !p.is_at_end() {
        decls.push(p.parse_decl()?);
    }
    Ok(Program { decls })
}
parser.ml
let parse_program parser =
  let rec loop decls =
    if is_at_end parser then List.rev decls
    else loop (parse_decl parser :: decls)
  in
  { decls = loop [] }

Step 2: parse_func_decl

Now for the interesting part. A function declaration has a distinctive shape: func name ( ret_type , params ) { body }. We parse it by consuming tokens in order, recursing into sub-parsers for the body block:

src/parser.rs
fn parse_func_decl(&mut self) -> Result<FuncDecl, Error> {
    self.expect(TokenKind::KwFunc)?;
    let name = self.expect_ident()?;
    self.expect(TokenKind::LParen)?;
    
    let ret_type = self.parse_type()?;
    
    let mut params = Vec::new();
    if !self.check(TokenKind::RParen) {
        loop {
            let param_type = self.parse_type()?;
            self.expect(TokenKind::Colon)?;
            let param_name = self.expect_ident()?;
            params.push(Param { name: param_name, ty: param_type });
            if !self.check(TokenKind::Comma) { break; }
            self.advance();
        }
    }
    self.expect(TokenKind::RParen)?;
    let body = self.parse_block()?;
    Ok(FuncDecl { name, ret_type, params, body })
}
parser.ml
let parse_func_decl parser =
  expect parser KwFunc;
  let name = expect_ident parser in
  expect parser LParen;
  let ret_type = parse_type parser in
  let params =
    if check parser RParen then []
    else let rec loop acc =
      let ty = parse_type parser in
      expect parser Colon;
      let name = expect_ident parser in
      let acc = { name; ty } :: acc in
      if check parser Comma then (advance parser; loop acc)
      else List.rev acc
    in loop []
  in
  expect parser RParen;
  let body = parse_block parser in
  { name; ret_type; params; body }
Why the unusual syntax? Remember: func name(ret_type, param: type). The return type comes before the parameter list. This is deliberate — it means we see the return type immediately after the opening paren, before we encounter any parameter names. This simplifies the parser because we always know what we are parsing next without lookahead.

Step 3: parse_statement

Statements are where parsing gets tactical. We look at the current token and decide what kind of statement we are looking at. This is called dispatch on first token:

src/parser.rs
fn parse_statement(&mut self) -> Result<Stmt, Error> {
    match self.peek().kind {
        TokenKind::KwReturn => self.parse_return_stmt(),
        TokenKind::KwLet    => self.parse_var_decl(),
        TokenKind::KwIf     => self.parse_if_stmt(),
        TokenKind::KwWhile  => self.parse_while_stmt(),
        TokenKind::LBrace   => self.parse_block().map(Stmt::Block),
        _                   => self.parse_expr_stmt(),
    }
}
parser.ml
let parse_statement parser =
  match peek_kind parser with
  | KwReturn -> parse_return_stmt parser
  | KwLet    -> parse_var_decl parser
  | KwIf     -> parse_if_stmt parser
  | KwWhile  -> parse_while_stmt parser
  | LBrace   -> Stmt::Block (parse_block parser)
  | _        -> parse_expr_stmt parser

Step 4: parse_var_decl

Variable declarations follow let name : type = expr ;. Straightforward:

src/parser.rs
fn parse_var_decl(&mut self) -> Result<Stmt, Error> {
    self.expect(TokenKind::KwLet)?;
    let name = self.expect_ident()?;
    self.expect(TokenKind::Colon)?;
    let ty = self.parse_type()?;
    self.expect(TokenKind::Equals)?;
    let init = self.parse_expression()?;
    self.expect(TokenKind::Semicolon)?;
    Ok(Stmt::VarDecl { name, ty, init })
}
parser.ml
let parse_var_decl parser =
  expect parser KwLet;
  let name = expect_ident parser in
  expect parser Colon;
  let ty = parse_type parser in
  expect parser Equals;
  let init = parse_expression parser in
  expect parser Semicolon;
  Stmt::VarDecl { name; ty; init }

The Hard Part: Expressions

Expressions are where parsing gets really interesting. Consider a + b * c. A naive parser would see a, then +, then b, then *, then c — and build a flat list. But we need a tree that respects precedence: multiplication binds tighter than addition.

The classic approach is to write one function per precedence level:

parse_expression → parse_term → parse_factor → parse_primary (lowest) (highest)

This works, but it is rigid. Every time we add a new operator with a new precedence, we need to thread it through the cascade. After a few operators, the code becomes a house of cards.

Pratt Parsing: A Better Way

In 1973, Vaughan Pratt proposed an alternative. Instead of a fixed function cascade, we give each operator a binding power (a number), and write a single loop that uses these numbers to decide what to parse next.

Here is the binding power table for our language:

OperatorBinding PowerAssociativity
==, !=5Left
<, >, <=, >=10Left
+, -15Left
*, /20Left
Unary -, !30Prefix
Function call ()40Postfix

Higher number = tighter binding. Multiplication (20) binds tighter than addition (15), so a + b * c parses as a + (b * c).

The core of Pratt parsing is a single function, parse_expr_bp(min_bp):

src/parser.rs
fn parse_expr_bp(&mut self, min_bp: u32) -> Result<Expr, Error> {
    // Parse a prefix expression (literal, ident, unary operator)
    let mut lhs = self.parse_prefix()?;

    // Consume postfix/infix operators as long as they bind tightly enough
    loop {
        let token = self.peek();
        let (l_bp, r_bp) = match infix_bp(token.kind) {
            Some(bp) => bp,
            None => break,
        };
        if l_bp < min_bp { break; }
        self.advance();
        let rhs = self.parse_expr_bp(r_bp)?;
        lhs = Expr::Binary { op, lhs: Box::new(lhs), rhs: Box::new(rhs) };
    }
    Ok(lhs)
}

fn infix_bp(kind: TokenKind) -> Option<(u32, u32)> {
    match kind {
        TokenKind::Plus       => Some((15, 16)),
        TokenKind::Star       => Some((20, 21)),
        TokenKind::EqualsEquals => Some((5, 6)),
        TokenKind::LessThan   => Some((10, 11)),
        _ => None,
    }
}
parser.ml
let rec parse_expr_bp parser min_bp =
  (* Parse a prefix expression *)
  let lhs = parse_prefix parser in
  
  (* Consume operators that bind tightly enough *)
  let rec loop lhs =
    let kind = peek_kind parser in
    match infix_bp kind with
    | None -> lhs
    | Some (l_bp, r_bp) when l_bp < min_bp -> lhs
    | Some (_, r_bp) ->
        advance parser;
        let rhs = parse_expr_bp parser r_bp in
        loop (Expr::Binary { op; lhs; rhs })
  in
  loop lhs

let infix_bp = function
  | Plus        -> Some (15, 16)
  | Star        -> Some (20, 21)
  | EqualsEquals -> Some (5, 6)
  | LessThan    -> Some (10, 11)
  | _           -> None
Why (l_bp, r_bp)? Left binding power controls when the loop stops: a + b * c, when we are parsing + at bp 15, the next operator * has left bp 20. Since 20 ≥ 15 (the +'s right binding power), the loop continues and b * c is parsed as a unit. Right binding power is what we pass as min_bp to the recursive call, which is usually l_bp + 1 for left-associative operators.

Prefix Parsing

The parse_prefix function handles atomic expressions and prefix operators. It looks at the current token and dispatches:

src/parser.rs
fn parse_prefix(&mut self) -> Result<Expr, Error> {
    match self.advance().kind {
        TokenKind::Int(n)  => Ok(Expr::Int(n)),
        TokenKind::Ident(s) => Ok(Expr::Ident(s)),
        TokenKind::KwTrue  => Ok(Expr::Bool(true)),
        TokenKind::KwFalse => Ok(Expr::Bool(false)),
        TokenKind::Minus   => {
            let rhs = self.parse_expr_bp(30)?;
            Ok(Expr::Unary { op: UnaryOp::Neg, rhs: Box::new(rhs) })
        }
        TokenKind::Bang    => {
            let rhs = self.parse_expr_bp(30)?;
            Ok(Expr::Unary { op: UnaryOp::Not, rhs: Box::new(rhs) })
        }
        TokenKind::LParen  => {
            let expr = self.parse_expression()?;
            self.expect(TokenKind::RParen)?;
            Ok(expr)
        }
        _ => Err(self.error("expected expression")),
    }
}
parser.ml
let parse_prefix parser =
  match advance parser with
  | Int n      -> Expr::Int n
  | Ident s    -> Expr::Ident s
  | KwTrue     -> Expr::Bool true
  | KwFalse    -> Expr::Bool false
  | Minus      ->
      let rhs = parse_expr_bp parser 30 in
      Expr::Unary { op = Neg; rhs }
  | Bang       ->
      let rhs = parse_expr_bp parser 30 in
      Expr::Unary { op = Not; rhs }
  | LParen     ->
      let expr = parse_expression parser in
      expect parser RParen;
      expr
  | _          -> error parser "expected expression"

Walking Through an Example

Let us trace the parser on func add(i32, a: i32, b: i32) { return a + b; }.

Token stream: KwFunc Ident("add") LParen Ident("i32") Comma Ident("a") Colon Ident("i32") Comma Ident("b") Colon Ident("i32") RParen LBrace KwReturn Ident("a") Plus Ident("b") Semicolon RBrace

1. parse_program enters its loop. The first token is KwFunc. It dispatches to parse_decl, which sees KwFunc and calls parse_func_decl.

2. parse_func_decl consumes KwFunc and Ident("add"). It sees LParen and enters the parameter section.

3. Return type: It parses Ident("i32") as the return type. Sees Comma, so there are parameters to parse.

4. First param: Parses Ident("i32") (type), consumes Colon, parses Ident("a") (name). Sees Comma — continues.

5. Second param: Same pattern: Ident("i32") : Ident("b"). Sees RParen — parameter loop ends.

6. Body: Consumes RParen. Calls parse_block, which consumes LBrace and calls parse_statement.

7. Return statement: parse_statement sees KwReturn. Calls parse_return_stmt, which consumes KwReturn and calls parse_expression.

8. Expression a + b: parse_expr_bp(0) calls parse_prefix, which yields Expr::Ident("a"). Then it sees Plus with left bp 15 ≥ 0. It advances, passes right bp 16 to recursion. The recursive call parses Ident("b"), sees no more operators, and returns. The result is Binary { op: Add, lhs: a, rhs: b }.

9. Finish: Consumes Semicolon, then RBrace. Returns a FuncDecl wrapped in a Program.

Error Recovery

What happens when the input is malformed? We don't want to stop at the first error — that would be frustrating for the user, who has to fix one error, recompile, find the next, fix, recompile…

The trick is error recovery. When we encounter an unexpected token, we:

  1. Report the error (add it to a list, don't crash)
  2. Synthesize a placeholder node (so the parser has something to return)
  3. Advance to a "recovery point" — usually a semicolon or a closing brace
src/parser.rs
fn expect(&mut self, kind: TokenKind) -> Result<TokenKind, Error> {
    if self.check(kind) {
        Ok(self.advance().kind)
    } else {
        let found = self.peek().kind;
        self.error("expected ", kind, ", found ", found);
        // Skip tokens until we find something that makes sense
        self.synchronize();
        Err(Error)
    }
}

fn synchronize(&mut self) {
    while !self.is_at_end() {
        if self.peek().kind in [Semicolon, RBrace, KwFunc, KwLet] {
            return;
        }
        self.advance();
    }
}
parser.ml
let expect parser kind =
  if check parser kind then
    advance parser
  else begin
    error parser ("expected " ^ show_kind kind);
    synchronize parser
  end

let synchronize parser =
  let rec go () =
    if is_at_end parser then ()
    else
      let kind = peek_kind parser in
      match kind with
      | Semicolon | RBrace | KwFunc | KwLet -> ()
      | _ -> advance parser; go ()
  in go ()

Synchronization tokens (semicolon, right brace, function keyword) act as landmarks. They tell the parser "you can safely resume here." This is not perfect — some errors will confuse the parser and cause cascading errors — but it is vastly better than stopping at the first mistake.

The Complete Picture

Here is the full parser structure, showing how the pieces fit together:

parse_program └─ parse_decl ├─ parse_func_decl │ ├─ parse_type (return type) │ ├─ parse_param_list (name: type, ...) │ └─ parse_block │ └─ parse_statement (loop) │ ├─ parse_return_stmt │ │ └─ parse_expression │ │ └─ parse_expr_bp(0) │ │ ├─ parse_prefix (atomics) │ │ └─ infix loop │ ├─ parse_var_decl │ │ └─ parse_expression │ ├─ parse_if_stmt │ │ ├─ parse_expression (condition) │ │ ├─ parse_block (then) │ │ └─ parse_block (else, optional) │ ├─ parse_while_stmt │ │ ├─ parse_expression (condition) │ │ └─ parse_block (body) │ └─ parse_expr_stmt │ └─ parse_expression └─ parse_var_decl (top-level)

What We Built

We took a flat list of tokens and turned it into a structured tree. The parser is the bridge between the lexer's world (flat sequences, surface syntax) and the compiler's world (nested trees, deep meaning).

But what do these trees actually look like? We have been vaguely referring to FuncDecl, Stmt::Return, Expr::Binary — but we have not defined them precisely. That is the next chapter, where we will build the Abstract Syntax Tree (AST) that serves as the backbone of the entire compiler.

Key insight The parser is a translator between representations. It converts the linear, token-based representation of the source code into a tree-based representation that the rest of the compiler can work with. Every function in the parser corresponds to a node type in the tree.

Exercises

1. Ternary operator Add support for cond ? then_expr : else_expr. What binding power should ? have? (Hint: it should be very low — lower than ==.) What is the right binding power of :?
2. Array subscripting Add support for arr[index]. This is a postfix operator similar to function calls. What binding power should it have? How does parse_prefix need to change?
3. Error-tolerant expression parsing Modify parse_expr_bp so that if it encounters an unexpected token, it inserts a dummy expression and continues. Test with a + * b — does it produce a reasonable error?
4. Negative literals Our language supports unary minus in expressions like -x. But what about -42? Should the lexer produce Int(-42) or should the parser combine Minus and Int(42)? What are the tradeoffs?
5. Assignment as expression In C, a = b = c is valid: assignment is right-associative and returns the assigned value. Add assignment = as an infix operator with binding power 2 (right associative). What implications does this have for let x: i32 = 42;? How do you disambiguate?
