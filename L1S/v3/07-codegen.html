Building a Compiler

Chapter 07: Speaking the Machine's Language

We have our intermediate representation. Clean, simple, low-level. Instructions like %0 = add %1, %2 that operate on virtual registers and jump between basic blocks. It's almost assembly already — except it isn't. No real processor understands virtual registers. No real processor executes br_cond %0, @block1, @block2.

We need to translate IR into something a real CPU can execute. For us, that means x86-64 assembly. This is the moment where all our careful abstractions meet the hard reality of a physical machine.

This chapter is different Everything before this was about transforming the source program while keeping it abstract. Now we make concrete choices: which register holds which value, where things go in memory, how to pass arguments, how to call functions. These decisions are constrained by the architecture, not by our design preferences.

The Problem

Here's a piece of IR our compiler might produce:

main:
  @entry:
    const %0, 42
    ret %0

We need to turn this into something like:

main:
  push rbp
  mov  rbp, rsp
  sub  rsp, 8
  mov  qword [rbp-8], 42
  mov  rax, [rbp-8]
  mov  rsp, rbp
  pop  rbp
  ret

That's a big jump. We need to solve a stack of sub-problems:

  • Virtual vs. physical registers: Our IR uses %0, %1, %2, ... The CPU has rax, rbx, rcx, ... How do we map one to the other?
  • Memory management: Where do local variables live? How does the stack work?
  • Control flow: How do we turn basic blocks and branches into labels and jump instructions?
  • Function calls: How do we pass arguments and get return values?

Let's solve each one, starting from the bottom.

x86-64: A Quick Primer

We're targeting x86-64 in 64-bit mode. Here's what you need to know:

Registers

The x86-64 has sixteen general-purpose registers. Each is 64 bits wide, but we can also access their lower 32 bits (eax, ebx, ...), 16 bits (ax, bx, ...), and 8 bits (al, ah, ...). For our compiler, we'll mostly use the full 64-bit names:

RegisterPurposeSaved by
raxReturn value, scratchCaller
rbxScratchCallee
rcx4th argument, scratchCaller
rdx3rd argument, scratchCaller
rdi1st argument, scratchCaller
rsi2nd argument, scratchCaller
r85th argument, scratchCaller
r96th argument, scratchCaller
r10–r11ScratchCaller
r12–r15ScratchCallee
rspStack pointer—
rbpBase pointer (frame pointer)Callee

"Caller-saved" means the caller must save the register before a call if it wants the value preserved. "Callee-saved" means the called function must save and restore the register if it uses it.

Key Instructions

InstructionMeaningExample
movCopy source to destinationmov rax, 42; mov [rbp-8], rax

cmpCompare, set flagscmp rax, rcx

jccConditional jumpje .Lthen; jne .Lend

popPop value from stackpop rbp

Square brackets mean memory access. mov rax, [rbp-8] loads the 8 bytes at address rbp-8 into rax. mov [rbp-8], rax stores rax into memory.

The Stack

The stack grows downward. rsp points to the top of the stack (the lowest address currently in use). push decrements rsp and stores; pop loads and increments rsp.

When a function runs, it sets up a stack frame — a region of the stack that holds its local variables and saved registers. The convention we'll follow:

; Prologue — set up the frame
push rbp          ; save the caller's base pointer
mov  rbp, rsp     ; our frame starts here
sub  rsp, <N>     ; allocate N bytes for locals

; ... function body ...

; Epilogue — tear down the frame
mov  rsp, rbp     ; restore stack pointer
pop  rbp          ; restore caller's base pointer
ret               ; return to caller

After the prologue, every local variable lives at a negative offset from rbp: [rbp-8], [rbp-16], [rbp-24], and so on. The frame size N must be a multiple of 16 to maintain stack alignment (required by the SysV ABI for call instructions).

Strategy: Spill Everything

The simplest possible register allocation strategy is: don't use registers at all. Put every virtual register on the stack. Access them through rbp-relative loads and stores.

This is called "spilling everything." It's naive — every operation becomes a load-compute-store sequence — but it's correct, and it's the foundation every real register allocator improves upon.

StackAllocator

We walk the IR, find every virtual register, and assign each a unique stack slot:

src/codegen.rs — StackAllocator
use std::collections::HashMap;

pub struct StackAllocator {
    offsets: HashMap<u32, i32>,
    frame_size: i32,
}

impl StackAllocator {
    pub fn new() -> Self {
        Self { offsets: HashMap::new(), frame_size: 0 }
    }

    pub fn allocate(&mut self, vreg: u32) -> i32 {
        if let Some(&offset) = self.offsets.get(&vreg) {
            return offset;
        }
        self.frame_size += 8;
        let offset = -self.frame_size;
        self.offsets.insert(vreg, offset);
        offset
    }

    pub fn frame_size(&self) -> i32 {
        self.frame_size
    }
}
codegen.ml — StackAllocator
module VRegMap = Map.Make(Int)

type t = {
  offsets: int VRegMap.t;
  frame_size: int;
}

let create () = { offsets = VRegMap.empty; frame_size = 0 }

let allocate alloc vreg =
  match VRegMap.find_opt vreg alloc.offsets with
  | Some offset -> (alloc, offset)
  | None ->
    let frame_size = alloc.frame_size + 8 in
    let offset = -frame_size in
    let offsets = VRegMap.add vreg offset alloc.offsets in
    ({ offsets; frame_size }, offset)

let frame_size alloc = alloc.frame_size

Every virtual register gets a dedicated 8-byte slot. The first vreg gets [rbp-8], the second gets [rbp-16], and so on. The frame size tells us how much to sub rsp in the prologue.

Instruction Selection

Now the real work: map each IR instruction to one or more x86-64 instructions. The pattern is almost always the same:

  1. Load operand virtual registers from their stack slots into physical registers
  2. Perform the operation
  3. Store the result back to the destination's stack slot

Let's see how each IR instruction translates:

IRx86-64 Translation
const %d, V mov rax, V; mov [rbp-OFF_d], rax
add %d, %a, %b mov rax, [rbp-OFF_a]; mov rcx, [rbp-OFF_b]; add rax, rcx; mov [rbp-OFF_d], rax
sub %d, %a, %b Same but sub rax, rcx
mul %d, %a, %b Same but imul rax, rcx
eq %d, %a, %b mov rax, [rbp-OFF_a]; cmp rax, [rbp-OFF_b]; sete al; mov [rbp-OFF_d], rax
lt %d, %a, %b Same but setl al
load %d, %p mov rax, [rbp-OFF_p]; mov rax, [rax]; mov [rbp-OFF_d], rax
store %p, %v mov rax, [rbp-OFF_v]; mov rcx, [rbp-OFF_p]; mov [rcx], rax
br @block jmp .Lblock
br_cond %c, @t, @f mov rax, [rbp-OFF_c]; cmp rax, 0; jne .Lt; jmp .Lf
ret %v mov rax, [rbp-OFF_v]; (epilogue); ret

Here's the implementation of this translation in both compilers:

src/codegen.rs — Instruction selection
pub fn codegen_instr(
    instr: &IrInstr,
    alloc: &mut StackAllocator,
    out: &mut Emitter,
) {
    match instr {
        IrInstr::Const(dst, val) => {
            let off = alloc.allocate(*dst);
            out.emit(&format!("mov rax, {}", val));
            out.emit(&format!("mov qword [rbp-{}], rax", -off));
        }
        IrInstr::Add(dst, a, b)
        | IrInstr::Sub(dst, a, b)
        | IrInstr::Mul(dst, a, b) => {
            let op = match instr {
                IrInstr::Add(..) => "add",
                IrInstr::Sub(..) => "sub",
                IrInstr::Mul(..) => "imul",
                _ => unreachable!(),
            };
            let off_a = alloc.allocate(*a);
            let off_b = alloc.allocate(*b);
            let off_d = alloc.allocate(*dst);
            out.emit(&format!("mov rax, [rbp-{}]", -off_a));
            out.emit(&format!("mov rcx, [rbp-{}]", -off_b));
            out.emit(&format!("{} rax, rcx", op));
            out.emit(&format!("mov qword [rbp-{}], rax", -off_d));
        }
        IrInstr::Eq(dst, a, b) | IrInstr::Lt(dst, a, b) => {
            let cc = match instr {
                IrInstr::Eq(..) => "sete",
                _ => "setl",
            };
            let off_a = alloc.allocate(*a);
            let off_b = alloc.allocate(*b);
            let off_d = alloc.allocate(*dst);
            out.emit(&format!("mov rax, [rbp-{}]", -off_a));
            out.emit(&format!("cmp rax, [rbp-{}]", -off_b));
            out.emit(&format!("{} al", cc));
            out.emit("movzx rax, al");
            out.emit(&format!("mov qword [rbp-{}], rax", -off_d));
        }
        IrInstr::Br(label) => {
            out.emit(&format!("jmp .L{}", label));
        }
        IrInstr::BrCond(cond, t, f) => {
            let off = alloc.allocate(*cond);
            out.emit(&format!("mov rax, [rbp-{}]", -off));
            out.emit("cmp rax, 0");
            out.emit(&format!("jne .L{}", t));
            out.emit(&format!("jmp .L{}", f));
        }
        IrInstr::Ret(val) => {
            let off = alloc.allocate(*val);
            out.emit(&format!("mov rax, [rbp-{}]", -off));
        }
        // load, store, call handled separately
        _ => {}
    }
}

    emit out (Printf.sprintf "jmp .L%s" f);
    alloc

  | Ret v ->
    let alloc, off = allocate alloc v in
    emit out (Printf.sprintf "mov rax, [rbp-%d]" (-off));
    alloc

Control Flow: Labels and Jumps

Our IR organizes code into basic blocks with an explicit control-flow graph. Code generation is straightforward: each basic block becomes a label, and each branch becomes a jump.

Consider this IR for a simple if-else:

main:
  @entry:
    const %0, 1
    const %1, 42
    const %2, 0
    lt %3, %0, %2
    br_cond %3, @then, @else
  @then:
    ret %1
  @else:
    const %4, 7
    ret %4

This translates to:

main:
  push rbp
  mov  rbp, rsp
  sub  rsp, 32

  .Lentry:
    mov  rax, 1
    mov  qword [rbp-8], rax
    mov  rax, 42
    mov  qword [rbp-16], rax
    mov  rax, 0
    mov  qword [rbp-24], rax
    mov  rax, [rbp-8]
    cmp  rax, [rbp-24]
    setl al
    movzx rax, al
    mov  qword [rbp-32], rax
    mov  rax, [rbp-32]
    cmp  rax, 0
    jne  .Lthen
    jmp  .Lelse

  .Lthen:
    mov  rax, [rbp-16]
    mov  rsp, rbp
    pop  rbp
    ret

  .Lelse:
    mov  rax, 7
    mov  qword [rbp-24], rax
    mov  rax, [rbp-24]
    mov  rsp, rbp
    pop  rbp
    ret

The mapping is mechanical. Each block label becomes .Llabel in assembly. Each br_cond becomes a compare-and-jump sequence. The epilogue is duplicated at every return point.

The Calling Convention

Function calls are the most complex part of code generation. We follow the System V AMD64 ABI, which is the standard on Linux and most Unix-like systems:

ArgumentRegister
1strdi
2ndrsi
3rdrdx
4thrcx
5thr8
6thr9
7+On the stack (right to left)

Return value goes in rax. The caller-saved registers (rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11) may be clobbered by the call. Since we spill everything to the stack, we don't need to save them — all live values are already in memory.

Here's how we generate code for a call instruction:

src/codegen.rs — Call instruction
IrInstr::Call(dst, name, args) => {
    // Allocate stack slots for all vregs first
    for arg in args {
        alloc.allocate(*arg);
    }
    let off_d = alloc.allocate(*dst);

    // Load arguments into argument registers
    let arg_regs = ["rdi", "rsi", "rdx", "rcx", "r8", "r9"];
    for (i, arg) in args.iter().enumerate() {
        if i < arg_regs.len() {
            let off = alloc.offset(*arg).unwrap();
            out.emit(&format!("mov {}, [rbp-{}]", arg_regs[i], -off));
        }
    }
    out.emit(&format!("call {}", name));
    out.emit(&format!("mov qword [rbp-{}], rax", -off_d));
}
codegen.ml — Call instruction
| Call (dst, name, args) ->
    let arg_regs = [|"rdi"; "rsi"; "rdx"; "rcx"; "r8"; "r9"|] in
    let alloc = List.fold_left (fun alloc arg -> fst (allocate alloc arg)) alloc args in
    let alloc, off_d = allocate alloc dst in
    let _ =
      List.iteri (fun i arg ->
        if i < Array.length arg_regs then
          let (_, off) = allocate alloc arg in
          emit out (Printf.sprintf "mov %s, [rbp-%d]" arg_regs.(i) (-off)))
      args
    in
    emit out (Printf.sprintf "call %s" name);
    emit out (Printf.sprintf "mov qword [rbp-%d], rax" (-off_d));
    alloc
Why we don't need to save registers In the spill-everything strategy, all live values live in stack slots, not registers. When a call happens, the callee might clobber rax, rcx, etc., but that's fine — we just reload from the stack after the call. This is the key advantage of spilling: function calls are free.

The Full Function Codegen

Now let's put it all together. Generating code for a function means:

  1. Emit the function label
  2. Emit the prologue (push rbp, mov rbp rsp, sub rsp frame_size)
  3. For each basic block, emit the block label and then each instruction
  4. Emit the epilogue (mov rsp rbp, pop rbp, ret)
src/codegen.rs — Function codegen
pub fn codegen_func(func: &IrFunc, out: &mut Emitter) {
    let mut alloc = StackAllocator::new();

    // First pass: allocate all vregs
    for block in &func.blocks {
        for instr in &block.instrs {
            allocate_vregs(&mut alloc, instr);
        }
    }

    // Align frame to 16 bytes
    let frame_size = align16(alloc.frame_size());

    // Emit function prologue
    out.emit(&format!("global {}", func.name));
    out.emit(&format!("{}:", func.name));
    out.emit("push rbp");
    out.emit("mov rbp, rsp");
    if frame_size > 0 {
        out.emit(&format!("sub rsp, {}", frame_size));
    }

    // Emit each basic block
    for block in &func.blocks {
        out.emit(&format!(".L{}:", block.label));
        for instr in &block.instrs {
            codegen_instr(instr, &mut alloc, out);
        }
    }

    // Emit epilogue (fallthrough — each ret handles its own)
    out.emit(".Lepilogue:");
    out.emit("mov rsp, rbp");
    out.emit("pop rbp");
    out.emit("ret");
}
codegen.ml — Function codegen
let codegen_func out func =
  let alloc = create () in

  (* First pass: allocate all vregs *)
  let alloc =
    List.fold_left (fun alloc block ->
      List.fold_left (fun alloc instr ->
        allocate_vregs alloc instr) alloc block.instrs)
      alloc func.blocks
  in

  (* Align frame to 16 bytes *)
  let frame_size = align16 alloc.frame_size in

  (* Prologue *)
  emit out (Printf.sprintf "global %s" func.name);
  emit out (Printf.sprintf "%s:" func.name);
  emit out "push rbp";
  emit out "mov rbp, rsp";
  if frame_size > 0 then
    emit out (Printf.sprintf "sub rsp, %d" frame_size);

  (* Basic blocks *)
  List.iter (fun block ->
    emit out (Printf.sprintf ".L%s:" block.label);
    let _ = List.fold_left (fun alloc instr ->
      codegen_instr alloc out instr) alloc block.instrs
    in ()
  ) func.blocks;

  (* Epilogue *)
  emit out "mov rsp, rbp";
  emit out "pop rbp";
  emit out "ret"

What the Output Looks Like

Let's trace through our small but growing language and see what each feature produces.

Minimal: Return a Constant

func main(i32) { return 42; }

Generated assembly (via IR → codegen):

global main
main:
  push rbp
  mov  rbp, rsp
  sub  rsp, 16
  mov  rax, 42
  mov  qword [rbp-8], rax
  mov  rax, [rbp-8]
  mov  rsp, rbp
  pop  rbp
  ret

Notice the redundancy: we load 42 into rax, store it to the stack, then immediately load it back. This is the cost of spilling everything. Our first exercise will fix this.

Arithmetic: Add Two Parameters

func add(i32, a: i32, b: i32) {
  return a + b;
}

Generated assembly:

global add
add:
  push rbp
  mov  rbp, rsp
  sub  rsp, 32

  ; Store incoming args to their stack slots
  mov  qword [rbp-8], rdi   ; a
  mov  qword [rbp-16], rsi  ; b

  ; a + b
  mov  rax, [rbp-8]
  mov  rcx, [rbp-16]
  add  rax, rcx
  mov  qword [rbp-24], rax

  ; return
  mov  rax, [rbp-24]
  mov  rsp, rbp
  pop  rbp
  ret
Parameter passing Parameters arrive in argument registers (rdi, rsi, ...). Our codegen must spill them to stack slots at the start of the function so they're accessible like any other virtual register.

Loops: While

func sum_to(i32, n: i32) {
  let i: i32 = 0;
  let s: i32 = 0;
  while (i < n) {
    s = s + i;
    i = i + 1;
  }
  return s;
}

The IR for this loop has three basic blocks: a pre-header for initialization, a loop header for the condition check, and a loop body. The codegen produces the expected label/jump pattern:

.Lpre_header:
  mov  qword [rbp-8], 0   ; i = 0
  mov  qword [rbp-16], 0  ; s = 0
  jmp  .Lloop_header

.Lloop_header:
  ; i < n → if !(i < n) goto .Lend
  mov  rax, [rbp-8]
  cmp  rax, [rbp-24]  ; n
  setl al
  movzx rax, al
  cmp  rax, 0
  jne  .Lbody
  jmp  .Lend

.Lbody:
  ; s = s + i
  mov  rax, [rbp-16]
  mov  rcx, [rbp-8]
  add  rax, rcx
  mov  qword [rbp-16], rax
  ; i = i + 1
  mov  rax, [rbp-8]
  add  rax, 1
  mov  qword [rbp-8], rax
  jmp  .Lloop_header

.Lend:
  mov  rax, [rbp-16]  ; return s
  mov  rsp, rbp
  pop  rbp
  ret

This is real assembly that will run on any x86-64 machine. The loop actually loops. The condition actually branches. We turned source code into machine instructions.

The Emitter

One small but essential piece: we need a utility that collects assembly text and writes it out. Nothing fancy — just a buffer with emit and write:

pub struct Emitter {
    buffer: String,
}

impl Emitter {
    pub fn new() -> Self {
        Self { buffer: String::new() }
    }

    pub fn emit(&mut self, line: &str) {
        self.buffer.push_str(line);
        self.buffer.push('\n');
    }

    pub fn write(&self, path: &str) -> io::Result<()> {
        fs::write(path, &self.buffer)
    }
}

The "Aha" Moment

Take a step back. We just did something remarkable.

We started with a high-level programming language — functions, let-bindings, if-else, while loops. We transformed it through a series of well-defined stages into x86-64 assembly. Every transformation was mechanical. Every step was something we understood and controlled.

func main(i32) { return 42; } │ ▼ Lexer ────────────────────────────────── [Func, Ident("main"), ...] │ ▼ Parser ────────────────────────────────── Func("main", [], ..., Const(42)) │ ▼ Type Checker ────────────────────────────────── (same tree, with types attached) │ ▼ IR Builder ────────────────────────────────── @entry: const %0, 42; ret %0 │ ▼ Codegen ────────────────────────────────── main: push rbp mov rbp, rsp sub rsp, 16 mov rax, 42 mov qword [rbp-8], rax mov rax, [rbp-8] mov rsp, rbp pop rbp ret │ ▼ Assembler + Linker ────────────────────────────────── ./main → exit code 42

This is the entire compiler pipeline, end to end. Every line of assembly above was generated by code we wrote. Every detail — the prologue, the stack offsets, the calling convention — is something we decided and implemented.

If you run this assembly through NASM and link it, you get a real executable. It runs on real hardware. It returns 42.

Try it yourself Save the generated assembly to a file, run nasm -f elf64 file.asm -o file.o, then cc -o file file.o -no-pie, then ./file; echo $?. You'll see 42 printed. Your code, running on real silicon.

Exercises

1. Constant optimization When we emit const %0, 42, we generate: mov rax, 42; mov [rbp-8], rax; mov rax, [rbp-8]. The final load is redundant if the constant is immediately returned. More generally, if the next instruction uses the value, we waste a store-load. Optimize: for a const followed immediately by ret, just load the constant directly into rax and skip the stack slot entirely.
2. Live range analysis Our stack allocator assigns every vreg a slot, even if two vregs never exist at the same time (their live ranges don't overlap). Add live range analysis: compute which vregs are live at each point, and reuse stack slots for vregs with non-overlapping live ranges.
3. Simple register allocation Instead of spilling everything, reserve a few x86-64 registers (say, r10, r11, r12) for frequently-used vregs. A vreg that's used often in a loop body could stay in a register instead of going to the stack. Track register usage and only spill when registers run out.
4. Handle more than 6 arguments The System V ABI passes the 7th and subsequent arguments on the stack. Extend the call codegen to handle this: for arguments beyond the 6th, push them onto the stack before the call (in reverse order) and adjust rsp afterward.
5. Tail call optimization For a function that ends with return f(x) (a tail call), we can emit jmp f instead of call f; ret. This saves a stack frame and enables recursive functions to run in constant stack space. Implement tail call detection and codegen.
