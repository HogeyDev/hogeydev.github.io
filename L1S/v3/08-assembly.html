Building a Compiler

Chapter 08: The Final Mile

We can generate assembly text from source code. But assembly text isn't an executable. It's a string — characters in a file. How do we cross that final gap from .asm to a program that actually runs?

The answer involves two tools we don't build ourselves: an assembler (NASM) and a linker (cc/ld). Our job is to orchestrate them. We become the conductor of a pipeline that starts with a .is source file and ends with a binary that the OS can load and execute.

The Driver Pipeline

The driver is the top-level orchestrator. It connects every piece of the compiler into a single coherent flow:

Source file (.is) │ ▼ 1. Read source into string │ ▼ 2. Lex → tokens │ ▼ 3. Parse → AST │ ▼ 4. Type check → annotated AST │ ▼ 5. Build IR → IrModule │ ▼ 6. Codegen → assembly string │ ▼ 7. Write assembly to .asm file │ ▼ 8. Invoke NASM: nasm -f elf64 file.asm -o file.o │ ▼ 9. Invoke linker: cc -o file file.o -no-pie │ ▼ Executable (file)

Steps 1-6 are the compiler itself — everything we've built in chapters 1-7. Steps 8-9 delegate to external tools. Step 7 is the bridge between them.

Error Handling Strategy

A good compiler doesn't stop at the first error. It collects as many errors as possible, then presents them to the user all at once. Our pipeline reflects this: each phase collects errors rather than panicking.

The pattern is simple: every phase returns a result that can be either a valid output or a list of errors. Errors flow downward; if any phase fails, we skip codegen and assembly and just print the errors.

Read source → string │ ▼ Lex → Result<Tokens, Errors> │ successes │ failures ▼ ▼ Parse → Result<AST, Errors> Print errors & exit │ successes │ failures ▼ ▼ Type check → Result<AST, Errors> Print errors & exit │ successes │ failures ▼ ▼ Build IR → Result<IrModule, Errors> Print errors & exit │ successes ▼ Codegen → assembly string │ ▼ Write .asm, assemble, link
Error-aware types In Rust, we use Result<T, Vec<Error>>. In OCaml, we use (T, Error list) Result.t. Multi-error accumulation is crucial: finding all bugs in one compile pass is much faster than fixing one at a time.

The Driver

Here's the main function of the compiler, connecting every piece:

src/main.rs
use std::env;
use std::fs;
use std::process;

mod span;
mod token;
mod lexer;
mod ast;
mod parser;
mod symbol;
mod types;
mod check;
mod ir;
mod codegen;
mod emitter;

fn main() {
    let args: Vec<String> = env::args().collect();
    if args.len() < 2 {
        eprintln!("usage: {} <file.is>", args[0]);
        process::exit(1);
    }

    let path = &args[1];

    // 1. Read source
    let source = match fs::read_to_string(path) {
        Ok(s) => s,
        Err(e) => {
            eprintln!("error reading {}: {}", path, e);
            process::exit(1);
        }
    };

    let file = span::SourceFile::new(path, &source);
    let mut errors: Vec<span::Error> = Vec::new();

    // 2. Lex
    let mut lexer = lexer::Lexer::new(&file);
    let tokens = lexer.lex();
    errors.extend(lexer.errors);
    if !errors.is_empty() { report_and_exit(&file, &errors); }

    // 3. Parse
    let mut parser = parser::Parser::new(&tokens, &file);
    let ast = parser.parse();
    errors.extend(parser.errors);
    if !errors.is_empty() { report_and_exit(&file, &errors); }

    // 4. Type check
    let mut checker = check::Checker::new(&file);
    let ast = checker.check(ast);
    errors.extend(checker.errors);
    if !errors.is_empty() { report_and_exit(&file, &errors); }

    // 5. Build IR
    let mut ir_builder = ir::Builder::new();
    let ir_module = ir_builder.build(&ast);
    errors.extend(ir_builder.errors);
    if !errors.is_empty() { report_and_exit(&file, &errors); }

    // 6. Codegen
    let mut emitter = emitter::Emitter::new();
    codegen::codegen_module(&ir_module, &mut emitter);

    // 7. Write assembly
    let asm_path = path.replace(".is", ".asm");
    emitter.write(&asm_path).unwrap();
    println!("wrote {}", asm_path);

    // 8. Assemble with NASM
    let obj_path = path.replace(".is", ".o");
    let nasm_status = process::Command::new("nasm")
        .args(&["-f", "elf64", &asm_path, "-o", &obj_path])
        .status()
        .expect("failed to run nasm (is it installed?)");
    if !nasm_status.success() {
        process::exit(1);
    }

    // 9. Link with cc
    let bin_path = path.replace(".is", "");
    let link_status = process::Command::new("cc")
        .args(&["-o", &bin_path, &obj_path, "-no-pie"])
        .status()
        .expect("failed to run cc (is a C compiler installed?)");
    if !link_status.success() {
        process::exit(1);
    }

    println!("built {}", bin_path);
}

fn report_and_exit(file: &span::SourceFile, errors: &[span::Error]) {
    for error in errors {
        eprintln!("{}", error.display(file));
    }
    process::exit(1);
}
bin/main.ml
let report_and_exit file errors =
  List.iter (fun e -> eprintf "%s\n" (Span.display_error file e)) errors;
  exit 1

let () =
  let args = Sys.get_argv () in
  if Array.length args < 2 then (
    eprintf "usage: %s <file.is>\n" args.(0);
    exit 1
  );
  let path = args.(1) in

  (* 1. Read source *)
  let source = match In_channel.read_all path with
    | s -> s
    | exception e -> eprintf "error reading %s: %s\n" path (Printexc.to_string e); exit 1
  in

  let file = Span.make_source_file path source in
  let errors = ref [] in
  let add_errors es = errors := !errors @ es in

  (* 2. Lex *)
  let tokens, lex_errors = Lexer.lex file in
  add_errors lex_errors;
  if !errors <> [] then report_and_exit file !errors;

  (* 3. Parse *)
  let ast, parse_errors = Parser.parse tokens file in
  add_errors parse_errors;
  if !errors <> [] then report_and_exit file !errors;

  (* 4. Type check *)
  let ast, check_errors = Checker.check file ast in
  add_errors check_errors;
  if !errors <> [] then report_and_exit file !errors;

  (* 5. Build IR *)
  let ir_module, ir_errors = Ir.build ast in
  add_errors ir_errors;
  if !errors <> [] then report_and_exit file !errors;

  (* 6. Codegen *)
  let emitter = Emitter.create () in
  Codegen.codegen_module emitter ir_module;

  (* 7. Write assembly *)
  let asm_path = Filename.with_extension path "asm" in
  Emitter.write emitter asm_path;
  printf "wrote %s\n" asm_path;

  (* 8. Assemble with NASM *)
  let obj_path = Filename.with_extension path "o" in
  let nasm_rc = Sys.command (Printf.sprintf
    "nasm -f elf64 %s -o %s" asm_path obj_path)
  in
  if nasm_rc <> 0 then exit 1;

  (* 9. Link with cc *)
  let bin_path = Filename.remove_extension path in
  let link_rc = Sys.command (Printf.sprintf
    "cc -o %s %s -no-pie" bin_path obj_path)
  in
  if link_rc <> 0 then exit 1;

  printf "built %s\n" bin_path

That's it. Every piece of the compiler, connected end-to-end, in about 80 lines of code. The pipeline is linear — each stage feeds the next — and the error handling is a simple gate: any error stops the pipeline and prints diagnostics.

Testing the Pipeline

With the driver in place, we can write tests at every level of the compiler. The key question at each level is: "what's the smallest unit we can test independently?"

Build Systems

How do you build the compiler itself? The answer depends on which language you're using:

Cargo.toml
[package]
name = "our-compiler"
version = "0.1.0"
edition = "2021"

[[bin]]
name = "isc"
path = "src/main.rs"

[dev-dependencies]
tempfile = "3"
dune-project & dune files
; dune-project
(lang dune 3.0)
(name compiler)

; bin/dune
(executable
 (name main)
 (public_name isc)
 (libraries unix))

With the build system in place, building the compiler is a single command: cargo build or dune build. The output is a binary called isc (our "IS compiler") that you can invoke on any source file.

A Complete Compilation Session

Let's see the whole thing in action — from source file to running executable:

$ cat > test.is
func main(i32) {
    return 42;
}
$ cargo run -- test.is
   Compiling our-compiler v0.1.0
    Finished dev [unoptimized + debuginfo] target(s) in 0.32s
     Running `target/debug/isc test.is`
wrote test.asm
$ nasm -f elf64 test.asm -o test.o
$ cc -o test test.o -no-pie
$ ./test; echo $?
42

That 42 at the end is the exit code of our compiled program. It was written in our language, compiled by our compiler, assembled by NASM, linked by cc, and executed by the OS. Every piece of that chain is something we understand.

What the compiler produced The file test.asm contains the x86-64 assembly we generated in chapter 7. NASM turns that text into machine code (a .o file). The linker (cc) resolves any external references and produces the final ELF binary. The OS loads that binary and starts executing at main.

Debugging and Diagnostics

When something goes wrong, you need visibility into each pipeline stage. Here are the debugging techniques that save the most time:

Dump the IR

Print the IR module before codegen to verify that the IR builder produced what you expect. A --dump-ir flag is invaluable:

$ ./isc --dump-ir test.is
@.main:
  @entry:
    const %0, 42
    ret %0

Dump the Assembly

Print the generated assembly to stdout instead of writing to a file. A --dump-asm flag lets you inspect what the codegen produced without creating files.

Single-Step Through Small Programs

Write the smallest possible test case for the feature you're debugging. Start with return 42 and work up. If arithmetic is broken, test return 1 + 1. If control flow is broken, test a simple if (1) { return 10; } else { return 20; }.

Use GDB for the hard cases If the assembly looks right but the program produces wrong results, run the binary under GDB: gdb ./test. Set a breakpoint at main, step through instruction by instruction with stepi, and inspect registers with info registers. You'll see exactly where the computation diverges from your expectations.

Exercises

1. Add a --dump-ir flag Add a command-line flag that prints the IR to stderr before codegen. This is the single most useful debugging tool you can add. Implement it by checking env::args() for --dump-ir and calling a display_ir function after the IR builder runs.
2. Add a --dump-ast flag Similarly, add --dump-ast to print the pretty-printed AST after parsing and type checking. This helps debug parser and type-checker issues independently from later pipeline stages.
3. Add a REPL Implement a read-eval-print loop that takes expressions interactively, compiles them on the fly, and runs them. This requires dynamic code generation: write assembly to a temp file, assemble, link into a shared library with cc -shared, load it with dlopen/dlsym, and call the function. This is a taste of JIT compilation.
4. Parallel compilation If the source file contains multiple independent functions, compile them in parallel using threads (Rust's std::thread or OCaml's Thread module). Measure the speedup. This is what production compilers do, and it's surprisingly easy to add.
5. Error recovery in the parser Our parser stops at the first syntax error. Implement error recovery: when a parse error occurs, skip tokens until you find a synchronization point (like func or }) and continue parsing. This lets the compiler find multiple errors in one pass, which is much more user-friendly.
