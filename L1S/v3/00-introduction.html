Building a Compiler

Chapter 00: The Spark

You write code. The computer runs it. How?

It's such an ordinary thing that we barely think about it anymore. You open a file, type some text, save it, run a command, and a program executes. But the gap between what you write and what the computer does is enormous. A modern CPU understands nothing but numbers — opcodes, registers, memory addresses. You write:

int main() {
    return 42;
}

But the computer needs something more like:

55                ; push rbp
48 89 e5          ; mov rbp, rsp
b8 2a 00 00 00    ; mov eax, 42
5d                ; pop rbp
c3                ; ret

These aren't even the same kind of thing. One is a statement of intent. The other is a sequence of mechanical instructions. One says what should happen; the other says how to make it happen, at the level of individual register moves and memory operations.

Something has to bridge this gap. That something is a compiler.

What Is a Compiler, Really?

The word "compiler" sounds imposing. It conjures images of dark magic, of sprawling C++ codebases, of things that only PhDs in programming languages understand. But the core idea is simple:

A compiler is a translator. It takes a program written in one language (the source language) and produces an equivalent program in another language (the target language).

That's it. The source is your high-level code. The target is assembly or machine code. The compiler is the translator in between.

Translation happens in stages because each stage solves a specific subproblem:

  • Lexing: "What are the words?" — group characters into tokens like keywords, identifiers, numbers.
  • Parsing: "What is the structure?" — arrange tokens into a tree that reflects the grammar of the language.
  • Semantic analysis: "Does it make sense?" — check that names refer to real things, that types are consistent.
  • IR generation: "How do we get closer to the machine?" — translate the high-level tree into a simpler, lower-level representation.
  • Code generation: "What are the actual instructions?" — translate the IR into assembly language for a real CPU.
  • Assembly and linking: "How do we make an executable?" — turn assembly into machine code and package it up.

Each stage is simpler than the whole problem. That's the secret: a compiler is not one hard thing; it's several easy things chained together.

The Question Beneath the Question

When you ask "how does a computer run my code?", you're really asking a stack of questions, each nested inside the last:

  • How does text become meaning? (What does return 42 even mean?)
  • How does meaning become instructions? (How do you turn "return the value 42" into mov eax, 42; ret?)
  • How do instructions become electrons? (But that's computer architecture, not compilers.)

This book answers the first two questions. We will build two compilers — one in Rust and one in OCaml — that translate a small but real programming language all the way down to x86-64 assembly. By the time we're done, every transformation from source text to machine code will be something you can hold in your head.

The Pipeline at 30,000 Feet

Every compiler follows the same basic arc. It's worth seeing the whole thing before we dive into any one part:

Source Code (text) │ ▼ ┌──────────────────────┐ │ 1. Lexing │ ── characters → tokens ├──────────────────────┤ │ 2. Parsing │ ── flat tokens → tree (AST) ├──────────────────────┤ │ 3. Semantic │ ── names → definitions, types │ Analysis │ ├──────────────────────┤ │ 4. IR │ ── high-level tree → low-level │ Generation │ instructions (IR) ├──────────────────────┤ │ 5. Codegen │ ── IR → assembly ├──────────────────────┤ │ 6. Assembly & │ ── assembly → machine code │ Linking │ └──────────────────────┘ │ ▼ Executable (binary)

Each chapter of this book builds one stage of this pipeline. The output of each stage becomes the input of the next. By chapter 8, the pipeline is complete: we feed it source code and get back an executable file.

Let's walk through what happens when you run our eventual compiler on that tiny C-like program above. This is a preview of everything we'll learn:

Source: func main(i32) { return 42; } │ ▼ Lexer Tokens: [Func, Ident("main"), OpenParen, I32, CloseParen, OpenBrace, Return, NumLiteral(42), Semicolon, CloseBrace] │ ▼ Parser AST: Program └── Function "main" ├── ReturnType: I32 ├── Params: [] └── Body: Block └── Return └── NumLiteral(42) │ ▼ Semantic Analysis (checks: main exists, 42 is i32, return matches type) │ ▼ IR Generation IR: function main(): mov eax, 42 ret │ ▼ Codegen Assembly: .globl main main: push rbp mov rbp, rsp mov eax, 42 pop rbp ret │ ▼ Assembler + Linker Executable: a.out

Every arrow in this diagram is something we will build from scratch. Every transformation is a piece of code we will write, understand, and debug. By the time we're done, this pipeline won't be a diagram — it will be a running program on your computer.

What Does a Compiler Look Like?

Let's step back and think about the big picture before we dive into code. A compiler is, at its heart, a function:

// Input:  source code as a string
// Output: assembly code as a string (or an executable file)
fn compile(source: &str) -> String;

Everything inside that function is what we're building. The pipeline transforms the data step by step:

fn compile(source: &str) -> Result<String, Vec<Error>> {
    let tokens = lex(source)?;
    let ast = parse(tokens)?;
    let checked_ast = check(ast)?;
    let ir = lower(checked_ast);
    let asm = codegen(ir);
    Ok(asm)
}

Each function call is a compiler phase. The ? operator means "if this phase fails, stop and report the error." This is the entire architecture of our compiler expressed in nine lines. Each phase is a self-contained transformation, and the phases compose linearly. That's beautiful — it means we can build, test, and understand each phase independently.

The Language We'll Build

We need a language to compile. We could pick an existing one, but half the fun is designing our own. More importantly, building the language as we go means every feature is motivated by a concrete need. Nothing is arbitrary. The language doesn't descend from on high as a spec — it emerges naturally from the problems we're solving.

Let's look at a complete program in the language we're going to create:

func main(i32) {
    return 42;
}

This is the simplest program that does something observable: it returns the number 42. Let's break down each piece:

PieceWhat it means
func"I'm defining a function" — a named block of code you can call
mainThe name of this function. Every program starts here.
(i32)The function takes no parameters; i32 is the return type (a 32-bit integer)
{ ... }The body of the function — the instructions to execute
return 42;Exit the function and give back the value 42

Our language will grow as we need it to. When we need variables, we'll add let. When we need decisions, we'll add if. When we need repetition, we'll add while. Each addition teaches us something new about the compiler pipeline.

Here's a sneak peek of what our language will look like by the middle of the book:

func add(i32, a: i32, b: i32) {
    return a + b;
}

func main(i32) {
    let x: i32 = 42;
    let y: i32 = add(x, 10);
    if (y > 50) {
        return 1;
    } else {
        return 0;
    }
}

And later, when we need loops:

func sum_to_n(i32, n: i32) {
    let i: i32 = 0;
    let total: i32 = 0;
    while (i < n) {
        total = total + i;
        i = i + 1;
    }
    return total;
}
Naming the language Every compiler book needs a name for its language. We'll call ours Sparks — small, bright, self-contained pieces of computation that can combine into larger programs. The file extension will be .spark.

Why Two Compilers?

You might wonder: why build this in two languages instead of one? There are two reasons, and both are central to the philosophy of this book.

First, the concepts of compilation are language-independent. A lexer is a lexer whether you write it in Rust, OCaml, Haskell, or Python. By seeing the same ideas expressed in two very different languages, you'll recognize the ideas themselves rather than just memorizing syntax. If you know Rust, reading the OCaml side will teach you OCaml through the lens of something you already understand — and vice versa.

Second, Rust and OCaml represent two different philosophies of systems programming, and compilers sit at an interesting intersection of both. Rust gives you fine-grained control over memory and a powerful type system that enforces correctness at compile time. OCaml gives you a remarkably expressive type system with pattern matching and algebraic data types that make tree-shaped data (which compilers have everywhere) feel utterly natural. Each has something to teach.

Here's a quick comparison to illustrate:

AspectRustOCaml
Type systemOwnership, borrowing, lifetimesAlgebraic data types, parametric polymorphism
Pattern matchingPowerful, but limited by ownershipFirst-class, exhaustive, everywhere
Memory modelExplicit ownership and borrowingGarbage collected
MutabilityOpt-in with mutOpt-in with ref
Module systemModules + cratesModules + functors (advanced)
Expressiveness sweet spotSystems programming with safetySymbolic computation and DSLs

Both compilers in this book are designed to be idiomatic in their respective languages while expressing the same concepts. Reading both sides will make you a better programmer regardless of which language you prefer.

The Two Compilers

Here's the structure of each compiler. They're laid out identically so you can always find the corresponding file:

code/rust/ — directory structure
rust-compiler/
├── Cargo.toml
└── src/
    ├── main.rs          // driver: reads files, runs pipeline
    ├── span.rs          // text positions and source files
    ├── token.rs         // token definitions
    ├── lexer.rs         // character → token
    ├── ast.rs           // abstract syntax tree types
    ├── parser.rs        // token → AST
    ├── symbol.rs        // symbol table for name resolution
    ├── types.rs         // type representation and checking
    ├── check.rs         // semantic analysis pass
    ├── ir.rs            // intermediate representation
    ├── codegen.rs       // IR → x86-64 assembly
    └── emitter.rs       // assembly text output
code/ocaml/ — directory structure
ocaml-compiler/
├── Makefile
├── dune-project
└── bin/
    ├── main.ml          // driver: reads files, runs pipeline
    ├── span.ml          // text positions and source files
    ├── token.ml         // token definitions
    ├── lexer.ml         // character → token
    ├── ast.ml           // abstract syntax tree types
    ├── parser.ml        // token → AST
    ├── symbol.ml        // symbol table for name resolution
    ├── types.ml         // type representation and checking
    ├── check.ml         // semantic analysis pass
    ├── ir.ml            // intermediate representation
    ├── codegen.ml       // IR → x86-64 assembly
    └── emitter.ml       // assembly text output

Every file in the Rust compiler has a corresponding file in the OCaml compiler. When we talk about span.rs in a chapter, you can open span.ml and see the same concepts expressed in OCaml. This one-to-one mapping is deliberate: it lets you focus on the ideas, not the language.

The dependency order between modules also mirrors the pipeline. span.rs depends on nothing but the standard library. token.rs depends on span.rs. lexer.rs depends on both. parser.rs depends on the lexer and the token types. By the time we reach codegen.rs, it depends on everything before it. This layered design means you can read the files in order and never encounter a forward reference you don't understand.

Getting Started with the Companion Code

The companion code lives in two directories: code/rust/ and code/ocaml/. Each is a complete, buildable project. Here's how to get started:

Rust:

cd code/rust
cargo build           # Build the compiler
echo 'func main(i32) { return 42; }' | cargo run  # Compile a program

OCaml:

cd code/ocaml
make                  # Build the compiler
echo 'func main(i32) { return 42; }' | ./spark  # Compile a program

Both compilers read source from stdin by default and write assembly to stdout. You can also pass a filename as an argument:

cargo run -- hello.spark
# or
./spark hello.spark

Throughout the book, you'll see code blocks with filenames like src/span.rs or span.ml. These correspond directly to files in the companion code. When you're reading about a piece of code, you can open the companion file and follow along — the code in the book is the code in the repository.

Each chapter's exercises invite you to modify the compiler and see the effects. Don't be afraid to experiment. The compiler is your sandbox; the worst that can happen is it doesn't compile, and you learn something from fixing it.

A note on Rust editions and OCaml versions The Rust compiler targets the latest stable Rust (edition 2021 or later). The OCaml compiler targets OCaml 5.x with dune as the build system. If you're using an older version, the code should still work, but you may need to adjust the build configuration. Check the README in each directory for version-specific notes.

Why Build a Compiler?

This is a fair question. You could use someone else's compiler and never think about how it works. Why invest the time?

It changes how you read code. Once you've built a parser, you never look at a source file the same way again. You see the syntax tree. You recognize where the lexer had to disambiguate. You understand why some error messages are easy to generate and others are hard. Every language feature becomes something you think about in terms of "how would I implement this?"

It's not as hard as you think. A production compiler like LLVM or GHC is millions of lines of code, but the core of a compiler — the pipeline from source text to assembly — is something you can build in a weekend once you know how. The principles scale. A toy compiler isn't a toy version of a real compiler; it's the same thing, just smaller. The same techniques that compile our tiny Sparks language are the techniques that compile Rust, OCaml, and C.

It's the most satisfying kind of programming. There is an almost alchemical pleasure in watching text you wrote transform into instructions a machine follows. Each stage of the pipeline is a small miracle, and when you've built each stage yourself, the whole thing feels like magic that you're allowed to understand. There's a moment in every compiler project — usually around the time you first run a compiled program — where you realize: I made this. These bytes on disk came from my code.

A word on prerequisites We assume you know how to program — you've written loops, functions, and data structures. We assume basic familiarity with Rust or OCaml (or the willingness to learn as we go — both are well within reach for an experienced programmer). We do not assume you know anything about compilers, assembly, or computer architecture beyond the absolute basics (you know what a CPU, registers, and memory are). Everything else we'll build together.

Who This Book Is For

This book is for you if:

  • You've ever wondered what happens between "save the file" and "run the program"
  • You've used compilers for years and want to understand them from the inside out
  • You're a Rust programmer who wants to learn OCaml (or vice versa) through a shared project
  • You want to build something real that you can hold in your hands — a working compiler from scratch
  • You enjoy that moment when a bunch of disparate pieces click together into a unified whole

This book is not for you if:

  • You want a deep theoretical treatment of formal language theory, automata, or denotational semantics
  • You want to build a production-grade compiler that optimizes better than LLVM
  • You want to skip straight to the "interesting" parts without building the foundation

There are excellent books for those goals. This one is about the joy of building something real, from first principles, one piece at a time.

How to Read This Book

Each chapter has the same structure:

  1. A problem — something that the previous chapter's work makes necessary or possible. We always lead with a question, never with an answer.
  2. The core idea — the concept that solves the problem, explained intuitively with analogies and concrete examples.
  3. Implementation — the actual code, in both Rust and OCaml, with tabs to switch between them. Code comes with filenames so you can find it in the companion repository.
  4. Exercises — ways to extend what we've built, test your understanding, or explore further. Some are small (add a method), some are larger (add a language feature).

You can read the Rust side, the OCaml side, or both. The tabs let you pick your language and stay in it, or switch back and forth to compare. Both compilers are fully functional — you can run the exercises and see your changes in action.

The exercises are the heart of the book. Reading about compilers is one thing; building them is another. The exercises range from straightforward to challenging. At minimum, do the first exercise in each chapter — they build on each other and ensure you understand the core concepts before moving on.

A Note on Testing and Debugging

Compilers are unusual programs to debug. You can't just print out the answer and check if it's right — the "answer" is a compiled binary, and if it's wrong, you need to trace back through the pipeline to find the bug. Here's the approach we'll use throughout the book:

  • Test each phase in isolation. The lexer should be tested on its own before you feed its output to the parser. The parser should be tested on its own before you feed its output to the semantic checker. Each phase has a well-defined input and output, so you can test them independently.
  • Pretty-print everything. We'll write functions to display tokens, ASTs, and IR in a human-readable form. Seeing the actual output of each phase is the fastest way to find bugs.
  • Start with the simplest possible program. Our first test program is func main(i32) { return 42; }. It has no variables, no control flow, no function calls. If this compiles correctly, we add one feature at a time.
  • The golden path: single-stepping the pipeline. When something goes wrong, trace a single program through every phase. Lex it by hand. Parse it by hand. Check it by hand. Then compare with what your compiler actually produces. The bug will reveal itself.

Throughout the book, we'll include test programs alongside each implementation. Running the tests after each chapter should give you confidence that everything works before you move on to the next piece.

What We Won't Cover

No book can cover everything. Here's what we deliberately leave out, along with pointers to where you can learn more:

  • Optimization passes. Our compiler generates straightforward code without optimization. For that, study the LLVM optimization pipeline or the Dragon Book's chapter on data-flow analysis.
  • Garbage collection. Our language doesn't allocate heap memory (no strings, no arrays, no closures). For GC, read about the Cheney collector or generational GC in Jones's Garbage Collection.
  • Formal semantics. We're not proving correctness. For that, study Benjamin Pierce's Types and Programming Languages or the Software Foundations series.
  • Linkers and loaders. We use the system assembler and linker. For the deep story, read John Levine's Linkers and Loaders.

Consider these the "advanced" modules of our compiler — the places where you can take what we've built and extend it in any direction.

Both compile the same language to x86-64 assembly. Both are well under 2000 lines. Both are designed to be read alongside this book — the files are named the same as the code headers in each chapter.

About This Third Edition

If you've read earlier versions of this book, you'll notice this is a complete rewrite. The first edition was a tour of an existing compiler codebase. The second was a concept-first reboot. This third edition builds two complete compilers from first principles, side by side, in Rust and OCaml.

The biggest change is pedagogical: instead of showing you a finished compiler and explaining how it works, we're discovering the design together. Every language feature, every data structure, every design decision is motivated by a problem that arises naturally from what we've already built. The compiler doesn't exist as a blueprint we're following — it emerges from the journey.
