Building a Compiler

4. The Shape of a Program

The parser gives us structure — but what does that structure actually look like? When parse_func_decl returns, what does it return? We have been hand-waving with names like FuncDecl and Expr::Binary, but we have not pinned down what these things are.

This chapter fixes that. We will define the data types that represent parsed programs — the Abstract Syntax Tree (AST). The AST is the skeleton of our compiler. Every phase after parsing works with the AST, not with source text or tokens.

What the Parser Built

Consider this program: 

func add(i32, a: i32, b: i32) {
    return a + b;
}

The parser consumed tokens and produced a tree. Let us draw that tree:

Program decls FuncDecl name: "add" ret_type: Prim("i32") params: [ Param { name: "a", ty: Prim("i32") }, Param { name: "b", ty: Prim("i32") } ] body: Block stmts: Return expr: Binary op: Add lhs: Ident("a") rhs: Ident("b")

This tree is the AST. Every node is a data structure that captures the meaning of the corresponding source construct, stripped of purely syntactic decoration.

What we throw away Notice what is not in the tree: semicolons, braces, parentheses around the parameter list, commas between parameters. These are syntactic sugar — they helped the parser navigate, but the rest of the compiler does not need them. The AST is a cleaner, more abstract representation.

Defining the AST Types

Our language needs a handful of node types. We will define them once, and both our Rust and OCaml compilers will use the same conceptual structure.

The Top Level: Program and Declarations

A program is a list of declarations. A declaration is either a function or a global variable:

src/ast.rs
pub struct Program {
    pub decls: Vec<Decl>,
}

pub enum Decl {
    Func(FuncDecl),
    Var(VarDecl),
}

pub struct FuncDecl {
    pub name: String,
    pub ret_type: Type,
    pub params: Vec<Param>,
    pub body: Block,
}

pub struct Param {
    pub name: String,
    pub ty: Type,
}

pub struct VarDecl {
    pub name: String,
    pub ty: Type,
    pub init: Box<Expr>,
}
ast.ml
type program = {
  decls: decl list;
}

type decl =
  | Func of func_decl
  | Var  of var_decl

and func_decl = {
  name:     string;
  ret_type: typ;
  params:   param list;
  body:     block;
}

and param = {
  name: string;
  ty:   typ;
}

and var_decl = {
  name: string;
  ty:   typ;
  init: expr;
}
and in OCaml OCaml uses and to connect mutually recursive type definitions. Since decl mentions func_decl, which mentions block, which mentions stmt, which mentions expr — these are all circular. The and keyword lets us define them together.

Statements

A Block is a list of statements. Each statement is one of the things that can appear inside a function body:

src/ast.rs
pub struct Block {
    pub stmts: Vec<Stmt>,
}

pub enum Stmt {
    Return {
        expr: Box<Expr>,
    },
    VarDecl {
        name: String,
        ty: Type,
        init: Box<Expr>,
    },
    If {
        cond: Box<Expr>,
        then_block: Block,
        else_block: Option<Block>,
    },
    While {
        cond: Box<Expr>,
        body: Block,
    },
    Block(Block),
    Expr(Box<Expr>),
}
ast.ml
and block = {
  stmts: stmt list;
}

and stmt =
  | Return  of { expr: expr }
  | VarDecl of { name: string; ty: typ; init: expr }
  | If      of { cond: expr; then_block: block; else_block: block option }
  | While   of { cond: expr; body: block }
  | Block   of block
  | Expr    of expr

Each variant corresponds to a production rule in our grammar. The Return variant wraps a single expression. The If variant has a condition, a then-block, and an optional else-block. The Expr variant handles expression statements like x = 5;.

Expressions

Expressions are the richest part of the AST. They nest arbitrarily through the Binary and Unary variants:

src/ast.rs
pub enum Expr {
    Int(i64),
    Bool(bool),
    Ident(String),
    Binary {
        op: BinaryOp,
        lhs: Box<Expr>,
        rhs: Box<Expr>,
    },
    Unary {
        op: UnaryOp,
        rhs: Box<Expr>,
    },
    Call {
        callee: Box<Expr>,
        args: Vec<Expr>,
    },
    Cast {
        expr: Box<Expr>,
        target: Type,
    },
    Assign {
        target: Box<Expr>,
        value: Box<Expr>,
    },
}

pub enum BinaryOp {
    Add, Sub, Mul, Div,
    Eq, Neq, Lt, Gt, Le, Ge,
}

pub enum UnaryOp {
    Neg, Not,
}
ast.ml
and expr =
  | Int    of int
  | Bool   of bool
  | Ident  of string
  | Binary of { op: binary_op; lhs: expr; rhs: expr }
  | Unary  of { op: unary_op; rhs: expr }
  | Call   of { callee: expr; args: expr list }
  | Cast   of { expr: expr; target: typ }
  | Assign of { target: expr; value: expr }

and binary_op =
  | Add | Sub | Mul | Div
  | Eq | Neq | Lt | Gt | Le | Ge

and unary_op =
  | Neg | Not

Types

Types are simple for now — primitive names (like i32), pointers, and arrays:

src/ast.rs
pub enum Type {
    Prim(String),     // i32, bool, etc.
    Ptr(Box<Type>),    // *i32
    Array(Box<Type>, Option<usize>), // [i32; 10] or []i32
}
ast.ml
and typ =
  | Prim  of string           (* i32, bool, etc. *)
  | Ptr   of typ              (* *i32 *)
  | Array of typ * int option (* [i32; 10] or []i32 *)
Why Box in Rust? Rust needs to know the size of every type at compile time. An Expr contains Binary, which contains two Exprs — this would be infinitely large! Box<Expr> puts the child on the heap, giving us a fixed-size pointer. In OCaml, this is not an issue because all types are heap-allocated by default.

AST vs. CST: What We Discard and Why

The parser actually builds a Concrete Syntax Tree (CST) first — a tree that preserves every token, every parenthesis, every semicolon. Then we transform it into the AST by discarding the noise.

FeatureCST (Concrete)AST (Abstract)
SemicolonsPresentAbsent
Braces { }PresentAbsent (replaced by Block node)
Parentheses ( )Present (for grouping)Absent (grouping is implicit in tree shape)
CommasPresentAbsent
Keywords (return, let)Present as tokensReplaced by variant/constructor name
Whitespace/commentsAbsent (lexer stripped them)Absent
Operator precedenceFlat (from tokens)Hierarchical (from tree shape)

Why bother discarding things? Because the rest of the compiler — type checking, code generation — does not care about semicolons. They are syntax, not semantics. Carrying them forward would mean every later phase has to deal with noise. The AST is the essence of the program, stripped of the accidentals of textual representation.

Source text: return a + b; CST (concrete): ReturnKeyword Ident("a") Plus Ident("b") Semicolon AST (abstract): Return Binary(Add, Ident("a"), Ident("b"))

Walking Through the Full AST

Let us build the complete AST for a slightly larger program and visualize it: 

func main(i32) {
    let x: i32 = 10;
    let y: i32 = x * 2;
    if (y > 5) {
        return y;
    } else {
        return 0;
    }
}
Program decls: FuncDecl name: "main" ret_type: Prim("i32") params: [] body: Block stmts: [0] VarDecl name: "x" ty: Prim("i32") init: Int(10) [1] VarDecl name: "y" ty: Prim("i32") init: Binary op: Mul lhs: Ident("x") rhs: Int(2) [2] If cond: Binary op: Gt lhs: Ident("y") rhs: Int(5) then_block: Block stmts: Return expr: Ident("y") else_block: Block stmts: Return expr: Int(0)

Trace through it: the function body is a Block with three statements. The first two are variable declarations; their init expressions are an integer literal and a binary multiplication. The third is an If whose condition is a comparison. Each branch is another Block containing a single Return.

What the AST Enables

With the AST defined, we have a language-independent representation of the program. This is powerful because:

  • Multiple passes: We can walk the tree multiple times, each pass doing one thing. First pass: collect all function names. Second pass: type check. Third pass: generate code.
  • Correctness: The tree structure guarantees well-formedness. You cannot construct a Return with two expressions, or a Binary with one child — the type system prevents it.
  • Transformations: We can rewrite the tree. Want to inline a function? Replace a Call node with a copy of the function body. Want to constant-fold? Replace Binary(Add, Int(2), Int(3)) with Int(5).
  • Frontend/backend independence: The AST is the contract between the frontend (lexing + parsing + type checking) and the backend (code generation). As long as both sides agree on the AST types, we can change the source language syntax without touching the code generator.
The AST is the spine of the compiler Everything from here on operates on the AST. The lexer and parser are behind us. Type checking walks the AST. Code generation walks the AST. The AST is the world that the rest of the compiler inhabits.

Pretty-Printing the AST

A useful exercise is to write a function that converts the AST back into source code. This is called pretty-printing or unparsing. It is a great way to debug your parser — parse a file, then pretty-print the resulting AST, and compare with the original.

The pretty-printer is a recursive traversal of the tree, emitting strings for each node:

src/pretty.rs
fn pretty_expr(expr: &Expr) -> String {
    match expr {
        Expr::Int(n)     => n.to_string(),
        Expr::Bool(true) => "true".to_string(),
        Expr::Bool(false) => "false".to_string(),
        Expr::Ident(s)   => s.clone(),
        Expr::Binary { op, lhs, rhs } => {
            format!("({} {} {})",
                pretty_expr(lhs),
                pretty_op(op),
                pretty_expr(rhs))
        }
        Expr::Unary { op, rhs } => {
            format!("({}{})", pretty_unary(op), pretty_expr(rhs))
        }
        _ => "<expr>".to_string(),
    }
}
pretty.ml
let rec pretty_expr = function
  | Int n      -> string_of_int n
  | Bool true  -> "true"
  | Bool false -> "false"
  | Ident s    -> s
  | Binary { op; lhs; rhs } ->
      Printf.sprintf "(%s %s %s)"
        (pretty_expr lhs) (pretty_op op) (pretty_expr rhs)
  | Unary { op; rhs } ->
      Printf.sprintf "(%s%s)" (pretty_unary op) (pretty_expr rhs)
  | _ -> "<expr>"

What Comes Next

We have defined the shape of our programs. The AST is a clean, recursive data structure that captures everything the compiler needs to know about the source code, and nothing it does not.

But the AST is just a tree of nodes. It does not know what names are in scope, what types expressions have, or whether return is used in a non-void function. Those are semantic questions, and they are the subject of the next chapter.

Exercises

1. Pretty-printer for declarations Extend the pretty-printer to handle FuncDecl and VarDecl. Pretty-print func add(i32, a: i32, b: i32) { return a + b; } and verify the output matches the input (modulo whitespace).
2. AST differ Write a function that compares two ASTs for structural equality (ignoring positions/spans). Use it to verify that parsing and pretty-printing is a round-trip: source == pretty(parse(source)).
3. Add a node type: Break Extend the AST with a Break statement for breaking out of loops. What does this require in Stmt? Does it change any existing nodes? How does the parser know it is inside a loop?
4. String literals Add string literals to the AST. What node type do you need? What about escape sequences like \n? Does the lexer or the parser handle escapes?
5. AST visitor pattern Implement a generic AST visitor that walks every node in the tree and calls a callback. Then implement a "count variables" pass using only the visitor. This pattern is useful for analyses that do not need to modify the tree.
