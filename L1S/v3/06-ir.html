Building a Compiler

6 · Bridging the Gap

The AST is tied to source language syntax. But the machine doesn't know about if-statements, while-loops, or function declarations. We need something in between.

Think about what we have: a beautiful, type-checked AST. Every node represents some concept from our source language — If, While, FuncDecl, Let. These are perfect for analysis. We can ask "what type does this If return?" or "what variables does this While reference?"

Now think about what the machine wants: flat sequences of instructions. Load a value into a register. Add two registers. Compare. Jump if equal. The machine doesn't have if. It has cmp and jne. It doesn't have while. It has jmp and conditional branches.

We need a translator. Someone who speaks both "high-level source" and "low-level machine" fluently. That translator is an intermediate representation (IR).

Why not go directly from AST to assembly? We could. Early compilers did. But separating IR generation from assembly generation gives us enormous flexibility: we can optimize the IR, target multiple architectures from the same IR, and reason about programs at a level that's simpler than the full AST. Think of IR as the compiler's lingua franca — a universal language that every compiler phase understands.

The Problem with the AST

Let's look at a concrete example. Here's a simple function in our language: 

func max(i32, a: i32, b: i32) {
  if (a > b) {
    return a;
  } else {
    return b;
  }
}

The AST for this looks like a tree:

FuncDecl "max" ret=i32 params=[a:i32, b:i32] └── Block └── If ├── cond: Binary Gt │ ├── Ident "a" │ └── Ident "b" ├── then: Block │ └── Return │ └── Ident "a" └── else: Block └── Return └── Ident "b"

This tree is great for analysis. We can traverse it, check types, find unused variables, and so on. But to generate code, we need to linearize it. We need to decide: what order do we evaluate things? Where do we store intermediate results? How do we encode the branch?

The IR answers all three questions. It's a linear representation of the computation. Every expression becomes a sequence of instructions that compute into virtual registers. Every control flow construct becomes basic blocks connected by branches.

Designing Our IR

Our IR has three levels of structure:

Concept IR Structure Description
Program IrModule List of functions + global variables
Function IrFunction Name, parameters, list of basic blocks
Block IrBlock Label, instruction list, terminator
Value %0, %1, ... Virtual register (unlimited supply)

The key insight: virtual registers are an unlimited resource. Unlike real CPU registers (which number 16 on x86-64), we can create as many virtual registers as we want. The register allocator (later) will map them to real registers or spill them to memory. For now, we just number them: %0, %1, %2, and so on.

IR Instruction Set

Our IR is a simple three-address code. Most instructions take the form: %dest = op %src1, %src2.

Instruction Format Description
const %d = const val Load an integer constant into a vreg
add %d = add %a, %b Integer addition
sub %d = sub %a, %b Integer subtraction
mul %d = mul %a, %b Integer multiplication
div %d = div %a, %b Integer division
eq %d = eq %a, %b Equality comparison (result is 0 or 1)
lt %d = lt %a, %b Less-than comparison
gt %d = gt %a, %b Greater-than comparison
and %d = and %a, %b Bitwise AND
or %d = or %a, %b Bitwise OR
not %d = not %a Bitwise NOT
neg %d = neg %a Arithmetic negation
load %d = load slot Load from a stack slot (local variable)
store store slot, %a Store to a stack slot
call %d = call name(%a, %b) Call a function

And the terminators (which end a basic block):

Terminator Format Description
ret ret %a Return from function with value
br br label Unconditional branch to a block
br_cond br_cond %c, tlab, flab Branch on condition (non-zero = true)

Building IR from AST

Now we reach the heart of this chapter: translating the AST into IR. The process follows a simple pattern for every AST node:

  • Statements produce side effects (stores, calls) and linear control flow.
  • Expressions produce a value in a virtual register.
  • Control flow (if, while) produces branch instructions between basic blocks.

Let's define our IR data structures first:

The IR Builder

The IR builder is a stateful object that carries a few things:

  • A counter for generating fresh virtual registers (%0, %1, ...)
  • A counter for generating fresh block labels
  • The current block we're emitting instructions into
  • A map from variable names to stack slot indices
  • The list of completed blocks (for the function we're building)

Here's how we build IR for expressions. Each expression returns the virtual register that holds its result:

src/ir_builder.rs
pub struct IrBuilder {
  vreg_counter: usize,
  block_counter: usize,
  slots: HashMap<String, usize>,
  pub current_block: IrBlock,
  pub blocks: Vec<IrBlock>,
}

impl IrBuilder {
  pub fn new() -> Self {
    IrBuilder {
      vreg_counter: 0,
      block_counter: 0,
      slots: HashMap::new(),
      current_block: IrBlock::new("entry".into()),
      blocks: vec![],
    }
  }

  fn fresh_vreg(&mut self) -> VReg {
    let r = self.vreg_counter;
    self.vreg_counter += 1;
    r
  }

  fn fresh_block(&mut self) -> String {
    let b = format!("b{}", self.block_counter);
    self.block_counter += 1;
    b
  }

  fn emit(&mut self, instr: IrInstr) {
    self.current_block.instrs.push(instr);
  }

  fn finish_block(&mut self, term: IrTerm) -> String {
    self.current_block.term = term;
    let label = self.current_block.label.clone();
    let block = std::mem::replace(
      &mut self.current_block,
      IrBlock::new(self.fresh_block()),
    );
    self.blocks.push(block);
    label
  }
}

  let label = b.current_block.label in
  let finished = b.current_block in
  b.current_block <- {
    label = fresh_block b;
    instrs = [];
    term = Ret 0;
  };
  b.blocks <- b.blocks @ [finished];
  label

Expressions to IR

Translating an expression is straightforward: we emit instructions that compute the value into a fresh virtual register:

src/ir_builder.rs (continued)
fn expr_to_ir(&mut self, expr: &Expr) -> VReg {
  match expr {
    Expr::IntLit(_, val) => {
      let r = self.fresh_vreg();
      self.emit(IrInstr::Const(r, *val));
      r
    }

    Expr::BoolLit(_, val) => {
      let r = self.fresh_vreg();
      self.emit(IrInstr::Const(r,
        if *val { 1 } else { 0 }));
      r
    }

    Expr::Ident(_, name) => {
      let slot = self.slots[name];
      let r = self.fresh_vreg();
      self.emit(IrInstr::Load(r, slot));
      r
    }

    Expr::Binary(_, op, lhs, rhs) => {
      let l = self.expr_to_ir(lhs);
      let r = self.expr_to_ir(rhs);
      let d = self.fresh_vreg();
      let instr = match op {
        BinOp::Add => IrInstr::Add(d, l, r),
        BinOp::Sub => IrInstr::Sub(d, l, r),
        BinOp::Mul => IrInstr::Mul(d, l, r),
        BinOp::Div => IrInstr::Div(d, l, r),
        BinOp::Eq  => IrInstr::Eq(d, l, r),
        BinOp::Lt  => IrInstr::Lt(d, l, r),
        BinOp::Gt  => IrInstr::Gt(d, l, r),
        BinOp::And => IrInstr::And(d, l, r),
        BinOp::Or  => IrInstr::Or(d, l, r),
      };
      self.emit(instr);
      d
    }

    Expr::Call(_, name, args) => {
      let arg_regs: Vec<VReg> = args
        .iter()
        .map(|a| self.expr_to_ir(a))
        .collect();
      let r = self.fresh_vreg();
      self.emit(IrInstr::Call(r, name.clone(), arg_regs));
      r
    }

    Expr::Cast(.., inner) => {
      // Cast is a no-op in IR — same bits, different interpretation
      self.expr_to_ir(inner)
    }
  }
}
ir_builder.ml (continued)
let rec expr_to_ir b expr =
  match expr with
  | Expr.IntLit (_, n) ->
    let r = fresh_vreg b in
    emit b (Const (r, n));
    r

  | Expr.BoolLit (_, v) ->
    let r = fresh_vreg b in
    emit b (Const (r, if v then 1L else 0L));
    r

  | Expr.Ident (_, name) ->
    let slot = Hashtbl.find b.slots name in
    let r = fresh_vreg b in
    emit b (Load (r, slot));
    r

  | Expr.Binary (_, op, lhs, rhs) ->
    let l = expr_to_ir b lhs in
    let r = expr_to_ir b rhs in
    let d = fresh_vreg b in
    let instr = match op with
      | Add -> Add (d, l, r)
      | Sub -> Sub (d, l, r)
      | Mul -> Mul (d, l, r)
      | Div -> Div (d, l, r)
      | Eq  -> Eq (d, l, r)
      | Lt  -> Lt (d, l, r)
      | Gt  -> Gt (d, l, r)
      | And -> And (d, l, r)
      | Or  -> Or (d, l, r)
    in
    emit b instr;
    d

  | Expr.Call (_, name, args) ->
    let arg_regs = List.map (expr_to_ir b) args in
    let r = fresh_vreg b in
    emit b (Call (r, name, arg_regs));
    r

  | Expr.Cast (_, _, inner) ->
    (* Cast is a no-op in IR *)
    expr_to_ir b inner

Statements and Control Flow

Statements are where things get interesting. A statement doesn't produce a value — it emits instructions and may split control flow into multiple blocks. 

fn stmt_to_ir(&mut self, stmt: &Stmt) {
  match stmt {
    Stmt::Let(_, name, _, init) => {
      if let Some(init) = init {
        let val = self.expr_to_ir(init);
        let slot = self.slots.len();
        self.slots.insert(name.clone(), slot);
        self.emit(IrInstr::Store(slot, val));
      }
    }

    Stmt::Return(_, expr) => {
      let val = self.expr_to_ir(expr);
      self.finish_block(IrTerm::Ret(val));
    }

    Stmt::If(_, cond, then_blk, else_blk) => {
      let cond_reg = self.expr_to_ir(cond);
      let then_label = self.fresh_block();
      let else_label = self.fresh_block();
      let after_label = self.fresh_block();

      // Branch on condition
      self.finish_block(
        IrTerm::BrCond(cond_reg, then_label, else_label));

      // Then block
      self.current_block.label = then_label;
      for s in then_blk {
        self.stmt_to_ir(s);
      }
      // If block didn't end with return, branch to after
      if !ended_with_return(then_blk) {
        self.finish_block(IrTerm::Br(after_label));
      }

      // Else block
      if let Some(else_blk) = else_blk {
        self.current_block.label = else_label;
        for s in else_blk {
          self.stmt_to_ir(s);
        }
        if !ended_with_return(else_blk) {
          self.finish_block(IrTerm::Br(after_label));
        }
      } else {
        // No else: branch directly to after
        self.current_block.label = else_label;
        self.finish_block(IrTerm::Br(after_label));
      }

      // After block (continue point)
      self.current_block.label = after_label;
    }

    Stmt::While(_, cond, body) => {
      let header_label = self.fresh_block();
      let body_label = self.fresh_block();
      let after_label = self.fresh_block();

      // Branch to loop header
      self.finish_block(IrTerm::Br(header_label));

      // Header: evaluate condition
      self.current_block.label = header_label;
      let cond_reg = self.expr_to_ir(cond);
      self.finish_block(
        IrTerm::BrCond(cond_reg, body_label, after_label));

      // Body
      self.current_block.label = body_label;
      for s in body {
        self.stmt_to_ir(s);
      }
      // Loop back to header
      if !ended_with_return(body) {
        self.finish_block(IrTerm::Br(header_label));
      }

      // After loop
      self.current_block.label = after_label;
    }

    Stmt::Expr(_, expr) => {
      self.expr_to_ir(expr);
    }
  }
}
let rec stmt_to_ir b stmt =
  match stmt with
  | Stmt.Let (_, name, _, init) -> (
    match init with
    | None -> ()
    | Some init_expr ->
      let val = expr_to_ir b init_expr in
      let slot = Hashtbl.length b.slots in
      Hashtbl.replace b.slots name slot;
      emit b (Store (slot, val))
  )

  | Stmt.Return (_, expr) ->
    let val = expr_to_ir b expr in
    finish_block b (Ret val)

  | Stmt.If (_, cond, then_blk, else_blk) ->
    let cond_reg = expr_to_ir b cond in
    let then_lab = fresh_block b in
    let else_lab = fresh_block b in
    let after_lab = fresh_block b in

    finish_block b (BrCond (cond_reg, then_lab, else_lab));

    (* then block *)
    b.current_block <- { b.current_block with label = then_lab };
    List.iter (stmt_to_ir b) then_blk;
    if not (ended_with_return then_blk) then
      finish_block b (Br after_lab);

    (* else block *)
    (match else_blk with
    | None ->
      b.current_block <- { b.current_block with label = else_lab };
      finish_block b (Br after_lab)
    | Some blk ->
      b.current_block <- { b.current_block with label = else_lab };
      List.iter (stmt_to_ir b) blk;
      if not (ended_with_return blk) then
        finish_block b (Br after_lab));

    (* after block *)
    b.current_block <- { b.current_block with label = after_lab }

  | Stmt.While (_, cond, body) ->
    let header_lab = fresh_block b in
    let body_lab = fresh_block b in
    let after_lab = fresh_block b in

    finish_block b (Br header_lab);

    b.current_block <- { b.current_block with label = header_lab };
    let cond_reg = expr_to_ir b cond in
    finish_block b (BrCond (cond_reg, body_lab, after_lab));

    b.current_block <- { b.current_block with label = body_lab };
    List.iter (stmt_to_ir b) body;
    if not (ended_with_return body) then
      finish_block b (Br header_lab);

    b.current_block <- { b.current_block with label = after_lab }

  | Stmt.Expr (_, expr) ->
    expr_to_ir b expr

Walkthrough: func main(i32) { return 1 + 2; }

Let's trace the IR builder on a tiny function. The AST is:

FuncDecl "main" ret=i32 params=[] └── Block └── Return └── Binary Add ├── IntLit 1 └── IntLit 2

The builder starts in b0 (entry). It processes the Return statement:

  1. Evaluate 1: emit %0 = const 1
  2. Evaluate 2: emit %1 = const 2
  3. Evaluate 1 + 2: emit %2 = add %0, %1
  4. Finish block with ret %2

Resulting IR:

func main: b0 (entry): %0 = const 1 %1 = const 2 %2 = add %0, %1 ret %2

Beautiful. The tree has been flattened into a linear sequence of instructions. Every intermediate value has a name (a virtual register). The control flow is trivial — just one block that returns.

Walkthrough: max with if/else

Now the more interesting case: 

func max(i32, a: i32, b: i32) {
  if (a > b) {
    return a;
  } else {
    return b;
  }
}

The AST for max has an If node with a comparison, two branches, and two Return statements. The IR builder transforms this into:

func max: ; Parameters live in stack slots 0 (a) and 1 (b) b0 (entry): %0 = load 0 ; a %1 = load 1 ; b %2 = gt %0, %1 ; a > b br_cond %2, b1, b2 b1: ret %0 b2: ret %1

The if became a br_cond. The two branches became separate blocks. The control flow is explicit — you can trace every path through the function just by following the arrows:

entry | v ┌─ b0 ──────┐ │ gt %0, %1 │ └──┬──────┬──┘ true │ │ false v v ┌─ b1 ┐ ┌─ b2 ┐ │ ret │ │ ret │ │ %0 │ │ %1 │ └─────┘ └─────┘
Note on parameter slots: Function parameters are stored in stack slots just like local variables. The caller pushes arguments onto the stack, and the IR builder assigns them slot numbers. Slot 0 is the first parameter, slot 1 is the second, and so on. When we generate x86-64 assembly, these will map to the actual calling convention (registers first, then stack).

The "Aha" Moment

Here's what just happened: all high-level control flow becomes just branches and labels. The if statement — with its condition, then-branch, and else-branch — collapsed into three blocks: one for the condition check, one for the then-path, one for the else-path, connected by br_cond.

The while loop — with its condition re-check at every iteration — collapsed into three blocks: one for the header (check condition, then branch into body or exit), one for the body (do work, then branch back to header), and one for after the loop.

This is the point where we stop thinking about programs as structured control flow and start thinking about them as control flow graphs. The IR is the bridge between these two worldviews. The AST is the "what the programmer wrote." The IR is the "what the machine needs."

From here, everything else becomes simpler. Want to optimize? Work on the IR. Want to generate assembly? Emit machine instructions for each IR instruction. Want to add a new backend? Just translate IR to the target architecture. The IR isolates the rest of the compiler from the source language's idiosyncrasies.

Variable Slots and Stack Layout

One detail we glossed over: how do variables become stack slots? Each local variable and parameter gets a numbered slot. The slot index is just the order in which we encountered the variable. When we generate x86-64 assembly, slot n lives at rbp - (n + 1) * 8 (or however many bytes the type requires).

Parameter slots work the same way. When a function is called, the arguments are stored in the first N slots. The return address and frame pointer are handled by the calling convention, not the IR.

Printing IR

A simple printer makes IR debugging much easier: 

pub fn print_module(module: &IrModule) {
  for func in &module.functions {
    println!("func {}:", func.name);
    for block in &func.blocks {
      println!("  {}:", block.label);
      for instr in &block.instrs {
        println!("    {}", display_instr(instr));
      }
      println!("    {}", display_term(&block.term));
    }
  }
}

fn display_instr(instr: &IrInstr) -> String {
  match instr {
    IrInstr::Const(d, v) => format!("%{} = const {}", d, v),
    IrInstr::Add(d, a, b) => format!("%{} = add %{}, %{}", d, a, b),
    IrInstr::Load(d, s) => format!("%{} = load {}", d, s),
    IrInstr::Store(s, v) => format!("store {}, %{}", s, v),
    IrInstr::Call(d, n, a) => {
      let args = a.iter()
        .map(|r| format!("%{}", r))
        .collect::<Vec<_>>()
        .join(", ");
      format!("%{} = call {}({})", d, n, args)
    }
    _ => format!("{:?}", instr),
  }
}

Why This Matters

The IR is where the compiler starts to look like a compiler and not just a fancy parser. With the IR, we've crossed the boundary from "understanding the program" to "computing the program." Every subsequent phase — optimization, register allocation, instruction selection — operates on the IR. The AST fades into the background.

IR Design Space Our IR is a custom design for this book. Real compilers use more sophisticated IRs: LLVM uses LLVM IR (static single assignment, typed), GCC uses GIMPLE and RTL, and many research compilers use Sea of Nodes or Thorn. The principles are the same: linearize the computation, make control flow explicit, provide unlimited temporaries.

Exercises

1. Short-circuit && evaluation

Our IR builder translates And to a single and IR instruction. But this evaluates both operands unconditionally. Add proper short-circuit semantics: a && b should only evaluate b if a is true. Hint: emit a br_cond + phi-like merge, or just use two blocks.

2. Phi nodes

After translating if (c) { x = 1; } else { x = 2; }, the variable x has two possible definitions. In SSA form, this requires a phi node. Add a IrInstr::Phi variant and modify the IR builder to emit phi nodes when a variable is defined in both branches of an if-else. What does the phi look like in our IR syntax?

3. Break and continue

Our language currently doesn't have break or continue in while loops. Add them. How do the stmt_to_ir methods for break and continue differ from a normal statement? You'll need to track the current loop's header and after labels in the IR builder.

4. IR verification pass

Write a verifier that checks IR for common errors: undefined virtual registers (used before defined), blocks that don't end with a terminator, terminators that reference nonexistent block labels. This catches bugs in the IR builder early.

5. Dead block elimination

After building IR, some blocks might be unreachable (especially after short-circuit &&). Write a dead block elimination pass that removes blocks not reachable from the entry block. How do you compute reachability?

What's Next

We have IR. We have virtual registers, basic blocks, and control flow graphs. Now the real work begins: we need to turn this IR into actual machine code. We need to pick real x86-64 registers, lay out a stack frame, and emit assembly instructions that a real CPU can execute. This is code generation, and it's where the compiler stops being abstract and starts being real.
