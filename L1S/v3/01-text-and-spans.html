Building a Compiler

Chapter 01: Raw Materials

The Problem: Where Did That Error Happen?

Imagine you're using our compiler and you write this:

func main(i32) {
    return 42
}

Something's wrong. We forgot the semicolon after the return statement. When our compiler detects this, what should it say?

Error: expected ';'

That's not very helpful. Where is the missing semicolon? The user has to scan the entire file to find the problem. A good compiler tells you not just what went wrong, but where. Something like:

Error: expected ';' after return statement
  ┌─ main.capi:2:11
  │
2 │     return 42
  │           ^^
  │           │
  │           expected ';' here

That's immediately useful. You can see the file, the line, the column, and the exact characters involved. To produce this, the compiler needs to track where things are in the source text.

This chapter builds the foundation that makes every error message in our compiler useful. We'll define two things: spans (ranges in a source file) and source files (the thing we read spans from).

The Core Idea: Spans

A span is just a pair of positions: where something starts and where it ends. The positions are offsets from the beginning of the file — byte counts, character counts, it doesn't matter at first.

"func main(i32) { return 42; }" ^ ^ ^ | | | 0 21 26 Span of "return": start=17, end=23 Span of "42": start=24, end=26

Why offsets? Because they're simple, cheap, and universal. Every character in a file has an offset from the start. If I tell you "this token starts at offset 17 and ends at offset 23," you can go directly to those characters in the source text and see exactly what it covers.

A span can represent any kind of source range: a single token, an expression, a function body, or even a whole file. When an error occurs, we attach the span of the thing that caused it, and the error reporter can highlight the relevant region of source code.

Byte offsets vs. character offsets For ASCII text (which our language uses), byte offsets and character offsets are the same thing. If we were handling Unicode, we'd need to think harder — a single "character" can be multiple bytes. We'll stick with byte offsets for now, which keeps things simple and matches what most real compilers do internally.

Implementing Span

Let's write the type. It's deceptively simple looking — just two numbers — but it's one of the most important types in the entire compiler.

src/span.rs
/// A range of source text, from start offset to end offset.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Span {
    pub start: usize,
    pub end: usize,
}

impl Span {
    pub fn new(start: usize, end: usize) -> Self {
        Self { start, end }
    }

    /// Combine two adjacent spans into one covering both.
    pub fn join(&self, other: &Span) -> Span {
        Span {
            start: self.start.min(other.start),
            end: self.end.max(other.end),
        }
    }

    /// Check if this span contains a given offset.
    pub fn contains(&self, offset: usize) -> bool {
        offset >= self.start && offset < self.end
    }
}
span.ml
(* A range of source text, from start offset to end offset. *)
type span = {
  start: int;
  end_: int;  (* 'end' is reserved in OCaml, so we use end_ *)
}

let make_span start end_ = { start; end_ }

(* Combine two adjacent spans into one covering both. *)
let join_span a b =
  { start = min a.start b.start;
    end_ = max a.end_ b.end_ }

(* Check if a span contains a given offset. *)
let span_contains span offset =
  offset >= span.start && offset < span.end_

There are a few subtle design choices here worth noticing:

  • Start inclusive, end exclusive. A span from 17 to 23 covers characters at offsets 17, 18, 19, 20, 21, and 22 — that's 6 characters for the word "return". This convention is the same as Rust's slices (s[17..23]) and avoids off-by-one errors when computing lengths (end - start gives the length).
  • The join method. When we have an expression like a + b, the parser might produce individual spans for a, +, and b. If there's an error in the whole expression, we want to point at all of it. join merges two spans into a containing range.
  • Why not line/column? Humans think in lines and columns, but computers think in offsets. Offsets are faster to compare and don't require looking at the source text. We can compute line/column on demand from the offset when we need it for error messages.
End is reserved in OCaml If you're reading the OCaml side, you'll notice we use end_ instead of end. That's because end is a reserved keyword in OCaml (it marks the end of a block). The trailing underscore is a common convention for keyword conflicts.

SourceFile: Bringing a File to Life

A span is just two numbers. To turn those numbers into a useful error message, we need access to the actual source text. That's what SourceFile does: it holds the filename, the complete text, and a few extra pieces of data that make it fast to answer questions like "what line is offset 42 on?"

Why Offsets (and Not Line/Column)?

You might be wondering: if humans think in lines and columns, why store offsets? There are three practical reasons, and they matter for performance and correctness:

Offsets are cheap to compare. Is position 150 before position 200? That's a single integer comparison. With line/column, you'd need to compare the line first, then break ties with the column. And because offsets are monotonic (they always increase as you scan the file), they're ideal for binary searching the line starts table.

Offsets are compact. A span is two machine words — 16 bytes on a 64-bit system. A span with line/column for both start and end would be four or six words — 32 or 48 bytes. In a compiler processing thousands of tokens and AST nodes, that difference adds up.

Offsets separate data from presentation. The compiler's internal representation should answer "where is this?" as precisely as possible (byte offset). The error reporter decides how to present that location to the user (line 5, column 12). This separation means the error reporting style can change without changing the core data structures.

This pattern — using opaque, compact identifiers internally and converting to human-readable form only when needed — shows up repeatedly in compiler design. It's the same reason we'll use numeric IDs for variables instead of storing their names everywhere.

src/span.rs (continued)
use std::rc::Rc;

/// A source file: filename, content, and cached line-start offsets.
pub struct SourceFile {
    pub name: String,
    pub source: Rc<String>,
    line_starts: Vec<usize>,
}

impl SourceFile {
    pub fn new(name: String, source: String) -> Self {
        let line_starts = Self::compute_line_starts(&source);
        Self { name, source: Rc::new(source), line_starts }
    }

    fn compute_line_starts(source: &str) -> Vec<usize> {
        let mut starts = vec![0];
        for (i, ch) in source.char_indices() {
            if ch == '\n' {
                starts.push(i + 1);
            }
        }
        starts
    }

    /// Convert an offset to (1-based line, 1-based column).
    pub fn offset_to_linecol(&self, offset: usize) -> (usize, usize) {
        let line = match self.line_starts.binary_search(&offset) {
            Ok(i) => i,
            Err(i) => i - 1,
        };
        let col = offset - self.line_starts[line];
        (line + 1, col + 1)
    }

    /// Get the line text at a given line index (0-based).
    pub fn get_line(&self, line_idx: usize) -> &str {
        let start = self.line_starts[line_idx];
        let end = if line_idx + 1 < self.line_starts.len() {
            self.line_starts[line_idx + 1] - 1  // exclude '\n'
        } else {
            self.source.len()
        };
        &self.source[start..end]
    }
}

type source_file = {
  name: string;
  source: string;
  line_starts: int array;
}

let compute_line_starts source =
  let starts = ref [0] in
  String.iteri (fun i ch ->
    if ch = '\n' then
      starts := (i + 1) :: !starts)
    source;
  Array.of_list (List.rev !starts)

let make_source_file name source = {
  name;
  source;
  line_starts = compute_line_starts source;
}

(* Convert an offset to (1-based line, 1-based column). *)
let offset_to_linecol sf offset =
  let line = match Array.binary_search sf.line_starts offset with
    | `Found i -> i
    | `Insert_at i -> i - 1
  in
  let col = offset - sf.line_starts.(line) in
  (line + 1, col + 1)

(* Get the line text at a given line index (0-based). *)
let get_line sf line_idx =
  let start = sf.line_starts.(line_idx) in
  let end_ =
    if line_idx + 1 < Array.length sf.line_starts
    then sf.line_starts.(line_idx + 1) - 1
    else String.length sf.source
  in
  String.sub sf.source start (end_ - start)

The key insight is the line_starts array. Instead of recomputing "what line is this offset on?" every time we need it (which would require scanning the file from the beginning), we compute the line start positions once when we read the file. Then offset_to_linecol answers the question in O(log n) time using a binary search.

The Rc<String> in Rust is intentional. Multiple parts of the compiler (the lexer, the parser, the error reporter) all need access to the source text. Using a reference-counted pointer means we can share the string without copying it.

A Real Error Message

Now we have everything we need to produce a good error message. Here's what happens when our compiler encounters that missing semicolon:

src/main.rs (error reporting sketch)
fn report_error(sf: &SourceFile, span: &Span, msg: &str) {
    let (line, col) = sf.offset_to_linecol(span.start);
    eprintln!("Error: {}", msg);
    eprintln!("  ┌─ {}:{}:{}", sf.name, line, col);
    eprintln!();
    let line_text = sf.get_line(line - 1);
    eprintln!("{:>4} │ {}", line, line_text);
    eprintln!("     │ {:1}",
        "",
        "^",
        indent = col - 1,
        width = (span.end - span.start).max(1),
    );
}
bin/main.ml (error reporting sketch)
let report_error sf span msg =
  let line, col = offset_to_linecol sf span.start in
  Printf.eprintf "Error: %s\n" msg;
  Printf.eprintf "  ┌─ %s:%d:%d\n\n" sf.name line col;
  let line_text = get_line sf (line - 1) in
  Printf.eprintf "%4d │ %s\n" line line_text;
  Printf.eprintf "     │ %s%s\n"
    (String.make (col - 1) ' ')
    (String.make (max (span.end_ - span.start) 1) '^')

And the output:

Error: expected ';' after return statement ┌─ main.capi:2:11 │ 2 │ return 42 │ ^^

The span points directly at the problem. The user sees exactly where the compiler got confused. The 2:11 tells you the line and column, the line text shows you the context, and the ^^ underlines the specific characters involved. This is possible only because we tracked positions from the very beginning — before we even started lexing.

There's a subtle detail worth noting: the error message doesn't point at the missing semicolon itself. How could it? The semicolon isn't there. Instead, it points at the last token before the expected semicolon — the 42 — and says "expected semicolon here." This is a common pattern in compiler error reporting: when something is missing, point at where it should have been.

A note on style You'll see that in the Rust version we use usize for offsets, which is the natural type for array indexing. In the OCaml version we use int, which is the default integer type (and on 64-bit systems, OCaml's int is 63 bits — plenty of range for any source file). The conventions differ, but the concept is identical.

What We Have Now

We've built something small but essential. Every other part of the compiler will depend on Span and SourceFile. When the lexer produces a token, it will attach a span showing where that token came from. When the parser builds an AST node, it will attach a span covering the whole node. When an error occurs, the span gives us the information we need to produce a useful, actionable message.

The best part is that this foundation is invisible when everything works. When you compile a correct program, spans are just numbers that flow through the pipeline and never get displayed. But the moment something goes wrong, they become the difference between "Error: expected ';'" and a report that tells you exactly where to look.

The Big Picture

Before moving on, let's see where our Span and SourceFile types sit in the overall compiler:

Source File (on disk) │ ▼ ┌──────────────┐ ┌──────────────────┐ │ SourceFile │────→│ Span │ │ name │ │ start: usize │ │ source │ │ end: usize │ │ line_starts │ └──────────────────┘ └──────────────┘ │ │ ▼ │ ┌──────────────────┐ │ │ Token │ └─────────────→│ kind: TokenKind │ │ span: Span │ └──────────────────┘ │ ▼ ┌──────────────────┐ │ AST Node │ │ kind: NodeKind │ │ span: Span │ └──────────────────┘

Every token and every AST node carries a span. The span always points back into the source field of the SourceFile. This means we can always answer: "what source code does this piece of the compiler correspond to?"

Exercises

1. Extend Span with convenience methods Add a method length() -> usize that returns the number of bytes covered by the span. Add a method contains_offset(offset) -> bool that checks if an offset falls within the span (we already did this — but try writing it from scratch). Add a static method from_len(start, len) -> Span that constructs a span from a start offset and a length.
2. Add line and column to Span Sometimes it's useful to cache line/column information directly in the span rather than computing it on demand. Create an alternative RichSpan struct that includes start_line, start_col, end_line, and end_col in addition to the byte offsets. Write a function that converts a regular Span to a RichSpan given a SourceFile.
3. Implement SourceFile::from_path Write a static method that reads a file from disk and constructs a SourceFile. In Rust, use std::fs::read_to_string. In OCaml, use the standard library's In_channel module. Handle the case where the file doesn't exist or isn't valid UTF-8.
4. Multi-line spans Our error reporter only handles single-line spans. Extend it to handle spans that cross multiple lines. When the span covers multiple lines, show all the affected lines with an underline on each one. This is harder than it sounds — think about how to handle the transition from one line to the next gracefully.
5. Span arithmetic Sometimes during parsing we need to create a span for a constructed node (like a binary expression a + b) without having a specific token for it. The join method handles this. But what about an empty span — a span at a single point, where start equals end? When might this be useful? Add an empty_at(offset) -> Span constructor and think about where it could appear in a compiler.
